# GitHub Actions 執行階段升級慣例

編輯 GitHub Actions 工作流程以處理由於動作執行階段所引起之棄用警告時（例如 Node.js 執行階段遷移），請使用此技能。

## 何時使用此技能

- 工作流程記錄回報動作正執行於已棄用的執行階段。
- 您正在升級 `.github/workflows/*.yml` 或 `.github/workflows/*.yaml` 中的動作版本。
- 您需要在將動作相依性現代化的同時，保持現有的工作流程行為。

## 升級規則

- 優先升級至與工作流程相容之每個動作的最新穩定**主要**版本。
- 優先使用不可變的釘選：將目標版本解析為完整的認可 SHA，並在 `uses:` 中使用該 SHA。
- 在最終建議中，請勿釘選至可變的標籤或分支（例如 `@v4` 或 `@main`）。
- 每次認可僅升級一個動作（或一個緊密相關的群組），以便於隔離失敗。
- 在升級執行階段/相依性動作時，保持現有的工作流程行為不變。

## 我們在此存放庫中追蹤的動作

當出現警告時，優先對這些群組進行執行階段審查：

- 任何位於 `actions/*` 下的第一方動作
- 特別是位於 `actions/setup-*` 下的設定動作（例如 `setup-node`、`setup-python`、`setup-dotnet`）
- 工作流程記錄中的執行階段棄用警告明確指出的任何其他動作

## 釘選模式

```yaml
steps:
  - uses: actions/checkout@8ade135a41bc03ea155e62e844d188df1ea18608 # v4.3.1
  - uses: actions/setup-node@60edb5dd545a775178f52524783378180af0d1f8 # v4.0.4
```

建議升級時，請先識別最新相容的版本，然後使用對應的認可 SHA，並可選擇加上版本註解。

## 驗證檢查清單

變更動作版本後：

1. 確保所有編輯過的工作流程仍然可以解析，並保持相同的觸發條件/權限，除非是有意變更。
2. 執行受影響的工作流程（或對等的本地建構/測試命令），並確認升級的步驟成功完成。
3. 確認發行/簽章/成品步驟在適用時仍產生預期的輸出。
4. 檢查工作流程執行記錄，以尋找任何新的棄用警告或執行階段遷移說明。

## PR 說明

包含在 PR 摘要中：

- 哪些動作已升級（從 -> 到）。
- 是否有任何動作無法移至新的主要版本，以及其原因。
- 重新執行了哪些工作流程來驗證此變更。

## 這如何與 Dependabot 互補

Dependabot 可以自動執行許多更新，但此技能在以下情況下仍然有幫助：

- 存放庫中的工作流程未啟用 Dependabot。
- 在自動更新可用之前出現了執行階段警告。
- 在動作版次變更後，工作流程需要保留行為的驗證。
