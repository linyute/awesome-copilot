## 12. 跨對話交接協定

每個衝刺對話在結束前必須執行以下操作：

1. 撰寫 `docs/sprint-N/done.md` — 建立了什麼、哪些未完成、哪些需要手動設定、更改/建立的檔案
2. 更新 PROJECT_BRIEF.md：第 7 節 (標記衝刺完成) + 第 8 節 (重寫目前狀態)
3. 使用具描述性的訊息提交所有變更：`sprint-N: <摘要>`

這是內容在對話之間延續的方式。如果跳過此步驟，下一個對話將在盲目狀態下開始，並可能覆蓋工作或重複勞動。儲存庫是共享的記憶 — 請保持其準確性。

## 13. 錯誤與修復追蹤

錯誤在儲存庫中作為 GitHub Issues 進行追蹤。這是所有團隊的單一事實來源。

**對於 QA：** 將錯誤提報為 GitHub Issues，並加上標籤 (`bug`, `severity:blocker/major/minor`)。包含：元件、重現步驟、預期 vs 實際。當未發現阻礙者時：撰寫 `docs/qa/sprint-N-signoff.md`，包含測試數量、通過率、明確的「無阻礙者」聲明。

**對於開發團隊：** 在開始工作前檢查 GitHub Issues。在磨光前先修復阻礙者 (blockers) 和重大 (majors) 問題。在提交中使用 GitHub 關閉關鍵字：`fix: 說明 (Fixes #42)`。僅供參考時，使用 `Refs #42`。

**對於 DevOps：** 提報基礎設施問題並加上標籤 `infra`。

**對於功能構想：** 新增至 `docs/ideas-backlog.md`。

## 14. 多儲存庫設定

每個團隊在儲存庫的獨立複製版本中工作。不使用工作樹 (worktrees)。每個人在自己的分支上工作，推送到 origin，建立 PR。

**團隊：**
- 製作人在 `main` (協調中心)
- 開發團隊在 `feature/sprint-N`
- QA 在 `feature/qa-N`
- DevOps 在 `feature/devops-N` (僅在需要時)

**設定：**
```bash
git clone <儲存庫> <資料夾名稱>
cd <資料夾名稱>
git checkout -b <分支名稱>
npm install
```

**分支策略：** 功能分支 → PR → 一般合併到 main。絕不直接推送到 main。絕不壓縮 (squash)。絕不對功能分支進行變更基底 (rebase) (會導致提交遺失)。
