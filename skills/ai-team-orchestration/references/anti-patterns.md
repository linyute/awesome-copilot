# 反模式

從真實的多代理專案中學到的教訓。每個反模式都至少遇到過一次，並造成了實際的問題。

## Git 與分支

| 不要 | 應該改做 | 為什麼 |
|-------|------------|-----|
| 變更功能分支的基底 (Rebase) | 一般合併 | Rebase 會重寫紀錄並導致提交遺失。當多個對話對同一個分支做出貢獻時，Rebase 會導致連鎖的退化 (regressions)。 |
| 壓縮合併 (Squash merge) PR | 一般合併 | Squash 會隱藏個別的提交，導致無法還原單一修復。 |
| 在共享分支上使用工作樹 (Worktrees) | 獨立的複製版本 (Clones) | 工作樹共享 Git 索引。平行團隊互相干擾彼此的暫存區 (staging area) 會造成混亂。 |
| 直接推送到 main 分支 | 功能分支 → PR → 合併 | 直接推送會繞過審查，且無法乾淨地還原。 |
| 強制推送 (`--force`) | 向前修復或還原 | 強制推送會破壞其他團隊可能已經拉取 (pull) 的遠端紀錄。 |

## 團隊角色

| 不要 | 應該改做 | 為什麼 |
|-------|------------|-----|
| 製作人撰寫程式碼 | 製作人僅負責計畫、合併、提報問題 | 當協調者開始撰寫程式碼時，他們會失去對大局的掌握。製作人對話中的修復常與開發團隊的工作發生衝突。 |
| 一個代理包辦所有事 | 將開發、QA、協調交給不同的代理 | 內容隔離可防止交叉污染。QA 不應擁有編輯工具。 |
| 跳過集思廣益 (Brainstorm) | 執行集思廣益 → 計畫 → 執行 | 直接進入開發會產生通用的結果。集思廣益能及早發現邊緣案例。 |
| 模糊的集思廣益提示（「你們是團隊」） | 命名每個代理並賦予鮮明的觀點 | 具有定義傾向的具名代理能產生真實的辯論。通用的提示會產生平淡的共識。 |

## 衝刺管理

| 不要 | 應該改做 | 為什麼 |
|-------|------------|-----|
| 批量的「修復所有問題」提交 | 每個修復一個提交並附上問題編號 | 批量提交導致無法追蹤修復內容。如果某個修復導致退化，你無法僅還原該修復。 |
| 僅在對話中保留錯誤 | 提報 GitHub Issues | 對話結束後，對話內容就會消失。Issues 則會在所有對話和團隊中持續存在。 |
| 跳過交接文件 (done.md) | 強制要求 done.md + 更新 PROJECT_BRIEF | 沒有交接文件，下一個對話會像盲人摸象。它可能會覆蓋已完成的工作或重複勞動。 |
| 跳過進度追蹤器 | 每個階段後更新 progress.md | 沒有進度追蹤器，內容溢出後的復原將變得不可能。新對話不知道舊對話在哪裡中斷。 |
| 以時間壓力催促 AI | 「慢慢來，正確地完成它」 | 時間壓力會讓 LLM 跳過邊緣案例、撰寫較少的測試，並產生品質較低的程式碼。「不急」能產生更好的結果。 |

## 測試與 QA

| 不要 | 應該改做 | 為什麼 |
|-------|------------|-----|
| 測試前合併 | 遊戲測試 → 提報問題 → 修復 → 合併 | 合併未經測試的程式碼會建立一個損壞的 main 分支。QA 無法對著一個變動的目標進行測試。 |
| QA 修改原始程式碼 | QA 僅提報問題，由開發團隊修復 | QA 的修復往往缺少背景資訊並引入新錯誤。關注點分離。 |
| 在未驗證的情況下關閉問題 | 開發修復 → QA 驗證 → 關閉 | 自行關閉問題會跳過驗證。該修復可能實際上並未生效。 |

## 內容與溝通

| 不要 | 應該改做 | 為什麼 |
|-------|------------|-----|
| 假設對話共享記憶體 | 檔案是共享的記憶體 | 每個對話都是一個全新的內容環境。PROJECT_BRIEF.md 和 progress.md 是唯二能留存下來的東西。 |
| 將決策保留在對話中 | 將決策寫入檔案 | 對話中做出的決策在對話關閉後就會遺失。請寫入 docs/ 或 GitHub Issues。 |
| 在團隊之間傳遞原始錯誤紀錄 | 總結並提報為 GitHub Issue | 原始紀錄會浪費內容 Token。總結：元件、步驟、預期、實際。 |
