# 企業級新手引導

將企業客戶從簽約引導至價值實現的四階段框架。目標不僅是讓系統上線 — 而是確保能持續採用，不會在第 12 週出現採用率陡降。

## 何時使用

**觸發情境：**
- 「我們該如何引導這位企業客戶？」
- 「客戶上線了但採用率很低」
- 「客戶總是在上線後 3 個月流失」
- 「從概念驗證 (POC) 到正式環境的轉銜」
- 「我該如何防止客戶在第四週消失？」
- 「客戶成功團隊的新手引導框架」

**背景：**
- 企業或中型市場交易
- 複雜的技術需求
- 涉及多方關係人
- 30 到 90 天的實作時程
- 第一年內流失的風險

---

## 核心框架

### 1. 第四週消失問題（以及如何預防）

**模式：**

第一週：開踢會議 (Kickoff) 進行得很順利，每個人都很興奮。
第二至三週：技術探索、需求收集，進展依然良好。
第四週：客戶停止回覆，會議被取消，理由是「太忙了」。

**發生了什麼事？**

你在客戶內部尚未達成共識前就開始了新手引導。

**誰在客戶內部主導這個專案？**
- 業務代表？（已經去談下一筆交易了）
- 技術支持者 (Champion)？（被日常工作淹沒了）
- 高管贊助人 (Executive sponsor)？（授權下去，不親自推動）
- 沒人主導？（**這就是為什麼他們會消失的原因**）

**框架：內部負責人驗證**

在開踢會議前，請確認以下問題的答案：

**客戶方由誰負責：**
- 參加每週專案會議？（不是被「邀請」，而是「真的會出席」）
- 排除採購、法律、安全方面的障礙？（具備權限）
- 在終端使用者中推動採用？（具備影響力）
- 在進度停滯時向上呈報？（具備與高層溝通的管道）

**如果你無法為每項職責指名道姓，那麼你就沒有專案負責人。你有的只是一份簽了名卻沒人推動的合約。**

**如何修復：**

**在業務轉交至客戶成功團隊 (CS) 期間（客戶開踢會議前）：**

業務代表必須識別出：
- 首席專案負責人（姓名，而非職稱）
- 他們的投入程度（專職還是兼任？）
- 他們的權限（他們能排除障礙嗎？）
- 他們的動機（這對他們個人有什麼好處？）

**如果沒有明確的負責人：**

先不要開始新手引導。請業務代表引薦你給財務買家：

「在我們正式啟動實作之前，我們希望確保貴司已指派合適的專案負責人。根據我們的經驗，當有人負責每週推動進度時，實作才會成功。貴團隊中哪位成員將與我們對接？」

**常見錯誤：**

假設有人會負責。請明確詢問。如果他們指派不出人選，這筆交易就有風險。

### 2. 採用率陡降（第 12 週問題）

**模式：**

第 6 週正式上線，使用量激增，大家都在慶祝。

第 8 週：使用量進入平原期。
第 10 週：使用量開始下降。
第 12 週：使用量較高峰期下降了 50%。

**為何會這樣：**

你把上線當成了終點線。**事實上，上線才是起跑線。**

**什麼能推動持續採用：**

**不是**：功能完整性、技術整合、培訓課程

**而是**：持續的價值展示、使用者成功案例、擴展使用案例

**框架：上線後的採用階段**

**第 1-6 週（實作）：** 讓系統運作起來
- 指標：技術設定完成百分比
- 負責人：技術領導人

**第 6-12 週（初始採用）：** 讓大家開始使用
- 指標：活躍使用者人數、使用頻率
- 負責人：推廣部門 (Enablement) / 開發者關係團隊 (DevRel)

**第 12-26 週（持續採用）：** 證明持續的價值
- 指標：使用案例擴展、團隊滲透率
- 負責人：客戶成功團隊 (CS)

**第 26 週以上（擴展）：** 在帳號內成長
- 指標：新團隊、新使用案例、升級觸發點
- 負責人：客戶經理 (AE) + CS

**多數團隊遺漏的交接：**

第 6 週（上線）→ 第 12 週（持續採用）

大多數客戶成功團隊在慶祝上線後就轉向下一位客戶了。**這正是流失種子埋下的時刻。**

**第 6-12 週該做什麼：**

**第 7 週**：首份價值報告
「這是貴團隊在第一週達成的成果：[特定指標]。第 12 週的理想目標應該是：[目標]。」

**第 9 週**：使用者成功案例
「[團隊名稱] 的 [使用者姓名] 本週節省了 [X 小時/減少了 Y 個錯誤]。這是他們的使用方式。」

**第 11 週**：使用案例擴展對話
「你們目前將產品用於 [首要使用案例]。像你們這樣的團隊也會將其用於 [相鄰使用案例]。有興趣探索嗎？」

**常見錯誤：**

衡量「上線完成率」而非「持續活躍使用量」。上線不代表成功，第 26 週的留存採用才是成功。

### 3. 前置新手引導：成功建立在初次通話前

**模式：**

大多數新手引導的失敗都可以追溯到開踢會議前的缺口。

**容易遺漏的部分：**

**業務未充分告知 CS 資訊：**
- 不清楚交易驅動因素
- 關係人動向不明
- 技術需求僅靠假設

**未識別內部專案負責人：**
- CS 主動聯繫，無人回應
- 會議邀約對象錯誤
- 決策無法落實

**客戶時間表不切實際：**
- 他們想在 2 週內上線
- 技術設定最少需要 6 週
- 從第一天起期望就未對齊

**框架：開踢會議前檢查清單**

在排定開踢會議前，請驗證：

**帳號情報：**
- [ ] 業務交接完成（交易驅動因素、關係人、技術需求）
- [ ] 已檢閱過去的互動記錄（演示筆記、提案、電子郵件）
- [ ] 已對應組織結構（團隊規模、彙報關係）
- [ ] 已記錄使用案例（首要 + 未來）

**內部準備：**
- [ ] 建立內部 Slack 頻道 (#account-[客戶名稱])
- [ ] CRM 中的帳號計畫已更新
- [ ] 專案計畫範本已備妥
- [ ] 已指派角色（CSM 負責人、技術負責人、高管贊助人）

**客戶就緒度：**
- [ ] 專案負責人已指名道姓（而非只是「他們的 DevRel 團隊」）
- [ ] 雙方高管贊助人已確認
- [ ] 時間表具備現實性（他們的目標 vs 你們的標準時程）
- [ ] 已記錄已知障礙（採購、安全、法律）

**時間表驗證：**
- [ ] 考慮技術需求後，客戶的上線日期是否合理
- [ ] 內部頻寬充足（未超載）
- [ ] 已識別相依性（SSO、整合、資料遷移）

**決策標準：**

僅在四個部分皆驗證後才排定開踢會議。若存在缺口，請在接觸客戶前向業務或高管贊助人反應。

**常見錯誤：**

在內部資訊不明確的情況下開始新手引導。這會導致混亂、錯過截止日期，並侵蝕客戶信心。

### 4. 四階段新手引導流程

**第一階段：開踢會議 (Week 1)**

**目標**：對齊目標、時間表、成功指標

**出席者**：高管贊助人 + 專案負責人 + 技術負責人

**議程**：
1. 介紹與角色說明 (5 min)
2. 高層對齊策略目標 (5 min)
3. 定義成功：「3/6/12 個月後的成功看起來像什麼？」(10 min)
4. 時間表與里程碑 (5 min)
5. 會議節奏（每週專案團隊會議、每月高層檢討）(5 min)
6. 下一步（技術探索通話、成功計畫檢閱）(5 min)

**交付物**：24 小時內寄出開踢會議紀要，包含成功指標、時間表、後續會議安排。

**第二階段：探索與規劃 (Week 2-3)**

**目標**：理解技術環境、對應使用案例、規劃推行策略

**三個並行工作流**：

**工作流 1：技術探索**
- 目前基礎架構（地端、雲端、混合雲）
- 現有的工具與整合
- 安全/合規需求
- 時間限制

**工作流 2：成功規劃**
- 使用案例優先級（從最高價值的開始）
- 定義成功指標（如何衡量採用率）
- 識別培訓需求（誰需要什麼）

**工作流 3：技術設定**
- SSO / 身份識別設定
- 所需的整合
- 資料遷移（若適用）
- 識別試點 (Pilot) 小組

**交付物**：客戶成功計畫文件，包含使用案例、指標、時間表、里程碑。

**第三階段：實作 (Week 4-6)**

**目標**：部署至試點小組、驗證使用案例、為全面推行做準備

**三個並行軌道**：

**軌道 1：管理與設定**
- SSO 設定完成
- 整合上線
- 資料遷移完成（若適用）

**軌道 2：使用者賦能**
- 針對試點小組的培訓課程
- 分享文件資料
- 排定辦公時間 (Office hours)

**軌道 3：試點與回饋**
- 試點小組開始使用產品
- 每週收集回饋
- 優先分級並解決問題

**交付物**：上線就緒檢查清單完成，試點小組驗證完畢。

**第四階段：正式上線與持續成功 (Week 6+)**

**目標**：全面推行、維持採用、擴展使用案例

**第 6-8 週（推行）：**
- 向所有團隊全面推行
- 排定培訓課程
- 提供支援管道（Slack、電子郵件、辦公時間）

**第 8-12 週（價值展示）：**
- 首份價值報告（第 7 週）
- 分享使用者成功案例（第 9 週）
- 使用案例擴展對話（第 11 週）

**第 12-26 週（持續採用）：**
- 每月業務檢討 (MBR)
- 採用率追蹤（活躍使用者、頻率、使用案例）
- 識別擴展機會

**常見錯誤：**

將正式上線視為完成。第四階段才是決定留存率的關鍵。

---

### 5. 「並行軌道」反模式

**模式：**

大多數新手引導團隊採取 **循序漸進** 的工作方式：
1. 技術設定（第 1-2 週）
2. 然後培訓（第 3-4 週）
3. 最後試點（第 5-6 週）

**總計時間：6 週**

**更有效的做法：並行軌道**

**同時** 進行技術設定、培訓和試點：
- 第 1 週：技術探索 + 識別試點小組 + 排定培訓
- 第 2 週：SSO 設定 + 試點小組培訓 + 試點開始
- 第 3 週：整合設定 + 更廣泛的培訓 + 試點回饋

**總計時間：3 週**

**為何並行奏效：**

1. 縮短價值實現時間 (time-to-value)
2. 保持客戶參與感（每週都有進展）
3. 及早發現障礙（試點小組在全面推行前就能暴露出問題）

**如何執行：**

為每個軌道指定明確的負責人：
- 軌道 1（管理）：技術負責人
- 軌道 2（賦能）：培訓 / DevRel 負責人
- 軌道 3（試點）：CSM + 試點小組支持者

每週跨軌道對齊，以發現相依性與障礙。

**常見錯誤：**

等待「完美的技術設定」完成後才開始試點。讓試點小組及早開始使用，即使設定還不完美也沒關係。他們的回饋會讓全面推行變得更好。

## 決策樹

### 我應該開始客戶新手引導嗎？

```
業務是否已指名道姓識別出專案負責人？
├─ 否 → 在開踢會議前先識別出專案負責人
└─ 是 → 繼續...
    │
    考量典型部署情況，他們的時間表是否具備現實性？
    ├─ 否 → 在開踢會議前重新校準期望
    └─ 是 → 繼續...
        │
        你具備內部產能嗎？
        ├─ 否 → 延後開踢會議或爭取更多資源
        └─ 是 → 進行開踢會議
```

### 這次新手引導是否有風險？

```
客戶是否回應會議邀約？
├─ 否 → 第四週消失，向上呈報至高管贊助人
└─ 是 → 繼續...
    │
    他們是否在完成自己的行動項？
    ├─ 否 → 缺乏專案負責人，識別誰來推動此事
    └─ 是 → 繼續...
        │
        試點小組是否在使用產品？
        ├─ 否 → 試點小組找錯人或產品未解決痛點
        └─ 是 → 按計畫進行
```

### 正式上線後採用率是否能維持？

```
活躍使用者在第 6 週到第 12 週之間是否有成長？
├─ 是 → 採用狀況健康
└─ 否 → 繼續...
    │
    活躍使用者是否在下降？
    ├─ 是 → 採用率陡降，立即介入
    └─ 否 (進入平原期) → 有風險，開始價值展示
```

## 常見錯誤

**1. 在內部對齊前就開始客戶新手引導**
   - 浪費前 2-3 週時間、造成混亂、損害公信力。

**2. 未預先識別真實的專案負責人**
   - 到第四週才發現問題，必須重來或導致交易停滯。

**3. 在沒有技術需求的情況下過度承諾時間表**
   - 在實作中期才發現障礙，錯過截止日期。

**4. 缺乏內部溝通中心**
   - 決策無法跨團隊傳達，導致重工。

**5. 將上線視為專案結束**
   - 第 12 週出現採用率陡降，帳號面臨風險。

**6. 循序漸進而非並行軌道**
   - 實作時間延長一倍，客戶失去動能。

**7. 上線後未持續追蹤指標**
   - 太晚發現採用問題，無法挽救帳號。

## 快速參考

**開踢會議前驗證：**
- [ ] 業務交接完成（交易驅動因素、關係人、需求）
- [ ] 已在客戶方指名道姓識別出專案負責人
- [ ] 時間表具備現實性（他們的目標 vs 標準部署）
- [ ] 內部角色已指派 (CSM, 技術, 高管贊助人)

**開踢會議議程 (30-45 min)：**
1. 介紹 (5 min)
2. 高層對齊 (5 min)
3. 定義成功 (10 min)
4. 時間表與里程碑 (5 min)
5. 會議節奏 (5 min)
6. 下一步 (5 min)

**採用率追蹤 (Week 6-26)：**
- 第 7 週：首份價值報告
- 第 9 週：使用者成功案例
- 第 11 週：使用案例擴展對話
- 第 13 週：首次每月業務檢討
- 第 26 週：擴展就緒度評估

**四個階段：**
1. 開踢會議 (Week 1)：對齊
2. 探索與規劃 (Week 2-3)：計畫
3. 實作 (Week 4-6)：部署至試點
4. 正式上線與持續成功 (Week 6+)：推行、價值展示、擴展

**紅燈訊號：**
- 客戶在第 4 週無回應 → 缺乏專案負責人
- 試點小組在第 5 週未使用產品 → 組別錯誤或使用案例錯誤
- 活躍使用者在第 8-12 週下降 → 採用率陡降正在形成
