# 事故事後分析 (Incident Post-Mortem)

引導團隊在生產事故後編寫一份結構化、無責備的事後分析。產出是一份文件，旨在建立共同理解、在不指責的情況下識別根本原因，並產生具體的行動項以防止再次發生。

## 無責備原則 (Blameless Principle)

系統會失敗，人不會。目標是理解事故是「如何」發生的，而不是「誰」造成的。避免使用如「X 忘記了」、「Y 應該知道」之類的語言。使用「系統沒有」、「流程缺乏」、「警報沒有觸發」。

## 何時使用

- 生產停機或服務降級已解決
- 發生了重大虛驚事件 (如果稍後才發現就會演變成事故)
- 發生了面向用戶的錯誤、數據丟失或違反 SLA
- 團隊希望在背景資訊淡忘前捕捉學習成果

**不適用於：** 在預發布環境 (staging) 中發現的小錯誤、計劃中的維護窗口或沒有學習價值的事故。

## 輸入要求

在編寫事後分析之前收集這些詳細信息。詢問任何缺失的部分：

### 事故元數據 (Incident Metadata)
- 事故標題 (簡短且具描述性)
- 檢測日期和時間 (帶時區)
- 解決日期和時間
- 嚴重程度 / 影響級別 (P1–P4 或同等)
- 事故指揮官 / 值班負責人

### 影響 (Impact)
- 受影響的服務和系統
- 面向用戶的影響 (錯誤、緩慢、完全停機)
- 估計受影響的用戶數量
- 數據丟失或損壞 (是/否，範圍)
- 違反 SLA/SLO (是/否，程度)

### 時間線事件 (Timeline Events)
要重建的關鍵時刻：
- 第一個症狀出現
- 警報觸發 (或手動注意到)
- 值班人員收到傳呼 / 宣布事故
- 開始調查
- 根本原因確定
- 應用緩解措施
- 確認完全解決
- 發送客戶溝通 (如果有)

### 促成因素 (Contributing Factors)
詢問團隊：「是什麼讓這件事變得比本應有的情況更糟？」——而不是「誰失敗了」。例如：
- 警報閾值過高 / 警報未觸發
- 操作手冊 (Runbook) 缺失或過時
- 部署缺乏用於回滾的特性標誌 (Feature Flag)
- 監控未覆蓋此故障模式
- 值班移交遺漏了背景資訊

## 流程 (Process)

### 步驟 1 — 收集元數據
如果用戶未提供完整的事故詳情，請按部分詢問。在獲得標題、時間、嚴重性、受影響服務和至少一個粗略的時間線之前，不要開始編寫。

### 步驟 2 — 重建時間線
與用戶合作建立精確的按時間順序的時間線。對於每個事件：
- 精確時間 (首選 UTC)
- 發生了什麼 (系統事件或人為操作)
- 誰觀察到了或採取了行動
- 日誌 / 警報 / Slack 消息的鏈接 (如果可用)

標記間隙：「我們不知道 14:32 到 14:47 之間發生了什麼 —— 值得檢查日誌。」

### 步驟 3 — 根本原因分析 (Root Cause Analysis)
迭代使用 **5 Whys**：

```
為什麼用戶看到 500 錯誤？
→ API pod 正在崩潰循環中。

為什麼它們在崩潰循環中？
→ 超過了內存限制。

為什麼超過了限制？
→ 一個新查詢正在將完整的結果集加載到內存中。

為什麼這在部署前沒有被發現？
→ 負載測試僅覆蓋了 p50 情況，而不是高基數帳戶。

為什麼負載測試僅覆蓋 p50？
→ 我們沒有針對大帳戶的測試固定裝置 (test fixtures)。
```

當你達到一個可以修復的系統/流程間隙時停止。最後一個「為什麼」應該指向一個行動項。

區分：
- **根本原因 (Root cause)** — 最深層的系統性間隙 (一個或兩個)
- **促成因素 (Contributing factors)** — 使情況變得更糟但不是根本原因的條件

### 步驟 4 — 影響量化
幫助用戶做到精確：
- 持續時間：從檢測到解決 (不是從症狀開始到解決 —— 將這些分開)
- 高峰錯誤率與正常基準的對比
- 受影響流量的百分比
- 收入 / 業務影響 (如果已知)

### 步驟 5 — 行動項 (Action Items)
針對每個根本原因和促成因素，生成至少一個行動項：

| # | 行動 | 負責人 | 截止日期 | 優先級 |
|---|--------|-------|----------|----------|
| 1 | 為大於 10k 記錄的帳戶添加負載測試固定裝置 | @eng-team | 2026-07-01 | 高 |
| 2 | 將內存警報閾值從 90% 降低到 75% | @platform | 2026-06-23 | 高 |
| 3 | 為內存 OOM pod 添加操作手冊 | @on-call-rotation | 2026-06-30 | 中 |

行動項必須有負責人 (一個人，而不是一個團隊) 和截止日期。像「改進監控」這樣模糊的行動是不可接受的 —— 將它們分解為具體的交付物。

### 步驟 6 — 編寫文件
使用下面的模板編寫完整的事後分析。保存到 `docs/postmortems/YYYY-MM-DD-<slug>.md`。

## 常見錯誤 (Common Mistakes)

| 錯誤 | 修復 |
|---------|-----|
| 「Bob 忘記檢查配置」 | 「部署清單不包含配置驗證」 |
| 根本原因是「人為錯誤」 | 繼續詢問「為什麼」—— 人為錯誤始終是症狀 |
| 沒有負責人的行動項 | 每個項目都需要一個具名的個人，而不是團隊 |
| 憑記憶重建時間線 | 在編寫前檢查日誌、警報、Slack、PagerDuty |
| 將「改進監控」作為行動 | 具體說明：哪個服務、哪個指標、什麼閾值、何時完成 |
| 幾週後才編寫事後分析 | 在 48–72 小時內、背景資訊尚新鮮時編寫 |
