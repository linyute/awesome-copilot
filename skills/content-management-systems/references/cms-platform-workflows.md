# CMS 平台工作流程 (CMS Platform Workflows)

本參考文件將高階平台對照圖與技能保持緊密連結，以便代理程式能快速選擇正確的銜接點 (seam)。

## 平台對照圖 (Platform Map)

| 平台 | 主要擴充表面 | 媒體與資產慣例 | 附註 |
| --- | --- | --- | --- |
| WordPress | 佈景主題、外掛程式、範本組件、Hook | 佈景主題資產位於作用中的佈景主題內；經撰寫的媒體位於 uploads 風格的路徑下 | 非常適合處理範本階層、分類、自訂欄位以及本地/靜態匯出工作流程 |
| WooCommerce | WordPress 佈景主題與外掛程式，加上產品/型錄擴充功能 | 與 WordPress 相同的基礎慣例，產品圖片為經撰寫的媒體 | 先將其視為 WordPress 處理，再套用商務特定的內容與管理員規則 |
| Shopify | 佈景主題、Liquid 區段、區塊、應用程式、Metafields | 佈景主題資產與代管的商店媒體是分開的關注點 | 當資料必須在重新設計中保留時，優先選擇應用程式或 Metafield 銜接點，而非僅限佈景主題的臨時做法 |
| Wix | 網站建立器介面、應用程式、內容集合、自訂元素 | 代管的媒體庫加上編輯器管理的資產 | 傾向於對編輯器安全的變更，避免假設具備檔案系統層級的存取權限 |
| Squarespace | 範本、程式碼注入、內容集合、商務設定 | 透過平台管理的代管資產庫 | 預期會有較窄的擴充點與較強的代管限制 |
| Drupal | 佈景主題、模組、內容類型、視圖、分類 | 受管理檔案與佈景主題資產是分開的 | 非常適合結構化內容、企業級工作流程以及遷移繁重的變更 |
| Joomla | 範本、模組、元件、外掛程式 | 受管理媒體加上範本擁有的資產 | 範本與擴充功能之間有類似的拆分；注意路由與內容元件的界線 |
| HubSpot CMS Hub | 佈景主題、模組、範本、無伺服器函式、CRM 連結內容 | 代管的檔案管理員加上佈景主題資產 | 內容、行銷與 CRM 關注點緊密耦合 |
| Webflow | 設計器 (Designer)、CMS 集合、元件、內嵌 (embeds)、有限的程式碼匯出 | 代管資產與 CMS 集合媒體 | 匯出限制很重要；區分哪些內容在匯出後能存活，哪些依賴代管的 CMS 功能 |
| Adobe Experience Manager | 元件、範本、內容片段 (Content Fragments)、體驗片段 (Experience Fragments)、工作流程 | DAM 管理的資產加上元件資源 | 企業治理、撰寫工作流程與內容片段模型驅動了大部分變更 |

## 媒體規則準則 (Media Rule of Thumb)

- 佈景主題擁有的圖片屬於佈景主題或範本套件。
- 使用者撰寫的圖片屬於平台的上傳或媒體庫流程。
- 如果專案同時支援兩者，請在設定與程式碼路徑中保持它們的區別。

## 通用 CMS 職責地圖 (Generic CMS Responsibility Map)

大多數 CMS 程式碼庫將行為分組為相同的幾種職責。在任何專案中定位負責的銜接點時，請將此作為檢查清單：

- 執行階段組裝與請求路由
- 佈景主題或範本系統與共享的範本協助工具
- 管理員與編輯器控制器及其視圖範本
- 內容、分類與設定的持久化（存放庫、模型、結構描述/遷移）
- 內容轉換公用程式（Markdown、短代碼、區塊渲染器）
- 靜態匯出、部署或渲染管線進入點

先將專案對應至這些職責，然後進行最小程度的變更以保留平台的結構。
