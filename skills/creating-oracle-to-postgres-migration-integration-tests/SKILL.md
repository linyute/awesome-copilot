# 為 Oracle 到 PostgreSQL 遷移建立整合測試 (Creating Integration Tests for Oracle-to-PostgreSQL Migration)

為單一目標專案中的資料存取成品產生整合測試案例。測試在針對 Oracle 或 PostgreSQL 執行時會驗證行為的一致性。

## 前提條件

- 測試專案必須已經存在且可編譯 (單獨建構)。
- 在撰寫測試前，請閱讀現有的基礎測試類別和種子管理員慣例。

## 工作流程 (Workflow)

```
測試建立：
- [ ] 步驟 1：探索測試專案慣例
- [ ] 步驟 2：識別可測試的資料存取成品
- [ ] 步驟 3：建立種子資料
- [ ] 步驟 4：撰寫測試案例
- [ ] 步驟 5：檢閱決定性 (Determinism)
```

**步驟 1：探索測試專案慣例**

閱讀基礎測試類別、種子管理員和專案檔案，以理解繼承模式、交易管理和種子檔案慣例。

**步驟 2：識別可測試的資料存取成品**

僅限於目標專案範圍。列出與資料庫互動的資料存取方法 — 存放庫、DAO、預存程序呼叫器、查詢建構器。

**步驟 3：建立種子資料**

- 遵循現有專案的種子檔案位置和命名慣例。
- 盡可能重複使用現有的種子檔案。
- 避免使用 `TRUNCATE TABLE` — 保持現有的資料庫資料完整。
- 不要提交種子資料；測試在會回滾的交易中執行。
- 確保種子資料不會與其他測試發生衝突。
- 在斷言依賴種子資料之前，先載入並驗證它。

**步驟 4：撰寫測試案例**

- 繼承自基礎測試類別，以獲得自動交易建立/回滾。
- 斷言邏輯輸出（列、欄、計數、錯誤類型），而非平台特定的訊息。
- 斷言特定的預期值 — 當可從種子資料獲得具體值時，絕不要僅斷言值為非 Null 或非空。
- 避免測試不存在的程式碼路徑或斷言無法發生的行為。
- 避免針對相同方法的測試之間出現多餘的斷言。

**步驟 5：檢閱決定性 (Determinism)**

重新檢查針對非 Null 值的每個斷言。確認每個斷言對於種子資料都是具決定性的。修正任何依賴於測試控制之外的資料庫狀態的斷言。

## 關鍵限制

- **Oracle 是黃金來源** — 測試捕捉 Oracle 的預期行為。
- **與資料庫無關的斷言** — 斷言中不包含平台特定的錯誤訊息或語法。
- **僅針對 Oracle 播種** — 測試專案稍後將遷移到 PostgreSQL。
- **限定於單一專案** — 不要為目標專案以外的成品建立測試。
