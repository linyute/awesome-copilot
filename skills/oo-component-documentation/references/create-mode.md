# 建立模式 (Create mode)

當輸入是元件原始路徑或使用者要求從程式碼產生新文件時，請使用此工作流。

## 輸入處理

- 接受代表元件的單一檔案或資料夾路徑。
- 如果輸入是資料夾，則分析該資料夾中相關的原始檔案以及附近的支援檔案。
- 如果輸入是單一檔案，則將其視為主要元件進入點，並根據需要檢查相鄰檔案以瞭解協作對象和介面。

## 建立特定需求

- 將新文件儲存在 `/docs/components/`。
- 將檔案命名為 `[元件名稱]-documentation.md`。
- 將 `component_path` 設定為使用者提供的原始路徑。
- 將 `date_created` 設定為目前日期。
- 僅當目標區域的儲存庫慣例期望在建立時設定時，才設定 `last_updated`。
- 僅當可以可靠地推斷時，才填入選填的 Metadata，例如 `version`、`owner` 和 `tags`。

## 建立特定分析重點

- 從程式碼結構中確定主要元件名稱和職責。
- 識別初始系統上下文、範圍邊界、相依項、設計模式和協作模型。
- 從實際的公開介面建立介面表和使用範例。
- 建立一個 Mermaid 圖表，為第一次閱讀文件的讀者介紹元件結構、相依項和資料流。

## 建立特定輸出指南

- 使用來自 `../assets/documentation-template.md` 的完整章節結構。
- 將介紹撰寫為關於元件功能和存在原因的新鮮概觀。
- 偏好完整的章節而非佔位符；如果資訊不可用，請用簡短的限制備註標記該章節，而不要留下範本文字。
- 僅當程式碼或儲存庫歷史記錄中有先前版本或遷移問題的證據時，才包含變更歷史記錄或遷移備註。
