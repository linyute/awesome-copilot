# 更新模式 (Update mode)

當輸入是現有的文件 Markdown 檔案或使用者要求重新整理現有的元件文件時，請使用此工作流。

## 輸入處理

- 先閱讀現有的文件，以瞭解目前的結構、術語和任何 front matter Metadata。
- 當存在 `component_path` front matter 時，從中擷取元件原始路徑。
- 如果缺少 `component_path`，則從文件內容和周圍的儲存庫結構中推斷元件路徑。
- 當文件與程式碼不一致時，以目前的實作為基準事實。

## 更新特定需求

- 保留現有的文件檔案作為輸出目標。
- 保留 `date_created`。
- 將 `last_updated` 更新為目前日期。
- 當版本歷史記錄和擁有權 Metadata 仍然準確時，請予以保留；僅當程式碼或儲存庫證據顯示其已變更時才進行重新整理。
- 在現有組織仍然有用時保留它，但確保內容與共享範本的預期保持一致。

## 更新特定分析重點

- 將現有文件與目前程式碼進行比較，以識別過時的 API、過時的圖表、重新命名的相依項和已變更的使用模式。
- 突顯明顯的破壞性變更、已過時的功能或重大的架構轉移。
- 重新整理方法表、範例、圖表、相依項清單和品質屬性備註，以匹配目前的實作。
- 僅在元件已成長或目前文件遺漏維護者現在需要的資訊時，才新增缺少的章節。

## 更新特定輸出指南

- 保留現有文件中具建設性的編輯選擇，但移除過時或誤導性的內容。
- 更新範例，使其在概念上能針對目前的 API 形狀進行編譯。
- 重新整理 Mermaid 圖表，而不是用通用的佔位符取代它們。
- 當更新揭示了重要的相容性或行為變更時，新增遷移備註或變更歷史記錄。
