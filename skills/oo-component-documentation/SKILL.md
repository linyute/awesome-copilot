# 物件導向元件文件 (OO Component Documentation)

透過分析目前的實作，為物件導向元件建立新文件，或更新現有的元件文件檔案。

## 首先確定模式

在編寫任何內容之前選擇工作流：

1. 當使用者提供現有的 Markdown 文件檔案、指向文件路徑或明確要求重新整理或修改現有文件時，請使用 **更新模式 (update mode)**。遵循 [references/update-mode.md](references/update-mode.md)。
2. 當使用者提供原始檔案或資料夾、指向元件路徑或要求從程式碼產生文件時，請使用 **建立模式 (create mode)**。遵循 [references/create-mode.md](references/create-mode.md)。
3. 如果同時提供了程式碼和現有的文件檔案，請將現有的文件檔案視為輸出目標，並將目前的原始程式碼視為基準事實。
4. 如果要求不明確，請盡可能從路徑類型推斷模式：現有的 Markdown 文件檔案表示更新模式；原始程式碼/元件路徑表示建立模式。

## 文件標準

- DOC-001：遵循 C4 模型文件層級 (Context, Containers, Components, Code)
- DOC-002：與 Arc42 軟體架構文件範本保持一致
- DOC-003：符合 IEEE 1016 軟體設計描述 (Software Design Description) 標準
- DOC-004：使用敏捷文件 (Agile Documentation) 原則 (僅保留能增加價值的必要文件)
- DOC-005：以開發人員和維護者為主要受眾

## 共享分析指南

- ANA-001：確定主要元件邊界，以及輸入代表的是資料夾、檔案還是現有的文件目標
- ANA-002：檢查原始程式碼檔案中的類別結構、繼承、組合和介面
- ANA-003：識別設計模式、架構決策和整合點
- ANA-004：記錄或重新整理公開 API、介面、相依項和使用模式
- ANA-005：擷取方法參數、傳回值、非同步行為、例外狀況和生命週期考量
- ANA-006：評估效能、安全性、可靠性、可維護性和延展性特徵
- ANA-007：推斷資料流、協作模式以及與周圍元件的關係
- ANA-008：使文件立足於實作；避免虛構程式碼不支援的行為

## 共享輸出需求

- 使用 [assets/documentation-template.md](assets/documentation-template.md) 作為權威的章節檢查表和基礎結構。
- 輸出保持為 Markdown 格式，具有清晰的標題階層、有用的表格、範例程式碼區塊，以及在需要視覺化架構關係時使用 Mermaid 圖表。
- 使範例和介面描述符合目前的實作，而非使用通用的佔位符。
- 僅包含可由程式碼、專案結構、設定或明確說明的假設所支持的資訊。
- 當原始程式碼涵蓋範圍不完整時，請明確記錄該限制而非猜測。

## 語言特定最佳化

- LNG-001：**C#/.NET** - async/await、相依性注入、設定、釋放 (disposal)、選項模式
- LNG-002：**Java** - Spring 框架、註釋 (annotations)、例外處理、套件封裝、相依性注入
- LNG-003：**TypeScript/JavaScript** - 模組、非同步模式、型別、npm 相依項、執行階段邊界
- LNG-004：**Python** - 套件、虛擬環境、型別提示、測試、相依性管理

## 錯誤處理

- ERR-001：如果路徑不存在，說明預期的路徑，以及該技能需要原始路徑還是現有的文件檔案
- ERR-002：如果未找到相關的原始檔案，記錄此缺口並建議下一個要檢查的可能位置
- ERR-003：如果無法從要求中推斷出文件目標，說明歧義並僅在無法推斷時才詢問缺少的路徑
- ERR-004：如果程式碼使用非標準架構模式，請記錄該自訂方法，而非強行將其納入通用模式
- ERR-005：如果原始程式碼存取不完整，請根據可用證據繼續進行，並清楚標示任何不受支援的章節

## 工作流

1. 確定任務是建立模式還是更新模式。
2. 檢查元件實作以及瞭解其公開介面範圍和內部結構所需的任何相關檔案。
3. 使用 [assets/documentation-template.md](assets/documentation-template.md) 作為共享的文件支架。
4. 套用 [references/create-mode.md](references/create-mode.md) 或 [references/update-mode.md](references/update-mode.md) 中的模式特定規則。
5. 產生或修訂文件，使圖表、範例、介面、相依項和品質屬性反映目前的實作。

## 完成標準

- 文件清楚地識別了元件目的、架構、介面、實作細節、使用模式、品質屬性和參考資料。
- Front matter 欄位對於所選模式是準確的。
- 範例和圖表與實作相符。
- 任何未知事項、缺口或假設都已明確標出。
