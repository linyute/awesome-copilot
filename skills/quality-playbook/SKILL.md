# 品質劇本產生器 (Quality Playbook Generator)

**當此技能啟動時，在執行任何操作前請先顯示此橫幅：**

```
Quality Playbook v1.2.0 — by Andrew Stellman
https://github.com/andrewstellman/
```

為特定的程式碼庫量身打造完整的品質系統。不同於僅從原始碼機械式運作的測試虛設產出器，此技能會先探索專案 — 了解其領域、架構、規格以及失敗歷史 — 然後根據其發現產出植基於實際情況的品質劇本。

## 為什麼需要這個

大多數軟體專案都有測試，但很少有品質「系統」。測試檢查程式碼是否運作。品質系統則回答更難的問題：對於這個特定專案，「正確運作」意味著什麼？有哪些測試無法捕捉到的失敗方式？每個開發人員 (無論是人類還是 AI) 在接觸這段程式碼之前應該知道什麼？

如果沒有品質劇本，每個新貢獻者 (以及每個新的 AI 工作階段) 都必須從零開始 — 猜測什麼是重要的，撰寫看起來不錯但無法捕捉真實錯誤的測試，並重新發現幾個月前就已發現並修正的失敗模式。品質劇本讓門檻變得明確、持久且可繼承。

## 此技能產出的內容

這六個檔案共同構成了一個可重複的品質系統：

| 檔案 | 目的 | 為什麼重要 | 是否執行程式碼？ |
|------|---------|----------------|----------------|
| `quality/QUALITY.md` | 品質憲法 — 涵蓋率目標、適合性案例、防止劇場行為 | 每個 AI 工作階段都會先閱讀此檔案。它告訴他們「足夠好」是什麼意思，以免他們瞎猜。 | 否 |
| `quality/test_functional.*` | 源自規格的自動化功能測試 | 安全網。測試與規格說「應該發生什麼」掛鉤，而不僅是程式碼「做了什麼」。使用專案的語言：`test_functional.py` (Python)、`FunctionalSpec.scala` (Scala)、`functional.test.ts` (TypeScript)、`FunctionalTest.java` (Java) 等。 | **是** |
| `quality/RUN_CODE_REVIEW.md` | 帶有防止幻覺發現之安全防護的程式碼檢閱協定 | 缺乏安全防護的 AI 程式碼檢閱會產出自信但錯誤的發現。安全防護 (行號、聲稱前先執行 grep、閱讀主體) 通常能提升準確性。 | 否 |
| `quality/RUN_INTEGRATION_TESTS.md` | 整合測試協定 — 跨所有變體的端對端管線 | 單元測試通過了，但系統是否真的能與實際的外部服務端對端協作？ | **是** |
| `quality/RUN_SPEC_AUDIT.md` | 三人議會多模型規格稽核協定 | 沒有任何單一 AI 模型能捕捉所有問題。三個具有不同盲點的獨立模型能捕捉到任何單一模型都會遺漏的缺陷。 | 否 |
| `AGENTS.md` | 為任何處理此專案的 AI 工作階段提供引導脈絡 | 「必讀」檔案。若無此檔案，AI 工作階段的前一小時都會浪費在搞清楚狀況。 | 否 |

外加輸出目錄：`quality/code_reviews/`、`quality/spec_audits/`、`quality/results/`。

關鍵的交付成果是功能測試檔案 (根據專案語言與測試框架慣例命名)。Markdown 協定是為人類與 AI 代理程式提供的文件。功能測試則是自動化的安全網。

## 如何使用

將此技能指向任何程式碼庫：

```
Generate a quality playbook for this project.
```

```
Update the functional tests — the quality playbook already exists.
```

```
Run the spec audit protocol.
```

如果品質劇本已經存在 (`quality/QUALITY.md`、功能測試等)，請先閱讀現有檔案，然後根據驗證階段的自我檢查基準進行評估。不要假設現有檔案是完整的 — 請將其視為起點。

## 第一階段：探索程式碼庫 (先不要動筆)

花費第一階段的時間來了解專案。品質劇本必須植基於這個特定的程式碼庫 — 而非通用的建議。

**為什麼要先探索？** AI 產出的品質劇本最常見的失敗是產出通用的內容 — 適用於任何專案的涵蓋率目標、描述理論性失敗的案例、練習語言內建功能而非專案程式碼的測試。探索可以防止這種情況，強制要求每個產出都參考真實的事物：特定的函式、特定的結構描述、特定的防禦性程式碼模式。如果您無法指出某個東西在程式碼中的位置，您就是在瞎猜 — 而瞎猜會產出沒人信賴的品質劇本。

**針對大型程式碼庫的縮放：** 對於擁有超過 ~50 個原始碼檔案的專案，不要嘗試閱讀所有內容。將探索重點放在 3-5 個核心模組上 (處理主要資料流、最複雜邏輯以及最容易出錯之操作的模組)。閱讀各子系統中具代表性的測試，而非讀完每個測試檔案。目標是對重要事物進行深度探索，而非對所有內容進行廣度掃描。

### 步驟 0：詢問開發歷史

在探索程式碼之前，請詢問使用者一個問題：

> 「您是否有開發此專案時導出的 AI 聊天紀錄 — Claude 導出、Gemini 匯出、ChatGPT 導出、Claude Code 逐字稿或類似內容？如果有，請將我指向該資料夾。這些聊天紀錄中的設計討論、事故報告以及品質決策將使產出的品質劇本大幅提升。」

如果使用者提供了聊天紀錄資料夾：

1. **先掃描索引檔案。** 尋找名為 `INDEX*`、`CONTEXT.md`、`README.md` 或類似的導覽輔助檔案。如果存在，請閱讀它 — 它會告訴您那裡有什麼以及如何找到。
2. **搜尋與品質相關的對話。** 尋找提到以下關鍵字的訊息：品質 (quality)、測試 (testing)、涵蓋率 (coverage)、錯誤 (bugs)、失敗 (failures)、事故 (incidents)、崩潰 (crashes)、驗證 (validation)、重試 (retry)、恢復 (recovery)、規格 (spec)、適合性 (fitness)、稽核 (audit)、檢閱 (review)。同時搜尋專案名稱。
3. **擷取設計決策與事故歷史。** 最有價值的內容是：(a) 事故報告 — 哪裡出錯了、影響了多少記錄、如何被偵測到，(b) 設計討論 — 為什麼選擇特定方法、拒絕了哪些替代方案，(c) 品質框架討論 — 涵蓋率目標、測試哲學、模型檢閱經驗，(d) 跨模型回饋 — 不同 AI 模型對程式碼產生分歧的地方。
4. **不要嘗試閱讀所有內容。** 聊天紀錄可能非常龐大。使用索引找到最相關的對話，然後在其中搜尋品質相關內容。10 分鐘的有針對性搜尋勝過 2 小時的窮舉式閱讀。

這些背景資訊非常珍貴。一份開發者討論過「為什麼我們選擇這個並發模型」或「那次我們在生產環境遺失了 1,693 筆記錄」的聊天紀錄，能將通用的案例轉化為具權威性的案例。

如果使用者沒有聊天紀錄，請照常進行 — 技能仍可運作，只是脈絡較少。

### 步驟 1：辨識領域、技術棧與規格

閱讀 README、現有文件以及建構組態 (`pyproject.toml` / `package.json` / `Cargo.toml`)。回答以下問題：

- 此專案是做什麼的？ (一句話說明。)
- 使用什麼語言與關鍵相依項？
- 它與哪些外部系統進行通訊？
- 主要產出是什麼？

**尋找規格說明。** 規格是功能測試的事實來源。按此順序搜尋：根目錄下的 `AGENTS.md`/`CLAUDE.md`、`specs/`、`docs/`、`spec/`、`design/`、`architecture/`、`adr/`，然後是根目錄下的 `.md` 檔案。記錄路徑。

**如果不存在正式的規格文件**，技能仍可運作 — 但您需要從其他來源彙整需求。依優先順序如下：

1. **詢問使用者** — 他們通常知道需求，即使沒有寫下來。
2. **README 與內嵌文件** — 許多專案將需求嵌入在其 README、API 文件或程式碼註釋中。
3. **現有的測試套件** — 測試是隱含的規格。如果一個測試斷言 `process(x) == y`，那就是一項需求。
4. **型別特徵標記與驗證規則** — 結構描述、型別標註與驗證器定義了系統接受與拒絕的內容。
5. **從程式碼行為推論** — 作為最後手段，閱讀程式碼並推論其預期行為。將這些在 QUALITY.md 中標記為「推論出的需求」，並標註供使用者確認。

當使用非正式需求時，請為每個案例與測試加上**需求標記**，包含信心等級與來源：

- `[Req: 正式 — README §3]` — 由人類撰寫在規格文件中的內容。具備權威性。
- `[Req: 使用者確認 — 「必須處理空輸入」]` — 使用者陳述但不在正式文件中的內容。視為具備權威性。
- `[Req: 推論 — 源自 validate_input() 行為]` — 從程式碼推導出的內容。標註供使用者檢閱。

在 QUALITY.md 案例、功能測試文件與規格稽核發現中，請使用此精確的標記格式。它能清晰區分哪些需求是權威性的，哪些需要驗證。

### 步驟 2：映射架構

列出原始碼目錄及其用途。閱讀主要進入點，追蹤執行流程。辨識：

- 3-5 個主要子系統
- 資料流 (輸入 → 處理 → 產出)
- 最複雜的模組
- 最脆弱的模組

### 步驟 3：閱讀現有測試

閱讀現有的測試檔案 — 對於中小型專案閱讀全部，大型專案則從每個子系統挑選具代表性的樣本。辨識：測試數量、涵蓋率模式、缺口，以及任何涵蓋率劇場 (看起來不錯但無法捕捉真實錯誤的測試)。

**至關重要：記錄匯入模式。** 現有測試如何匯入專案模組？每種語言都有其慣例 (Python 的 `sys.path` 操作、Java/Scala 的套件匯入、TypeScript 的相對路徑或別名、Go 的套件/模組路徑、Rust 的 `use crate::` 或 `use myproject::`)。您「必須」在您的功能測試中使用完全相同的模式 — 弄錯了這點會導致每個測試都因匯入/解析錯誤而失敗。詳見 `references/functional_tests.md` 中的「匯入模式」章節，其中包含六種語言的矩陣。

**辨識整合測試執行器。** 尋找能對實際外部服務 (API、資料庫等) 執行端對端系統練習的腳本或測試檔案。記錄其模式 — 您在撰寫 `RUN_INTEGRATION_TESTS.md` 時會需要用到。

### 步驟 4：閱讀規格說明

逐章逐節閱讀每份規格文件。針對每一節，詢問：「這陳述了什麼可測試的需求？」記錄沒有對應測試的規格需求 — 這些是功能測試必須填補的缺口。

如果使用推論出的需求 (源自測試、型別或程式碼行為)，請使用步驟 1 中定義的 `[Req: 等級 — 來源]` 格式為每一項加上信心等級標記。推論出的需求會進入 QUALITY.md 案例，並應在第四階段標註供使用者檢閱。

### 步驟 4b：閱讀函式特徵標記與真實資料

在撰寫任何測試之前，您必須確切了解每個函式是如何被呼叫的。針對您在步驟 2 中辨識出的每個模組：

1. **閱讀實際的函式特徵標記** — 參數名稱、型別、預設值。不要從使用脈絡中猜測 — 請閱讀函式定義與任何說明文件 (Python docstrings、Java/Scala Javadoc/ScalaDoc、TypeScript 型別標註、Go godoc 註釋、Rust doc 註釋與型別特徵標記)。
2. **閱讀真實資料檔案** — 如果專案具有項目檔案、fixture 檔案、組態檔案或範例資料 (在 `pipelines/`、`fixtures/`、`test_data/`、`examples/` 中)，請閱讀它們。您的測試 fixture 必須與真實資料形狀完全匹配。
3. **閱讀現有的測試 fixture** — 現有測試如何建立測試資料？複製它們的模式。如果它們使用特定鍵值建構組態字典，請使用完全相同的鍵。
4. **檢查函式庫版本** — 檢查專案的相依項清單 (`requirements.txt`、`build.sbt`、`package.json`、`pom.xml`/`build.gradle`、`go.mod`、`Cargo.toml`) 以了解實際可用的內容。不要撰寫依賴於未安裝之函式庫功能的測試。如果相依項可能缺失，請使用測試框架的跳過機制 — 參見 `references/functional_tests.md` 的「函式庫版本感知」章節以取得框架特定的範例。

記錄一個**函式呼叫對映表**：針對您計畫測試的每個函式，寫下其名稱、模組、參數以及回傳內容。此對映表能防止最常見的測試失敗：使用錯誤的引數呼叫函式。

### 步驟 5：尋找「骷髏」 (防禦性模式)

這是最重要的一步。搜尋防禦性程式碼模式 — 每一個模式都是過去失敗或已知風險的證據。

**為什麼這很重要：** 開發人員不會無緣無故撰寫 `try/except` 區塊、空值檢查或重試邏輯。每一段防禦性程式碼的存在都是因為有人曾經被燙傷過。圍繞著 JSON 解析的 `try/except` 表示生產環境曾發生過格式錯誤的 JSON。對欄位進行空值檢查表示該欄位曾發生過不應有的缺失。這些模式是程式碼庫在輕聲訴說其失敗歷史。每一項都會轉化為一個適合性案例以及邊界測試。

**請閱讀 `references/defensive_patterns.md`** 以獲取系統性搜尋方法、grep 模式，以及如何將發現轉化為適合性案例與邊界測試。

最低門檻：每個核心原始碼檔案至少找出 2-3 個防禦性模式。如果您找出的較少，表示您只是在略讀 — 請閱讀函式主體，而不僅是特徵標記。

### 步驟 5a：追蹤狀態機

如果專案有任何形式的狀態管理 — 狀態欄位、生命週期階段、工作流程階段、模式旗標 — 請完整地追蹤狀態機。這能捕捉到防禦性模式分析會遺漏的一類錯誤：存在但未被處理的狀態。

**如何尋找狀態機：** 在模型、列舉或常數中搜尋狀態/狀態欄位 (例如：`status`, `state`, `phase`, `mode`)。搜尋在執行動作前檢查狀態的防護 (例如：`if status == "running"`, `match self.state`)。搜尋狀態轉換 (對狀態欄位的賦值)。

**針對您找到的每個狀態機：**

1. **列舉所有可能的狀態。** 閱讀列舉、常數，或對每個賦值給狀態欄位的地方執行 grep 搜尋。列出所有狀態。
2. **針對每個狀態的使用者** (UI 處理程序、API 端點、控制流防護)，檢查：它是否處理了所有可能的狀態？一個沒有具意義預設值的 `switch`/`match`，或一個未涵蓋所有狀態的 `if/elif` 鏈，就是一個缺口。
3. **針對每個狀態轉換**，檢查：您是否能到達每個狀態？是否存在您可以進入但永遠無法離開的狀態？是否存在阻礙了本應可用之操作的狀態？
4. **將缺口記錄為發現。** 一個允許「執行中」狀態執行動作 X 卻不允許「卡住」狀態執行的防護，如果使用者需要在卡住的程序上執行動作 X，那就是一個真實的錯誤。一個進入終端狀態卻從未觸發清理的程序，也是一個真實的錯誤。

**為什麼這很重要：** 狀態機缺口產生的錯誤在正常運作時是不可見的，但在壓力或邊緣條件下會浮上水面 — 正好是在您最需要系統運作的時候。一個處於「卡住」狀態卻無法被終止的批次處理器，或者一個在所有工作完成後卻從不自我終止的監視程式，或者一個拒絕恢復「待處理」執行的 UI，都是狀態處理不完整的徵兆。這些錯誤不會出現在防禦性模式分析中，因為程式碼並非在防禦它們 — 程式碼只是單純地完全沒有處理它們。

### 步驟 5b：映射結構描述型別

如果專案具有驗證層 (Python 的 Pydantic 模型、JSON Schema、TypeScript 介面/Zod 結構描述、Java Bean Validation 註釋、Scala case class 編解碼器)，現在請閱讀結構描述定義。針對每個您找到防禦性模式的欄位，記錄結構描述接受與拒絕的內容。

**請閱讀 `references/schema_mapping.md`** 以獲取對映格式，以及為什麼這對於撰寫有效的邊界測試至關重要。

### 步驟 6：辨識品質風險 (程式碼 + 領域知識)

每個專案都有不同的失敗剖面。此步驟使用**兩個來源** — 不僅是程式碼探索，還有您關於類似系統中什麼會出錯的訓練知識。

**從程式碼探索中**，詢問：
- 此專案中「隱性的錯誤」看起來像什麼？
- 哪些外部相依項可能會在無預警的情況下變更？
- 哪裡看起來簡單，實際上卻很複雜？
- 橫切關注點隱藏在哪裡？

**從領域知識中**，詢問：
- 「這類系統通常會出什麼問題？」 — 如果是批次處理器，思考崩潰恢復、等冪性 (idempotency)、隱性資料遺失、狀態損壞。如果是網頁應用程式，思考驗證邊緣案例、競爭條件 (race conditions)、輸入驗證繞過。如果處理隨機性或統計數據，思考種子設定 (seeding)、相關性、分佈偏差。
- 「什麼會產出看起來正確但實際上錯誤的產出？」 — 這是最危險的一類錯誤：通過所有檢查但被細微損壞的產出。
- 「在 10 倍規模下會發生什麼 1 倍時不會發生的事？」 — 塊邊界 (chunk boundaries)、速率限制、逾時連鎖、記憶體壓力。
- 「當此程序在最糟糕的時刻被終止會發生什麼？」 — 寫入中途、交易中途、批次提交中途。
- 「使用者在提交一個不可逆或昂貴的操作之前，需要什麼資訊？」 — 預先執行的成本估算、範圍確認 (特別是當擴展或分發會成倍增加工作量時)、資源警告。如果系統會默默地讓使用者承擔數小時的處理或顯著的成本，卻不向他們展示他們即將執行的操作，那就是缺失的安全防護。搜尋啟動長效程序、提交批次作業或觸發擴展/分發的操作 — 並檢查使用者在到達不可逆轉點之前，是否看到了帶有實際數據的預覽、估算或確認。
- 「當一個長效程序結束時 — 它真的停止了嗎？」 — 輪詢迴圈、監視程式、背景執行緒以及運行到完成的守護程序都應該有明確的終止條件。如果迴圈只檢查「還有更多工作嗎？」而從不檢查「所有工作都完成了嗎？」，它在完成後仍會永遠運行。這在批次處理器和佇列取用端中尤為常見。

根據這些知識產生現實的失敗案例。您不需要觀察到這些失敗 — 您從訓練中知道它們會發生在此類系統上。將它們撰寫為帶有具體數量與後果的**架構漏洞分析**。將每一項架構成「此架構允許以下失敗模式」 — 而非虛構一個事故報告。使用具體的數字讓嚴重性不容置疑：「如果在 10,000 筆記錄的批次處理中途 `save_state()` 崩潰且缺乏原子重新命名模式，會留下一個損壞的狀態檔案 — 下一次執行會遇到 JSONDecodeError 且在無手動介入的情況下無法恢復。」然後將其植基於您探索過的實際程式碼：「閱讀 persistence.py 第 ~340 行 (save_state)：驗證暫存檔 + 重新命名模式。」

## 第二階段：產生品質劇本

現在開始撰寫這六個檔案。針對每一個檔案，遵循下方的結構，並諮詢相關的參考檔案以獲取詳細指導。

**為什麼要產出六個檔案而不僅是測試？** 測試捕捉迴歸，但無法防止新類別的錯誤。品質憲法 (`QUALITY.md`) 在未來的工作階段開始撰寫程式碼之前，就先告訴他們什麼是「正確」。協定 (`RUN_*.md`) 提供了結構化的檢閱、整合測試與規格稽核流程，能產出可重複的結果 — 而非讓品質取決於 AI 想要檢查什麼。這些檔案結合在一起，建立了一個品質系統，其中每一塊都強化了其他部分：QUALITY.md 中的案例對映到功能測試檔案中的測試，這些測試由整合協定驗證，並由三人議會進行稽核。

### 檔案 1：`quality/QUALITY.md` — 品質憲法

**請閱讀 `references/constitution.md`** 以獲取完整範本與範例。

品質憲法包含六個章節：

1. **目的** — 品質對此專案意味著什麼，植基於戴明 (建構出來而非檢驗出來)、朱蘭 (適合使用)、克勞斯比 (品質是免費的)。具體套用這些原則：對於「此系統」而言，「適合使用」意味著什麼？不是「測試通過」，而是實際的維運需求。
2. **涵蓋率目標** — 將每個子系統對映到一個目標值，其理由應參考實際風險。每個目標都必須有一個植基於具體案例的「為什麼」 — 若無此理由，未來的 AI 工作階段會爭論降低目標。
3. **防止涵蓋率劇場** — 根據您在探索期間所見內容，提供專案特定的虛假測試範例。(原因：AI 產出的測試常會墊高涵蓋率數字，卻無法捕捉真實錯誤 — 例如斷言匯入成功、字典具有特定鍵值，或模擬物件回傳了它被配置要回傳的內容。明確指出這一點可以阻止此類行為。)
4. **適合使用案例** — 核心部分。每個案例都記錄一個現實的失敗模式，並附帶程式碼參考與驗證方法。目標是每個核心模組有 2 個以上的案例 — 中型專案通常總共 8-10 個，小型專案較少，複雜專案較多。品質比數量更重要：一個精確捕捉到真實架構漏洞的案例，比三個通用的案例更有價值。(原因：涵蓋率百分比告訴您有多少程式碼執行過，而非它們是否執行正確。一個系統可能擁有 95% 的涵蓋率，卻仍會默默遺失記錄。適合性案例定義了「正確執行」在具體條款下的實際含義，無人能對此產生異議。)
5. **AI 工作階段品質規範** — 每個 AI 工作階段都必須遵守的規則
6. **人類閘門** — 需要人類判斷的事項

**案例語調至關重要。** 將「發生了什麼」撰寫為具有具體數量、連鎖後果與偵測難度的架構漏洞分析 — 而非抽象的規格說明。「因為 `save_state()` 缺乏原子重新命名模式，在 10,000 筆記錄的批次處理中途崩潰會留下一個損壞的狀態檔案 — 下一次執行會遇到 JSONDecodeError 且無法恢復。在大規模情況下，此風險會導致隱性遺失 1,693 筆以上的記錄，且無偵測機制。」讀到這段文字的 AI 工作階段將不會爭論標準。利用您對類似系統的知識產生現實的失敗案例，然後將其植基於您探索過的實際程式碼。案例來自程式碼探索與關於此類系統常見錯誤的領域知識。

每個案例的「如何驗證」都必須對映到功能測試檔案中至少一個測試。

### 檔案 2：功能測試

**這是最重要的交付成果。** 請閱讀 `references/functional_tests.md` 以獲取完整指南。

將測試組織成三個邏輯群組 (類別、describe 區塊、模組，或測試框架使用的任何形式)：

- **規格需求** — 每個可測試的規格章節一個測試。每個測試的文件皆引用其驗證的規格需求。
- **適合性案例** — QUALITY.md 每個案例一個測試。1:1 對映，命名須匹配。
- **邊界與邊緣案例** — 每個來自步驟 5 的防禦性模式一個測試。

關鍵規則：
- **完全匹配現有的匯入模式。** 閱讀現有測試如何匯入專案模組並照做。弄錯這點會導致所有測試失敗。
- **在呼叫前閱讀每個函式的特徵標記。** 閱讀實際的 `def` 行 — 參數名稱、型別、預設值。閱讀專案中的真實資料檔案以了解資料形狀。不要猜測函式參數或 fixture 結構。
- **不使用佔位符測試。** 每個測試都必須匯入並呼叫實際的專案程式碼。如果測試本體是 `pass` 或斷言是微不足道的 (例如 `assert isinstance(x, list)`)，請將其刪除。不執行專案程式碼的測試會墊高計數並產生虛假信心。
- **測試計數啟發式** = (可測試的規格章節) + (QUALITY.md 案例) + (防禦性模式)。對於中型專案 (5-15 個原始碼檔案)，通常會產出 35-50 個測試。顯著偏少表示遺漏了需求或探索過淺。顯著偏多沒關係，只要每個測試都有意義即可 — 不要為了湊數字而填充。
- **跨變體啟發式：~30%** — 如果專案處理多種輸入類型，目標是讓約 30% 的測試能參數化涵蓋所有變體。確切的百分比並不重要，重要的是確保每個橫切屬性都在所有變體中得到測試。
- **測試結果而非機制** — 斷言規格說「應該發生什麼」，而非程式碼如何「實作它」。
- **使用符合結構描述的變動** — 邊界測試必須使用結構描述接受的數值 (來自步驟 5b)，而非其拒絕的數值。

### 檔案 3：`quality/RUN_CODE_REVIEW.md`

**請閱讀 `references/review_protocols.md`** 以獲取範本。

關鍵章節：引導檔案、對映至架構的焦點區域，以及這些強制的安全防護：

- 行號是強制的 — 無行號，無發現
- 閱讀函式主體，不僅是特徵標記
- 若不確定：標記為問題 (QUESTION)，而非錯誤 (BUG)
- 聲稱缺失前先執行 grep
- 「不要」建議風格變更 — 僅標記不正確的事項

**第二階段：迴歸測試。** 在檢閱產出「錯誤 (BUG)」發現後，在 `quality/test_regression.*` 中撰寫能重現每項錯誤的迴歸測試。每個測試都應該在目前實作中失敗，以確認錯誤是真實存在的。報告結果為一個確認表 (BUG CONFIRMED / FALSE POSITIVE / NEEDS INVESTIGATION)。參見 `references/review_protocols.md` 以獲取完整的迴歸測試協定。

### 檔案 4：`quality/RUN_INTEGRATION_TESTS.md`

**請閱讀 `references/review_protocols.md`** 以獲取範本。

必須包含：安全限制、執行前檢查、具備特定通過標準的測試矩陣、執行體驗 (Execution UX) 章節，以及結構化的報告格式。涵蓋成功路徑、跨變體一致性、產出正確性以及元件邊界。

**所有指令必須使用相對路徑。** 產出的協定應包含「工作目錄」章節，說明所有指令皆從專案根目錄使用相對路徑執行。切勿產生使用 `cd` 切換至絕對路徑的指令 — 這在不同機器或目錄執行協定時會出錯。請使用 `./scripts/`、`./pipelines/`、`./quality/` 等。

**包含一個執行體驗 (Execution UX) 章節。** 當有人指示 AI 代理程式「執行整合測試」時，代理程式需要知道如何呈現其工作。協定應指定三個階段：(1) 在執行任何操作前以帶編號的表格顯示計畫，(2) 在每個測試執行時報告一行的進度更新 (`✓`/`✗`/`⧗`)，(3) 顯示帶有通過/失敗計數與建議的摘要表。參見 `references/review_protocols.md` 的「執行體驗 (Execution UX)」章節以獲取範本與範例。若無此項，代理程式會傾印原始輸出或保持沉默 — 這兩者都沒用。

**此協定必須練習真實的外部相依項。** 如果專案與 API、資料庫或外部服務通訊，整合測試協定必須對這些服務執行真實的端對端執行 — 而不僅是本機驗證檢查。圍繞專案實際的執行模式與外部相依項設計測試矩陣。在探索期間尋找 API 金鑰、提供者抽象以及現有的整合測試腳本，並以此為基礎進行建構。

**從程式碼衍生品質閘門，而非通用的檢查。** 探索期間閱讀驗證規則、結構描述列舉與產生邏輯。將它們轉化為每條管線的品質檢查，包含特定的欄位與可接受的數值範圍。「所有單位皆已驗證」是不夠的 — 協定必須驗證領域特定的正確性。

**撰寫並行執行腳本，而非僅是描述。** 將執行任務分組，以便獨立的執行 (不同的提供者) 能夠同時運行。包含帶有 `&` 和 `wait` 的實際 bash 指令。每次針對一個提供者執行一個任務，以避免速率限制。

**針對專案校準單位數量。** 閱讀 `chunk_size` 或對等的組態。使用足夠的單位來跨越至少 2 個區塊，並足以驗證分佈檢查。通常整合測試建議使用 10-30 個。

**深度的執行後驗證。** 不要停留在「程序已完成」。驗證日誌檔案、資訊清單狀態、產出資料的存在性、範例記錄的內容，以及任何現有的品質檢查腳本 — 針對每一次執行。

**尋找並使用現有的驗證工具。** 搜尋現有的驗證產出品質的腳本 (例如：`integration_checks.py`、驗證函式、品質閘門函式)。如果存在，請從協定中呼叫它們。如果專案具有 TUI 或儀表板，請在執行後核取清單中包含 TUI 驗證指令 (例如：`--dump` 旗標)。

**撰寫品質閘門前請先建構「欄位參考表」。** 這是確保協定準確性最重要的步驟。AI 模型即使在閱讀過結構描述後，也常會信誓旦旦地寫錯欄位名稱 — `document_id` 變成了 `doc_id`，`sentiment_score` 變成了 `sentiment`，`float 0-1` 變成了 `int 0-100`。修正方法是程序性的：**在撰寫每行表格內容之前，「立即」重新閱讀每個結構描述檔案。** 不要依賴對話稍早讀過的內容 — 您的欄位名稱記憶會隨著數千個權杖而產生偏移。從檔案內容逐字複製欄位名稱。包含結構描述中的「所有」欄位。參見 `references/review_protocols.md` 的「欄位參考表」章節以獲取完整流程與格式。請勿跳過此步驟 — 它能防止協定中單一最常見的不準確之處。

### 檔案 5：`quality/RUN_SPEC_AUDIT.md` — 三人議會

**請閱讀 `references/spec_audit.md`** 以獲取完整協定。

三個獨立的 AI 模型針對規格稽核程式碼。為什麼需要三個？因為每個模型都有不同的盲點 — 在實務上，不同的稽核員會發現不同的問題。交叉比對能捕捉到任何單一模型都會遺漏的缺陷。

協定定義了：一個帶有安全防護的可直接複製貼上的稽核提示詞、專案特定的詳查區域、一個篩選流程 (按信心等級合併發現結果)，以及修正執行規則 (按子系統分小批次進行，而非特大號提示詞)。

### 檔案 6：`AGENTS.md`

如果 `AGENTS.md` 已經存在，請更新它 — 不要取代。新增一個「品質文件 (Quality Docs)」章節，指向所有產出的檔案。

如果是從零開始建立：包含專案描述、設定指令、建構與測試指令、架構概覽、關鍵設計決策、已知的特性，以及品質文件指標。

## 第三階段：驗證

**為什麼需要驗證階段？** AI 產出的結果可能看起來很專業，但實際上卻有細微的錯誤。參考了未定義 fixture 的測試會報告 0 個失敗但有 16 個錯誤 — 而「0 個失敗」聽起來像是成功。整合協定可能會列出實際結構描述中不存在的欄位名稱。驗證階段在使用者發現這些問題前先將其捕捉，這非常重要，因為對自動產出之品質劇本的信賴是非常脆弱的 — 一個錯誤的欄位名稱就會瓦解對其他所有內容的信心。

### 自我檢查基準

在宣告完成之前，請檢查每一項基準。**請閱讀 `references/verification.md`** 以獲取包含所有 13 項基準的完整核取清單。

關鍵的檢查項目：

1. **測試計數** 接近啟發式目標 (規格章節 + 案例 + 防禦性模式)
2. **案例涵蓋率** — 案例測試數量與 QUALITY.md 案例數量相符
3. **跨變體涵蓋率** — 約 30% 的測試參數化涵蓋所有輸入變體
4. **邊界測試計數** ≈ 步驟 5 的防禦性模式計數
5. **斷言深度** — 大多數斷言檢查的是數值，而非僅檢查存在性
6. **層級正確性** — 測試斷言的是結果 (規格說什麼)，而非機制 (程式碼如何實作)
7. **變動有效性** — 每個 fixture 變動都使用了來自步驟 5b 的符合結構描述的數值
8. **所有測試皆通過 — 零失敗且零錯誤。** 使用專案的測試執行器執行測試套件 (Python: `pytest -v`、Scala: `sbt testOnly`、Java: `mvn test`/`gradle test`、TypeScript: `npx jest`、Go: `go test -v`、Rust: `cargo test`) 並檢查摘要。來自缺失 fixture、匯入失敗或未解析相依項的錯誤，都算作損壞的測試。如果您看到設定錯誤，表示您忘記建立 fixture/設定檔案，或引用了未定義的測試輔助工具。
9. **現有測試未損壞** — 新檔案沒有破壞任何既有功能。
10. **整合測試品質閘門是根據「欄位參考表」撰寫的。** 驗證您是否在撰寫品質閘門之前「重新閱讀」了每個結構描述檔案來建立欄位參考表，且品質閘門中的每個欄位名稱都是從該表複製而來的 — 而非憑記憶。如果您跳過了該表，請現在回去補做。

如果任何一項基準失敗，請先修正後再繼續。

## 第四階段：呈現、探索、改進 (互動式)

在產生與驗證之後，清晰地呈現結果，並讓使用者控制接下來的操作。此階段包含三個部分：一個可掃描的摘要、按需深入探討，以及一個改進路徑選單。

**不要跳過此階段。** 來自第一至第三階段的自主產出是一個堅實的起點，但使用者需要了解產出了什麼、探索對他們而言重要的部分，並選擇如何改進。只有當專案的擁有者信任並理解品質劇本時，它才是有用的。僅丟出六個檔案而不加說明，會產出沒人閱讀的構件。

### 第一部分：摘要表

呈現一個使用者可在 10 秒內掃描完畢的表格：

```
這是我產出的內容：

| 檔案 | 作用 | 關鍵指標 | 信心等級 |
|------|-------------|------------|------------|
| QUALITY.md | 品質憲法 | 10 個案例 | ██████░░ 高 — 植基於程式碼，但案例是推論出的，非源自真實事故 |
| 功能測試 | 自動化測試 | 47 個通過 | ████████ 高 — 所有測試皆通過，35% 為跨變體 |
| RUN_CODE_REVIEW.md | 程式碼檢閱協定 | 8 個焦點區域 | ████████ 高 — 源自架構 |
| RUN_INTEGRATION_TESTS.md | 整合測試協定 | 9 次執行 × 3 個提供者 | ██████░░ 中 — 品質閘門需要調優閾值 |
| RUN_SPEC_AUDIT.md | 三人議會稽核 | 10 個詳查區域 | ████████ 高 — 已包含安全防護 |
| AGENTS.md | AI 工作階段引導 | 已更新 | ████████ 高 — 事實陳述 |
```

根據您實際產出的內容調整表格 — 檔案名稱、指標與信心等級會因專案而異。信心等級欄位最為重要：它告訴使用者該在哪裡投注注意力。

**信心等級：**
- **高 (High)** — 直接源自程式碼、規格或結構描述。不太需要修訂。
- **中 (Medium)** — 合理的推論，但可能錯誤。從使用者輸入中獲益良多。
- **低 (Low)** — 最佳推測。絕對需要使用者輸入才能發揮作用。

在表格之後，新增一個「快速入門」區塊，提供可直接複製的提示詞來執行每個構件：

```
若要使用這些構件，請開啟新的 AI 工作階段並嘗試以下提示詞之一：

• 執行程式碼檢閱：
  「閱讀 quality/RUN_CODE_REVIEW.md 並按照其指示檢閱 [模組或檔案]。」

• 執行功能測試：
  「[測試執行指令，例如：pytest quality/ -v, mvn test -Dtest=FunctionalTest 等]」

• 執行整合測試：
  「閱讀 quality/RUN_INTEGRATION_TESTS.md 並按照其指示執行。」

• 開始規格稽核 (三人議會)：
  「閱讀 quality/RUN_SPEC_AUDIT.md 並按照其指示，使用 [模型名稱] 執行。」
```

根據實際專案調整測試執行指令與模組名稱。重點是提供使用者可複製貼上的提示詞 — 而非做什麼的描述，而是他們會輸入的實際文字。

在快速入門區塊後，新增一行：

> 「您可以詢問我關於其中任何一項的細節 — 例如：『向我展示案例 3』或『帶我了解整合測試矩陣』。」

### 第二部分：按需深入探討

當使用者詢問特定項目時，提供專注的摘要 — 而非整個檔案，而是關鍵決策以及您不確定的地方。範例：

- **「告訴我關於案例 4 的資訊」** → 展示案例文字，說明其來源 (哪個防禦性模式或領域知識)，並標註您推論出的內容與您確知的內容。
- **「展示整合測試矩陣」** → 展示執行組別，說明並行策略，並註明哪些品質閘門是源自結構描述，哪些是猜測的。
- **「功能測試是如何運作的？」** → 展示三個測試群組，說明與規格與案例的對映關係，並標註您信心最低的測試。

在準備好改進任何內容之前，使用者可能會進行多次深入探討。這沒關係 — 讓他們按照自己的步調探索。

### 第三部分：改進選單

在使用者看過摘要 (並選用深入探討細節) 後，呈現改進選項：

> 「有三種方法可以讓這份劇本變得更好：」
>
> **1. 檢閱並強化個別項目** — 挑選任何案例、測試或協定章節，我將帶領您逐步檢閱。適用於：收緊特定品質閘門、修正推論出的案例、新增缺失的邊緣案例。
>
> **2. 引導式問答** — 我會向您提出 3-5 個針對我無法從程式碼推論之內容的特定問題：事故歷史、預期分佈、成本容忍度、模型偏好。適用於：填補知識缺口，讓案例更具權威性。
>
> **3. 檢閱開發歷史** — 將我指向導出的 AI 聊天紀錄 (Claude, Gemini, ChatGPT 導出, Claude Code 逐字稿)，我將從中挖掘應包含在 QUALITY.md 中的設計決策、事故報告與品質討論。適用於：將案例植基於真實專案歷史而非推論。
>
> 「您可以按任何順序執行這些選項的任何組合。您想從哪一個開始？」

### 執行各項改進路徑

**路徑 1：檢閱並強化。** 使用者挑選一個項目。帶領他們檢閱：展示目前文字，說明您的理由，詢問是否準確。根據他們的回饋進行修訂。如果功能測試有變動，請重新執行測試。

**路徑 2：引導式問答。** 提出 3-5 個衍生自您探索發現的問題。這些類別涵蓋了最常見的高槓桿缺口：

- **案例的事故歷史。** 「我發現了 [特定防禦性程式碼]。是什麼失敗導致了這個？影響了多少記錄？」
- **品質閘門閾值。** 「我正在檢查 [欄位] 是否包含 [數值]。什麼樣的分佈是正常的？什麼代表出現問題？」
- **整合測試規模與成本。** 「此協定執行 [N] 個測試，成本約為 $[X]。我應該增加還是減少涵蓋率？」
- **測試範圍。** 「我產出了 [N] 個功能測試。您的現有套件已涵蓋了 [其他區域]。是否還有缺口？」
- **規格稽核的模型偏好。** 「您使用哪些 AI 模型？是否注意到特定的優勢？」

使用者回答後，修訂產出的檔案並重新執行測試。

**路徑 3：檢閱開發歷史。** 如果使用者提供了聊天紀錄資料夾：

1. 掃描索引檔案並導覽至品質相關對話 (與步驟 0 方法相同，但現在有特定目標 — 您知道哪些案例需要根據事實說明、哪些品質閘門需要閾值、哪些設計決策需要理由)。
2. 擷取：帶有具體數值的事故故事、防禦性模式的設計理由、品質框架討論、跨模型稽核結果。
3. 使用真實事故細節修訂 QUALITY.md 案例。使用真實數值更新整合測試閾值。如果稽核結果存在，則新增三人議會的實證資料。
4. 修訂後重新執行測試。

如果使用者已在步驟 0 提供過聊天紀錄，您應該已經挖掘過它了 — 但他們可能想指向特定的對話，或要求您更深入地挖掘某個特定主題。

### 反覆運算

使用者可以多次循環執行這些路徑。每一次循環都會讓品質劇本更加植基於事實。當他們感到滿意時，就會自然地進入下一個階段 — 這裡沒有明確的「完成」步驟。

## Fixture 策略

`quality/` 資料夾與專案的單元測試資料夾是分開的。請根據專案語言建立適當的測試設定：

- **Python：** 使用 `quality/conftest.py` 定義 pytest fixture。如果 fixture 是內嵌定義的 (這在 pytest 的 `tmp_path` 模式中很常見)，則優先使用內嵌方式而非共享 fixture。
- **Java：** 使用帶有 `@BeforeEach`/@BeforeAll` 設定方法的測試類別，或共享的測試公用程式類別。
- **Scala：** 混入測試規格中的 trait (例如：`trait FunctionalTestFixtures`)，或使用內嵌資料產生器。
- **TypeScript/JavaScript：** 使用帶有 `beforeAll`/`beforeEach` 勾點的 `quality/setup.ts`，或使用內嵌測試工廠。
- **Go：** 在同一個 `_test.go` 檔案中使用輔助函式或共享的 `testutil_test.go`。使用 `t.Helper()` 標記測試輔助工具。Go 的慣例偏好內嵌測試設定而非共享 fixture。
- **Rust：** 在 `#[cfg(test)] mod tests` 區塊中使用輔助函式，或使用共享的 `test_utils.rs` 模組。使用產生器 (builder) 模式來建構測試資料。

檢閱現有的測試檔案以了解它們如何設定測試資料。無論現有測試使用什麼模式，請複製它。研究現有的 fixture 模式以獲得現實的資料形狀。

## 詞彙表

- **功能測試 (Functional testing)** — 程式碼產出的產出是否符合規格？不同於單元測試 (個別函式的隔離測試)。
- **整合測試 (Integration testing)** — 元件是否能端對端協作，包括實際的外部服務？
- **規格稽核 (Spec audit)** — AI 模型閱讀程式碼並與規格進行比較。不執行程式碼。捕捉程式碼與文件不符之處。
- **涵蓋率劇場 (Coverage theater)** — 產出高涵蓋率數字卻無法捕捉真實錯誤的測試。範例：斷言函式未拋出例外卻未檢查其產出。
- **適合使用 (Fitness-to-purpose)** — 程式碼在現實條件下是否執行了它應有的功能？一個系統可能擁有 95% 的涵蓋率，卻仍會默默遺失記錄。

---

## 原則

1. 適合使用優於涵蓋率百分比
2. 案例來自程式碼探索與領域知識
3. 具體的失敗模式讓標準不容置疑 — 抽象需求會引發合理化
4. 安全防護能改變 AI 檢閱品質 (行號、閱讀主體、聲稱前先執行 grep)
5. 修正前先篩選 — 許多「缺陷」其實是規格錯誤或設計決策
