# 反覆運算模式參考 (Iteration Mode Reference)

> 此檔案包含各個反覆運算策略的詳細指令。
> 代理程式在執行反覆運算時會閱讀此檔案 — 所有操作細節都位於此處，
> 而非位於提示或基準測試執行器中。

## 反覆運算週期

建議的反覆運算順序為：**gap → unfiltered → parity → adversarial**。每種策略都能發現不同類別的漏洞，按此順序執行可最大化累計產出。每次反覆運算後，該技能會列印針對下一個策略的建議提示 — 請遵循此週期，直到遇到報酬遞減或決定停止為止。

```
基準線執行 (Baseline run)                             # 結構化的三階段探索
→ gap         掃描先前的涵蓋範圍，探索缺口          # 發現未涵蓋子系統中的漏洞
→ unfiltered  純領域驅動，無結構化限制              # 發現結構化方式會抑制的漏洞
→ parity      跨路徑比較與 diff 差異分析            # 發現平行實作之間的不一致性
→ adversarial 挑戰已解編/已降級的發現               # 恢復先前分類中的第二類錯誤 (Type II errors)
```

## 適用於所有策略的共享規則

這些規則適用於每種反覆運算策略：

1. **ITER 檔案命名**：將發現寫入 `quality/EXPLORATION_ITER{N}.md` — 檢查已存在的反覆運算檔案並使用下一個數字（例如，第一次反覆運算使用 `EXPLORATION_ITER2.md`，第二次使用 `EXPLORATION_ITER3.md`）。

2. **「不要」刪除或封存 quality/ 目錄**：您是在現有執行的基礎上進行建構，而非取代它。

3. **內容預算紀律**：初次執行的 EXPLORATION.md 可能有 200-400 列。一次將其全部載入內容視窗會導致深度調查空間不足。先前執行的掃描應佔用約 20-30 列的內容空間。有針對性的深度閱讀應總計佔用約 40-60 列。這為新的探索留下了大部分內容預算。

4. **合併**：完成特定策略的探索後，建立或更新 `quality/EXPLORATION_MERGED.md`，以結合「所有」反覆運算的發現。對於每個章節，串接發現內容並標註明確的歸屬資訊（`[Iteration 1]` / `[Iteration 2: gap]` / `[Iteration 3: unfiltered]` 等）。在歸屬資訊中包含策略名稱，以便下游階段可以看到每項發現的來源方法。「候選漏洞」(Candidate Bugs) 章節應根據所有反覆運算的所有發現進行重新彙整。如果 `EXPLORATION_MERGED.md` 已存在於先前的反覆運算中，請將新的反覆運算發現合併進去，而非從頭開始。

   **已降級候選漏洞清單 (Demoted Candidates Manifest)（EXPLORATION_MERGED.md 中的強制項）**：在重新彙整「候選漏洞」章節後，於 EXPLORATION_MERGED.md 的末尾新增或更新一個 `## Demoted Candidates` 章節。此章節追蹤在任何反覆運算期間被解編、降級或取消優先順序的發現 — 它們是 adversarial 策略的原始素材。針對每個已降級的候選漏洞，記錄：

   ```
   ### DC-NNN：[簡短標題]
   - **來源：** [哪次反覆運算和策略首先發現了此項]
   - **解編原因：** [為何被降級 — 例如，「分類為設計選擇」、「證據不足」、「需要執行階段確認」]
   - **程式碼位置：** [檔案:列 引用]
   - **重新提升標準：** [能將此項翻轉為已確認候選漏洞的具體證據 — 例如，「證明寬容行為違反了記錄在案的合約」、「追蹤程式碼路徑以證明邊緣情況是可觸及的」、「演示輸出與規格要求的不同之處」]
   - **狀態：** DEMOTED (已降級) | RE-PROMOTED (已重新提升) [反覆運算] | FALSE POSITIVE (假陽性) [反覆運算]
   ```

   「重新提升標準」是最重要的欄位 — 它告訴 adversarial 策略具體要收集哪些證據。不接受「需要更多調查」之類模糊的標準；請撰寫不同的代理程式對話可以在無需額外背景資訊的情況下執行的標準。如果隨後的操作重新提升或明確證偽了某個已降級的候選漏洞，請更新其狀態並新增註解說明解決方案。

5. **繼續執行階段 2-6**：使用 `EXPLORATION_MERGED.md` 作為階段 2 成品產出的主要輸入。所有下游成品（REQUIREMENTS.md、程式碼審查、規格稽核）都應引用合併後的探索。

   **TDD 對於反覆運算執行是強制性的 (v1.3.49)**：反覆運算執行必須為每個新確認的漏洞執行完整的 TDD 紅-綠週期，這與基準線執行完全一致。這意味著：對於此反覆運算中確認的每個新 BUG-NNN，建立一個迴歸測試補丁，針對未修補的程式碼執行測試以產生 `quality/results/BUG-NNN.red.log`，如果存在修復補丁，則針對修補後的程式碼執行測試以產生 `quality/results/BUG-NNN.green.log`。階段 5 中的「TDD 日誌結案閘道」同樣適用於反覆運算執行 — 缺失日誌檔案將導致 `quality_gate.sh` 失敗。請勿因為這「僅僅是反覆運算」或先前漏洞已有日誌而跳過 TDD。新漏洞需要新日誌。如果該專案語言的測試執行器不可用，請建立第一行為 NOT_RUN 的日誌檔案並說明原因 — 該檔案必須存在。

6. **反覆運算模式完成閘道**：在進入階段 2 之前（適用於所有策略）：
   - `quality/ITERATION_PLAN.md` 存在並命名了所使用的策略
   - 此反覆運算的 `quality/EXPLORATION_ITER{N}.md` 存在，且至少包含 80 列實質內容
   - `quality/EXPLORATION_MERGED.md` 存在，且包含來自所有反覆運算的發現
   - 合併後的「候選漏洞」章節至少具有 2 個先前反覆運算中不存在的新候選漏洞
   - 至少有一項發現涵蓋了先前反覆運算中未探索的程式碼區域，「或者」是以新的證據重新確認了先前被解編的發現

7. **建議的下一次反覆運算**：在階段 6 結束時，在寫入最終的 PROGRESS.md 摘要後，列印週期中下一個反覆運算策略的建議提示。如果目前的策略是：
   - **gap** → 建議：`使用 unfiltered 策略執行品質播放手冊的下一個反覆運算。`
   - **unfiltered** → 建議：`使用 parity 策略執行品質播放手冊的下一個反覆運算。`
   - **parity** → 建議：`使用 adversarial 策略執行品質播放手冊的下一個反覆運算。`
   - **adversarial** → 建議：`從頭開始執行品質播放手冊。`（週期完成）
   - **基準線 (無策略)** → 建議：`使用 gap 策略執行品質播放手冊的下一個反覆運算。`

   清楚地格式化建議內容，以便使用者複製貼上：
   ```
   ────────────────────────────────────────────────────────
   下一次反覆運算建議：
   "使用 [策略] 策略執行品質播放手冊的下一個反覆運算。"
   ────────────────────────────────────────────────────────
   ```

## 中繼策略：`all` — 按順序執行每種策略

`all` 策略是執行器層級的便利功能，按順序執行 gap → unfiltered → parity → adversarial，每個策略作為單獨的代理程式對話執行。單個代理程式對話無法執行多個策略（內容預算限制），因此 `all` 是由協調器代理程式或基準測試執行器作為反覆運算呼叫的迴圈實作的。如果任何策略未發現新漏洞，請提早停止（報酬遞減）。

用法（協調器代理程式）：「執行所有反覆運算 (Run all iterations)」 — 代理程式依序執行 gap → unfiltered → parity → adversarial。
用法（基準測試執行器）：`python3 bin/run_playbook.py --next-iteration --strategy all <目標>`（基準測試工具，不隨技能發佈）。`--strategy` 也接受以逗號分隔的有順序子集，例如 `--strategy unfiltered,parity,adversarial`。

## 策略：`gap` (預設) — 找出先前執行遺漏的內容

掃描先前執行的涵蓋範圍，並刻意探索其他地方。最適合初次執行結構良好但僅涵蓋了程式碼庫子集的情況。

1. **涵蓋範圍掃描 (輕量級)**：使用分而治之的策略閱讀先前的 `quality/EXPLORATION.md` — 「不要」一次將整個檔案載入內容視窗。相反地：
   - 僅閱讀章節標題和每個章節的前 2-3 列，以建立涵蓋範圍地圖
   - 對於每個章節，記錄：章節名稱、涵蓋的子系統、發現數量、深度層級（淺 = 單一函式提及，深 = 多函式追蹤）
   - 將涵蓋範圍地圖寫入 `quality/ITERATION_PLAN.md`

2. **缺口識別**：根據涵蓋範圍地圖，識別：
   - 完全未被探索的子系統或模組
   - 發現較淺的章節（僅幾列內容、單一函式提及、無程式碼路徑追蹤）
   - 已列出但從未追蹤到具體程式碼的「品質風險」情境
   - 本應適用但未被選取的模式深鑽 (來自適用性矩陣)
   - 步驟 6 中未被處理的領域知識問題

3. **具針對性的深度閱讀**：僅針對 2-3 個最薄弱或缺口最多的章節，從先前的 EXPLORATION.md 中閱讀完整章節內容。這可以為您提供關於已發現內容的具體背景資訊，而不會耗盡您針對新探索的全部內容預算。

4. **缺口探索**：執行一次集中的階段 1 探索，僅針對識別出的缺口。使用相同的三階段方法（開放探索 → 品質風險 → 選定模式），但範圍限定於未涵蓋的區域。使用相同的範本結構將發現寫入 `quality/EXPLORATION_ITER{N}.md`。

## 策略：`unfiltered` — 無結構化限制的純領域驅動探索

完全忽略三階段閘道結構。像經驗豐富的開發人員那樣探索程式碼庫 — 閱讀程式碼、跟隨直覺、追蹤可疑路徑 — 沒有模式範本、適用性矩陣或章節格式要求。此策略刻意移除結構化架構，以便讓領域專業知識在不受約束的情況下驅動發現。

**為什麼存在此策略**：在基準測試中，v1.3.25-v1.3.26 技能版本所使用的未過濾領域驅動方法，發現了結構化三階段方法始終遺漏的漏洞，尤其是在 Web 框架和 HTTP 程式庫中。結構化方法擅長系統化涵蓋，但可能會過度約束探索，導致模型將內容預算花費在格式合規性而非深度程式碼閱讀上。unfiltered 策略能恢復該遺失的發現能力。

1. **輕量級先前執行掃描**：僅閱讀先前執行的 `## Candidate Bugs for Phase 2` 章節和 `quality/BUGS.md`，以瞭解已發現的內容。「不要」閱讀完整的 EXPLORATION.md — 您需要一個新鮮的視角，不要被先前的探索路徑所束縛。在 `quality/ITERATION_PLAN.md` 中撰寫簡短註解，列出先前執行的發現並確認您正在使用 unfiltered 策略。

2. **未過濾探索**：使用純粹的領域知識從頭開始探索程式碼庫。無必要章節、無模式適用性矩陣、無閘道自我檢查。相反地：
   - 深入閱讀原始碼 — 進入點、熱路徑、錯誤處理、邊緣情況
   - 遵循您的領域專業知識：「此領域的專家會覺得什麼地方可疑？」
   - 針對每項可疑發現，追蹤跨 2 個以上函式的程式碼路徑並提供 檔案:列 引用
   - 直接產生漏洞假設 — 不是「要調查的區域」，而是「此處檔案:列的特定程式碼產生了錯誤行為，原因為 [原因]」
   - 將發現寫入 `quality/EXPLORATION_ITER{N}.md` 作為平坦的發現清單，每項發現都有 檔案:列 參考和漏洞假設。無需結構化範本 — 深度和具體性勝過章節格式
   - 最低門檻：10 個具備 檔案:列 參考的具體發現，其中至少 5 個追蹤了跨 2 個以上函式的程式碼路徑

3. **領域知識問題**：結合您剛剛探索的程式碼「以及」您的領域知識完成下列問題。在您的發現中內嵌撰寫答案，而非在單獨的閘道章節中：
   - 相似方法之間存在哪些 API 表面不一致？
   - 程式碼在何處對結構化格式執行臨時的字串解析？
   - 領域專家會嘗試哪些開發人員可能不會測試的輸入？
   - 哪些 Metadata 或設定值可能發生靜默錯誤？

## 策略：`parity` — 跨路徑比較與 diff 差異分析

系統性地列舉相同合約的平行實作，並針對不一致性執行 diff 差異分析。此策略透過比較理應表現一致但實際上並非如此的程式碼路徑來發現漏洞。

**為什麼存在此策略**：在基準測試中，v1.3.40 技能版本使用「後備路徑一致性」和「跨實作一致性」作為明確的探索模式，在 virtio 中發現了 5 個漏洞。其中三個漏洞（MSI-X slow_virtqueues 重新連接、Spinlock 下的 GFP_KERNEL、INTx admin queue_idx）是透過排列平行程式碼路徑並找出差異點而發現的 — 而不是透過探索個別子系統。gap、unfiltered 和 adversarial 策略雖然探索區域或挑戰決策，但都沒有明確比較平行路徑。此策略填補了該項空白。

1. **列舉平行路徑**：掃描程式碼庫，尋找實作相同合約或透過不同路徑處理相同邏輯操作的程式碼群組。常見類別：
   - **傳輸/後端變體**：相同介面的多個實作 (例如，PCI vs MMIO vs vDPA, 同步 vs 非同步, HTTP/1.1 vs HTTP/2)
   - **後備鏈**：主要路徑 → 後備方案 → 最後手段 (例如，MSI-X → shared → INTx, 詳細錯誤 → 通用錯誤)
   - **設定 vs 清理/重設**：同一個資源的初始化路徑 vs 清理/重設路徑
   - **正常路徑 vs 錯誤路徑**：同一個操作的正常流程 vs 異常/錯誤處理
   - **公共 API 變體**：重載方法、便利包裝函式、理應產生等效結果的格式專屬解析器
   - 將列舉內容寫入 `quality/ITERATION_PLAN.md`，並簡要描述每個平行群組。

2. **成對比較**：針對每個平行群組，並排閱讀程式碼路徑，並系統性地檢查下方的每個比較子類型。並非每個子類型都適用於每個平行群組 — 但明確地考慮每一項能防止策略僅發現「顯而易見」的差異而遺漏結構性差異。

   **比較子類型清單** (針對每個平行群組檢查每一項)：

   - **資源生命週期一致性**：比較資源的設定/初始化操作與同一個資源的清理/重設/清除操作。設定中獲取的每個資源都必須在清理中釋放 — 且順序相同、範圍相同。尋找設定已建立但重設卻遺漏的資源 (例如，探測期間填入但在重設期間未清空的清單)。
   - **分配上下文一致性**：比較跨平行路徑的分配旗標、鎖定上下文和中斷狀態。如果一條路徑使用 `GFP_KERNEL` (可睡眠) 進行分配，但卻在另一條路徑未持有的微調鎖 (spinlock) 下執行，那就是漏洞。檢查：持有哪些鎖？在該上下文中哪些分配旗標是有效的？平行路徑是否一致？
   - **識別碼與索引一致性**：比較平行路徑如何為同一個邏輯實體計算索引、偏移量或識別碼。如果設定使用 `queue_index + admin_offset` 但重設卻使用 `raw_queue_index`，則該不匹配即是漏洞候選項。
   - **功能/特徵位元一致性**：比較每條平行路徑檢查或設定哪些特徵位元、旗標或功能。如果 MSI-X 路徑檢查了慢速路徑向量清單，但 INTx 後備路徑卻未檢查，則在後備後向量可能會被誤導。
   - **錯誤/異常一致性**：比較路徑間的錯誤處理。如果主要路徑優雅地處理了錯誤，但後備路徑卻讓錯誤傳播，則後備方案的強健性反而不如主要方案 — 這是不合理的。
   - **反覆運算/集合一致性**：比較各個路徑反覆執行的集合。如果設定反覆執行 `all_queues` 但重設僅反覆執行 `active_queues`，則非作用中佇列的資源會發生洩漏。

   針對發現的每項差異，追蹤兩條程式碼路徑並提供 檔案:列 引用，並判斷該差異是故意的（有記錄、有測試或結構上必要）還是漏洞。

3. **跨檔案合約追蹤**：針對最有希望的差異，追蹤跨檔案的呼叫鏈以驗證：
   - 每條路徑在哪種鎖定/中斷上下文中執行
   - 在該上下文中哪些分配旗標是有效的
   - 合約（記錄在規格、註解或標頭中）是否要求一致性
   - 將發現寫入 `quality/EXPLORATION_ITER{N}.md`，並為每項發現引用兩條程式碼路徑。

4. **最低門檻**：列舉至少 5 個平行群組，追蹤至少 8 個具備 檔案:列 參考的成對比較，至少 3 個具體的差異發現。

## 策略：`adversarial` — 挑戰先前執行的結論

重新調查先前執行中所解編、降級或標記為 SATISFIED (已滿足) 的內容。此策略假設先前執行發生了第二類錯誤 (Type II errors)（因過於保守而遺漏了真實漏洞），並系統性地挑戰那些決策。

**為什麼存在此策略**：在基準測試中，分類步驟會透過要求過多證據、將模糊案例標記為「設計選擇」或在沒有深度驗證的情況下接受程式碼審查的 SATISFIED 判定，進而可靠地降級合法的發現。adversarial 策略專門針對這些失敗模式。

1. **載入先前的決策**：從先前執行中讀取這些檔案（使用分而治之法 — 先讀標題，再進行針對性深讀）：
   - `quality/EXPLORATION_MERGED.md` — 特別是 `## Demoted Candidates` 章節（這是您的主要輸入 — 它包含每項被解編發現的結構化重新提升標準）
   - `quality/BUGS.md` — 已確認的內容（以避免重複發現相同的漏洞）
   - `quality/spec_audits/*triage*` — 分類期間解編或降級的內容，以及原因
   - `quality/code_reviews/*.md` — 階段 2 的 SATISFIED/VIOLATED 判定
   - `quality/EXPLORATION.md` — 僅閱讀 `## Candidate Bugs for Phase 2` 章節，看看哪些候選漏洞未變成已確認漏洞
   - 向 `quality/ITERATION_PLAN.md` 寫入摘要，列出：(a) 來自清單中帶有重新提升標準的已降級候選漏洞，(b) 尚未包含在清單中的其他已解編分類發現，(c) 未被提升的候選漏洞，(d) 證據薄弱卻被標記為 SATISFIED 的需求

2. **以較低的證據門檻重新調查已解編的發現**：adversarial 策略刻意使用比早期策略更低的證據標準。baseline 和 gap 策略為了在初步發現期間避免假陽性，理所應當地要求強有力的證據。但到了 adversarial 反覆運算，剩餘的未發現漏洞正是那些保守分類一直拒絕的漏洞 — 它們看起來很模糊，可能是「設計選擇」，且缺乏顯著的執行階段失敗。針對這些發現：
   - 展示可觀察語意飄移（輸出與規格或合約要求不同）的程式碼路徑追蹤，足以用於確認 — 您不需要執行階段崩潰或顯著失敗
   - 「寬容行為」並不自動代表是設計選擇 — 檢查規格、文件或 API 合約是否定義了預期行為。如果程式碼偏離了記錄在案的合約，無論該偏離是否為「寬容」，皆屬漏洞
   - 如果「已降級候選漏洞清單」包含重新提升標準，請嘗試特別滿足那些標準。每項標準在撰寫時都是具備行動力的 — 請遵循它
   - 閱讀發現中所引用的具體程式碼位置
   - 獨立追蹤程式碼路徑 — 不要依賴先前執行的分析
   - 結合新鮮證據做出明確的 CONFIRMED (已確認) / FALSE-POSITIVE (假陽性) 判定
   - 更新「已降級候選漏洞清單」：將狀態更改為 RE-PROMOTED 或 FALSE POSITIVE，並標註反覆運算歸屬

3. **挑戰 SATISFIED 判定**：針對程式碼審查中證據薄弱（單列引用、無程式碼路徑追蹤，或與 3 個以上其他需求分組在同一個引用下）而標記為 SATISFIED 的每個需求：
   - 透過閱讀引用程式碼並追蹤行為，重新驗證該需求
   - 檢查該需求是否確實得到滿足，還是審查只是走馬看花

4. **探索相鄰程式碼**：針對每項重新確認或新確認的發現，在周邊程式碼中探索相關漏洞 — 漏洞通常成群出現。如果一個函式有一個漏洞，其呼叫者和同級函式很可能也有相關問題。

5. 將所有發現寫入 `quality/EXPLORATION_ITER{N}.md`。每項發現必須包含：原始來源（分類解編、候選降級或 SATISFIED 挑戰）、新鮮證據以及新的判定結果。
