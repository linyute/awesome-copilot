# 審查協定 (Review Protocols) (檔案 3 與 4)

## 檔案 3：程式碼審查協定 (Code Review Protocol) (`RUN_CODE_REVIEW.md`)

### 範本

```markdown
# 程式碼審查協定：[專案名稱]

## 引導 (Bootstrap) (請先閱讀)

在開始審查前，請閱讀下列檔案以獲取背景資訊：
1. `quality/QUALITY.md` — 品質憲法與適用性情境
2. `quality/REQUIREMENTS.md` — 在產生播放手冊期間衍生的可測試需求
3. [主要架構文件]
4. [關鍵設計決策文件]
5. [任何其他必要背景資訊]

## 傳遞 1：結構化審查 (Structural Review)

閱讀程式碼並回報任何看起來有誤的地方。不預設需求或焦點區域 — 請運用您自身對程式碼正確性的知識。尋找：競爭條件 (race conditions)、Null 指標危害、資源洩漏、差一錯誤 (off-by-one errors)、型別不匹配、錯誤處理缺口以及任何看起來可疑的程式碼。

### 護欄 (Guardrails)

- **行號是強制性的。** 如果您無法引用確切的行號，請勿包含該項發現。
- **閱讀函式主體，而非僅看簽章。** 不要根據函式名稱假設其運作正常。
- **如果不確定某項內容是漏洞還是故意設計**，請將其標記為 QUESTION (問題) 而非 BUG (漏洞)。
- **在聲稱缺失前先執行 grep 搜尋。** 如果您認為某項特徵缺失，請先搜尋程式碼庫。如果在其他檔案中找到，那是位置缺陷，而非功能缺失。
- **「請勿」建議樣式變更、重構或改進。** 僅標記不正確或可能導致失敗的事項。

### 輸出

針對審查的每個檔案：

#### filename.ext
- **第 NNN 列：** [BUG / QUESTION / INCOMPLETE] 描述。預期 vs. 實際。重要性說明。

## 傳遞 2：需求驗證 (Requirement Verification)

閱讀 `quality/REQUIREMENTS.md`。針對每項需求，檢查程式碼是否滿足它。這是一次純粹的驗證傳遞 — 您唯一的工作是「程式碼是否滿足此需求？」。

「不要」同時進行一般性的程式碼審查。「不要」尋找其他漏洞。「不要」評估程式碼品質。僅檢查每項需求。

針對每項需求，回報下列其中之一：
- **SATISFIED (已滿足)**：程式碼實作了此需求。引用具體程式碼。
- **VIOLATED (已違反)**：程式碼「未」滿足此需求。解釋程式碼的操作與需求的規定有何不同。引用程式碼。
- **PARTIALLY SATISFIED (部分滿足)**：實作了某些方面，但遺漏了其他方面。解釋兩者。
- **NOT ASSESSABLE (不可評估)**：無法從審查的檔案中進行檢查。

### 輸出

針對每項需求：

#### REQ-N：[需求文字]
**狀態**：SATISFIED / VIOLATED / PARTIALLY SATISFIED / NOT ASSESSABLE
**證據**：[檔案:列] — [程式碼引用]
**分析**：[說明]
[若為 VIOLATED] **嚴重程度**：[影響描述]

## 傳遞 3：跨需求一致性 (Cross-Requirement Consistency)

比較 `quality/REQUIREMENTS.md` 中引用相同欄位、常數、範圍或安全性原則的需求對。針對每對需求，檢查其約束是否互通一致。

要尋找的內容：
- **數值範圍 vs. 位元寬度**：如果一項需求說有效範圍是 [0, N)，而另一項需求說該欄位寬度為 M 位元，是否滿足 N = 2^M？
- **安全性原則傳播**：如果一項需求說配置了 CA 檔案，那麼所有關於應使用該連線的需求是否確實引用了它？
- **驗證邊界 vs. 編碼限制**：一個檔案中的驗證檢查是否與另一個檔案中的存儲容量一致？
- **生命週期一致性**：如果資源是由一項需求的程式碼建立的，是否由另一項需求的程式碼清理？

針對共享相同概念的每一對需求，驗證其相對於實際程式碼的一致性。

### 輸出

針對每個共享概念：

#### 共享概念：[名稱]
**需求**：REQ-X, REQ-Y
**REQ-X 的主張**：[摘要]
**REQ-Y 的主張**：[摘要]
**一致性**：CONSISTENT (一致) / INCONSISTENT (不一致)
**程式碼證據**：[來自兩個位置的引用]
**分析**：[說明]
[若為 INCONSISTENT] **影響**：[觸發矛盾時發生的情況]

## 綜合摘要 (Combined Summary)

| 來源 | 發現 | 嚴重程度 | 狀態 |
|--------|---------|----------|--------|
| 傳遞 1 | [結構化發現] | [嚴重程度] | BUG / QUESTION |
| 傳遞 2, REQ-N | [需求違規] | [嚴重程度] | VIOLATED |
| 傳遞 3, REQ-X vs REQ-Y | [一致性問題] | [嚴重程度] | INCONSISTENT |

- 各傳遞與嚴重程度的發現總數
- 整體評估：SHIP (可發佈) / FIX BEFORE MERGE (合併前修正) / BLOCK (封鎖)
```

### 執行要求

**所有三次傳遞皆為強制性。** 請勿將多次傳遞合併為單次審查。每次傳遞都會產生截然不同的發現，因為它們使用的是不同的視角：

- **傳遞 1** 發現結構化漏洞（競爭條件、Null 危害、資源洩漏）
- **傳遞 2** 發現需求違規（行為缺失、規格偏離）
- **傳遞 3** 發現需求間的矛盾（不一致的範圍、衝突的保證）

**將每次傳遞寫入輸出檔案中清楚標註的章節**。使用標題 `## Pass 1: Structural Review`、`## Pass 2: Requirement Verification`、`## Pass 3: Cross-Requirement Consistency` 以及 `## Combined Summary`。

**如果某次傳遞沒有發現，請說明原因。** 不要只寫「無發現」。請撰寫您檢查了什麼，以及為什麼沒有問題。例如：「審查了 lib/response.js 中的 12 個函式的 Null 危害、資源洩漏與錯誤處理缺口。未確認任何漏洞 — 所有錯誤路徑皆會擲出異常或傳回定義明確的預設值。」無發現且無說明的傳遞視為未執行的傳遞。

**針對大型程式碼庫的範圍縮減：** 如果專案的需求超過 50 個，傳遞 2 不需要針對每個檔案驗證每項需求。相反地，請將傳遞 2 集中在與被審查檔案最相關的需求上 — 檢查引用了那些檔案的需求，或者治理那些檔案所實作之行為領域的需求。目標是深入審查中的檔案，而非橫跨所有需求。

**結束前的自我檢查：** 在寫完所有三次傳遞和綜合摘要後，請驗證：(1) 輸出中存在所有三個傳遞章節，(2) 傳遞 2 引用了帶有 SATISFIED/VIOLATED 判定結果的具體 REQ-NNN 編號，(3) 傳遞 3 至少識別出需求之間的一個共享概念（即使是一致的），(4) 每個 BUG 發現都在 `quality/test_regression.*` 中具有對應的迴歸測試（見下方的階段 2），(5) 每個迴歸測試都練習了發現中引用的實際程式碼路徑（見下方的測試與發現一致性檢查）。如果任何檢查失敗，請返回並補足缺失的工作。

### 當文件可用時採取對抗性立場

如果播放手冊是配合補充文件（reference_docs/、社群文件、使用者指南、API 參考）產生的，程式碼審查必須將該文件「對抗 (against)」程式碼，而非為其辯護。文件告訴您程式碼「應該」做什麼。您的工作是找出它「沒有」做什麼的地方。

**不要讓文件說明成為程式碼缺陷的藉口。** 如果文件說「函式庫會優雅地處理 X」，但程式碼並未檢查 X，那就是漏洞 — 文件讓它「更」像是一個漏洞，而非減輕其程度。對意圖的更豐富理解應使您對程式碼「更嚴格」，而非更寬容。

這解決了以下失敗模式：當模型可以存取文件時，它們會建構一個更豐富的軟體心理模型，並對大致符合該模型的程式碼變得更「寬容」。文件給了模型相信程式碼可以運作的理由，從而抑制了偵測。請透過將文件視為控方的證據來對抗這一點 — 它定義了程式碼承諾的內容，而您的工作是尋找背棄承諾的地方。

### 測試與發現一致性檢查

針對每個聲稱重現特定發現的迴歸測試，驗證該測試是否確實練習了被引用的程式碼路徑。針對不同函式、不同分支或不同於其聲稱重現之發現的失敗模式的測試，比沒有測試更糟 — 它會建立虛假的信心。

**驗證程序：** 針對每個迴歸測試：
1. 閱讀發現：記下具體的檔案、行號、函式與失敗條件。
2. 閱讀測試：識別它呼叫了哪個函式以及它斷言了什麼條件。
3. 確認一致性：測試必須呼叫發現中引用的函式，觸發發現描述的具體條件，並斷言發現指出的錯誤行為。

如果測試未練習被引用的程式碼路徑，請修復測試或將該發現標記為 UNCONFIRMED (未確認)。不要發佈因為與發現無關的原因而通過或失敗的迴歸測試。

### 結案授權 (Closure mandate)

每個確認的 BUG 發現都必須在 `quality/test_regression.*` 中產出一個迴歸測試。測試必須是專案語言的可執行原始檔案 — 而不是 Markdown 檔案，不是散文文件，也不是描述測試操作的註解區塊。如果專案使用 Java，請撰寫 `.java` 檔案。如果是 Python，則撰寫 `.py` 檔案。測試必須能被專案的測試框架編譯（或解析）且可執行。

**無語言豁免。** 如果擔心在修復前引入失敗的測試，請使用語言的「預期失敗」機制。保護機制必須是該框架 **最早的語法保護** — 在慣用的地方使用裝飾器 (decorator) 或註釋 (annotation)，否則使用測試主體中的第一行可執行程式碼：

- **Python (pytest)**：`@pytest.mark.xfail(strict=True, reason="BUG-NNN: [描述]")` — 位於 `def test_...():` 之上的裝飾器。當漏洞存在時：XFAIL (預期)。當漏洞已修復但未移除標記時：XPASS → strict 模式會失敗，發出應移除保護機制的訊號。
- **Python (unittest)**：`@unittest.expectedFailure` — 測試方法上方的裝飾器。
- **Go**：`t.Skip("BUG-NNN: [描述] — 在套用 quality/patches/BUG-NNN-fix.patch 後取消跳過")` — 測試函式內部的第一行。注意：Go 的 `t.Skip` 會隱藏測試 (報告為 SKIP，而非 FAIL)，這比 Python 的 xfail 證據力弱。
- **Rust**：測試函式上的 `#[ignore]` 屬性 — 標準的「不在預設套件中執行」機制。僅對恐慌 (panic) 形狀的漏洞使用 `#[should_panic]`。
- **Java (JUnit 5)**：測試方法上方的 `@Disabled("BUG-NNN: [描述]")` 註釋。
- **TypeScript/JavaScript (Jest)**：`test.failing("BUG-NNN: [描述]", () => { ... })`
- **TypeScript/JavaScript (Vitest)**：`test.fails("BUG-NNN: [描述]", () => { ... })`
- **JavaScript (Mocha)**：`it.skip("BUG-NNN: [描述]", () => { ... })` 或在測試主體內部使用 `this.skip()` 進行條件式跳過。

每個保護機制都必須引用漏洞 ID (BUG-NNN 格式) 與修復補丁路徑，以便遇到被跳過測試的人確切知道如何解決它。

這些模式確保每個漏洞都具有可執行的測試，在修復發佈時可以啟用，而不會因為預期失敗而污染 CI。

**TDD 紅/綠與跳過保護機制的互動。** 在 TDD 驗證期間，紅相和綠相必須暫時繞過跳過保護機制：
- **紅相 (絕不跳過)**：移除或停用跳過/xfail 保護機制，針對未修補的程式碼執行。必須失敗。記錄結果後重新啟用保護機制。**針對每個確認的漏洞，紅相是強制性的，即使沒有修復補丁。** 將其記錄為 `verdict: "confirmed open"`，其中 `red_phase: "fail"` 且 `green_phase: "skipped"`。請勿使用 `verdict: "skipped"` — 該值已棄用。
- **綠相**：移除或停用保護機制，套用修復補丁，執行。必須通過。如果修復將被還原，請重新啟用保護機制。如果沒有修復補丁，請記錄 `green_phase: "skipped"`。
- **成功的紅→綠之後**：產出位於 `quality/writeups/BUG-NNN.md` 的每漏洞說明檔案（參見 SKILL.md 檔案 7，「漏洞說明檔案產生」）。在 `tdd-results.json` 中將路徑記錄為 `writeup_path`。寫入 `tdd-results.json` 後，重新開啟並驗證所有必填欄位、列舉值以及無額外未記錄的根索引鍵（參見 SKILL.md 的寫入後驗證步驟）。兩個附屬 JSON 檔案都必須使用 `schema_version: "1.1"`。
- **TDD 週期後**：保護機制保留在已提交的迴歸測試檔案中，僅在修復永久合併時移除。

**唯一可接受的豁免情況** 是當迴歸測試確實無法撰寫時 — 例如，漏洞需要無法可靠重現的多執行緒計時，或者需要測試環境中無法取得的外部服務。在這種情況下，請在綜合摘要中撰寫明確的豁免註解並解釋原因，並包含一個展示如果您可以測試時會如何測試的最小程式碼草圖。

沒有可執行迴歸測試或明確豁免註解的發現是不完整的。綜合摘要不得包含未解決的發現 — 每個 BUG 都必須結案。

### 迴歸測試語意慣例

所有迴歸測試都必須斷言 **期望的正確行為**，並被標記為在目前程式碼上預期失敗。請勿撰寫斷言目前損壞行為並通過的測試。這兩者之間的區別很重要：

- **正確**：測試說「此輸入應產出 X」 → 因為有漏洞的程式碼產出 Y，測試失敗 → 標記為 `xfail` / `@Disabled` / `t.Skip` → 當漏洞修復後，測試通過並移除跳過標記。
- **錯誤**：測試說「此輸入產出 Y (漏洞輸出)」 → 在有漏洞的程式碼上通過 → 當漏洞修復後，測試默默失敗 → 成為斷言錯誤行為的陳舊測試。

`xfail(strict=True)` 模式 (Python/pytest) 是黃金標準：如果漏洞存在，它會失敗 (預期)；如果漏洞已修復但未移除 `xfail` 標記，它也會失敗 (strict 模式)。其他語言應以「跳過 + 原因」來近似實現此效果。

### 審查後結案驗證 (Post-review closure verification)

在撰寫完所有迴歸測試與綜合摘要後，請執行此檢查清單：

1. **統計綜合摘要中的 BUG 數量。** 這是預期的數量。
2. **統計 `quality/test_regression.*` 中的測試函式數量。** 此數量應等於或超過 BUG 數量（有些 BUG 可能需要多個測試）。
3. **針對摘要中的每一列 BUG**，驗證其是否具有下列其中之一：
   - 引用測試函式名稱的 `REGRESSION TEST:` 行，「或者」
   - 解釋為什麼未撰寫測試的 `EXEMPTION:` 行
4. **如果任何 BUG 兩者皆無**，請在宣告審查完成前返回撰寫測試或豁免說明。

此檢查清單是結案授權的執行機制。若沒有它，授權就只是空談 — 代理程式會在傳遞摘要中完整記錄漏洞，但會跳過迴歸測試並繼續前進。

### 規格稽核後的迴歸測試

結案授權同樣適用於規格稽核確認的程式碼漏洞，而不僅僅是程式碼審查漏洞。在規格稽核分類對發現進行類別劃分後，每個分類為「真實程式碼漏洞」(Real code bug) 的發現都必須獲得一個迴歸測試 — 使用與程式碼審查迴歸測試相同的慣例（可執行原始檔案、預期失敗標記、測試與發現的一致性）。

**為什麼這是一個單獨的步驟：** 程式碼審查迴歸測試是在程式碼審查後立即撰寫的，早於規格稽核的執行。這意味著規格稽核漏洞被系統性地遺孤了 — 它們出現在分類報告中，但從未進入迴歸測試檔案。在跨 8 個存放庫的 v1.3.4 執行中，規格稽核漏洞佔所有發現的 ~30%，且 8 個存放庫中僅有 1 個 (httpx) 為其撰寫了迴歸測試。

**程序：**
1. 規格稽核分類後，讀取分類摘要中分類為「真實程式碼漏洞」的發現。
2. 針對每一項，使用與程式碼審查迴歸測試相同的格式在 `quality/test_regression.*` 中撰寫一個迴歸測試。使用規格稽核報告作為來源引用：`[來自 spec_audits/YYYY-MM-DD-triage.md 的 BUG]`。
3. 執行測試以確認其失敗（預期）或通過（需要調查）。
4. 使用測試參考更新 PROGRESS.md 中的累計 BUG 追蹤器。

如果規格稽核未產生已確認的程式碼漏洞，請跳過此步驟 — 但請在 PROGRESS.md 中記錄該情況，以便稽核線索完整。

### 規格稽核翻轉後的清理

當規格稽核推翻了程式碼審查發現（將 BUG 分類為設計選擇或假陽性）時，對應的迴歸測試必須被刪除，或移至單獨的檔案 (`quality/design_behavior_tests.*`)，以記錄有意為之的行為。一個針對記錄在案的正確行為卻失敗的測試比沒有測試更糟 — 它會造成雜訊並侵蝕對迴歸套件的信任。

在規格稽核分類後，請檢查：`quality/test_regression.*` 中的任何測試是否對應到被重新分類為非缺陷的發現？如果是，請將其從迴歸檔案中移除。

### 為什麼採用三次傳遞而非焦點區域

先前的實驗 (QPB NSQ 基準測試) 顯示焦點區域並不能可靠地改進 AI 程式碼審查。一個通用的「審查漏洞」提示得分為 65.5%，而帶有 7 個具名焦點區域的播放手冊得分為 48.3% — 焦點區域縮小了模型的注意力範圍並抑制了偵測。

三步管線之所以奏效，是因為每步都做一件事且各步之間無交叉污染：
- **傳遞 1** 讓模型執行其原本擅長的工作（結構化審查，約佔缺陷的 65%）
- **傳遞 2** 擷取結構化審查會遺漏的個別需求違規（缺失型漏洞、規格偏離）
- **傳遞 3** 擷取各自正確的程式碼片段之間的矛盾（跨檔案算術漏洞、安全性原則缺口）

在 NSQ 程式碼庫上的實驗顯示，此管線在對具體漏洞零知識的情況下，發現了在所有結構化審查條件下都不可見的 3 個缺陷中的 2 個。發現的缺陷是一個跨檔案數值不匹配（驗證邊界 vs. 位元欄位寬度）和一個安全性設計缺口（配置的 CA 未傳播到外發的驗證用戶端）。

### 階段 2：針對已確認漏洞的迴歸測試

程式碼審查產生髮現後，撰寫重現每個 BUG 發現的迴歸測試。這將審查從「這裡有潛在漏洞」轉變為「這裡有具備失敗測試的已證明漏洞」。

**為什麼這很重要：** 沒有重現器的程式碼審查發現只是一個意見。具有失敗測試的發現則是一個事實。在多個程式碼庫（Go, Rust, Python）中，根據程式碼審查發現撰寫的迴歸測試以極高比率確認了漏洞 — 包含資料競爭、跨租用戶資料洩漏、狀態機違規以及靜默內容遺失。迴歸測試也是修復漏洞的驗收標準：當測試通過時，漏洞即已修復。

**如何產生迴歸測試：**

1. **針對每個 BUG 發現**，撰寫一個滿足以下條件的測試：
   - 目標對準發現中確切的程式碼路徑與行號。
   - 在目前的實作中失敗，以確認漏洞存在。
   - 使用 Mocking/Monkeypatching 來與外部服務隔離。
   - 在測試 docstring 中包含發現描述以實現可追溯性。

2. **將測試檔案命名為** `quality/test_regression.*`，並使用專案語言：
   - Python: `quality/test_regression.py`
   - Go: `quality/regression_test.go` (或在相關套件的測試目錄中)
   - Rust: `quality/regression_tests.rs` 或相關 Crate 中的 `tests/regression_*.rs` 檔案
   - Java: `quality/RegressionTest.java`
   - TypeScript: `quality/regression.test.ts`

3. **每個測試都應記錄其來源：**
   ```
   # Python 範例
   def test_webhook_signature_raises_on_malformed_input():
       """[來自 2026-03-26-reviewer.md, 第 47 列的 BUG]
       Webhook 簽章驗證在遇到格式錯誤的輸入時會引發異常而非傳回 False，
       面臨 500 錯誤而非乾淨的 401 錯誤的風險。"""

   // Go 範例
   func TestRestart_DataRace_DirectFieldAccess(t *testing.T) {
       // 來自 2026-03-26-claude.md, 第 3707 列的 BUG
       // Restart() 在未獲取鎖的情況下寫入受互斥鎖保護的欄位
   }
   ```

4. **執行測試並將結果報告** 為確認表格：
   ```
   | 發現 | 測試 | 結果 | 是否確認？ |
   |---------|------|--------|------------|
   | Webhook 簽章在格式錯誤的輸入上引發異常 | test_webhook_signature_... | FAILED (預期中) | 是 — 漏洞已確認 |
   | 佇列訊息在處理前被刪除 | test_message_queue_... | FAILED (預期中) | 是 — 漏洞已確認 |
   | 執行緒活動檢查在開啟時失敗 | test_is_thread_active_... | PASSED (非預期) | 否 — 需要調查 |
   ```

5. **如果測試發生非預期通過**，請進行調查 — 或是該發現為假陽性，或是測試未練習正確的程式碼路徑。回報為 NEEDS INVESTIGATION (需要調查)，而非已確認漏洞。

**各語言專屬提示：**

- **Go**：使用 `go test -race` 來確認資料競爭發現。競爭偵測器是決定性的 — 如果它觸發，則競爭是真實存在的。
- **Rust**：使用 `#[should_panic]` 或斷言特定的錯誤條件。對於原子性 (atomicity) 漏洞，在注入失敗後斷言清理狀態。
- **Python**：使用 `monkeypatch` 或 `unittest.mock.patch` 來隔離外部相依項。針對異常路徑漏洞使用 `pytest.raises`。
- **Java**：使用 Mockito 或類似工具來隔離相依項。針對異常路徑漏洞使用 `assertThrows`。

**儲存迴歸測試輸出** 以及程式碼審查：如果審查位於 `quality/code_reviews/2026-03-26-reviewer.md`，則迴歸測試位於 `quality/test_regression.*`，確認結果則作為增補內容放入審查檔案中或放在 `quality/results/` 中。

### 為什麼這些護欄很重要

這些護欄透過減少模糊與幻覺發現，實質性地改進了 AI 程式碼審查品質：

1. **行號** 強制模型實際定位問題，而不僅僅是描述一般性的疑慮。
2. **閱讀主體** 避免了僅憑函式名稱就假設其正常運作的常見錯誤。
3. **QUESTION vs BUG** 減少了浪費人力時間的假陽性。
4. **聲稱缺失前先執行 grep** 避免了最常見的 AI 審查幻覺：聲稱某項內容不存在，但其實它在另一個檔案中。

「不建議樣式變更」規則使審查專注於正確性。樣式建議會稀釋訊號並浪費審查時間。

### 撰寫良好整合檢查的提示

- 每個檢查項應練習真實的端對端流程，而非僅呼叫單個函式。
- 通過標準必須具體且可驗證 — 不是「看起來正確」而是「輸出包含確切 N 筆具備屬性 X 的記錄」。
- 在相關的情況下包含對時間的期望（特別是針對批次/管線專案）。
- 如果專案具有多種執行模式（批次 vs. 實時，不同提供者），請測試每種組合。

### 針對外部服務的實時執行

整合測試必須練習專案實際的外部相依項 — API、資料庫、服務、檔案系統。僅測試本機驗證與設定解析的協定不是整合測試協定，而是偽裝成它的單元測試套件。

探索期間請識別：
- **專案呼叫的外部 API** — 尋找 .env 檔案中的 API 金鑰、環境變數參考、提供者/用戶端抽象化、HTTP 用戶端設定。
- **執行模式** — 批次 vs. 實時、同步 vs. 非同步、不同的提供者後端。
- **現有的整合測試執行器** — 已練習端對端流程的指令碼或測試檔案。

接著將測試矩陣設計為 **提供者 × 管線 × 模式** 網格。例如，如果專案支援 3 個 API 提供者和 3 條管線，具備批次和實時模式，則協定應在該矩陣中執行真實的調用 — 而不僅僅是在本地驗證配置。

**為平行處理結構化執行任務。** 針對執行任務進行分組，以便最多只有一個執行任務在同一個外部提供者上同時執行（以避免頻率限制）。在群組內使用背景程序與 `wait` 指令來實現並行執行。

**推導管線專屬的品質閘道。** 每條管線產出的輸出各異，其正確性標準也不同。協定必須指明每條管線應檢查哪些欄位、哪些數值範圍是可接受的 — 而不僅僅是「輸出已存在」。

**包含執行後的驗證清單。** 針對每次執行，驗證：記錄檔已存在且包含完成訊息、資訊清單顯示終端狀態、驗證後的輸出檔案已存在且包含可解析的資料、樣本記錄具有預期的已填充欄位，以及任何現有的自動化品質檢查指令碼皆通過。

**預檢必須檢查 API 金鑰。** 如果缺少金鑰，請停止並詢問 — 不要默默跳過實時測試。

目標是執行此協定能練習真實世界條件下的完整系統，擷取本機端測試會遺漏的問題：提供者專屬的回應格式差異、逾時行為、頻率限制以及使用真實 LLM 回應的輸出正確性。

### 平行處理與頻率限制感知

循序執行的整合任務浪費時間。將執行任務進行分組，以便獨立的任務能並行執行，並遵守以下約束：

- **同一時間在同一個外部提供者上最多僅執行一個任務**，以避免頻率限制。
- **使用背景程序與 `wait` 指令** 來在群組內實現並行執行。
- **在對話開始時產生一個共享時間戳記**，以便一致地命名執行目錄。

具有 3 條管線和 3 個提供者（9 個以上執行任務）的專案的分組範例：

```
第 1 組 (平行)：管線_A × 提供者_1 | 管線_B × 提供者_2 | 管線_C × 提供者_3
第 2 組 (平行)：管線_A × 提供者_2 | 管線_B × 提供者_3 | 管線_C × 提供者_1
第 3 組 (平行)：管線_A × 提供者_3 | 管線_B × 提供者_1 | 管線_C × 提供者_2
```

此模式可最大化吞吐量，同時絕不會以並行請求衝擊同一個提供者。根據專案實際的管線和提供者數量調整分組。

在產出的協定中，包含使用 `&` 進行背景執行的實際 Bash 命令，以及群組之間的 `wait`。不要只描述平行處理 — 要寫成指令碼。

### 從程式碼推導品質閘道

通用性的通過/失敗標準（「所有單元皆已驗證」）會遺漏領域專屬的正確性問題。請從程式碼本身推導管線專屬的品質檢查：

1. **閱讀驗證規則。** 如果專案會驗證輸出（結構描述驗證器、斷言函式、業務規則檢查），這些規則定義了什麼是「正確」的內容。將其轉化為品質閘道：「欄位 X 對於所有輸出記錄都必須滿足條件 Y」。

2. **閱讀結構描述列舉。** 如果結構描述定義了列舉欄位（例如 `outcome: ["fell_in_water", "reached_ship"]`），品質閘道為：「所有輸出都必須使用來自此集合的數值，且分佈應為非簡併的 (non-degenerate)（非 100% 同一個值）」。

3. **閱讀產生邏輯。** 如果專案產生測試資料（項目檔案、種子資料、排列組合策略），請了解應出現哪些變體。如果有 3 種性格類型，品質閘道為：「所有 3 種類型都必須出現在輸出中，且具有足夠的樣本大小」。

4. **閱讀現有的品質檢查。** 搜尋已在驗證輸出品質的指令碼或函式（例如 `integration_checks.py`、執行後呼叫的驗證函式）。在協定中引用或呼叫它們。

針對專案中的每條管線，整合協定應具有專屬的「品質檢查」章節，列出 2-4 項從上述探索中推導出的具體檢查與預期值。請勿使用「輸出已存在」等通用的檢查 — 每項檢查必須引用具體的欄位與可接受的數值範圍。

### 欄位參考表 (Field Reference Table) (在撰寫品質閘道前必備)

**為什麼這很重要：** AI 模型即使讀過了結構描述，也會自信地寫錯欄位名稱。這是因為模型在探索期間讀取結構描述，但在幾小時（或數千個權杖）後憑記憶撰寫協定。記憶會發生飄移：`document_id` 變成 `doc_id`，`sentiment_score` 變成 `sentiment`，`float 0-1` 變成 `int 0-100`。協定看起來很權威，但欄位名稱是幻覺。當有人對真實資料執行品質閘道時，它們會失敗 — 使用者對整個產出的播放手冊的信任也會隨之崩塌。

**修復方法是程序性的，而非指示性的。** 不要只告訴自己「稍後交叉檢查」 — 請 **先** 建立參考表，然後透過複製來撰寫品質閘道。

在撰寫任何引用輸出欄位名稱的品質閘道之前，請透過重新閱讀每個結構描述檔案來建立 **欄位參考表**：

```
## 欄位參考表 (根據結構描述建立，非憑記憶)

### 管線：WeatherForecast
結構描述：pipelines/WeatherForecast/schemas/analyze.json
| 欄位 | 型別 | 約束 |
|-------|------|-------------|
| region_name | string | — |
| temperature | number | 最小值：-50, 最大值：60 |
| condition | string | enum: ["sunny", "cloudy", "rain", "snow"] |

### 管線：SentimentAnalysis
結構描述：pipelines/SentimentAnalysis/schemas/evaluate.json
| 欄位 | 型別 | 約束 |
|-------|------|-------------|
| document_id | string | — |
| sentiment_score | number | 最小值：-1.0, 最大值：1.0 |
| classification | string | enum: ["positive", "negative", "neutral"] |
...
```

**程序：**
1. **在撰寫每列表格之前，「立即」重新閱讀每個結構描述檔案。** 不要憑記憶撰寫任何列。檔案讀取與表格列撰寫必須相鄰 — 讀取檔案，撰寫資料列，讀取下一個檔案，撰寫下一列。如果您在對話早期閱讀了所有結構描述，那不算數 — 您必須在此處「再次」閱讀，因為您的欄位名稱記憶會隨著數千個權杖而飄移。
2. **從檔案內容中字元對字元地複製欄位名稱。** 不要重新輸入。`document_id` 不是 `doc_id`。`sentiment_score` 不是 `sentiment`。`classification` 不是 `category`。即使是細微的差異也會破壞品質閘道。
3. **包含來自結構描述的「所有」欄位，而不僅僅是您認為重要的欄位。** 如果結構描述有 8 個必填欄位，表格就有 8 列。如果您寫的列數少於結構描述的欄位數，說明您跳過了某些欄位。
4. 透過從已完成的表格中複製欄位名稱來撰寫品質閘道。
5. 撰寫後計算欄位數量：如果品質閘道提到了表格中沒有的欄位，說明您產生了幻覺。請將其移除。

此表格是一個中間產出物 — 請將其包含在協定本身中（作為參考章節），以便協定的未來使用者可以驗證欄位準確性。重點是將其作為產生讀取結構描述證據的具體步驟來建立，而不是因為您「已經知道」欄位而跳過它。

### 校準規模 (Calibrating Scale)

每次整合測試執行的單元/記錄/反覆運算次數非常重要：

- **太少 (1–3)：** 快速且便宜，但會遺漏並行漏洞，分佈檢查會失敗（無法用 2 筆記錄驗證「25–75% 比例」），且無法在現實規模下測試扇出/擴展邏輯。
- **太多 (100+)：** 對於測試協定來說過於昂貴且緩慢。適用於生產環境，但不適用於品質驗證。
- **正確範圍：** 足以有意義地練習系統。指引：
  - 如果專案具有分塊/批次邏輯，使用的數量應跨越至少 2 個分塊（例如，如果 chunk_size=10，使用 15–30 個單元）。
  - 如果專案具有分佈檢查，使用至少類別數量的 5–10 倍（例如，3 種結果類型 → 至少 15 個單元）。
  - 如果專案具有扇出/擴展功能，使用的數量應產生非微不足道數量的子項。

在專案中尋找 `chunk_size`、`batch_size` 或類似配置進行校準。如有疑慮，10–30 筆記錄通常是整合測試的正確範圍 — 足以擷取真實問題，且不會耗盡 API 預算。

### 針對技能與 LLM 自動化工具的整合測試

當受測專案是 AI 技能、包裝 LLM 呼叫的 CLI 工具或任何主要執行路徑涉及調用 AI 模型的軟體時，整合測試協定必須包含 **LLM 自動化整合測試** — 透過命令列 AI 代理程式端對端執行工具，並在結構上驗證產出的測試。

這與標準整合測試不同，因為受測系統沒有確定性的 API 可供呼叫。「整合」是：將技能安裝到測試存放庫中，透過 CLI 代理程式（GitHub Copilot CLI、Claude Code 或類似工具）進行調用，並驗證產出的成品符合結構與內容預期。

**為什麼這很重要：** 技能與 LLM 工具無法透過直接呼叫函式來測試 — 它們的執行路徑是經由 AI 代理程式，該代理程式負責解讀指令、讀取檔案並產出成品。測試技能是否運作的唯一方法就是執行它。手動執行對於開發來說是可以的，但品質播放手冊應將測試編碼為可重複的協定。

**技能/LLM 整合測試的協定結構：**

```markdown
## 技能整合測試協定

### 先決條件
- 已安裝並設定 CLI 代理程式 (例如 `gh copilot`, `claude`, `npx @anthropic-ai/claude-code`)
- 已準備測試存放庫，並將技能安裝於 `.github/skills/SKILL.md` (或等效位置)
- 乾淨的 `quality/` 目錄 (無先前執行的成品)
- 選項：用於基準線 vs 帶有文件比較執行的 `reference_docs/` 資料夾

### 測試矩陣

| 測試 | 方法 | 通過標準 |
|------|--------|---------------|
| 完整執行 | 透過 CLI 以「執行」提示語執行技能 | `quality/` 中存在所有預期成品 |
| PROGRESS.md 完整性 | 讀取 `quality/PROGRESS.md` | 所有階段皆標記為已完成，漏洞追蹤器已填入 |
| 成品結構檢查 | 驗證每個預期檔案 | 檔案非空，包含預期章節 |
| 漏洞追蹤器結案 | 統計 BUG 條目 vs 迴歸測試 | 每個 BUG 皆具備測試參考或豁免說明 |
| 基準線 vs 帶有文件 (選配) | 執行兩次：無及有 reference_docs/ | 帶有文件的執行產生 >= 基準線的需求計數 |

### 執行

```bash
# 將技能安裝到測試存放庫
cp -r path/to/skill/.github test-repo/.github

# 透過 CLI 代理程式執行 (根據您的代理程式調整命令)
cd test-repo
gh copilot -p "閱讀 .github/skills/SKILL.md 及其參考檔案。執行此專案的品質播放手冊。" \
    --model gpt-5.4 --yolo > quality_run.output.txt 2>&1
```

### 結構驗證 (自動化)

執行後，在結構上驗證產出：

```bash
# 必要的成品存在且非空
for f in quality/QUALITY.md quality/REQUIREMENTS.md quality/CONTRACTS.md \
         quality/COVERAGE_MATRIX.md quality/COMPLETENESS_REPORT.md \
         quality/PROGRESS.md quality/RUN_CODE_REVIEW.md \
         quality/RUN_INTEGRATION_TESTS.md quality/RUN_SPEC_AUDIT.md; do
    [ -s "$f" ] || echo "FAIL: $f 缺失或為空"
done

# 功能測試檔案存在 (語言慣用名稱)
ls quality/test_functional.* quality/FunctionalSpec.* quality/functional.test.* 2>/dev/null \
    || echo "FAIL: 無功能測試檔案"

# PROGRESS.md 中所有階段皆已勾選
grep -c '\[x\]' quality/PROGRESS.md  # 應等於總階段數

# 漏洞追蹤器中有條目 (如果發現了漏洞)
grep -c '^| [0-9]' quality/PROGRESS.md

# 程式碼審查與規格稽核產出了實質性檔案
find quality/code_reviews -name "*.md" -size +500c | wc -l  # 應 >= 1
find quality/spec_audits -name "*triage*" -size +500c | wc -l  # 應 >= 1
```
```

**基準線 vs 帶有文件比較模式：** 對同一個存放庫執行兩次技能 — 一次沒有補充文件，一次有包含專案歷史記錄的 `reference_docs/` 資料夾。比較：需求計數、情境計數、漏洞計數以及管線完成情況。帶有文件的執行應產出等於或更多的需求與漏洞。如果基準線在漏洞偵測方面優於帶有文件的執行，那是關於文件品質的發現，而非技能失敗。

**何時產生此協定：** 每當分析的專案是技能、包裝 AI 呼叫的 CLI 工具或用於建構 AI 驅動工具的框架時，請在 `RUN_INTEGRATION_TESTS.md` 中產生技能整合測試章節。尋找：`SKILL.md` 檔案、提示範本、LLM 用戶端設定、代理程式協調程式碼，或程式碼庫中對 AI 模型的引用。

### 執行後驗證深度

完成且無錯誤的執行仍可能是錯誤的。針對每次整合測試執行，在多個層級進行驗證：

1. **程序級：** 程序是否乾淨退出？檢查記錄檔是否有完成訊息，而不僅僅是檢查結束代碼。
2. **狀態級：** 執行是否處於終端狀態？檢查執行資訊清單/狀態檔案是否為「完成」(不是卡在「執行中」或「已提交」)。
3. **資料級：** 輸出資料是否存在且能正確解析？讀取實際的輸出檔案，驗證其包含有效的 JSON/CSV 等。
4. **內容級：** 輸出記錄是否在關鍵欄位填充了合理的數值？讀取 2–3 個樣本記錄並檢查關鍵欄位。
5. **品質級：** 管線專屬的品質閘道是否通過？執行任何現有的品質檢查指令碼。
6. **UI 級 (如有)：** 如果專案有儀表板/TUI/UI，驗證執行結果是否正確顯示。

在產出的協定執行後檢查清單中包含所有適用的層級。常見的失敗是停在第 2 層 (程序已完成)，而未檢查第 3–5 層。
