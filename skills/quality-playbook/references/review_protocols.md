## 檔案 3：程式碼檢閱協定 (`RUN_CODE_REVIEW.md`)

### 範本

```markdown
# 程式碼檢閱協定：[專案名稱]

## 引導程序 (請先閱讀)

在開始檢閱之前，請閱讀以下檔案以獲取脈絡：
1. `quality/QUALITY.md` — 品質憲法與適合使用案例
2. [主要架構文件]
3. [關鍵設計決策記錄]
4. [任何其他必要的背景資訊]

## 檢查項目

### 焦點區域 1：[子系統/風險區域名稱]

**位置：** [特定的檔案與函式]
**內容：** [具體要尋找的事項]
**原因：** [如果此項不正確會發生什麼問題]

### 焦點區域 2：[子系統/風險區域名稱]

[為 4-6 個焦點區域重複上述內容，並根據探索得到的架構與風險區域進行對映]

## 安全防護

- **行號為必填項。** 如果您無法引用具體的行號，請勿包含該項發現。
- **閱讀函式主體，而不僅僅是特徵標記。** 不要根據函式名稱假設其運作正確。
- **如果不確定某項是錯誤還是刻意為之**，請將其標記為「問題 (QUESTION)」而非「錯誤 (BUG)」。
- **在聲稱缺失之前請先執行 grep 搜尋。** 如果您認為某項功能缺失，請搜尋程式碼庫。如果在不同的檔案中找到，那是位置缺陷，而非功能缺失。
- **請勿建議風格變更、重構或改進。** 僅標記不正確或可能導致失敗的事項。

## 輸出格式

將發現儲存至 `quality/code_reviews/YYYY-MM-DD-檢閱者.md`

針對每個檢閱的檔案：

### filename.ext
- **第 NNN 行：** [BUG / QUESTION / INCOMPLETE] 描述。預期 vs. 實際。為什麼這很重要。

### 摘要
- 按嚴重程度統計總發現數量
- 無發現的檔案
- 總體評估：可出貨 (SHIP IT) / 先修正 (FIX FIRST) / 需討論 (NEEDS DISCUSSION)
```

### 第二階段：針對已確認錯誤的迴歸測試

在程式碼檢閱產出發現後，撰寫能重現每個「錯誤 (BUG)」發現的迴歸測試。這將檢閱從「這裡可能有錯誤」轉化為「這裡有已證實的錯誤且附帶失敗測試」。

**為什麼這很重要：** 一項沒有重現器的程式碼檢閱發現只是一個意見。一項附帶失敗測試的發現則是一個事實。在多個程式碼庫（Go、Rust、Python）中，根據程式碼檢閱發現撰寫的迴歸測試以極高的比例確認了錯誤 — 包括資料競爭 (data races)、跨租戶資料外洩、狀態機違規以及隱性的脈絡遺失。迴歸測試也可以作為修復錯誤的驗收標準：當測試通過時，錯誤即已修復。

**如何產生迴歸測試：**

1. **針對每項「錯誤 (BUG)」發現**，撰寫一個測試，其具備以下特點：
   - 針對發現中確切的程式碼路徑與行號
   - 在目前的實作中會失敗，以確認錯誤存在
   - 使用模擬 (mocking)/猴子補丁 (monkeypatching) 以與外部服務隔離
   - 在測試 docstring 中包含發現描述以供追蹤

2. **命名測試檔案** 為 `quality/test_regression.*`，並使用專案對應的語言：
   - Python：`quality/test_regression.py`
   - Go：`quality/regression_test.go` (或位於相關套件的測試目錄中)
   - Rust：`quality/regression_tests.rs` 或相關 crate 中的 `tests/regression_*.rs` 檔案
   - Java：`quality/RegressionTest.java`
   - TypeScript：`quality/regression.test.ts`

3. **每個測試都應記錄其來源：**
   ```
   # Python 範例
   def test_webhook_signature_raises_on_malformed_input():
       """[源自 2026-03-26-reviewer.md 第 47 行的 BUG]
       Webhook 簽章驗證在格式錯誤的簽章上會引發例外，而非回傳 False，
       存在導致 500 錯誤而非乾淨 401 錯誤的風險。"""

   // Go 範例
   func TestRestart_DataRace_DirectFieldAccess(t *testing.T) {
       // 源自 2026-03-26-claude.md 第 3707 行的 BUG
       // Restart() 在未取得鎖的情況下寫入受 mutex 保護的欄位
   }
   ```

4. **執行測試並將結果報告** 為確認表：
   ```
   | 發現 | 測試 | 結果 | 已確認？ |
   |---------|------|--------|------------|
   | Webhook 簽章在格式錯誤的輸入上引發例外 | test_webhook_signature_... | 失敗 (符合預期) | 是 — 錯誤已確認 |
   | 佇列訊息在處理前被刪除 | test_message_queue_... | 失敗 (符合預期) | 是 — 錯誤已確認 |
   | 執行緒活躍檢查失敗並保持開啟 | test_is_thread_active_... | 通過 (不符預期) | 否 — 需要調查 |
   ```

5. **如果測試出乎意料地通過**，請進行調查 — 可能是發現為誤判，或者是測試未執行到正確的程式碼路徑。報告為「需要調查」，而非已確認的錯誤。

**語言特定提示：**

- **Go：** 使用 `go test -race` 來確認資料競爭發現。競爭偵測器是權威性的 — 如果它觸發，競爭就是真實存在的。
- **Rust：** 使用 `#[should_panic]` 或針對特定的錯誤條件進行斷言。對於原子性錯誤，在注入失敗後對清理狀態進行斷言。
- **Python：** 使用 `monkeypatch` 或 `unittest.mock.patch` 來隔離外部相依項。針對例外路徑錯誤使用 `pytest.raises`。
- **Java：** 使用 Mockito 或類似工具來隔離相依項。針對例外路徑錯誤使用 `assertThrows`。

**將迴歸測試輸出儲存** 在程式碼檢閱結果旁邊：如果檢閱檔案位於 `quality/code_reviews/2026-03-26-reviewer.md`，則迴歸測試放在 `quality/test_regression.*` 中，確認結果則作為附錄放在檢閱檔案中或放在 `quality/results/` 中。

### 為什麼這些安全防護很重要

這四個安全防護通常能透過減少模糊與虛假的發現來提升 AI 程式碼檢閱品質：

1. **行號** 強制模型實際定位問題，而不僅僅是描述一般的疑慮
2. **閱讀主體** 防止常見的失敗：根據函式名稱假設其運作正確
3. **問題 vs. 錯誤** 減少浪費人類時間的誤判
4. **聲稱缺失前先執行 grep** 防止最常見的 AI 檢閱幻覺：在該內容位於不同檔案時聲稱其不存在

「不建議風格變更」的規則讓檢閱專注於正確性。風格建議會稀釋訊號並浪費檢閱時間。

- **元件邊界** — 模組 A 的產出是否能正確饋送給模組 B？

## 執行體驗 (執行此協定時如何呈現)

當 AI 代理程式執行此協定時，它應分三個階段進行溝通，以便使用者無需閱讀原始輸出即可跟隨進度：

### 第一階段：計畫

在執行任何操作之前，向使用者展示即將發生的事：

```

**要執行的測試：**

| # | 測試 | 檢查內容 | 預估時間 |
|---|------|---------------|-----------|
| 1 | [測試名稱] | [一句話描述] | ~30s |
| 2 | [測試名稱] | [一句話描述] | ~2m |
| ... | | | |

**總計：** N 個測試，預估 M 分鐘
```

這讓使用者有機會在任何操作開始前說出「跳過測試 4」或「其實，不要執行即時 API 測試」。

### 第二階段：進度

在每個測試執行時，報告一行狀態更新。保持簡練 — 使用者想要心跳訊號，而非日誌傾印：

```

✓ 測試 2：結構描述驗證 — 通過 (0.1s)
⧗ 測試 3：即時管線 (Gemini, realtime)... 執行中
```

使用 `✓` 表示通過，`✗` 表示失敗，`⧗` 表示進行中。如果測試失敗，顯示一行脈絡資訊（錯誤訊息或失敗的斷言），而非完整的堆疊追蹤。使用者若需要詳細資訊可以再詢問。

### 第三階段：結果

所有測試完成後，顯示摘要表與建議：

```

### 撰寫良好整合檢查的提示

- 每個檢查都應練習一個真實的端對端流程，而非僅呼叫單一函式
- 通過標準必須具體且可驗證 — 不是「看起來正確」，而是「產出包含確切 N 筆具備屬性 X 的記錄」
- 在相關處包含時間預期 (特別是針對批次/管線專案)
- 如果專案有多種執行模式 (批次 vs. 即時，不同的提供者)，請測試每個組合

### 針對外部服務的即時執行

整合測試必須練習專案實際的外部相依項 — API、資料庫、服務、檔案系統。僅測試本機驗證與組態解析的協定不是整合測試協定；它只是披著整合測試外衣的單元測試套件。

在探索期間，識別：
- **專案呼叫的外部 API** — 在 .env 檔案中尋找 API 金鑰、環境變數參照、提供者/用戶端抽象、HTTP 用戶端組態
- **執行模式** — 批次 vs. 即時、同步 vs. 非同步、不同的提供者後端
- **現有的整合測試執行器** — 已經在練習端對端流程的腳本或測試檔案

然後將測試矩陣設計為 **提供者 (provider) × 管線 (pipeline) × 模式 (mode)** 格網。例如，如果專案支援 3 個 API 提供者和 3 個具有批次與即時模式的管線，該協定應跨該矩陣執行真實的執行 — 而不僅是在本機驗證組態。

**為平行處理結構化執行任務。** 將執行任務分組，以便最多同時執行一個針對特定提供者的任務（以避免速率限制）。在組別內使用背景程序與 `wait` 進行並行執行。

**定義每管線品質閘門。** 每個管線產生不同的產出，具備不同的正確性標準。協定必須指定為每個管線檢查哪些欄位以及哪些數值是可接受的 — 而不僅是「產出存在」。

**包含執行後驗證核取清單。** 對於每次執行，驗證：日誌檔案存在且包含完成訊息、資訊清單 (manifest) 顯示終端狀態、經過驗證的產出檔案存在且包含可解析的資料、範例記錄的關鍵欄位已填寫，以及任何現有的自動化品質檢查腳本皆通過。

**執行前檢查必須檢查 API 金鑰。** 如果金鑰缺失，請停止並詢問 — 不要靜默地跳過即時測試。

其目標是透過執行此協定，在現實條件下練習整個系統，捕捉本機測試會遺漏的問題：提供者特定的回應格式差異、逾時行為、速率限制，以及真實 LLM 回應的產出正確性。

### 平行處理與速率限制意識

循序的整合執行會浪費時間。將執行任務分組，以便獨立的任務能並行執行，並遵循以下限制：

- **最多同時執行一個針對特定外部提供者的任務**，以避免速率限制
- **使用背景程序與 `wait`** 在組別內進行並行執行
- **在工作階段開始時產生一個共享的時間戳記**，以便維持執行目錄命名的一致性

針對具備 3 個管線與 3 個提供者（共 9+ 次執行）之專案的範例分組：

```
第 1 組 (平行)：管線_A × 提供者_1 | 管線_B × 提供者_2 | 管線_C × 提供者_3
第 2 組 (平行)：管線_A × 提供者_2 | 管線_B × 提供者_3 | 管線_C × 提供者_1
第 3 組 (平行)：管線_A × 提供者_3 | 管線_B × 提供者_1 | 管線_C × 提供者_2
```

此模式能在不對同一個提供者發送並行請求的情況下極大化吞吐量。請根據專案實際的管線與提供者數量調整分組方式。

在產出的協定中，包含實際的 bash 指令，使用 `&` 進行背景執行，並在組別之間使用 `wait`。不要只描述平行處理 — 請將其寫成腳本。

### 從程式碼衍生品質閘門

通用的通過/失敗標準（「所有單位皆已驗證」）會遺漏領域特定的正確性問題。請從程式碼本身衍生管線特定的品質檢查：

1. **閱讀驗證規則。** 如果專案會驗證產出（結構描述驗證器、斷言函式、業務規則檢查），這些規則定義了什麼是「正確」。將它們轉化為品質閘門：「欄位 X 必須針對所有產出記錄滿足條件 Y。」

2. **閱讀結構描述列舉 (enums)。** 如果結構描述定義了列舉欄位（例如：`outcome: ["fell_in_water", "reached_ship"]`），品質閘門即為：「所有產出必須使用此集合中的數值，且分佈應為非退化的 (non-degenerate)（非 100% 都是同一個值）。」

3. **閱讀產生邏輯。** 如果專案會產生測試資料（項目檔案、種子資料、排列組合策略），請了解應該出現哪些變體。如果有 3 種人格類型，品質閘門即為：「所有 3 種類型都必須出現在產出中，且具備足夠的樣本大小。」

4. **閱讀現有的品質檢查。** 搜尋已經在驗證產出品質的腳本或函式（例如：`integration_checks.py`、執行後呼叫的驗證函式）。直接從協定中參考或呼叫它們。

對於專案中的每個管線，整合協定都應具備專屬的「品質檢查」章節，列出 2-4 個源自上述探索的具體檢查項目。不要使用「產出存在」之類的通用檢查 — 每個檢查都必須參考具體的欄位與可接受的數值範圍。

### 欄位參考表 (撰寫品質閘門前必備)

**為什麼這很重要：** AI 模型即使在閱讀過結構描述後，也常會信誓旦旦地寫錯欄位名稱。這是因為模型在探索階段閱讀結構描述，但在幾小時（或數千個權杖）後憑記憶撰寫協定。記憶會產生偏移：`document_id` 變成了 `doc_id`，`sentiment_score` 變成了 `sentiment`，`float 0-1` 變成了 `int 0-100`。協定看起來很有權威性，但欄位名稱卻是幻覺。當有人針對真實資料執行品質閘門時，它們會失敗 — 且使用者會對整個產出的劇本失去信心。

**修正方法是程序上的，而非指示上的。** 不要只告訴自己「稍後交叉檢查」 — 請先建立參考表，然後透過複製表格內容來撰寫品質閘門。

在撰寫任何參考產出欄位名稱的品質閘門之前，請透過重新閱讀每個結構描述檔案來建立**欄位參考表**：

```
## 欄位參考表 (根據結構描述建立，非憑記憶)

### 管線：WeatherForecast
結構描述：pipelines/WeatherForecast/schemas/analyze.json
| 欄位 | 型別 | 限制 |
|-------|------|-------------|
| region_name | string | — |
| temperature | number | min: -50, max: 60 |
| condition | string | enum: ["sunny", "cloudy", "rain", "snow"] |

### 管線：SentimentAnalysis
結構描述：pipelines/SentimentAnalysis/schemas/evaluate.json
| 欄位 | 型別 | 限制 |
|-------|------|-------------|
| document_id | string | — |
| sentiment_score | number | min: -1.0, max: 1.0 |
| classification | string | enum: ["positive", "negative", "neutral"] |
...
```

**程序如下：**
1. **在撰寫每個表格列之前，立即重新閱讀每個結構描述檔案。** 不要憑記憶撰寫任何列。檔案讀取與表格列的撰寫必須相鄰 — 讀取檔案，撰寫該列，讀取下一個檔案，撰寫下一列。如果您在對話稍早閱讀過所有結構描述，那不算數 — 您必須在這裡「再次」閱讀，因為您的欄位名稱記憶會隨著數千個權杖而產生偏移。
2. **從檔案內容逐字複製欄位名稱。** 不要重新輸入。`document_id` 不是 `doc_id`。`sentiment_score` 不是 `sentiment`。`classification` 不是 `category`。即使是微小的差異也會破壞品質閘門。
3. **包含結構描述中的所有欄位，而不僅僅是您認為重要的欄位。** 如果結構描述有 8 個必填欄位，則表格應有 8 列。如果您撰寫的列數少於結構描述的欄位數，表示您跳過了欄位。
4. 透過從完成的表格中複製欄位名稱來撰寫品質閘門。
5. 撰寫完成後，清點欄位：如果品質閘門提到的欄位不在表格中，表示您產生了幻覺。請將其移除。

此表格是一個中間產物 — 請將其包含在協定本身中（作為參考章節），以便未來的協定使用者能驗證欄位準確性。重點是將其作為產出結構描述讀取證據的具體步驟，而非因為您「已經知道」欄位而跳過它。

### 校準規模

每次整合測試執行的單位/記錄/反覆運算數量非常重要：

- **太少 (1-3)：** 快速且便宜，但會遺漏並行錯誤、分佈檢查會失敗（無法以 2 筆記錄驗證 「25-75% 比例」），且無法在現實規模下測試分散式分發/擴張邏輯。
- **太多 (100+)：** 對於測試協定而言過於昂貴且緩慢。適用於生產環境，但不適用於品質驗證。
- **正確範圍：** 足以有意義地練習系統。指南：
  - 如果專案具備切塊 (chunking)/分批邏輯，請使用能跨越至少 2 個區塊的數量（例如：如果 chunk_size=10，則使用 15-30 個單位）
  - 如果專案具備分佈檢查，請至少使用類別數量的 5-10 倍（例如：3 種結果類型 → 至少 15 個單位）
  - 如果專案具備分散式分發/擴張功能，請使用能產出非顯著數量之子項目的數量

在專案中尋找 `chunk_size`、`batch_size` 或類似的組態來進行校準。不確定時，10-30 筆記錄通常是整合測試的正確範圍 — 足以捕捉真實問題，又不會燒掉 API 預算。

### 執行後驗證深度

執行完成且無錯誤並不代表其正確。對於每次整合測試執行，請在多個層次進行驗證：

1. **程序層級：** 程序是否乾淨地結束？檢查日誌檔案中是否有完成訊息，而不僅僅是檢查結束代碼。
2. **狀態層級：** 執行是否處於終端狀態？檢查執行資訊清單/狀態檔案是否顯示「已完成」(complete)（而非卡在「執行中」或「已提交」）。
3. **資料層級：** 產出資料是否存在且能正確解析？讀取實際產出的檔案，驗證它們包含有效的 JSON/CSV 等格式。
4. **內容層級：** 產出記錄的預期欄位是否已填入合理的數值？讀取 2-3 筆範例記錄並檢查關鍵欄位。
5. **品質層級：** 管線特定的品質閘門是否通過？執行任何現有的品質檢查腳本。
6. **UI 層級 (如果適用)：** 如果專案具備儀表板/TUI/UI，驗證執行結果是否正確顯示在該處。

將所有適用的層次包含在產出的協定之執行後核取清單中。常見的失敗是停留在第 2 層（程序已完成）而未檢查第 3-5 層。
