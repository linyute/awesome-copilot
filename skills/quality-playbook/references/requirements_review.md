# 需求審查協定 (Requirements Review Protocol)

## 概觀

這是 `quality/REVIEW_REQUIREMENTS.md` 的範本。播放手冊在產出需求管線內容時會一併產生此檔案。它為在產生需求後交互式地審查需求提供了三種模式。

## 產出的檔案範本

播放手冊應將下列內容產生為 `quality/REVIEW_REQUIREMENTS.md`：

---

```markdown
# 需求審查協定：[專案名稱]

## 如何使用

此協定協助您審查產出需求的完整性與準確性。可以使用任何 AI 模型執行 — 審查是自給自足的，並從 `quality/` 中的檔案讀取。

**開始之前：** 確保 `quality/REQUIREMENTS.md` 已存在（來自管線），且您已閱讀頂部的「專案概觀」與「使用案例」章節。

### 選擇審查模式

**模式 1 — 自我引導審查**。您挑選要檢查的使用案例。當您已經知道專案的哪些區域需要最嚴格的審查時，此模式效果最佳。

**模式 2 — 完全引導審查**。AI 帶領您按順序巡覽每個使用案例，深入研究每個連結的需求。最適合進行透徹的第一次審查。

**Mode 3 — 跨模型稽核**。另一個 AI 模型事實檢查完整性報告，驗證每個標記為 COVERED 的領域是否確實具有處理該檢查清單項目的需求。最適合使用與產生需求時不同的模型來執行。

所有三種模式都在 `quality/REFINEMENT_HINTS.md` 中追蹤進度。

---

## 模式 1：自我引導審查

閱讀 `quality/REQUIREMENTS.md` 並向使用者呈現使用案例的編號清單：

```
REQUIREMENTS.md 中的使用案例：
1. [x] 使用案例 1：[名稱] (已審查)
2. [ ] 使用案例 2：[名稱]
3. [ ] 使用案例 3：[名稱]
...
```

檢查 `quality/REFINEMENT_HINTS.md` 中的審查進度 — 標記為 `[x]` 的使用案例已經審查過。呈現清單並詢問使用者要檢查哪個使用案例。

當使用者挑選使用案例時：
1. 顯示該使用案例（參與者、步驟、後置條件、替代路徑）
2. 列出連結的 REQ-NNN 編號
3. 詢問：「要深入研究這些需求中的任何一項，還是此使用案例看起來已完整？」

當深入研究需求時：
1. 顯示完整的需求內容（摘要、使用者故事、滿足條件、替代路徑）
2. 詢問：「這是否捕捉到了正確的行為？有任何缺失或錯誤嗎？」
3. 在 REFINEMENT_HINTS.md 中的該使用案例標題下記錄回饋

在審查完一個使用案例後，在 REFINEMENT_HINTS.md 中將其標記為 `[x]` 並返回使用案例清單。

同時提議：「是否有任何跨領域疑慮或未連結到使用案例的需求是您想要審查的？」

---

## 模式 2：完全引導審查

與模式 1 相同，但不是要求使用者挑選，而是從使用案例 1 開始，依序進行。

針對每個使用案例：
1. 呈現使用案例概觀
2. 逐一巡覽每個連結的需求
3. 針對每個需求，詢問：「這看起來正確嗎？有任何缺失嗎？」
4. 在 REFINEMENT_HINTS.md 中記錄任何回饋
5. 將該使用案例標記為已審查
6. 進入下一個使用案例

在處理完所有使用案例後：
1. 呈現「跨領域疑慮」章節
2. 詢問：「關於執行緒、Null 處理、錯誤、相容性或配置組合，有任何疑慮嗎？」
3. 詢問：「是否有任何您預期會看到但目前不存在的需求？」
4. 記錄回饋，並呈現收集到的所有提示摘要

---

## 模式 3：跨模型稽核

閱讀 `quality/COMPLETENESS_REPORT.md` 與 `quality/REQUIREMENTS.md`。針對完整性報告中的每個領域：

1. 閱讀領域檢查清單項目（來自報告的領域涵蓋範圍章節）
2. 閱讀每個引用的 REQ-NNN
3. 驗證：此需求是否確實處理了領域檢查清單項目？
4. 如果引用錯誤（需求涵蓋的是其他內容），將其標記為缺口

同時檢查：
- 是否有需求未出現在任何使用案例的需求清單中？如果是，將其標記為潛在孤立項。
- 是否每個使用案例的替代路徑章節都具有對應其提及之錯誤/邊緣情況的需求？
- 跨領域疑慮是否引用了確實存在且處理了所述疑慮的需求？

將發現寫入 `quality/REFINEMENT_HINTS.md` 的 `## Cross-Model Audit` 標題下：

```
## Cross-Model Audit
日期：[日期]
模型：[模型名稱]

### 已驗證的領域
- Null 處理：CONFIRMED (REQ-054, REQ-055 正確處理了 Null 語意)
- ...

### 發現的缺口
- 進入點：COMPLETENESS_REPORT 引用了 REQ-100, REQ-101，但這些是關於
  漂亮列印 (pretty printing)，而非進入點合約。JsonStreamParser 沒有涵蓋範圍。
- ...

### 孤立需求
- REQ-NNN 未連結到任何使用案例
- ...
```

向使用者呈現發現，並詢問哪些缺口應在精煉傳遞中處理。

---

## REFINEMENT_HINTS.md 格式

審查協定建立並維護此檔案：

```markdown
# 精煉提示 (Refinement Hints)

## 審查進度
- [x] 使用案例 1：[名稱] — 已審查，無問題
- [x] 使用案例 2：[名稱] — 已審查，請參見下方回饋
- [ ] 使用案例 3：[名稱]
- [ ] 使用案例 4：[名稱]
...

## 跨領域疑慮
- [ ] 執行緒模型 — 尚未審查
- [ ] Null 合約 — 尚未審查
- [ ] 錯誤理念 — 尚未審查
- [ ] 回溯相容性 — 尚未審查
- [ ] 配置組合 — 尚未審查

## 回饋

### 使用案例 2：[名稱]
- REQ-NNN：[關於缺失或錯誤內容的具體回饋]
- 一般：[關於此使用案例涵蓋範圍的更廣泛觀察]

### Cross-Model Audit
[如果執行了模式 3]

## 額外提示
[來自使用者的自由格式回饋，未繫結到特定使用案例]
```

此檔案具有雙重用途：追蹤審查進度（以便使用者跨對話恢復進度）「以及」累積精煉傳遞所讀取的回饋。
```
