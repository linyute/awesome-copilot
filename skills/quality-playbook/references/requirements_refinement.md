# 需求精煉協定 (Requirements Refinement Protocol)

## 概觀

這是 `quality/REFINE_REQUIREMENTS.md` 的範本。播放手冊在產出需求管線內容時會一併產生此檔案。它為根據審查回饋更新需求提供了結構化的程序，且可以使用任何模型執行。

## 產出的檔案範本

播放手冊應將下列內容產生為 `quality/REFINE_REQUIREMENTS.md`：

---

```markdown
# 需求精煉協定：[專案名稱]

## 如何使用

此協定讀取來自 `quality/REFINEMENT_HINTS.md` 的回饋，並更新 `quality/REQUIREMENTS.md` 以彌補識別出的缺口。它可以使用任何 AI 模型執行 — 此協定是自給自足的。

**多模型精煉：** 您可以使用不同的模型多次執行此協定。每次執行都會備份目前版本、進行具針對性的改進、提升次版本號並記錄變更。您可以根據需要執行任意數量的模型，直到遇到報酬遞減。

---

## 開始之前

1. 閱讀 `quality/REFINEMENT_HINTS.md` — 這包含要處理的審查回饋。
2. 閱讀 `quality/REQUIREMENTS.md` — 要更新的目前需求。
3. 閱讀 `quality/CONTRACTS.md` — 以獲取在新增條件時的合約級詳細資訊。
4. 閱讀 `quality/VERSION_HISTORY.md` — 以確定目前版本號。

## 第 1 步：備份與版本控制

1. 從 `quality/VERSION_HISTORY.md` 讀取目前版本。
2. 將 `quality/` 中的所有檔案複製到 `quality/history/vX.Y/` (目前版本號)。
3. 提升次版本號：v1.2 變為 v1.3。
4. 更新 `quality/REQUIREMENTS.md` 頂部的版本標記。

## 第 2 步：處理回饋

閱讀 REFINEMENT_HINTS.md 中的每個項目並將其分類：

- **缺口 — 缺失需求**：某個行為合約或領域區域沒有需求。使用 7 欄位範本建立一個新需求。
- **缺口 — 缺失條件**：現有需求未涵蓋特定情境。向現有需求新增一項滿足條件。
- **缺口 — 缺失使用案例涵蓋範圍**：使用案例未連結到治理其步驟的需求。將 REQ-NNN 新增至使用案例的需求清單中。
- **精確化 — 模糊條件**：滿足條件過於模糊，無法測試。使用具體的通過/失敗標準重寫它。
- **修正 — 錯誤內容**：需求說明了不正確的事項。修正特定的欄位。
- **跨模型稽核發現**：完整性報告中某個領域標記為 COVERED，但引用的需求實際上並未處理它。新增缺失的需求。
- **移除（僅限使用者指示）**：使用者明確指出某項需求不正確且應移除（例如：「REQ-047 不正確，因為 X — 請移除」）。僅在提示清楚來自使用者而非自動化傳遞時才處理移除。在變更報告中記錄移除及其原因。

## 第 3 步：執行變更

針對每個回饋項目：

1. **新需求**：新增至適當的類別章節末尾。延續現有的編號順序。嚴格遵循 7 欄位範本。
2. **修改需求**：編輯需要變更的特定欄位。不要重寫未標記的需求。
3. **使用案例更新**：將新建立的 REQ 編號新增至相關使用案例的需求清單中。
4. **跨領域疑慮**：如果新需求影響跨領域疑慮，請更新那些章節。

### 規則

- **在自動化精煉期間，不要刪除或削弱現有的需求。** 目前存在的每項需求在精煉後必須繼續存在，且至少具有相同的滿足條件 — 除非使用者已明確標記某項需求為移除並提供原因。使用者指示的移除是唯一例外。
- **不要為現有需求重新編號。** 新需求獲取下一個可用編號。這保證了跨版本的可追溯性。
- **不要重組文件結構。** 敘述階段已經建立了結構。精煉是手術式的 — 新增、精確化或修正個別項目。
- **每項變更都必須可追溯到某個回饋項目。** 不要進行未被要求的變更。

## 第 4 步：報告變更

在執行所有變更後，向 `quality/REFINEMENT_HINTS.md` 附加摘要：

```
## 精煉傳遞 — v[新版本]
日期：[日期]
模型：[模型名稱]

### 已執行的變更
- REQ-NNN (NEW)：[簡短描述] — 處理回饋：「[引用的提示]」
- REQ-NNN：新增了關於 [內容] 的滿足條件 — 處理回饋：「[引用的提示]」
- REQ-NNN：精確化條件 #N：[變更內容] — 處理回饋：「[引用的提示]」
- 使用案例 N：向需求清單新增了 REQ-NNN

### 未處理的回饋項目
- 「[引用的提示]」 — 原因：[為什麼此項不可執行或不在範圍內]

### 摘要
新增了 N 個新需求，修改了 N 個現有需求，更新了 N 個使用案例。
需求總數：N (之前為 N)。
```

## 第 5 步：更新版本歷程記錄

向 `quality/VERSION_HISTORY.md` 新增一列：

```
| vX.Y | YYYY-MM-DD | [模型] | [作者] | N | [變更摘要] |
```

## 第 6 步：更新完整性報告

如果新增的新需求處理了領域檢查清單的缺口，請更新 `quality/COMPLETENESS_REPORT.md` 中相關的領域條目，以引用新的 REQ 編號。

---

## 執行多次精煉傳遞

每次精煉傳遞遵循相同的協定：
1. 讀取最新的 REFINEMENT_HINTS.md（該檔案現在含有先前傳遞的報告）
2. 只專注於標記為「未處理」的回饋項目或自上次傳遞以來新增的回饋
3. 備份、提升版本、進行變更並產生報告

使用者可以在傳遞之間直接編輯 REFINEMENT_HINTS.md 以新增提示。下一次精煉傳遞會自動擷取這些新增項目。

使用者也可以在精煉傳遞之間執行一次新的跨模型稽核（審查協定的模式 3），以找出先前精煉未發現的缺口。這會形成一個「審查 → 精煉 → 審查 → 精煉」的循環，最終朝完整性收斂。
```
