# 用於漏洞發現的探索模式

此參考檔案定義了階段 1 在程式碼庫探索期間套用的探索模式。這些模式針對的是當探索僅停留在子系統或架構層級時，最常被遺漏的漏洞類別。

需求問題是修復成本最高的，因為它們直到實作後才會被發現。探索階段即是需求引出 (requirements elicitation) — 它決定了程式碼審查和規格稽核將尋找什麼。未衍生的需求即是永遠無法發現的漏洞。這些模式的存在是為了系統性地發掘廣泛探索所遺漏的需求。

每種模式都包含定義、其針對的漏洞類別、來自不同領域的多樣化範例，以及 EXPLORATION.md 預期的輸出格式。

**重要提示：這些模式是對自由探索的補充 — 它們不能取代自由探索。** 階段 1 從由領域知識和程式碼庫理解驅動的開放式探索開始。在該開放式探索之後，套用下方的模式作為結構化的第二次傳遞，以擷取特定的漏洞類別。如果您發現自己只尋找模式所描述的事項，那麼您的用法是錯誤的。這些模式是在您已經對程式碼庫的風險形成自己的理解之後，需要執行的清單。

## 模式 1：後備與降級路徑的一致性 (Fallback and Degradation Path Parity)

### 定義

當程式碼為實現同一個目標提供多種策略時 — 一條主要路徑和一條或多條後備路徑 — 每條後備路徑必須保留與主要路徑相同的行為不變量。後備方案可能使用不同的機制，但可觀察到的合約必須是等效的。

### 漏洞類別

後備路徑的撰寫時間較晚，測試較少，且受到的審查嚴格程度低於主要路徑。它們通常會遺漏主要路徑執行的步驟（驗證、清理、索引分配、資源釋放），因為開發人員複製了主要路徑並為「降級」情況進行了簡化。結果是一個函式在常見情況下運作正確，但在後備方案啟動時違反了其合約。

### 跨領域範例

- **身分驗證**：Web 服務嘗試 OAuth 權杖驗證，後備為 API 金鑰查閱，再後備為對話 Cookie。每個後備方案都必須執行相同的授權範圍。漏洞：API 金鑰後備方案跳過了範圍驗證並授予了完整存取權限。
- **連線池**：資料庫用戶端嘗試主要連線池，後備為次要池，再後備為建立一次性連線。每條路徑都必須套用相同的逾時和交易隔離設定。漏洞：一次性連線後備方案使用了驅動程式預設的隔離層級，而非設定的層級。
- **資源分配**：記憶體分配器嘗試快速的 slab 路徑，後備為慢速的分頁級路徑。兩者都必須將敏感欄位初始化為零。漏洞：慢速路徑傳回未初始化的記憶體，因為填零僅在 slab 快速路徑中執行。
- **HTTP 重新導向處理**：當重新導向跨越來源邊界時，用戶端遵循重新導向且必須移除安全性敏感標頭（Authorization, Proxy-Authorization, cookies）。漏洞：重新導向路徑移除了 Authorization 但未移除 Proxy-Authorization，導致代理伺服器認證洩露給重新導向後的來源。
- **序列化後備**：訊息代理程式嘗試二進位序列化，後備為 JSON，再後備為字串編碼。每條路徑都必須保留相同的欄位順序和 Null 處理語意。漏洞：JSON 後備方案默默捨棄了二進位序列化所保留的 Null 欄位。

### 如何套用

針對每個核心模組，尋找：嘗試一種方法然後落入另一種方法的條件鏈；在執行階段選取多個實作的策略/配接器模式；每次嘗試使用不同策略的重試邏輯；功能協商級聯，其中功能決定執行哪條程式碼路徑；必須保留或移除標頭的 HTTP 重新導向/重試邏輯。

針對發現的每個級聯：
1. 列出主要路徑和每個後備方案。
2. 對於每個後備方案，檢查其是否執行與主要路徑相同的關鍵操作（驗證、資源設定、索引分配、清理、錯誤報告、標頭移除、資源釋放）。
3. 主要路徑中存在但在後備方案中缺失的任何操作都是候選需求。

### EXPLORATION.md 輸出格式

```
## 後備路徑分析

### [級聯名稱]
- **主要路徑：** [函式, 檔案:列] — [其功能]
- **後備方案 1：** [函式, 檔案:列] — [其功能，有何不同]
- **後備方案 2：** [函式, 檔案:列] — [其功能，有何不同]
- **一致性缺口：** [主要路徑中存在但在後備方案中缺失的特定操作]
- **候選需求：** REQ-NNN：[後備方案必須執行 X]
```

## 模式 2：分派器傳回值的正確性 (Dispatcher Return-Value Correctness)

### 定義

當一個函式根據輸入類型或條件進行分派且必須傳回一個狀態值時，該傳回值必須對所有輸入組合都是正確的 — 而不僅僅是主要情況。處理多種事件類型、要求類型或狀態轉換的分派器，在邊緣組合中特別容易出現傳回值錯誤。

### 漏洞類別

分派器通常是針對常見情況撰寫和測試的。當主要事件觸發時，傳回值是正確的。但當不尋常的組合發生時（僅觸發次要事件、完全沒有事件、多個並行事件），傳回值邏輯可能是錯誤的 — 對於已處理的事件傳回「未處理」，對部分失敗傳回成功，或傳回前一次反覆運算的過期值。

### 跨領域範例

- **HTTP 中間件**：要求分派器檢查身分驗證、頻率限制和路由。當觸發頻率限制但身分驗證已設定時，分派器傳回身分驗證狀態碼，而非頻率限制狀態碼。漏洞：受頻率限制的要求收到 401 而非 429。
- **CORS 處理常式鏈**：CORS 預檢處理常式設定 400（被拒絕），然後缺失的 OPTIONS 處理常式路徑設定 404，接著 AFTER 處理常式將 404 標準化為 200（旨在用於允許的來源）。漏洞：被拒絕的預檢要求收到 200，因為狀態碼被下游處理常式覆寫。
- **事件迴圈**：輪詢/選取迴圈處理可讀、可寫和錯誤條件。當僅在沒有擱置讀取的通訊端上觸發錯誤條件時，迴圈會傳回「無事件」，因為可讀檢查為 false。漏洞：連線錯誤被默默忽略。
- **狀態機轉換**：狀態機分派函式處理有效轉換、無效轉換和無操作 (no-op) 轉換。當發生無操作轉換（目前狀態 == 目標狀態）時，函式傳回旨在用於無效轉換的錯誤代碼。漏洞：冪等 (idempotent) 操作在應成功時卻失敗。
- **中斷處理常式**：硬體中斷處理常式檢查多種事件類型（資料就緒、組態變更、錯誤）。當僅觸發次要事件（例如組態變更但無資料）時，處理常式傳回「不是我的」，因為主要事件檢查失敗，且次要路徑未設定已處理旗標。漏洞：合法的次要事件被報告為虛假中斷。

### 如何套用

針對每個核心模組，尋找：具有傳回狀態之 switch/case 或 if-else 鏈的函式；處理多種事件類型的中斷/事件處理常式；在傳回前檢查多個條件的要求分派器；狀態機轉換函式；多個處理常式寫入同一個回應狀態的中間件鏈。

針對發現的每個分派器：
1. 列舉所有輸入組合（不只是具有明確 case 標籤的組合 — 還有隱含的 "else" 和 "default" 路徑）。
2. 針對每種組合，追蹤傳回值在整個處理常式鏈中的傳遞（不只是立即的函式）。
3. 傳回值與預期語意不符的任何組合都是候選需求。

### EXPLORATION.md 輸出格式

```
## 分派器傳回值分析

### [函式名稱] 位於 [檔案:列]
- **輸入類型：** [函式分派所根據的條件/事件清單]
- **已檢查的組合：**
  - [僅條件 A]：傳回 [X] — 正確/錯誤，原因為 [原因]
  - [僅條件 B]：傳回 [X] — 正確/錯誤，原因為 [原因]
  - [A 和 B 皆成立]：傳回 [X] — 正確/錯誤，原因為 [原因]
  - [A 和 B 皆不成立]：傳回 [X] — 正確/錯誤，原因為 [原因]
- **候選需求：** REQ-NNN：[當僅有 B 觸發時，函式必須傳回 Y]
```

## 模式 3：跨實作合約一致性 (Cross-Implementation Contract Consistency)

### 定義

當多個函式針對不同上下文（不同傳輸方式、不同後端、不同通訊協定版本）實作相同的邏輯操作時，它們都應滿足相同的規格需求。規格中強制要求的步驟必須出現在每個實作中 — 某個實作中缺失而另一個實作中存在的步驟是一個強烈的漏洞訊號。

### 漏洞類別

當同一個操作在多個地方實作時，每個實作通常由不同的開發人員或在不同時間編寫。規格說明「重設必須等待完成」，實作 A 的開發人員編寫了等待迴圈，但實作 B 的開發人員只編寫了重設觸發器而忘記了等待。在單獨測試實作 B 時，該漏洞是不可見的，因為它在快速硬體上可以「正常運作」 — 競爭條件僅在負載下或在慢速裝置上才會顯現。

### 跨領域範例

- **裝置重設**：規格說明「驅動程式必須寫入零，然後輪詢直到狀態暫存器讀回為零」。PCI 實作包含輪詢迴圈。MMIO 實作寫入零但未輪詢。漏洞：MMIO 重設可能與重新初始化發生競爭。
- **資料庫驅動程式**：連線關閉規格說明「驅動程式必須發送終止訊息，等待確認，然後釋放通訊端」。PostgreSQL 驅動程式執行了這三個步驟。MySQL 驅動程式發送終止訊息並釋放通訊端，而未等待確認。漏洞：伺服器可能在通訊端被重複使用後才處理終止訊息。
- **HTTP 標頭編碼**：Headers 類別建構函式根據 RFC 7230 將原始位元組解碼為 Latin-1。變更方法 (`__setitem__`) 將值編碼為 UTF-8。漏洞：透過「取得再設定」循環處理 Latin-1 標頭會導致數值損壞，因為編碼發生了變更。
- **快取失效**：快取規格說明「失效操作必須移除項目並通知所有訂閱者」。記憶體內快取執行了這兩項操作。分散式快取移除了項目但未廣播通知。漏洞：其他節點提供過期資料。
- **檔案鎖定**：儲存規格說明「獲取鎖定必須設定逾時並在失敗時清理」。本機檔案系統實作設定了逾時。NFS 實作使用封鎖式鎖定且無逾時。漏洞：NFS 鎖定爭用可能導致程序無限期掛起。

### 如何套用

針對每個核心模組，尋找：在多個檔案或類別中實作的同一個操作名稱；跨不同後端的介面/特徵 (trait) 實作；同一個訊息的通訊協定版本專屬實作；同一個生命週期操作的傳輸專屬實作；同一個邏輯操作的建構函式與變更實作。

針對每一對（或每一組）實作：
1. 識別它們共享的規格需求。
2. 列出規格中的強制性步驟。
3. 檢查每個實作中的每個步驟。
4. 一個實作中存在但另一個實作中缺失的任何步驟都是候選需求。

**檢查每個跨傳輸操作，而不僅僅是最明顯的一個。** 如果程式碼庫具有多種傳輸方式 (PCI, MMIO, vDPA) 或後端 (PostgreSQL, MySQL)，請列舉所有具有跨實作等效項的操作 — 重設、中斷處理、功能協商、佇列設定、設定存取 — 並檢查每一項。您找到的第一個跨實作缺口很少是唯一的一個。常見的失敗模式是徹底分析了重設，但跳過了具有相同跨傳輸結構的中斷分派。

### EXPLORATION.md 輸出格式

```
## 跨實作一致性

### [操作名稱] — [規格參考]
- **實作 A：** [函式, 檔案:列] — 執行步驟：[1, 2, 3]
- **實作 B：** [函式, 檔案:列] — 執行步驟：[1, 3] (缺失步驟 2)
- **缺口：** [實作 B 缺失步驟 2：描述]
- **候選需求：** REQ-NNN：[X 的所有實作都必須執行步驟 2]
```

## 模式 4：列舉與表示的完整性 (Enumeration and Representation Completeness)

### 定義

當程式碼庫維護一組封閉的公認值時 — 一個 switch/case 白名單、一個有效常數陣列、一個列舉/標記聯集定義、一個特徵/訪問者方法家族、一組結構描述關鍵字、一個公認條目註冊表 — 每個由規格、上游定義或程式碼庫自身公共 API 表面指出應被接受的值都必須出現在該集合中。不在集合中的值會被默默捨棄、拒絕或錯誤處理，且條目的缺失在呼叫端是不可見的。

### 漏洞類別

封閉集合在撰寫後很少被重新檢視。當規格或上游標頭新增了新功能時，定義該功能的程式碼（常數、特徵旗標、列舉變體）會被更新，使用該功能的程式碼也會被更新，但用於過濾該功能是否能存活的封閉集合卻被遺忘了。該功能看似受支援 — 它已定義、已協商、已使用 — 但它被一個沒人記得更新的過濾函式默默移除了。該漏洞在正常測試中是不可見的，因為該功能根本沒有啟動，而缺乏啟動看起來像是「另一端不支援」。

此模式還涵蓋了必須鏡像公共 API 的 **內部表示**。如果程式碼庫的公共 API 接受 i128/u128 整數，但用於去標記 (untagged) 和內部標記列舉還原序列化的內部緩衝表示 (如 `Content` 列舉) 僅具有 i64/u64 變體，則通過緩衝區的值會被默默截斷或拒絕 — 儘管公共 API 承諾會處理它們。

### 跨領域範例

- **功能協商過濾器**：傳輸層維護一個特徵位元的 switch/case 白名單。UAPI 標頭新增了新特徵 (`RING_RESET`) 並被更高層級的程式碼使用，但從未被新增到白名單中。漏洞：該特徵在協商期間被默默清除，進而停用了驅動程式聲稱支援的功能。
- **序列化內部表示**：序列化程式庫的公共 `Deserializer` 特徵支援 `deserialize_i128()`/`deserialize_u128()`。用於去標記和內部標記列舉還原序列化的內部緩衝表示 (`Content` 列舉) 僅具有 `I64`/`U64` 變體。漏洞：通過緩衝區的 128 位元整數會因「無 i128 變體」錯誤而被拒絕，儘管公共 API 聲稱支援它們。
- **結構描述關鍵字匯入器**：驗證程式庫匯入 JSON 結構描述文件。規格為陣列定義了 `uniqueItems`、`contains`、`minContains`、`maxContains`。匯入器能識別這些關鍵字（無解析錯誤）但未執行強制檢查。漏洞：匯入的結構描述會默默接受違反原始約束的陣列。
- **權限系統**：授權中間件維護一個公認權限字串的陣列。角色定義中新增了新權限 (`audit:write`)，但未新增到中間件的白名單中。漏洞：具有 `audit:write` 角色的使用者被默默拒絕存取，因為中間件無法識別該權限。
- **通訊協定訊息類型**：訊息路由器為公認的訊息類型維護 switch/case 分派。通訊協定規格和序列化層中新增了新訊息類型，但未新增到路由器中。漏洞：新訊息類型被路由器的 default case 默默捨棄，且發送者未收到錯誤。

### 如何套用

針對每個核心模組，尋找：具有明確 case 標籤且 default 會捨棄/清除/拒絕的 switch/case 陳述式；用於過濾或驗證的公認值陣列或集合；必須手動新增新條目的註冊函式；鏡像規格或公共 API 的列舉/標記聯集定義；每個方法處理一個變體的特徵/訪問者方法家族；必須處理規格定義之每個關鍵字的結構描述匯入器；必須涵蓋公共介面全範圍的內部表示（緩衝區、IR, AST）。

針對發現的每個封閉集合：
1. 識別定義哪些值應為有效的權威來源。這可能是：規格、標頭檔、上游列舉、通訊協定定義，**或者是程式碼庫自身的公共 API 表面**（特徵方法、函式簽章、型別定義）。
2. 機械式地擷取封閉集合（將 case 標籤、列舉變體、訪問者方法、陣列條目或結構描述關鍵字儲存到檔案中）。
3. 將擷取的集合與權威來源進行比較。權威來源中存在但封閉集合中缺失的每個值都是候選需求。

**呼叫端的補償不能作為遺漏條目的理由。** 如果共享/通用函式中的封閉集合缺失了一個條目，那就是漏洞 — 即使特定的呼叫者在函式執行後透過恢復數值來進行補償。該補償是一種因應措施，而非修復。任何不知道要進行補償的新呼叫者都會默默繼承該漏洞。將每個缺失的條目報告為一個發現，並註明哪些呼叫者（如果有）執行了補償，但不要因為補償而忽略該發現。

### [函式/型別名稱] 位於 [檔案:列]
- **用途：** [此封閉集合過濾的對象 — 例如，「在傳輸過濾後保留的特徵位元」或「緩衝區可以持有的整數變體」]
- **權威來源：** [定義有效值之處 — 例如，「include/uapi/linux/virtio_config.h」或「公共 Deserializer 特徵方法」]
- **擷取的條目：** [封閉集合中的值清單，或機械式擷取檔案的參考]
- **缺失的條目：** [權威來源中存在但封閉集合中缺失的值]

## 模式 5：API 表面一致性 (API Surface Consistency)

### 定義

當同一個邏輯操作可以透過多個 API 表面執行時 — 直接方法 vs. 檢視/包裝器、建構函式 vs. 變更方法 (mutator)、同步 vs. 非同步變體、主要 API vs. 便利別名 — 所有表面對於相同的輸入都必須產生等效的可觀察行為。兩條通往同一個操作的路徑之間的差異即是漏洞，因為呼叫者有理由預期無論使用哪個表面，行為都是一致的。

### 漏洞類別

程式碼庫通常透過多個介面公開相同的底層資料：直接方法與集合檢視 (`add()` vs. `asList().add()`)、建構函式與 Setter、同步與非同步變體。這些表面在不同時間實作，通常由不同開發人員完成，且它們的邊緣情況處理方式互有分歧 — 特別是在 Null/哨兵值、編碼、排序和錯誤報告方面。這種分歧在正常測試中是不可見的，因為測試通常針對每個操作僅練習一個表面。

### 跨領域範例

- **JSON Null 處理**：`JsonArray.add(null)` 將 Null 轉換為 `JsonNull.INSTANCE` 並成功執行。`JsonArray.asList().add(null)` 拋出 `NullPointerException`，因為檢視的包裝器無條件拒絕 Null。漏洞：同一個操作的兩個方法具有矛盾的 Null 語意。
- **HTTP 標頭編碼**：`Headers([(b"X-Custom", b"\xe9")])` 從 Latin-1 位元組建構標頭。`headers["X-Custom"] = b"\xe9"` 以 UTF-8 儲存值。漏洞：透過「取得再設定」循環處理標頭會默默變更編碼。
- **WebSocket 通訊協定協商**：`WebSocketUpgrade::protocols()` 傳回一個 `BTreeSet<HeaderValue>`，它會對用戶端按偏好排序的通訊協定清單進行排序和去重複。漏洞：應用程式看到的順序與用戶端發送的順序不同，打破了基於偏好的協商。
- **設定選項傳播**：`res.sendFile(path, { etag: false })` 應為此回應停用 ETag。但程式碼在傳遞給底層 `send` 模組之前將選項轉換為布林值，遺失了「強」與「弱」ETag 模式。漏洞：單次呼叫的 ETag 設定被默默忽略或發生有損轉換。
- **Map 重複項偵測**：`map.put(key, value)` 傳回先前的值以發出重複訊號。當先前的值合法地為 `null` 時，`put()` 傳回 `null` — 與「無先前條目」傳回的值相同。漏洞：當第一個值為 Null 時，重複鍵無法偵測。

### 如何套用

針對每個核心模組，尋找：由 `asList()`、`asMap()`、`unmodifiableView()`、`stream()`、`iterator()` 等方法傳回的檢視/包裝器物件；建構函式與變更方法對；同一個操作的同步與非同步變體；委派給主要實作的便利別名；接受選項/設定物件的方法。

針對每一對表面：
1. 識別它們共享的邏輯操作。
2. 在兩個表面上測試相同的邊緣情況輸入（Null、空值、邊界值、特殊字元、區分順序的資料）。
3. 行為上的任何差異（不同的異常、不同的編碼、不同的順序、一個成功另一個失敗）都是候選需求。

```

### [操作名稱] — [比較的兩個表面]
- **表面 A：** [方法, 檔案:列] — [邊緣輸入的行為]
- **表面 B：** [方法, 檔案:列] — [同一邊緣輸入的行為]
- **分歧：** [不同之處 — 異常類型、編碼、順序、Null 處理]
- **候選需求：** REQ-NNN：[兩個表面對於輸入 X 必須表現得等效]
```

## 模式 6：符合規格結構的解析保真度 (Spec-Structured Parsing Fidelity)

### 定義

當程式碼解析由正式語法或規格定義的值時 — HTTP 標頭、URL、MIME 類型、CLI 旗標、JSON 結構描述關鍵字、檔案路徑 — 解析必須符合語法的實際規則。捷徑（子字串匹配、完全相等、錯誤的分隔符號、無邊界檢查的前綴匹配）會產生適用於常見輸入但在有效邊緣情況下失敗或接受無效輸入的解析器。

### 漏洞類別

開發人員經常實作「夠用就好」的解析器來處理常見情況：使用 `header.contains("gzip")` 而非按逗號分記 (tokenize) 並修剪空白；使用 `url.startsWith("/api")` 而非檢查路徑段 (path segment) 邊界；使用 `connection == "Upgrade"` 而非不區分大小寫的權杖清單成員檢查。這些捷徑能通過所有單元測試，因為測試使用格式正確的輸入，但它們在現實世界的邊緣情況下會崩潰，例如 `gzip;q=0`（明確拒絕）、`Connection: keep-alive, Upgrade`（權杖清單）或 `/api-docs`（無邊界的前綴匹配）。

### 跨領域範例

- **HTTP Accept-Encoding**：中間件檢查 `accept.contains("gzip")` 以決定是否壓縮。這匹配了 `gzip;q=0`（用戶端明確拒絕 gzip）和 `xgzip`（不是有效的編碼）。漏洞：在用戶端說明不壓縮的情況下壓縮了回應。
- **WebSocket Connection 標頭**：程式碼檢查 `connection == "Upgrade"`（精確匹配）。根據 RFC 7230，`Connection` 是逗號分隔的權杖清單；`Connection: keep-alive, Upgrade` 是有效但無法通過精確匹配。漏洞：合法的 WebSocket 升級要求被拒絕。
- **SPA 後備路由**：單頁應用程式處理常式使用 `path.startsWith("/app")` 匹配路徑。這同時匹配了 `/app/users`（正確）和 `/api-docs`（不正確的同級路由）。漏洞：API 文件要求被 SPA 處理常式吞沒。
- **MIME 類型參數處理**：內容協商將 `text/html;level=1` 與處理常式索引鍵進行比較，但在匹配前去除了參數。漏洞：協商期間選取的 `level=1` 參數在回應 Content-Type 中遺失。
- **URL 主機標準化**：程式碼透過檢查 `host.startsWith("xn--")` 來偵測國際化域名。根據 IDNA，只有個別標籤以 `xn--` 開頭；`foo.xn--example.com` 的 Punycode 標籤位於中間。漏洞：國際化子域名未被解碼。

### 如何套用

針對每個核心模組，尋找：對 RFC 或規格定義的值（標頭、URL、MIME 類型、編碼名稱）進行的字串比較；針對結構化值的 `contains()` / `indexOf()` / `startsWith()` / `endsWith()`；規格要求不區分大小寫但實作為區分大小寫的比較；在錯誤的分隔符號上拆分或完全不拆分；無路徑段或權杖邊界的前綴/字尾匹配。

針對發現的每個解析器：
1. 識別定義語法的規格（RFC, ABNF, JSON 結構描述規格, POSIX 等）。
2. 檢查實作是否處理了：權杖清單（逗號分隔）、帶引號字串、參數（分號分隔）、大小寫轉換、空白修剪、邊界條件。
3. 建構一個符合規格但會使實作捷徑解析器失敗的輸入。該輸入是候選測試案例，且解析缺口是候選需求。

### [解析器位置] 位於 [檔案:列]
- **規格：** [哪個語法/RFC/標準定義了該格式]
- **實作技術：** [contains/equals/startsWith/split-on-X]
- **會破壞解析器的規格有效輸入：** [具體範例]

## 模式 7：組合與掛載上下文感知 (Composition and Mount-Context Awareness)

### 定義

當程式碼在組合上下文 (composed context) 中運作時 — 掛載於子路由、巢狀於父模組、範圍限定於子容器、被框架配接器包裝 — 框架通常會維護活動狀態的 *標準 (canonical)* 表示（目前活動上下文聲稱為真的內容），以及來自外部呼叫位置的 *原始 (raw)* 表示（外部呼叫者最初傳遞的內容）。在需要標準表示時讀取或寫入原始表示的程式碼（或反之亦然），在外部層級可以正常運作（兩者在那裡恰好相同），但在組合情況下會發生靜默失敗（兩者在那裡產生分歧）。

### 漏洞類別

程式碼是在外部層級撰寫和測試的 — 頂層路由、根模組、單租戶部署、預設範圍 — 在那裡標準和原始狀態是相同的。當相同的程式碼在組合上下文中執行時，框架會更新標準狀態（掛載的子路徑、限定範圍的記錄器、交易限定範圍的連線、語言環境感知比較器），但原始狀態仍反映外部呼叫。缺陷以兩個對稱的方向顯現：讀取原始狀態而需要標準狀態的函式會看到過時資料並產生靜默飄移（永遠不匹配、洩漏父項上下文、傳回錯誤輸出）；從標準狀態寫入向外輸出的數值的函式會產出取用者無法使用的輸出（遺失掛載前綴、傳回父項用戶端無法遵循的子項相對路徑）。無論哪種情況，測試套件通常僅練習外部層級，從未看到分歧。

### 跨領域範例

- **HTTP 路由中間件 (掛載上下文)**：將要求路徑與設定終端節點進行比較的中間件讀取 `r.URL.Path`。當掛載於子路由時，框架的標準「活動路由路徑」（例如 chi 中的 `RoutePath`, Express 子應用程式中的 `req.url`）是子相對路徑，而 `r.URL.Path` 仍是完整的 URL 路徑。漏洞：中間件在掛載的子路徑內永遠不匹配，因為它讀取了錯誤的路徑表示。
- **資料庫交易上下文**：儲存庫方法透過連線池開啟其自身的連線。當在明確交易中被呼叫時，框架的標準「當前交易」上下文是該明確交易，但該方法直接從連線池讀取。漏洞：該方法的寫入不參與周邊交易；還原 (rollback) 會留下孤立資料列。
- **記錄上下文傳播**：程式庫透過 `logging.getLogger(__name__)` 進行記錄。當在限定了基於 contextvar 的關聯 ID 的非同步工作或工作者池中調用時，記錄器不讀取 contextvar。漏洞：程式庫的記錄列缺少框架傳播的關聯 ID，打破了可追溯性。
- **語言環境敏感比較**：排序函式使用 `str.lower()` 進行不區分大小寫的比較。當在語言環境感知上下文（土耳其語 "i" / "İ" / "ı" 語意）中被呼叫時，框架的標準語言環境已設定，但 `str.lower()` 讀取的是預設語言環境。漏洞：相等性比較會根據哪個語言環境在標準狀態下為活動狀態而產生靜默差異。
- **授權範圍繼承**：ACL 檢查讀取 `request.user`（原始驗證主體）。當在模擬 (impersonation) 上下文中被調用時，框架的標準 `request.effective_user` 是模擬主體，但檢查仍讀取原始主體。漏洞：權限提升 — 檢查授權了錯誤的主體。

### 如何套用

識別讀取或寫入在組合下可能出現「標準與原始」差異之狀態的每個函式或元件。檢查點是：當其呼叫者被組合在更大的上下文中時，此程式碼路徑是否保持不變？如果是，它觀察（或產生）的狀態是否發生相應變化？

**與模式 4 的區別。** 模式 4（列舉與表示完整性）是關於封閉的值集合：漏洞是「識別器的封閉集合中缺失了值」。模式 7 是關於狀態變數的選擇：漏洞是「函式在組合下讀取或寫入了錯誤的狀態表示」。如果兩個框架似乎都適用，請優先選取 REQ 更具可測試性的那一個。這兩個模式很少在同一個缺陷上重疊；當重疊時，標準與原始的框架通常更直接地指向修復方法。

**預算。** 將候選項目限制在每次傳遞中 3-5 個影響最大的組合縫隙。如果僅從此模式中就產生了超過 10 個候選項目，則網撒得太廣，且該模式被過度套用 — 請帶著更嚴格的「此框架在組合下實際維護什麼標準狀態」過濾器重新檢視步驟 1。

針對發現的每個候選項目：

1. **識別讀取和寫入的標準與原始表示。** 框架在組合下將什麼維護為「此問題的活動狀態」？函式實際讀取了什麼？然後詢問對稱的輸出問題：當此函式建構一個向外的數值（重新導向目標、衍生路徑、記錄的關聯 ID、授權的資源控制代碼）時，該數值是否針對其取用者使用了正確的表示來建構？讀取端和寫入端的缺陷同樣常見；請檢查兩個方向。
2. **追蹤組合縫隙。** 框架在何處更新標準狀態？函式是否在該更新位置的下游？它讀取的是標準表示還是原始表示？
3. **建構組合測試。** 此程式碼在組合父項（掛載路由器、巢狀交易、限定範圍記錄器、模擬上下文）內執行的最小範例是什麼？函式的行為是否符合外部層級行為，還是會默默飄移？
4. **記錄發生的情況。** 在組合下飄移的函式即是候選需求：「函式 `<X>` 必須讀取 `<canonical_state>` (而不是 `<raw_state>`) [或為向外輸出寫入 `<raw_state>`]，以便在組合進 `<parent_context>` 時行為保持正確。」

值得明確檢查的常見組合縫隙：

- **路由框架：** `RoutePath` / `req.url` / `request.path_info` 相對於原始 URL。每一層掛載都會更新標準路徑；原始 URL 則不會。
- **交易管理員：** 明確交易上下文相對於連線池的自動提交預設值。組合後的程式碼必須讀取活動交易。
- **記錄 / 追蹤：** 限定範圍的 contextvar 關聯 ID 相對於執行緒區域或預設記錄器。組合後的程式碼必須讀取 contextvar。
- **授權 / 模擬：** 有效主體相對於原始使用者。組合後的程式碼必須讀取有效主體。

### [函式/元件名稱] 位於 [檔案:列]
- **組合進：** [父項上下文 — 例如，「任何呼叫 Mount() 來附加此中間件的 chi 路由器」]
- **標準表示：** [框架在組合下維護的內容 — 例如，「rctx.RoutePath，活動的掛載路徑」]

## 擴充此清單

這些模式源自於對跨越 7 種語言、11 個開源存放庫的 56 個已確認漏洞的分析。每種模式代表了一類廣泛架構摘要始終遺漏的需求。

若要新增模式：
1. 識別一個探索時遺漏但可以透過特定分析技術找到的已確認漏洞。
2. 將技術一般化：探索者應該對程式碼提出什麼問題？
3. 提供至少 5 個來自不同領域的多樣化範例（不全來自同一個專案）。
4. 定義 EXPLORATION.md 預期的輸出格式。
5. 將模式新增至此檔案，並將對應章節新增至 SKILL.md 中的 EXPLORATION.md 範本。

目標是建立一個系統化探索技術庫，隨著新漏洞類別的發現而與時俱進。
