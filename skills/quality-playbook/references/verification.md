# 驗證核取清單 (第三階段)

在宣告品質劇本完成之前，請檢查下列各項基準。如果任何一項失敗，請回去進行修正。

## 自我檢查基準

### 1. 測試計數

計算啟發式目標值：(可測試的規格章節) + (QUALITY.md 案例) + (來自步驟 5 的防禦性模式)。

- **遠低於目標** → 您可能遺漏了規格需求或在防禦性模式上走馬看花。請回去重新檢查。
- **接近目標** → 檢閱您是否測試了負面案例與邊界。
- **高於目標** → 沒問題，只要每個測試都有意義即可。請勿為了湊數字而填充內容。

### 2. 案例涵蓋率

計算 QUALITY.md 中的案例數量。計算功能測試檔案中的案例測試函式數量。這兩個數字必須完全相符。

### 3. 跨變體涵蓋率

如果專案處理 N 個輸入變體，有多少百分比的測試會練習所有 N 個變體？

計算方式：循環執行或參數化所有變體的測試數量 / 總測試數量。

**啟發式目標：約 30%。** 如果遠低於此值，請尋找應該參數化的單一變體測試。候選對象通常包括：結構完整性、身分驗證、必要欄位存在性、資料關聯、語義正確性。確切的百分比並不重要，重要的是確保橫切屬性已在所有變體中完成測試。

### 4. 邊界與負面測試計數

計算來自步驟 5 的防禦性模式數量。計算您的邊界/負面測試數量。比例應接近 1:1。如果顯著較低，請針對未測試的防禦性模式撰寫更多測試。

### 5. 斷言深度

掃描您的斷言。有多少是檢查存在性，有多少是檢查數值？如果超過一半僅檢查存在性 (`assert x is not None`, `assert x in output`)，請強化它們以檢查實際數值。

### 6. 層級正確性

針對每個測試，詢問：「我是在測試『需求』還是在測試『機制』？」如果任何測試僅斷言引發了特定的錯誤型別，而未驗證管線產出，則它是在測試機制。請重寫以測試結果。

### 7. 變動有效性

針對每個變動 fixture 的測試，驗證變動後的數值是否在步驟 5b 結構描述對映表的「接受」欄位中。如果任何變動使用了結構描述拒絕的型別，測試會因為驗證錯誤而失敗，而無法測試到防禦性程式碼。請修正此問題。

### 8. 所有測試皆通過 — 零失敗且零錯誤

使用專案的測試執行器執行測試套件：

- **Python：** `pytest quality/test_functional.py -v`
- **Scala：** `sbt "testOnly *FunctionalSpec"`
- **Java：** `mvn test -Dtest=FunctionalTest` 或 `gradle test --tests FunctionalTest`
- **TypeScript：** `npx jest functional.test.ts --verbose`
- **Go：** `go test -v` 並指定產出的測試檔案所屬套件 — 請遵循專案現有的模組與套件配置
- **Rust：** `cargo test` 並指定產出的測試 — 可以是 `tests/` 中的整合測試目標，或者是內嵌的 `#[cfg(test)]` 測試，須符合專案慣例

**檢查「失敗 (Failures)」與「錯誤 (Errors)」。** 大多數測試框架會區分測試失敗（斷言錯誤）與測試錯誤（設定失敗、缺失 fixture、匯入/解析錯誤、初始化期間的例外）。兩者皆屬於損壞的測試。一個常見的錯誤：產出的測試引用了不存在的共享 fixture 或輔助工具。這些會顯示為設定錯誤，而非斷言失敗 — 但它們同樣是損壞的。

執行後，檢查：
- 所有測試皆已通過 — 通過數量必須等於總測試數量
- 零失敗
- 零錯誤/設定失敗

如果存在設定錯誤，表示您忘記建立 fixture/設定檔案，或者您引用了不存在的輔助工具。請回去建立它們或將測試重寫為自包含 (self-contained) 的形式。

### 9. 現有測試未損壞

執行專案完整的測試套件（不只是您的新測試）。您的新檔案不應破壞任何既有功能。

## 文件驗證

### 10. QUALITY.md 案例引用了實際程式碼並標註了來源

每個案例都應提到程式碼庫中實際存在的函式名稱、檔案名稱或模式。請執行 grep 搜尋每個引用以確認其存在。

如果從非正式需求開始工作，請驗證每個案例與測試皆包含符合規範的需求標記：`[Req: 正式 — README §3]`、`[Req: 推論 — 源自 validate_input() 行為]`、`[Req: 使用者確認 — 「必須處理空輸入」]`。推論出的需求應在第四階段標記供使用者檢閱。

### 11. RUN_CODE_REVIEW.md 是自包含的

沒有任何背景知識的 AI 應該能讀取它並執行有用的檢閱。檢查：它是否列出了引導檔案？它是否有特定的焦點區域？安全防護措施是否存在？

### 12. RUN_INTEGRATION_TESTS.md 是可執行且欄位準確的

每個指令都應該能運作。每個檢查都應有具體的通過/失敗標準 — 不是「驗證看起來正確」，而是具體的預期結果。

**驗證整合測試品質閘門是根據「欄位參考表」撰寫的，而非憑記憶。** 檢查下列各項：

1. RUN_INTEGRATION_TESTS.md 中是否存在一個欄位參考表，且針對每個結構描述中的每個欄位都有一列
2. **欄位計數檢查：** 針對每個結構描述，計算實際結構描述檔案中的欄位數量，並計算您表格中的列數。如果數字不符，表示您遺漏了欄位或虛構了欄位。最常見的失敗情況：一個結構描述有 8 個欄位，但表格僅包含 2-3 個「重要」欄位。
3. **字元逐一檢查：** 現在重新閱讀每個結構描述檔案，並將您表格中的每個欄位名稱與檔案內容進行比較。`document_id` ≠ `doc_id`。`sentiment_score` ≠ `sentiment`。`classification` ≠ `category`。
4. 每個型別與限制皆與結構描述相符（`float 0-1` 不是 `int 0-100`，`string enum` 不是 `integer`）

如果任何欄位名稱、計數或型別錯誤，請先修正後再繼續。表格是基礎 — 如果表格錯誤，根據它建立的每個品質閘門都會是錯誤的。

### 13. RUN_SPEC_AUDIT.md 提示詞是可直接複製貼上的

權威性的稽核提示詞應能直接貼入 Claude Code、Cursor 或 Copilot 中執行而無需修改（檔案引用語法除外）。

## 快速核取清單格式

使用此清單進行最後確認：

- [ ] 測試計數接近啟發式目標 (規格章節 + 案例 + 防禦性模式)
- [ ] 案例測試計數與 QUALITY.md 中的案例計數相符
- [ ] 跨變體測試約佔總數 30% (涵蓋每個橫切屬性)
- [ ] 邊界測試數量 ≈ 防禦性模式數量
- [ ] 大多數斷言檢查的是數值，而非僅檢查存在性
- [ ] 所有測試斷言的是結果，而非機制
- [ ] 所有變動皆使用符合結構描述的數值
- [ ] 所有新測試皆通過 (零失敗且零錯誤 — 檢查是否有 fixture 錯誤)
- [ ] 所有現有測試仍能通過
- [ ] QUALITY.md 案例參考了實際程式碼並包含 `[Req: 等級 — 來源]` 標記
- [ ] 如果使用推論需求：所有 `[Req: 推論 — ...]` 項目皆已標記供使用者檢閱
- [ ] 程式碼檢閱協定是自包含的
- [ ] 整合測試品質閘門是根據欄位參考表撰寫的 (而非憑記憶)
- [ ] 整合測試具有具體的通過標準
- [ ] 規格稽核提示詞可直接複製貼上，並使用 `[Req: 等級 — 來源]` 標記格式
