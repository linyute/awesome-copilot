# 撰寫功能測試 (Functional Tests)

這是最重要的產出物。Markdown 檔案是文件，而功能測試檔案則是自動化安全網。請使用專案的慣例命名：`test_functional.py` (Python/pytest)、`FunctionalSpec.scala` (Scala/ScalaTest)、`FunctionalTest.java` (Java/JUnit)、`functional.test.ts` (TypeScript/Jest)、`functional_test.go` (Go) 等。

## 結構：三個測試群組

使用測試框架提供的任何結構 — 類別 (Python/Java)、describe 區塊 (TypeScript/Jest)、traits (Scala) 或子測試 (Go) — 將測試組織成三個邏輯群組：

```
規格需求 (Spec Requirements)
    — 每個可測試的規格章節一個測試
    — 每個測試的文件中應引用其驗證的規格需求

適用性情境 (Fitness Scenarios)
    — 每個 QUALITY.md 情境一個測試 (1:1 對應)
    — 命名應相符：test_scenario_N_易記名稱 (或等效慣例)

邊界與邊緣情況 (Boundaries and Edge Cases)
    — 針對步驟 5 中的每個防禦性模式進行一個測試
    — 目標是 Null 防護、try/catch、正規化、後備方案
```

## 測試計數啟發式方法

**目標 = (可測試規格章節數) + (QUALITY.md 情境數) + (步驟 5 中的防禦性模式數)**

範例：12 個規格章節 + 10 個情境 + 15 個防禦性模式 = 以 37 個測試為目標。

對於中型專案（5-15 個來源檔案），這通常會產生 35-50 個功能測試。如果數量顯著減少，則可能表示遺漏了需求或探索不夠深入。請勿為了湊數而湊數 — 每個測試都應練習真實的專案程式碼並驗證有意義的屬性。

## 匯入模式：與現有測試相符

在撰寫任何測試程式碼之前，請閱讀 2-3 個現有的測試檔案，並識別它們如何匯入專案模組。這至關重要 — 專案處理匯入的方式各不相同，弄錯會導致所有測試因解析錯誤而失敗。

識別專案中使用的匯入慣例。無論現有測試使用什麼模式，請完全照抄。不要猜測或發明不同的模式。

各語言的常見模式：

- **Python**：`sys.path.insert(0, "src/")` 然後進行裸匯入；套件匯入 (`from myproject.module import func`)；使用 conftest.py 進行路徑操作的相對匯入。
- **Go**：同套件測試 (`package mypackage`) 可以存取未匯出的識別碼；黑箱測試 (`package mypackage_test`) 僅測試匯出的 API；內部套件可能需要特定的匯入路徑。
- **Java**：`import com.example.project.Module;` 需與套件結構相符；測試原始碼根目錄必須鏡像主原始碼根目錄。
- **TypeScript**：搭配相對路徑使用 `import { func } from '../src/module'`；來自 `tsconfig.json` 的路徑別名 (例如 `@/module`)。
- **Rust**：針對同一個 Crate 內的單元測試使用 `use crate::module::function;`；針對 `tests/` 中的整合測試使用 `use myproject::module::function;`。
- **Scala**：`import com.example.project._` 或 `import com.example.project.{ClassA, ClassB}`；SBT 佈局會在 `src/test/scala/` 中鏡像 `src/main/scala/`。

## 在撰寫測試前建立測試設定

每個測試框架都有共享設定的機制。如果您的測試使用共享 fixture 或測試資料，您「必須」在撰寫測試前建立設定檔案。測試框架不會自動探索其他目錄中的 fixture。

識別您的框架設定機制 (fixtures, `@BeforeEach`, `beforeAll`, 協助程式函式, 建構器模式等)，並遵循專案現有測試中已使用的慣例。

**規則：每個被引用的 fixture 或測試協助程式都必須被定義。** 如果測試依賴於不存在的共享設定，測試將在設定期間出錯（而不是在斷言期間失敗）— 這會產生看似通過但實際已損壞的測試。

**所有語言的偏好方法：** 撰寫在行內建立自身資料的測試。這消除了跨檔案相依性。使用框架的暫存目錄支援和常值資料結構，直接在每個測試函式中建立測試資料。

**在撰寫完所有測試後，執行測試套件並檢查設定錯誤。** 無論框架如何對其進行分類，設定錯誤（找不到 fixture、匯入失敗）都被視為已損壞的測試。

## 不使用預留位置測試 (No Placeholder Tests)

每個測試都必須匯入並呼叫實際的專案程式碼。如果測試主體是 `pass`，或其唯一斷言是 `assert isinstance(errors, list)`，或者它檢查的是微不足道的屬性（如 `assert hasattr(cls, 'validate')`），請將其刪除並撰寫真實測試或完全捨棄。不練習專案程式碼的測試比沒有測試更糟 — 它會虛報計數並造成虛假的信心。

如果您確實無法為防禦性模式撰寫有意義的測試（例如，它需要執行中的伺服器或外部服務），請在註解中註明為不可測試，而不是撰寫預留位置測試。

## 撰寫前先閱讀：函式呼叫地圖

在撰寫單一測試之前，先建構一個函式呼叫地圖。針對您計劃測試的每個函式：

1. **閱讀函式/方法簽章** — 不只是名稱，還有每個參數、其型別和預設值。
2. **閱讀文件** — docstrings, Javadoc, TSDoc, ScalaDoc。它們通常會說明傳回型別、異常和邊緣情況行為。
3. **閱讀一個呼叫該函式的現有測試** — 現有測試會向您展示確切的呼叫慣例、fixture 形狀和斷言模式。
4. **閱讀真實的資料檔案** — 如果函式處理設定、結構描述或資料檔案，請閱讀專案中的實際檔案。您的測試 fixture 必須與此形狀完全匹配。

**常見失敗模式**：代理程式探索了架構，從概念上理解了函式的功能，然後撰寫了帶有猜測參數的測試呼叫。測試失敗是因為真實的函式採用的是 `(config, items_data, limit)` 而不是 `(items, seed, strategy)`。閱讀實際的簽章只需 5 秒鐘，且能完全防止這種情況。

**程式庫版本感知**：檢查專案的相依性清單 (`requirements.txt`, `build.sbt`, `package.json`, `pom.xml`/`build.gradle`, `go.mod`, `Cargo.toml`) 以驗證可用內容。針對選用相依性使用測試框架的跳過機制 (例如 `pytest.importorskip()`, `Assumptions.assumeTrue()`, `t.Skip()`, `#[ignore]` 等)。

## 撰寫規格衍生測試

逐章節閱讀每份規格文件。針對每個章節，問自己：「這說明了什麼可測試的需求？」然後撰寫測試。

每個測試都應該：
1. **設定** — 載入 fixture、建立測試資料、設定系統。
2. **執行** — 呼叫函式、執行管線、發出要求。
3. **斷言** — 規格要求的特定屬性。

每個測試都應包含追蹤標註（透過 docstring、顯示名稱或註解），引用其驗證的規格章節，例如 `[Req: formal — Design Doc §N] X should produce Y`。

## 什麼是良好的功能測試

- **具備可追溯性** — 測試名稱、顯示名稱或文件註解說明了其驗證的規格需求。
- **具體性** — 檢查特定的屬性，而不僅僅是「發生了某些事」。
- **強健性** — 使用真實資料（來自實際系統的 fixture），而非合成資料。
- **跨變體** — 如果專案處理多種輸入類型，請測試所有類型。
- **在正確的層級進行測試** — 測試您關心的「行為」。如果需求是「無效資料不會產生錯誤輸出」，請測試管線輸出 — 不要只測試結構描述驗證器拒絕了輸入。

## 跨變體測試策略

如果專案處理多種輸入類型，跨變體涵蓋範圍即是隱藏漏洞之處。目標是讓大約 30% 的測試在所有變體中執行 — 確切百分比不如確保每個跨領域屬性都在所有變體中得到測試來得重要。

使用您的框架參數化機制 (例如 `@pytest.mark.parametrize`, `@ParameterizedTest`, `test.each`, 表格驅動測試, 對個案進行反覆運算) 來在所有變體中執行相同的斷言邏輯。

如果參數化不適用，請在單個測試中明確執行迴圈。

**哪些測試應該是跨變體的？** 任何驗證無論輸入類型為何都「應」成立之屬性的測試：實體身分、結構化屬性、必填連結、時間欄位、領域特定語意。

**在撰寫完所有測試後，執行跨變體稽核。** 計算跨變體測試數除以總測試數。如果低於 30%，請轉換更多測試。

## 應避免的非預期模式 (Anti-Patterns)

這些模式看起來像測試，但無法擷取真實漏洞：

- **僅存在性檢查** — 找到一個正確結果並不代表所有結果都是正確的。還要檢查數量或進行全面驗證。
- **僅存在性斷言** — 斷言某個數值存在僅能證明其存在，不能證明其正確性。斷言其實際數值。
- **單變體測試** — 僅測試一種輸入類型並寄望其他類型也能運作。請使用參數化。
- **僅正面測試** — 您必須測試無效輸入「不會」產生不良輸出。
- **不完整的負面斷言** — 在測試拒絕情況時，斷言「所有」後果皆不存在，而不僅僅是一個。
- **擷取異常而非檢查輸出** — 測試程式碼以特定方式崩潰，並不是在測試它是否正確處理了輸入。請測試輸出。

### 詳解「擷取異常」非預期模式

```python
# 錯誤：測試驗證機制
def test_bad_value_rejected(fixture):
    fixture.field = "invalid"  # 結構描述會拒絕此值！
    with pytest.raises(ValidationError):
        process(fixture)
    # 此測試完全未說明輸出情況

# 正確：測試需求
def test_bad_value_not_in_output(fixture):
    fixture.field = None  # 結構描述接受選填欄位為 None
    output = process(fixture)
    assert field_property not in output  # 輸出中不應出現錯誤資料
    assert expected_type in output  # 其餘部分仍運作正常
```

每種語言的模式都相同：不要測試驗證機制是否拒絕了不良輸入 — 要測試當系統接收到結構描述接受的邊緣情況輸入時，是否產生了正確的輸出。錯誤的方法測試實作 (驗證器)；正確的方法測試需求 (輸出)。

在選擇變更值之前，務必檢查您的步驟 5b 結構描述對應表。

## 在正確的層級測試

問自己：「規格說明『應該』發生什麼？」規格說明「無效資料不應出現在輸出中」 — 而不是「驗證層應該拒絕它」。請測試規格，而不是實作。

**例外情況**：當規格明確強制要求特定機制時（例如，「必須在結構描述層級快速失敗」），測試該機制是適當的。但這種情況很少見。

## 適用性情境測試 (Fitness-to-Purpose Scenario Tests)

針對 QUALITY.md 中的每個情境撰寫一個測試。這是一對一的對應。每個測試應包含引用該情境的追蹤標註，例如 `[Req: formal — QUALITY.md Scenario 1]`，且其命名應與該情境的易記名稱相符。

## 邊界與負面測試

針對步驟 5 中的每個防禦性模式進行一個測試。每個測試應包含引用該防禦性模式的追蹤標註，例如 `[Req: inferred — from function_name() guard] guards against X`。

對於每個邊界測試：
1. 變更輸入以觸發防禦性程式碼路徑（使用結構描述接受的數值）。
2. 處理變更後的輸入。
3. 斷言優雅地處理 — 儘管輸入為邊緣情況，結果仍為有效。

在選擇變更值時使用您的步驟 5b 結構描述對應表。每次變更都必須使用結構描述接受的數值。

系統化方法：
- **缺失欄位** — 選填欄位缺失？設定為 Null。
- **錯誤型別** — 欄位獲得不同型別？使用結構描述接受的替代方案。
- **空值** — 空清單？空字串？空字典？
- **邊界值** — 零、負數、最大值、第一個、最後一個。
- **跨模組邊界** — 模組 A 產生不尋常但有效的輸出 — 模組 B 能處理嗎？

如果您發現了超過 10 個防禦性模式但僅撰寫了 4 個邊界測試，請返回並撰寫更多。目標是 1:1 的比例。
