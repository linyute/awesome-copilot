# Council of Three 規格稽核協定 (檔案 5)

這是一個靜態分析協定 — AI 模型閱讀原始碼並將其與規格進行比較。此過程不執行任何程式碼。它能擷取與測試不同類別的問題：規格與程式碼分歧、未記錄的功能、虛幻規格以及缺失的實作。

## 為什麼需要三位模型？

不同的 AI 模型具有不同的盲點 — 它們對不同的事項有信心，也會遺漏不同的事項。交叉參考三份獨立的審查，可以擷取任何單一模型都會遺漏的缺陷。

## 範本

```markdown
# 規格稽核協定：[專案名稱]

## 決定性的稽核提示 (The Definitive Audit Prompt)

將此提示完全相同地提供給三個獨立的 AI 工具（例如 Claude、GPT、Gemini）。

---

**待閱讀的上下文檔案：**
1. [列出所有規格/意圖文件及其路徑]
2. [架構文件]
3. [設計決策記錄]

**任務：** 擔任測試員。閱讀 [原始碼目錄] 中的實際程式碼，並將其與上方列出的規格進行比較。

**需求信賴度分層：**
需求標有 `[Req: tier — source]`。根據層級權衡您的發現：
- **formal (正式)** — 由真人在規格文件中撰寫。具備權威性。分歧是真實的發現。
- **user-confirmed (使用者確認)** — 由使用者陳述但不在正式文件中。除非與其他證據相矛盾，否則視為具備權威性。
- **inferred (推論)** — 從程式碼行為推導而來。信賴度較低。將分歧回報為 NEEDS REVIEW (需要審查)，而非確定的缺陷。

**規則：**
- 僅列出缺陷。不要摘要相符的內容。
- 針對「每項」缺陷，引用具體的檔案與行號。
  如果您無法引用行號，請勿包含該項發現。
- 在聲稱缺失前，先對程式碼庫執行 grep 搜尋。
- 在聲稱存在前，先閱讀實際的函式主體。
- 分類每項發現：MISSING (缺失) / DIVERGENT (分歧) / UNDOCUMENTED (未記錄) / PHANTOM (虛幻)
- 對於針對推論需求的發現，請註明：NEEDS REVIEW

**缺陷分類：**
- **MISSING** — 規格有要求，程式碼未實作。
- **DIVERGENT** — 規格與程式碼皆有處理，但兩者不一致。
- **UNDOCUMENTED** — 程式碼有執行，但規格未提及。
- **PHANTOM** — 規格有描述，但實際實作方式與描述不同。

**專案專屬的審查區域：**

[5–10 個強制稽核員閱讀最關鍵程式碼的具體問題。目標如下：]

1. [最脆弱的模組 — 強制稽核員閱讀特定的函式]
2. [外部資料處理 — 驗證、正規化、錯誤恢復]
3. [可能不成立的假設 — 欄位存在性、數值範圍、格式一致性]
4. [跨模組邊界的功能]
5. [文件與實作之間的落差]
6. [來自 QUALITY.md 情境的特定邊緣情況]

**輸出格式：**

### [filename.ext]
- **第 NNN 列：** [MISSING / DIVERGENT / UNDOCUMENTED / PHANTOM] [Req: tier — source] 描述。
  規格說明：[引用或參考]。程式碼執行：[實際發生的情況]。
  *(包含 `[Req: tier — source]` 標籤，以便發現能追溯回其需求與信賴層級。)*

---

## 稽核前文件驗證（必填的分類章節）

無論 `reference_docs/` 是否存在，分類報告都必須包含 `## Pre-audit docs validation` 章節。此章節記錄了稽核員用作其事實基準的內容。

**如果 `reference_docs/` 存在**：在執行稽核前，抽查收集到的文件的事實準確性。過時或錯誤的文件會扭曲稽核信賴度 — 模型如果在文件中讀到「函式庫透過執行 Y 來處理 X」，即使文件有誤，它也會對分歧發現給予較高評分。

**快速驗證程序（最多 5 分鐘）：**
1. 從 `reference_docs/` 中挑選 2–3 個描述具體執行階段行為的事實主張（例如，「無效輸入引發 ValueError」、「欄位 X 預設為 Y」、「不支援格式 Z」）。
2. 對原始碼執行 grep 搜尋引用的行為。程式碼是否與文件匹配？
3. 如果任何主張有誤，請在分類標頭中註明：「reference_docs/ 包含 N 個已知的不準確之處：[列表]。依賴這些主張的發現降級為 NEEDS REVIEW。」

**關於程式碼內容的抽查主張必須擷取，而非僅僅斷言。** 當規格稽核提示或預先驗證包含諸如「函式 X 在第 Z 列處理常數 Y」之類的主張時，分類步驟必須閱讀引用的行並回報它們實際包含的內容。不要透過檢查函式是否存在或常數是否定義在某處來確認主張 — 請透過展示引用行的確切文字來確認。將每個抽查結果格式化為：

```
主張：「vring_transport_features() 在第 3527 列保留了 VIRTIO_F_RING_RESET」
實際第 3527 列：`default:`
結果：主張為偽 — 第 3527 列是 default 分支，不是 RING_RESET 的 case 標籤
```

從產出的需求或收集的文件（而非程式碼本身）衍生的抽查主張是 **待測試的假設**，而非要確認的事實。此規則可防止 v1.3.17 中觀察到的污染鏈，當時一個錯誤的抽查主張（「3527-3528 列的 RING_RESET」）在未閱讀實際行的情況下被接受為「準確」，進而導致三位稽核員皆繼承了虛幻的程式碼存在性主張。

**如果 `reference_docs/` 不存在**：明確說明此點：「未提供補充文件。稽核員僅依賴存放庫內規格與程式碼。」這確認了缺失是故意的，而非疏忽。

此章節在每次分類中都會觸發，不論文件是否存在。在 v1.3.5 跨存放庫測試中，它僅在 1/8 的存放庫中觸發，因為它是條件式的 — 將其設為必填可確保稽核線索始終記錄了事實基準。

## 執行稽核

1. 將相同的提示提供給三個 AI 工具。
2. 每個稽核員獨立作業 — 互不污染。
3. 收集所有三份報告。

## 分類程序 (Triage Process)

在所有三個模型回報後，合併發現。

**記錄有效 Council 人數。** 如果某個模型未傳回可用報告（逾時、輸出為空、拒絕執行），請在分類標頭中記錄：

```
## Council 狀態
- 模型 A：已接收新鮮報告 (YYYY-MM-DD)
- 模型 B：已接收新鮮報告 (YYYY-MM-DD)
- 模型 C：TIMEOUT — 無可用報告。有效 Council：2/3。
```

當有效 Council 為 2/3 時，降級信賴度分層：「全部三個」變為不可能，「三個中的兩個」變為上限。當有效 Council 為 1/3 時，所有發現皆為「需要驗證」，不論該單一模型的信心有多強。不要默默用先前執行的陳舊報告替代 — 如果模型未針對此次執行產出新鮮報告，則視為未參與。

| 信賴度 | 發現者 | 行動 |
|------------|----------|--------|
| 最高 | 全部三個 | 幾乎確定為真 — 修正或更新規格 |
| 高 | 三個中的兩個 | 可能為真 — 驗證並修正 |
| 需要驗證 | 僅一個 | 可能是真也可能是幻覺 — 部署驗證探針 |

**當有效 Council 為 2/3 或更少時：** 在分類中明確區分單一稽核員發現與多稽核員發現。在 2/3 Council 的情況下，兩個現有稽核員皆有的發現具備「高」信賴度。僅由一個現有稽核員發現的項目則為「需要驗證」 — 它不能在沒有驗證探針的情況下提升為已確認 BUG，因為缺失的稽核員可能會反對它。不要僅因為 Council 不完整就將所有發現視為等效。

在分類摘要表中，新增一欄稽核員共識：「2/2 在場」、「1/2 在場」等。這使信賴層級可見且可稽核。

**針對列舉/分派檢查的不完整 Council 閘道。** 如果有效 Council 少於 3/3，且執行包含白名單/列舉/分派函式檢查（關於函式處理哪些常數的主張），則在沒有執行機械式證明的情況下，稽核不得得出「無確認缺陷」的結論。檢查針對每個相關函式是否存在 `quality/mechanical/<函式>_cases.txt` 成品。如果存在且顯示常數存在，則主張獲得確認。如果顯示常數缺失，則無論任何稽核員寫了什麼，該主張皆為偽。如果不存在機械式成品，請在關閉列舉檢查前產生一個。此規則的存在是因為 v1.3.18 的有效 Council 為 1/3，且單一模型的分類虛構了列舉主張的行內容 — 機械式成品本可以發現此矛盾。

### 驗證探針 (The Verification Probe)

當模型對事實主張產生分歧時，請部署一個唯讀探針：給予一個模型有爭議的主張，並要求其閱讀程式碼並報告地面實況 (ground truth)。絕不要透過多數決來解決事實爭議 — 多數人對於程式碼實際執行的操作也可能出錯。

**驗證探針必須產生可執行的證據。** 無論是確認還是拒絕，文字推理都是不夠的。每個驗證探針都必須產生一個測試斷言，以機械式證明判斷結果：

**對於拒絕 (Rejections)**（發現是假陽性）：撰寫一個「通過」的斷言，證明稽核員的主張是錯誤的：
```python
# 拒絕證明：函式 X 確實在第 247 列檢查了 null
assert "if (ptr == NULL)" in source_of("X"), "X has null check at line 247"
```
如果您無法撰寫通過的斷言，**請不要拒絕該發現**。無法產生機械式證明本身就是發現可能為真的證據。

**對於確認 (Confirmations)**（發現是真實漏洞）：撰寫一個「失敗」(預期失敗) 的斷言，證明漏洞存在：
```python
# 確認證明：RING_RESET 不是白名單中的 case 標籤
assert "case VIRTIO_F_RING_RESET:" in source_of("vring_transport_features"), \
    "RING_RESET should be in the switch but is not — cleared by default at line 3527"
```

**每個斷言必須引用一個確切的行號**，用於說明其引用的證據。不是「第 3527-3528 列」，而是「第 3527 列：`default:`」 — 顯示該列實際包含的內容。

**為什麼這項規則很重要：** 在 v1.3.16 virtio 測試中，分類步驟收到了一個正確的少數派發現，即 VIRTIO_F_RING_RESET 在 switch/case 白名單中缺失。分類步驟執行了一個驗證探針，聲稱第 3527-3528 列「明確保留了 RING_RESET」 — 但那些列實際上包含的是 `default:` 分支。該探針虛構了合規性。如果它被要求撰寫 `assert "case VIRTIO_F_RING_RESET:" in source`，斷言本會失敗，從而揭露幻覺。要求提供可執行證據，使虛構的拒絕行為自取其滅。

### 分類每項確認的發現

- **規格漏洞** — 規格有誤，程式碼正常 → 更新規格
- **設計決策** — 需要人為判斷 → 進行討論與決定
- **真實程式碼漏洞** — 按子系統分小批次進行修復
- **文件缺口** — 功能存在但未記錄 → 更新文件
- **缺失測試** — 程式碼正確但無測試驗證 → 新增至功能測試檔案
- **推論需求錯誤** — 推論的需求不符合實際意圖 → 在 QUALITY.md 中移除或修正

最後一類是規格稽核與測試套件之間的橋樑。每項尚未被測試涵蓋的確認發現都應轉化為一項測試。

### 遺留與歷史指令碼

被記錄為「歷史性的」、「已棄用」或「不屬於當前工作流程」的指令碼，有時會在分類期間降級，理由是它們不影響當前運作。當指令碼確實從不執行時，這是正確的。但如果指令碼的漏洞已經體現在規範成品中 — 發佈檔案中的重複條目、受控快取中的過期資料、下游工具使用的錯誤對應 — 則該漏洞不是歷史性的。它是存放庫發佈狀態中的現存缺陷。

**規則：如果遺留指令碼的漏洞在規範成品中已可見，則不論指令碼狀態如何，皆將其提升為已確認 BUG。** 指令碼可能是歷史性的，但它留下的損害是現實的。迴歸測試應針對成品（重複條目、過期對應），而非指令碼 — 因為成品才是使用者遇到的東西。

此規則的存在是因為 QPB 的 v1.3.5 自舉執行發現了由「歷史性」指令碼產生的重複變更日誌條目與過期的快取對應。兩份分類報告都降級了這些發現，因為指令碼是歷史性的。但重複條目已存在於發佈的程式庫中，每位使用者都能看到。

### 跨成品一致性檢查

分類後，將規格稽核發現與來自 `quality/code_reviews/` 的程式碼審查發現進行比較。如果程式碼審查與規格稽核對同一個事實主張產生分歧（一方說是真實漏洞，另一方稱其為假陽性），請標記該分歧並部署驗證探針。程式碼審查與規格稽核使用不同的方法（結構化閱讀 vs. 規格比較），因此分歧是有資訊價值的，而非錯誤。但關於程式碼實際執行操作的事實矛盾在任一報告被信任前都需要解決。

## 偵測部分對話與延續的成品

### 部分對話偵測

提早終止的對話（逾時、內容耗盡、崩潰）可能會產生架構（目錄結構、空範本），而不產出實際的審查或稽核內容。執行指令碼中的重試機制可以重新產生架構，但無法恢復分析工作。

**在任何對話完成後，檢查是否有部分結果：**
1. 如果 `quality/code_reviews/` 存在但未包含具有實際發現的 `.md` 檔案（或僅包含無 BUG/VIOLATED/INCONSISTENT 條目的範本標頭），則說明程式碼審查未執行。在 PROGRESS.md 中將其標記為 FAILED (失敗)，而非「完成但無發現」。
2. 如果 `quality/spec_audits/` 存在但未包含分類摘要，則說明規格稽核未執行。
3. 如果 `quality/test_regression.*` 存在但僅包含匯入而無測試函式，則說明未撰寫迴歸測試。

部分對話不是「無發現的乾淨執行」 — 它是失敗的執行，需要重新執行。PROGRESS.md 應清楚記錄此點：「階段 6：FAILED — 程式碼審查對話在產出發現前終止。需重新執行。」

### 延續成品的出處標頭 (Provenance Headers)

當新的播放手冊執行在（存檔後）發現來自先前執行的現有成品時，或者當成品從失敗的對話中存活下來時，它們必須帶有出處標頭，以便讀者瞭解其來源。

**如果任何成品「不是」在當前執行中新鮮產生的**，請新增一個出處標頭：

```markdown
<!-- PROVENANCE: 此檔案是從先前的執行 ([日期]) 延續而來的。
     它並未由當前的 v1.3.5 執行重新產生。
     請將發現視為可能已過時 — 在採取行動前，請根據當前原始碼進行驗證。 -->
```

這可防止 v1.3.4 中觀察到的失敗模式，當時 express 與 zod 默默保留了 v1.3.3 的程式碼審查與規格稽核，而未標記為存檔。閱讀這些成品的使用者會誤以為它們是新鮮的 v1.3.4 結果。

## 修復執行規則

- 按子系統對修復進行分組，而非按缺陷編號。
- 絕不要使用單個超大提示來處理所有修復。
- 每一批次：實作、測試，並讓三位審查人員皆驗證 diff。
- 至少兩位稽核員必須確認修復通過，才能標記為完成。

## 輸出

將稽核報告儲存到 `quality/spec_audits/YYYY-MM-DD-[模型].md`
將分類摘要儲存到 `quality/spec_audits/YYYY-MM-DD-triage.md`
```

## 四個護欄（對所有稽核員至關重要）

某些模型會在未檢查程式碼的情況下自信地聲稱特徵缺失。這四條嵌入稽核提示中的規則透過減少模糊與幻覺發現，實質性地改進了輸出品質：

1. **強制行號** — 如果您無法引用行號，請勿包含該項發現。這消除了模糊的主張。
2. **聲稱缺失前先執行 grep** — 在說特徵不存在前，先搜尋程式碼庫。它可能在另一個檔案中。
3. **閱讀函式主體，而非僅看簽章** — 不要僅憑函式名稱就假設其運作正常。
4. **分類缺陷類型** — 強制進行結構化思考 (MISSING/DIVERGENT/UNDOCUMENTED/PHANTOM)，而非模糊的「這看起來不對」。

這些護欄已嵌入上方的範本中。它們對於傾向於提出自信但未經檢查之主張的模型最為重要。

## 模型選取備註

不同的模型具有不同的稽核優勢。實務上：

- **專注於架構的模型** (如 Claude) 傾向於以最少的假陽性發現最多的問題，在靜默資料遺失、跨函式資料流以及狀態機漏洞方面表現出色。
- **專注於邊緣情況的模型** (如基於 GPT 的工具) 傾向於擷取其他模型遺漏的邊界條件（零長度輸入、檔案衝突、差一錯誤），並可作為有效的驗證交叉檢查器。
- **需要結構的模型** (如某些 Gemini 變體) 在開放式稽核提示下可能表現不佳，但對上方的四個護欄反應劇烈。

表現優異的具體模型會隨時間而變。原則不變：使用多個具有不同優勢的模型，且務必包含四個護欄。

### 最低模型能力要求

稽核協定要求閱讀函式主體、引用行號、聲稱缺失前執行 grep 以及分類缺陷類型。輕量級或針對速度優化的模型（Haiku 等級、GPT-4o-mini 等級）不適合作為稽核員。它們傾向於略讀而非閱讀，會跳過 grep 步驟，並在強大模型能發現真實漏洞的程式碼庫上產出淺顯或空洞的報告 (「未發現缺陷」)。請在所有三個稽核員位置都使用具備強大程式碼閱讀能力的模型。平庸的稽核員不僅會遺漏發現 — 它還會使 Council 從三個獨立視角縮減為兩個。

## 撰寫審查區域的提示

審查區域是提示中最重要的部分。通用的問題如「檢查程式碼是否符合規格」會產出通用的答案。具體的問題如果命名了函式、檔案與邊緣情況，則會產出具體的發現。

良好的審查區域：
- 「閱讀 `pipeline.py` 第 45–120 列中的 `process_input()`。規格說明它應透過替換預設值來處理缺失欄位。它有做到嗎？哪些欄位有預設值，哪些會默默產生 null？」
- 「架構文件說明模組 A 將驗證後的資料傳遞給模組 B。閱讀這兩個模組。是否存在未驗證資料到達模組 B 的路徑？」

不良的審查區域：
- 「檢查程式碼是否正確」
- 「尋找漏洞」
- 「驗證實作是否符合規格」
