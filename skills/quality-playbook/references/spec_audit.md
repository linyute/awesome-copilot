# 三人議會 (Council of Three) 規格稽核協定 (檔案 5)

這是一個靜態分析協定 — AI 模型閱讀程式碼並將其與規格說明進行比較。不執行任何程式碼。它能捕捉與測試不同類別的問題：規格與程式碼的分歧、未記載的功能、虛假規格以及缺失的實作。

## 為什麼需要三個模型？

不同的 AI 模型有不同的盲點 — 它們對不同事物有信心，也會遺漏不同的事物。交叉比對三個獨立的檢閱結果，能捕捉到任何單一模型都會遺漏的缺陷。

## 範本

```markdown
# 規格稽核協定：[專案名稱]

## 權威性的稽核提示詞

將此提示詞原封不動地提供給三個獨立的 AI 工具（例如：Claude、GPT、Gemini）。

---

**待閱讀的脈絡檔案：**
1. [列出所有具有路徑的規格/意圖文件]
2. [架構文件]
3. [設計決策記錄]

**任務：** 擔任測試員。閱讀 [原始碼目錄] 中的實際程式碼，並將其與上述規格說明進行比較。

**需求信心等級：**
需求標有 `[Req: 等級 — 來源]` 標記。請根據等級加權您的發現：
- **正式 (formal)** — 由人類撰寫在規格文件中。具備權威性。分歧即為真實的發現。
- **使用者確認 (user-confirmed)** — 由使用者陳述但不在正式文件中。除非與其他證據相左，否則視為具備權威性。
- **推論 (inferred)** — 從程式碼行為中演繹而來。信心較低。將分歧報告為「需檢閱 (NEEDS REVIEW)」，而非肯定的缺陷。

**規則：**
- 「僅」列出缺陷。不要總結符合的部分。
- 對於「每一個」缺陷，請引用特定的檔案與行號。
  如果您無法引用行號，請勿包含該項發現。
- 在聲稱缺失之前，請先執行 grep 搜尋程式碼庫。
- 在聲稱存在之前，請先閱讀實際的函式本體。
- 將每項發現分類為：MISSING / DIVERGENT / UNDOCUMENTED / PHANTOM
- 針對針對推論需求的發現，請加上：NEEDS REVIEW

**缺陷分類：**
- **MISSING (缺失)** — 規格有要求，但程式碼未實作
- **DIVERGENT (分歧)** — 規格與程式碼皆有提及，但兩者不一致
- **UNDOCUMENTED (未記載)** — 程式碼有執行，但規格未提及
- **PHANTOM (虛假)** — 規格有描述，但實際實作方式與描述不同

**專案特定的詳查區域：**

[提供 5-10 個強迫稽核員閱讀最關鍵程式碼的特定問題。目標：]

1. [最脆弱的模組 — 強迫稽核員閱讀特定函式]
2. [外部資料處理 — 驗證、標準化、錯誤恢復]
3. [可能不成立的假設 — 欄位存在性、數值範圍、格式一致性]
4. [跨越模組邊界的功能]
5. [文件與實作之間的鴻溝]
6. [來自 QUALITY.md 案例的特定邊緣案例]

**輸出格式：**

### [filename.ext]
- **第 NNN 行：** [MISSING / DIVERGENT / UNDOCUMENTED / PHANTOM] [Req: 等級 — 來源] 描述。
  規格說明：[引用或參考]。程式碼執行：[實際發生的情況]。
  *(請包含 `[Req: 等級 — 來源]` 標記，以便將發現追蹤回其需求與信心等級。)*

---

## 執行稽核

1. 將相同的提示詞提供給三個 AI 工具
2. 每個稽核員獨立工作 — 禁止交叉污染
3. 收集所有三份報告

## 篩選 (Triage) 流程

在所有三個模型完成報告後，合併發現結果：

| 信心等級 | 發現者 | 採取行動 |
|------------|----------|--------|
| 最高 | 全部三個 | 幾乎肯定屬實 — 修正程式碼或更新規格 |
| 高 | 三分之二 | 很可能屬實 — 驗證並修正 |
| 需驗證 | 僅一個 | 可能是真實的也可能是幻覺 — 部署驗證探針 |

### 驗證探針

當模型對事實性主張意見不一時，部署一個唯讀探針：將爭議點交給一個模型，並要求其閱讀程式碼並報告真實情況。永遠不要透過多數決來解決事實爭議 — 多數也可能對程式碼的實際行為產生錯誤認知。

### 為每個確認的發現分類

- **規格錯誤 (Spec bug)** — 規格有誤，程式碼沒問題 → 更新規格
- **設計決策 (Design decision)** — 需要人類判斷 → 進行討論並決定
- **真實程式碼錯誤 (Real code bug)** — 按子系統分小批次進行修正
- **文件缺口 (Documentation gap)** — 功能存在但未記載 → 更新文件
- **缺失測試 (Missing test)** — 程式碼正確但無測試驗證 → 新增至功能測試檔案
- **推論需求錯誤 (Inferred requirement wrong)** — 推論出的需求與實際意圖不符 → 從 QUALITY.md 中移除或修正

最後一類是規格稽核與測試套件之間的橋樑。每個尚未被測試涵蓋的確認發現都應轉化為測試。

## 修正執行規則

- 按子系統分組修正，而非按缺陷編號
- 絕不使用一個大型提示詞來處理所有修正
- 每一批次：實作、測試、並讓所有三個檢閱者驗證差異 (diff)
- 至少需要兩個稽核員確認修正通過後，才可標記為完成

## 產出

將稽核報告儲存至 `quality/spec_audits/YYYY-MM-DD-[模型名稱].md`
將篩選摘要儲存至 `quality/spec_audits/YYYY-MM-DD-triage.md`
```

## 四個安全防護 (對所有稽核員至關重要)

某些模型會在未檢查程式碼的情況下，信誓旦旦地宣稱功能缺失。嵌入在稽核提示詞中的這四條規則，能透過減少模糊與幻覺發現，實質地提升產出品質：

1. **強制行號** — 如果您無法引用行號，請勿包含該項發現。這能消除模糊的主張。
2. **聲稱缺失前先執行 grep** — 在聲稱功能缺失前，請搜尋程式碼庫。它可能在另一個檔案中。
3. **閱讀函式主體，而不僅是特徵標記** — 不要根據函式名稱假設其運作正確。
4. **分類缺陷類型** — 強制進行結構化思考 (MISSING/DIVERGENT/UNDOCUMENTED/PHANTOM)，而非模糊的「這看起來不對」。

這些安全防護已嵌入在上述範本中。對於那些傾向於給出未經檢查但自信的主張的模型，這些防護尤為重要。

## 模型選擇備註

不同的模型具有不同的稽核優勢。在實務上：

- **專注於架構的模型** (例如：Claude) 傾向於以最低的誤判率找出最多的問題，擅長發現隱性資料遺失、跨功能資料流以及狀態機錯誤。
- **專注於邊緣案例的模型** (例如：基於 GPT 的工具) 傾向於捕捉其他模型會遺漏的邊界條件 (零長度輸入、檔案衝突、差一錯誤 (off-by-one errors))，並可作為有效的驗證交叉檢查員。
- **需要結構的模型** (例如：某些 Gemini 變體) 在處理開放式稽核提示詞時表現可能不佳，但對上述四個安全防護的反應非常顯著。

表現優異的特定模型會隨時間改變。原則不變：使用具有不同優勢的多個模型，並始終包含四個安全防護。

## 撰寫詳查區域的提示

詳查區域是提示詞中最重要的部分。通用的問題（如「檢查程式碼是否符合規格」）會產生通用的答案。指名函式、檔案與邊緣案例的特定問題則會產出特定的發現。

良好的詳查區域：
- 「閱讀 `pipeline.py` 第 45-120 行的 `process_input()`。規格說明它應透過替換預設值來處理缺失欄位。它有做到嗎？哪些欄位有預設值，哪些會默默地產出 null？」
- 「架構文件說明模組 A 會將經過驗證的資料傳遞給模組 B。閱讀這兩個模組。是否有任何路徑會讓未經驗證的資料到達模組 B？」

不良的詳查區域：
- 「檢查程式碼是否正確」
- 「尋找錯誤」
- 「驗證實作是否符合規格」
