# 需求管線 (Requirements Pipeline)

## 概觀

此文件定義了品質播放手冊 (Quality Playbook) 步驟 7 的五階段需求產生管線。此管線將合約發現與需求推導分開，使用基於檔案的外部記憶體，以便模型不需要同時在對話中處理所有內容，並包含一個帶有完整性閘道的機械式驗證。

**為什麼需要管線？** 單次傳遞的需求產生會在產生大約 70 個需求後耗盡注意力，因為模型同時在發現合約並撰寫正式需求。將這些任務分成具有基於檔案移交的獨立階段，能產生顯著更完整的涵蓋範圍。在對 Gson（81 個原始檔案，約 2.1 萬列程式碼）的測試中，單次傳遞產生了 48 個需求；而此管線則產生了 110 個。

## 產出的檔案

| 檔案 | 用途 |
|------|---------|
| `quality/CONTRACTS.md` | 從原始碼擷取的原始行為合約 |
| `quality/REQUIREMENTS.md` | 帶有敘述的可測試需求（主要產出物） |
| `quality/COVERAGE_MATRIX.md` | 合約對應到需求的追蹤性 |
| `quality/COMPLETENESS_REPORT.md` | 最終的完整性評估與判定結果 |
| `quality/VERSION_HISTORY.md` | 包含版本表格和出處的審查日誌 |
| `quality/REFINEMENT_HINTS.md` | 審查進度與回饋（在審查期間建立） |

版本化備份位於 `quality/history/vX.Y/`。

## 階段 A：擷取行為合約

**輸入：** 專案中的所有原始檔案（或限定範圍的子系統 — 參見下方的規模檢查）。
**輸出：** `quality/CONTRACTS.md`

### 規模檢查

在開始擷取之前，計算專案中的原始檔案數量（排除測試、產出的程式碼、供應商相依性和組建產出物）。

- **標準專案 (≤ 300 個原始檔案)**：正常進行 — 從所有檔案中擷取合約。此範圍內的專案已通過端對端測試（例如，Gson 在約 81 個原始檔案下產生了 110 個具備完全涵蓋範圍的需求）。
- **大型專案 (301–500 個原始檔案)**：專注於階段 1 步驟 2 中識別的 3–5 個核心子系統。從這些模組及其內部相依項中擷取合約。在 CONTRACTS.md 標頭中註明範圍，以便審查人員知道涵蓋了哪些內容。
- **極大型專案 (> 500 個原始檔案)**：建議使用者一次針對一個子系統執行管線。每個子系統都有其自身的管線執行，產出其自身的 REQUIREMENTS.md、CONTRACTS.md 等。告知使用者：「此專案有 N 個原始檔案。為了獲得最佳結果，請針對每個主要子系統分別執行需求管線 (例如：『產生身分驗證模組的需求』)。針對整個程式碼庫執行單次管線將會因為對話內容限制而遺漏合約。」

如果使用者明確要求在大型程式碼庫上執行全專案範圍，請滿足要求，但警告涵蓋範圍將比子系統層級的執行更薄。

### 初始階段的範圍廣度

在第一次管線執行時，優先考慮廣度而非深度。涵蓋所有主要子系統和模組，而不是深入研究少數幾個。目標是建立一個廣泛的基準線，供後續的自我精煉迴圈和稍後的審查/精煉階段深化。如果您專注於 3 個模組而跳過了其他 8 個，則完整性檢查無法在它從未見過的模組中發現缺口。

對於同時具有核心函式庫和支援模組（中間件、外掛程式、配接器、擴充功能）的專案，請在階段 A 中至少包含核心和風險最高的支援模組。在 CONTRACTS.md 標頭中註明範圍，以便清楚瞭解涵蓋了哪些內容。精煉階段稍後可以擴大範圍，但初始階段應撒下內容視窗所允許的最廣泛的網。

### 合約擷取

閱讀每個（範圍內的）原始檔案，並列出它實作或應該實作的每個行為合約。行為合約是程式碼向其呼叫者做出的任何承諾：

- **METHOD (方法)**：公有方法對傳回值、副作用、異常、執行緒安全性的保證
- **NULL**：傳遞、傳回或儲存 Null 時會發生什麼
- **CONFIG (配置)**：設定選項在其邊界處的效果
- **ERROR (錯誤)**：擲出哪些異常、何時擲出，以及帶有什麼診斷資訊
- **INVARIANT (不變量)**：必須始終成立的屬性
- **COMPAT (相容性)**：為了回溯相容性而保留的行為
- **ORDER (順序)**：輸出/反覆運算順序是穩定的、有記錄的還是未定義的
- **LIFECYCLE (生命週期)**：資源建立/清理、初始化順序
- **THREAD (執行緒)**：執行緒安全性保證或需求

### 合約擷取規則

- **務必徹底。** 對於一個 200 列的檔案，預期會有 5–15 個合約。對於一個 1000 列的檔案，預期會有 20–40 個。如果您在一個具有實際邏輯的檔案中發現的合約少於 3 個，說明您跳過了某些內容。
- **包含內部檔案。** 內部合約很重要，因為公有 API 依賴於它們。
- **包含「應存在」的合約** — 程式碼未執行但根據其領域應該執行的事項。這些能擷取到缺失性漏洞。
- **閱讀程式碼，而非僅看 Javadoc/docstrings。** 當文件與程式碼不一致時，請將兩者都列出。
- **這是發現過程，而非評判。** 列出所有事項，即使看起來顯而易見。

### 輸出格式

```
# 行為合約擷取 (Behavioral Contract Extraction)
產生日期：[日期]
分析的原始檔案數：N
擷取的總合約數：N

## 依類別摘要
- METHOD: N
- NULL: N
- CONFIG: N
[依此類推]

### 路徑/至/檔案.副檔名 (N 個合約)

1. [METHOD] 類別名稱.方法名稱()：保證內容的描述
2. [NULL] 類別名稱.方法名稱()：傳遞/傳回 Null 時發生的情況
[依此類推]
```

---

### 需求標題格式

REQUIREMENTS.md 中的所有需求都必須使用 `### REQ-NNN: 標題` 格式，其中 NNN 是以零填充的三位數序號，標題是簡短的描述性名稱。「請勿」使用其他格式，如 `### REQ-NNN — 標題`、`### REQ-NNN. 標題`、`**REQ-NNN**: 標題` 或無編號的自由格式標題。一致的格式可讓自動化工具解析並交叉引用需求。

## 階段 B：從合約衍生需求

**輸入：** `quality/CONTRACTS.md`、專案文件、SKILL.md 步驟 7 範本。
**輸出：** `quality/REQUIREMENTS.md`

### 如何作業

**B.1 — 將相關合約分組。** 不同檔案中的許多合約都服務於同一個行為需求。按行為疑慮進行分組，而非按檔案分組。不要僅因為合約在同一個檔案中就合併不相關的合約。

**B.2 — 充實意圖。** 對於每個組群，從文件中尋找使用者故事：GitHub 問題說明了使用者的期望，使用者指南說明了預期行為，疑難排解文件揭示了已知的邊緣情況，設計文件解釋了設計目標。「以便 (so that)」子句必須源自對關注者及其原因的理解。

**B.3 — 撰寫需求。** 使用來自 SKILL.md 步驟 7 的 7 欄位範本。滿足條件 (conditions of satisfaction) 來自組群中的個別合約 — 每個合約成為一項滿足條件。

**B.4 — 檢查孤立合約。** 在撰寫完所有需求後，驗證 CONTRACTS.md 中的每個合約皆已涵蓋。未涵蓋的合約將成為新的需求，或被新增至現有需求的滿足條件中。

### 規則

- **「不要」限制需求數量。** 根據合約內容撰寫所需數量的需求。
- **每個合約都「必須」對應到至少一個需求。**
- **每個不同的行為疑慮對應一個需求。** 不要僅因為「執行緒安全性」和「Null 處理」在同一個類別中就將其合併。
- **「不要」修改 CONTRACTS.md。** 僅供讀取。

## 階段 C：驗證涵蓋範圍（迴圈，最多 3 次反覆運算）

**輸入：** `quality/CONTRACTS.md`、`quality/REQUIREMENTS.md`
**輸出：** `quality/COVERAGE_MATRIX.md`、更新後的 `quality/REQUIREMENTS.md`

對於 CONTRACTS.md 中的每個合約，判定其是否被某個需求涵蓋。如果某項需求的滿足條件明確測試了該行為，則該合約被視為「已涵蓋」。如果僅僅是附帶提及、暗示但未說明，或同一個檔案的其他方面已被涵蓋但此特定合約未被涵蓋，則該合約「未」被涵蓋。

### 輸出格式

```
# 合約涵蓋範圍矩陣 (Contract Coverage Matrix)
產生日期：[日期]
合約總數：N
已涵蓋：N (百分比)
未涵蓋：N (百分比)
部分涵蓋：N (百分比)

## 完全涵蓋的合約
[檔案]：[合約摘要] → REQ-NNN (滿足條件 #M)

## 部分涵蓋的合約
[檔案]：[合約摘要] → REQ-NNN 涵蓋了概略區域，但遺漏了 [特定方面]

## 未涵蓋的合約
[檔案]：[合約摘要] → 無需求處理此行為
```

在寫入矩陣後，修正 REQUIREMENTS.md 中的缺口：向現有需求新增缺失的條件，或建立新的需求。報告變更內容。

**迴圈終止條件：** 如果未涵蓋計數達到 0，請繼續執行階段 D。否則，重新產生矩陣並再次檢查。最多執行 3 次反覆運算。

## 階段 D：完整性檢查

**輸入：** `quality/REQUIREMENTS.md`、`quality/CONTRACTS.md`、`quality/COVERAGE_MATRIX.md`、原始碼樹。
**輸出：** `quality/COMPLETENESS_REPORT.md`、更新後的 `quality/REQUIREMENTS.md`

這是進入敘述階段前的最後一個閘道。執行三項檢查：

### 檢查 1：領域完整性

下列行為領域「必須」具備需求。請檢查每一項。此清單僅為最低要求 — 如果您注意到應該針對此專案領域具備需求但未列出的領域，請自行新增。

- [ ] **Null 處理**：明確的 Null、缺失的欄位、Null 鍵、集合中的 Null 值
- [ ] **型別強制轉換**：字串↔數字、字串↔布林值、數字精度、溢位
- [ ] **原始型別 vs 包裝類別 (Primitive vs wrapper)**：還原序列化期間原始型別與物件 Null 語意的差異（適用於具有此區別的語言）
- [ ] **泛型 (Generic types)**：清除邊界 (erasure boundaries)、萬用字元處理、遞迴泛型（適用於具有泛型的語言）
- [ ] **執行緒安全性**：並行存取、發佈安全性、快取可見性
- [ ] **錯誤診斷**：異常類型、路徑上下文、位置資訊
- [ ] **資源管理**：串流關閉、Reader/Writer 生命週期
- [ ] **回溯相容性**：有線格式 (wire format) 穩定性、API 行為穩定性
- [ ] **安全性**：DoS 防護（巢狀深度、字串長度）、注入預防
- [ ] **編碼**：Unicode、BOM、代理對 (surrogate pairs)、逸出序列
- [ ] **日期/時間**：格式優先順序、時區處理、精度
- [ ] **集合 (Collections)**：陣列、清單、集合、對應表、佇列 — 空值、Null 元素、排序
- [ ] **列舉 (Enums)**：名稱解析、別名、未知值
- [ ] **多型 (Polymorphism)**：執行階段型別 vs 宣告型別、配接器/處理常式委派
- [ ] **樹狀模型 / 中間表示 (IR)**：變更語意、深層複製結構獨立性、Null 正規化
- [ ] **配置 (Configuration)**：建構器不可變性、執行個體隔離、選項組合
- [ ] **進入點**：每個不同的公有進入點都必須有其自身的合約 — 基於字串、基於串流、基於樹狀、獨立解析、多值解析。如果程式庫有 N 種方式啟動讀取或寫入，則必須有 N 組合約。
- [ ] **輸出逸出**：預設逸出哪些字元、停用逸出會改變什麼、建構器層級與寫入器層級控制如何互動
- [ ] **內建型別處理常式合約**：對於處理標準函式庫型別的每個內建處理常式，說明其對格式、精度、正規化和來回保真度 (round-trip fidelity) 的承諾。需求應指明處理常式的承諾，而不僅僅是處理常式的存在。
- [ ] **欄位/屬性序列化順序**：輸出順序是遵循宣告順序、繼承順序、字母順序還是未定義。說明順序是承諾的合約，還是僅為觀察到的行為。
- [ ] **公用型別的身分合約**：公用模型型別上的 `toString()`、`hashCode()`/`equals()` (或等效語言語法)。這些是使用者依賴於比較、記錄和集合鍵使用的行為合約。
- [ ] **輸入驗證**：對於每個具有領域約束的配置欄位，說明有效範圍以及是否存在驗證。

對於每個領域，請引用涵蓋它的 REQ-NNN 編號，或將其標記為缺口。

### 檢查 2：可測試性稽核

針對每個需求，檢查其滿足條件是否確實可測試。審查人員能否根據此條件編寫具體的測試案例？通過/失敗是否明確？條件是否涵蓋了失敗模式，而非僅涵蓋正常路徑？

### 檢查 3：跨需求一致性

檢查引用相同概念的需求對。範圍是否一致？Null 處理規則是否一致？執行緒安全性保證是否與生命週期合約衝突？配置預設值在不同需求之間是否匹配？

### 檢查 4：跨成品一致性（如果存在程式碼審查或規格稽核結果）

如果 `quality/code_reviews/` 或 `quality/spec_audits/` 包含來自先前或當前執行的結果，請閱讀它們。對於狀態為 VIOLATED (已違反)、BUG (漏洞) 或 INCONSISTENT (不一致) 的每項發現，請檢查需求是否處理了該發現所針對的行為疑慮。如果程式碼審查在需求未涵蓋的壓縮標頭解析中發現了漏洞，那就是一個完整性缺口 — 請新增需求或滿足條件以補足。

**如果存在未處理的發現，完整性報告不能聲稱為 COMPLETE (完整)。** 如果來自程式碼審查或規格稽核的任何 VIOLATED/BUG/INCONSISTENT 發現針對的是需求未涵蓋的行為，判定結果必須為 INCOMPLETE (不完整)，並列出具體缺口。

此檢查的存在是因為管線的早期版本產生的完整性報告標記為「COMPLETE」，而同一執行中的程式碼審查卻發現了需求違規。完整性報告必須與所有其他品質成品保持一致。

### 審查後完整性重新整理（強制性）

**在程式碼審查和規格稽核完成後**，重新閱讀 `quality/COMPLETENESS_REPORT.md` 並更新它。初始的完整性報告是在程式碼審查和規格稽核執行前撰寫的，因此它無法反映這些發現。此重新整理步驟將完整性判定結果與實際審查結果進行協調。

**程序：**
1. 閱讀來自 `quality/code_reviews/` 的綜合摘要 — 統計 VIOLATED 和 BUG 發現的數量。
2. 閱讀來自 `quality/spec_audits/` 的分類摘要 — 統計已確認的程式碼漏洞數量。
3. 對於每項發現，檢查 REQUIREMENTS.md 是否有涵蓋該行為的需求。
4. 向 COMPLETENESS_REPORT.md 附加 `## Post-Review Reconciliation` (審查後對帳) 章節：

```
## Post-Review Reconciliation
更新日期：[日期]

### 程式碼審查發現：N 個 VIOLATED，M 個 BUG
- [發現摘要] → 已由 REQ-NNN 涵蓋 / 未涵蓋 (缺口)
- ...

### 規格稽核發現：N 個已確認程式碼漏洞
- [發現摘要] → 已由 REQ-NNN 涵蓋 / 未涵蓋 (缺口)
- ...

### 更新後的判定結果
[如果所有發現皆已由需求涵蓋，則為 COMPLETE；如果仍有缺口，則為 INCOMPLETE]
```

5. 如果原始判定結果為 COMPLETE 但存在未處理的發現，請將判定結果更改為 INCOMPLETE。

### 解決程式碼審查與規格稽核之間的衝突

當程式碼審查與規格稽核對同一個行為主張產生分歧時 — 一方說是 BUG，另一方說是設計選擇或假陽性 — 對帳程序必須解決該衝突，而非敷衍了事。

**解決程序：**
1. 識別爭議核心的事實主張。程式碼實際執行了什麼？
2. 部署驗證探針：為模型提供有爭議的主張和相關原始碼，並要求其報告地面實況 (ground truth)。（參見 `spec_audit.md` §「驗證探針 (The Verification Probe)」）
3. 在「審查後對帳」章節記錄解決方案：
   ```
   ### 衝突已解決
   - [發現描述]：程式碼審查說是 [X]，規格稽核說是 [Y]。
     驗證探針：[程式碼實際執行的內容]。
     解決方案：[BUG CONFIRMED / FALSE POSITIVE / DESIGN CHOICE]。[說明。]
   ```
4. 如果解決方案確認為 BUG，請確保它具有迴歸測試。如果解決方案推翻了 BUG，請按 `review_protocols.md` §「規格稽核翻轉後的清理」清理迴歸測試。

**不要透過預設採用其中一個來源來解決衝突。** 程式碼審查和規格稽核都沒有絕對的權威性 — 它們使用不同的方法（結構化閱讀 vs. 規格比較），且具有不同的盲點。驗證探針是決定性的關鍵。

**此重新整理不是選填的。** 早於程式碼審查的完整性報告只是一個時間戳記，而非品質閘道。重新整理將其轉化為實際的對帳。

### 輸出格式

```
# 完整性報告 (Completeness Report)
產生日期：[日期]

## 領域涵蓋範圍
[針對每個領域：COVERED (REQ-NNN, REQ-NNN) 或 GAP (描述)]

## 可測試性問題
[針對每個模糊需求：REQ-NNN — 條件 N 不可測試，因為……]

## 一致性問題
[針對每個衝突：REQ-NNN 與 REQ-NNN 在……方面不一致]

## 跨成品缺口 (若程式碼審查/規格稽核結果存在)
[針對每項未處理的發現：發現摘要 → 缺失的需求或條件]

## 判定結果
COMPLETE 或 INCOMPLETE，並附帶建議動作
```

然後盡可能修復：為領域缺口新增需求、精確化模糊條件、解決一致性問題，並彌補跨成品缺口。

**重要提示：** 這是最後一項檢查。請採取對抗性立場。假設先前的傳遞並不完美。對於每個標記為 COVERED 的領域，請驗證引用的需求是否確實處理了清單項目 — 不要只是勾選核取方塊。

### 自我精煉迴圈（最多 3 次反覆運算）

在初始完整性檢查後，執行最多 3 次精煉反覆運算，以彌補階段 D 識別的缺口：

1. **閱讀完整性報告。** 識別所有 GAP 條目、可測試性問題和一致性問題。
2. **修正 REQUIREMENTS.md 中的缺口。** 針對每個 GAP：使用 7 欄位範本新增需求，或向現有需求新增滿足條件。針對可測試性問題：精確化條件。針對一致性問題：解決衝突。
3. **重新執行所有三項檢查**（領域完整性、可測試性稽核、跨需求一致性）。將更新後的結果寫入 COMPLETENESS_REPORT.md。
4. **計算增量。** 在此反覆運算中新增了多少個新需求，或修改了多少個現有需求？
5. **短路檢查：** 如果增量少於 3 個變更，請停止 — 您已遇到報酬遞減。繼續執行階段 E。

**為什麼這樣做有效：** 初始完整性檢查識別了缺口，但模型可能無法在一次傳遞中修復所有缺口，特別是模型需要重新閱讀原始檔案以瞭解缺失內容的概念性缺口。每次反覆運算都會縮小缺口。三次反覆運算足以彌補機械性缺口；剩餘的概念性缺口則是跨模型稽核和人工審查發揮作用之處。

**為什麼它有限制：** 這是自我精煉 — 同一個模型檢查其自身的工作。它可以擷取模型在被指出後能看到的缺口（未涵蓋領域、模糊條件、數值不一致），但無法擷取模型未識別為缺口的盲點。這是設計使然。審查與精煉協定的存在，是為了透過不同的模型或人工輸入來彌補這些更深層次的缺口。

迴圈完成（或短路）後，繼續執行階段 E。

## 階段 E：敘述傳遞

**輸入：** `quality/REQUIREMENTS.md`、`quality/CONTRACTS.md`、專案文件、原始碼樹。
**輸出：** 重組後的 `quality/REQUIREMENTS.md`

**開始之前：** 儲存備份：`cp quality/REQUIREMENTS.md quality/REQUIREMENTS_pre_narrative.md`

此階段將規格說明書轉化為指引。新增說明性內容，以便新的小組成員、程式碼審查人員或 AI 代理程式可以從頭到尾閱讀文件並理解該軟體。

### E.1 — 專案概觀 (全新，位於文件頂部)

撰寫 400–600 字的連貫散文，解釋：該軟體是什麼、誰在使用它以及原因（主要角色與目標）、資料如何流經主要元件，以及設計理念（關鍵的架構決策及其原因）。

### E.2 — 使用案例 (全新，位於概觀之後)

以《應用軟體專案管理 (Stellman & Greene)》的風格撰寫 6–8 個使用案例。每個案例包含：

- **名稱**：簡短的描述性名稱
- **參與者**：誰發起了此案例
- **前置條件**：開始前必須為真的條件
- **步驟**：編號的參與者/系統動作序列
- **後置條件**：成功時為真的條件
- **替代路徑**：變體與錯誤案例
- **需求**：此使用案例練習了哪些 REQ-NNN 編號

涵蓋主要的使用模式。使用案例是「軟體的功能」與「需求規格」之間的橋樑。

### E.3 — 跨領域疑慮 (全新，位於使用案例之後)

記錄跨越多個類別的架構不變量：執行緒模型、Null 合約、錯誤理念、回溯相容性策略、配置組合。每一項都應引用特定的 REQ-NNN 編號。以散文段落形式撰寫。

### E.4 — 類別敘述 (充實現有內容)

針對每個需求類別，在第一個需求前新增 2–4 個句子，解釋該類別涵蓋的內容、它與其他類別的關係，以及審查人員應注意的事項。

### E.5 — 依由上而下的流程重新排序

按從使用者導向（進入點、配置）到基礎設施（錯誤處理、回溯相容性）的順序重新排列類別。將任何雜項章節併入適當的類別中。

### E.6 — 依序重新編號

重新排序後，依照文件順序為所有需求重新編號 REQ-001 到 REQ-NNN。更新所有內部交叉引用。

### 規則

- **「不要」在自動化精煉期間刪除、合併或削弱任何現有需求。**
- **「不要」在此次傳遞中新增需求。**
- **從使用者的角度撰寫概觀與使用案例。**
- **使用案例必須引用特定的 REQ 編號。**

## 版本控制協定

### 版本方案：主版本號.次版本號 (major.minor)

- **主版本號 (Major)** 提升：結構性變更（新的管線架構、新增敘述傳遞、重大範圍擴張）。由使用者提升。
- **次版本號 (Minor)** 提升：精煉傳遞、彌補缺口、精確化條件。在每次管線執行或精煉傳遞時自動增加。

### VERSION_HISTORY.md

在 `quality/VERSION_HISTORY.md` 維護版本歷程記錄檔案：

```markdown
# 需求版本歷程記錄

## 目前版本：vX.Y

| 版本 | 日期 | 模型 | 作者 | 需求數 | 摘要 |
|---------|------|-------|--------|------|---------|
| v1.0 | YYYY-MM-DD | [模型] | 品質播放手冊 | N | 初始管線產生 |
| v1.1 | YYYY-MM-DD | [模型] | [作者] | N | [變更內容] |

## 待審查
[若審查正在進行中，則顯示來自 REFINEMENT_HINTS.md 的狀態]
```

**作者 (Author)** 欄位記錄出處：「品質播放手冊」用於自動化管線執行，人名用於手動編輯，模型名稱用於精煉傳遞。

### 備份協定

在每次版本變更前，將 `quality/` 中的所有檔案複製到 `quality/history/vX.Y/`：

```
quality/history/
├── v1.0/
│   ├── REQUIREMENTS.md
│   ├── CONTRACTS.md
│   ├── COVERAGE_MATRIX.md
│   └── COMPLETENESS_REPORT.md
├── v1.1/
│   └── ...
└── v2.0/
    └── ...
```

每個版本資料夾都是一個完整的快照。使用者可以對任意兩個版本執行 diff。

### 版本標記

REQUIREMENTS.md 標頭包含目前版本：

```markdown
# 行為需求 — [專案名稱]
版本：vX.Y
產生日期：[日期]
管線：帶有敘述傳遞的合約擷取 v2
```

## 管線之後：審查與精煉

管線產出了一個堅實的基準線，但 AI 並非 100% 可靠。此技能提供了兩個用於反覆運算改進的獨立工具：

### 需求審查 (`quality/REVIEW_REQUIREMENTS.md`)

對按使用案例組織的需求進行互動式或引導式審查。三種模式：
- **自我引導**：挑選要深入研究的使用案例
- **完全引導**：依序逐一瀏覽使用案例
- **跨模型稽核**：由不同的模型事實檢查完整性報告

進度與回饋追蹤於 `quality/REFINEMENT_HINTS.md`。有關完整協定，請參閱產出的 `quality/REVIEW_REQUIREMENTS.md`。

### 需求精煉 (`quality/REFINE_REQUIREMENTS.md`)

讀取 `quality/REFINEMENT_HINTS.md` 並更新 `quality/REQUIREMENTS.md` 以彌補識別出的缺口。可以使用任何模型執行。備份目前版本，提升次版本號，報告所有變更。有關完整協定，請參閱產出的 `quality/REFINE_REQUIREMENTS.md`。

### 多模型精煉

使用者可以使用不同的模型執行精煉傳遞，以捕捉不同的盲點。每次傳遞：備份 → 精煉 → 版本提升 → 在 VERSION_HISTORY.md 中記錄。可以執行任意數量的模型，直到遇到報酬遞減為止。
