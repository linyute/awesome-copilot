# 文件與現代化

針對使用者已在本機簽出 (check out) 的儲存庫，提供兩個互補的工作流程，並打包為一個 skill：

- **文件模式** — 從磁碟上的程式碼產生一份具權威性且有引用來源的架構文件。非常適合用於上線引導、系統設計圖，或作為現代化工作的實證基礎。
- **現代化模式** — 將該架構轉化為分階段、安全階梯化的計劃，以升級、遷移或重寫舊有系統。


## 模式選擇

- 如果使用者想要**理解、記錄、對照、研究或加入**某個程式碼庫，請執行**文件模式**。
- 如果使用者想要**對系統進行現代化、遷移、升級或重寫**，請執行**現代化模式**。現代化模式是自給自足的：如果目前尚無架構文件，它會先執行**文件模式**工作流程（在同一次執行中），然後直接繼續執行計劃。

如有疑問，請先產生架構文件 — 它是這兩種模式所依賴且經過審查的實證基礎。


## 文件模式

針對使用者已在本機簽出的儲存庫，產生一份具權威性且有引用來源的架構文件。其目標是撰寫一份可以交給新工程師作為其上線培訓參考的文件 — 內容要足夠廣泛以涵蓋整個系統，對困難的部分要足夠深入才有用，並且因為每個主張都可以追溯到磁碟上的檔案而值得信賴。


### 為什麼優先考慮本地

從本機簽出的內容（而非 GitHub API 或網頁）讀取是刻意設定的**預設行為**。它速度更快、免費、免於 API 速率限制，而且最重要的是，它描述的是*您眼前的確切程式碼*，而不是遠端 `main` 分支當下的樣子。唯一的折衷是無法看到僅限遠端的事實（星數、完整的 CI 執行歷史、同層級儲存庫）。這沒關係：請將這些聲明為超出範圍或將其標記為 `[UNVERIFIED]`，而不是靠猜測。

本機優先並不代表「絕不存取遠端」：網頁/API 查詢是刻意保留的**最後手段備用方案**，僅用於確實無法從磁碟判定且對文件有實質影響的事實。當您真的需要存取遠端時，請明確標示結果（例如 `[UNVERIFIED]` / 來源為遠端），好讓讀者知道它並非來自簽出的內容，並且絕不要讓它成為取代閱讀磁碟程式碼的便捷路徑。


### 工作流程

1. **首先確立身分。** 執行 `git remote -v`、`git branch --show-current` 和 `git log -1`，讓文件錨定到特定的遠端、分支和提交 (commit)。讀者必須能夠判別此文件描述的是哪一個快照。遠端 URL 可能包含內嵌的憑證（例如 `https://<token>@github.com/...`） — 在文件中記錄 URL 之前，請**刪除任何憑證/權杖**。
2. **偵測，而非假設。** 讀取真實的資訊清單（如 `go.mod`、`package.json`、`Cargo.toml`、`pyproject.toml`、`pom.xml` 等）、`Makefile`/任務執行器、CI 設定以及任何儲存庫特定的代理程式或貢獻者文件（如 `AGENTS.md`、`CONTRIBUTING`、`README`、`docs/`）。這些是技術堆疊和命令的單一事實來源 — 請優先使用它們，而非您先前對該框架的知識。
3. **先對照全局，再深入探討細節。** 首先建構整個儲存庫的對照地圖（下方的三個視角），然後挑選 2-3 個最困難的子系統並對其進行深入探討。
4. **邊執行邊驗證。** 開啟您引用的檔案。如果您引用了行號，您應該要確實讀過該行。在這裡，無根據的主張比遺漏更糟糕 — 此文件的全部價值在於它是值得信賴的。


### 輸出結構

依此順序，產生**單一 Markdown 檔案**並包含以下章節。根據實際專案調整標題（例如 CLI 工具沒有「前端」視角 — 將該位置併入對該儲存庫重要的任何部分中），但保留三視角外觀與驗證紀律。


#### 第一部分－全程式碼庫技術深度解析
- 該儲存庫是什麼（一個段落，引用至 README）。
- 技術堆疊偵測表：圖層 | 技術 | 證據（檔案+行號）。
- 進入點（後端、前端、CLI — 任何適用的項目）。
- **命令與驗證清單** — 專案標準命令的表格（`command | purpose | evidence`），經由任務執行器 / 資訊清單 / CI 設定所驗證，而非憑空猜測。涵蓋建構 (build)、執行/伺服 (run/serve)、測試（以及如何執行單一測試）、Lint、格式化，以及（如果存在）型別檢查、端對端/冒煙測試、合約測試以及任何其他閘門命令，加上執行這些命令的 CI 工作流程及其觸發條件。**同時記錄 CI 是否被*強制執行*** — 亦即是否有任何工作流程是**必要狀態檢查 / 分支保護規則**且實際會阻擋合併，而非僅僅是執行 — 因為這種區別是手動、由人員設定的，現代化模式必須將其呈現出來，而非假設。強制執行狀態通常無法單憑本機簽出內容來判定：請詢問使用者，或除非已從權威來源確認，否則標記為 `[UNVERIFIED]`（依據上述本機優先規則，任何遠端查詢都是已標記的最後手段）。此清單是下游規劃（現代化模式）所引用的單一事實來源，因此其結束條件是可執行的，而非流於形式。請根據各個生態系統（npm/yarn/pnpm、`make`、`just`、`cargo`、`go`、`poetry`/`tox`/`nox`、`gradle`/`maven` 等）偵測這些內容 — 不要假定技術堆疊。將任何您無法驗證的命令標記為 `[UNVERIFIED]`。
- 每個主要區域的目錄配置，每個目錄附帶一行的用途說明。
- **部署與執行階段介面** — 系統*執行*（而不僅僅是建構）時所固定之語言/執行階段及後端服務版本的每一個位置的表格：容器基礎映像檔（`Dockerfile`/`Containerfile`、`docker-compose*` 建構上下文）、CI 執行器映像檔 / `setup-*` 版本、`engines`/`.nvmrc`/`.tool-versions`/`runtime.txt`、無伺服器/lambda 執行階段，以及具狀態之資料儲存庫映像檔標記（資料庫/快取/代理器/搜尋）。引用時請註明檔案+行號。此介面是後續平台/執行階段升級時必須同步移動的部分 — 請在此處標示建構執行階段與執行執行階段之間的任何偏差，以便在撰寫現代化計劃之前使其清晰可見。
- **EOL / 停用依賴項掃描** — 指出已達壽命結束 (EOL)、無人維護，或在可能的目標主版本中被移除的框架、執行階段、基礎映像檔和函式庫（例如，其下一個主版本重新命名命名空間或捨棄某個元件家族的框架）。適當地將每一項標記為 `[INFERRED]`/`[UNVERIFIED]`。這是現代化模式可行性探討和危害紅隊演練所基於的原始資料。
- 資料/儲存層、API、外掛程式/擴充功能、背景工作、CI/CD、測試。


#### 第二部分—背景與生態系統
- 本機簽出身分表（遠端、分支、HEAD 提交、版本、授權條款）。
- 存在之儲存庫特定代理程式/貢獻者文件，以及它們所編碼的規則。
- 開發人員注意事項（測試監控模式預設值、建構緩慢、產生的程式碼必須提交、pre-commit 攔截器）— 各自引用。
- 此專案如何與更廣泛的生態系統或同層級服務相關聯，*如同從磁碟上可見的*（建構標記、選用的連結儲存庫、可單獨部署的元件）。請勿匯入遠端生態系統的瑣碎資訊。


#### 第三部分—建築藍圖
- 技術堆疊摘要（可參考 Part 1 的表格）。
- 以 Mermaid 呈現的 C4 風格圖表：第 1 層系統上下文、第 2 層容器、第 3 層具代表性的請求/元件生命週期。
- 分層與依賴關係規則（何者可以依賴何者，以及由何者強制執行）。
- 橫切關注點 (cross-cutting concerns) 表格：驗證、設定、記錄、計量/追蹤、秘密資訊、錯誤處理、功能旗標 — 每一項都附有其位置與證據。
- 推導出的架構決策記錄 (Architectural Decision Records)（從程式碼 + 文件重構）。
- 治理與強制執行機制（CI 閘門、程式碼產生驗證、CODEOWNERS、審查閘門、相容性規則）。
- 「如何新增功能」指南以及常見陷阱。


#### 子系統深度分析
識別出 2-3 個最複雜或在架構上最重要的子系統 — 亦即新工程師最感吃力的部分，例如評估/排程引擎、外掛程式載入器管線、狀態機，或轉譯/遷移框架。為每個子系統新增一個專用子章節，涵蓋其內部結構、生命週期或狀態機、關鍵型別和資料流，並附帶本機檔案+行號引用，以及在有助於釐清流程時提供一個小型 Mermaid 圖表。這正是實用的上線培訓文件與目錄列表的區別所在 — 請在此投入真正的精力。


#### 置信度評估
主要主張區域的表格，評等為 **High / Inferred / Unverified**，讓讀者確切知道哪些部分可以直接信賴，哪些部分需要重複確認。


#### 註腳－本地文件引用
此文件所依賴之關鍵本機檔案的清單，每個檔案附帶一行說明其確立的內容。


### 使文件可信的慣例

這些習慣將此 Skill 的輸出與一般性概述區隔開來。它們非常重要，因為文件的全部價值在於讀者可以直接依賴它，而不需要重新推導所有內容。

- **將每個非顯而易見的主張都引用**至本機路徑，並註明標定具體內容的行號（`pkg/server/server.go#L39-L41`）。從儲存庫根目錄開始的相對路徑可保持連結為可點選狀態。
- **誠實標記不確定性。** 對於您推論出來但未見明確說明的內容，請使用 `[INFERRED]`；對於您轉述但未確認的內容（例如來自您未重新測量的文件中的建構耗時主張），請使用 `[UNVERIFIED]`。誠實面對落差比虛假的自信更有用。
- **解決矛盾，而非重述矛盾。** 如果兩個來源不一致（例如程式碼中的版本常值與資訊清單不符），請去閱讀程式碼，判斷真正的答案，並標記為 `[Resolved contradiction]` 並附帶說明。讓讀者自己去思考衝突是失敗的模式。
- **記錄儲存庫強制的相容性與部署節奏規則** — 獨立的 FE/BE PR、雙向儲存相容性、僅限累加的 protobuf 變更 — 因為這些是新進人員最容易違反的規則。
- **偏好精確的數量而非模糊的描述。** 「73 個服務套件」、「89 個工作流程檔案」（來自目錄列表）讀起來像是經過驗證的；「許多服務」讀起來像是猜測。


### 範圍控制

保持文件立足於簽出的內容。這很容易向外漂移到專案更廣泛的生態系統（相關產品、README 行銷、衛星儲存庫） — 除非它在磁碟上清晰可見，否則請拒絕這麼做，並明確標記來自本機樹狀結構之外的任何內容。讀者要求的是忠實記錄*此程式碼庫*。


## 現代化模式

為舊有程式碼庫產生一份完整且具操作性的現代化計劃。此 Skill 專注於前瞻性工作 — 要現代化什麼、原因、順序以及如何進行 — 但它是**自給自足的**：它會先確保有架構文件存在，在需要時透過文件模式產生一份。

**假定預設為失效狀態。** 人們之所以尋求現代化，恰恰是因為舊有的技術堆疊很難或不可能升級 — 壽命結束 (EOL) 的執行階段、無法編譯的原生模組、已停用的套件鏡像站、已廢棄的框架。因此，此 Skill **不**假設您可以在接觸任何事物之前，先復活舊有的工具鏈並建立一個完全綠燈的 CI 閘門。那種「先凍結再提升 (freeze-then-lift)」的方法是*幸運*的情況，而不是預設情況。相反地，此 Skill 會執行一個**有時間限制的可行性探討**，然後根據系統的實際存活狀態，選擇遷移策略和**安全策略**。在一個真正失效的應用程式上，建構一個綠燈的舊有閘門*本身就是一個現代化專案* — 這是此 Skill 旨在避免的循環陷阱。

三個概念驅動著整個計劃，並在下方介紹：**可測試性里程碑 (Testability Milestone)**（即針對各個元件，系統何時可以實際重新建構、執行並通過測試）、**安全階梯 (safety ladder)**（您實際上可以達到的最高迴歸安全階級，將降級視為有幸得到的結果，而非失敗），以及 **CI 里程碑 (CI Milestone)**（哪一個階段首先建立 CI — 並提醒*強制執行* CI 作為必要檢查是一個人手動的步驟，而非代理程式可以執行的操作）。


### 先決條件

此 Skill 在規劃前需要先瞭解系統的架構。請依下列方式解決：

1. 如果架構文件已經存在 — 由上方的**文件模式**產生，或是使用者指向的 README / ARCHITECTURE.md，或足夠的先前對話上下文 — 請直接使用它並跳至下方的生命週期工作流程。
2. **如果尚無架構文件，請先執行上方的文件模式工作流程**以產生一份有引用來源的架構文件，然後在同一次執行中**直接繼續**進行下方的 Phase 1。不要停下來要求使用者分開執行，也不要在兩份文件之間暫停進行審查。

其結果是產生兩個 Artifacts：架構文件（經審查的實證基礎）和此現代化計劃（前瞻性行動方案）。

**在規劃之前，請確認存在「命令與驗證清單」。** 只有在結束條件為*可執行*時，它才有價值，因此計劃必須能夠引用專案的標準建構 / 執行 / 測試 / Lint / 型別檢查 / e2e / 合約命令以及 CI 閘門。文件模式會在 Part 1 產生此清單；如果您是根據缺乏此清單的 README 或先前上下文進行工作，請在撰寫結束條件之前，自行偵測並記錄這些命令（依據各個生態系統 — npm/yarn/pnpm、`make`、`just`、`cargo`、`go`、`poetry`/`tox`/`nox`、`gradle`/`mvn` 等）。絕不要發明您未曾對照任務執行器 / 資訊清單 / CI 設定所驗證過的命令。


### 工作流程


#### Phase 1: 評估現況（基於現有文件）

閱讀架構文件並擷取：

1. **技術堆疊清單** — 語言、框架、執行階段、有標示版本的依賴項。
2. **功能/領域地圖** — 系統的主要功能領域。
3. **已識別出的痛點** — 現有文件中指出的技術債、疑慮、已棄用的依賴項、違反架構的處置。
4. **部署與基礎設施** — 目前運行的情形。

請勿重新閱讀每個原始檔案。請信賴架構文件。僅在現代化問題需要驗證細節時（例如「此依賴項是確實被使用，還是僅僅被宣告？」），才開啟特定的檔案。


#### Phase 2: 可行性探索、策略分歧與安全階梯

這是整個計劃的關鍵樞紐。**請在推薦目標架構或撰寫任何階段之前執行此步驟。** 它的工作是快速且誠實地回答：*這個系統有多大程度存活、適合哪種遷移形式，以及我們實際上可以達到多少迴歸安全性？*


##### 2a. 進行限時可行性測試

設定一個**嚴格的時間限制**（例如一天）。此可行性探討的問題**不是**「我們能讓 CI 變綠燈嗎？」 — 而是「我們能否讓它執行甚至*一次*以擷取其行為？」 針對每個元件 / 可部署單元進行探測：

- 它的依賴項今天是否可以**安裝**（理想情況下是**從已提交的鎖定檔案安裝，無需手動修補**）？
- **原生/建構步驟**在**目前受支援的**工具鏈上是否成功？
- 它是否能**開機 / 啟動**？
- **測試執行器是否能執行**，以及**是否有 ≥1 個有意義的測試通過**？

記錄您實際觀察到的情況 — 包括部分成功（在 monorepos 中很常見：一個服務安裝且測試綠燈，而另一個服務無法編譯）。**不要**投入兩個衝刺 (sprint) 來復活一具屍體以發現它是無法復活的。當時間限制到期時，請根據您擁有的證據做出決定。


##### 2b. 找到測試性里程碑（每個組件）

**「可測試性」並非您可以在失效應用程式上滿足的前置條件 — 對於失效的應用程式來說，它是現代化的*輸出*，而非輸入。** 因此，每份計劃都必須預先指明其**可測試性里程碑 (Testability Milestone)**：亦即系統（或特定元件）首次同時滿足**所有四個**條件的具體階段：

1. 執行於**受支援的**（非 EOL）執行階段。
2. 依賴項可**從鎖定檔案安裝，無需手動修補**。
3. 原生 / 建構步驟在**目前的工具鏈上成功**。
4. **測試執行器可執行，且 CI 中有 ≥1 個有意義的測試通過**。

請**針對各個元件**進行評估 — 正如 Nylas 參考執行所示，一個接近現代化的雲端服務可以立即跨越該界線，而舊有的 Electron 用戶端則要到好幾個階段後才能跨越。**此里程碑是計劃中唯一最重要的標記。** 請明確指出每個元件的里程碑落在何處。


##### 2c. 給每個階段貼上「測試前」或「測試後」的標籤

可測試性里程碑將工作劃分為兩個具有**不同安全規則**的體制 (regimes)：

- **可測試性之前（「黑暗」）體制：** 元件無法被信賴以執行。安全性來自**非測試**來源（請參見下方的安全階梯）。**在此處要求 CI 測試閘門是類別錯誤 (category error)** — 您正在向尚未存活的項目要求綠燈測試。
- **可測試性之後（「明亮」）體制：** 它在存活且受支援的技術堆疊上建構並執行。*此時*真正的 CI、特性描述 (characterization) 和端對端 (e2e) 測試是可達成的，且值得投入。經典的綠燈閘門屬於**這裡**。

**規則：在元件跨越其自身的可測試性界線之前，絕不要要求對該元件設定自動化測試閘門。**


##### 2d. 選擇遷移策略－A/B 分叉

可行性探討決定了適用哪兩種形式（這在 monorepo 中是針對各個元件的呼叫）：

- **(A) 先凍結再提升 (Freeze-then-lift)。** 照原樣網住 (net) 舊有的應用程式，然後在網內進行升級。要求舊有應用程式是**可以以可接受的成本復活的** — 意即它*幾乎*可以執行。這是「在變更任何事物之前先進行特性描述測試」的傳統路徑。**屬於選用 (opt-in)，而非預設。**
- **(B) 先建立據點再擴張 (Beachhead-then-expand)（步進骨架 / walking skeleton）。** 不要網住屍體。在現代技術堆疊上驅動**最薄的端對端切片**，直到它可以建構、啟動並進行測試；然後逐漸將其餘部分絞殺 (strangle) 進來，並在進行的過程中**在新的技術堆疊上**撰寫測試。**這是失效應用程式的預設路徑。** 在策略 B 中，第一階段是*「用最小的切片達到可測試性里程碑」*，而非*「凍結整個舊有工具鏈」*。

如果元件**根本無法建構或執行**，那麼根據定義，在*舊有*技術堆疊上進行特性描述測試是不可能的 — 請反轉此方法：**停止嘗試網住屍體，並以漸進的方式在*目標*技術堆疊上建立網**，並使用舊有的程式碼/輸出作為參考預測源 (oracle)。


##### 2e. 選擇安全梯上能夠達到的最高等級

迴歸安全性是一個階梯，而非二分法。請針對每個元件選擇您**實際上可以達到的最高階級**；**只要指明殘留風險，降級至較低階級是被允許且一流的結果，而不是失敗。**

- **L4 — 完整自動化閘門：** CI 中的綠燈 Lint + 單元測試 + 特性描述測試 + 端對端測試。
- **L3 — 部分閘門：** 某些測試套件綠燈 + 鎖定檔案 + 可達成之子集上的 CI；其餘部分以具名清單**隔離 (quarantined)**。
- **L2 — 僅限特性描述 / 黃金母片 (golden-master)：** 無法在 CI 中執行，但可擷取 I/O 快照 / 行為差異（手動是可接受的）。
- **L1 — 基於可回復性：** 無可執行的測試；安全性來自微小的可回復步驟、絞殺者模式 (strangler) / 平行執行、冒煙測試清單以及審查。
- **L0 — 無法建立網：** 視為規格驅動的重寫（舊程式碼 = 規格）或封存。

**安全性不單指測試。** 至少有五種來源，而在失效的應用程式上，2-4 的重要性通常超過 1：

1. **測試**（特性描述 / 單元 / 端對端）— 最強，但在失效應用程式上往往無法達成。
2. **可回復性** — 微小的提交、每次變更一個分支、簡單的回滾。
3. **隔離** — 功能旗標、絞殺者模式 (strangler-fig)、平行執行新舊系統並**比對輸出差異**。
4. **將運行中的舊系統作為預測源 (oracle)** — 記錄/重播真實 I/O，以在沒有測試控件的情況下描繪行為特徵。
5. **人工審查 + 領域知識。**

**將安全性錨定在行為*接縫 (seams)*，而非單元。** 任何應用程式持久且與技術堆疊無關的合約是其外部可觀察的行為：HTTP 端點、資料庫綱要/查詢、檔案/有線格式、CLI 輸出、協定。將*這些*固定為黃金母片 / 核准快照 — **它們在重寫中得以存活；當您替換模組時，失效舊有程式碼上的單元測試會立即被刪除。** 根據可用資源取得**預測源 (oracle)**，並進行分級：

- **最佳 — 它在*某處*執行**（生產環境、虛擬機器、舊容器，甚至是唯讀狀態），**且您被允許使用它**：記錄真實的 I/O 並在新的程式碼上重播。一次成功的擷取勝過永久綠燈的舊有 CI。*（不要假設此階級隨時可用 — 忠實的第三方重新實作或復活的舊有二進位檔案可能會因原則、授權或「僅使用我們擁有的資源」的明確決策而被限制。如果是這樣，請跳至自我凍結階級。）*
- **備用 — 它根本無法執行：** 舊的**原始碼就是規格**；輔以生產環境日誌、資料庫傾卸 (dumps)、文件、工單，並接受在**新實作上**撰寫特性描述測試，以在驗證正確後鎖定它。
- **自我凍結黃金母片 — 不存在或不允許外部參考：** 當沒有任何值得信賴的東西可以執行，*且*您不得借用外部預測源時，**現代化後的元件會在它首次執行時（在其可測試性里程碑）記錄其*自身*的黃金母片**，且隨後的每個階段都會與該凍結的快照進行差異比對。這是一個真實、可用的階級 — 但**請明確指出其殘留風險**：它保證了*後續重構的自我一致性*，**而非**首次啟動的正確性（快照的正確性取決於產生它的那次執行）。將其與人工審查 / 規格交叉檢查配對，以核准該首次執行。


##### 2f. 將經濟/預言分類納入選擇考量

在選擇中納入經濟/預測源評估 (economic/oracle triage) 時權衡**迴歸成本 (regression cost)**：應用程式是否**在生產環境中執行**？它是否有**真實使用者**？**舊系統是否仍可執行**以作為參考？一個擁有零使用者的廢棄應用程式其迴歸成本接近於零 — 在此設定昂貴的閘門是**過度工程 (over-engineering)**。一個擁有使用者的生產環境系統則應採用較高的階級與真實的預測源。


##### 2g. 找到持續整合里程碑（指出確立持續整合的階段）

正如可測試性里程碑指出*元件何時可以首次執行與測試*一樣，**CI 里程碑**指出了*何時首次建立持續整合 (CI)* — 且必須在**藍圖中明確指明，而非埋藏在階段內容中。** 現代化計劃常見的失敗是讓讀者無法判別哪一個階段引進了 CI；請明確指出它。

**規則：CI 在*第一個明亮階段*建立 — 亦即在元件跨越其可測試性里程碑的同時或緊接在後 — 絕不能在此之前。** 元件在尚無法建構/執行時，CI 不可能是綠燈，因此在黑暗體制期間連接 CI *閘門*是與要求測試閘門相同的類別錯誤。（微不足道的鷹架 — 例如僅 Lint 或僅建構的工作流程 — 可能會較早出現，但具權威性的測試閘門則降落在 CI 里程碑。）

**CI 包含兩個不同的步驟，且第二個步驟是僅限人工操作的** — 請明確指出兩者：
1. **撰寫 CI 工作流程檔案**（例如 `.github/workflows/*.yml`、`.gitlab-ci.yml`、Azure Pipelines）以執行閘門。**代理程式可以做到這一點。**
2. **使該工作流程成為*強制執行的*閘門** — 亦即**必要狀態檢查 / 分支保護規則 / 合併請求核准規則**。這是手動的平台 UI / 管理員步驟（GitHub → Settings → Branches；GitLab 受保護的分支 + MR 設定等），**代理程式無法執行。** 在人員對其進行設定之前，CI 會在 PR 上*執行*，但**不會** *阻擋*合併。

因為步驟 2 超出代理程式的範圍，**每個建立 CI 的計劃都必須將其作為明確的使用者行動項目發出**（在該階段的結束條件中，以及在「§9 利益關係人需要決定的開放問題/決策」中），以便讀者知道在他們開啟它之前，此閘門不是自我強制執行的。

**Phase 2 的輸出：** 針對每個元件 — 可行性探討結果、選擇的策略（A 或 B）、定位的可測試性里程碑、前/後標記，以及具名殘留風險的目標安全階梯階級 — **加上全計劃範圍的 CI 里程碑**（哪一個階段建立 CI）以及強制執行是一個人員手動步驟的提醒。下游的每件事都取決於這些。


#### Phase 2.5: 每個階段都需進行紅隊演練，以核對危險目錄（強制）

上述的策略鷹架（可測試性、安全階梯、CI 里程碑）是必要但**不足夠的**。現代化計劃也常在*戰術性、生態系統可預測的*危害上失敗 — 不完整的隔離集、被遺忘的命名空間 codemod、導致基礎映像檔落後的執行階段升級、被當作映像檔升級的具狀態資料儲存庫主版本更新、捨棄匿名路由類別的邊緣重寫。這些非常頻繁地發生，以至於它們擁有自己的目錄：**`references/migration-hazards.md`** (H1–H8)。

**規則：在實作任何階段之前，請針對 `references/migration-hazards.md` 中的每個危害對該階段計劃進行紅隊演練。** 對於每個危害，詢問「此階段是否會觸發它？」，針對真實的儲存庫執行該危害的偵測探針，並在撰寫程式碼**之前**將其計劃行動併入該階段的任務/結束條件中。在規劃中捕獲的危害是任務；在審查中捕獲的是重工；在生產環境中捕獲的則是事件。請記錄您檢查並**清除 (cleared)** 的危害，而不僅僅是觸發的危害 — 靜默跳過與遺漏是無法區分的。

這是獨立評論機制 (independent-critic pass) 的制度化形式。請透過走訪目錄來自行執行此操作，並且 — 如果有獨立的第二意見（例如橡皮鴨 / 評論代理程式）可用時 — 請使用它：在 PiggyMetrics 參考執行中，此類審查在**每個階段都捕獲了阻擋性的計劃差距**，且每個差距都是 H1–H8 類別的實例。將結果寫入每個階段的「已做決策」和結束條件中，好讓執行代理程式能承接已清除的檢查清單，而不是重新發現危害。


#### Phase 3: 推薦目標架構

**決策框架 — 始終按以下順序評估：**

針對每個元件/依賴項，從破壞性最小到破壞性最大依次評估這些級別。在解決問題的第一個級別停止：

1. **就地升級 (Upgrade in place)** — 您能否提高主版本並修正中斷處？
   （例如，Node 14→22、Python 2→3、React 16→19、Rails 5→7）。這是預設答案。只有在升級路徑官方不支援、重大變更範圍太廣以至於超過重寫成本，或者框架本身被廢棄/EOL 時，才拒絕此選項。

2. **更換依賴項 (Swap the dependency)** — 如果特定的函式庫已停用但模式運作良好，您能否更換為具有相同介面外觀的已維護替代方案？
   （例如，Moment.js → date-fns、Request → got/fetch、CoffeeScript → TypeScript 搭配 decaffeinate）。在存在直接替換或機械式遷移工具時，偏好使用它們。

3. **包裝/適配 (Wrap/adapt)（絞殺者無花果 / Strangler Fig）** — 如果元件的介面健全但內部無法維護，您能否在其前端放置一個乾淨的介面，並在其後方逐步替換內部實作？當耦合是可控的且系統可以與新舊元件共存執行時，此方法有效。

4. **重寫 (Rewrite)** — 僅在以下情況下使用：框架被廢棄且無升級路徑、架構根本無法支援所需的新功能（例如，必須變為即時的純同步設計），或者誠實的估算顯示升級成本超過重寫成本。每項重寫建議都需要明確的理由 — 「它太舊了」是不夠的。

5. **移除 (Remove)** — 廢棄的程式碼、使用率為零的功能、被其他系統取代的已棄用功能。在建議移除之前驗證使用情況聲明。

**偏向保守。** 最便宜的遷移是您不做的遷移。
每一級的干擾都會增加風險、時間線和成本。在較舊框架上運作的系統勝過在新框架上只完成一半的重寫。

**機械式遷移附加條件（適用於建議為升級、更換或移除的任何時候）。** 這些是首稿計劃中經常遺漏的目標架構決策部分；每一項都對應到 `references/migration-hazards.md` 中的一個危害類別：

- **遞移隔離完整性 (Transitive-quarantine completeness) (H1)。** 當移除/切換整個依賴項家族時，請在**所有**資訊清單中搜尋 (grep) 該已失效的 Artifact，並列出*完整*的依賴對象集，這些對象必須被移動或隔離 — 而不是您起初想到的子集。明確說明切換後的建構目標。
- **框架主版本 Codemods (H2)。** **主版本**的升級會帶來與您的邏輯無關的機械式重大變更 — 命名空間重新命名、移除已棄用的 API、變更**測試引擎**、變更設定金鑰。將每一項列為其自己的任務，並且在大程式碼樹上，偏好使用生態系統的自動化**配方/codemod 工具**（OpenRewrite, jscodeshift/react-codemod, `2to3`/`pyupgrade` 等），而非手動編輯。不要忘記測試框架的遷移 — 這是最常被遺忘的一個。
- **執行階段與部署同步 (Runtime↔deployment lockstep) (H3)。** 如果建議提升了語言/執行階段的**主版本**，則系統*執行*（而非僅建構）時固定執行階段的每個位置 — 基礎映像檔、`docker-compose` 建構上下文、CI 執行器映像檔、`engines`/`.nvmrc`/`.tool-versions`、無伺服器執行階段 — 必須在**同一個階段**移動。建構執行階段與執行執行階段絕不能跨階段邊界產生偏差，否則應用程式建構為綠燈，然後執行失敗。

產生一個**目標架構 (Target Architecture)** 章節，其中包含：

- **推薦的技術堆疊**，並說明每個選擇的理由（框架、語言、執行階段、資料庫、基礎設施）
- **架構模式**建議（單體應用程式 → 模組化單體應用程式 → 服務、事件驅動、無伺服器等）並說明理由
- **保留與捨棄的內容** — 使用上述決策框架明確分類每個主要元件：
  - ✅ 照原樣保留（仍屬現代/足夠）
  - ⬆️ 就地升級（提高版本，修正中斷處）
  - 🔀 更換依賴項（用已維護的等效項目替換已失效的函式庫）
  - 🔄 包裝/適配（新增一個介面層，逐步替換內部實作）
  - 🔁 重寫（根本不相容 — 需要合理理由）
  - 🗑️ 移除（廢棄程式碼、已棄用功能、已被取代）

針對每項建議，撰寫一個行內 ADR：

```markdown
#### ADR: [決策標題]
- **Context:** [為什麼需要此決策]
- **Decision:** [我們選擇了什麼，以及它屬於決策框架的哪一個級別]
- **Alternatives considered:** [還評估了哪些其他方案，以及它們被捨棄的原因]
- **Consequences:** [接受的折衷方案]
```


#### Phase 4: 按特徵遷移分析

針對架構文件中識別出的每個主要功能/領域，產生一個章節，涵蓋：

1. **目前實作** — 包含檔案參考的單一段落摘要
2. **遷移策略** — 針對此元件在 Phase 2 所做的 A/B 選擇（先凍結再提升 vs. 先建立據點/步進骨架），加上具體戰術：絞殺者模式 (Strangler Fig)、大爆炸重寫 (Big Bang rewrite)、遷移與提升 (Lift & Shift)、漸進式重構 (Incremental Refactor)，或原地保留 (Leave In Place)
3. **可測試性狀態** — 此元件的**可測試性里程碑**落在何處，以及其**安全階梯階級**（L0–L4）並註明殘留風險
4. **依賴關係與耦合** — 還有哪些其他部分會因此中斷或必須隨之移動
5. **工作量估算** — T-shirt 尺寸（XS/S/M/L/XL）並說明理由
6. **風險評估** — 可能出錯之處、資料遷移考量、使用者影響
7. **驗證條件** — 您如何得知遷移已成功（針對所選預測源/接縫合約表示，而非針對尚未存活的測試套件）


#### Phase 5: 分階段實施計劃

產生一個有序的、分階段的計劃。每個階段都應該是可以獨立部署的（沒有無法在生產環境中執行的半遷移狀態）。

**階段閘門是可感知體制的（適用於每個階段）。** 在分階段計劃的頂部聲明此規則，並在每個階段的結束條件中遵守它。在驗證與結束條件通過之前，階段**不算完成**。這些條件必須是：(a) **客觀可驗證的**；(b) 在下一個階段開始之前**實際執行並記錄**；以及 (c) **受控的** — 在證明滿足階段 N 的條件之前，不要推進到階段 N+1。**但哪些條件有效，取決於該元件所在的體制 (Phase 2c)：**

- **可測試性之後（「明亮」）階段：** 結束條件是可執行的命令 / 綠燈 CI — **該階段的分支/PR 上的綠燈 CI 是具權威性的訊號。**
- **可測試性之前（「黑暗」）階段：** 元件尚未存活，因此綠燈 CI 測試閘門是**類別錯誤**。結束條件來自可達成的安全階梯階級 — 擷取到的接縫/預測源快照、已證實存在的可回復性鷹架、已通過的冒煙測試清單、記錄的審查。**不要**因為元件還無法執行的測試套件而阻擋黑暗體制的階段。

向使用者報告任何通過/失敗狀態不明的階段，而非假設其已通過。

**第一階段建立最大*可達成*安全性 — 而非固定的「綠燈舊有閘門」。** 請勿硬編碼「復活所有內容並讓 CI 變綠燈」。根據 Phase 2 的輸出，針對各元件塑造第一階段：

- **策略 A（先凍結再提升），元件是可復活的：** 第一階段*就是*經典的安全網與基準線 — 純粹是累加性的，無行為變更：
  - 在鎖定目前可觀察行為（協定追蹤、API 快照、輸出黃金檔）的**接縫 (seams)** 處進行**特性描述 / 黃金檔 / 合約測試**。
  - 連接 **CI + Lint + 格式化工具**並呈綠燈（作為後續明亮體制階段的權威性閘門）。
  - **固定依賴項 / 提交鎖定檔案**並記錄一個已知的良性基準線（版本、工具鏈、它今天如何建構與執行）。
  - **「證明網會失敗。」** 故意變更程式碼（例如，變更一個協定代碼或傳回值），確認新的測試變為**紅燈**，然後回復。一個從不失敗的安全網無法證明任何事情 — 請證明它是有作用的。
- **策略 B（先建立據點/步進骨架），元件已失效：** 第一階段是**「透過最小切片達到可測試性里程碑」** — 在現代技術堆疊上驅動最薄的端對端路徑，直到它建構、啟動並執行 ≥1 個真實測試呈綠燈，並從舊系統中擷取接縫/預測源合約來進行驗證。**不要試圖網住屍體。** 其結束條件是可達成的安全階梯階級並指明殘餘風險 — **被允許的降級不是失敗。**

只有在滿足第一階段的條件後（無論在哪個階級），後續的基礎設施和功能階段才會開始。

將每個階段結構化為：

```markdown
### Phase N: [名稱] (T-shirt 尺寸: M)

**目標：** [一句話]
**體制：** [可測試性之前 ("dark") | 可測試性之後 ("lit")] — 針對各個元件
**安全階級：** [L0–L4，若低於 L4 則指明殘留風險]
**前置條件：** [哪些階段必須先完成]
**時程估算：** [相對時間，而非日曆時間 — 例如「2-4 個衝刺」]


#### 任務
| ID | 任務 | 元件 | 受阻於 |
|----|------|-----------|------------|
| N.1 | ... | ... | — |
| N.2 | ... | ... | N.1 |


#### 風險與緩解措施
- **風險：** ... → **緩解措施：** ...


#### 已做出的決定
- [此階段所依賴的每個子決策，都已在此處解決並記錄，因此實作此階段時無需進一步的使用者輸入。對於任何被裁剪的內容，明確聲明「已捨棄」或「已延後」。]


#### 驗證與退出標準（完成的定義）
- [ ] [適用於該體制的條件。對於明亮 (LIT) 階段：可執行的命令 / 綠燈 CI 檢查，引用來自「命令與驗證清單」的實際命令。對於黑暗 (DARK) 階段：可達成的安全階梯階級的證據 — 擷取到的接縫/預測源快照、證實存在的可回復性、通過的冒煙測試清單 — 絕非該元件尚無法執行的綠燈測試套件。]
- [ ] [每當變更旨在保留行為時，納入針對所選預測源的等效性/特性描述檢查。]
- [ ] [如果是純粹累加性的，請宣告它：「未變更任何行為/依賴關係/邏輯。」]
- [ ] [如果低於 L4 結束，請說明殘留風險以及哪一個後續階段會關閉它（例如「用戶端測試閘門延後至用戶端遷移階段」）。]
```

**排序原則：**
- **首先定位可測試性里程碑**，針對個別元件 — 儘早安排可達到此里程碑的最薄切片，而不是在開始時要求閘門卻將「可測試」延後到最後。
- **在元件跨越其自身的可測試性界線之前，絕不要要求對該元件設定自動化測試閘門。**
- **在藍圖中指明 CI 里程碑。** 明確指出哪一個階段建立 CI（第一個*明亮*階段，在可測試性里程碑之時/之後），好讓讀者永遠不必猜測 CI 何時到達 — 並標示*強制執行*它（必要狀態檢查 / 分支保護）是一個手動的人工步驟，而非代理程式的任務。
- **先建立第一階段的安全性，再變更行為** — 在修改行為之前，先確立最高可達成的階級（不一定是完全綠燈的閘門）。
- 接下來是基礎設施和橫切關注點（驗證、設定、記錄、CI/CD）
- 接下來是最高風險或最高價值的核心功能（儘早證明模式可行）
- 最後是低風險、低耦合的功能（容易平行處理的簡單獲勝點）
- 資料遷移有其專屬的階段，並附帶回滾計劃


#### Phase 6: 執行治理

現代化計劃失敗在於執行，而非在於紙面。請加入確保每個階段落實與可回復的治理機制（適用於任何生態系統的通用規則）：

- **每個階段一個分支。** 絕不要將階段工作直接提交到預設分支。每個階段開啟一個 PR；針對**明亮體制階段**，讓 PR 上的 CI 作為合併前的權威性綠燈訊號。對於黑暗體制階段，PR 仍然攜帶該安全階級的證據（擷取的合約、冒煙測試結果、可回復性鷹架）。
- **在下一個階段開始前合併至主幹 (trunk)；絕不堆疊 (H7)。** 每個階段分支都是**自主幹**切出，且其 PR 在**下一個階段開始前合併至主幹**。絕不要將某個階段分支或其 PR 建立在*同層級*階段分支之上 — 這會導致 PR 互相合併，主幹默默落後好幾個階段，並迫使後續階段陷入不得不違反無堆疊規則的困境。在開始一個階段之前，請驗證前一個階段已合併至主幹，且在建立分支時 `git log origin/<trunk>..HEAD` 是空的。**如果受控的堆疊確實無法避免，則僅在需要對帳 PR**（將堆疊內容合併至主幹）**並附帶明確的殘留風險說明**的情況下才被允許 — 絕不能作為預設的默許行為。
- **確認主幹名稱；停用舊有的預設分支。** 固定儲存庫的真實主幹（例如 `main`），如果存在第二個預設性質的分支（例如 `master`），請將其標記為「僅限歷史記錄 — 請勿以此為目標」，以防階段工作誤入錯誤的分支。
- **閘門需與體制相符。** 明亮階段在綠燈 CI 上推進；黑暗階段在指明殘留風險的可達成安全階梯階級上推進 — 而不是憑藉「看起來完成了」的判斷，也不是憑藉元件尚無法執行的測試套件。
- **在 CI 里程碑建立 CI，並將強制執行移交給人員。** 指明引進 CI 的階段（第一個明亮階段）。區分兩個步驟：撰寫工作流程檔案是代理程式可做的；使其成為**強制執行的必要狀態檢查 / 分支保護（或 MR 核准）規則是人員手動的平台 UI 步驟，代理程式無法執行。** 在人員對其進行設定之前，CI 會*執行*但不會*阻擋*合併 — 因此請將「啟用分支保護 / 必要檢查」作為明確的使用者行動項目記錄（在 CI 階段的結束條件中以及在 §9 利益關係人決策中），絕不能當作已完成的代理程式任務。
- **保留介面且可獨立部署。** 每個階段都讓系統保持可執行狀態，因此回滾 = 重新部署前一個版本。偏好保留介面的重寫（在穩定介面後包裝內部實作），並使用接縫/預測源合約（若存在特性描述/黃金檔測試）證明其等效性。
- **動態計劃。** 當滿足每個階段的結束條件時，更新 `MODERNIZATION_PLAN.md` 狀態標記 — ✅ complete / ⏭️ descoped / 🗑️ dropped — 記錄實作期間所做的決策，並在元件跨越其可測試性界線時**更新安全階梯階級 / 殘留風險註冊表**。
- **同時保持可執行文件的時效性 (H8)。** 動態計劃的紀律不僅限於計劃檔案本身。任何變更**拓撲**（模組/反應器成員資格、移除的服務、重新命名的分支、變更的命令、新增/移除的端點）的階段，都必須在**同一個 PR 中**更新 `.github/copilot-instructions.md`、`README` 以及任何模組/拓撲列表。將這些文件更新視為該階段「已完成的定義」的一部分，而不是後續工作 — 過時的「隔離模組」列表或硬編碼的舊分支連結會主動誤導下一個代理程式/人員。
- **在實作每個階段之前對其進行紅隊演練 (Phase 2.5)。** 對照 `references/migration-hazards.md` (H1–H8) 走訪階段計劃，並在撰寫程式碼之前將修正方案併入任務和結束條件中。將已清除的檢查清單記錄在該階段的「已做決策」中。
- **發出可編輯的指令。** 根據 `references/copilot-instructions.template.md` 的範本，產生一個 `.github/copilot-instructions.md`（若缺法則建立 `.github/` 目錄） — 包含命令表格以及可感知體制的階段閘門與分支/PR 規則 — 好讓執行計劃的代理程式/人員能在 GitHub Copilot 自動載入的位置擁有標準命令和閘門。使用者可將其編輯為其自己的閘門。如果 `.github/copilot-instructions.md` 已經存在，請勿覆寫它 — 將命令/閘門/分支章節合併進去，或在旁邊寫入一個產生的 `.github/copilot-instructions.modernization.md` 並提示使用者合併。


#### Phase 7: 遷移安全網

記錄所需的運作護欄：

- **功能旗標** — 哪些內容加上旗標，以及新舊系統如何共存
- **資料遷移策略** — 綱要變更、回填、雙寫期間。對於使用持續磁碟區的**具狀態資料儲存庫主版本升級** (H5)，引擎映像檔/標記的更新**不是**遷移：請選擇並記錄一條路徑 — 附帶步驟 + 回滾的循序/功能相容性版本升級，**或**在資料為暫時/展示用途時進行明確的**破壞性重設**（刪除磁碟區、重新植入資料） — 作為審慎決策說明，並保留先前主版本的映像檔以便回滾。（例如，在現有的 `/data/db` 磁碟區上進行 Mongo 3→7 升級，展示用途需要 `down -v` + 重新植入，而真實資料則需要分階段 FCV 升級。）
- **回滾計劃** — 每個階段中，如果發生失敗該如何還原
- **過渡期不安全狀態註冊表 (H6)** — 為了在遷移中期保持系統可執行而引入的每個刻意削弱的狀態（允許所有的墊片、停用 CSRF、開放的端點、預留位置秘密資訊），並列出：為什麼需要它、**關閉**它的階段、殘留風險，以及給審查人員的「在階段 N 之前以此設計」說明。盡可能收緊每個削弱的範圍（例如，允許所有的鏈仍應明確停用表單登入/HTTP-Basic）。預先宣告這些內容，可將 scanner/reviewer 引起的重複 churn 轉化為已追蹤的決策。
- **預測源與接縫合約** — 哪一個運行中的執行個體 / 記錄的 I/O / 黃金快照（或在無外部參考可用或不允許時的**自我凍結黃金母片**）作為行為參考，以及如何對照它們進行新工作差異比對
- **測試策略** — 需要哪些新的測試覆蓋範圍（當元件跨越其可測試性界線時，在*目標*技術堆疊上建構），以及哪些內容保持隔離
- **可觀測性** — 證明新系統與舊系統相符的指標/警示


### 輸出結構

產生一個名為 `MODERNIZATION_PLAN.md` 的主要 Markdown 檔案，其中包含：

1. 執行摘要（一個段落：內容、原因、概略範圍）
2. 目前狀態評估（來自架構文件）
3. **可行性探討結果與策略** — 針對個別元件：探討發現、A/B 遷移策略、定位的**可測試性里程碑**、目標**安全階梯階級**，以及針對低於 L4 的任何項目的**殘留風險註冊表**；加上全計劃範圍的 **CI 里程碑**（哪一個階段建立 CI），並附註*強制執行*它是人員手動的步驟
4. 包含 ADR 的目標架構
5. 個別功能遷移分析（附帶可測試性狀態 + 安全階級）
6. 分階段實作計劃 — 第一階段建立最高*可達成*的安全階級（而非固定的綠燈舊有閘門）；每個階段都標示了可測試性之前/之後，並攜帶與體制相符的受控結束條件；**明確指出 CI 里程碑階段**，並將「啟用分支保護 / 必要狀態檢查」列為人員手動的行動項目
7. 執行治理（每個階段一個分支、感知體制的閘門、動態計劃狀態）
8. 遷移安全網（預測源與接縫合約、旗標、回滾、可觀測性）
9. 利益關係人需要決定的開放問題/決策 — 包括代理程式無法執行的任何手動平台設定交接（例如啟用必要的狀態檢查 / 分支保護，使 CI 成為強制執行的閘門）

同時，根據 `references/copilot-instructions.template.md` 範本，發出一個附帶的 **`.github/copilot-instructions.md`**（GitHub Copilot 自動載入的路徑；若缺法則建立 `.github/`），填入專案的標準命令和階段閘門 / 分支規則，供執行計劃的人員編輯與採用。絕不要覆寫已存在的 `.github/copilot-instructions.md` — 將其合併或在旁邊寫入一個同層級的 `.github/copilot-instructions.modernization.md` 並提示使用者。


### 公約

- **引用架構文件**來參考目前狀態，而不是重述所有內容。使此文件保持前瞻性。
- **展現明確的意見。** 做出具體的建議，而不是「您可以做 A 或 B」。說明您會挑選什麼以及原因。使用者可以覆寫。
- **務實控管範圍。** 企圖一次變更所有內容的現代化計劃是不切實際的。識別什麼是階段 1，什麼是「未來考量」。
- **在規劃期間解決每個階段內決策。** 階段計劃必須攜帶其實作所依賴的所有子決策，並已在規劃中解決和記錄（「已做決策」區塊），以便執行階段時無需返回詢問使用者。對於任何被裁剪的內容，明確聲明**「已捨棄」與「已延後」** — 它們是不同的。將 `[DECISION NEEDED]` 保留給會阻擋階段的真正*利益關係人/業務*選擇（預算、團隊大小、產品方向、時程壓力）；在規劃期間與使用者一起解決這些決策，而不是在可實作的階段中留下未決事項。
- **不要過度工程 (don't gold-plate)。** 如果某個元件在舊有的技術堆疊上運行良好且沒有維護負擔，「保持原樣」是一個有效的建議。
- **假定預設為失效狀態；讓安全性具有適應性。** 不要預先要求一個完全綠燈的舊有 CI 閘門。執行可行性探討，挑選最高可達成的安全階梯階級，並將**降級視為有幸得到的結果**且指明殘留風險 — 絕非失敗。
- **在時間線上定位可測試性。** 每個計劃都必須指明每個元件的**可測試性里程碑**，且在元件跨越其自身的可測試性界線之前，絕不要要求對該元件設定自動化測試閘門。
- **指出 CI 里程碑，將強制執行交給人員。** 說明哪一個階段建立 CI（第一個明亮階段），並明確表示將 CI 轉為*強制執行的*必要檢查 / 分支保護規則是代理程式無法執行的手動平台步驟 — 將其呈現為使用者行動項目，而非已完成的任務。
- **網住接縫，而非屍體。** 將安全性錨定在外部可觀察的合約（協定、綱要、有線/檔案格式、端點），並搭配分級的預測源（執行中的實例 → 記錄的 I/O → 程式碼作為規格 → 不允許或不存在外部參考時的**自我凍結黃金母片**）。不要投入多個衝刺復活已失效的工具鏈，只為了執行會在重寫時被刪除的測試。
- **保持通用性。** 用*專案的*套件管理器 / 測試執行器 / CI 來表示命令與閘門 — 絕不假設特定的技術堆疊。
- **在實作前對每個階段進行紅隊演練 (H1–H8)。** 對照 `references/migration-hazards.md` 走訪每個階段計劃，並先將修正方案併入任務/結束條件中 — 在規劃中捕獲的危害是任務；在審查中是重工；在生產環境中則是事件。這些戰術性危害（不完整隔離、框架 codemods、執行階段/基礎映像檔同步、路由類別列舉、資料儲存庫升級路徑、過渡期不安全狀態干擾、堆疊 PR/主幹偏差、動態文件偏差）是單憑策略性鷹架容易遺漏的。
