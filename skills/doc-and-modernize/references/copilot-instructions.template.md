## 階段閾值（ regime 感知；適用於每個階段）

在 `MODERNIZATION_PLAN.md` 中的**驗證與結束標準**通過之前，該階段**尚未完成**。這些標準必須是：

1. **客觀可驗證的** — 可執行的命令 / 擷取的辨識證據，而非主觀判斷。
2. **實際執行並記錄的** — 在開始下一個階段之前，執行它們並在計畫中記錄結果。
3. **閘口控制的** — 在階段 N 的標準被證實滿足之前，**請勿**前進到階段 N+1。

**哪些標準有效取決於元件的 regime：**

- **測試後 ("光") 階段：** 結束於可執行的命令 / **該分支/PR 上的綠色 CI —— 最權威的信號。** 共享基準：`<Lint 命令>` 和 `<測試命令>` 必須通過；如果變更觸及了用戶端 / 傳輸層 / 協定 / 任何公共介面，則相關的端對端和合約/特徵測試也必須保持綠色。
- **測試前 ("暗") 階段：** 該元件尚未啟用，因此綠色 CI 測試閘口是一個**分類錯誤**。請改為結束於該階段可達到的**安全階梯階層**——擷取的縫隙/預測指標快照、經證實存在的還原支架、通過的冒煙檢查表、記錄的審閱。**不要因為測試套件還無法執行而阻擋暗階段。** 降級到較低階（並命名殘留風險）是一個有效的結束，而不是失敗。

**在元件跨越其自身的測試可用性里程碑之前，絕不要求該元件的自動化測試閘口。** 向使用者報告通過/失敗狀態不明的任何階段，而不是假設其已通過。

## 決策

- 當規劃某個階段時，**所有子決策必須在計畫中得到解決和記錄**，以便實作時不需要進一步的使用者輸入。
- 對於任何裁剪掉的內容，請明確說明是**「捨棄 (dropped)」**還是**「延後 (deferred)」**——不要使其保持模糊。
- 當每個階段的結束標準滿足時，更新 `MODERNIZATION_PLAN.md` 的狀態標記（✅ 完成 / ⏭️ 取消範圍 / 🗑️ 捨棄），並在元件跨越其測試可用性里程碑時，**更新安全階梯階層 / 殘留風險登記表**。

## 分支與 PR

每個階段都在它**自己的分支**上進行開發——絕不要將階段工作直接提交到預設分支（`<預設分支>`）。在階段開始時建立一個新分支（例如 `phase-N-<簡稱>`）。一旦該階段的結束標準被滿足並記錄，推送分支並向預設分支發起 PR。對於**測試後 ("光")** 階段，在合併前讓 PR 上的 CI 成為權威的綠色信號。對於**測試前 ("暗")** 階段，PR 則攜帶了可達到的安全階梯階層的證據（擷取的合約、冒煙結果、還原支架）並命名殘留風險。

**在下一個階段之前合併到主幹；絕不堆疊。** 從**主幹（`<預設分支>`）**切出每個階段分支，並**在開始下一個階段之前將其 PR 合併到主幹。** 絕不要將階段分支或其 PR 建構在*同儕* `phase-N` 分支之上——堆疊的階段 PR 會互相合併，主幹會靜默落後好幾個階段，導致後面的階段陷入困境。在開始某個階段之前，請驗證前一個階段已合併，且在建立分支時 `git log origin/<預設分支>..HEAD` 為空。**只有在確實無法避免的情況下才允許受控堆疊——且必須有將堆疊合併到主幹的協調 PR，加上明確的殘留風險說明。**

> **確認主幹名稱；停用舊版預設名稱。** 釘選真實的主幹（`<預設分支>`）。如果存在第二個類似預設的分支（例如與 `main` 並存的舊版 `master`），請將其標記為「僅限歷史記錄 — 請勿作為目標」，以防階段工作誤入錯誤的分支。

**保持可執行文件與程式碼同步啟用。** 改變**拓撲結構**的任何階段——模組/反應器成員資格、移除的服務、重新命名的分支、變更的命令、新/移除的端點——必須**在同一個 PR 中**更新**此檔案**、`README` 和任何模組/拓撲列表。過期的模組列表或寫死的舊分支連結會主動誤導下一個代理程式/人類。

**登記過渡期的不安全狀態。** 如果某個階段為了保持可執行而刻意引入暫時的薄弱狀態（允許全部的 shim、停用 CSRF、開放的端點、預留位置金鑰），請將其與**關閉**該狀態的階段一起記錄，並加上「在階段 N 之前以此設計」的備註，並儘可能縮減其範圍——這可以預防掃描器/審閱者的重複擾動。

> **CI 強制執行是一個僅限人工的手動步驟。** 代理程式可以編寫並執行 CI 工作流程，但將其設為**必要狀態檢查 / 分支保護規則**以使其真正*阻擋*合併，是代理程式無法執行的平台 UI/管理員動作（`<GitHub → Settings → Branches; GitLab protected branches; 等等>`）。在您設定它之前，請將「綠色 CI 是權威的」視為您手動強制執行的慣例。在 CI 里程碑階段或之後執行此操作一次。
