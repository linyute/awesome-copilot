# 移轉風險目錄

一個通用的、**與技術堆疊無關**的移轉風險目錄，這些風險在*各個生態系統中都是可以預測的*，但在初版現代化計畫中經常被遺漏。每個階段計畫在**實作前都應該對照此清單進行紅隊演練（Red-teamed）**（參見現代化模式在實作前的紅隊步驟）。

每個風險都寫為：**觸發條件**（何時適用） → **風險**（什麼會損壞） → **偵測探針**（您現在可以執行的具體檢查） → **計畫行動**（要加入該階段的工作） → **範例**（各生態系統的說明，僅供參考）。

> 這些是*風險類別*，而不是針對某個堆疊的固定檢核表。範例（Java/Spring/Mongo/Docker 等）是取自真實執行的說明——請將每個類別對應到您實際移轉的任何生態系統。如果某個風險不適用，請明確說明，而不是靜默略過。

---

## 如何使用此目錄

在實作**任何**階段之前：

1. 檢視下方的每個風險，並詢問「此階段是否會觸發它？」
2. 對於每個被觸發的風險，對真實的存放庫執行**偵測探針**。
3. 在撰寫程式碼*之前*，將**計畫行動**納入該階段的工作和結束標準中——在規劃中發現的風險是一項工作；在審閱中發現的是重做；在生產環境中發現的則是事故。
4. 記錄您檢查過且已排除的風險，而不僅是觸發的風險——靜默略過與遺漏是無法區分的。

根據經驗（PiggyMetrics 執行，階段 0-6），獨立的評論者在**每個階段中都發現了阻礙計畫的缺口**，且每個缺口都是下方類別之一的實例。執行此目錄就是將該評論者的機制制度化。

## H1 — 不完整的隔離 / 移除集

**觸發條件：** 您正在移除、隔離或切換整個相依性家族（一個失效的框架、一個已移除的執行階段 API、一個 EOL 函式庫集），且只有部分模組被修改。

**風險：** 您*未*列出的模組仍然遞迴相依於已移除的成品（Artifact），因此預設的建構/測試目標在切換後會失敗——通常發生在計畫從未提及的模組中。

**偵測探針：** 在**所有**資訊清單（Manifest）中搜尋（Grep）失效的成品/座標，而不僅是您正在編輯的清單（`pom.xml`/`build.gradle`, `package.json`, `*.csproj`, `requirements*.txt`/`pyproject.toml`, `go.mod` 等）。每個搜尋命中都是一個在相同階段必須移動或隔離的模組。

**計畫行動：** 預先列出失效成品相依項的**完整遞迴集**，並將每一項都列入該階段的反應器（Reactor）/隔離決策中。明確說明切換後的建構目標（哪些模組包含在內，哪些被隔離）。

**範例：**
- *JVM:* Spring Cloud 2023 移除了 Netflix Hystrix/Turbine；即使計畫僅隔離了 `gateway` + `auth-service`，`monitoring` 和 `turbine-stream-service` 仍然拉取了 `netflix-hystrix-dashboard` / `netflix-turbine-stream` 並破壞了 `mvn verify`。
- *JS:* 捨棄 `request` —— 搜尋每個 `require('request')` / `from 'request'`，而不僅是您記住的那個服務。
- *Python:* 移除 C 延伸模組函式庫 —— 檢查每個 `import` 點和額外的項目。

## H2 — 框架主要版本的機械式程式碼修改（命名空間 / API / 測試引擎）

**觸發條件：** 框架、語言或平台的**主要（Major）**版本升級（而非修補程式/次要版本升級）。主要版本升級會帶來與您自身邏輯無關的*機械式*重大變更。

**風險：** 計畫將「升級版本」視為單一工作，而遺漏了升級所強制的整棵樹的機械式移轉——命名空間重新命名、已移除的棄用 API、已變更的測試執行器/引擎、已變更的設定鍵——導致建構甚至無法編譯。

**偵測探針：** 閱讀目標版本的**移轉/升級指南**，並識別已知的重大變更處方。在樹狀結構中搜尋舊的命名空間 / API 介面以衡量變更規模。檢查是否存在**生態系統程式碼修改/處方工具（Codemod/Recipe tool）**。

**計畫行動：** 將每個機械式移轉列為其**專屬的明確工作**，並在可靠的地方，偏好使用生態系統的自動化處方工具，而不是手動編輯。審慎地決定使用工具還是手動（小型樹狀結構手動可能更安全；大型樹狀結構需要工具）。將測試框架移轉納入清單中——這經常被遺忘。

**範例：**
- *Spring Boot 2→3:* `javax.* → jakarta.*` 命名空間移轉**以及** JUnit 4→5（加入 `junit-vintage-engine` 以保持 JUnit 4 測試執行，移轉 `OutputCapture` 樣式的 Jupiter 專用情況）。工具：**OpenRewrite**。
- *JS:* CommonJS→ESM，或 React 類別→hooks。工具：**jscodeshift**, `react-codemod`。
- *Python:* `2to3`/`pyupgrade`；`async` API 重新命名。
- *.NET:* `Newtonsoft`→`System.Text.Json`，`HttpClient` 模式。

## H3 — 執行階段升級未與部署成品同步

**觸發條件：** 您升級了用於*建構*應用程式的**語言/執行階段主要版本**（JDK, Node, Python, Ruby, Go）。

**風險：** 建構現在會為新的執行階段發行成品，但**部署/執行介面**仍然釘選舊的版本——基礎映像檔、CI 執行器版本、buildpack、`engines` 欄位、lambda 執行階段——導致應用程式建構為綠燈，但接著*執行*失敗（位元組碼/ABI/語法不相容），通常僅在冒煙/e2e 步驟中顯現，或在生產環境中靜默失敗。

**偵測探針：** 清點釘選執行階段版本的每個地方：`Dockerfile`/`Containerfile` 基礎映像檔、`docker-compose*` 建構上下文、CI 工作流程 `setup-*`/執行器映像檔、`.tool-versions`/`.nvmrc`/`runtime.txt`、`package.json` 中的 `engines`、無伺服器執行階段鍵。將每一項與新的目標進行交叉比對。

**計畫行動：** 新增一個明確的工作，在與建構升級相同的階段中移動**每個**部署/執行階段釘選。絕不要讓建構執行階段和執行期執行階段漂移跨越階段邊界。驗證冒煙路徑是否實際建構/執行了新的成品（且不建構被隔離的模組）。

**範例：**
- *JVM:* 原始碼建構的服務發行 JDK 21 位元組碼，但每個 `Dockerfile` 仍然是 `FROM java:8-jre` → 容器無法啟動；開發用 compose 仍然*建構*了被隔離的 `auth-service`。
- *Node:* CI 升級到 Node 22，但 Docker 映像檔仍然是 `node:14-alpine`。
- *Python:* venv 在 3.12 上，但容器基礎映像檔是 `python:3.7`。

## H4 — 流量流量 / 路由類別列舉缺口（邊緣與驗證重寫）

**觸發條件：** 重寫邊緣/閘道、驗證/身分識別層，或任何請求路由/安全性邊界。

**風險：** 計畫僅釘選了顯而易見的已驗證正常路徑（Happy path），而靜默地丟失了一個或多個*其他*流量類別——匿名流程（註冊、密碼重設、公共資產、健康狀況/指標收集）、服務對服務呼叫、webhook——導致這些在切換時中斷。

**偵測探針：** 列舉邊界必須服務的**每個路由類別**：(1) 匿名/公共、(2) 終端使用者已驗證、(3) 服務對服務（機器權杖/範圍）、(4) 基礎架構/可觀測性（健康狀況、就緒狀態、指標收集）、(5) webhook/回呼。對於每一項，確認新的設定有明確的規則。

**計畫行動：** 將路由類別列舉作為該階段的明確產出（路由類別 → 新規則 → 範圍的表格）。指出掃描器/安全性預期基礎架構端點（例如指標收集）保持可連線。為服務對服務（s2s）呼叫決定權杖/範圍的精細度（窄範圍，而非通用的「伺服器」授權）。

**範例：**
- *BFF/OAuth2:* SCG BFF + TokenRelay 涵蓋了登入，但遺漏了**匿名註冊 POST + 註冊資產**，且 `/uaa/users` 是 `permitAll` 而不是需要窄範圍的 `user.write` 服務範圍。
- *指標:* 資源伺服器安全性必須保持 `/actuator/prometheus` 開放，否則收集會靜默中斷。

## H5 — 狀態化資料儲存庫主要版本升級被視為映像檔/版本升級

**觸發條件：** 升級具有**持續性資料/磁碟卷 (volumes)** 的狀態化支援服務（資料庫、快取、訊息代理、搜尋索引）的**主要版本**。

**風險：** 引擎主要版本通常無法在不經過循序/FCV（Feature Compatibility Version）升級路徑的情況下，在舊主要版本的磁碟資料上啟動。天真的映像檔升級會導致無法啟動、行為因本地磁碟卷狀態而異，或者資料損毀。

**偵測探針：** 檢查儲存庫是否具有持續性磁碟卷/資料目錄，並閱讀該引擎支援的升級路徑（循序主要版本？功能相容性版本步驟？僅限傾印/還原？）。確定該資料是生產/耐久的還是暫時/示範的。

**計畫行動：** 選擇並記錄**一條**路徑：(a) 包含步驟 + 還原的循序/FCV 移轉，或 (b) 當資料是暫時/示範資料時，進行**明確的破壞性重設**（捨棄磁碟卷、重新植入資料）——將此作為決策而非意外記錄。將此納入移轉安全網中並附帶還原計畫。

**範例：**
- *MongoDB 3→7:* 在現有的匿名 `/data/db` 磁碟卷上直接進行映像檔升級是不安全的；對於示範專案，祝福的路徑是 `docker compose down -v` + 透過原生 `initdb.d` 重新植入，並保留舊的 `mongo:3` 映像檔用於還原。
- *Postgres 主要版本:* 需要 `pg_upgrade` 或傾印/還原，而不能僅變更標籤。

## H6 — 過渡期的不安全狀態重複觸發掃描器與審閱者

**觸發條件：** 某個階段刻意引入暫時的、蓄意削弱的安全狀態以保持系統在移轉中期可執行——允許全部的安全性 shim、停用 CSRF、開放的端點、寫死的預留位置金鑰、停用的檢查。

**風險：** 每個程式碼掃描器（CodeQL, Dependabot, SAST）和每個自動化審閱者都會在**每次審閱**中標記它，產生重複的噪音和重複的論證，且——更糟的是——原本旨在作為暫時性的 shim 由於沒有人追蹤其關閉階段而靜默地發行到生產環境。

**偵測探針：** 對於每個過渡期削弱，詢問：哪個掃描器會標記它，以及哪個後續階段會關閉它？削弱是否受到*最小*範圍的限制（例如允許全部的鏈仍應明確停用表單登入/HTTP-Basic，以便安全表面確切是所預期的）？

**計畫行動：** 維持一個**過渡期不安全狀態登記表**：每個 shim 都有 (a) 為什麼需要它、(b) 關閉它的階段、(c) 殘留風險、(d) 給審閱者的備註，說明這是「在階段 N 之前以此設計的」。儘可能縮減每個削弱的範圍。預先宣告可以將審閱擾動轉變為已解決、被追蹤的決策。

**範例：**
- 階段 5 允許全部的 `SecurityConfig` shim 在每個服務、每次審閱中都會觸發 CodeQL "CSRF disabled"；修正方法是明確停用表單登入/基本驗證，並註記「無狀態且允許全部的 shim，在階段 N 中關閉」。

## H7 — 堆疊 PR 陷阱與主幹分裂腦（分階段治理）

**觸發條件：** 多階段移轉，其中每個階段都相依於前一個階段，採用分支/階段的開發方式。

**風險：** 如果階段 N+1 的分支切自階段 N 的分支（而不是主幹），且階段 N 從未合併到主幹，PR 就會合併**到彼此之中**，主幹會靜默地落後好幾個階段，後面的階段被迫堆疊——產生了必須覆寫「無堆疊 PR」規則的困境。一個平行的失敗：具有**兩個類似預設分支**（`master` + `main`）的存放庫中，階段工作鎖定了錯誤的分支。

**偵測探針：** 在開始某個階段之前，驗證前一個階段的 PR 已**合併到主幹**，且新分支是**切自主幹**（在建立分支時 `git log origin/<主幹>..HEAD` 應為空）。確認存放庫的真實主幹名稱，且沒有鎖定舊版的預設分支。

**計畫行動：** 強制要求**在開始下一個階段之前，將每個階段的 PR 合併到主幹**。絕不要將階段分支/PR 建構在同儕階段分支上。如果確實受阻且控制下的堆疊是無法避免的，要求一個將堆疊合併到主幹的**協調 PR** 以及一個明確的**殘留風險說明**。釘選主幹名稱，並將任何舊版預設分支標記為「僅限歷史記錄 — 請勿作為目標」。

**範例：**
- PiggyMetrics 階段 3 和 4 被合併到彼此之中；`main` 卡在階段 2；`master`/`main` 分裂腦——只能在階段 5 必須打破無堆疊規則後，透過專用的 PR #6 進行協調。

## H8 — 活文件漂移（拓撲結構變更超出文件更新的速度）

**觸發條件：** 某個階段變更了系統的**拓撲結構（Topology）**——模組/反應器成員資格、移除的服務、重新命名的分支、變更的命令、新/移除的端點。

**風險：** 只有計畫的狀態標記得到更新；**可執行/引導文件**（`.github/copilot-instructions.md`, `README`, 模組清單, 命令表）過期，並開始主動誤導下一個代理程式/人類——例如，「隔離模組」清單與反應器不再相符。

**偵測探針：** 在任何拓撲結構變更後，比對變更與*描述*拓撲的每個文件：代理程式指示檔案、README、架構文件的模組/命令表、compose/服務清單。

**計畫行動：** 將**活計畫紀律**擴展到計畫檔案之外：改變拓撲結構的任何階段必須**在同一個 PR 中**更新 `copilot-instructions.md`、README 和任何模組/拓撲清單。將這些文件視為階段完成定義（Definition of Done）的一部分，而不是後續工作。

**範例：**
- 在 `gateway` 移入反應器且 `monitoring`/`turbine` 被移除後，`copilot-instructions.md` 隔離文字保持過期；README 寫死了一個在主幹變為 `main` 後產生 404 的 `master` 分支連結；actuator 暴露遺漏了 `refresh`，破壞了 `@RefreshScope` 工作流程。
