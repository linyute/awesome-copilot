# Salesforce Flow 設計與驗證

將這些檢查套用至您設計、建構或檢閱的每個 Flow。

## 步驟 1 — 確認 Flow 是正確的工具

在設計 Flow 之前，請驗證較輕量級的宣告式選項是否無法解決問題：

| 需求 | 最佳工具 |
|---|---|
| 計算欄位值且無副作用 | 公式欄位 |
| 透過使用者訊息防止不正確的記錄儲存 | 驗證規則 |
| 加總或計算父項上的子項記錄 | 累計摘要欄位 (Roll-up Summary) |
| 複雜的多物件邏輯、圖說文字 (Callouts) 或高載量 | Apex (Queueable / Batch) — 而非 Flow |
| 其他所有事項 | Flow ✓ |

如果您正在建構一個可以被公式欄位或驗證規則取代的 Flow，請要求使用者確認該需求確實更加複雜。

## 步驟 2 — 選擇正確的 Flow 類型

| 使用案例 | Flow 類型 | 關鍵限制 |
|---|---|---|
| 在記錄儲存前更新同一筆記錄上的欄位 | 儲存前記錄觸發 (Before-save Record-Triggered) | 無法傳送電子郵件、進行圖說文字 (Callouts) 或變更相關記錄 |
| 建立/更新相關記錄、電子郵件、圖說文字 (Callouts) | 儲存後記錄觸發 (After-save Record-Triggered) | 在提交後執行 — 避免遞迴陷阱 |
| 引導使用者完成多步驟 UI 程序 | 螢幕流程 (Screen Flow) | 無法由記錄事件自動觸發 |
| 可重複使用的背景邏輯由另一個 Flow 呼叫 | 自動啟動 (子流程 Autolaunched / Subflow) | 輸入/輸出變數定義協定 |
| 從 Apex `@InvocableMethod` 叫用的邏輯 | 自動啟動 (可叫用 Autolaunched / Invocable) | 必須宣告輸入/輸出變數 |
| 基於時間的批次處理 | 排程流程 (Scheduled Flow) | 在批次內容中執行 — 遵守管理員限制 (Governor Limits) |
| 回應事件 (平台事件 / CDC) | 平台事件觸發 (Platform Event–Triggered) | 非同步執行 — 最終一致性 |

**決策規則**：當您只需要變更觸發記錄本身的欄位時，請選擇儲存前 (Before-save)。一旦您需要觸及相關記錄、傳送電子郵件或進行圖說文字 (Callouts)，請改用儲存後 (After-save)。

## 步驟 3 — 批次安全性檢查清單

這些模式在規模化時會導致管理員限制失敗。在啟用 Flow 之前，請檢查所有這些項目。

### 迴圈中的 DML — 自動判定失敗

```
迴圈元件
  └── 建立記錄 / 更新記錄 / 刪除記錄  ← ❌ 迴圈內的 DML
```

修正：將迴圈內的記錄收集到集合變數中，然後在迴圈**外部**執行 DML 元件。

### 迴圈中的取得記錄 — 自動判定失敗

```
迴圈元件
  └── 取得記錄  ← ❌ 迴圈內的 SOQL
```

修正：在迴圈**之前**執行「取得記錄」查詢，然後對集合變數進行迴圈。

### 正確的批次模式

```
取得記錄 — 在單次查詢中收集所有記錄
└── 對集合變數進行迴圈
    └── 決策 / 指派 (無 DML，無取得記錄)
└── 迴圈結束後：建立/更新/刪除記錄 — 單次 DML 作業
```

### 轉換 vs 迴圈
當目標是重塑集合 (例如將欄位值從一個物件對應到另一個物件) 時，請使用**轉換 (Transform)** 元件，而不是「迴圈 + 指派」模式。轉換設計上就是批次安全的，並且能產生更乾淨的 Flow 圖表。

## 步驟 4 — 錯誤路徑需求

每個在執行階段可能會失敗的元件都必須有錯誤連接器 (fault connector)。沒有錯誤路徑的 Flow 會向使用者呈現原始系統錯誤。

### 需要錯誤連接器的元件
- 建立記錄
- 更新記錄
- 刪除記錄
- 取得記錄 (當存取可能不存在的必要記錄時)
- 傳送電子郵件
- HTTP 圖說文字 (Callout) / 外部服務動作
- Apex 動作 (可叫用)
- 子流程 (如果子流程可能擲回錯誤)

### 錯誤處理常式模式
```
錯誤連接器 → 記錄錯誤 (在記錄物件上建立記錄或觸發平台事件)
           → 帶有易懂訊息的螢幕元件 (螢幕流程)
           → 停止 / 結束元件 (記錄觸發流程)
```

切勿將錯誤路徑連接回發生錯誤的同一個元件 — 這會建立無限迴圈。

## 步驟 5 — 自動化密度檢查

在部署之前，請驗證同一個物件和觸發事件上沒有重疊的自動化：

- 同一 物件 + 何時執行 組合上的其他作用中記錄觸發流程 (Record-Triggered Flow)
- 同一物件上仍具作用中的舊版流程建立器 (Process Builder) 規則
- 在相同欄位變更時觸發的工作流程規則 (Workflow Rules)
- 同樣在 before insert / after update 內容中執行的 Apex 觸發程式

重疊的自動化可能會導致非預期的順序、遞迴和管理員限制失敗。在啟用前記錄該物件的自動化清單。

## 步驟 6 — 螢幕流程 (Screen Flow) UX 指導方針

- 螢幕流程的每個路徑都必須到達**結束**元件 — 不得有孤立的分支。
- 在多步驟流程中提供**返回**導覽選項，除非返回導覽會損毀資料。
- 對所有使用者輸入使用 lightning-input 和符合 SLDS 的元件 — 不要使用 HTML 表單元素。
- 在使用者可以進入下一步之前，在螢幕上驗證必要的輸入 — 在螢幕上使用 Flow 驗證規則。
- 如果流程可能需要等待跨工作階段的使用者動作，請處理**暫停 (Pause)** 元件。

## 步驟 7 — 部署安全性

```
部署為草稿    →   使用 1 筆記錄進行測試   →   使用 200 筆以上記錄進行測試   →   啟用
```

- 始終先部署為**草稿 (Draft)**，並在啟用前進行徹底測試。
- 對於記錄觸發流程：使用確切的進入條件進行測試 (例如 ISCHANGED(Status) — 確保測試資料實際觸發了條件)。
- 對於排程流程：在正式環境中啟用之前，先在沙箱中進行小批次測試。
- 檢查物件的自動化密度分數 — 單一物件上超過 3 個作用中的自動化會增加執行順序風險。

## 快速參考 — Flow 反模式 (Anti-patterns) 摘要

| 反模式 | 風險 | 修正 |
|---|---|---|
| 迴圈內部的 DML 元件 | 管理員限制異常 | 將 DML 移到迴圈外部 |
| 迴圈內部的取得記錄 | SOQL 管理員限制異常 | 在迴圈之前進行查詢 |
| DML/電子郵件/圖說文字元件上無錯誤連接器 | 向使用者呈現未處理的異常 | 為每個此類元件新增錯誤路徑 |
| 在儲存後流程中更新觸發記錄且無遞迴保護 | 無限觸發迴圈 | 新增進入條件或遞迴保護變數 |
| 直接在 $Record 集合上進行迴圈 | 規模化時行為不正確 | 先指派給集合變數，然後進行迴圈 |
| 流程建立器 (Process Builder) 仍與新 Flow 同時具作用中 | 重複執行、非預期的順序 | 在啟用 Flow 之前停用流程建立器 |
| 螢幕流程在所有分支上都沒有結束元件 | 執行階段錯誤或使用者受阻 | 確保每個分支都解析為結束元件 |
