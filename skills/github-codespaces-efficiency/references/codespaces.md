# Codespaces 與 Devcontainer 效率

僅在任務涉及 `.devcontainer/`、Codespaces 大小調整、預構建 (prebuilds) 或工作區成本時，才載入此參考。

如果儲存庫是首次導入 Codespaces，請先定義最低限度的基準，然後再進行優化。

## 稽核順序

請依照此順序檢查：

1. 檢查是否存在 `.devcontainer/`。
2. 如果不存在，請先收集基準需求：語言/工具鏈、所需的 CLI、預期的啟動目標、機器大小限制，以及是否需要預構建。
3. 審查 `.devcontainer/devcontainer.json`。
4. 確保存在 `.devcontainer/devcontainer-lock.json`；如果遺失，請建議新增，因為許多儲存庫早於鎖定檔 (lock-file) 支援。
5. 審查相關的 Dockerfile、功能 (features) 和設定指令碼。
6. 審查建議機器大小、預構建或啟動預期相關的文件。

對於新的設定（步驟 2），請從最小化開始，並避免使用選用工具，直到使用數據證明它們的必要性為止。

## 常見浪費來源

- 基礎映像檔過大
- 不必要的軟體包或擴充功能
- 緩慢的後建立 (post-create) 引導步驟
- 為過多分支啟用預構建
- 遺失機器大小調整或閒置逾時規律的指引

## 建議修復順序

1. 移除不必要的軟體包、功能和擴充功能
2. 減少啟動指令和後建立安裝
3. 建議能保持輸送量 (throughput) 的最小機器大小
4. 將預構建範圍縮小至持續使用的分支（預設分支、活躍的發行分支，以及每週擁有超過五個 Codespaces 的任何分支）
5. 新增或加強閒置逾時和清理指引

## 安全變更規則

- 不要透過移除團隊每天實際需要的工具來優化啟動。
- 區分儲存庫變更與組織或使用者設定。
- 當有效的控制權位於儲存庫之外時，請優先使用文件說明。
- 除非團隊明確需要，否則請避免將 devcontainer 轉變為類似生產環境的映像檔。

## 報告重點

在報告 Codespaces 改進時，請分開報告：

- 更快的啟動
- 更低的穩態工作區成本
- 更低的預構建支出
- 僅建議類型的建議，仍需要組織或使用者採取行動
