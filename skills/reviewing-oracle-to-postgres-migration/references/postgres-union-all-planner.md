# PostgreSQL UNION ALL 計畫器風險指南

目的：避免已移轉的 `UNION ALL` 查詢在 PostgreSQL 中執行速度遠低於預期的效能退化。

## 問題

`UNION ALL` 會保留重複的資料列並直接合併分支輸出，但 PostgreSQL 並非總是能像孤立查詢那樣主動最佳化每個分支。在大型資料集中，這可能會產生不佳的計畫（例如，在預期使用基於索引之計畫的地方進行了全表掃描）。

## 為什麼會發生

- 透過 `UNION ALL` 分支的述詞下推（Predicate pushdown）可能會受限，具體取決於查詢的形狀。
- 跨分支的基數（Cardinality）估計可能會偏斜。
- 當最佳化器評估合併查詢時，可能不會選擇分支局部索引。

## 審閱檢核表

- [ ] 針對合併的 `UNION ALL` 查詢和分支孤立變體，比較 `EXPLAIN (ANALYZE, BUFFERS)` 計畫。
- [ ] 確認分支述詞是明確的，而不是隱藏在非 sargable 運算式中。
- [ ] 檢查大表中任一分支是否存在未預期的循序掃描。
- [ ] 驗證每個分支的篩選和聯結述詞是否存在索引。

## 緩和模式

1. 獨立測試每個分支以驗證預期的索引使用情況。
2. 將篩選器下推至各分支，而不是僅在外部查詢中進行篩選。
3. 如果計畫品質仍然不佳，請將查詢分割為兩個單獨執行的陳述式，並在應用程式程式碼中合併結果。
4. 僅在測量確認有益處時，才考慮在暫存/中間結構中具體化分支結果。

## 驗證備註

即使功能測試結果與 Oracle 相符，也請將 `UNION ALL` 效能行為視為移轉審閱項目。
