# PostgreSQL 具體化檢視表重新整理指南

目的：確保已移轉的應用程式在基底資料表變更後，保持具體化檢視表（Materialized view）為最新狀態。

## 問題

PostgreSQL 具體化檢視表是靜態快照。對來源資料表的更新**不會**自動重新整理相依的具體化檢視表。

## 移轉風險

- Oracle 時代關於衍生資料會立即更新的假設可能不再成立。
- 除非明確管理重新整理時間，否則讀取路徑可能會傳回過期的資料列。
- 如果重新整理順序不是確定性的，整合測試可能會通過一次，然後斷斷續續地失敗。

## 必要的檢視項目

對於寫入至餵給具體化檢視表之資料表的每個已移轉路徑，請驗證應用程式工作流程中包含明確的重新整理策略。

## 重新整理模式

- 當需要即時性時，在寫入工作流程中立即重新整理：
  ```sql
  REFRESH MATERIALIZED VIEW my_view;
  ```
- 並行重新整理（在支援且有索引時）以減少讀取鎖定：
  ```sql
  REFRESH MATERIALIZED VIEW CONCURRENTLY my_view;
  ```
- 當可以接受資料過期窗口時，進行排程/批次重新整理。

## 整合測試預期事項

- [ ] 修改來源資料表的測試，必須在預期的重新整理動作之後，才斷言具體化檢視表的內容。
- [ ] 測試在適用時斷言重新整理前的過期行為。
- [ ] 測試記錄每個受影響功能的即時性是立即的還是最終一致的。
