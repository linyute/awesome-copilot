# Oracle 到 PostgreSQL 資料庫遷移 (Oracle-to-PostgreSQL Database Migration)

揭示遷移風險，並根據 `references/` 資料夾中記錄的已知 Oracle/PostgreSQL 行為差異來驗證遷移工作。

## 何時使用

1. **規劃 (Planning)** — 在開始對預存程序、觸發程序、查詢或 Refcursor 用戶端進行遷移工作之前。識別哪些參考見解適用，以便預先處理風險。
2. **驗證 (Validating)** — 遷移工作完成後，確認每個適用的見解都已處理，且整合測試涵蓋了新的 PostgreSQL 語義。

## 工作流程 (Workflow)

確定任務類型：

**正在規劃遷移？** 請遵循風險評估工作流程。
**正在驗證已完成的工作？** 請遵循驗證工作流程。

### 風險評估工作流程 (規劃)

```
風險評估：
- [ ] 步驟 1：識別遷移範圍
- [ ] 步驟 2：篩選每個見解的適用性
- [ ] 步驟 3：記錄風險與建議行動
```

**步驟 1：識別遷移範圍**

列出受影響的資料庫物件（預存程序、觸發程序、查詢、檢視表）以及呼叫它們的應用程式程式碼。

**步驟 2：篩選每個見解的適用性**

檢閱 [references/REFERENCE.md](REFERENCE.md) 中的參考索引。針對每個項目，確定遷移範圍是否包含受該見解影響的模式。僅在見解可能相關時才閱讀完整的參考檔案。

**步驟 3：記錄風險與建議行動**

針對每個適用的見解，記錄特定風險以及參考檔案中建議的修復模式。標記任何需要設計決策的見解（例如：是否保留 Oracle 的「空字串視為 NULL」語義，還是採用 PostgreSQL 行為）。

### 驗證工作流程 (遷移後)

```
驗證：
- [ ] 步驟 1：對應遷移成品
- [ ] 步驟 2：交叉檢查適用的見解
- [ ] 步驟 3：驗證整合測試涵蓋範圍
- [ ] 步驟 4：對結果進行管制 (Gate the result)
```

**步驟 1：對應遷移成品**

識別已遷移的物件並總結變更集。

**步驟 2：交叉檢查適用的見解**

針對 [references/REFERENCE.md](REFERENCE.md) 中的每個參考項目，確認該行為或測試需求在遷移工作中已獲得承認並處理。

**步驟 3：驗證整合測試涵蓋範圍**

確認測試執行了適用的見解中強調的正常路徑與失敗情境（例外處理、排序、Refcursor 取用、並行交易、時戳等）。

**步驟 4：對結果進行管制 (Gate the result)**

回傳一份檢核表，斷言每個適用的見解都已處理、遷移指令碼已執行，且整合測試已通過。
