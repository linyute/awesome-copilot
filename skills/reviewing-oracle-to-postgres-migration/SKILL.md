# Oracle 至 PostgreSQL 資料庫遷移

依據 `references/` 資料夾中記載的已知 Oracle/PostgreSQL 行為差異，浮現遷移風險並驗證遷移工作。

## 使用時機

1. **規劃階段** — 在開始對程序、觸發器、查詢或 refcursor 用戶端進行遷移工作之前。識別哪些參考洞見適用，以便提前處理風險。
2. **驗證階段** — 遷移工作完成後，確認每個適用的洞見均已處理，且整合測試涵蓋了新的 PostgreSQL 語義。

## 工作流程

判斷任務類型：

**正在規劃遷移？** 請遵循風險評估工作流程。
**正在驗證已完成的工作？** 請遵循驗證工作流程。

### 風險評估工作流程（規劃）

```
風險評估：
- [ ] 步驟 1：識別遷移範圍
- [ ] 步驟 2：篩選每個洞見的適用性
- [ ] 步驟 3：記錄風險與建議行動
```

**步驟 1：識別遷移範圍**

列出受影響的資料庫物件（程序、觸發器、查詢、檢視）以及呼叫它們的應用程式程式碼。

**步驟 2：篩選每個洞見的適用性**

檢閱 [references/REFERENCE.md](references/REFERENCE.md) 中的參考索引。針對每個條目，判斷遷移範圍是否包含受該洞見影響的模式。僅在洞見可能相關時才讀取完整的參考檔案。

**步驟 3：記錄風險與建議行動**

針對每個適用的洞見，記錄參考檔案中的具體風險與建議修正模式。標記任何需要設計決策的洞見（例如，是否要保留 Oracle 空字串視為 NULL 的語義，或採用 PostgreSQL 行為）。

### 驗證工作流程（遷移後）

```
驗證：
- [ ] 步驟 1：對應遷移成品
- [ ] 步驟 2：交叉比對適用洞見
- [ ] 步驟 3：驗證整合測試覆蓋率
- [ ] 步驟 4：結果把關
```

**步驟 1：對應遷移成品**

識別已遷移的物件並摘要說明變更集。

**步驟 2：交叉比對適用洞見**

針對 [references/REFERENCE.md](references/REFERENCE.md) 中的每個參考，確認遷移工作中已確認並處理了該行為或測試需求。

**步驟 3：驗證整合測試覆蓋率**

確認測試涵蓋了適用洞見中強調的正常路徑與失敗情境（例外狀況、排序、`UNION ALL` 行為/效能風險、refcursor 消費、並行交易、時間戳記、物化檢視新鮮度等）。

**步驟 4：結果把關**

返回一份清單，斷言每個適用洞見均已處理、遷移腳本已執行、整合測試均通過。
