# 分析原則 — 安全分析方法論

本檔案包含所有如何針對安全威脅進行程式碼分析的規則。它是自成一體的 — 執行正確且基於證據的安全分析所需的所有資訊都在這裡。

---

## ⛔ 重要：回報前請先驗證

**在確認安全漏洞存在之前，絕不要回報它。** 許多平台都有安全的預設值。

### 三步驟驗證

1. **在聲明缺少安全性之前，先檢查安全基礎架構元件**：
   - 憑證授權單位 (Dapr Sentry, cert-manager, Vault)
   - 服務網格 (Service mesh) 控制平面 (Istio, Linkerd, Dapr)
   - 策略引擎 (Policy engines) (OPA, Kyverno, Gatekeeper)
   - 秘密管理員 (Secret managers) (Vault, Azure Key Vault, AWS Secrets Manager)
   - 身分識別提供者 (Identity providers) (MISE, OAuth 代理, OIDC)

2. **了解平台預設值** — 在假設之前先進行研究：
   - Dapr：部署 Sentry 時預設啟用 mTLS
   - Kubernetes：自 v1.6 起預設啟用 RBAC
   - Istio：預設為 PERMISSIVE 模式的 mTLS，可使用 STRICT 模式
   - Azure：許多服務預設對靜態資料進行加密

3. **區分組態狀態**：
   - **明確停用**：`enabled: false` → 回報為發現事項 (finding)
   - **未設置**：不存在設定 → 先檢查平台預設行為
   - **隱含啟用**：預設行為是安全的 → 記錄為控制措施，而非漏洞

### 證據品質要求

對於每個發現事項：
- 顯示證明該漏洞的特定組態/程式碼（而不僅僅是缺少組態）
- 對於「缺少安全性」的聲明，請證明預設情況是不安全的
- 不確定時請交叉引用平台文件

## 安全基礎架構清單

在進行 STRIDE-A 分析之前，請識別程式碼庫中存在的所有安全強化元件：

| 類別 | 要尋找的元件 | 它們提供的安全性 |
|----------|----------------------|----------------------|
| 服務網格 | Dapr, Istio, Linkerd, Consul Connect | mTLS、流量策略、觀測性 |
| 憑證管理 | Sentry, cert-manager, Vault PKI | 自動憑證發行/輪換 |
| 驗證 | MISE, OAuth2-proxy, Dex, Keycloak | 權杖驗證、SSO |
| 授權 | OPA, Kyverno, Gatekeeper, RBAC | 策略執行 |
| 秘密 | Vault, External Secrets, CSI 驅動程式 | 秘密注入、輪換 |
| 網路 | NetworkPolicy, Calico, Cilium | 微分段 (Microsegmentation) |

**如果這些元件存在，除非明確停用，否則它們的安全功能很可能是啟用的。**

## 安全分析視角

在分析期間應用這些框架：

- **零信任 (Zero Trust)**：顯式驗證、最小權限、假設遭受入侵
- **深度防禦 (Defense in Depth)**：識別缺少的安全層
- **濫用個案 (Abuse Cases)**：商業邏輯濫用、工作流操作、功能誤用

## 全面涵蓋要求

**不要針對大型程式碼庫縮減分析。** 所有元件都必須接受相同的分析深度。

### Sidecar 安全分析

⚠️ **Sidecar (Dapr, MISE, Envoy 等) 不是 DFD 中的獨立元件** — 它們與主容器位於同一個 pod 中（參見 [diagram-conventions.md](diagram-conventions.md) 規則 2）。然而，仍必須分析 Sidecar 通訊的安全漏洞。

**如何分析 Sidecar 威脅：**
- 具有獨特威脅面的 Sidecar (例如 MISE 驗證繞過、Dapr mTLS) 在 `2-stride-analysis.md` 中有其專屬的 `## Component` 章節 — 但它們並不是獨立的 DFD 節點（參見 [diagram-conventions.md](diagram-conventions.md) 規則 2）
- 使用格式：威脅標題包含 Sidecar 名稱，例如：「Dapr Sidecar 明文通訊」
- 常見 Sidecar 威脅：
  - **資訊洩漏 (Information Disclosure, I)**：Dapr/MISE Sidecar 與主容器在 pod 內透過明文 HTTP 進行通訊
  - **竄改 (Tampering, T)**：Dapr 發布/訂閱 (pub/sub) 訊息未簽署或未加密
  - **冒充 (Spoofing, S)**：若 Sidecar 遭到入侵，MISE 權杖驗證被繞過
  - **權限提升 (Elevation of Privilege, E)**：Sidecar 以主容器不需要的提升權限執行
- CWE 映射：CWE-319 (明文傳輸)、CWE-311 (遺失加密)、CWE-250 (不必要的權限)
- 這些威脅出現在 Sidecar 自己的 STRIDE 章節（如果它具有獨特的威脅面）或主元件的表格下（如果 Sidecar 只是簡單的基礎架構代理）
- 如果 Sidecar 漏洞值得作為發現事項，請將其列在 Sidecar 元件下並註明：「影響 [Dapr/MISE] Sidecar 通訊」

1. **最低涵蓋範圍**：`0.1-architecture.md` 中的每個元件都必須在 `2-stride-analysis.md` 中有對應的章節，並包含實際的威脅列舉（而不僅僅是「未發現威脅」）。
2. **發現事項密度檢查**：作為準則，預計每 2-3 個重要元件約有 1 個發現事項。如果一個儲存庫有超過 15 個元件，而您的發現事項少於 8 個，請重新檢查分析不足的元件。
3. **使用子代理進行規模化**：對於擁有 10 個以上元件的儲存庫，將特定元件的 STRIDE 分析委託給子代理以保持深度。每個子代理應分析 3-5 個元件。
4. **OWASP 清單掃描**：在元件層級的 STRIDE 之後，使用下方的 OWASP Top 10:2025 清單進行交叉檢查。這可以捕捉到元件層級分析可能會遺漏的系統性問題（遺失驗證、無稽核記錄、無速率限制、未簽署映像）。
5. **基礎架構層檢查**：明確檢查：容器安全上下文、網路策略、資源限制、映像簽署、秘密管理、備份/災難復原控制以及監控/告警漏洞。
6. **詳盡的發現事項合併**：在 STRIDE 分析完成後，掃描 STRIDE 輸出以查找所有識別出的威脅。每個威脅都必須映射到以下之一：
   - `3-findings.md` 中的一個發現事項（與相關威脅合併）
   - 威脅涵蓋範圍驗證表中的 `🔄 Mitigated by Platform` 條目（僅限平台處理的威脅）

   **⛔ 每個 `Open` 威脅都必須有一個發現事項。** 工具無權接受風險、推遲威脅或決定威脅是「可接受的」。這是工程團隊的決定。工具的工作是識別所有威脅並為其建立發現事項。涵蓋範圍表應對每個 Open 威脅顯示 `✅ Covered (FIND-XX)` — 絕不要顯示 `⚠️ Accepted Risk`。

   如果您在 STRIDE 中有 40 多個威脅，但只有 10 個發現事項，表示您的合併程度不足。檢查是否遺漏了資料儲存驗證、操作控制、憑據管理和供應鏈問題。

   **⛔ 禁止「接受風險」(強制執行)：**
   - **絕不要在涵蓋範圍表狀態中使用 `⚠️ Accepted Risk`。** 此標籤暗示工具已代表工程團隊接受了風險。事實並非如此，它也無法這麼做。
   - **絕不要將 `Accepted` 作為 STRIDE 狀態值。** 僅使用 `Open`、`Mitigated` 或 `Platform`。
   - 如果您想寫「接受風險」 → 請改為建立發現事項。發現事項的補救章節會告訴團隊該怎麼做。團隊會決定是否接受、修復或延遲。

   **⛔ 「需審查 (NEEDS REVIEW)」限制 (強制執行)：**
   - **第 1 層威脅（必要條件 = `None`）絕不能被歸類為「⚠️ 需審查」。** 未經身分驗證的外部攻擊者可利用的威脅不能推遲 — 它必須成為一個發現事項。
   - **如果 STRIDE 分析中列出了某個威脅的緩解措施，它應該成為一個發現事項。** 緩解措施文字即為補救措施 — 用它來撰寫發現事項。僅在緩解措施確實不可執行的情況下，才推遲到「需審查」。
   - **必要條件為 `None` 的 DoS 威脅是第 1 層發現事項**，而不是強化的機會。未經身分驗證的攻擊者在沒有速率限制的情況下湧入 API 是直接可利用的漏洞 (CWE-770, CWE-400)。
   - **不要將整個 STRIDE 類別批量歸類為「需審查」。** 每個威脅都必須根據其必要條件和可利用性進行單獨評估。
   - **「⚠️ 需審查」保留給：** 無法進行技術緩解的第 2/3 層威脅（例如社交工程），或需要工具所不具備的商業背景的威脅。
   - **自動化分析無權接受風險** — 它僅識別風險。「需審查」信號表示必須由人工決定。
   - **最大「需審查」比例**：如果超過 30% 的威脅被歸類為「需審查」，請重新檢查 — 您可能少報了發現事項。典型比例：分析良好的程式碼庫為 10-20%。
7. **按儲存庫大小劃分的最低發現事項門檻：**
   - 小型儲存庫（< 20 個原始檔案）：預期 8 個以上發現事項
   - 中型儲存庫（20-100 個原始檔案）：預期 12 個以上發現事項
   - 大型儲存庫（100 個以上原始檔案）：預期 18 個以上發現事項

   如果低於門檻，請系統地審查：各元件驗證、程式碼中的秘密、容器安全、網路分段、記錄/監控、輸入驗證。

8. **內容感知的平台比例限制 (強制執行)：**

   完成安全基礎架構清單（第 1 步）後，偵測部署模式：

   | 模式 | 偵測信號 | 平台限制 |
   |---------|-----------------|----------------|
   | **K8s Operator** | go.mod/go.sum 中包含 `controller-runtime`、`kubebuilder` 或 `operator-sdk`；原始碼中包含 `Reconcile()` 函式 | **≤35%** |
   | **獨立應用程式** | 所有其他儲存庫（網頁應用程式、CLI 工具、服務） | **≤20%** |

   **為何 K8s 運算子 (Operator) 的平台比例較高**：運算子將安全性委託給 K8s 平台（用於 CR 存取的 RBAC、etcd 加密、API 伺服器 TLS、webhook 憑證驗證、Azure AD 權杖驗證）。運算子程式碼無法實作這些控制措施 — 它們是平台的責任。將其歸類為平台是正確的。

   **當平台比例超過限制時的操作**：
   - 審查每個歸類為平台的威脅
   - 如果運算子可以採取行動（例如：新增輸入驗證、在啟動時新增 RBAC 檢查） → 重新分類為 `Open` 並建立一個發現事項
   - 如果運算子確實無法採取行動（例如：etcd 加密是叢集管理員的事項） → 歸類為平台是正確的
   - 在 `0-assessment.md` 的「分析背景與假設」中記錄偵測到的模式和比例

## 技術特定安全清單

**在完成 STRIDE 分析後**，掃描程式碼庫中是否存在以下技術。對於發現的每項技術，驗證對應的安全檢查是否已包含在發現事項中或記錄為已緩解。這可以捕捉到元件層級 STRIDE 常會遺漏的特定漏洞。

| 發現的技術 | 必須檢查 | 常見發現事項 |
|-----------------|---------------|----------------|
| **Redis** | `requirepass` 已停用、無 TLS、無 ACL | 預設停用驗證 → 發現事項 |
| **Milvus** | `authorizationEnabled: false`、無 TLS、公開 gRPC 連接埠 | 預設停用驗證 → 發現事項 |
| **PostgreSQL/SQL DB** | 超級使用者使用、`ssl=false`、SQL 注入、連接字串憑據 | 輸入驗證 + 驗證 |
| **MongoDB** | 驗證已停用、無 TLS、`--noauth` 旗標 | 預設停用驗證 |
| **NGINX/Ingress** | 遺失 TLS、server_info 標頭、程式碼片段注入、速率限制 | 組態強化 |
| **Docker/容器** | 以 root 身分執行、無 `USER` 指令、主機掛載、無 seccomp/AppArmor、未簽署映像 | 容器強化 |
| **ML/AI 模型** | 未經身分驗證的推論端點、模型中毒、提示注入、無輸入驗證 | 端點驗證 + 輸入驗證 |
| **LLM/Cloud AI** | PII/秘密傳送到外部 LLM、無內容過濾、提示注入、資料外洩 | 資料暴露給雲端 |
| **Kubernetes** | 無 NetworkPolicy、無 PodSecurityPolicy/Standards、無資源限制、RBAC 漏洞 | 網路分段 + 資源限制 |
| **Helm Charts** | values.yaml 中硬編碼秘密、無映像標記固定、無安全上下文 | 組態 + 供應鏈 |
| **金鑰管理** | 硬編碼 RSA/HMAC 金鑰、弱金鑰生成、無輪換、程式碼中有金鑰 | 加密失敗 |
| **CI/CD 管線** | 記錄中有秘密、無構件簽署、可變相依性、腳本注入 | 供應鏈 |
| **REST APIs** | 遺失驗證、無速率限制、冗長錯誤、無輸入驗證 | 驗證 + 注入 |
| **gRPC 服務** | 無 TLS、無驗證攔截器、在正式環境啟用反射 | 驗證 + 加密 |
| **訊息佇列** | 發布/訂閱無驗證、無加密、無訊息簽署 | 驗證 + 完整性 |
| **NFS/檔案共享** | 路徑遍歷、無存取控制、全域可讀掛載 | 存取控制 |
| **稽核/記錄** | 無安全事件記錄、記錄注入、無防竄改保護 | 監控漏洞 |

**流程：** 在撰寫 3-findings.md 之後，掃描此表以查找儲存庫中存在的技術。對於每項技術，根據該技術的實際使用方式評估其常見的技術特定威脅模式，並確保在評估中說明任何相關風險。僅在識別出實際威脅或有意義的緩解缺口時才新增發現事項。

## OWASP Top 10:2025 清單

分析期間請檢查這些漏洞類別：

| ID | 類別 | 檢查內容 |
|----|----------|----------|
| A01 | 權限控制失效 (Broken Access Control) | 遺失授權、權限提升、IDOR、CORS 配置錯誤 |
| A02 | 安全設定錯誤 (Security Misconfiguration) | 預設憑據、冗長錯誤、不需要的功能、遺失強化 |
| A03 | 軟體與供應鏈失敗 (Software Supply Chain Failures) | 有漏洞的相依性、惡意套件、受損的 CI/CD |
| A04 | 加密機制失效 (Cryptographic Failures) | 弱演算法、暴露的秘密、不當金鑰管理、明文資料 |
| A05 | 注入 (Injection) | SQL, NoSQL, OS 命令, LDAP, XSS, 範本注入 |
| A06 | 不安全設計 (Insecure Design) | 架構層級遺失安全控制、威脅建模漏洞 |
| A07 | 驗證機制失效 (Authentication Failures) | 驗證受損、弱工作階段、憑據填充、遺失 MFA |
| A08 | 軟體與資料完整性失敗 (Software/Data Integrity Failures) | 不安全的反序列化、未簽署的更新、CI/CD 竄改 |
| A09 | 安全記錄與告警失敗 (Security Logging & Alerting Failures) | 遺失稽核記錄、無告警、記錄注入、監控不足 |
| A10 | 例外狀況處理不當 (Mishandling of Exceptional Conditions) | 錯誤處理不佳、競態條件、資源耗盡 |

參考：https://owasp.org/Top10/2025/

## 平台安全預設值參考

在回報缺少安全性之前，請檢查以下常見的安全預設行為：

| 平台 | 功能 | 預設行為 | 如何驗證 |
|----------|---------|------------------|---------------|
| **Dapr** | mTLS | 部署 Sentry 時啟用 | 檢查是否包含 `dapr_sentry` 或 `sentry` 元件 |
| **Dapr** | 存取控制 | 若定義了策略則拒絕 | 在組態中查找 `accessControl` |
| **Kubernetes** | RBAC | 自 v1.6 起啟用 | 檢查 `--authorization-mode` 是否包含 RBAC |
| **Kubernetes** | 秘密 | Base64 編碼（非加密） | 檢查加密提供者組態 |
| **Istio** | mTLS | 預設為 PERMISSIVE | 檢查 PeerAuthentication 資源 |
| **Azure Storage** | 靜態加密 | 預設啟用 | 始終加密，檢查金鑰管理 |
| **Azure SQL** | TDE | 預設啟用 | 透明資料加密已開啟 |
| **PostgreSQL** | SSL | 通常預設停用 | 檢查 `ssl` 參數 |
| **Redis** | 驗證 | 預設停用 | 檢查 `requirepass` 組態 |
| **Milvus** | 驗證 | 預設停用 | 檢查 `authorizationEnabled` |
| **NGINX Ingress** | TLS | 預設未啟用 | 檢查 Ingress 中的 TLS 秘密 |
| **Docker** | 使用者 | 預設為 Root | 檢查 Dockerfile 中的 `USER` |

**關鍵見解**：服務網格 (Dapr, Istio, Linkerd) 通常會自動啟用 mTLS。資料庫 (Redis, Milvus, MongoDB) 通常預設停用驗證。

## 可利用性層級 (Exploitability Tiers)

威脅根據必要條件分為三個可利用性層級：

| 層級 | 標籤 | 必要條件 | 分配規則 |
|------|-------|---------------|----------------|
| **第 1 層** | 直接暴露 | `None` | 可由未經身分驗證的外部攻擊者在沒有事先存取權限的情況下利用。 |
| **第 2 層** | 條件風險 | 單一必要條件 | 剛好需要一種存取形式：`Authenticated User`、`Privileged User`、`Internal Network` 或單一 `{Boundary} Access`。 |
| **第 3 層** | 深度防禦 | 多個必要條件或基礎架構存取 | 需要 `Host/OS Access`、`Admin Credentials`、`{Component} Compromise`、`Physical Access` 或帶有 `+` 的多個必要條件。 |

### 層級分配規則

**⛔ 規範必要條件 → 層級映射（確定性，無例外）：**

必要條件必須僅使用這些值（封閉列舉）。層級會機械式地跟隨：

| 必要條件 | 層級 | 理由 |
|-------------|------|----------|
| `None` | **第 1 層** | 未經身分驗證的外部攻擊者，無事先存取權限 |
| `Authenticated User` | **第 2 層** | 需要有效的憑據 |
| `Privileged User` | **第 2 層** | 需要管理員/運算子角色 |
| `Internal Network` | **第 2 層** | 需要在內部網路中的位置 |
| `Local Process Access` | **第 2 層** | 需要在同一主機上執行程式碼（localhost 監聽器, IPC） |
| `Host/OS Access` | **第 3 層** | 需要檔案系統、主控台或對主機的偵錯存取權限 |
| `Admin Credentials` | **第 3 層** | 需要管理員憑據 + 主機存取權限 |
| `Physical Access` | **第 3 層** | 需要實體接觸 (USB, 序列埠) |
| `{Component} Compromise` | **第 3 層** | 需要先前對另一個元件的入侵 |
| 任何 `A + B` 組合 | **第 3 層** | 多個必要條件 = 始終為第 3 層 |

**⛔ 禁止的必要條件值**：`Application Access`、`Host Access`（含糊不清 — 請使用 `Local Process Access` 或 `Host/OS Access`）。

**部署背景覆蓋**：如果部署分類為 `LOCALHOST_DESKTOP` 或 `LOCALHOST_SERVICE`，則所有元件均禁止使用 `None` 作為必要條件 — 請改用 `Local Process Access` 或 `Host/OS Access`。層級則根據修正後的必要條件決定。

### ⛔ 必要條件判定（強制執行 — 基於證據，而非基於判斷）

**必要條件必須根據部署組態證據判定，而非根據一般知識或假設。** 對同一程式碼進行兩次獨立分析必須分配相同的必要條件，因為它們是關於部署的客觀事實。

**通用決策程序（適用於所有環境）：**

1. **網路暴露檢查 — 元件是否可從外部存取？**
   - 在程式碼庫中尋找外部暴露的證據：
     - 指向元件的 API 閘道 / 反向代理路由
     - 防火牆規則或安全群組組態
     - 負載平衡器組態
     - DNS 記錄或公開端點定義
   - 如果存在任何外部路由 → 對於基於網路的威脅，必要條件 = `None`
   - 如果不存在外部路由，且元件位於僅限內部的網路 → 必要條件 = `Internal Network`

2. **驗證檢查 — 端點是否需要憑據？**
   - 在元件程式碼中尋找驗證中介軟體、裝飾器或過濾器：
     - `@require_auth`, `[Authorize]`, `@login_required`, Express/FastAPI 中的驗證中介軟體
     - 要求處理常式中的 API 金鑰驗證
     - OAuth/OIDC 權杖驗證
     - mTLS 憑證要求
   - 如果在所有端點強制執行驗證 → 必要條件 = `Authenticated User`
   - 如果驗證是選用的或透過組態旗標停用 → 必要條件 = `None`（停用的驗證 = 無障礙）
   - 如果驗證存在但有繞過路由（例如：無需驗證的 `/health`、`/metrics`） → 這些特定路由的必要條件 = `None`

3. **授權檢查 — 需要什麼層級的存取權限？**
   - 如果除了驗證之外沒有 RBAC/角色檢查 → 必要條件保持為 `Authenticated User`
   - 如果需要管理員/運算子角色 → 必要條件 = `Privileged User`
   - 如果需要特定權限 → 必要條件命名該權限（例如：`ClusterAdmin Role`）

4. **實體/本地存取檢查：**
   - 如果元件僅監聽 `localhost`/`127.0.0.1` → 必要條件 = `Local Process Access` (T2)
   - 如果存取需要主控台/SSH/檔案系統 → 必要條件 = `Host/OS Access` (T3)
   - 如果存取需要實體接觸 (USB, 序列埠) → 必要條件 = `Physical Access` (T3)
   - 如果元件沒有監聽器（主控台應用程式、函式庫、僅限外連） → 必要條件 = `Host/OS Access` (T3)

5. **預設規則**：如果您無法從組態中判定暴露情況 → 請在元件暴露表中查找該元件的「最低必要條件」。如果表尚未填寫，則假設 `Local Process Access` (T2) 作為未知元件的安全預設值。**絕不要在沒有外部可達性確切證據的情況下假設 `None`。****絕不要在沒有網路限制證據的情況下假設 `Internal Network`。**

**平台特定證據來源：**

| 平台 | 在何處檢查暴露 | 內部指標 | 外部指標 |
|----------|------------------------|--------------------|--------------------|
| **Kubernetes** | 服務類型、Ingress 規則、values.yaml | `ClusterIP` 服務、無 Ingress | `LoadBalancer`/`NodePort`、存在 Ingress 路徑 |
| **Docker Compose** | `ports:` 映射、網路組態 | 無 `ports:` 映射、僅限內部網路 | `ports: "8080:8080"` 映射到主機 |
| **Azure App Service** | 應用程式設定、存取限制 | VNet 整合、私用端點 | 公開 URL、無 IP 限制 |
| **VM / 裸機** | 防火牆規則、NSG、iptables | 連接埠在防火牆/NSG 中被封鎖 | 連接埠開啟、綁定公用 IP |
| **無伺服器 (Functions)** | 函式驗證層級、API 管理 | `authLevel: function/admin` | `authLevel: anonymous` |
| **.NET / Java / Node** | 啟動組態、中介軟體管線 | 強制執行 `app.UseAuthentication()` | 無驗證中介軟體，或驗證已停用 |
| **Python (FastAPI/Flask)** | 中介軟體、相依性注入 | 路由上有 `Depends(get_current_user)` | 無驗證相依性、開放路由 |

**⛔ 絕不要基於「看起來合理」或架構假設分配必要條件。** 請檢查實際的部署組態。相同的元件在不同分析執行中必須獲得相同的必要條件，因為組態在執行之間不會改變。

**常見違規：**
- 將 `Internal Network` 分配給具有 ingress 路由的元件 → 隱藏了實際的外部暴露
- 在未檢查資料庫是否有公用端點或 ingress 路由的情況下，假設資料庫是「僅限內部」的
- 假設 ML 模型伺服器是「內部」的，但它們可能暴露於直接的推論請求

### CVSS 與層級的一致性檢查 (強制執行)

**分配 CVSS 向量和層級後，交叉檢查是否存在矛盾：**

| CVSS 指標 | 值 | 層級隱含意義 |
|-------------|-------|------------------|
| `AV:L` (攻擊向量：本地) | 需要本地存取 | **不能為第 1 層** — 必須為 T2 或 T3 |
| `AV:A` (攻擊向量：鄰近) | 需要鄰近網路 | **不能為第 1 層** — 必須為 T2 或 T3 |
| `AV:P` (攻擊向量：實體) | 需要實體存取 | **必須為第 3 層** |
| `PR:H` (所需權限：高) | 需要管理員/特權存取 | **不能為第 1 層** — 必須為 T2 或 T3 |
| `PR:L` (所需權限：低) | 需要已驗證使用者 | **不能為第 1 層** — 必須為 T2 |
| `PR:N` + `AV:N` | 無權限、網路可存取 | 第 1 層候選者（確認無部署覆蓋） |

⚠️ **如果發現事項具有 `AV:L` 且為 `第 1 層`，這始終是一個錯誤。** 修復方法如下之一：
- 將層級更改為 T2/T3（適用於僅限本地服務的正確方法），或
- 如果服務實際上是網路可存取的，將 CVSS AV 更改為 `AV:N`（罕見）

⚠️ **如果發現事項具有 `PR:H` 且為 `第 1 層`，這始終是一個錯誤。** 需要管理員權限的發現事項至少為 T2。

### 部署背景影響層級分類

**重要：當特定的部署條件適用時，本章節會覆蓋上述的預設層級規則。**

在分配層級之前，從程式碼、文件和架構中判定系統的部署模型。在 `0.1-architecture.md` 中記錄 **部署分類 (Deployment Classification)** 和 **元件暴露表 (Component Exposure Table)**。

**部署分類及其層級影響：**

| 分類 | 描述 | 允許 T1？ | 最低必要條件 |
|----------------|-------------|-------------|------------------|
| `LOCALHOST_DESKTOP` | 控制台/GUI 應用程式、無網路監聽器（或僅限本地）、單一使用者工作站 | ❌ **不允許** — 所有發現事項均為 T2+ | `Host/OS Access` (T3) 或 `Local Process Access` (T2) |
| `LOCALHOST_SERVICE` | 僅綁定到 127.0.0.1 的守護程序/服務 | ❌ **不允許** — 所有發現事項均為 T2+ | `Local Process Access` (T2) |
| `AIRGAPPED` | 無網際網路連線 | ❌ 對於源自網路的攻擊 | `Internal Network` |
| `K8S_SERVICE` | 具有 ClusterIP/LoadBalancer 的 Kubernetes 部署 | ✅ 允許 | 取決於服務類型 |
| `NETWORK_SERVICE` | 公開 API、雲端端點、面向網際網路 | ✅ 允許 | `None`（若無驗證） |

**`0.1-architecture.md` 中的元件暴露表設定了每個元件的最低必要條件。** 任何威脅或發現事項的必要條件都不能低於該表所允許的條件。此表在第 1 步填寫，對所有後續分析步驟具有約束力。

**遺留覆蓋表（仍作為備案適用）：**

| 部署指標 | 層級覆蓋規則 |
|---------------------|-------------------|
| 僅綁定到 `localhost`/`127.0.0.1` | 不能為 T1 — 需要本地存取（最低 T2） |
| 隔離網路 (Air-gapped) / 無網際網路 | 將基於網路的攻擊降低一個層級 |
| 單一管理員工作站工具 | 除非可由非管理員本地使用者利用，否則不能為 T1 |
| 單機上的 Docker/容器 | Docker socket 存取 = T2（需要本地管理員） |
| 具名管道 (Named pipe) / Unix socket | 不能為 T1 — 需要本地程序存取 |

**如何應用：**
1. 在第 1 步（內容收集）中，識別部署模型並記錄在 0.1-architecture.md
2. 在第 6/7 步（發現事項驗證）中，根據上表檢查每個 T1 候選者
3. 如果適用任何覆蓋規則，降級為 T2（若有多個則為 T3）
4. 在發現事項的說明中記錄覆蓋理由

**範例：** 隔離工作站上的 Kusto 容器，監聽連接埠 80 且無驗證：
- 預設分類：T1（未經驗證，連接埠 80）
- 覆蓋：僅限本地 + 單一管理員 → **T2**（攻擊者需要對管理員工作站的本地存取權限）

**不要對以下情況進行覆蓋：**
- Kubernetes 服務（任何 pod 都能接觸它們 → 橫向移動是現實的 → 保持 T1）
- 網路暴露的 API（任何網路使用者都能接觸它們 → 保持 T1）
- 雲端端點（公開網際網路 → 保持 T1）
- **網路暴露的 API**：監聽連接埠上的未經驗證 API 是第 1 層。

第 1 層的必要條件是 `None` — 意指**未經身分驗證的外部攻擊者**在沒有事先存取權限的情況下即可存取。如果利用漏洞需要本地管理員存取權限、OS 層級存取權限或實體接觸，則不能是第 1 層。

## 發現事項分類

在記錄每個發現事項之前，請驗證：

- [ ] **存在正面證據**：您能否顯示證明漏洞的組態/程式碼？
- [ ] **非安全預設值**：您是否已檢查平台是否預設啟用安全性？
- [ ] **已檢查安全基礎架構**：您是否尋找過 Sentry/cert-manager/Vault 等？
- [ ] **明確 vs 隱含**：安全性是明確停用的，還是只是未明確啟用的？
- [ ] **已諮詢平台文件**：不確定時，請查閱官方文件進行驗證

**分類結果：**
- **已確認 (Confirmed)**：漏洞的正面證據 → 記錄為 `3-findings.md` 中的發現事項
- **需驗證 (Needs Verification)**：無法確認但有潛在風險 → 新增到 `0-assessment.md` 中的「需驗證」
- **非發現事項 (Not a Finding)**：確認預設安全或已明確啟用 → 不予記錄

### 補救心力 (Remediation Effort)
- **低 (Low)**：組態變更、旗標切換或單一檔案修復
- **中 (Medium)**：多檔案程式碼變更、新的驗證邏輯或相依性更新
- **高 (High)**：架構變更、新元件或跨團隊協作

### STRIDE 範圍規則
- **外部服務** (AzureOpenAI, AzureAD, Redis, PostgreSQL) **確實**會有 STRIDE 章節 — 從您的系統角度來看，它們是攻擊面
- **外部執行者** (Operator, EndUser) **不會**有 STRIDE 章節 — 他們是威脅來源，而非目標
- 如果您有 20 個元素，其中 2 個是外部執行者，則您要撰寫 18 個 STRIDE 章節

**⚠️ 不要包含時間估計。** 絕不要在輸出中的任何地方新增 "(hours)", "(days)", "(weeks)", "~1 hour", "~2 hours" 或任何持續時間/修復心力估計。心力層級（低/中/高）已足夠。

### 緩解類型 (OWASP 對齊)
- **重新設計 (Redesign)**：透過變更架構消除威脅 (OWASP: Avoid)
- **標準緩解 (Standard Mitigation)**：應用眾所周知且經過驗證的安全控制措施 (OWASP: Mitigate)
- **自訂緩解 (Custom Mitigation)**：實作針對此系統的特定程式碼修復 (OWASP: Mitigate)
- **現有控制措施 (Existing Control)**：團隊已建立了解決此威脅的控制措施 — 請記錄它 (OWASP: Fix)
- **接受風險 (Accept Risk)**：承認並記錄殘留風險（需要理由） (OWASP: Accept)
- **轉移風險 (Transfer Risk)**：將責任轉移給使用者/運算子/第三方（例如：組態選擇、SLA） (OWASP: Transfer)
