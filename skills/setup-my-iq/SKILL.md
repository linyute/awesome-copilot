# Setup My IQ: 建立與更新上下文檔案庫

建立並維護個人上下文檔案庫：一組結構化 markdown 檔案，代表您是誰、您如何工作以及什麼對您重要。這些檔案可以存在於磁碟上的任何位置（OneDrive 資料夾、git 倉庫、本地目錄等）。將它們與代理連繫起來的是使用者自定義指令檔案（`AGENTS.md`、`copilot-instructions.md`、`CLAUDE.md` 或主機加載到會話中的任何 `*.instructions.md`）中的一個指標。

## 此技能如何工作

調用時，確定目前狀態並採取正確的行動。

### 發現現有上下文

掃描會話以查找對此技能建立的檔案的引用。主機已經將所有相關的自定義指令檔案合併到會話中：`AGENTS.md`（工作區或使用者範圍）、`copilot-instructions.md`、`CLAUDE.md`、任何 `*.instructions.md` 檔案以及類似的主機特定等效項。指標來自哪個檔案並不重要，只需查找檔案名稱即可：

- `identity.md`
- `role-and-responsibilities.md`
- `team.md`
- `tools-systems-and-config.md`
- `communication-style.md`
- `preferences-and-constraints.md`

對於每個檔案，在加載的指令文本中的任何位置查找一個 `@<絕對路徑>`（或等效的路徑引用）。那就是檔案的位置。不要依賴路徑旁邊的標籤或變數名稱：使用者可能對指標進行不同的命名（`identityProfile`、`me`、`who-i-am` 等）。匹配路徑末尾的檔案名稱。

對於每個檔案名稱，分類為：

- **未引用** — 會話中未加載此檔案名稱的路徑。
- **已引用，檔案缺失** — 已加載路徑，但磁碟上不存在該檔案。
- **存在** — 已加載路徑且檔案存在。讀取它以檢查佔位符。

如果這六個檔案名稱處於混合狀態，請按此優先順序處理它們：首先透過訪談建立任何「未引用」的檔案，然後填充「已引用，檔案缺失」的條目，最後修復「存在」檔案中的不完整欄位。在開始之前向使用者總結組合狀態，以便他們知道可以期待什麼。

具體例子。假設 `identity.md` 和 `team.md` 為「存在」且完整，`tools-systems-and-config.md` 為「存在」但在 ADO 部分有 TODO 佔位符，`role-and-responsibilities.md` 和 `communication-style.md` 為「已引用」但磁碟上不存在該檔案，且 `preferences-and-constraints.md` 為「未引用」。該技能應告訴使用者：「您已經完成了身分和團隊。工具配置有缺口。角色和溝通風格已引用但缺失。偏好尚未設定。我將首先就偏好對您進行訪談，然後建立角色和溝通風格，最後填補工具配置中的缺口。可以嗎？」

### 1. 未引用上下文檔案
加載的指令中沒有出現任何預期的檔案名稱。將此視為首次設定。運行下方的 **首次設定**。

### 2. 某些檔案已引用但磁碟上缺失
使用者之前開始了設定但未完成，或者路徑存在但背後沒有檔案。**從他們離開的地方繼續：**
- 列出哪些檔案存在，哪些缺失。
- 告訴使用者：「您已有 {existing files}。仍然需要： {missing files}。想要從您離開的地方繼續嗎？」
- 選擇下一個要訪談的檔案時，*發現現有上下文* 中的優先順序仍然適用。在平局中，使用此預設順序：identity -> role -> team -> tools -> communication-style -> preferences-and-constraints。

### 3. 檔案存在但有不完整欄位

檔案存在，但某些欄位包含未填充的佔位符數值。**填補缺口：**

- 掃描所有引用的檔案，查找以下任何不完整欄位模式：
  - `<!-- TODO -->` — 明確的佔位符
  - 形式為 `<!-- ... -->` 的 HTML 註解，用作替身數值（例如：`<!-- 您的姓名 -->`、`<!-- 組織名稱 -->`、`<!-- 經理姓名 -->`）
  - 任何僅包含 HTML 註解的表格單元格或欄位
- 列出缺口：「我在 {files} 中發現了不完整欄位。想要填充它們嗎？」
- 對於每個缺口，詢問該欄位的具體問題（而不是完整的訪談）。
- 在適當的位置使用使用者的答案更新檔案。

### 4. 檔案已完成，使用者想要更新
使用者要求更改某些內容，或者順便提到了上下文更改（例如：「我的經理換了」、「我們僱傭了新人」、「我搬到了另一個團隊」）。**有針對性的更新：**
- 如果使用者在執行另一項任務期間順便提到了更改，請提議：「聽起來您的 {file} 可能需要更新。要我現在修復嗎？」
- 如果使用者明確要求，請直接繼續。
- 從 *發現現有上下文* 中解析出的路徑讀取檔案。
- 進行編輯並與使用者確認。

### 5. 檔案已完成，使用者想要全面刷新
使用者想要從頭開始重做某個檔案。按照與首次設定相同的步驟，為該特定檔案重新運行訪談。

## 首次設定

此技能中的所有 `assets/...` 路徑均相對於此 SKILL.md 的目錄。使用您的檔案讀取工具從這些路徑加載捆綁的模板。如果由於某種原因模板不可讀，請回退到此檔案底部示例演練中顯示的欄位結構。

### 1. 選擇上下文目錄

詢問：「您的上下文檔案應該存放在哪裡？任何持久的位置都可以：同步雲端資料夾（OneDrive、Dropbox、iCloud）、本地目錄或 git 倉庫。只需挑選一個您以後可以再次找到的路徑即可。」

- 如果目錄不存在，則建立它。
- 存儲路徑以供後續步驟使用。
- 在 macOS 或 Linux 上，建議將 `~/my-iq-context` 或 `~/OneDrive/my-iq-context` 作為預設值。在技能稍後產生的任何路徑中，用 `$HOME` 替換 `%USERPROFILE%`。

### 2. 開始訪談

介紹過程：

「我將對您進行訪談並產出您的個人上下文檔案庫：一組代表您是誰、您如何工作以及什麼對您重要的 markdown 檔案。任何 AI 技能、代理或插件都可以讀取這些檔案並立即了解其工作對象。

我們將一次處理一個檔案。我會問您問題，起草檔案，然後請您告訴我哪裡做錯了。您可以跳過現在不想處理的任何檔案。準備好開始了嗎？」

### 3. 從可用數據源預填事實欄位

**在提問任何訪談問題之前，此步驟是強制性的。** 即使您預計它不會返回任何內容，也不要跳過它。

在詢問使用者任何問題之前，請嘗試從您的環境提供的任何連接源收集事實數據。使用者不應該輸入連接系統已經知道的資訊。

**尋找什麼（按檔案）：**

| 檔案 | 要預填的事實欄位 |
|------|---------------------------|
| `identity.md` | 姓名、角色/職稱、組織、團隊、經理 |
| `role-and-responsibilities.md` | 支援的團隊、節奏（來自日曆）、匯報線 |
| `team.md` | 直接下屬、頻繁合作者、組織架構圖數據 |
| `tools-systems-and-config.md` | ADO 組織/專案/團隊/區域路徑、倉庫、使用的工具 |
| `communication-style.md` | 來自郵件/文件的寫作樣本（先徵得許可） |
| `preferences-and-constraints.md` | 無預填（這些是主觀邊界） |

**如何執行：**

1. 檢查您的環境中哪些數據源可用（工作個人資料、目錄、日曆、ADO、郵件等）。僅使用已授權的來源。
2. 對於您即將進行訪談的檔案，查詢相關事實欄位的來源。
3. 將發現結果作為提案呈現：「我從您的工作個人資料中提取了一些資訊。這是我的發現 — 請在我們繼續之前確認或更正。」
4. 將每個數值標記為自動填充，以便使用者知道什麼來自系統，什麼是您要求他們提供的。
5. 僅在確認預填提案後（或者如果沒有可用來源），才繼續詢問剩餘問題。

**如果沒有可用數據源：** 明確說明（「在此環境中，我無權訪問工作個人資料或目錄，因此我將直接詢問您」）並繼續進行訪談問題。不要默默跳過此步驟。

請參閱訪談規則下 *提問前從可用數據源預填* 部分中的完整規則和限制。

### 4. 為 identity.md 進行訪談

從此技能捆綁包中的 `assets/templates/identity.md` 加載檔案結構模板。

**首先：應用步驟 3（預填）。** 查詢可用來源以獲取姓名、角色、組織、團隊和經理。呈現任何發現結果以供確認。然後針對預填未涵蓋的欄位繼續進行以下提問。

要詢問的開放式問題（這些總是需要使用者）：

- 如果您必須向晚宴上的某人解釋您實際做什麼，您會說什麼？
- 人們為了什麼來找您？

僅在預填未涵蓋事實問題時詢問：

- 您的姓名和目前的角色是什麼？
- 您在哪個組織或公司工作？
- 您在哪個團隊？
- 您的經理是誰？

一旦您有足夠的資訊起草（不要在此之後繼續提問），請起草檔案。透過反應測試（見下方的訪談規則）呈現它。修訂，然後寫入上下文目錄。

過渡：「身分部分已完成。接下來是角色與職責，它捕捉了您每週的實際情況。準備好了嗎？」

### 5. 為 role-and-responsibilities.md 進行訪談

加載模板 `assets/templates/role-and-responsibilities.md`。

**首先：應用步驟 3（預填）。** 查詢可用來源以獲取支援的團隊、定期日曆節奏和匯報線。在提問前呈現任何發現結果以供確認。

關鍵問題：

- 請帶我了解典型的一週。有哪些定期發生的事情？
- 您支援哪些團隊或小組？
- 對於每個團隊：您負責什麼？您執行什麼節奏？
- 您定期做出哪些決定？不是重大的戰略決策，而是例行公事。
- 您產出什麼？報告、電子郵件、儀表板？
- 您向誰匯報？
- 是否有塑造您工作的每月或每季度節律？

一旦您有足夠的資訊起草，請起草。按團隊組織：每個團隊都有自己的職責、節奏和交付物部分。呈現、修訂、寫入。

過渡：「角色與職責部分已完成。接下來是團隊，涵蓋您工作生活中的關鍵人物。準備好了嗎？」

### 6. 為 team.md 進行訪談

加載模板 `assets/templates/team.md`。

**首先：應用步驟 3（預填）。** 查詢可用來源以獲取直接下屬、組織架構圖、頻繁合作者（來自日曆/聊天）。在提問前呈現任何發現結果以供確認。

關鍵方法：

- 誰是與您互動最多的人？從您自己的團隊開始。
- 詢問團隊名冊（使用者可以粘貼電子郵件列表）
- 對於每個團隊，識別負責人和層級結構
- 對於關鍵人物：他們需要您提供什麼，您需要他們提供什麼？
- 在已知情況下記錄每個人的關注領域或支柱
- 記錄互動模式（1:1、站會、同步會）
- 是否有任何代理應該知道的關於與特定人員合作或為與其互動做準備的事情？

使用來自先前檔案的資訊。如果他們在角色訪談期間提到了合作者，請在此處引用他們，而不是重新詢問。起草、呈現、修訂、寫入。

過渡：「團隊部分已完成。最後一個：工具、系統與配置。這是我們記錄所有內容如何設定的地方。準備好了嗎？」

### 7. 為 tools-systems-and-config.md 進行訪談

加載模板 `assets/templates/tools-systems-and-config.md`。

**首先：應用步驟 3（預填）。** 查詢可用來源以獲取 ADO 組織/專案/團隊/區域路徑、倉庫和工具配置。在提問前呈現任何發現結果以供確認。

關鍵問題：

- 您每天使用哪些工具和平台？
- 您如何組織會議記錄？它們存放在哪裡？
- 哪些標籤或模式可以按類型和團隊對您的會議進行分類？

然後針對使用者支援的每個工作流/團隊（使用來自角色與職責訪談的資訊）：

如果使用者支援三個以上的工作流，請先詢問他們是想現在配置所有工作流，還是先從最重要的一個或兩個開始，稍後再添加其餘工作流。對於類似的工作流，詢問是否適用相同的配置，並僅針對差異進行訪談。

- 此工作流是否使用 Azure DevOps？如果是：哪個組織、專案、團隊、區域路徑？如果不是，請跳過此工作流的 ADO 部分。
- （僅限 ADO 工作流）是否有應從衝刺計分卡中排除的工作項標題模式？（例如：估算或審閱佔位符項）
- 此工作流的戰略支柱或部分是什麼？這些在每週更新和衝刺報告中用作分組標題。即使工作流不使用 ADO，也要定義它們。
- 對於 ADO 工作流：哪些 ADO Epic 對應於每個支柱？
- 衝刺更新保存在哪裡？檔案前綴是什麼？
- 每週更新保存在哪裡？檔案前綴是什麼？
- 此工作流是否有路線圖檔案？

將常用設定放在頂部，特定於工作流的配置放在下方的各個部分中。起草、呈現、修訂、寫入。

過渡：「工具與配置部分已完成。接下來是溝通風格，以便代表您寫作的任何代理聽起來都像您。準備好了嗎？」

### 8. 為 communication-style.md 進行訪談

加載模板 `assets/templates/communication-style.md`。

這部分比其他部分更難。人們用陳腔濫調描述自己的聲音。兩條更強大的路徑：詢問負面情況，以及提議分析真實的寫作樣本。兩者都用。

從提供樣本路徑開始：

> 「想要分享一些您真實寫作的樣本嗎？粘貼 2 到 5 個您發送過的內容：每週狀態更新、領導層電子郵件、Loop 更新、評論條目、您引以為豪的 Teams 訊息。我會閱讀它們，提取我看到的語音特徵，並提議應該在您的檔案中放入什麼。您在寫入任何內容之前進行批准。我不會複製原始內容，只會複製風格模式。」

如果使用者分享樣本：

- 僅閱讀每個樣本的風格：語域、句子長度、開頭、過渡、使用或避免的特定單詞、框架模式、他們如何處理功勞、他們如何處理壞消息。
- 請勿將原始樣本內容複製到檔案中。提取模式並將其作為檔案結構中的提議項目呈現。
- 呈現提議的草案：「這是我提取的內容。告訴我哪裡錯了或缺失了。」
- 照常應用反應測試。

如果使用者未分享樣本（或除了樣本之外），請一次詢問一個問題：

1. 「如果同事用一句話描述您的寫作方式，他們會說什麼？」
2. 「讀一下這句話：*'我會鼓勵你倚重這次機會，成為團隊真正的力量倍增器。'* 就您的語音而言，它有什麼問題？」（這比正面問題更快地找出避免清單。）
3. 「當您向領導層撰寫每週狀態時，您希望自動化哪些部分，哪些部分您最終總是會重寫？」
4. 「對您來說，最難寫信給誰？當您寫信給他們時，會有什麼變化？」
5. 「當您在 Connect 或績效評估中撰寫關於自己的內容時，您發現自己寫過並立即刪除的短語是什麼？」

起草、呈現、修訂、寫入。

寫入後，告訴使用者：

> 「我不會隨時間自動發現模式。如果您注意到我重複錯誤、交換短語或與您的編輯衝突，請要求我更新此檔案，我會將更正內容併入其中。此檔案在您要求時演變。」

過渡：「最後一個：偏好與限制。這是您希望 AI 在您身邊如何表現。準備好了嗎？」

### 9. 為 preferences-and-constraints.md 進行訪談

加載模板 `assets/templates/preferences-and-constraints.md`。

此處的大多數規則是關於代理不應單方面做什麼。透過回憶具體的挫折而不是詢問抽象的規則來引出它們。一次問一個問題：

1. 「上一次 AI 工具做了您不想要的事情是什麼時候？它做了什麼，它應該改做什麼？」（一個答案通常會產出 3 條邊界規則。）
2. 「代理在不詢問您的情況下做什麼是安全的？什麼絕對需要您的批准？」
3. 「速度還是徹底。如果必須選一個，選哪個？」
4. 「您是否希望 AI 為您做任何決定，還是您總是希望提供選項？」
5. 「您不希望 AI 在哪些地方代表您發言？」（Teams、電子郵件、會議、公共聊天。）
6. 「您是否使用任何具有持久記憶的工具（VS Code 會話記憶、代理記憶功能等）？如果是，有什麼是您想要存儲在那裡的，或者特別是不想存儲在那裡的？」如果答案是否定的，請跳過記憶部分。

起草、呈現、修訂、寫入。應用反應測試。

過渡：「訪談結束了。現在我將把這些檔案連繫到您的自定義指令中，以便每個代理都能找到它們。」

### 10. 將檔案連繫到自定義指令檔案中

向使用者的自定義指令檔案中添加指標條目，以便其他技能和代理可以找到新檔案。**建議使用使用者範圍，而不是工作區範圍。** 這種上下文的重點在於它隨處可用，而不是與一個倉庫綁定。

快速地圖：10a 檢測工具，10b 選擇目標檔案，10c（僅當有多個工具時）提議將它們符號連結到規範檔案，10d 使用捆綁的模板寫入指標。

在 macOS 或 Linux 上，下方的每個命令和路徑都有 POSIX 等效項。在每個路徑中用 `$HOME` 替換 `%USERPROFILE%`，使用 `ln -s <canonical-target> <harness-file>` 而不是 `New-Item -ItemType SymbolicLink`，並完全跳過開發者模式檢查（符號連結在 macOS 或 Linux 上不需要提升權限）。

**步驟 10a. 檢測使用者使用哪些工具。** 檢查是否存在以下任何使用者範圍路徑（使用檔案系統；無需先詢問）：

| 工具 | 使用者範圍自定義指令路徑 |
|---------|--------------------------------------|
| VS Code Copilot Chat / Claude Code | `%USERPROFILE%\.claude\CLAUDE.md` |
| GitHub Copilot CLI | `%USERPROFILE%\.copilot\copilot-instructions.md` |

VS Code Copilot Chat 預設不會讀取 `%USERPROFILE%\.agents\AGENTS.md`，但它確實會讀取 `%USERPROFILE%\.claude\CLAUDE.md`，這與 Claude Code 使用的檔案相同。因此 `.claude\CLAUDE.md` 是這兩個工具共享的使用者範圍檔案。

如果 `%USERPROFILE%\.agents\AGENTS.md` 存在，則將其視為規範目標。在某些設定中，上述特定於工具的檔案會符號連結到它，因此單次寫入會流向每個連繫的工具。

**步驟 10b. 選擇目標。** 應用第一條匹配的規則：

1. 如果 `%USERPROFILE%\.agents\AGENTS.md` 存在，則預設使用它。特定於工具的檔案是它的符號連結，因此單次寫入會流向每個連繫的工具。在寫入前與使用者確認。
2. 否則，如果恰好存在一個特定於工具的檔案，則提議寫入它。確認。同時提到步驟 10c 中的規範加符號連結選項。
3. 否則，如果兩個特定於工具的檔案都存在（`copilot-instructions.md` 和 `CLAUDE.md`），列出它們並詢問要更新哪一個。強烈建議建立 `.agents\AGENTS.md` 作為規範檔案，並將其他檔案符號連結到它，以便單次編輯即可觸及所有工具（步驟 10c）。
4. 否則（任何地方都不存在使用者範圍自定義指令檔案），詢問：「您最常使用哪種 AI 工具：VS Code Copilot Chat、Claude Code、GitHub Copilot CLI 還是其他？」然後建立匹配的使用者範圍檔案（前兩個為 `.claude\CLAUDE.md`，Copilot CLI 為 `.copilot\copilot-instructions.md`），並帶有 `## Personal Context` 部分。

如果使用者明確要求工作區範圍，請寫入 `<workspace>\AGENTS.md` 並告訴他們權衡：它僅適用於該倉庫。

**步驟 10c. 提議將其他工具符號連結到規範檔案。** 僅當使用者擁有（或想要）多個工具時才有意義。詢問：「要我將 `%USERPROFILE%\.agents\AGENTS.md` 作為規範檔案，並將您的其他工具檔案符號連結到它嗎？這樣您只需編輯一次，每個工具都能看到更改。」

如果是，對於使用者想要連結的每個其他工具檔案：

1. **先決條件。** Windows 上的檔案符號連結需要啟用開發者模式或提升權限的 PowerShell。使用以下命令檢查開發者模式：

   ```powershell
   Get-ItemPropertyValue 'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\AppModelUnlock' AllowDevelopmentWithoutDevLicense -ErrorAction SilentlyContinue
   ```

   如果開發者模式已開啟，則返回 `1`。如果未開啟，請向使用者提供兩個選項並讓其選擇：

   - **選項 A：啟用開發者模式**（一次性，無需重新啟動）。
     設定 -> 隱私與安全性 -> 開發者選項 -> 開啟開發者模式。或在提升權限的 PowerShell 中運行此命令：

     ```powershell
     New-ItemProperty -Path 'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\AppModelUnlock' -Name AllowDevelopmentWithoutDevLicense -Value 1 -PropertyType DWord -Force
     ```

   - **選項 B：自己在提升權限的 PowerShell 中運行符號連結命令。** 給他們準確的命令（填入真實路徑），並請他們在以管理員身分打開的 PowerShell 視窗中運行。然後在他們確認成功後繼續：

     ```powershell
     New-Item -ItemType SymbolicLink -Path '<harness-file>' -Target "$env:USERPROFILE\.agents\AGENTS.md"
     ```

   在未開啟開發者模式或未確認使用者自己運行了提升權限的命令之前，請勿繼續。
2. **將現有檔案備份** 到 `<file>.bak-<yyyyMMddHHmmss>`，以免丟失任何內容。絕不刪除。
3. **建立指向 `%USERPROFILE%\.agents\AGENTS.md` 的符號連結**（如果使用者在選項 B 下已自己運行，則跳過）：

   ```powershell
   New-Item -ItemType SymbolicLink -Path '<harness-file>' -Target "$env:USERPROFILE\.agents\AGENTS.md"
   ```

上述內嵌符號連結步驟足以進行上下文連繫。

如果使用者拒絕符號連結，請繼續執行按工具寫入並繼續。不要推銷兩次。

**步驟 10d. 寫入指標。** 使用 `assets/templates/AGENTS.md`（相對於此 SKILL.md）中捆綁的 AGENTS.md 模板作為 `## Personal Context` 區塊結構（標題、分錄標籤、路由表、規則）的事實來源。不要重新發明格式。

行為：

- **如果目標檔案尚未存在於磁碟上**，請建立它。建立任何缺失的父目錄（例如：`%USERPROFILE%\.copilot\` 或 `%USERPROFILE%\.claude\`），將整個捆綁的模板複製到目標路徑中，將 `<CONTEXT_DIR>` 替換為步驟 1 中選擇的上下文目錄的絕對路徑，然後寫入。
- **如果目標檔案存在但沒有 `## Personal Context` 部分**，請將該部分附加到檔案末尾（如果 `## Safety` 部分也缺失，則從模板添加），並替換 `<CONTEXT_DIR>`。不要觸動其上方現有的內容。
- **如果 `## Personal Context` 部分已存在**，僅添加或更新更改的具體分錄。不要重寫整個部分。保留現有的標籤和順序；僅更換 `@<path>` 行。
- 僅連繫磁碟上實際存在的檔案。不要為使用者跳過的檔案添加 `@<path>` 指標。

### 11. 確認並總結

「那是您的上下文檔案庫。任何讀取您自定義指令的 AI 技能、代理或插件都可以找到這些檔案並立即了解其工作對象。

隨著您隨時間更新檔案，檔案會變得更好。專案會變化，優先順序會轉移，特別是您的溝通風格檔案，當您發現我重複錯誤並要求我將更正內容併入其中時，它會演變。將這些視為動態文件，而不是成品。」

向使用者展示：
- 建立的檔案列表及其路徑。
- 寫入指標的自定義指令檔案路徑。

## 訪談規則

### 語氣
直接、溫暖且具體。您是訪談者，而不是教練。不要對使用者的回答發表評論、讚美或提供意見。

### 提問前從可用數據源預填
在引導使用者完成問題列表之前，請檢查您的任何可用工具或數據源是否可以為您回答事實欄位。使用者不應該輸入連接系統已經知道的資訊。

常見例子（使用您的環境實際提供的任何內容）：

- 連接的工作個人資料或目錄可能知道使用者的姓名、職稱、組織、團隊、經理和直接下屬 — 對於 identity.md 和 team.md 很有用。
- 日曆、郵件或聊天記錄可能會顯示頻繁的合作者和定期節奏 — 對於 team.md 和 role-and-responsibilities.md 很有用。
- 專案管理或源代碼控制集成可能知道使用者的組織/專案/區域路徑或倉庫 — 對於 tools-systems-and-config.md 很有用。
- 郵件或文件中的現有寫作樣本（徵得使用者許可）可以為 communication-style.md 提供資訊，而無需新的樣本。

規則：

1. 僅使用使用者已授權的來源。不要在訪談中未經詢問就尋找任何新內容。
2. 預填是提議，而不是答案。展示您的發現，將其標記為自動填充，並在將每個數值放入檔案之前請使用者確認或更正。
3. 不要向使用者枚舉工具名稱。說一些類似於「我可以從您的工作個人資料中提取其中一些內容」的話，而不是列出具體的產品名稱。
4. 當沒有可用來源，或者數據模糊、陳舊或敏感時，請回退到直接詢問。
5. 開放式欄位（您實際做什麼、人們為了什麼來找您、語音、邊界）仍然需要詢問。不要從間接信號中虛構它們。

### 問題
- 一次問一個問題。切勿呈現問題列表。
- 切勿詢問複合問題。
- 使用您在先前檔案中學到的內容來為後續訪談提供資訊。不要重複詢問您已經知道的事情。
- 如果使用者談論到對後續檔案有用的題外話，請記錄下來並在到達該處時使用。
- 當使用者給出模糊的答案時，爭取具體細節：「您能給我舉個例子嗎？」或「這在日常生活中實際是什麼樣子的？」
- 當您有足夠的資訊起草時，請說明並起草。不要繼續提問。「足夠」意味著您的答案涵蓋了該檔案模板中的核心欄位（對於 identity：姓名、角色、組織、團隊、經理以及他們做什麼）。如果早期的回答已經涵蓋了資訊，則不需要遍歷所有列出的問題。

### 反應測試
在起草每個檔案後，呈現它並說：
「這是我的草案。閱讀一下，告訴我聽起來不對勁的地方。任何感覺不對的地方、我假設錯的地方、任何缺失的地方。」

如果使用者說看起來不錯，沒有任何更改，請推動一次：
「挑選一個最弱或最通用的句子。什麼能讓它更具體地代表您？」

如果他們再次推託並說沒問題，請接受並繼續。

### 檔案品質
- 保持檔案簡潔。一頁，而不是五頁。
- 這些檔案聽起來應該像使用者，而不是像 AI 在寫關於他們的內容。
- 使用他們的語言、他們的架構、他們的正式程度。
- 絕不使用長破折號 (em dashes)。
- 如果使用者想跳過某個檔案，請隨其便。
- 如果使用者需要在檔案中途停止，請告訴他們您的進度，以便他們稍後恢復。
