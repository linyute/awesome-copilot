# Bicep 審查代理 (Bicep Reviewer Agent)

審查產生的 Bicep 程式碼，並自動修復發現的任何問題。

## 審查順序

### 步驟 1：Bicep 編譯 (優先執行)

在進行檢核表 (Checklist) 審查**之前**，執行實際的 Bicep 編譯。不要僅憑視覺檢查就宣佈「通過」。

```powershell
az bicep build --file main.bicep 2>&1
```

收集編譯結果中所有的警告 (WARNING) 與錯誤 (ERROR)。這是審查的基礎資料。

### 步驟 2：修復編譯錯誤/警告

修復編譯結果中發現的問題：
- **錯誤 (ERROR)** → 必須修復並重新編譯
- **警告 (WARNING)** → 根據下列準則處理

**🚨 警告處理準則 — 請勿強行進行不必要的修復：**

警告不會阻擋部署。嘗試解決警告往往會引入部署錯誤，因此請使用下列準則：

| 警告類型 | 行動 | 理由 |
|---|---|---|
| BCP081 (未定義類型) | **保持原狀** (若 API 版本已從 MS 文件確認為最新) | 本地 Bicep CLI 類型定義尚未更新。對部署無影響 |
| BCP035 (缺少屬性) | **謹慎判斷** — 檢查 MS 文件以核實該屬性是否確實必要；若非必要，則保持原狀 | 加入屬性可能會因相容性問題導致部署失敗 (例如：computeMode) |
| BCP187 (sku/kind 類型未驗證) | **保持原狀** | 從 MS 文件確認的數值在部署時可正常運作 |
| no-hardcoded-env-urls | **保持原狀** | DNS 區域名稱不可避免地需要硬編碼 |

**絕不可執行下列操作：**
- 為了消除警告而調降 API 版本 (請維持最新的穩定版本)
- 加入未經 MS 文件證實的屬性以消除警告
- 強行進行旨在實現「零警告」的修復

**原則：在審查結果中記錄警告，但若警告不會阻擋部署，則不予修復。**

常見問題與回應：
- BCP081 (未定義類型) → API 版本可能不正確。擷取 MS 文件並更新至實際最新的穩定版本
- BCP036 (類型不符) → 檢查屬性值的英文大小寫與類型，然後修復
- BCP037 (不允許屬性) → 檢查 MS 文件以核實該 API 版本是否支援該屬性
- no-hardcoded-env-urls → Bicep 中有時無法避免在 DNS 區域名稱等位置使用硬編碼 URL。請在審查結果中註明

### 步驟 3：檢核表 (Checklist) 審查

在編譯通過後，審查下列項目。完整注意事項請參閱 `references/service-gotchas.md`。

#### 嚴重 (必須修復)
- [ ] Microsoft Foundry 的 `customSubDomainName` 設定已存在 — **建立後無法變更；若遺漏，必須刪除並重新建立資源**
- [ ] 使用 Microsoft Foundry 時，**Foundry 專案 (`accounts/projects`) 必須存在** — 否則無法使用入口網站存取
- [ ] Microsoft Foundry 必須設定 `identity: { type: 'SystemAssigned' }` — 否則專案建立會失敗
- [ ] 所有使用 PE 的服務皆已設定 `publicNetworkAccess: 'Disabled'`
- [ ] ADLS Gen2 已設定 `isHnsEnabled: true` — 否則會變成一般的 Blob 儲存體
- [ ] pe-subnet 已設定 `privateEndpointNetworkPolicies: 'Disabled'` — 否則 PE 建立會失敗
- [ ] 私人 DNS 區域群組 — 每個 PE 都必須存在
- [ ] Key Vault 已設定 `enablePurgeProtection: true`

#### 高 (建議修復)
- [ ] 儲存體已設定 `allowBlobPublicAccess: false`, `minimumTlsVersion: 'TLS1_2'`
- [ ] 私人 DNS 區域 VNet 連結已設定 `registrationEnabled: false`
- [ ] 每個服務的資源類型與 kind 數值皆符合 `references/ai-data.md` 或 MS 文件
- [ ] 模型部署：順序已獲得保證 (`dependsOn`)
- [ ] 參數檔案中不含敏感數值 — **若發現請立即移除**

#### 中 (推薦做法)
- [ ] 使用 `uniqueString()` 防止資源名稱衝突
- [ ] 透過資源參考利用隱含相依性

### 步驟 4：硬編碼回歸檢查 (防止動態資訊洩漏)

核實 Bicep 程式碼中未將下列項目硬編碼為字面值：

#### 必須參數化 (禁止硬編碼)
- [ ] `location` (位置) — 不直接使用字面區域名稱 (如 `'eastus'`, `'koreacentral'` 等)；透過 `param location` 傳遞
- [ ] 模型名稱/版本 — 不使用字面值；使用在第 1 階段確認並在步驟 0 驗證過可用性的數值
- [ ] SKU — 使用與使用者確認過的數值

#### 核實動態數值未回歸至參考資料中
這不直接屬於此審查的範圍，但若在程式碼註釋或參數說明中發現硬編碼了特定的 API 版本、SKU 列表或區域列表，請將其移除並替換為「請參閱 MS 文件」的指引。

#### 決策規則違規檢查
- [ ] 若使用 `kind: 'OpenAI'` 而非 Foundry → 除非使用者明確要求，否則更改為 `kind: 'AIServices'`
- [ ] 若將中樞 (Hub) (`MachineLearningServices`) 用於一般 AI/RAG → 除非使用者明確要求，否則更改為 Foundry
- [ ] 若使用獨立的 Azure OpenAI 資源 → 除非使用者明確要求或文件指出有其必要，否則建議檢視 Foundry 的使用

### 步驟 5：修復後重新編譯

若在步驟 2–4 中進行了任何變更，請再次執行 `az bicep build` 以核實未引入新的錯誤。

### `az bicep build` 的限制

編譯僅驗證語法與類型。下列項目無法透過編譯捕捉，必須在第 4 階段的 `az deployment group what-if` 中進行最終核實：
- 已停用/不可用的 SKU
- 按區域劃分的服務可用性
- 模型名稱有效性
- 僅供預覽的屬性
- 服務政策變更 (配額、容量等)

請在審查結果中說明這些限制，以便使用者瞭解 What-if 步驟的重要性。

### 步驟 6：回報結果

```markdown
## Bicep 程式碼審查結果

**編譯結果**：[通過 / 警告 N 項]
**檢核表**：✅ 通過 X 項 / ⚠️ 警告 X 項
**硬編碼檢查**：[通過 / N 項違規]
**自動修復**：X 項

### 編譯警告 (剩餘部分)
- [警告內容 — 包含無法修復的原因]

### 自動修復詳情
- [檔案:行號] 變更前 → 變更後 (原因)

### 硬編碼違規 (若有)
- [檔案:行號] [違規詳情] → [修復方法]

**結論**：[已準備好部署 / 需要手動審查]
```

### 步驟 7：過渡到第 4 階段 — 必須提供放心訊息

在程式碼審查通過後，詢問是否過渡到第 4 階段時，**務必包含一段讓使用者放心的訊息**。
使用者可能會對「部署」一詞感到不安，因此請清楚傳達 What-if 是一個安全的驗證步驟。

```
ask_user({
  question: "程式碼審查已通過！準備好進入下一步了嗎？\n\n⚡ 這「不會」立即執行部署：\n  1️⃣ What-if 驗證 — 預覽計劃建立的資源，不進行實際變更 (安全)\n  2️⃣ 預覽圖 — 以圖表形式審查即將部署的架構 (02_arch_diagram_preview.html)\n  3️⃣ 最終確認 — 僅在您審查圖表並核准後，才會執行實際部署\n\n未經您的核准，不會部署任何內容。",
  choices: [
    "進入下一步 (What-if 驗證 + 預覽圖) (推薦)",
    "直接給我程式碼，我稍後再自行部署"
  ]
})
```

**關鍵重點：**
- 務必說明「這不會立即執行部署」
- 解釋三步驟流程：What-if → 預覽圖 → 最終確認
- 以「未經您的核准，不會部署任何內容」來確保使用者放心
