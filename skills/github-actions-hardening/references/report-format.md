# 報告格式 (Report Format)

為每次工作流安全強化審查使用此結構。

## 1. 摘要表（始終置頂）

```
GitHub Actions 安全強化 — <經審查的工作流檔案>

| 嚴重程度   | 計數 |
| ---------- | ----- |
| 🔴 嚴重 (CRITICAL) | 1     |
| 🟠 高 (HIGH)     | 2     |
| 🟡 中 (MEDIUM)   | 1     |
| 🔵 低 (LOW)      | 1     |
| ⚪ 資訊 (INFO)     | 0     |
```

如果未發現任何問題：`未發現問題。已檢查：觸發器、注入接收點、權限、Action 固定、秘密處理。`

## 2. 發現結果（按問題類型分組，而非按檔案）

為每個發現結果使用一張卡片：

```
### 🔴 嚴重 — 在高權限觸發器上透過 PR 標題進行指令注入

檔案：.github/workflows/triage.yml (第 14 行)
觸發器：pull_request_target

有問題的代碼：
    - run: echo "New PR: ${{ github.event.pull_request.title }}"

風險：pull_request_target 以讀/寫令牌和儲存庫秘密運行，任何貢獻者都可以開啟一個標題如 "; <attacker-command> # 的 PR，該標題將作為 Shell 執行。這允許秘密外洩和使用工作流令牌進行推送。

修復：
    - env:
        PR_TITLE: ${{ github.event.pull_request.title }}
      run: echo "New PR: $PR_TITLE"

信心：高
```

## 3. 修復前後對比區塊

每個「嚴重」和「高」風險發現結果都應包含具體的修復前後對比。保留原作者的縮進、步驟名稱和周圍結構 — 僅更改修復問題的部分，並在不明顯的地方添加一行註解解釋更改。

## 4. 結語

結尾請註明以下內容：

> 在提交之前請審閱每項更改。尚未修改任何內容。

## 風格規則

* 引用確切的有問題行並提供其位置。
* 用通俗易懂的語言解釋風險 — 攻擊者實際上做了什麼，而不僅僅是規則名稱。
* 每個發現結果的信心級別：高 / 中 / 低。
* 不要誇大嚴重程度：來自 Fork 的 `pull_request`（唯讀令牌，無秘密）執行不受信任的代碼本身並不構成「嚴重」。
