# 當 Qdrant 需要更多容量時該怎麼辦

優先考慮垂直擴充：操作更簡單，無網路開銷，根據維度和量化情況，單個節點可承載約 1 億個向量。在以下情況下考慮水平擴充：資料量超過單個節點容量、需要容錯能力、需要隔離租戶，或者是 IOPS 受限 (更多節點 = 更多獨立的 IOPS)。

## 最基礎的分散式組態

- 3 個節點，3 個分片，並設定 `replication_factor: 2` 以實現零停機時間縮放

至少 3 個節點對於達成共識和容錯非常重要。擁有 3 個節點，您可以損失 1 個節點而不會停機。若只有 2 個節點，損失 1 個節點會導致集合操作停機。
複製因子 (replication factor) 為 2 表示每個分片都有 1 個副本，因此您擁有 2 份資料。這允許進行零停機時間的縮放與維護。若 `replication_factor: 1`，則即使是點層級的操作也無法保證零停機，且叢集維護需要停機。

## 選擇分片數量

分片是資料分佈的單位。
更多的分片允許更多節點和更好的分佈，但會增加開銷。較少的分片可減少開銷，但會限制水平縮放能力。

對於 3-6 個節點的叢集，建議的分片數量為 6-12 個。
這允許每個節點分配 2-4 個分片，從而平衡分佈與開銷。

## 變更分片數量

適用於：分片數量無法被節點數量整除，導致分佈不均，或者需要重新平衡。

重新分片 (Resharding) 是昂貴且耗時的操作，應作為常規資料分佈無法實現時的最後手段。
重新分片旨在對使用者操作透明，更新和搜尋在重新分片期間應仍可運作，僅有微小的效能影響。

但重新分片操作本身非常耗時，且需要在節點之間移動大量資料。

- 在 Qdrant Cloud 中提供 [重新分片](https://search.qdrant.tech/md/documentation/operations/distributed_deployment/?s=resharding)
- 重新分片功能不適用於自行代管的部署。

更好的替代方案：初始時超額配置分片數量，或者啟動一個具備正確組態的新叢集並遷移資料。


## 應避免的做法

- 在竭盡垂直擴充選項前不要跳到水平擴充 (增加複雜度卻無實質收益)
- 不要設定無法被節點數量整除的 `shard_number` (會導致分佈不均)
- 如果您需要容錯能力，不要在生產環境使用 `replication_factor: 1`
- 不要只新增節點而不重新平衡分片 (使用 shard move API 進行重新分佈)
- 在沒有進行負載測試的情況下不要縮減 RAM 規模 (快取置換會導致長達數天的延遲事件)
- 不要因為每個租戶使用一個集合而達到集合限制 (請使用 Payload 分割/分區)
