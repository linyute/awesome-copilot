# CodeQL 警示管理參考

了解、審閱、關閉及解決由 CodeQL 產生的程式碼掃描警示指南。

## 警示嚴重性層級

### 標準嚴重性

所有程式碼掃描警示都具有以下嚴重性層級之一：

| 層級 | 描述 |
|---|---|
| `Error` | 應該修復的高信賴度、高影響性問題 |
| `Warning` | 中等信賴度或中等影響性的問題 |
| `Note` | 低信賴度或資訊性發現 |

### 安全性嚴重性

安全性警示另外具有衍生自 CVSS 分數的安全性嚴重性：

| 層級 | CVSS 分數範圍 | 描述 |
|---|---|---|
| `Critical` | > 9.0 | 需要立即處理的嚴重漏洞 |
| `High` | 7.0 – 8.9 | 應優先處理的重要漏洞 |
| `Medium` | 4.0 – 6.9 | 在正常工作流程中處理的中等漏洞 |
| `Low` | 0.1 – 3.9 | 安全影響有限的次要問題 |

當存在安全性嚴重性時，它在顯示和排序時優先於標準嚴重性。

### 如何計算安全性嚴重性

對於新增至預設或延伸套件的每個 CodeQL 安全性查詢：
1. 識別與查詢的 CWE 標記相符的所有 CVE
2. 計算這些 CVE 的 CVSS 分數的第 75 百分位數
3. 該分數成為查詢的安全性嚴重性
4. 數值分數根據 CVSS 定義對應到 Critical/High/Medium/Low per CVSS 定義

## 警示標籤

非應用程式程式碼中的警示會收到類別標籤：

| 標籤 | 描述 |
|---|---|
| **Generated** | 由建構程序產生的程式碼 |
| **Test** | 測試程式碼 (根據檔案路徑偵測) |
| **Library** | 函式庫或第三方程式碼 |
| **Documentation** | 文件檔案 |

這些標籤是根據檔案路徑自動指派的。它們無法手動覆蓋。

## 提取要求中的警示審閱

### 提取要求 (PR) 警示的工作原理

- 警示會以註釋形式出現在 **對話** 索引標籤和 **檔案已變更** 索引標籤中
- **程式碼掃描結果** 檢查會摘要所有發現
- 只有當所有識別出的行都存在於 PR diff 中時，警示才會出現在 PR 中
- 顯示已變更行上的新警示；不會顯示預先存在的警示

### PR 檢查失敗行為

預設情況下，如果警示的嚴重性為 `error`、`critical` 或 `high`，則檢查失敗。可透過儲存庫「設定」→「規則」→「規則集」→「程式碼掃描」來覆蓋此閾值。

### 合併保護

設定規則集以在以下情況下封鎖 PR 合併：
- 所需工具發現與嚴重性閾值相符的警示
- 所需工具的分析仍進行中
- 所需工具未針對該儲存庫進行設定

## Copilot 自動修復

GitHub Copilot 自動修復 (Autofix) 會自動為提取要求中的 CodeQL 警示產生修復建議。

### 可用性
- 所有公開儲存庫均可免費使用
- 具有 GitHub 程式碼安全授權的私人存放庫可用
- 不需要 Copilot 訂閱
- 支援 CodeQL 查詢的子集 (並非全部)

### 工作原理
1. 程式碼掃描偵測到 PR 中的警示
2. 警示資訊會傳送至 LLM 進行分析
3. 修復建議會以具有內嵌程式碼變更的 PR 留言形式發佈
4. 開發人員檢視、編輯並認可 (commit) 建議的修復

### 使用自動修復建議
- 按一下 **編輯** 以直接在 GitHub 上或透過 GitHub CLI 套用修復
- 使用 **檢視自動修復修補程式** 以在本地套用
- 在認可之前始終檢視並測試修復
- 修復可能包含對原始 PR diff 中不存在的檔案進行變更 (例如，在 `package.json` 中新增相依性)

### 關閉自動修復
按一下留言上的 **關閉建議** 以拒絕建議。

## 關閉警示

### 何時關閉

在以下情況下關閉警示：
- 發現是誤報 (程式碼使用了 CodeQL 無法辨識為安全的模式)
- 程式碼僅用於測試且風險可接受
- 修復的工作量大於收益

### 關閉原因

選擇適當的原因 — 它會影響查詢是否繼續執行：

| 原因 | 何時使用 |
|---|---|
| **誤報** | 警示不正確；程式碼實際上是安全的 |
| **不予修復** | 風險已被接受或程式碼即將被淘汰 |
| **用於測試** | 易受攻擊的模式僅存在於測試程式碼中 |

### 關閉留言
- 新增說明關閉理由的留言
- 留言儲存在警示時間軸中，以便進行稽核/合規性檢查
- 可透過 REST API 在 `alerts/{alert_number}` → `dismissed_comment` 存取

### 貢獻改進
對於來自不支援的淨化函式庫的誤報，請考慮向 CodeQL 儲存庫貢獻內容以提高分析準確性。

## 解決警示

### 修復並重新掃描
1. 修復原始碼中的漏洞
2. 認可並推送變更
3. 下一次程式碼掃描執行將驗證修復
4. 當修復確認後，警示會自動關閉

### 移除過時的組態
如果警示持續來自舊的/已停用的組態：
1. 導覽至警示的 **受影響的分支** 區段
2. 識別過時的組態
3. 刪除過時的組態以移除過時的警示

## 警示資料流

對於 `path-problem` 查詢，警示包含資料流資訊：

- **來源 (Source)** — 不受信任資料進入之處 (例如：使用者輸入)
- **接收器 (Sink)** — 資料被不安全使用之處 (例如：SQL 查詢、HTML 輸出)
- **路徑 (Path)** — 資料從來源到接收器所採取的過渡步驟

按一下警示註釋上的 **顯示路徑** 以視覺化完整的資料流。

## 多重組態警示

當多個程式碼掃描組態分析同一個檔案時：
- 由同一個查詢偵測到的同一個問題會顯示為單一警示
- **受影響的分支** 區段會顯示哪些組態發現了該警示
- 不同的組態可能會顯示不同的狀態
- 重新執行過時的組態以同步警示狀態

## 檢視警示

### 儲存庫安全性索引標籤
- 導覽至 **安全性** → **程式碼掃描警示**
- 依以下條件篩選：工具、嚴重性、規則、分支、狀態
- 按一下警示以查看完整詳細資料、受影響的分支和資料流

### 提取要求檢查
- 檢視 PR 中的 **程式碼掃描結果** 檢查
- 按一下 **檢視所有分支警示** 以查看完整警示清單
- 註釋會內嵌出現在 **檔案已變更** 中

### REST API
- `GET /repos/{owner}/{repo}/code-scanning/alerts` — 列出警示
- `GET /repos/{owner}/{repo}/code-scanning/alerts/{alert_number}` — 取得警示詳細資料
- `PATCH /repos/{owner}/{repo}/code-scanning/alerts/{alert_number}` — 更新警示狀態
