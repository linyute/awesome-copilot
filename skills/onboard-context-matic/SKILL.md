# 入門導覽：ContextMatic MCP

此技能提供 `context-matic` MCP 伺服器的引導式互動導覽。請依序遵循每個階段。在每個互動點後請暫停，並等待使用者的回覆後再繼續。

> **代理程式行為規則 — 在整個技能期間皆須遵循：**
> - **絕不要敘述技能結構。** 不要說出階段名稱、步驟編號，或任何聽起來像是在閱讀指示的內容 (例如：「在第 1 階段中，我將...」、「步驟 1a：」、「根據技能...」)。請以自然對話方式進行導覽。
> - **在執行任何工具呼叫前皆須宣告。** 短短一句話即可 — 告訴使用者您即將查詢什麼內容以及原因，然後呼叫該工具。例如：*「讓我為您的專案語言調出可用 API 清單。」* 這能讓使用者隨時了解情況，並避免沈默、未經解釋的停頓。

## 階段 0 — 開場白與工具逐步說明

以簡單易懂的語言簡要說明伺服器功能作為開場。根據以下事實，以您自己的話表達：

> **context-matic** MCP 伺服器解決了 AI 輔助編碼的一個根本問題：通用模型是在公開程式碼上進行訓練的，這些程式碼通常已過期、不正確，或者完全缺少較新 SDK 版本的資訊。此伺服器充當 **即時、具版本意識的基礎接地層 (grounding layer)**。代理程式無須從訓練資料中猜測 SDK 用法，而是直接查詢伺服器，取得與當前 API 版本及專案程式語言相符的 *精確* SDK 模型、端點、驗證模式與可執行程式碼範例。

說明完伺服器解決的問題後，逐一介紹四個工具，就像向第一次使用的人介紹一樣。針對每個工具，請說明：
- **它是什麼** — 給出一個令人印象深刻的簡短說明
- **何時使用** — 一個具體、相關的情境
- **會得到什麼回饋** — 使用者將看到的輸出型別

使用以下事實作為來源，但請以對話方式表達 — 不要直接呈現原始表格：

> | 工具 | 功能 | 使用時機 | 獲得的結果 |
> |---|---|---|---|
> | `fetch_api` | 回傳與 API `key`/識別碼及語言的精確匹配，或列出特定語言的所有 API。`key` 是由 `fetch_api` 回傳的機器可讀識別碼 (例如 `paypal`)，而非人類可讀的顯示名稱 (例如 "PayPal Server SDK")。 | 「我可以使用哪些 API？」 / 開始新專案 / 「你們有 PayPal SDK 嗎？」 | 可用 API 的命名清單及其簡短說明 (完整目錄)，或當您提供識別碼/key 與語言時的單一精確 API 匹配結果 |
> | `ask` | 使用具版本準確性的指引與程式碼範例回答整合問題 | 「如何進行驗證？」、「顯示快速入門」、「做 X 的正確方式是什麼？」 | 紮根於實際 SDK 版本的逐步指引與可執行程式碼範例 |
> | `model_search` | 查詢 SDK 模型/物件定義及其型別化屬性 | 「Order 有哪些欄位？」、「此屬性是必要的嗎？」 | 模型名稱、說明，以及完整的型別化屬性清單 (必要 vs. 選用，巢狀型別) |
> | `endpoint_search` | 查詢端點方法、其參數、回應型別與可執行程式碼範例 | 「顯示如何呼叫 createOrder」、「getTrack 回傳什麼？」 | 方法簽章、參數型別、回應型別，以及可直接複製貼上的程式碼範例 |

在本節結束時，告訴使用者您將在導覽期間即時示範四個核心探索與整合工具，現在就從 `fetch_api` 開始。請明確指出，本導覽重點在於這些核心 ContextMatic 伺服器工具，而非廣泛工作流程中所有可能的輔助工具。

## 階段 1 — 顯示可用 API

### 1a. 偵測專案語言

在呼叫 `fetch_api` 之前，請檢查工作區檔案以判斷專案的主要程式語言：

- 尋找 `package.json` + `.ts`/`.tsx` 檔案 → `typescript`
- 尋找 `*.csproj` 或 `*.sln` → `csharp`
- 尋找 `requirements.txt`, `pyproject.toml` 或 `*.py` → `python`
- 尋找 `pom.xml` or `build.gradle` → `java`
- 尋找 `go.mod` → `go`
- 尋找 `Gemfile` 或 `*.rb` → `ruby`
- 尋找 `composer.json` 或 `*.php` → `php`
- 若未找到專案檔案，請預設為 `typescript`。

儲存偵測到的語言 — 您將在之後的每個工具呼叫中傳遞它。

### 1b. 擷取可用 API

告訴使用者您偵測到的語言，並告知您正在擷取可用 API — 例如：*「我看出這是一個 TypeScript 專案。讓我擷取 TypeScript 可用的 API。」*

呼叫 **`fetch_api`**，並將 `language` = 偵測到的語言，`key` = ""，以便工具回傳完整的 API 清單。

以格式化清單顯示結果，列出每個 API 的 **名稱** 及其 **描述** 的一句摘要。請勿截斷或跳過任何條目。

範例顯示格式 (請依據實際結果調整)：

```
以下是此伺服器目前可用的 API：

1. PayPal Server SDK   — 透過 PayPal REST API 進行支付、訂單、訂閱與保險庫管理。
2. Spotify Web API     — 音樂/播客探索、播放控制與媒體庫管理。
....
```

## 階段 2 — API 選擇 (互動)

詢問使用者：

> 「您想探索這些 API 中的哪一個？請直接說出名稱或編號。」

**請等待使用者的回覆後再繼續。**

儲存所選 API 來自 `fetch_api` 回應中的 `key` 數值 — 您將將其傳遞給所有後續工具呼叫。此外，請記下 API 名稱以便在說明文字中使用。

---

## 階段 3 — 說明所選的 API

在呼叫之前，說些類似的話：*「很棒的選擇 — 讓我為您整理 [API 名稱] 的概觀。」*

呼叫 **`ask`**，參數如下：
- `key` = 所選 API 的 key
- `language` = 偵測到的語言
- `query` = `"請給我此 API 的高層級概觀：它做什麼、主要的控制器或模組是什麼、驗證如何運作，以及開始使用的第一步是什麼。"`

以對話方式呈現回應。強調：
- API 能做什麼 (使用情境)
- 驗證如何運作 (憑證、OAuth 流程等)
- 主要的 SDK 控制器或命名空間
- 要安裝的 NPM/pip/NuGet 等套件名稱

## 階段 4 — 專案語言整合 (互動)

詢問使用者：

> 「是否有 [API 名稱] 的特定部分您想學習如何使用 — 例如建立訂單、搜尋曲目或管理訂閱？或者我應該直接向您展示完整的整合快速入門？」

**等待使用者的回覆。**

在呼叫之前，說些類似的話：*「沒問題 — 讓我來幫您查詢。」* 或 *「好的，讓我調出快速入門。」*

呼叫 **`ask`**，參數如下：
- `key` = 所選 API 的 key
- `language` = 偵測到的語言
- `query` = 使用者提出的目標，若使用者要求完整指南，則為 `"請向我展示完整的整合快速入門：安裝 SDK、設定憑證，並執行第一次 API 呼叫。"`

呈現回應，包含所有回傳的程式碼範例（請保持與回傳內容完全一致）。

## 階段 5 — 示範 `model_search`

告訴使用者：

> 「現在讓我向您展示 `model_search` 如何運作。此工具可讓您查詢任何 SDK 模型或物件定義 — 它的型別化屬性、哪些是必要的 vs. 選用的，以及它們使用的型別。它支援部份名稱與區分大小寫的匹配。」

在呼叫之前，說些類似的話：*「讓我搜尋 `[模型名稱]` 模型，這樣您就能看到結果的樣子。」*

從所選 API 中挑選一個 **具代表性的模型** (範例見下方)，並呼叫 **`model_search`**，參數如下：
- `key` = 之前選擇的 API key (例如 `paypal` 或 `spotify`)
- `language` = 偵測到的專案語言
- `query` = 您挑選的代表性模型名稱

| API key | 良好的示範查詢 |
|---|---|
| `paypal` | `Order` |
| `spotify` | `TrackObject` |

顯示結果，指出：
- 確切的模型名稱及其說明
- 一些有趣的型別化屬性 (強調選用 vs. 必要)
- 任何巢狀模型參考 (例如 `PurchaseUnit[] | undefined`)

告訴使用者：

> 「您可以依名稱搜尋任何模型 — 部分匹配也有效。每當您需要了解特定模型的結構時，請試著要求我查詢 [API 名稱] 的特定模型。」

## 階段 6 — 示範 `endpoint_search`

告訴使用者：

> 「同樣地，`endpoint_search` 會查詢任何 SDK 方法 — 確切參數、它們的型別、回應型別，以及您可以直接放入專案中的可執行程式碼範例。」

在呼叫之前，說些類似的話：*「讓我擷取 `[端點名稱]` 端點，這樣您就能看到參數與實作程式碼範例。」*

針對所選 API 挑選一個 **具代表性的端點**，並呼叫 **`endpoint_search`**，並使用明確的引數物件：

- `key` = 您正在示範的 API key (例如 `paypal` 或 `spotify`)
- `query` = 您要查詢的端點 / SDK 方法名稱 (例如 `createOrder` or `getTrack`)
- `language` = 使用者的專案語言 (例如 `"typescript"` or `"python"`)

例如：

| API key (`key`) | 端點名稱 (`query`) | 範例 `language` |
|---|---|---|
| `paypal` | `createOrder` | 使用者的專案語言 |
| `spotify` | `getTrack`   | 使用者的專案語言 |
顯示結果，指出：
- 方法名稱與說明
- 請求參數與其型別
- 回應型別
- 完整的程式碼範例 (請保持與回傳內容完全一致)

告訴使用者：

> 「請注意，此程式碼範例可直接使用 — 它匯入了正確的 SDK、初始化了用戶端、呼叫了端點並處理錯誤。您可以透過方法名稱或區分大小寫的片段來搜尋任何端點。」

## 代理程式注意事項

- 若使用者選擇的 API 不在 `fetch_api` 的結果中，請告知該 API 目前不可用，並提議繼續導覽其他可用的 API。
- 本技能中的所有工具呼叫皆為 **唯讀** — 除非使用者明確要求您繼續進行整合，否則它們不會修改專案、安裝套件或寫入檔案。
- 當從 `endpoint_search` 或 `ask` 顯示程式碼範例時，請使用正確的語言標籤呈現於圍欄程式碼區塊 (fenced code blocks) 中。
