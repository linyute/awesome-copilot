# 設定、資料與異步模式

當任務涉及設定載入、持久化、快取、計分板或背景重新整理時，請參考此文件。

## 真實外掛程式中的設定模式

### 多設定的小遊戲層

大型小遊戲可能會使用多個以設定為導向的類別：

- `PluginConfigManager` (外掛程式設定管理員)
- `GameConfigurationManager` (遊戲設定管理員)
- `KillMessagesConfig` (擊殺訊息設定)
- `MessagesConfig` (訊息設定)
- `LevelsConfig` (等級設定)
- `MapConfig` (地圖設定)
- `PlayerConfig` (玩家設定)

當各個遊戲系統分別擁有獨立的設定介面時，此模式運作成效良好。

權衡：

- 易於擴展特定領域的設定
- 除非經過仔細管理，否則較難保持預設值和重新載入語意的一致性

### 防禦性包裝設定層

持續對戰 (Persistent brawl) 模式可能會在主設定周圍使用防禦性包裝 (Defensive wrapper)，並針對其他領域檔案（如進度和英雄）使用獨立的載入器。

觀察到的優點：

- 備援值 (Fallback values)
- 無效設定的警告記錄
- 單一包裝層處理核心大廳和素材設定

指引：

- 如果專案仍處於小型規模，偏好使用帶有備援機制的類型化包裝。
- 僅當領域複雜度真正需要時，才拆分為多個設定管理員。

## 資料庫與玩家資料模式

### 重型小遊戲的觀察結果

- 使用一個資料庫管理員加上統計數據和品牌相關的儲存
- 在斷開連線前排清 (Flush) 待處理的更新

### 持續對戰的觀察結果

- 分離 `DatabaseManager`、儲存庫 (Repository) 和服務 (Service)
- 保留以 `UUID` 為鍵的記憶體內快取
- 追蹤「髒」(Dirty) 玩家（有資料異動的玩家）
- 異步排清
- 從快取資料重新建構排行榜
- 為側邊欄和階級 UI 使用「髒」集合，以便批次重新整理計分板
- 在向遊戲或 UI 程式碼呈現排行榜資料前，先對其進行快照 (Snapshot)

指引：

- 當持久化邏輯增加時，使用儲存庫或 DAO 邊界。
- 為熱點玩家統計數據保留快取。
- 記錄有異動的條目 (Dirty entries)，而不是在每個事件發生時都寫入。
- 定期以及在關閉時排清。
- 當擊殺、死亡或階級頻繁變動時，對排行榜重新建構進行去彈跳 (Debounce) 或批次處理。
- 從由異步維護的快取中發佈不可變的快照或複本。

## Paper 外掛程式的異步規則

觀察到的安全持續對戰模式：

- 異步預載快取
- 當資料準備就緒時，切換到主執行緒進行 Bukkit 端的 UI 重新整理
- 異步排行榜重新建構可避免阻塞遊戲進行

建議規則：

- 異步 (Async)：
  - SQL 查詢
  - 頻繁的檔案處理
  - 排行榜重新建構
  - 快取重新計算
- 主執行緒 (Main thread)：
  - 傳送 (Teleport)
  - 背包更改
  - 實體或世界異動
  - 計分板和可見 UI 更改（除非 API 明確聲明為執行緒安全）

## 背景任務模式

觀察到的持續對戰模式：

- 異步定期排行榜重新建構檢查
- 異步髒玩家排清任務
- 使用髒集合的節流 (Throttled) UI 重新整理策略
- 從 UUID 髒集合進行主執行緒計分板排清
- 異步快取預載，隨後進行主執行緒計分板、Tab 名稱和排行榜重新整理

觀察到的重型小遊戲模式：

- 主執行緒定期遊戲階段任務
- 針對每個工作階段的冷卻時間跳動 (Ticking)
- 與遊戲進度掛鉤的時間敏感型顯示重新整理

指引：

- 使用異步任務進行資料維護。
- 使用主執行緒任務進行遊戲進度。
- 如果一個任務可以由事件驅動並加上髒標記，則偏好此方式，而非在每個跳動中強行重新整理所有內容。
- 在異步儲存或排序前，先複製可變的快取資料。
- 避免從異步重複任務內部啟動異步工作，除非你需要獨立的生命週期控制。

## 設定貢獻指引

產生感知設定的程式碼時：

- 提供預設值
- 驗證輸入
- 記錄必要的鍵值
- 避免因無效的素材名稱或缺失的位置而導致默默崩潰
- 說明該子系統重新載入 (Reload) 是否安全
