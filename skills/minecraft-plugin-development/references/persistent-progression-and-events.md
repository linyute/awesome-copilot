# 持久化進度與事件系統

在處理長期執行的 Minecraft 伺服器外掛程式時，如果玩家進度、經濟、特長 (Perks)、增益 (Buffs)、任務或自訂戰鬥事件在單場比賽之外仍然持續存在，請參考此文件。

這最適用於 Pit 風格的 PvP 伺服器、類 MMORPG 的大廳、持續對戰模式、派系相關玩法，以及任何玩家個人檔案被許多獨立系統異動的外掛程式。

## 持久化個人檔案邊界

大型進度外掛程式通常會將持久的個人檔案資料與臨時的執行階段狀態分開。

持久化範例：

- 等級、聲望 (Prestige)、經驗值、硬幣、名望 (Renown)、賞金 (Bounty)、統計數據
- 擁有的特長、已選特長、任務進度、交易限制、郵件、勳章
- 類似背包的儲存空間，例如儲存的工具包 (Kits)、終界箱、倉庫或備份
- 結構描述或個人檔案格式版本

僅限執行階段的範例：

- 戰鬥計時器
- 目前的傷害歸屬映射表 (Map)
- 活躍的增益 (Buffs)
- 臨時連殺計數器
- 編輯模式
- 最後一次傷害時間戳記
- 快取的即時 `ItemStack` 引用

指引：

- 使用瞬時欄位 (Transient fields)、忽略的序列化欄位或專用的 DTO 來明確定義儲存邊界。
- 使用 `UUID` 作為個人檔案查詢和持久化的持久身分識別。
- 避免直接儲存活的 Bukkit 物件，除非專案已經具備其序列化處理。
- 當個人檔案形狀可能隨時間遷移時，保留一個個人檔案版本欄位。
- 偏好使用按內部名稱索引的映射表 (Maps) 進行熱點查詢，例如特長、任務、增益或解鎖狀態。

## 經濟與進度異動

進度功能很少只更改一個數字。一次擊殺獎勵可能會影響硬幣、經驗值、賞金、連殺、任務、助攻、計分板行、Boss 血量條以及持久化的髒狀態 (Dirty state)。

指引：

- 透過一個服務 (Service) 或領域方法路由獎勵，而不是在監聽器中重複計算數學公式。
- 在最終確定獎勵之前發送或呼叫領域級事件，以便其他系統可以修改結果。
- 限制 (Clamp) 並驗證面向玩家的數值，例如硬幣、經驗值、等級、生命值、賞金或冷卻時間。
- 在異動後將個人檔案標記為髒 (Dirty)，並在程式碼庫使用髒路徑追蹤時進行批量儲存。
- 在持久化或獎勵計算完成後，從主執行緒更新可見的 UI。

## 自訂領域事件

複雜的外掛程式通常會將 Bukkit 事件封裝在特定於外掛程式的事件中，例如傷害、擊殺、助攻、獎勵、任務完成、個人檔案已載入、藥水效果或連殺變更事件。

在以下情況使用自訂事件：

- 多個系統需要觀察或修改同一個遊戲決策
- 原始 Bukkit 事件對外掛程式的規則來說過於低階
- 獎勵、助攻、增益或特長需要一個一致的擴充點
- 測試或未來的核心功能將受益於單一的領域訊號

指引：

- 保持 Bukkit 監聽器輕量化：將原始事件翻譯為外掛程式的領域模型，然後進行委派。
- 定義自訂事件是否可取消 (Cancellable) 或可修改。
- 記錄監聽器是否可以更改傷害、獎勵、目標狀態或副作用。
- 如果處理程式可能觸及 Bukkit 狀態，避免從異步執行緒發送自訂事件。
- 防止當自訂事件處理程式觸發另一個原始 Bukkit 動作時產生的遞迴。

## 特長、增益、任務與事件註冊表

功能重型外掛程式通常使用工廠 (Factories) 或註冊表 (Registries) 來發現和公開擴充模組：

- 按介面分組的特長註冊表，例如傷害、擊殺、助攻、重生、射擊或跳動掛鉤
- 具有選用 Bukkit 監聽器註冊的增益註冊表
- 分為一般和重大事件的定時事件註冊表
- 按穩定內部名稱索引的任務或勳章註冊表

指引：

- 要求每個註冊模組都有一個穩定的內部名稱。
- 保持顯示名稱與內部名稱分離，以免在地化或格式化損壞儲存的資料。
- 僅為實際實作了 `Listener` 的模組註冊 Bukkit 監聽器。
- 當啟動可靠性很重要時，偏好顯式註冊；僅當專案已經依賴反射掃描時才使用。
- 對於重複的內部名稱發出嚴重的錯誤警告。
- 當其他系統只需要檢查已註冊模組時，公開唯讀檢視表。

## 定時事件與輪替的伺服器活動

長期執行的伺服器通常會獨立於競技場比賽來排程全域事件。

指引：

- 明確儲存活動事件、下一個事件、冷卻時間和準備階段。
- 分離準備、活動、非活動和清理掛鉤。
- 避免全域事件重疊，除非設計支援。
- 保持 Boss 血量條、計分板和廣播更新與活動事件狀態掛鉤。
- 當事件被取代、強制執行或外掛程式停用時，取消或使計時器過期。
- 在啟動高影響力事件之前，使用設定的線上玩家閾值。

## 版本與相依性邊界

持久化伺服器外掛程式通常依賴多個選用和必要的外掛程式，例如權限、經濟、點數、通訊協定、WorldEdit、資料庫驅動程式、封包函式庫或自訂伺服器分支。

指引：

- 保持 `plugin.yml` 中的 `depend` 和 `softdepend` 與啟動程式碼一致。
- 在使用其 API 之前偵測選用相依性。
- 當缺少軟相依性 (Soft dependency) 時，提供降級路徑或明確警告。
- 將 NMS、封包、反射和自訂分支呼叫隔離在配接器或工具類別之後。
- 在增加該伺服器產品線不存在的 API 之前，檢查目標 Minecraft 版本。

## 審查檢查表

在完成持久化進度程式碼的更改之前，請驗證：

- 個人檔案異動是透過預期的服務或領域方法發生的
- 持久化和執行階段欄位在序列化期間沒有被意外混淆
- 自訂事件發送一次且在正確的執行緒上
- 特長、增益、任務或事件模組具有穩定的內部名稱
- 重複的模組不會默默地互相覆蓋
- 經濟和進度更改一致地更新 UI 和髒儲存狀態
- 在使用前已檢查選用相依性
- 排程的全域事件會清理 Boss 血量條、任務和活動狀態
