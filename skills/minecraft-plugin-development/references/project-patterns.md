# 真實專案模式

本參考文件記錄了在真實 Paper 遊戲外掛程式中觀察到的架構模式：

- 具有地圖、團隊、能力選擇、Boss 波次、計分板和比賽模式覆蓋的重度比賽、多階段小遊戲。
- 具有英雄進度、地圖輪替、玩家資料服務和異步排行榜重新整理的基於職業的持續對戰模式。

當使用者詢問儲存庫結構、功能放置或如何組織一個日益增長的 Minecraft 外掛程式時，請使用此檔案。

## 模式 1：具有明確子系統的中央外掛程式引導 (Bootstrap)

這兩種風格都保留了一個主外掛程式類別，負責建構和連接核心子系統。

觀察到的範例：

- 一個重度比賽的小遊戲從 `onEnable` 連接資料庫、設定管理員、遊戲管理員、比賽管理員、GUI 物件、側邊欄服務、監聽器、命令和重複任務。
- 一個持續對戰外掛程式從 `onEnable` 連接設定、進度、資料庫、儲存庫 (Repository)、玩家資料服務、計分板管理員、地圖管理員、英雄註冊表、英雄服務、遊戲服務和英雄技能處理器。

指引：

- 保持外掛程式啟動過程可讀且有序。
- 在註冊依賴於核心服務的監聽器或命令之前，先建構這些服務。
- 將主外掛程式類別視為編排根節點，而不是遊戲邏輯的堆放場。

## 模式 2：具狀態的遊戲內容屬於專用的工作階段或管理員物件

這兩種風格都避免將所有遊戲資料直接儲存在監聽器上。

觀察到的範例：

- 重度比賽的小遊戲使用 `GameManager`、`MatchManager`、`Game`、`PlayerSession`、`GamePhase` 和 `MatchSession` 等物件。
- 持續對戰模式使用 `GameService`、`PlayerSession`、`PlayerState`、`HeroService` 和 `MapManager`。

指引：

- 將長生命週期的狀態放入工作階段 (Session)、管理員 (Manager) 或服務 (Service) 類別中。
- 保持監聽器輕量化：驗證事件、委派給相關服務，然後退出。
- 偏好明確的遊戲狀態模型，而非散亂的旗標。

## 模式 3：複雜的外掛程式按領域模組成長

隨著外掛程式範圍的擴大，這兩種風格都會將功能拆分為以領域為導向的套件 (Package)。

觀察到的重度比賽小遊戲領域：

- `Command` (命令)
- `Game` (遊戲)
- `Match` (比賽)
- `GameConfig` (遊戲設定)
- `InitialListener` (初始監聽器)
- `Shop` (商店)
- `tasks` (任務)
- `Database` (資料庫)

觀察到的持續對戰領域：

- `bootstrap` (引導)
- `command` (命令)
- `data` (資料)
- `game` (遊戲)
- `hero` (英雄)
- `listener` (監聽器)
- `map` (地圖)
- `gui` (介面)

指引：

- 按遊戲領域或子系統進行分組，而不是模糊的 "utils" 儲存桶。
- 當一項功能擁有自己的狀態、設定和生命週期時，請立即進行拆分。
- 保持套件名稱與你思考遊戲的方式一致。

## 模式 4：出現了兩種有效的架構風格

重度比賽的小遊戲偏向以管理員為中心的遊戲編排。

- 適用於：
  - 回合制
  - 團隊淘汰
  - 特定地圖的遊戲執行個體
  - 疊加在基礎遊戲規則之上的比賽覆蓋

持續對戰模式偏向以服務為中心的執行階段編排。

- 適用於：
  - 持續性的戰鬥大廳
  - 職業選擇和重生迴圈
  - 進度與玩家資料整合
  - 共享遊戲模式下的輪替地圖

指引：

- 對於每個比賽或每個競技場執行個體，偏好以管理員為中心的架構。
- 對於具有可重複使用系統的單一持續共享模式，偏好以服務為中心的架構。

## 模式 5：真實的外掛程式同時需要遊戲層和營運層

真實的遊戲外掛程式將遊戲程式碼與營運子系統相結合：

- 設定載入
- 資料庫連接性
- 計分板重新整理
- 地圖載入
- 命令和監聽器註冊
- 關閉時的清理

指引：

- 不要只設計戰鬥機制。
- 也要設計啟動、關閉、持久化、UI 重新整理和失敗路徑。

## 模式 6：執行個體重型外掛程式需要明確的隔離層

SkyWars 風格和地牢風格的外掛程式需要在每個活動的遊戲執行個體周圍建立清晰的邊界。

常見的隔離表面：

- 玩家與遊戲工作階段的所有權
- 世界或地圖執行個體的所有權
- 臨時怪物或實體的所有權
- 特定遊戲的聊天接收者
- 可見性群組
- 每個遊戲的計分板和標題
- 資源補給或波次計時器

指引：

- 將 `Game` 視為任何不應洩漏到另一個競技場之物的所有者。
- 讓全域管理員負責查詢和生命週期，而不是每一條遊戲規則。
- 在增加新的監聽器時，首先決定它是全域的、僅限大廳的，還是僅限遊戲執行個體的。
- 如果外掛程式可以同時執行多個遊戲，則每個事件都應在更改遊戲狀態之前解析所屬的遊戲。

## 何時重用哪種模式

- 具有回合、團隊和地圖執行個體的新小遊戲：
  - 偏好 `GameManager` + `Game` + `PlayerSession` + 選用的 `MatchManager`
- 具有職業選擇和地圖輪替的持續對戰大廳：
  - 偏好 `GameService` + `PlayerSession` + `HeroService` + `MapManager`
- SkyWars 風格的隔離競技場：
  - 偏好 `GameManager` + `Game` + `GameMap` + 倒數計時任務 + 每個遊戲的計分板和戰利品管理員
- PvE 地牢或波次進度：
  - 偏好 `Game` + 大廳地圖 + `RoundManager` 或階段管理員 + 實體所有權映射 (Map) + 明確的清理掛鉤
- 外掛程式已經龐大且混亂：
  - 在增加更多功能之前，先使用領域套件拆分模式
