# 建構、測試與執行階段驗證

當 Minecraft 外掛程式任務涉及建構設定、打包、產生的資源、部署到本地測試伺服器、選用的外掛程式整合、重新載入行為或發佈就緒時，請參考此文件。

這對於具有許多 YAML 資源、陰影函式庫 (Shaded libraries)、軟相依性 (Soft dependencies)（如 PlaceholderAPI 或 MythicMobs）以及用於驗證執行階段設定的管理員命令的 Maven 或 Gradle Paper 外掛程式特別有用。

## 建構 Metadata 與產生的資源

外掛程式專案通常將原始資源保留在 `src/main/resources` 下，而建構工具會將篩選後的輸出複製到 `target/classes` 或 `build/resources` 中。

指引：

- 編輯原始資源，而不是產生的複本。
- 將 `target/classes`、`build/classes`、`build/resources` 和複製的伺服器外掛程式 jar 檔案視為建構輸出。
- 當 `plugin.yml` 使用 `${project.version}` 等篩選值時，請確認 Maven 或 Gradle 中已啟用資源篩選。
- 保持 `plugin.yml` 中的 `name`、`main`、`api-version`、命令、權限、`depend`、`softdepend` 和 `libraries` 與程式碼及建構檔案一致。
- 如果套件或主類別發生變更，請在同一次變更中更新 `plugin.yml`。
- 如果命令名稱發生變更，請一併更新命令註冊、權限檢查、Tab 補全器和用法文字。

要識別的 Maven 結構範例：

```xml
<resources>
  <resource>
    <directory>src/main/resources</directory>
    <filtering>true</filtering>
  </resource>
</resources>
```

## 陰影處理 (Shading) 與相依性範圍

Paper 外掛程式通常結合了伺服器提供的 API、選用的外掛程式 API 以及必須綑綁的函式庫。

指引：

- 將 `paper-api`、Bukkit、Spigot 和選用的外掛程式 API 保持為 `provided` (已提供)，除非專案有明確的理由要綑綁它們。
- 對伺服器不會提供的函式庫進行陰影處理 (Shade)，例如小型 UI 輔助工具或獨立的公用程式函式庫。
- 當陰影函式庫可能與其他外掛程式發生衝突時，請重新定位 (Relocate) 它們。
- 停用或審閱「相依性減少的 POM」(dependency-reduced POM) 輸出，以免產生混亂或誤導性的儲存庫異動。
- 不要對列在 `depend` 或 `softdepend` 中的外掛程式 API 進行陰影處理。
- 如果 `plugin.yml` 使用了 Paper 的 `libraries` 機制，請確保伺服器版本支援該機制，且該相依性應由 Paper 載入而非進行陰影處理。

常見的重新定位範例：

```xml
<relocation>
  <pattern>fr.mrmicky.fastboard</pattern>
  <shadedPattern>com.example.plugin.libs.fastboard</shadedPattern>
</relocation>
```

## 選用的外掛程式整合

與 PlaceholderAPI、MythicMobs、WorldEdit、經濟外掛程式、權限系統、NPC 系統或自訂模型外掛程式整合的外掛程式，在該整合為選用時應能優雅地降級。

指引：

- 將必要的插件放入 `depend`，將選用的整合放入 `softdepend`。
- 在註冊引用了選用外掛程式事件類別的監聽器之前，先檢查 `PluginManager#isPluginEnabled`。
- 將選用整合的程式碼隔離在監聽器、配接器或服務中。
- 當缺少軟相依性 (Soft dependency) 且遊戲功能將因此受限時，記錄清晰的警告訊息。
- 確保選用整合的資料流經外掛程式正常的狀態機，而不是繞過獎勵、背包、任務或清理邏輯。

對於 MythicMobs 風格的資源怪物：

- 透過穩定的內部怪物名稱進行匹配，而非顯示名稱。
- 仔細解析直接擊殺者，因為直接擊殺者可能是拋射物或其他實體。
- 只有在外掛程式接受獎勵路徑的所有權後，才清除原始掉落物。
- 保持自訂怪物獎勵與正常遊戲獎勵掛鉤在相同的清理和持久化邏輯上。

## 執行階段設定驗證

設定繁重的外掛程式受益於一個非破壞性的驗證命令，該命令可以在啟動、重新載入和測試回合之前執行。

有用的驗證表面：

- 缺失的世界、大廳點、出生點、區域或試煉室
- 半徑為零或體積為零的區域
- 優先級可能模糊的重疊區域
- 未知的特長、工具包 (Kit)、目標、怪物、階級、素材或側邊欄識別碼
- 被側邊欄行、獎勵、目標或命令仍然引用的已停用系統
- 超出安全範圍的冷卻時間、獎勵分享比例和百分比總和

指引：

- 回傳警告和錯誤，而不是在每個不完美的設定上都導致崩潰。
- 快取最新的驗證結果，以便管理員命令在重新載入後可以顯示它。
- 在每個問題中包含設定檔案和路徑。
- 在重新載入後、重新整理執行階段服務之前進行驗證。
- 保持自動修復保守化；默默地更改地圖座標、獎勵或進度規則可能會讓伺服器操作員感到驚訝。

## 重新載入與服務重新整理

具狀態的外掛程式重新載入是有風險的，但有時對於活躍的伺服器疊代是必要的。

指引：

- 在重新建構取用設定的服務之前，先重新載入類型化設定模型。
- 重新啟動依賴於已更改設定的重複任務。
- 在重新載入後重新整理線上玩家狀態：區域、側邊欄、操作列、NPC 可見性、特長選擇和活動目標狀態。
- 避免在重新載入後堆疊重複的排程器、Boss 血量條、全息圖 (Holograms)、NPC 或監聽器。
- 偏好在擁有任務或產生的實體的服務上使用明確的 `stop()` 和 `start()` 方法。
- 如果重新載入僅獲得部分支援，請明確說明哪些系統需要重新啟動伺服器。

## 第一輪伺服器測試計畫

對於遊戲外掛程式，當程式碼變更難以透過單元測試證明時，請包含一個簡短的手動測試計畫。

建議檢查表：

- 使用已安裝所需相依性的目標 Paper 版本啟動伺服器。
- 確認外掛程式載入且無主控台異常。
- 執行外掛程式的驗證命令（如果有）。
- 在相關情況下，分別以玩家和主控台身分測試核心命令和 Tab 補全。
- 測試加入、離開、死亡、重生、斷開連線、重新連線和外掛程式停用清理。
- 測試更改功能的一個正常路徑 (Happy path)。
- 測試一個失敗路徑，例如缺少權限、冷卻中、目標無效、缺少相依性或錯誤的設定。
- 確認產生的實體、怪物、NPC、全息圖、計分板、Boss 血量條和臨時世界已清理乾淨。
- 確認個人檔案儲存、獎勵、排名更新和 UI 重新整理僅執行一次且在預期的執行緒上。

## 審閱檢查表

在完成建構或部署相關的更改之前，請驗證：

- 更改的是原始資源，而非產生的輸出。
- 如果儲存庫有要求，已更新產生的 README 或市集檔案。
- 建構檔案與 `plugin.yml` 在版本、主類別、相依性和命令上保持一致。
- 當容易發生衝突時，陰影相依性已重新定位。
- 選用整合具備保護機制和明確的降級行為。
- 驗證命令或測試檢查表涵蓋了已變更的遊戲路徑。
- 除非儲存庫有意追蹤，否則不提交本地建構輸出。
