# 小遊戲執行個體流程

在建構競技場風格的 PvP 遊戲、SkyWars 風格的島嶼遊戲、地牢或波次模式、輪替地圖執行個體，或任何多個玩家、世界、怪物、計分板和計時器屬於特定遊戲執行個體的外掛程式時，請參考此文件。

## 執行個體所有權模式

優秀的小遊戲外掛程式會使所有權變得明確：

- 全域 `GameManager` 或 `GameService` 追蹤活動中的遊戲。
- 每個 `Game` 擁有其地圖或世界執行個體。
- 玩家被映射到唯一的活動遊戲或大廳狀態。
- 臨時怪物、拋射物、NPC 或產生的物件會映射回其所屬的遊戲。
- 計分板、可見性、聊天接收者、戰利品和計時器的範圍都限制在所屬遊戲內。

建議的映射 (Maps)：

- `Map<String, Game>` 用於按產生的 ID 儲存活動中的遊戲。
- `Map<UUID, Game>` 或 `Map<UUID, PlayerSession>` 用於玩家所有權。
- `Map<UUID, Game>` 用於在使用 MythicMobs、自訂 NPC 或 Boss 波次時的怪物或實體所有權。

避免僅依賴世界名稱、背包內容、計分板文字或實體顯示名稱作為唯一事實來源。

## 競技場遊戲狀態機

常見的 PvP 競技場狀態：

- `WAITING` (等待中)
- `STARTING` (開始中)
- `STARTED` (已開始)
- `ENDED` (已結束)

常見的持續對戰狀態：

- `LOBBY` (大廳)
- `RESPAWNING` (重生中)
- `IN_BRAWL` (戰鬥中)

常見的地牢或路徑節點狀態：

- 大廳或路徑選擇階段
- 活躍戰鬥節點
- 和平房間、商店、事件或休息節點
- 勝利或清理階段

指引：

- 在套用遊戲效果之前，使用狀態來限制每個事件監聽器。
- 轉換狀態時，同時更新計時器、計分板、可見性、玩家背包和保護區域。
- 將狀態變更作為啟動倒數、補給任務、結束倒數和怪物生成的唯一觸發點。
- 如果某個任務僅在特定狀態下有意義，則在遊戲離開該狀態時應自我取消。

## 玩家隔離

多競技場外掛程式必須隔離的不僅僅是傳送目的地。

請檢查以下表面：

- 使用 `showPlayer` 和 `hidePlayer` 控制玩家可見性
- 聊天接收者，特別是在 `AsyncChatEvent` 中
- 計分板執行個體和側邊欄更新
- Tab 清單名稱或階級
- Boss 血量條和標題廣播
- 大廳物品和僅限遊戲的背包狀態
- 傷害、方塊、物品撿取、物品掉落、互動、傳送、重生和退出事件

規則：

- 如果兩名玩家不在同一個可見性群組中，請在雙方向上對彼此隱藏。
- 大廳玩家不應收到競技場聊天或競技場廣播。
- 競技場玩家不應收到無關的大廳或其他競技場遊戲訊息。
- 觀戰玩家應排除在戰鬥、撿取、方塊和互動邏輯之外，除非明確允許。

## 倒數與重複任務生命週期

典型任務：

- 開始倒數
- 結束倒數
- 箱子或資源補給倒數
- 計分板或側邊欄重新整理
- 階段或波次延遲
- 地圖輪替計時
- 髒資料 (Dirty data) 儲存

規則：

- 當需要取消功能時，請儲存 `BukkitTask` 控制代碼。
- 不要為同一個遊戲疑慮啟動第二個任務，而不取消或檢查第一個任務。
- 倒數任務應在每次跳動 (Tick) 時讀取目前遊戲狀態，並在過期時自我取消。
- 延遲任務應在執行前重新檢查遊戲是否仍然存在且仍在執行。
- 在遊戲結束和外掛程式關閉時清理任務。

## 戰利品與資源補給系統

空島和競技場外掛程式通常需要由設定驅動的資源分配。

有用的結構：

- 按角色（如島嶼、中間、中心、普通、稀有或超強）對容器位置進行分組。
- 從設定載入加權戰利品。
- 為一個群組建立總戰利品池。
- 在放置前對物品進行洗牌 (Shuffle)。
- 使用有上限的重試迴圈來選擇隨機的空背包格位。
- 透過發出警告並安全降級來驗證缺失或無效的戰利品設定。

容器安全性：

- 在讀取配置的箱子或木桶之前先載入區塊 (Chunk)。
- 在轉型 (Casting) 前驗證該方塊是否為 `Container`。
- 如果配置的方塊缺失，請有目的地重新建立，或發出警告並跳過。
- 如果設計預期完全重置，請在補給前清除先前的內容。
- 避免在頻繁觸發的事件中進行繁重的戰利品產生。

## 波次、怪物與路徑節點系統

地牢風格的外掛程式通常將路徑選擇與戰鬥波次相結合。

建議的模型：

- 為當前波次保留一個等待中的怪物 ID 佇列。
- 為產生的怪物保留一組活動怪物 UUID。
- 將每個產生的怪物 UUID 註冊到所屬的遊戲。
- 怪物死亡時，將其從活動集合中移除並註銷所有權。
- 只有當等待佇列和活動集合都為空時，波次才算清除。
- 波次清除後，安排下一個波次或返回路徑/大廳階段。

路徑投票指引：

- 只有存活且活動中的玩家才應投票或前進。
- 在檢查閾值前，針對當前活動的玩家 UUID 修剪投票映射表 (Map)。
- 使用降級方案或乾淨的勝利/結束路徑來處理缺失的階段範本。
- 和平房間應有明確的前進門檻，而非默默前進。

清理規則：

- 在怪物死亡、遊戲停止和外掛程式停用時註銷怪物。
- 只有在玩家安全傳送出後，才卸載舊的階段地圖。
- 在遊戲結束時清除波次佇列和活動怪物集合。
- 絕不要讓過時的怪物死亡事件推進一個已完成或已刪除的遊戲。

## 地圖執行個體與輪替規則

對於複製的臨時世界：

- 將來源地圖複製到唯一的活動資料夾中。
- 從該資料夾建立或載入世界。
- 在世界存在後配置遊戲規則 (Gamerules)。
- 有目的地設定自動儲存。
- 在卸載前將玩家傳送出去。
- 在刪除資料夾前先卸載世界。
- 卸載後刪除舊資料夾並記錄失敗原因。

對於輪替的共享戰場：

- 在輪替前警告玩家。
- 在載入下一個地圖前快照目前的參與者。
- 將參與者移動到新的出生點並重置暫時的職業或能力狀態。
- 輪替後重新整理計分板、Tab 名稱、排行榜和可見性。
- 如果顯示的倒數計時變化緩慢，請防止計分板每跳一次就重新整理一次。

## 監聽器啟發式方法

對於觸及遊戲狀態的每個監聽器：

- 如果實體或玩家沒有工作階段 (Session)，則提早返回。
- 僅解析一次所屬遊戲並重複使用。
- 在異動方塊、背包、生命值、掉落物或計分板之前檢查遊戲狀態。
- 在允許傷害之前檢查觀戰者、安全區域、團隊和戰鬥標籤規則。
- 保持世界與遊戲、實體與遊戲檢查的一致性。
- 不要假設 `Player#getKiller()` 足以應付自訂戰鬥；當拋射物、技能或能力傷害很重要時，請使用戰鬥標籤。

## 第三方外掛程式整合

常見整合：

- PlaceholderAPI 用於計分板、Tab 名稱或外部預留位置。
- MythicMobs 用於自訂怪物產生和技能。
- 選單外掛程式用於大廳選擇器。
- 資料庫函式庫用於玩家進度。

規則：

- 僅當啟動若無其則無法運作時，才在 `depend` 中宣告強制相依性。
- 針對選用功能使用 `softdepend` 並在執行階段檢查。
- 保持選用整合註冊與核心啟動程式碼隔離。
- 當使用外部技能或產生外部怪物時，優雅地處理缺失的 ID 和 API 失敗。
- 偏好服務邊界 (Service boundaries)，使其餘遊戲程式碼不會直接依賴於每個整合 API。
