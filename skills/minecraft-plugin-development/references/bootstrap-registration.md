# 引導 (Bootstrap) 與註冊

在變更 `onEnable`、`onDisable`、命令設定、事件串接或啟動順序時，請參考此文件。

## 真實外掛程式中的引導模式

### 重型小遊戲引導

共同特徵：

- 首先連線資料庫
- 需要時註冊設定序列化 (Configuration serialization)
- 初始化多個設定管理員
- 建立遊戲管理員和 GUI 輔助工具
- 向已在線上的玩家套用大廳 UI
- 明確註冊多個監聽器 (Listeners)
- 啟動重複任務
- 最後註冊命令和 Tab 補全器

當啟動過程必須在伺服器安全接受互動前組裝多個遊戲子系統時，此模式運作良好。

### 持續性職業對戰引導

共同特徵：

- 首先儲存預設設定和隨附資源
- 建構設定包裝器 (Wrapper) 和進度設定
- 連線資料庫並建立儲存庫 (Repository)/服務 (Service) 層
- 建構計分板、地圖、英雄和遊戲服務
- 將服務相依性串接在一起
- 啟動背景任務
- 預載異步快取，然後排程主執行緒 UI 重新整理
- 在服務圖 (Service graph) 準備就緒後註冊監聽器和命令

當玩家資料和異步服務是一等相依項 (First-class dependencies) 時，此模式運作良好。

## 命令註冊規則

觀察到的做法：

- 重型小遊戲通常在 `plugin.yml` 中宣告所有命令，然後在程式碼中設定執行器 (Executors) 和 Tab 補全器。
- 持續對戰模式可能宣告較小的命令表面（例如 `leave`），然後在分配執行器之前對 `getCommand` 進行空值檢查。

建議規則：

- 務必將新命令加入 `plugin.yml`
- 如果命令可能是選用的或會被重新命名，請對 `getCommand` 進行空值檢查
- 如果存在 Tab 補全，請在同一次變更中註冊它
- 保持用法 (Usage)、權限 (Permission) 和權限訊息 (Permission-message) 與實際程式碼行為一致

## 監聽器註冊規則

觀察到的做法：

- 兩種外掛程式都在啟動期間的一個地方註冊監聽器
- 監聽器建構函式僅接收它們實際需要的服務

建議規則：

- 保持註冊集中在主外掛程式類別或命名明確的引導輔助工具中
- 明確地注入服務，而不是讓監聽器在各處搜尋全域變數 (Globals)
- 如果一個監聽器依賴於任務、GUI 或管理員，請先建構該相依項

## 重複任務與啟動順序

觀察到的做法：

- 重型小遊戲通常直接從啟動開始定期的遊戲任務，用於邊界檢查和大廳 UI 重新整理
- 持續對戰模式通常透過玩家資料和遊戲服務等服務啟動背景任務

建議規則：

- 僅在必要的狀態持有者存在後才啟動重複任務
- 在關閉時取消它們
- 如果任務會異動遊戲狀態，請確保它在主執行緒上執行
- 如果任務執行 I/O 或快取重建，偏好異步執行並跳回主執行緒進行 Bukkit 工作

## 關閉預期

觀察到的做法：

- 重型小遊戲刪除遊戲、卸載地圖、排清 (Flush) 待處理的統計數據、關閉資料庫資源並清除側邊欄
- 持續對戰模式關閉遊戲服務、玩家資料服務、地圖管理員，然後斷開資料庫連線

建議規則：

- 停止任務
- 排清髒資料 (Dirty data)
- 分離或清除 UI 物件
- 如果你的外掛程式建立了臨時世界，請卸載它們
- 最後關閉資料庫連線池 (Pools)

## 此技能的貢獻指引

產生啟動或關閉程式碼時，請註明：

- 哪些設定或資源檔案必須存在
- 哪些命令或監聽器必須註冊
- 哪些任務必須啟動或取消
- 哪些資源在停用時需要清理
