# SemVer 決策規則

參考資料：https://semver.org/

---

## 版本格式

```
vMAJOR.MINOR.PATCH
```

- **MAJOR (主版本)** — 不相容的 API 變更
- **MINOR (次版本)** — 新的向下相容功能
- **PATCH (修補版本)** — 向下相容的臭蟲修復

1.0 版本前的注意事項：如果目前版本為 `0.x.y`，則規則較寬鬆 — 
針對重大變更進行 MINOR 升級是很常見的。一旦超過 `1.0.0`，下述規則將嚴格執行。

---

## 什麼算作 MAJOR 升級 (重大變更)

重大變更係指任何可能導致函式庫使用者在**不更改其自身程式碼**的情況下， 
遇到編譯錯誤、執行階段錯誤或行為變更的修改。

範例：
- 移除公開的函式、類別、方法或常量
- 重新命名公開的函式、類別、方法或常量
- 變更函式特徵 (新增必要參數、移除參數、變更參數類型、變更傳回類型)
- 變更呼叫者依賴的可觀察行為 (例如：拋出的錯誤類型、發出的事件名稱、傳回值的格式)
- 變更必要的設定鍵或其接受的值
- 停止支援先前支援的執行階段/語言版本
- 移除或重新命名公開匯出的模組路徑

**如有疑慮，請優先選擇 MAJOR 升級而非 MINOR。** 標示重大變更優於 
靜默地破壞使用者的程式。

---

## 什麼算作 MINOR 升級 (新功能)

- 新增新的公開函式、類別、方法或常量
- 為現有函式新增選填參數 (具備向下相容的預設值)
- 實作不影響現有協定/介面的新協定/介面
- 新增具有合理預設值的設定鍵
- 棄用 (但未移除) 公開 API — 移除將在未來的 MAJOR 版本中進行

---

## 什麼算作 PATCH 升級

- 修復行為與文件說明意圖不符的臭蟲
- 在不變更公開 API 的情況下改進效能
- 無外部可觀察差異的內部重構
- 文件更新
- 不變更函式庫自身公開界面的相依性更新
- CI/CD、測試、工具鏈變更
- 不破壞 API 的安全性修復

---

## 多項變更 — 優先順序

當發布中包含多種變更類型時，**最高優先順序者**勝出：

```
MAJOR > MINOR > PATCH
```

一個重大變更 + 十個新功能 = MAJOR 升級。

---

## 首次發布 (無過往標籤)

無論差異比較中包含什麼，預設為 `1.0.0`。請告知使用者。

---

## 邊緣案例

| 情況 | 建議 |
|---|---|
| 僅變更了內部/私有符號 | PATCH |
| 為先前未定義類型的函式新增類型註釋 | PATCH (非重大變更) |
| 變更選填參數的預設值 | 若呼叫者可能依賴舊的預設值，則視為 MAJOR |
| 在選填區塊中新增必要的設定選項 | 若區塊本身是選填的則為 MINOR，否則為 MAJOR |
| 完全還原先前的提交 | 遵循最終差異顯示的結果，而非還原訊息 |
