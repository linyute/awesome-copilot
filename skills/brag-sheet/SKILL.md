# 績效清單 (Brag Sheet) — 工作影響力撰寫程式

將工程工作轉化為有證據支持的影響力陳述，用於績效檢閱、自我檢閱、晉升資料包和每週更新。獨特之處在於挖掘 Copilot CLI 工作階段記錄檔、git 歷程記錄和 PR，以重構遺忘的工作。

適用於：「brag」、「log work」、「what did I do」、「backfill」、「performance review」、「self-review」、「promo packet」、「weekly update」、「status report」、「write impact statement」、「what did I ship」、「I forgot to log my work」、「review prep」、「accomplishments」
不要用於：專案管理、衝刺計劃、時間追蹤、票證建立

## 快速開始

| 使用者想要... | 模式 | 輸出 |
|---------------|------|--------|
| 記錄一項成就 | **Capture (擷取)** | 1 個影響力優先的條目 |
| 「我上週做了什麼？」 | **Backfill (補填)** | 按週分組的條目，挖掘自 git/PR/工作階段 |
| 為檢閱或晉升做準備 | **Review Pack (檢閱包)** | 按影響力主題分組的條目 + STAR 敘述 |

## 代理程式行為規則

1. **務必**在掃描來源之前確認時間範圍和範圍。不要假設是「上週」 — 請詢問。
2. **務必**在選擇工作流程之前檢查有哪些工具可用 (`save_to_brag_sheet`, `git`, `gh`)。
3. **務必**始終包含所有三個部分：動作 → 結果 → 證據。如果缺少證據，請寫下 `(需要證據)` — 絕不要靜默省略。
4. **務必**在儲存之前向使用者展示草擬的條目。未經確認絕不自動儲存。
5. **務必**將相關的提交分組為單個條目。同一個功能上的十個提交 = 一個條目。
6. **務必**保留使用者的語氣。為了影響力重新架構，但不要捏造成就或誇大範圍。
7. **絕對不要**造假數據、團隊規模或影響力數字。如果使用者沒有提供數字，不要自己發明。
8. **絕對不要**為使用者僅口頭描述而未經驗證的工作撰寫條目。請詢問：「這項工作上線了嗎？是否有我可以參考的 PR 或文件？」
9. **絕對不要**在掃描完成之前略過補填掃描步驟或草擬條目。
10. **絕對不要**用瑣碎的條目來填充工作較少的期間。誠實的空缺優於虛假的填充。

## 條目格式

每個條目都使用影響力優先的框架，包含三個必要部分：

```
做了 [動作] → [結果/影響] → [證據]
```

**除非包含所有三個部分，否則不要輸出條目。** 如果缺少證據，請要求提供或標註為「(需要證據)」。

### 反模式

| ❌ 不要 | ✅ 而是要 |
|---------|--------------|
| 「修復了身份驗證中的臭蟲」 | 「修復了權杖重新整理的競爭條件 → 消除了影響 12% API 呼叫的 401 錯誤 → PR #247」 |
| 「開發儀表板」 | 「在 Grafana 中建立了延遲儀表板 → 值班人員能在 2 分鐘內偵測到 P95 飆升 → 已部署到正式環境」 |
| 造假數據：「節省了 40% 的工程時間」 | 詢問：「您是否有粗略的估計，還是我應該保持定性描述？」 |
| 每次提交一個條目 | 將相關提交分組為一個條目，並採用最高影響力的框架 |
| 被動語態：「管線得到了改進」 | 主動語態：「建立了 CI 矩陣 → 在發布前發現了僅限 Windows 的臭蟲」 |
| 列出使用的技術 | 陳述結果：「將 4 個服務遷移到 IaC → 部署時間從 45 分鐘縮短至 8 分鐘」 |
| 靜默捨棄較弱的條目 | 標記為 `(需要證據)` 並交給使用者填寫 |

## 證據階梯

並非每個條目都需要數據。請使用可用的最強證據：

| 強度 | 證據類型 | 範例 |
|----------|--------------|---------|
| 🥇 最佳 | 量化數據 | 「將 P95 延遲從 800 毫秒降低至 120 毫秒」 |
| 🥈 強 | PR、提交或文件連結 | 「PR #312，維基中的設計文件」 |
| 🥉 良好 | 可觀察到的結果 | 「為團隊 X 解除阻塞」、「解決了 Sev2 事故 Y」 |
| ✅ 可接受 | 定性描述 + 背景 | 「減少了值班輪轉的瑣事 — 參見更新後的維運指南」 |
| ⚠️ 弱 | 僅活動 | 「開發身份驗證」 — 重新架構或標記為 `(需要證據)` |

絕不要發明數據來填補缺口。帶有背景資訊的定性證據勝過造假的數字。

## 如何幫助使用者

遵循此決策樹：

1. **如果 `save_to_brag_sheet` 工具可用** → 直接使用擴充工具 (`save_to_brag_sheet`, `review_brag_sheet`, `generate_work_log`)。除非確認可用，否則不要參考或嘗試呼叫這些工具。

2. **如果 git 或 gh CLI 可用** → 從提交和 PR 補填 (參見下方的「補填工作流程」)

3. **否則** → 引導式訪談：「您在做什麼？」、「誰受益了？」、「證據是什麼？」

針對每個條目，引導完成：**內容** (交付成果) → **原因** (誰受益) → **證據** (PR、數據、連結)。輸出格式化的 Markdown，以便使用者貼到檢閱文件中。

## 補填工作流程

當使用者詢問「我上週做了什麼」或「補填我的歷程記錄」時：

**請按順序執行以下步驟。在掃描完成之前不要草擬條目。**

### 步驟 1：掃描可用來源

檢查可用來源，然後挖掘每個來源：
```bash
git --version 2>/dev/null         # 用於挖掘提交
gh --version 2>/dev/null          # 用於挖掘 PR
ls ~/.copilot/session-state/ 2>/dev/null  # Copilot 工作階段記錄檔
```

**Git 提交** — 使用者在目前存放庫中最近的提交：
```bash
git log --author="$(git config user.email)" --since="2 weeks ago" \
  --pretty=format:'%h|%ad|%s' --date=short --no-merges
```

**PR 歷程記錄** — 跨存放庫的已合併 PR：
```bash
gh pr list --author @me --state merged --limit 20 \
  --json number,title,repository,mergedAt
```

**Copilot 工作階段歷程記錄** (此技能獨有)：
- 路徑：`~/.copilot/session-state/<工作階段 ID>/workspace.yaml`
- 讀取欄位：`summary`, `cwd`, `repository`, `branch`
- 略過不含 `summary` 欄位的工作階段
- 注意：此目錄可能並非在所有機器上都存在

如果這些來源皆不可用，則退回到引導式訪談。

### 步驟 2：將相關工作分組

將相關訊號聚集為一個條目：
- 同一個 PR + 及其提交 → 1 個條目
- 3 天內對同一個檔案/功能的多次提交 → 1 個條目
- 參考同一個存放庫 + 分支的 Copilot 工作階段 → 如果存在 PR 條目，則合併至其中

### 步驟 3：草擬條目

為每個分組撰寫影響力優先的條目。分配類別。

### 步驟 4：展示與精煉

向使用者展示所有草擬的條目。根據回饋進行調整。

### 步驟 5：輸出

格式化為按週分組的 Markdown：

```markdown
## 2025-04-14 當週

### 🚀 PR 與功能
- **將身份驗證服務遷移到受控識別** → 每季消除了 3 起秘密輪轉事故 → PR #312

### 🏗️ 基礎設施
- **建置了 copilot-brag-sheet 的 CI 管線** → 跨 3 個作業系統 × 3 個 Node 版本的 107 項測試 → 已發布 v1.0.0
```

## 績效檢閱準備

當使用者正在準備績效檢閱 (Connect、年度檢閱等) 時：

### 結構

1. **收集** — 從工作記錄中收集條目 (或使用上述工作流程補填)
2. **選擇** — 挑選前 3–5 個影響力最高的項目
3. **改寫** 每個項目，包含三個部分：
   - **我的工作內容** — 具體動作
   - **為什麼它很重要** — 誰受益了、改變了什麼
   - **證明** — PR 編號、數據變化、儀表板連結、客戶結果
4. **組織** 按影響力主題 (而非按時間順序)：
   - 交付結果 / 卓越營運
   - 客戶 / 團隊影響
   - 協作 / 指導 / 領導力
   - 成長 / 學習
5. **詢問缺口** — 如果缺少證據，請提示使用者：「改變了什麼數據？」、「誰的阻塞被解除了？」、「PR 或事故 ID 是什麼？」

### 強條目 vs 弱條目

| ✅ 強 | ❌ 弱 |
|----------|--------|
| 結果優先、量化 | 活動列表 (「負責 X」) |
| 與客戶/團隊影響掛鉤 | 未提及受益者 |
| 包含證據 (PR、數據) | 無衡量結果 |
| 展示所有權或領導力 | 純粹完成任務 |

### 敘述格式

對於較長的敘述章節，使用 STAR： **情境 (S)**ituation → **任務 (T)**ask → **行動 (A)**ction → **結果 (R)**esult。

對於使用 Connect 預設值的 Microsoft 員工，請圍繞核心優先順序 (Core Priorities) 構建條目：交付結果、客戶導向、團隊合作和成長思維。

## 輸出契約

在結束之前，請確保：
1. 每個條目都有 動作 → 結果 → 證據 (如果缺少，請標註 `(需要證據)`)
2. 不造假數據 — 僅使用使用者提供或源自已驗證的資料
3. 儲存前向使用者展示條目
4. 明確說明時間範圍
5. 輸出是可貼上的 Markdown，並已分配類別

## 注意事項

### 當生存放庫中沒有最近的提交
使用者可能跨多個存放庫工作。在得出沒有可補填內容的結論之前：
1. 詢問他們是否想掃描不同的存放庫或分支
2. 檢查 `gh pr list --author @me --state merged` 以獲取跨存放庫的 PR
3. 退回到引導式訪談 — 並非所有具影響力的工作都會留下 git 痕跡 (設計文件、事故回應、指導)

### 檢閱期間與 git 歷程記錄不符
績效檢閱通常涵蓋 6–12 個月。明確設定日期範圍：
```bash
git log --author="$(git config user.name)" --since="2024-07-01" --until="2025-01-01" --oneline
```
對於長時間範圍，PR 歷程記錄 (`gh pr list --state merged`) 比提交記錄更可靠。

### 使用者無法量化影響
並非每個條目都需要數字。請參閱上方的「證據階梯」。可接受的證據包括 PR 連結、「為團隊 X 解除阻塞」或帶有背景資訊的定性結果。絕不要發明數據來填補缺口。

### Copilot 工作階段目錄不存在
僅當使用者執行過 Copilot CLI 工作階段時，`~/.copilot/session-state/` 才會存在。不要報錯 — 靜默略過並註明：「找不到 Copilot 工作階段歷程記錄；僅掃描 git 和 PR。」

### 「brag」可能意味著其他事情
使用者可能會說「向我的團隊炫耀 (brag) 這個功能」(這是一個發布公告，而不是一個工作條目)。如果意圖模糊，請確認意圖。

### 結對程式設計或共同作者的提交
如果同一個提交中出現多個作者，請詢問：「我應該將這項工作歸功於您、視為共同工作，還是略過它？」

## 自動工作階段追蹤 (選填)

如需自動背景追蹤每個 Copilot CLI 工作階段 (編輯的檔案、建立的 PR、git 操作)，請安裝 [copilot-brag-sheet](https://github.com/microsoft/copilot-brag-sheet) 擴充功能。它會為每個工作階段新增 `save_to_brag_sheet`, `review_brag_sheet`, 和 `generate_work_log` 工具。
