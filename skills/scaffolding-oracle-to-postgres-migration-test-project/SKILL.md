# 為 Oracle 到 PostgreSQL 遷移建構整合測試專案 (Scaffolding an Integration Test Project for Oracle-to-PostgreSQL Migration)

為單一目標專案建立一個可編譯的、空的 xUnit 測試專案，其中包含交易管理和種子資料基礎設施。在撰寫測試前，每個專案執行一次。

## 工作流程 (Workflow)

```
進度：
- [ ] 步驟 1：檢查目標專案
- [ ] 步驟 2：建立 xUnit 測試專案
- [ ] 步驟 3：實作交易回滾基礎類別
- [ ] 步驟 4：實作種子資料管理員
- [ ] 步驟 5：驗證專案是否可編譯
```

**步驟 1：檢查目標專案**

讀取目標專案的 `.csproj` 以確定 .NET 版本和現有的套件參考。精確匹配這些版本 — 不要進行升級。

**步驟 2：建立 xUnit 測試專案**

- 目標設定為與受測應用程式相同的 .NET 版本。
- 加入用於 Oracle 資料庫連線和 xUnit 的 NuGet 套件。
- 僅加入對目標專案的專案參考 — 不加入其他應用程式專案。
- 加入一個已設定 Oracle 資料庫連線的 `appsettings.json`。

**步驟 3：實作交易回滾基礎類別**

- 建立一個基礎測試類別，在每個測試前開啟交易，並在測試後將其回滾。
- 擷取並處理所有例外狀況，以確保回滾。
- 使該模式可由所有下游測試類別繼承。

**步驟 4：實作種子資料管理員**

- 建立一個全域種子管理員，用於在交易範圍內載入測試資料。
- 不要提交種子資料 — 交易在每個測試後都會回滾。
- 不要使用 `TRUNCATE TABLE` — 保留現有的資料庫資料。
- 如果有現成的種子檔案，請重複使用。
- 建立種子檔案位置的命名慣例，供後續的測試建立遵循。

**步驟 5：驗證專案是否可編譯**

建構測試專案並確認其在結束前可編譯且錯誤數為零。

## 關鍵限制

- Oracle 是黃金行為來源 — 請先為 Oracle 建構環境。
- 遵循解決方案所使用的現有 .NET 和 C# 版本；不要導入較新的語言或執行階段功能。
- 產出物是僅包含基礎設施的空測試專案 — 無測試案例。
