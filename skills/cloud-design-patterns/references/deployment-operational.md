# 部署與營運模式

## 運算資源合併模式 (Compute Resource Consolidation Pattern)

**問題**：多個任務在隔離時會低效率地消耗資源。

**解決方案**：將多個任務或操作合併到單一運算單位中。

**使用時機**：
- 降低基礎設施成本
- 提高資源利用率
- 簡化部署與管理

**實作考量**：
- 將具有相似延展需求的相關任務分組
- 使用容器或微服務裝載 (hosting)
- 監控每個任務的資源使用情況
- 確保安全或可靠性所需的隔離
- 在合併與失敗隔離之間取得平衡

## 部署戳記模式 (Deployment Stamps Pattern)

**問題**：應用程式需要跨區域或跨客戶群體進行延展。

**解決方案**：部署多個獨立的應用程式元件複本（戳記），包括資料儲存，以服務不同的區域或客戶群體。

**使用時機**：
- 延展超出單一戳記的限制
- 提供區域性的資料落地 (data residency)
- 為了安全性或效能而隔離租戶

**實作考量**：
- 每個戳記都是一個完整、自給自足的部署
- 跨區域部署戳記以實現地理分佈
- 將請求路由到適當的戳記
- 使用一致的方式管理戳記部署 (IaC)
- 規劃戳記容量以及何時增加新戳記

## 外部設定儲存模式 (External Configuration Store Pattern)

**問題**：應用程式設定內嵌在部署套件中。

**解決方案**：將設定資訊從應用程式部署套件移至集中位置。

**使用時機**：
- 管理多個環境中的設定
- 在不重新部署的情況下更新設定
- 在多個應用程式之間共用設定

**實作考量**：
- 使用 Azure App Configuration、Key Vault 或類似服務
- 實作設定變更通知
- 在本機快取設定以減少相依性
- 保護敏感設定（連接字串、秘密值）
- 對設定變更進行版本控制

## Geode 模式 (Geode Pattern)

**問題**：不同區域的使用者會遇到高延遲。

**解決方案**：將後端服務部署到一組地理節點中，每個節點都可以服務任何區域的任何用戶端請求。

**使用時機**：
- 降低全球分佈使用者的延遲
- 提供跨區域的高可用性
- 實作主動-主動 (active-active) 的地理分佈

**實作考量**：
- 在多個區域部署應用程式執行個體
- 在全球範圍內複本資料（考量一致性影響）
- 將使用者路由到最近的健康區域
- 針對多主機寫入實作衝突解決方案
- 監控區域健康狀況與效能

## 靜態內容代管模式 (Static Content Hosting Pattern)

**問題**：從運算執行個體提供靜態內容效率低落。

**解決方案**：將靜態內容部署到雲端儲存服務，該服務可以將內容直接傳遞給用戶端。

**使用時機**：
- 代管圖片、影片、CSS、JavaScript 檔案
- 減輕網頁伺服器的負載
- 提高內容傳遞效能

**實作考量**：
- 使用 Blob 儲存、CDN 或靜態網站代管
- 為跨來源存取啟用 CORS
- 適當實作快取標頭
- 使用 CDN 進行全球內容分發
- 如果需要，使用 SAS 權杖保護內容
