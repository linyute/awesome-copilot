# 可靠性與復原能力模式

## 大使模式 (Ambassador Pattern)

**問題**：服務需要網路請求的 Proxy 功能 (記錄、監控、路由、安全性)。

**解決方案**：建立協助者服務，代表取用者服務或應用程式傳送網路請求。

**使用時機**：
- 卸載常見的用戶端連線工作 (監控、記錄、路由)
- 支援無法輕易修改的舊版應用程式
- 為遠端服務實作重試邏輯、斷路器或逾時處理

**實作考量**：
- 將大使部署為應用程式的 Sidecar 處理序或容器
- 考慮 Proxy 層引入的網路延遲
- 確保大使不會成為單一故障點

## 隔艙模式 (Bulkhead Pattern)

**問題**：一個元件中的故障可能會級聯並影響整個系統。

**解決方案**：將應用程式的元素隔離到集區中，以便在一個元素故障時，其他元素仍能繼續運作。

**使用時機**：
- 將關鍵資源與較不關鍵的資源隔離
- 防止一個區域的資源耗盡影響其他區域
- 對取用者和資源進行分割以提高可用性

**實作考量**：
- 為不同的後端建立獨立的連線集區
- 將服務執行個體劃分到不同的群組
- 為每個分割區設定資源限制 (CPU、記憶體、執行緒)
- 監控隔艙的健全狀況與容量

## 斷路器模式 (Circuit Breaker Pattern)

**問題**：應用程式可能會將資源浪費在可能失敗的作業上。

**解決方案**：防止應用程式重複嘗試執行可能失敗的作業，使其能在無需等待故障修復的情況下繼續運作。

**使用時機**：
- 防止級聯故障
- 當遠端服務無法使用時快速失敗
- 當服務停機時提供後備行為 (fallback)

**實作考量**：
- 定義觸發斷路器的臨界值 (失敗次數/時間範圍)
- 實作三種狀態：關閉 (Closed)、開啟 (Open)、半開啟 (Half-Open)
- 為作業設定適當的逾時值
- 記錄狀態轉換與失敗以進行診斷
- 向用戶端提供有意義的錯誤訊息

## 補償交易模式 (Compensating Transaction Pattern)

**問題**：分散式交易難以實作且可能不被支援。

**解決方案**：復原由一連串步驟所執行的工作，這些步驟共同構成一個最終一致的作業。

**使用時機**：
- 在分散式系統中實作最終一致性
- 回復執行到一半失敗的多步驟商務處理程序
- 處理無法使用 2PC (兩階段認可) 的長時間執行交易

**實作考量**：
- 為交易中的每個步驟定義補償邏輯
- 儲存足夠的狀態以復原作業
- 處理補償作業的冪等性 (Idempotency)
- 考慮補償動作之間的順序相依性

## 重試模式 (Retry Pattern)

**問題**：暫時性故障在分散式系統中很常見。

**解決方案**：讓應用程式能透過重試失敗的作業來處理預期的暫時性故障。

**使用時機**：
- 處理暫時性故障 (網路閃斷、暫時無法使用)
- 預期在短暫延遲後會成功的作業
- 謹慎考慮非冪等作業

**實作考量**：
- 在重試之間實作指數型後退 (Exponential Backoff)
- 設定最大重試次數以避免無限迴圈
- 區分暫時性故障與永久性故障
- 確保作業具備冪等性或追蹤重試嘗試
- 考慮使用抖動 (Jitter) 以避免驚群效應 (Thundering Herd)

## 健康端點監視模式 (Health Endpoint Monitoring Pattern)

**問題**：外部工具需要驗證系統的健全狀況與可用性。

**解決方案**：在應用程式中實作功能檢查，讓外部工具能定期透過公開端點進行存取。

**使用時機**：
- 監控網頁應用程式與後端服務
- 實作就緒探查 (Readiness Probe) 與存活探查 (Liveness Probe)
- 向協調器提供詳細的健全狀況資訊

**實作考量**：
- 公開健康端點 (例如：`/health`、`/ready`、`/live`)
- 檢查關鍵相依性 (資料庫、佇列、外部服務)
- 傳回適當的 HTTP 狀態碼 (200、503)
- 為敏感的健康資料實作驗證/授權
- 根據安全性內容提供不同層級的詳細資訊

## 領導者選舉模式 (Leader Election Pattern)

**問題**：分散式工作需要透過單一執行個體進行協調。

**解決方案**：藉由選舉一個執行個體作為領導者來管理共同作業的工作執行個體，進而協調分散式應用程式中的動作。

**使用時機**：
- 協調分散式工作
- 管理叢集中的共用資源
- 確保關鍵工作由單一執行個體執行

**實作考量**：
- 使用分散式鎖定機制 (Redis、etcd、ZooKeeper)
- 透過自動重新選舉來處理領導者故障
- 實作心跳 (Heartbeats) 以偵測領導者的健全狀況
- 確保隨從者 (Followers) 能快速成為領導者

## Saga 模式

**問題**：在沒有分散式交易的情況下維護跨微服務的資料一致性。

**解決方案**：在分散式交易案例中，使用一連串的本機交易來管理跨微服務的資料一致性。

**使用時機**：
- 跨越多個服務的長時間執行商務處理程序
- 不支援 2PC 的分散式交易
- 跨微服務的最終一致性需求

**實作考量**：
- 在協調 (Orchestration，集中式) 或舞步 (Choreography，事件式) 之間進行選擇
- 為回復案例定義補償交易
- 處理局部失敗與回復邏輯
- 為所有 Saga 步驟實作冪等性
- 提供明確的稽核線索與監控

## 循序護航模式 (Sequential Convoy Pattern)

**問題**：在不封鎖獨立訊息群組的情況下，按順序處理相關訊息。

**解決方案**：按定義的順序處理一組相關訊息，而不封鎖其他訊息群組。

**使用時機**：
- 訊息處理需要在群組內嚴格遵守順序
- 獨立的訊息群組可以平行處理
- 實作以工作階段 (Session) 為基礎的訊息處理

**實作考量**：
- 使用工作階段識別碼 (Session IDs) 或分割索引鍵來分組相關訊息
- 循序處理每個群組，但平行處理各個群組
- 適當處理工作階段內的訊息失敗
