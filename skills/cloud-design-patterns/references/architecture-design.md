# 架構與設計模式

## 反貪婪層模式 (Anti-Corruption Layer Pattern)

**問題**：新系統必須與使用過時模型或技術的舊有系統整合。

**解決方案**：在現代應用程式與舊有系統之間實作一個外觀 (façade) 或配接器 (adapter) 層，以防止舊有系統的限制影響新設計。

**使用時機**：
- 逐步從舊有系統遷移
- 與具有不同定義域模型 (domain models) 的第三方系統整合
- 保護現代架構免受舊有系統限制的影響

**實作考量**：
- 在定義域模型之間建立轉換層
- 在舊有與現代資料結構之間進行映射
- 將舊有系統介面隔離在抽象層之後
- 考量轉換對效能的影響
- 如果遷移完成，規劃最終的移除方案

## 前端後端分離 (Backends for Frontends, BFF) 模式

**問題**：單一後端可能無法最佳化地服務不同類型的用戶端。

**解決方案**：建立獨立的後端服務來服務特定的前端應用程式或介面。

**使用時機**：
- 不同類型的用戶端 (網頁、行動裝置、IoT) 有不同的需求
- 針對每個用戶端最佳化承載 (payload) 的大小與形狀
- 減少前端與共用後端之間的耦合

**實作考量**：
- 為每個使用者體驗或用戶端類型建立一個 BFF
- 根據前端需求量身打造 API 合約
- 避免在多個 BFF 之間重複商業邏輯
- 在 BFF 之間共用一般服務
- 管理增加的服務數量

## 閘道彙總模式 (Gateway Aggregation Pattern)

**問題**：用戶端需要來自多個後端服務的資料。

**解決方案**：使用閘道將多個個別請求彙總成單一請求。

**使用時機**：
- 減少用戶端與後端之間的通訊次數 (chattiness)
- 結合多個來源的資料以提供單一檢視
- 透過平行化後端呼叫來減少延遲

**實作考量**：
- API 閘道彙總來自多個服務的回應
- 盡可能平行執行後端呼叫
- 適當處理部分失敗的情況
- 考量彙總回應的快取
- 避免建立單一大型 (monolithic) 閘道

## 閘道卸載模式 (Gateway Offloading Pattern)

**問題**：共用功能在多個服務中重複實作。

**解決方案**：將共用或特化的服務功能卸載到閘道代理 (gateway proxy)。

**使用時機**：
- 集中處理跨切面關注點 (cross-cutting concerns)，如 SSL、驗證、記錄 (logging)
- 簡化服務實作
- 標準化共用功能

**實作考量**：
- 將 SSL 終止卸載到閘道
- 在閘道實作驗證與授權
- 處理速率限制與節流 (throttling)
- 提供請求/回應記錄
- 避免讓閘道成為瓶頸

## 閘道路由模式 (Gateway Routing Pattern)

**問題**：用戶端需要透過單一端點存取多個服務。

**解決方案**：使用單一端點將請求路由到多個服務。

**使用時機**：
- 為多個服務提供單一進入點
- 對用戶端隱藏後端服務拓撲
- 支援服務版本控制與遷移策略

**實作考量**：
- 根據 URL 路徑、標頭 (headers) 或查詢參數進行路由
- 支援 URL 重寫與轉換
- 支援 A/B 測試與金絲雀部署 (canary deployments)
- 為後端服務實作健康檢查
- 監控路由效能

## 側掛模式 (Sidecar Pattern)

**問題**：應用程式需要輔助功能，但不想與其耦合。

**解決方案**：將應用程式的元件部署到獨立的程序或容器中，以提供隔離與封裝。

**使用時機**：
- 在不修改應用程式的情況下為其增加功能
- 實作跨切面關注點 (記錄、監控、安全)
- 支援異質環境

**實作考量**：
- 將側掛與主應用程式一起部署
- 與主應用程式共用生命週期、資源與網路
- 用於代理、記錄、設定或監控
- 考量額外容器的資源開銷
- 在不同服務之間標準化側掛實作

## 絞殺者模式 (Strangler Fig Pattern)

**問題**：一次取代整個舊有系統風險很高。

**解決方案**：透過逐步將特定的功能片段替換為新的應用程式與服務，來遞增地遷移舊有系統。

**使用時機**：
- 現代化舊有應用程式
- 減少「大爆炸式」(big-bang) 遷移的風險
- 實現遞增的商業價值交付

**實作考量**：
- 識別要遞增遷移的功能
- 使用外觀或代理在舊系統與新系統之間路由
- 先遷移風險較低的元件
- 最初讓舊系統與新系統平行執行
- 規劃最終除役舊有系統的方案
