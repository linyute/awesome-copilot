# 訊息傳遞與整合模式

## 編排 (Choreography) 模式

**問題**：集中式的協調器 (orchestrators) 會產生耦合與單點失敗。

**解決方案**：透過事件驅動的協同作業，讓個別服務自行決定何時以及如何處理商業操作。

**使用時機**：
- 鬆散耦合的微服務架構
- 事件驅動系統
- 避免集中式協調的瓶頸

**實作考量**：
- 使用發佈-訂閱訊息傳遞進行事件散發
- 每個服務發佈定義域事件並訂閱相關事件
- 針對複雜的工作流程實作 Saga 模式
- 確保冪等性 (idempotency)，因為事件可能會多次傳遞
- 提供全面的記錄 (logging) 與分散式追蹤

## 領取支票 (Claim Check) 模式

**問題**：大型訊息可能會使訊息基礎設施負擔過重。

**解決方案**：將大型訊息拆分為領取支票（參考）與個別儲存的承載 (payload)。

**使用時機**：
- 訊息超過訊息系統的大小限制
- 減輕訊息匯流排的負載
- 以非同步方式處理大型檔案傳輸

**實作考量**：
- 將承載儲存在 Blob 儲存或資料庫中
- 僅透過訊息匯流排傳送參考/URI
- 為儲存的承載實作到期原則
- 處理承載儲存的存取控制
- 考量儲存成本與訊息傳輸成本

## 競爭取用者 (Competing Consumers) 模式

**問題**：單一取用者可能無法趕上訊息量。

**解決方案**：讓多個並行取用者處理來自同一個訊息通道的訊息。

**使用時機**：
- 高訊息輸送量需求
- 水平延展訊息處理
- 在多個執行個體之間進行負載平衡

**實作考量**：
- 確保訊息可以按任何順序處理
- 使用競爭取用者佇列 (Service Bus, RabbitMQ)
- 為訊息處理常式實作冪等性
- 透過重試與死信 (dead-letter) 原則處理毒藥訊息
- 根據佇列深度調整取用者數量

## 訊息橋接器 (Messaging Bridge) 模式

**問題**：不同系統使用不相容的訊息傳遞技術。

**解決方案**：建構中介程式，使原本不相容的訊息系統之間能夠進行通訊。

**使用時機**：
- 在訊息系統之間進行遷移
- 與舊有系統整合
- 連接雲端與地面端 (on-premises) 的訊息傳遞

**實作考量**：
- 在系統之間轉換訊息格式
- 處理通訊協定差異
- 如果需要，維持訊息順序
- 實作錯誤處理與重試邏輯
- 監控橋接器的效能與健康狀況

## 管道與篩選器 (Pipes and Filters) 模式

**問題**：複雜的處理任務難以維護與重複使用。

**解決方案**：將執行複雜處理的任務分解為一系列獨立、可重複使用的元素（篩選器），並透過通道（管道）連接。

**使用時機**：
- 處理具有多個轉換的資料串流
- 建構可重複使用的處理元件
- 實現獨立操作的平行處理

**實作考量**：
- 每個篩選器執行單一轉換
- 使用訊息佇列或串流連接篩選器
- 盡可能啟用平行執行
- 在篩選器內部或管道層級處理錯誤
- 支援篩選器的組合與重新排序

## 發佈者-訂閱者 (Publisher-Subscriber) 模式

**問題**：應用程式需要將資訊廣播給多個感興趣的取用者。

**解決方案**：讓應用程式能夠以非同步方式向多個取用者發佈事件，而無需將傳送者與接收者耦合。

**使用時機**：
- 向多個感興趣的對象廣播事件
- 將事件生產者與取用者解耦
- 實作事件驅動架構

**實作考量**：
- 使用基於主題 (topic-based) 或基於內容 (content-based) 的訂閱
- 確保訊息傳遞保證符合需求
- 實作訂閱篩選器以進行選擇性取用
- 處理取用者失敗而不影響發佈者
- 考量每個訂閱者的訊息順序需求

## 排程器代理監控器 (Scheduler Agent Supervisor) 模式

**問題**：分散式操作需要協調與監控。

**解決方案**：透過監控與管理工作流程的監控器，協調一組分散式服務與資源的操作。

**使用時機**：
- 協調多步驟工作流程
- 協調分散式交易
- 實作具備復原能力的長時間執行程序

**實作考量**：
- 排程器將任務分派給代理程式
- 代理程式執行工作並回報狀態
- 監控器監控進度並處理失敗
- 為失敗的步驟實作補償邏輯
- 維持狀態以供工作流程復原
