# 效能模式

## 非同步請求-回覆模式 (Asynchronous Request-Reply Pattern)

**問題**：用戶端應用程式預期同步回應，但後端處理是非同步的。

**解決方案**：在後端處理必須是非同步的，但前端需要明確回應的情況下，將後端處理與前端主機分離。

**使用時機**：
- 執行時間長的後端作業
- 用戶端應用程式無法等待同步回應
- 從網頁層卸載運算密集型作業

**實作考量**：
- 傳回 HTTP 202 (Accepted) 並包含用於狀態檢查的位置標頭 (location header)
- 為用戶端實作狀態端點以進行輪詢
- 考慮使用 Webhooks 進行回呼通知
- 使用關聯識別碼 (Correlation IDs) 來追蹤請求
- 為執行時間長的作業實作逾時

## 旁路快取模式 (Cache-Aside Pattern)

**問題**：應用程式重複從資料儲存區存取相同的資料。

**解決方案**：視需要將資料從資料儲存區隨選載入到快取中。

**使用時機**：
- 頻繁存取、讀取密集的資料
- 不常更改的資料
- 減少主要資料儲存區的負載

**實作考量**：
- 在存取資料儲存區之前檢查快取
- 在快取未命中 (lazy loading) 時將資料載入快取
- 設定適當的快取到期策略
- 實作快取失效策略
- 平穩處理快取失敗 (回復到資料儲存區)
- 在分散式案例中考慮快取一致性

## CQRS (命令查詢職責分離) 模式

**問題**：讀取和寫入工作負載具有不同的需求和調整規模的需求。

**解決方案**：使用不同的介面，將讀取資料的作業與更新資料的作業分開。

**使用時機**：
- 讀取和寫入工作負載具有截然不同的效能特性
- 不同的團隊負責讀取和寫入端
- 需要在共同作業案例中防止合併衝突
- 讀取和寫入之間的複雜商務邏輯不同

**實作考量**：
- 分離讀取和寫入模型
- 使用事件溯源 (Event Sourcing) 來同步模型
- 獨立調整讀取和寫入端的規模
- 考慮最終一致性的影響
- 為命令與查詢實作適當的安全性

## 索引表模式 (Index Table Pattern)

**問題**：查詢頻繁參考未有效編製索引的欄位。

**解決方案**：在資料儲存區中查詢頻繁參考的欄位上建立索引。

**使用時機**：
- 改善查詢效能
- 支援多種查詢模式
- 處理不具備原生索引功能的 NoSQL 資料庫

**實作考量**：
- 針對特定查詢建立經過最佳化的個別資料表/集合
- 使用事件或觸發程序以非同步方式維護索引
- 考慮重複資料的儲存開銷
- 處理索引更新失敗與不一致的情況

## 具體化檢視模式 (Materialized View Pattern)

**問題**：資料的格式不適合所需的查詢作業。

**解決方案**：當資料的格式不適合查詢作業時，在一個或多個資料儲存區的資料上產生預先填滿的檢視。

**使用時機**：
- 對正規化資料進行複雜查詢
- 改善複雜聯結/彙總的讀取效能
- 有效支援多種查詢模式

**實作考量**：
- 使用背景作業或觸發程序以非同步方式重新整理檢視
- 考慮具體化資料的過期容忍度
- 在儲存成本與查詢效能之間取得平衡
- 盡可能實作增量重新整理

## 優先權佇列模式 (Priority Queue Pattern)

**問題**：某些請求需要比其他請求更快的處理速度。

**解決方案**：優先處理傳送至服務的請求，讓優先權較高的請求能更快獲得處理。

**使用時機**：
- 為不同客戶提供不同的服務層級
- 在較不重要的作業之前處理關鍵作業
- 管理具有不同重要性的混合工作負載

**實作考量**：
- 使用訊息優先權 Metadata
- 為不同的優先權層級實作多個佇列
- 防止低優先權訊息產生飢餓現象
- 依優先權監控佇列深度與處理時間

## 以佇列為基礎的負載調節模式 (Queue-Based Load Leveling Pattern)

**問題**：間歇性的沉重負載可能會使服務負荷過重。

**解決方案**：使用佇列作為工作與服務之間的緩衝區，以平滑間歇性的沉重負載。

**使用時機**：
- 保護服務免於流量尖峰的影響
- 將製造者與取用者解耦
- 啟用非同步處理

**實作考量**：
- 選擇適當的佇列技術 (Azure Storage Queue、Service Bus 等)
- 監控佇列長度以偵測飽和度
- 根據佇列深度實作自動調整規模
- 設定適當的訊息存活時間 (TTL)
- 使用寄不出的信件佇列處理有毒訊息

## 速率限制模式 (Rate Limiting Pattern)

**問題**：必須控制服務耗用量以防止資源耗盡。

**解決方案**：控制應用程式、租用戶或服務對資源的耗用，以防止資源耗盡與節流。

**使用時機**：
- 保護後端服務免於超載
- 實作公平使用原則
- 防止單一租用戶壟斷資源

**實作考量**：
- 實作權杖貯槽 (token bucket)、漏桶 (leaky bucket) 或固定視窗演算法
- 當超過限制時傳回 HTTP 429 (Too Many Requests)
- 為用戶端提供 Retry-After 標頭
- 考慮針對不同用戶端/層級設定不同的限制
- 讓限制為可設定且可監控的

## 分區模式 (Sharding Pattern)

**問題**：單一資料儲存區在儲存容量與效能上可能有所限制。

**解決方案**：將資料儲存區劃分為一組水平分割區或分區。

**使用時機**：
- 規模擴充超出單一資料庫的限制
- 透過減少資料集大小來改善查詢效能
- 將負載分散到多個資料庫

**實作考量**：
- 選擇適當的分區索引鍵 (雜湊、範圍或以清單為基礎)
- 透過選取平衡的分區索引鍵來避免熱點分割區 (hot partitions)
- 小心處理跨分區查詢
- 規劃分區重新平衡與分割
- 考慮管理多個分區的營運複雜性

## 節流模式 (Throttling Pattern)

**問題**：必須限制資源耗用量以防止系統超載。

**解決方案**：控制應用程式、租用戶或服務所使用的資源耗用量。

**使用時機**：
- 確保系統在定義的容量內運作
- 防止在尖峰負載期間資源耗盡
- 強制執行以 SLA 為基礎的資源配置

**實作考量**：
- 在 API 閘道或服務層級實作
- 使用不同策略：拒絕請求、佇列或降低服務品質
- 傳回適當的 HTTP 狀態碼 (429、503)
- 向用戶端提供關於節流的明確回饋
- 監控節流指標以調整容量
