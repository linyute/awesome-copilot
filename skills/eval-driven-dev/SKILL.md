# Python LLM 應用程式的評估驅動開發

你正在建置一個**自動化評估管線**，用於對基於 Python 的 AI 應用程式進行端到端測試 — 以與真實使用者相同的方式、使用真實輸入來執行它 — 然後使用評估器對輸出進行評分，並透過 `pixie test` 產生通過/失敗結果。

**你測試的是應用程式本身** — 它的請求處理、背景組合 (Context assembly)（它如何收集資料、建構提示詞、管理對話狀態）、路由和回應格式化。該應用程式使用 LLM，這使得輸出具有非確定性 — 這就是為什麼你使用評估器（LLM 即評審、相似度分數）而不是 `assertEqual` — 但測試的對象是應用程式的程式碼，而非 LLM。

在評估期間，應用程式自身的程式碼會真實執行 — 路由、提示詞組合、LLM 呼叫、回應格式化 — 內容不會被模擬 (Mock) 或替換。但應用程式從外部來源（資料庫、快取、第三方 API、語音流）讀取的資料會透過檢測 (Instrumentations) 替換為測試指定的值。這意味著每個測試案例都能精確控制應用程式看到的資料，同時仍能測試完整的應用程式程式碼路徑。

**規則：應用程式的 LLM 呼叫必須發送到真實的 LLM。** 不要用虛假的實作來替換、模擬、截獲 LLM。LLM 是核心價值產生成分 — 替換它會使評估變成循環論證（你同時控制了輸入和輸出，因此分數毫無意義）。如果專案的測試套件包含 LLM 模擬模式，那是為了專案自身的單元測試 — 請「不要」將其套用於評估的 Runnable。

**交付成果是帶有真實分數、可運作的 `pixie test` 執行結果** — 不是計畫，不僅僅是檢測，也不僅僅是資料集。

此技能旨在執行工作，而非描述工作。請閱讀程式碼、編輯檔案、執行命令，並產出一個可運作的管線。

## 在開始之前

**首先，啟動虛擬環境**。識別專案正確的虛擬環境並啟動它。虛擬環境啟動後，執行此技能資源中包含的 `setup.sh`。
該指令碼會將 `eval-driven-dev` 技能和 `pixie-qa` Python 套件更新到最新版本，如果尚未初始化則初始化 pixie 工作目錄，並在背景啟動網頁伺服器以向使用者顯示更新。

**設定錯誤處理 — 哪些可以跳過 vs 哪些必須成功：**

- **技能更新失敗** → 可以繼續。現有的技能版本已足夠。
- **pixie-qa 升級失敗但已安裝過** → 可以使用現有版本繼續。
- **pixie-qa 未安裝且安裝失敗** → **停止。** 請求使用者協助。沒有 `pixie` 套件，工作流程無法進行。
- **`pixie init` 失敗** → **停止。** 請求使用者協助。
- **`pixie start` (網頁伺服器) 失敗** → **停止。** 請求使用者協助。檢查 pixie 根目錄下的 `server.log` 以進行診斷。常見原因：連接埠衝突、缺少相依性、環境反應緩慢。沒有網頁伺服器請勿繼續執行 — 使用者需要它來查看評估結果。

## 工作流程

依次執行步驟 1–6，不要停頓。不要在中間步驟請求使用者確認 — 請自行驗證每個步驟並繼續執行。

**如何工作 — 在執行任何操作前閱讀：**

- **一次只做一個步驟。** 僅閱讀當前步驟的指示。在執行步驟 1 時「不要」閱讀步驟 2–6。
- **僅在步驟指示時閱讀參考文件。** 每個步驟都會指名特定的參考檔案。當你到達該步驟時再閱讀 — 不要提前閱讀。
- **立即建立成品。** 在為某個子步驟閱讀程式碼後，先撰寫該子步驟的輸出檔案再繼續。不要在撰寫任何內容前累積對多個子步驟的理解。
- **驗證後再繼續。** 每個步驟都有一個檢查點。請驗證它，然後進入下一步。在驗證當前步驟時，不要規劃後續步驟。

**何時停止並請求協助：**

有些阻礙是無法且不應繞過的。當你遇到以下情況時，請**立即停止並請求使用者協助** — 不要嘗試規避：

- **應用程式因缺少環境變數或設定而無法執行**：應用程式需要未設定且無法推斷的環境變數或設定。請勿透過模擬、偽造或替換應用程式元件來規避 — 評估必須測試真實的生產程式碼。請要求使用者修正環境設定。
- **應用程式匯入失敗顯示專案損壞**：如果因缺少系統相依性或 Python 版本不相容（而不僅僅是你可以安裝的缺失 pip 套件）而無法匯入應用程式的核心模組，請要求使用者修正專案設定。
- **入口點模糊不清**：如果應用程式有多個同樣合理的入口點，且專案分析未能明確哪個最重要，請詢問使用者要針對哪一個。

你應自行解決的阻礙（無需詢問）：缺失 Python 套件（安裝它們）、缺失 `pixie` 套件（安裝它）、連接埠衝突（選擇另一個連接埠）、檔案權限問題（修復它們）。

**按順序執行步驟 1–6。** 如果使用者的提示明確表示早期步驟已完成（例如：「執行現有的測試」、「重新執行評估」），請跳至相應步驟。如有疑慮，請從步驟 1 開始。

### 步驟 1：理解應用程式並定義評估標準

**首先，檢查使用者提示中的具體要求。** 在閱讀應用程式程式碼之前，先檢查使用者要求的內容：

- **參考文件或規格**：提示中是否提到了要遵循的檔案（例如：「遵循 EVAL_SPEC.md 中的規格」、「使用 REQUIREMENTS.md 中的方法論」）？如果是，請**先閱讀該檔案** — 它可能指定了會覆蓋你預設設定的資料集、評估維度、通過標準或方法論。
- **指定的資料集或資料來源**：提示中是否引用了特定的資料檔案（例如：「使用來自 eval_inputs/research_questions.json 的問題」、「使用 call_scenarios.json 中的場景」）？如果是，請**閱讀這些檔案** — 你必須將其作為評估資料集的基礎，而非憑空捏造通用的替代方案。
- **指定的評估維度**：提示中是否指名了特定的品質面向（例如：「評估事實性、完整性和偏見」、「測試身分驗證和工具呼叫正確性」）？如果是，**每個指名的維度都必須在你的測試檔案中具備對應的評估器。**

如果提示指定了上述任何內容，它們具有優先權。請在繼續之前閱讀並納入它們。

步驟 1 有三個子步驟。每個子步驟都會閱讀其專屬的參考檔案並產生其專屬的輸出檔案。**在開始下一步之前，請完整完成當前的子步驟。**

#### 子步驟 1a：專案分析

> **參考**：現在閱讀 `references/1-a-project-analysis.md`。

在檢視程式碼結構或入口點之前，先了解此軟體在現實世界中的作用 — 它的目的、它的使用者、真實輸入的複雜度以及它在哪裡失敗。這種理解驅動著後續的每一個決策：哪些入口點最重要、要定義哪些評估標準、使用哪些追蹤輸入，以及要建立哪些資料集項目。在繼續下一步之前，請撰寫詳細的背景檔案。**註記**：專案可能包含 `tests/`、`fixtures/`、`examples/`、模擬伺服器和文件 — 這些是專案自身的開發基礎設施，**不是**你的評估管線的資料來源。在獲取追蹤輸入和資料集內容時請忽略它們。

> **檢查點**：已撰寫 `pixie_qa/00-project-analysis.md` — 涵蓋軟體的作用、目標使用者、能力清單（如果專案具備能力，至少列出 3 個）、現實輸入特性，以及硬性問題 / 失敗模式（至少 2 個）。

#### 子步驟 1b：入口點與執行流程

> **參考**：現在閱讀 `references/1-b-entry-point.md`。

閱讀原始程式碼以理解應用程式如何啟動以及真實使用者如何叫用它。使用來自 `pixie_qa/00-project-analysis.md` 的**能力清單**來排列入口點的優先順序 — 專注於測試最有價值能力的入口點，而不僅僅是第一個被發現的。在繼續下一步之前，請撰寫詳細的背景檔案。

> **檢查點**：已撰寫 `pixie_qa/01-entry-point.md` — 涵蓋入口點、執行流程、面向使用者的介面以及環境要求。

#### 子步驟 1c：評估標準

> **參考**：現在閱讀 `references/1-c-eval-criteria.md`。

定義應用程式的使用案例和評估標準。從 `pixie_qa/00-project-analysis.md` 中的**能力清單**衍生使用案例。從**硬性問題 / 失敗模式**中衍生評估標準 — 不要使用通用的品質維度。使用案例驅動資料集建立（步驟 4）；評估標準驅動評估器選擇（步驟 3）。在繼續下一步之前，請撰寫詳細的背景檔案。

> **檢查點**：已撰寫 `pixie_qa/02-eval-criteria.md` — 涵蓋使用案例、評估標準及其適用範圍。在此之前「不要」閱讀步驟 2 的指示。

### 步驟 2：檢測應用程式、執行並擷取參考追蹤

步驟 2 有三個子步驟。每個子步驟都會閱讀其專屬的參考檔案。**在開始下一步之前，請完成當前的子步驟。**

#### 子步驟 2a：使用 `wrap` 進行檢測

> **參考**：現在閱讀 `references/2a-instrumentation.md`。

在應用程式的資料邊界增加 `wrap()` 呼叫，以便評估控具可以注入受控輸入並擷取輸出。這使得應用程式在不改變邏輯的情況下可被測試。

> **檢查點**：已在所有資料邊界增加 `wrap()` 呼叫。`pixie_qa/02-eval-criteria.md` 中的每個評估標準都有對應的資料點。

#### 子步驟 2b：實作 Runnable

> **參考**：現在閱讀 `references/2b-implement-runnable.md`。

撰寫一個 Runnable 類別，讓評估控具能像真實使用者一樣叫用應用程式。Runnable 應該保持簡單 — 它只是將應用程式的真實入口點連接到控具介面。如果它變得很複雜，說明有些地方出錯了。

> **檢查點**：已撰寫 `pixie_qa/run_app.py`。Runnable 使用真實的 LLM 設定呼叫應用程式的真實入口點 — 沒有模擬、沒有偽造、沒有元件替換。

#### 子步驟 2c：擷取並驗證參考追蹤

> **參考**：現在閱讀 `references/2c-capture-and-verify-trace.md`。

透過 Runnable 執行應用程式並擷取追蹤 (Trace)。追蹤證明了檢測和 Runnable 運作正常，並提供了步驟 4 中建立資料集所需的資料形狀。

> **檢查點**：`pixie_qa/reference-trace.jsonl` 已存在。所有預期的 `wrap` 項目和 `llm_span` 項目皆已出現。`pixie format` 顯示了評估所需的所有資料點。在此之前「不要」閱讀步驟 3 的指示。

### 步驟 4：建置資料集

> **參考**：現在閱讀 `references/4-build-dataset.md` 以了解詳細子步驟。

**目標**：建立測試場景，將所有內容串聯在一起 — Runnable (步驟 2)、評估器 (步驟 3) 和使用案例 (步驟 1c)。每個資料集項目定義了要發送給應用程式的內容、應用程式應從外部服務中看到的資料，以及如何對結果進行評分。使用步驟 2 中的參考追蹤作為資料形狀和欄位名稱的事實來源。涵蓋 `pixie_qa/00-project-analysis.md` 中的能力清單，並包含針對該檔案中識別出的失敗模式的項目。**「不要」將專案自身的測試固定裝置、模擬伺服器或範例資料用作資料集的 `eval_input` 內容** — 請改用現實世界的資料。**應用程式中的每個 `wrap(purpose="input")` 在每個項目中的 `eval_input` 必須具有預先擷取的內容** — 當應用程式具有 Input Wrap 時，「不要」讓 `eval_input` 保持為空。

> **檢查點**：已在 `pixie_qa/datasets/<name>.json` 建立資料集 JSON，其中包含涵蓋所有使用案例的多樣化項目。**資料集真實性稽核通過** — 項目使用了具代表性規模的現實世界資料，無專案測試固定裝置污染，至少有一個項目針對結果不確定的失敗模式，且每個 `eval_input` 都具有針對所有 Input Wrap 擷取的內容。在此之前「不要」閱讀步驟 5 的指示。

### 步驟 5：執行 `pixie test` 並修復機械性問題

> **參考**：現在閱讀 `references/5-run-tests.md` 以了解詳細子步驟。

**目標**：端到端執行完整的管線，並使其在沒有機械性錯誤的情況下運作。此步驟嚴格僅限於修復 pixie QA 元件（資料集、Runnable、自訂評估器）中的設定和資料問題 — 「不是」修復應用程式本身或評估結果品質。一旦 `pixie test` 無誤完成並為每個項目產生真實的評估器分數，此步驟即告完成。

> **檢查點**：`pixie test` 執行完成。每個資料集項目都具有評估器分數（真實的 `EvaluationResult` 或 `PendingEvaluation`）。無設定錯誤、無匯入失敗、無資料驗證錯誤。
>
> 如果測試出錯，那是你的 QA 元件中的機械性錯誤 — 請修復並重新執行。但一旦測試產生了分數，請繼續。此處「不要」評估結果品質 — 那是步驟 6。

**在測試產生分數後，務必進入步驟 6。** 分析是至關重要的最後一步 — 沒有它，待處理的評估將永遠無法完成，使用者收到的將是未經解釋的原始分數，且沒有可執行的見解。在此處「不要」停下來詢問使用者是否繼續。

**疊代執行的循環規則**：每次成功的 `pixie test` 調用都會建立一個具體的 `pixie_qa/results/<test_id>` 目錄，並開始一個新的分析循環。在編輯應用程式程式碼、提示詞、資料集、評估器或重新執行 `pixie test` 之前，請為該特定的結果目錄完成步驟 6。不要跳過早期的循環而僅分析最後一次執行。

### 步驟 6：分析結果

> **參考**：現在閱讀 `references/6-analyze-outcomes.md` — 它包含完整的分析三階段流程、撰寫指引以及輸出格式要求。

**目標**：透過結構化、資料驅動的流程分析 `pixie test` 的結果，以針對測試案例品質、評估器品質和應用程式品質產生可執行的見解。此步驟會完成待處理的評估，撰寫針對每個項目和資料集的分析，並產生優先排序的行動計畫。每一項陳述都必須由評估執行的具體資料支撐 — 不得推測或含糊其辭。

**持久化的分析成品**：在此簡化流程中，僅在資料集層級和測試執行層級持久化分析。這些成品仍採用**詳細版本**（供代理取用：資料點、證據鏈、推理鏈）加上**摘要版本**（供人工審閱：2 分鐘內可讀完的簡潔 TLDR）。不要建立針對各別項目的分析檔案。

**硬性完成關卡**：滿足以下所有條件，步驟 6 始告完成：

- 每個 `pixie_qa/results/<test_id>/dataset-*/entry-*/evaluations.jsonl` 中所有的 `"status": "pending"` 項目都已被替換為包含 `score` 和 `reasoning` 的已評分結果。
- 每個資料集目錄都包含 `analysis.md` 和 `analysis-summary.md`。
- 測試執行根目錄包含 `action-plan.md` 和 `action-plan-summary.md`。
- 你已針對 `pixie_qa/results/<test_id>` 執行了此技能 `resources/` 目錄下的步驟 6 驗證指令碼，且回報成功。

**明確不充分的情況：**

- 僅撰寫單個頂層檔案，如 `pixie_qa/06-analysis.md`。
- 宣稱待處理的評估留待使用者在網頁 UI 中審查。
- 在未更新 `evaluations.jsonl` 的情況下，宣稱某個項目「可能通過」。

## 網頁伺服器管理

pixie-qa 在背景執行一個網頁伺服器，用於向使用者顯示背景資訊、追蹤記錄和評估結果。它由設定指令碼自動啟動（透過 `pixie start` 啟動一個分離的背景進程並立即傳回）。

當使用者完成評估驅動開發工作流程時，請告知他們網頁伺服器仍在執行，你可以透過以下命令將其關閉：

```bash
pixie stop
```

重要提示：網頁伺服器關閉後，網頁 UI 將無法存取。因此，請僅在使用者確認已完成所有網頁 UI 功能後再停止伺服器。如果他們想繼續使用網頁 UI，請「不要」停止伺服器。
