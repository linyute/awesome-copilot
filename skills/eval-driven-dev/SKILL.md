# 針對 Python LLM 應用程式的評估驅動開發 (Eval-Driven Development)

您正在建置一個**自動化 QA 管線**，用於對 Python 應用程式進行端對端測試 — 以與真實使用者相同的方式執行它，使用真實輸入 — 然後使用評估器 (evaluators) 為輸出評分，並透過 `pixie test` 產生通過/失敗結果。

**您正在測試的是應用程式本身** — 它的請求處理、內容組合 (它如何收集資料、建構提示字元、管理對話狀態)、路由和回應格式化。應用程式使用 LLM，這使得輸出具有非確定性 — 這就是為什麼您使用評估器 (LLM 作為評審、相似度分數) 而非 `assertEqual` 的原因 — 但受測對象是應用程式的程式碼，而不是 LLM。

在評估期間，應用程式自身的程式碼會真實執行 — 路由、提示字元組合、LLM 呼叫、回應格式化 — 任何部分都不會被模擬 (mocked) 或虛設 (stubbed)。但是應用程式從外部來源 (資料庫、快取、第三方 API、語音串流) 讀取的資料會透過檢測 (instrumentations) 替換為測試指定的數值。這意味著每個測試案例都能精確控制應用程式看到的資料，同時仍能執行完整的應用程式程式碼路徑。

**交付物是一個帶有真實分數且可運作的 `pixie test` 執行結果** — 而不是計畫，不僅僅是檢測，也不僅僅是資料集。

此技能的核心在於實作，而非描述。閱讀程式碼、編輯檔案、執行指令、產生一個可運作的管線。

## 在您開始之前

**首先，啟動虛擬環境**。識別專案正確的虛擬環境並啟動它。虛擬環境啟動後，執行技能資源中包含的 setup.sh。
該指令稿會將 `eval-driven-dev` 技能和 `pixie-qa` python 套件更新至最新版本，如果尚未初始化則初始化 pixie 工作目錄，並在背景啟動網頁伺服器以向使用者顯示更新。如果技能或套件更新失敗，請繼續執行 — 不要讓這些失敗阻礙其餘的工作流程。

## 工作流程

直接依序執行步驟 1–6，不要停頓。不要在中間步驟要求使用者確認 — 請自行驗證每個步驟並繼續。

**如何工作 — 在執行任何操作前請先閱讀：**

- **一次一個步驟。** 僅閱讀目前步驟的指示。在執行步驟 1 時，請勿閱讀步驟 2–6。
- **僅在步驟要求時閱讀參考文件。** 每個步驟都會指名特定的參考檔案。當您到達該步驟時再閱讀它 — 不要提前閱讀。
- **立即建立成品。** 閱讀完子步驟的程式碼後，在進入下一步之前，請先撰寫該子步驟的輸出檔案。不要在撰寫任何內容之前累積多個子步驟的理解。
- **驗證，然後繼續。** 每個步驟都有一個查核點 (checkpoint)。驗證它，然後進入下一個步驟。不要在驗證目前步驟時規劃未來的步驟。

**依序執行步驟 1–6。** 如果使用者的提示字元明確表示先前的步驟已經完成 (例如，「執行現有的測試」、「重新執行評估」)，請跳至適當的步驟。如有疑慮，請從步驟 1 開始。

### 步驟 1：理解應用程式並定義評估準則

**首先，檢查使用者的提示字元是否有特定需求。** 在閱讀應用程式程式碼之前，檢查使用者要求的內容：

- **參考文件或規格**：提示字元是否提到要遵循的檔案 (例如，「遵循 EVAL_SPEC.md 中的規格」、「使用 REQUIREMENTS.md 中的方法論」)？如果是，**請先閱讀該檔案** — 它可能指定了會覆寫您預設設定的資料集、評估維度、通過準則或方法論。
- **指定的資料集或資料來源**：提示字元是否參考了特定的資料檔案 (例如，「使用 eval_inputs/research_questions.json 中的問題」、「使用 call_scenarios.json 中的場景」)？如果是，**請閱讀這些檔案** — 您必須使用它們作為評估資料集的基礎，而不是虛構通用的替代方案。
- **指定的評估維度**：提示字元是否命名了要評估的特定品質面向 (例如，「評估事實性、完整性和偏差」、「測試身分驗證和工具呼叫正確性」)？如果是，**每個命名的維度都必須在您的測試檔案中具有對應的評估器**。

如果提示字元指定了上述任何內容，它們具有優先權。在繼續之前閱讀並納入它們。

步驟 1 有兩個子步驟。每個子步驟都有自己的參考檔案並產生自己的輸出檔案。**在開始下一個子步驟之前，請完整完成前一個子步驟。**

#### 子步驟 1a：進入點與執行流程

> **參考文件**：現在閱讀 `references/1-a-entry-point.md`。

閱讀原始碼以理解應用程式如何啟動，以及真實使用者如何呼叫它。在繼續之前，將您的發現寫入 `pixie_qa/01-entry-point.md`。

> **查核點**：`pixie_qa/01-entry-point.md` 已撰寫，包含進入點、執行流程、面向使用者的介面以及環境需求。

#### 子步驟 1b：評估準則

> **參考文件**：現在閱讀 `references/1-b-eval-criteria.md`。

定義應用程式的使用案例和評估準則。使用案例驅動資料集建立 (步驟 4)；評估準則驅動評估器選擇 (步驟 3)。在繼續之前，將您的發現寫入 `pixie_qa/02-eval-criteria.md`。

> **查核點**：`pixie_qa/02-eval-criteria.md` 已撰寫，包含使用案例、評估準則及其適用範圍。請勿閱讀步驟 2 的指示。

### 步驟 2：使用 `wrap` 檢測並擷取參考追蹤 (Reference Trace)

> **參考文件**：現在閱讀 `references/2-wrap-and-trace.md` 以取得詳細子步驟。

**目標**：透過控制應用程式的外部資料並擷取其輸出來使其可測試。在資料邊界呼叫 `wrap()` 讓測試控置台 (test harness) 能夠注入受控輸入 (替換真實的資料庫/API 呼叫) 並擷取輸出以供評分。`Runnable` 類別提供了 `pixie test` 用於設定、呼叫和卸載應用程式的生命週期介面。使用 `pixie trace` 擷取的參考追蹤可證明檢測功能正常，並提供建立步驟 4 資料集所需確切資料形狀。

> **查核點**：`pixie_qa/scripts/run_app.py` 已撰寫並驗證。`pixie_qa/reference-trace.jsonl` 已存在，且使用 `pixie format` 格式化時會顯示所有預期的資料點。請勿閱讀步驟 3 的指示。

> **查核點**：所有評估器皆已實作。`pixie_qa/03-evaluator-mapping.md` 已撰寫，包含準則對應評估器的關聯。請勿閱讀步驟 4 的指示。

### 步驟 4：建置資料集

> **參考文件**：現在閱讀 `references/4-build-dataset.md` 以取得詳細子步驟。

**目標**：建立將所有內容連繫在一起的測試情境 — 可執行程式 (步驟 2)、評估器 (步驟 3) 和使用案例 (步驟 1b)。每個資料集項目都定義了要傳送給應用程式的內容、應用程式應從外部服務看到的資料，以及如何對結果評分。使用步驟 2 中的參考追蹤作為資料形狀和欄位名稱的真實來源。

> **查核點**：資料集 JSON 已在 `pixie_qa/datasets/<name>.json` 建立，其中包含涵蓋所有使用案例的多樣化項目。請勿閱讀步驟 5 的指示。

### 步驟 5：執行以評估為基礎的測試

> **參考文件**：現在閱讀 `references/5-run-tests.md` 以取得詳細子步驟。

**目標**：端對端執行完整管線，並驗證其產生真實分數。此步驟旨在讓機制運作 — 修正任何設定或資料問題，直到每個資料集項目都能執行並獲得評分。測試產生結果後，執行 `pixie analyze` 進行模式分析。

> **查核點**：測試已執行並產生真實分數。已產生分析結果。
>
> 如果測試出錯，那是設定上的錯誤 — 請修正並重新執行。但如果測試產生了真實的通過/失敗分數，這就是交付物。
>
> **停止門檻 (STOP GATE) — 在測試產生分數後，執行任何操作前請先閱讀：**
>
> - 如果使用者的原始提示字元僅要求設定 (「設定 QA」、「新增測試」、「新增評估」、「設定評估」)，**請在此停止**。向使用者報告測試結果：「QA 設定已完成。測試顯示 N/M 通過。[簡短摘要]。是否需要我調查失敗原因並進行迭代？」請勿繼續執行步驟 6。
> - 如果使用者的原始提示字元明確要求迭代 (「修正」、「改進」、「偵錯」、「迭代」、「調查失敗」、「使測試通過」)，請繼續執行步驟 6。

## 網頁伺服器管理

pixie-qa 在背景執行一個網頁伺服器，用於向使用者顯示內容、追蹤和評估結果。它由 setup 指令稿自動啟動 (透過 `pixie start`，這會啟動一個分離的背景處理程序並立即傳回)。

當使用者完成評估驅動開發工作流程時，告知他們網頁伺服器仍在執行中，您可以透過以下指令進行清理：

```bash
pixie stop
```

重要提示：網頁伺服器停止後，網頁 UI 將無法存取。因此，請務必在使用者確認已完成所有網頁 UI 功能後再停止伺服器。如果他們想繼續使用網頁 UI，請勿停止伺服器。

每當您重新啟動工作流程時，請務必再次執行資源中的 setup.sh 指令稿，以確保網頁伺服器正在執行：
