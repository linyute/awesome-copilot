# 調查與反覆修正

此參考資料涵蓋評估驅動開發 (eval-driven-dev) 流程的步驟 6：調查測試失敗、找出根本原因並進行反覆修正。

---

## 停止 —— 在繼續之前請先檢查

**在進行任何調查或反覆修正工作之前，您必須決定是繼續進行還是停止並詢問使用者。**

如果使用者的原始提示詞明確要求進行反覆修正，請**立即繼續** —— 尋找如「修正 (fix)」、「改進 (improve)」、「除錯 (debug)」、「反覆修正 (iterate)」、「調查失敗 (investigate failures)」或「使測試通過 (make tests pass)」等詞彙。在這種情況下，請繼續執行下方的調查步驟。

**否則，請在此處停止。**向使用者報告測試結果：

> 「QA 設定已完成。測試顯示 N/M 通過。[如果有失敗，請簡短摘要]。需要我調查失敗原因並進行反覆修正嗎？」

**在使用者確認之前，請勿繼續進行調查。**這是預設做法 —— 大多數如「設定評估 (set up evals)」、「新增測試 (add tests)」、「設定 QA (set up QA)」或「新增評估 (add evaluations)」之類的提示詞僅要求進行設定，而非反覆修正。

---

## 逐步調查

當使用者已確認（或其原始提示詞明確關於反覆修正時），請繼續：

### 1. 閱讀分析報告

首先閱讀在步驟 5 中產生的分析報告。分析檔案位於 `{PIXIE_ROOT}/results/<test_id>/dataset-<index>.md`。這些檔案包含由 LLM 產生的深入見解，關於測試執行中成功與失敗的模式。使用分析報告來決定優先調查哪些失敗，並瞭解系統性問題。

### 2. 獲取詳細測試輸出

```bash
pixie test -v    # 顯示每個案例的分數和推論
```

擷取完整的詳細輸出。對於每個失敗的案例，請記錄：

- `entry_kwargs`（發送了什麼）
- `captured output`（應用程式產生了什麼）
- `expected_output`（預期是什麼，如果適用的話）
- 評估器分數和推論

### 3. 檢查追蹤 (trace) 資料

對於每個失敗的案例，查找完整追蹤以查看應用程式內部發生了什麼：

```python
from pixie import DatasetStore

store = DatasetStore()
ds = store.get("<dataset-name>")
for i, item in enumerate(ds.items):
    print(i, item.eval_metadata)   # trace_id 在這裡
```

然後檢查完整的 span 樹：

```python
import asyncio
from pixie import ObservationStore

async def inspect(trace_id: str):
    store = ObservationStore()
    roots = await store.get_trace(trace_id)
    for root in roots:
        print(root.to_text())   # 完整的 span 樹：輸入、輸出、LLM 訊息

asyncio.run(inspect("在此填入追蹤識別碼"))
```

### 4. 根本原因分析 (Root-cause analysis)

逐步檢查追蹤並確切找出失敗的源頭。常見模式如下：

**與 LLM 相關的失敗**（透過提示詞/模型/評估變更來修正）：

| 徵兆                                                   | 可能原因                                                      |
| ------------------------------------------------------ | ------------------------------------------------------------- |
| 儘管工具結果正確，但輸出內容在事實上是錯誤的           | 提示詞未指示 LLM 忠實使用工具輸出                             |
| 代理程式路由到錯誤的工具/交接                         | 路由提示詞或交接描述模糊不清                                  |
| 輸出格式錯誤                                           | 提示詞中缺失格式指示                                          |
| LLM 產生幻覺而非使用工具                               | 提示詞未強制執行工具使用                                      |

**非 LLM 失敗**（透過傳統程式碼變更修正，超出評估範圍）：

| 徵兆                                              | 可能原因                                                |
| ------------------------------------------------- | ------------------------------------------------------- |
| 工具傳回了錯誤的資料                              | 工具實作中的錯誤 —— 修正工具而非評估                    |
| 由於關鍵字不匹配，工具根本未被呼叫                | 工具選擇邏輯損壞 —— 修正程式碼                          |
| 資料庫傳回過時/錯誤的記錄                         | 資料問題 —— 獨立修正                                    |
| API 呼叫失敗並出現錯誤                            | 基礎設施問題                                            |

對於非 LLM 失敗：在調查日誌中記錄它們並建議程式碼修正，但**請勿調整評估期望或閾值來適應非 LLM 程式碼中的錯誤**。評估測試應在假設系統其餘部分正常運作的情況下，衡量 LLM 的品質。

### 5. 記錄發現

**每次失敗調查都應與修正方案一起記錄。**包含：

````markdown
### <日期> — 失敗調查

**資料集**：`qa-golden-set`
**結果**：3/5 案例通過 (60%)

#### 失敗案例 1：「哪些排數有額外腿部空間？」

- **entry_kwargs**：`{"user_message": "哪些排數有額外腿部空間？"}`
- **captured output**： 「抱歉，我沒有關於額外腿部空間的確切排數資訊...」
- **expected_output**： 「第 5-8 排是具有額外腿部空間的優選經濟艙 (Economy Plus)」
- **評估器分數**： 0.1 (Factuality)
- **評估器推論**： 「輸出聲稱不知道答案，但參考資料明確指出是第 5-8 排...」

**追蹤分析**：
檢查追蹤 `abc123`。span 樹顯示：

1. 分流代理程式 (Triage Agent) 路由到 FAQ 代理程式 ✓
2. FAQ 代理程式呼叫 `faq_lookup_tool("哪些排數有額外腿部空間？")` ✓
3. `faq_lookup_tool` 傳回 「抱歉，我不知道...」 ← **根本原因**

**根本原因**：`faq_lookup_tool` (customer_service.py:112) 使用關鍵字比對。
座位 FAQ 項目是由關鍵字 `["seat", "seats", "seating", "plane"]` 觸發的。
問題 「哪些排數有額外腿部空間？」 不包含任何這些關鍵字，因此
落入預設的 「我不知道」 回應。

**分類**：非 LLM 失敗 —— 關鍵字比對工具損壞。
LLM 代理程式正確路由到 FAQ 代理程式並使用了工具；工具
本身傳回了錯誤資料。

**修正**：在 `faq_lookup_tool` (customer_service.py:130) 的座位關鍵字清單中
加入 `"row"`、`"rows"`、`"legroom"`。這是一個傳統程式碼修正，
而非評估/提示詞變更。

**驗證**：修正後，重新執行：

```bash
pixie test -v      # 驗證
```
````

### 6. 修正並重新執行

進行針對性變更，視需要更新資料集，然後重新執行：

```bash
pixie test -v
```

在修正穩定後，再次執行分析以查看模式是否已改變：

```bash
pixie analyze <新測試識別碼>
```

---

## 反覆修正週期

1. 閱讀步驟 6 的分析報告 → 決定失敗調查的優先順序
2. 執行詳細測試輸出 → 識別特定失敗
3. 調查每個失敗 → 分類為 LLM 或非 LLM
4. 對於 LLM 失敗：調整提示詞、模型或評估準則
5. 對於非 LLM 失敗：建議或應用程式碼修正
6. 如果修正更改了應用程式行為，請更新資料集
7. 重新執行測試與分析
8. 重複執行直到通過或使用者滿意為止
