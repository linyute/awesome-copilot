# 步驟 1c：評估標準

根據你已記錄的專案分析 (`00-project-analysis.md`) 和入口點 (`01-entry-point.md`)，定義對此應用程式重要的品質維度。

此文件有兩個目的：

1. **資料集建立 (步驟 4)**：使用案例告訴你應該產生哪些類型的項目 — 資料集應包含每個使用案例的具代表性項目。
2. **評估器選擇 (步驟 3)**：評估標準告訴你應該選擇哪些評估器以及如何映射它們。

**從 `pixie_qa/00-project-analysis.md` 中的能力清單衍生使用案例**。**從硬性問題 / 失敗模式衍生評估標準** — 不要使用像「事實性」或「相關性」這樣的通用品質維度。

保持簡潔 — 這是一個規劃產物，而不是一份詳盡的規格書。

---

## 需要定義的內容

### 1. 使用案例

列出應用程式處理的不同場景。這些場景應衍生自 `pixie_qa/00-project-analysis.md` 中的**能力清單** — 每個能力應至少映射到一個使用案例。每個使用案例會成為資料集項目的一個類別。**每個使用案例的描述必須是簡潔的一行文字，同時傳達 (a) 輸入內容以及 (b) 預期的行為或結果。** 描述應足夠具體，以便不熟悉該應用程式的人也能理解該場景及其成功標準。

在可能的情況下，註明每個使用案例的**預期難度等級** — 例如：「例行性」(Routine) 適用於簡單場景，「具挑戰性」(Challenging) 適用於邊緣案例或失敗模式場景。這將引導資料集建立 (步驟 4) 包含一系列難度的項目，而不是集中在簡單案例上。

**好的使用案例描述：**

- 「在帳戶查詢遇到困難時重新路由至人工代理」
- 「使用 CRM 中的客戶方案詳情回答計費問題」
- 「拒絕回答支援領域之外的問題」
- 「總結研究發現，包含所有查詢的子主題」

**不好的使用案例描述（太模糊）：**

- 「處理計費問題」
- 「邊緣案例」
- 「錯誤處理」

### 2. 評估標準

定義**高階且特定於應用程式的評估標準** — 即對「此」應用程式而言重要的品質維度。每個標準都將在步驟 3 中映射到一個評估器。

**好的標準應特定於應用程式的目的**，並衍生自 `pixie_qa/00-project-analysis.md` 中的**硬性問題 / 失敗模式**。例如：

- 語音客戶支援代理：「代理在轉接前是否驗證了通話者的身份？」、「回應是否簡短到足以用於電話對話？」
- 研究報告產生器：「報告是否涵蓋了所有子問題？」、「聲明是否有擷取來源的支撐？」
- RAG 聊天機器人：「回答是否基於擷取的背景？」、「當背景缺失時是否會說『我不知道』？」
- 網頁抓取工具：「擷取的資料是否符合要求的結構描述欄位？」、「是否能在不崩潰或遺失資料的情況下處理格式錯誤的 HTML？」

**不好的標準只是偽裝成要求的通用評估器名稱。** 不要只說「事實準確性」或「回應相關性」— 請說明事實準確性或相關性對「此」應用程式意味著什麼。如果你的標準適用於任何聊天機器人（例如：「Groundedness」、「PromptRelevance」），則它們太過通用 — 請回到 `00-project-analysis.md` 中的失敗模式並從中衍生標準。

在此階段，不需要選擇評估器類別或閾值。這些內容將在步驟 3 中完成。

### 3. 檢查標準的適用性與可觀察性

針對每個標準：

1. **確定適用範圍** — 此標準是適用於「所有」使用案例，還是僅適用於其中一個子集？如果標準僅與某些場景相關（例如：「身分驗證」僅適用於與帳戶相關的請求，而不適用於一般的 FAQ），請清楚標記。此區分對步驟 4（建立資料集）至關重要，因為：
   - **通用標準** → 成為資料集層級的預設評估器。
   - **特定案例標準** → 僅成為相關資料列中的項目層級評估器。

2. **驗證可觀察性** — 對於每個標準，識別需要在應用程式中擷取哪個資料點作為 `wrap()` 呼叫以進行評估。這將驅動步驟 2 中的 Wrap 覆蓋範圍。
   - 如果標準是關於應用程式的最終回應 → 由 `wrap(purpose="output", name="response")` 擷取。
   - 如果標準是關於路由決策 → 由 `wrap(purpose="state", name="routing_decision")` 擷取。
   - 如果標準是關於應用程式獲取並使用的資料 → 由 `wrap(purpose="input", name="...")` 擷取。

## 擁有多個能力的專案

如果專案分析 (`pixie_qa/00-project-analysis.md`) 列出了多個能力，你應至少評估 **2-3 個最重要 / 最常用**的能力。當專案的價值來自其廣度時，不要將資料集限制在單一能力上。

對於除第一個能力之外的每個額外能力：

- 在 `02-eval-criteria.md` 中增加使用案例。
- 在步驟 2 中規劃一個單獨的追蹤（使用不同的入口點 / 設定執行 `pixie trace`）。
- 在步驟 4 中規劃涵蓋該能力的資料集項目。

如果時間或內容限制使得涵蓋所有能力不切實際，請在 `02-eval-criteria.md` 的末尾**記錄你涵蓋了哪些能力以及跳過了哪些能力**（並說明理由）。

## 標準品質檢查 (強制性自我檢查)

在撰寫 `02-eval-criteria.md` 之前，請對每個標準執行此項檢查：

> **針對每個標準，請自問：「如果應用程式回傳了結構正確但語義錯誤或產生幻覺的答案，此標準是否能捕捉到該問題？」**

- 如果對「所有」標準的回答都是「否」，則你的標準集是**僅結構化的** — 它僅檢查管道（欄位是否存在、資料是否流過），而不檢查品質（內容是否正確、完整、有無幻覺）。**你必須增加至少一個語義標準**來評估應用程式輸出的「內容」，而不僅僅是其「形狀」。
- 結構化標準（欄位存在性、JSON 有效性、格式檢查）雖然有用但並不充足。即使應用程式回傳了捏造或錯誤的資料，它們仍能通過檢查。

**結構化 vs 語義標準範例：**

| 結構化標準（檢查形狀） | 語義標準（檢查品質） |
| ------------------------------------------- | -------------------------------------------------------------------------- |
| 「輸出中存在必要欄位」 | 「擷取的值與來源內容匹配 — 無幻覺資料」 |
| 「來源類型與預期類型匹配」 | 「應用程式正確解釋了混亂的輸入，且未遺漏關鍵事實」 |
| 「輸出為有效的 JSON」 | 「摘要準確擷取了文件的主要觀點」 |
| 「回應包含至少 N 個字元」 | 「回應針對使用者的具體問題，而非通用主題」 |

一個好的標準集應**同時包含**結構化和語義標準。結構化標準可捕捉嚴重的失敗（應用程式崩潰、回傳空輸出）。語義標準可捕捉品質失敗（應用程式雖執行但回傳了錯誤/幻覺/不完整的內容）。
