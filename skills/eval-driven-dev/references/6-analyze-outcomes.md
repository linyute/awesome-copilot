# 步驟 6：分析結果

**為何需要此步驟**：`pixie test` 產生了原始分數。現在你需要分析這些結果以了解其意義 — 完成待處理的評估、識別模式、驗證假設，並制定具備可操作性的改進計畫。分析結構分為三個相互建構的階段：項目層級 → 資料集層級 → 行動計畫。

---

## 結果目錄結構

執行 `pixie test` 後，結果目錄如下所示：

```text
{PIXIE_ROOT}/results/<test_id>/
  meta.json
  dataset-{idx}/
    metadata.json
    entry-{idx}/
      config.json              # 評估器設定、說明、預期
      eval-input.jsonl         # 餵給應用程式/評估器的資料
      eval-output.jsonl        # 應用程式產生的資料
      evaluations.jsonl        # 已評分與待處理的評估
      trace.jsonl              # LLM 呼叫追蹤
```

閱讀 `meta.json` 以尋找 `<test_id>`。分析所需的所有資料都在此目錄中。

## 硬性完成關卡

你是步驟 6 的評分者。**待處理的評估不是要交接給使用者處理的，網頁 UI 也不能取代評分。** 你可以使用網頁 UI 瀏覽追蹤和輸出，但最終完成標誌是在磁碟上撰寫檔案。

在滿足以下所有條件之前，步驟 6 尚未完成：

- 每個 `evaluations.jsonl` 中所有 `"status": "pending"` 的項目都已被替換為包含 `score` (分數) 和 `reasoning` (理由) 的已評分項目。
- 每個資料集目錄都包含 `analysis.md` 和 `analysis-summary.md`。
- 測試執行根目錄包含 `action-plan.md` 和 `action-plan-summary.md`。
- 此技能 `resources/` 目錄下的步驟 6 驗證指令碼對目標結果目錄回報成功。

**禁止的捷徑：**

- 留下任何 `"status": "pending"` 項目。
- 告知使用者在網頁 UI 中審核待處理的評估。
- 僅撰寫單個頂層替代檔案，如 `pixie_qa/06-analysis.md`。
- 在未評分評估並更新 `evaluations.jsonl` 的情況下，撰寫如「可能通過」或「可能失敗」之類的詞句。

如果你執行了上述任何操作，步驟 6 即視為未完成。

## 疊代規則

如果你正在進行多個修復/測試循環的疊代，每次成功的 `pixie test` 執行都會建立一個新的 `pixie_qa/results/<test_id>` 目錄以及新的步驟 6 義務。該目錄一旦存在，就會成為當前循環的分析目標。

在編輯應用程式程式碼、提示詞、資料集、評估器或重新執行 `pixie test` 之前，請為該特定的結果目錄完成步驟 6。不要跳過早期的循環而僅分析最後一次執行。

**額外禁止的捷徑：**

- 不要在同一項任務中建立較新的 `pixie_qa/results/<test_id>`，卻留下較舊的結果目錄而未完成步驟 6 成品。

## 撰寫原則

你產生的每個分析**詳細**成品都必須遵循以下原則：

- **資料驅動**：每個觀點或陳述都必須由評估執行的具體資料支撐。引用分數、提及項目索引、參考特定的評估輸入/輸出內容。不要空談。寧可不寫，也不要寫未經證實的內容。
- **證據優先**：在得出結論前先呈現原始資料和證據。讀者（另一個編碼代理）應該能夠根據你引用的證據獨立驗證你的結論。
- **可追蹤**：對於每個結論，請提供鏈條：資料來源 → 觀察 → 推理 → 結論。另一個代理應能反向追蹤此鏈條以驗證或挑戰任何主張。
- **不推銷**：不要倡導、推廣或使用帶有價值判斷的詞彙（如「優秀」、「強大」、「令人印象深刻」、「設計精良」）。陳述資料顯示了什麼以及它暗示了什麼行動。讓讀者自行形成品質判斷。
- **行動導向**：每項分析都應有助於達成對評估管線或應用程式進行具體改進的最終目標。不要寫沒有後續行動的觀察。

每個持久化的分析**摘要**成品都必須遵循以下原則：

- **簡潔**：人類讀者應能在 2 分鐘內理解任何單個成品的關鍵發現和行動。
- **結論優先**：以讀者需要知道的內容（結果、發現、行動）開頭，而非方法論或背景。
- **簡單明瞭**：避免專業術語。非技術利害關係人應能看懂摘要。
- **一致性**：摘要結論必須與詳細版本的證據相符。絕不在摘要中加入詳細版本中沒有證據支撐的主張。

### 雙變體模式

此步驟中每個持久化的分析成品都有兩個檔案：

| 成品 | 詳細檔案 (給代理) | 摘要檔案 (給人類) |
| ---------------- | --------------------------- | ----------------------------------- |
| 資料集分析 | `dataset-{idx}/analysis.md` | `dataset-{idx}/analysis-summary.md` |
| 行動計畫 | `action-plan.md` | `action-plan-summary.md` |

**務必先撰寫詳細版本**，再從中推導出摘要。摘要應為詳細版本內容的嚴格子集 — 絕不應包含詳細版本中未出現的主張或結論。

## 第一階段：項目層級評分

逐一處理每個資料集項目。對於每個 `dataset-{idx}/entry-{idx}/`：

### 1a. 閱讀項目資料

閱讀該項目的以下檔案：

- `config.json` — 設定了哪些評估器、說明、預期。
- `eval-input.jsonl` — 餵給應用程式/評估器的資料。
- `eval-output.jsonl` — 應用程式產生的資料。
- `evaluations.jsonl` — 目前的評估結果（已評分與待處理）。
- `trace.jsonl` — 應用程式進行了哪些 LLM 呼叫（若可用）。

### 1b. 完成待處理的評估

如果 `evaluations.jsonl` 包含狀態為 `"status": "pending"` 的項目，你必須對其進行評分：

1. 閱讀待處理評估的 `criteria` (標準) 欄位。
2. 將標準套用到項目的評估輸入、評估輸出和追蹤資料。
3. 分配一個 0.0 到 1.0 之間的**分數**：
   - `1.0` — 完全符合標準。
   - `0.5`–`0.9` — 部分符合標準（說明缺失部分）。
   - `0.0`–`0.4` — 不符合標準。
4. 撰寫一段 **reasoning** (理由) 字串（1–3 句話，引用輸出或追蹤中的具體證據）。
5. 將 `evaluations.jsonl` 中的待處理項目替換為已評分的結果。**不要附加第二行而留下待處理行。請直接覆寫待處理行本身。**

**之前** (待處理)：

```json
{
  "evaluator": "ResponseQuality",
  "status": "pending",
  "criteria": "回應應該..."
}
```

**之後** (已評分)：

```json
{
  "evaluator": "ResponseQuality",
  "score": 0.85,
  "reasoning": "回應解決了主要問題，但遺漏了..."
}
```

**評分指南**：

- 以證據為基礎 — 每個分數都必須參考特定的輸出或追蹤內容。
- 逐字遵循標準 — 不要超出書面標準進行擴充或重新解釋。
- 考慮追蹤紀錄 — 區分應用程式邏輯問題與 LLM 品質問題。
- 保持校準 — 僅將 1.0 保留給真正完全滿足標準的輸出。
- 不要懲罰 LLM 的非確定性 — 正確答案的不同措辭不算是失敗。
- 不要推給使用者 — 如果證據足以讓你寫下「可能通過」，則足以分配分數並更新 `evaluations.jsonl`。

### 1c. 不要持久化項目層級分析檔案

在此簡化流程中，**不要撰寫 `entry-{idx}/analysis.md` 或 `entry-{idx}/analysis-summary.md`**。第一階段僅用於閱讀證據並將每個待處理的評估轉換為 `evaluations.jsonl` 中的已評分行。

你可以在推理時做臨時筆記，但它們不屬於交付成果。僅需持久化：

- 每個項目目錄中更新後的 `evaluations.jsonl`。
- 第二階段的資料集層級分析檔案。
- 第三階段的執行層級行動計畫檔案。

## 第二階段：資料集層級分析

在分析完一個資料集中的所有項目後，產生資料集層級的分析。在資料集目錄中撰寫 `analysis.md` (`dataset-{idx}/analysis.md`)。

### 2a. 彙總資料

總結資料集中的所有項目：

- 通過/失敗計數以及總體通過率。
- 每個評估器的統計數據（通過率、最小/最大/平均分數）。
- 哪些項目在哪些評估器上失敗了（失敗叢集）。

### 2b. 形成並驗證假設

針對以下三個維度，提出**正好 3 個高置信度的假設**：

1. **測試案例品質** — 測試案例集是否充分且有效地驗證了應用程式的能力？是否涵蓋了重要的失敗模式？是否有盲點？

2. **評估標準/評估器品質** — 評估器是否具備適當的粒度和評分機制來捕捉真實問題？是否有「橡皮圖章」評估器（全都是 1.0）？是否有不穩定的評估器（在程式碼未變更的情況下變異度很高）？標準是太模糊還是太嚴格？

3. **應用程式品質** — 根據評估結果，應用程式的優點和缺點是什麼？它在哪裡產生高品質輸出？在哪裡失敗？

對於每個假設：

- 用一句話清楚地**陳述假設**。
- **引用證據** — 項目索引、評估器名稱、分數、理由引言、追蹤資料。
- **驗證或證偽** — 查看實際的評估輸入/輸出資料和程式碼以確認或駁回。
- **結論** — 此假設暗示了什麼行動？

即使資料有限，提出 3 個假設始終是可能的。如果評估資料無法對應用程式品質給出確切答案，這本身就是關於測試案例或評估器缺口的信號。

### 2c. 撰寫資料集分析 (兩個檔案)

為資料集分析產生**兩個檔案**。先撰寫詳細版本，再推導出摘要。

#### 詳細版本：`dataset-{idx}/analysis.md`

此檔案供 **代理取用** — 它提供完整的資料彙總、帶有證據鏈的假設形成，以及編碼代理可以直接執行的已驗證結論。

**撰寫原則：**

- **在解釋之前呈現所有資料。** 在任何假設之前，先以原始彙總（通過/失敗、每評估器統計數據、失敗矩陣）開頭。資料應能獨立說明問題。
- **對於每個假設，呈現：資料 → 推理 → 結論。** 讀者應能循著你的邏輯一步步推導，並獨立得出相同的結論。
- **直接交叉引用原始項目證據。** 引用證據時，請參考具體的項目索引以及底層檔案/資料點（例如：`entry-3/evaluations.jsonl`、`entry-3/eval-output.jsonl` 或 `entry-3/trace.jsonl`）。
- **區分相關性與因果關係。** 如果兩個項目在同一個評估器上失敗，這是一種模式。但根本原因可能不同 — 請透過檢查實際輸出資料來驗證，不要憑空假設。
- **若無標記則不要推測。** 如果結論不確定，請說明「假設 (未驗證)：...」並解釋哪些額外資料可以確認或證偽。

**內容：**

1. **概觀** — 資料集名稱、項目計數、總體通過率。
2. **原始彙總資料**
   - 每評估器統計數據表（通過率、分數範圍、平均值、標準差）。
   - 失敗矩陣：項目 × 評估器，顯示分數，標示失敗處。
   - 失敗叢集：按共同失敗評估器分組的項目。
3. **假設 1：測試案例** — 假設陳述、帶有項目/評估器參考的證據、採取的驗證步驟、具體行動結論。
4. **假設 2：評估器** — 結構同上。
5. **假設 3：應用程式** — 結構同上。
6. **未解決的問題** — 資料未給出確切答案的任何事項，並建議哪些額外資料會有所幫助。

#### 摘要版本：`dataset-{idx}/analysis-summary.md`

此檔案供 **人工審核** — 資料集結果、關鍵發現和建議行動的簡明總覽。

**範本：**

```markdown
# 資料集分析 — 摘要

**資料集**：<名稱> | **項目數**：<N> | **通過率**：<X/N (Y%)>

## 結果一覽

| 評估器 | 通過率 | 平均分數 | 註記 |
| --------- | --------- | --------- | ---------------------- |
| ...       | ...       | ...       | <若有值得注意的事項則寫下一行說明> |

## 關鍵發現

1. <發現>：<1-2 句話說明結論及其影響>
2. ...
3. ...

## 建議行動 (按優先順序)

1. <行動>：<要做什麼以及預期影響，1-2 句話>
2. ...
3. ...
```

摘要上限約 40 行。

## 第三階段：行動計畫 (兩個檔案)

在分析完所有資料集後，產生行動計畫。在測試執行根目錄下撰寫**兩個檔案**。先撰寫詳細版本，再推導出摘要。

### 詳細版本：`{PIXIE_ROOT}/results/<test_id>/action-plan.md`

此檔案供 **代理取用** — 它提供具備完整證據鏈的具體、可實施的改進項目，以便編碼代理可以直接處理任何項目而無需額外收集背景。

**撰寫原則：**

- **每個項目都必須是自成一體的。** 編碼代理在僅閱讀一個優先項目時，應有足夠的背景（證據參考、檔案路徑、預期變更）來實施它。
- **將每個項目追溯至證據。** 每個優先項目必須引用：哪個假設（來自哪個資料集分析）、哪些項目/評估器提供了證據，以及具體資料顯示了什麼。
- **具體說明「如何做」。** 不要只說「改進提示詞」 — 請說「在 `scrapegraphai/prompts/generate_answer.py` 第 45 行，增加指令：'...'」。越具體越具備可操作性。
- **不包含推測性項目。** 每個項目都必須有經過驗證的證據。如果某個項目是基於未驗證的假設，請先驗證或將其排除。

**結構：**

```markdown
# 行動計畫 (詳細)

## 摘要

- 分析了 X 個資料集，總計 Y 個項目，總體通過率 Z%
- [1-2 句高階評估]

## 優先順序 1：[最具影響力的改進]

- **內容**：[要做的具體變更]
- **原因**：[來自哪個資料集分析的哪個假設，並附帶項目/評估器參考]
- **證據**：[支撐此項目的具體分數、輸出節錄、追蹤資料]
- **預期影響**：[這將改進哪些項目/評估器，以及預測的分數變動]
- **方式**：[包含檔案路徑和行號的具體實作步驟]
- **驗證**：[如何驗證修復有效 — 要重新執行哪些項目，預期得分為何]

## 優先順序 2：...

...
```

### 摘要版本：`{PIXIE_ROOT}/results/<test_id>/action-plan-summary.md`

此檔案供 **人工審核** — 優先排序的改進清單，人類應能在 2 分鐘內理解並核准。

**範本：**

```markdown
# 行動計畫 — 摘要

**總體情況**：<X 個項目，通過率 Y%。1 句話評估。>

## 行動 (按優先順序)

1. **<行動標題>**：<變更內容及原因，2-3 句話。預期影響。>
2. **<行動標題>**：<變更內容及原因，2-3 句話。預期影響。>
3. ...
```

摘要上限約 30 行。

**優先順序標準：**

- 系統性問題（影響多個項目/資料集）優先於孤立問題。
- 具有清晰、已驗證證據的問題優先於推測性問題。
- 應用程式品質缺口優先於評估器精煉優先於測試案例增加。
- 快速修復優先於大型重構。

行動計畫應包含 3–5 個項目。每項都必須追溯至第二階段已驗證的假設。不要包含推測性或缺乏證據的項目。

## 流程總結

1. **第一階段** (按項目)：閱讀資料 → 為待處理評估評分 → 更新 `evaluations.jsonl`
2. **第二階段** (按資料集)：彙總 → 形成 3 個假設 → 驗證 → 撰寫 `dataset-{idx}/analysis.md` + `dataset-{idx}/analysis-summary.md`
3. **第三階段** (按測試執行)：綜合 → 排序優先順序 → 撰寫 `action-plan.md` + `action-plan-summary.md`

並行處理資料集內的項目（若有子代理可用）。依序處理各個階段 — 第二階段依賴第一階段的輸出，第三階段依賴第二階段的輸出。
