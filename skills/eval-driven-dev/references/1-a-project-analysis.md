# 步驟 1a：專案分析

在檢視程式碼結構、入口點或撰寫任何檢測 (Instrumentation) 之前，請先了解此軟體在現實世界中的作用。此分析是後續每個步驟的基礎 — 它決定了優先處理哪些入口點、定義哪些評估標準、使用哪些追蹤輸入，以及建立哪些資料集項目。

---

## 調查內容

閱讀專案的 README、說明文件和頂層原始檔案。你需要尋找以下五個問題的答案：

### 1. 此軟體的作用是什麼？

撰寫一段簡短的人讀摘要。它解決了什麼問題？成功的執行看起來是什麼樣子？

### 2. 誰使用它以及為什麼？

目標使用者是誰？主要的使用案例是什麼？它解決了哪些其他方案無法解決的問題？這有助於你理解「品質」對此應用程式意味著什麼 — 客戶對話機器人的品質要求與綜合多來源報告的研究代理不同。

### 3. 能力清單

列出應用程式提供的不同能力、模式或功能。請具體說明。例如：

- 對於抓取函式庫：單頁抓取、多頁抓取、基於搜尋的抓取、語音輸出、腳本產生。
- 對於語音代理：問候、FAQ 處理、帳戶查詢、轉接人工、通話摘要。
- 對於研究代理：主題研究、多來源綜合、引用產生、報告格式化。

每種能力可能需要自己的入口點、自己的追蹤和自己的資料集項目。此清單直接提供給步驟 1c（使用案例）和步驟 4（資料集多樣性）。

### 4. 現實輸入是什麼？

描述應用程式處理的現實世界輸入 — 而非簡單的玩具範例：

- 對於網頁抓取工具：「包含導航欄、廣告、動態內容、表格、巢狀結構的混亂 HTML 頁面 — 通常為 5KB 到 500KB 的 HTML」。
- 對於研究代理：「需要綜合多個來源的開放式研究問題，包含 3 到 10 個子問題」。
- 對於語音代理：「包含背景雜訊、插嘴和模糊請求的多輪對話」。

請具體說明**規模** (多大)、**複雜度** (多混亂/多樣) 和**種類**。這直接為追蹤輸入的選擇（步驟 2）提供依據 — 如果你不在這裡描述現實輸入，最終會使用繞過應用程式真實邏輯的玩具輸入。

**此章節是運作約束，而不僅僅是文件。** 步驟 2c (追蹤輸入) 和 4c (資料集項目) 將交叉引用這些特性，以驗證追蹤輸入和資料集項目是否符合現實世界的規模和複雜度。請務必具體且量化 — 撰寫「5KB 到 500KB 的 HTML 頁面」，而不是「各種 HTML 頁面」。

### 5. 硬性問題 / 失敗模式是什麼？

什麼使得此應用程式的工作變得很困難？它在實務中在哪裡失敗？這些將成為最有價值的評估場景：

- 對於抓取工具：「格式錯誤的 HTML、動態 JS 渲染內容、複雜的巢狀結構描述、超出內容視窗的極大頁面」。
- 對於研究代理：「衝突的來源、需要多步推理的問題、幻覺產生的引用」。
- 對於語音代理：「模糊的通話者意圖、帳戶查詢失敗、同步的工具呼叫」。

每種失敗模式應至少映射到一個評估標準（步驟 1c）和至少一個資料集項目（步驟 4）。

### 品質檢查

在繼續之前，請驗證：

- 「軟體的作用」章節使用非技術使用者也能理解的語言描述應用程式的目的 — 而不僅僅是「它執行圖」或「它呼叫 OpenAI」。
- 能力清單至少包含 3 個能力（如果專案具備的話）— 如果你只找到 1 個，可能只看了程式碼庫的一部分。
- 現實輸入特性描述了現實世界的規模和複雜度，而非最簡單的輸入。
- 失敗模式特定於此應用程式的領域，而非通用的（「錯誤輸入」不是失敗模式；「帶有未閉合標籤且導致解析器崩潰的格式錯誤 HTML」才是）。

### 專案中應忽略的內容

專案可能包含其自身開發/測試基礎設施的一部分目錄和檔案 — `tests/`、`fixtures/`、`examples/`、`mock_server/`、`docs/`、示範指令碼等。這些是為專案開發人員準備的，**不是**為你的評估管線準備的。

**關鍵提示**：「不要」將專案的測試固定裝置 (Fixtures)、模擬伺服器、範例資料或單元測試基礎設施用作評估追蹤或資料集項目的輸入。它們是為了開發速度和隔離而設計的 — 體量小、乾淨、確定性強的資料會避開每個現實世界的困難。使用它們會產生微不足道的評估結果，無法捕捉真實的品質問題。

當你在分析過程中遇到這些目錄時，請留意其存在，但將其視為專案的實作細節 — 而非你的 QA 管線的資料來源。你的 QA 管線必須針對現實世界條件測試應用程式，而非針對專案自身的測試捷徑。
