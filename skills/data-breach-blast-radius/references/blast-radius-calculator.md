# 影響範圍計算機

用於量化分析中的程式碼庫若發生資料外洩時，受影響的人員、紀錄和系統數量的公式、評分矩陣和估計啟發式方法。

---

## 核心影響範圍公式

```
影響範圍得分 (BRS) = 層級權重 × 暴露可能性 × 母體規模 × 完整性因素 × 背景乘數
```

**得分範圍：**
- 0–25：**低** — 暴露有限，紀錄較少
- 26–50：**中** — 有意義的暴露，目標母體明確
- 51–75：**高** — 重大暴露，廣泛的監管後果
- 76–100：**關鍵** — 災難性暴露，需要立即採取行動

---

## 因素 1：層級權重 (T)

根據 `data-classification.md` 中的資料分類層級：

| 層級 | 標籤 | 權重 |
|------|-------|--------|
| T1 | 災難性 | 5.0 |
| T2 | 關鍵 | 4.0 |
| T3 | 高 | 3.0 |
| T4 | 提升 | 2.0 |
| T5 | 標準 | 1.0 |

**規則：** 當同一個暴露向量中存在多個層級時，使用**最高**的層級權重。

**彙總提升：** 如果 3 個以上不同層級的欄位同時暴露，則在最高層級權重上加 +0.5（彙總攻擊風險）。

---

## 因素 2：暴露可能性 (E)

在現實的外洩情境中，此向量被利用的可能性有多大？

| 可能性得分 | 標籤 | 標準 |
|-----------------|-------|---------|
| 1.0 | **確定** | 資料目前可公開存取 (無需驗證) |
| 0.9 | **幾乎確定** | 驗證繞過非常簡單 (例如：連續 ID 的 IDOR、損毀的 JWT 驗證) |
| 0.8 | **非常可能** | 需要驗證，但此特定端點缺失驗證；或資料外洩到大多數工程師可存取的紀錄中 |
| 0.7 | **可能** | 需要驗證，但權限過於寬泛 (所有使用者都能看到所有資料)；缺失欄位級存取控制 |
| 0.6 | **中等** | 需要權限提升或與另一個漏洞串聯；具有廣泛開發者存取權限的內部系統 |
| 0.5 | **有可能** | 需要攻擊者付出巨大努力，但無縱深防禦；可從開發環境存取資料庫 |
| 0.3 | **不太可能** | 已實施多項安全性控制；但控制措施未經程式碼庫審查驗證 |
| 0.1 | **極低** | 強大的縱深防禦：加密、欄位遮罩、正確的授權、頻率限制、異常偵測皆具備 |

---

## 因素 3：母體規模 (P)

將受影響紀錄的估計數量標準化為 0–1 的量表。

### 估計紀錄數量

**步驟 1：在程式碼庫中尋找明確訊號**
```
# 強訊號 (如果發現則使用)：
- README 提及使用者數量 (「服務 500 萬名使用者」)
- 帶有紀錄數量的 Seeder/fixture 檔案
- 遷移註釋 (「為 5 萬名使用者增加索引」)
- 提及規模的分析儀表板或監控設定
- 基礎設施設定 (資料庫執行個體大小隱含規模)：
  - db.t3.micro → < 1 萬名活躍使用者
  - db.r5.large → 1 萬 – 50 萬名使用者
  - db.r5.4xlarge / Aurora Serverless → > 50 萬名使用者

# 中訊號：
- 應用程式類別 (SaaS 產品 → 較高，內部工具 → 較低)
- 多租戶 vs. 單租戶架構
- 資料庫結構描述中存在分片 (sharding) 或分割 (partitioning)

# 弱訊號：
- 僅靠技術堆疊 (與使用者數量無可靠關聯)
```

**步驟 2：未發現訊號時套用預設估計值**

| 應用程式類型 | 保守估計 | 典型估計 |
|-----------------|----------------------|-----------------|
| 內部企業工具 | 100 – 1,000 | 500 |
| B2B SaaS (小型/初創) | 1,000 – 10,000 | 5,000 |
| B2B SaaS (成熟型) | 10,000 – 100,000 | 50,000 |
| B2C 應用程式 (消費型初創) | 10,000 – 100,000 | 50,000 |
| B2C 應用程式 (成長階段) | 100,000 – 1,000,000 | 500,000 |
| B2C 應用程式 (規模化) | 1,000,000 – 100,000,000 | 10,000,000 |
| 醫療保健系統 | 1,000 – 100,000 | 20,000 |
| 金融服務 | 5,000 – 500,000 | 50,000 |
| 政府 / 公共部門 | 10,000 – 10,000,000 | 1,000,000 |

**請務必說明所使用的假設。**

### 母體規模得分 (P)

| 風險紀錄數 | 得分 |
|----------------|-------|
| < 100 | 0.1 |
| 100 – 1,000 | 0.2 |
| 1,000 – 10,000 | 0.3 |
| 10,000 – 50,000 | 0.4 |
| 50,000 – 100,000 | 0.5 |
| 100,000 – 500,000 | 0.6 |
| 500,000 – 1,000,000 | 0.7 |
| 100 萬 – 1000 萬 | 0.8 |
| 1000 萬 – 1 億 | 0.9 |
| > 1 億 | 1.0 |

## 因素 4：完整性因素 (C)

暴露的資料對攻擊者而言有多完整/有用？

| 因素 | 得分 | 說明 |
|--------|-------|-------------|
| **完整設定檔** | 1.0 | 完整的身份紀錄 (姓名 + email + 電話 + 地址 + 敏感欄位) |
| **部分 + 可關聯** | 0.9 | 資料不完整，但可與其他資料表關聯以補全；同一次外洩讓攻擊者獲得關聯金鑰 |
| **Email + PII** | 0.8 | 電子郵件地址加上 1 個以上敏感欄位 — 足以進行有針對性的網路釣魚 + 攻擊利用 |
| **僅限敏感欄位** | 0.7 | 僅有敏感欄位 (SSN、健康、財務) 而無聯絡資訊 — 依然非常嚴重 |
| **僅限聯絡資訊** | 0.5 | 僅有 email / 電話 — 可進行垃圾郵件、網路釣魚，但無即時危害 |
| **碎片化** | 0.3 | 欄位缺少背景資訊，若無此次外洩以外的其他資料則無法重新識別身份 |
| **匿名化** | 0.1 | 已正確匿名化 — 重新識別需要大量的外部資料連結 |

---

## 因素 5：背景乘數 (M)

針對特定背景將這些乘數套用到最終得分：

| 背景 | 乘數 | 依據 |
|---------|-----------|-----------|
| 包含兒童資料 (COPPA / GDPR 第 8 條) | × 2.0 | 全球最高法律風險 |
| 健康紀錄 (HIPAA / GDPR 特殊類別) | × 1.8 | 特殊類別資料，涉及民事 + 刑事責任 |
| 生物辨識資料 (GDPR 第 9 條, 伊利諾州 BIPA) | × 1.8 | 不可變資料 — 外洩後無法「更改」 |
| 金融帳戶憑證 | × 1.7 | 可能導致直接的財務竊盜 |
| 政府 ID (SSN, 護照) | × 1.6 | 身份竊用可能持續多年 |
| 性取向 / 宗教 / 政治觀點 | × 1.6 | GDPR 特殊類別，歧視風險 |
| 由醫療保健提供者持有的資料 | × 1.5 | HIPAA 商業夥伴 (Business Associate) 風險 |
| 雲端區域與使用者司法管轄區不符的資料 | × 1.3 | 跨境傳輸違規 (GDPR 第 V 章) |
| 備份/封存儲存 (常被遺忘) | × 1.2 | 外洩圍堵時常遺漏備份 |

## 影響範圍得分計算範例

### 範例 1：電子商務結帳系統

**暴露向量：** API 端點 `/api/users/{id}/payment-methods` — 無擁有權檢查 (IDOR)
- 層級：T2 (卡片末 4 碼 + 帳單地址) = 4.0
- 暴露可能性：0.9 (連續 ID 的 IDOR，幾乎確定會被利用)
- 母體規模：10 萬名使用者 = 0.6
- 完整性：部分設定檔 + 可與使用者資料表關聯 = 0.9
- 背景乘數：付款資料 = 1.7

```
BRS = 4.0 × 0.9 × 0.6 × 0.9 × 1.7 = 3.30 (原始) → 標準化為 66/100 → 高 (HIGH)
```

### 範例 2：內部 HR 工具

**暴露向量：** 所有公司使用者皆可透過 `/api/employees` 檢視員工資料表
- 層級：T2 (薪資 + 住家地址 + SSN) = 5.0 (SSN 為 T1)
- 暴露可能性：0.7 (需要驗證，但無 RBAC；任何員工皆可檢視全部)
- 母體規模：2,000 名員工 = 0.3
- 完整性：完整設定檔 = 1.0
- 背景乘數：政府 ID (SSN) = 1.6

```
BRS = 5.0 × 0.7 × 0.3 × 1.0 × 1.6 = 1.68 (原始) → 標準化為 34/100 → 中 (MEDIUM)
```

然而 — **財務影響**在此處優先於得分，因為 SSN 暴露屬於第 1 級。不論得分為何，皆標記為「高 (HIGH)」。

---

## 得分標準化

原始公式輸出通常範圍在 0–8。將其標準化為 0–100：

```
標準化 BRS = min(100, (原始 BRS / 8.0) × 100)
```

## 影響範圍摘要表 (按暴露向量)

回報時請使用此格式：

```markdown
| # | 暴露向量 | 層級 | 可能性 | 風險母體 | BRS | 嚴重性 | 司法管轄區 |
|---|----------------|------|-----------|-------------|-----|----------|--------------|
| 1 | /api/users 端點 - 回應中包含 SSN | T1 | 0.9 | 5 萬 | 87 | 關鍵 | GDPR, CCPA |
| 2 | 紀錄包含明文電子郵件 | T3 | 0.6 | 5 萬 | 45 | 中 | GDPR |
| 3 | Redis 快取儲存完整使用者物件 | T2 | 0.5 | 5 萬 | 38 | 中 | GDPR, CCPA |
| 4 | S3 儲存貯體 - 使用者頭像公開讀取 | T4 | 1.0 | 5 萬 | 28 | 低 | - |
```

## 組織總體影響範圍

在為所有暴露向量評分後，計算：

**最大同時暴露量 (MSE)：** 如果單一攻擊者獲得廣泛的資料庫存取權限，可能受影響的唯一人數 (最壞情況)。這是監管報告中使用的數字。

**預期外洩暴露量 (EBE)：** 基於最可能的攻擊向量的典型暴露 (可能性最高的發現，而非影響最大的發現)。

**監管觸發計數：** 觸發的相異監管制度數量 (每項制度都有其自身的通知義務和罰款公式)。

```markdown
## 組織影響範圍摘要

| 指標 | 值 |
|--------|-------|
| 最大風險紀錄數 | [數量] |
| 具有第 1 級資料的使用者 | [數量] |
| 具有第 2 級資料的使用者 | [數量] |
| 具有第 3 級以上資料的使用者 | [數量] |
| 觸發的法規 | GDPR, CCPA, [其他] |
| 最壞情況 BRS | [分數] |
| 最可能的攻擊向量 | [說明] |
| 估計偵測時間 | [產業平均：若無 SIEM 為 194 天] |
| 估計圍堵時間 | [產業平均：73 天] |
```
