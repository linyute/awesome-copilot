# 推送保護參考

GitHub 推送保護的詳細參考 — 防止祕密進入儲存庫、略過工作流程以及委派略過設定。

## 推送保護的運作方式

推送保護在推送過程中掃描祕密，並封鎖包含偵測到之祕密的推送。它作為一種預防性控制措施運作，這與在提交後偵測祕密的標準祕密掃描不同。

### 掃描內容

| 介面 | 已掃描 |
|---|---|
| 命令列推送 | ✅ |
| GitHub UI 提交 | ✅ |
| 檔案上傳至儲存庫 | ✅ |
| REST API 內容建立請求 | ✅ |

### 推送保護類型

**儲存庫推送保護：**
- 需要啟用 GitHub 祕密保護
- 預設為停用；由儲存庫管理員、組織擁有者或安全性管理員啟用
- 在「安全性」索引標籤中產生略過行為的警示
- 可以在儲存庫、組織或企業層級啟用

**使用者推送保護：**
- 對於所有 GitHub.com 帳戶預設為啟用
- 封鎖向包含支援祕密的公開儲存庫進行的推送
- 略過時不會產生警示 (除非儲存庫也啟用了推送保護)
- 透過個人帳戶設定進行管理

## 解決遭封鎖的推送 — 命令列

當推送保護封鎖推送時，錯誤訊息包含：
- 偵測到的祕密類型
- 包含該祕密的提交 SHA
- 檔案路徑和行號
- 用於略過的 URL (如果獲准)

### 從最新提交中移除祕密

```bash
# 編輯檔案以移除祕密
# 修正 (amend) 提交
git commit --amend --all

# 再次推送
git push
```

### 從較早的提交中移除祕密

```bash
# 1. 審閱推送錯誤，找出所有包含祕密的提交
# 2. 尋找包含祕密的最早提交
git log

# 3. 在該提交之前進行互動式變更基準 (rebase)
git rebase -i <EARLIEST-COMMIT>~1

# 4. 將有問題的提交之 'pick' 更改為 'edit'
# 5. 從檔案中移除祕密
# 6. 暫存並修正
git add .
git commit --amend

# 7. 繼續變更基準
git rebase --continue

# 8. 推送
git push
```

### 略過推送保護

1. 造訪錯誤訊息中的 URL (必須是進行推送的同一位使用者)
2. 選擇原因：
   - **用於測試** → 建立一個已關閉的警示 (解決為「用於測試」)
   - **這是誤判 (false positive)** → 建立一個已關閉的警示 (解決為「誤判」)
   - **我稍後再修復** → 建立一個未關閉的警示
3. 點擊 **允許我推送此祕密**
4. 在 **3 小時** 內重新推送 (超過時間後，請重複略過程序)

> 當儲存庫啟用了祕密掃描時，必須提供略過原因。對於僅啟用了使用者推送保護 (無儲存庫推送保護) 的公開儲存庫，不需要理由且不會產生警示。

## 解決遭封鎖的推送 — GitHub UI

在 GitHub UI 中建立或編輯檔案時：
1. 會出現橫幅，警告偵測到祕密
2. 內嵌顯示移除祕密或略過的選項
3. 套用與命令列相同的略過原因

## 解決遭封鎖的推送 — REST API

推送保護也適用於 REST API 內容建立端點。遭封鎖時：
- API 會傳回包含偵測到之祕密詳細資訊的錯誤回應
- 在請求中包含略過原因以繼續進行

## 委派略過

委派略過讓組織能夠精細地控制誰可以略過推送保護。

### 運作方式

1. 組織擁有者/儲存庫管理員建立使用者、角色或小組的 **略過清單**
2. 略過清單上的使用者可以直接略過推送保護 (需提供原因)
3. 所有其他貢獻者必須 **提交略過請求** 以供審閱
4. 略過請求會顯示在「安全性」索引標籤 →「推送保護略過」頁面
5. 請求若未經審閱，將在 **7 天** 後過期

### 誰可以始終略過 (無需請求)

- 組織擁有者
- 安全性管理員
- 已新增至略過清單的小組/角色中的使用者
- 擁有「審閱並管理祕密掃描略過請求」權限之自訂角色的使用者

### 啟用委派略過

**儲存庫層級：**
1. 設定 → 進階安全性 → 推送保護
2. 啟用「限制誰可以略過推送保護」
3. 將使用者、小組或角色新增至略過清單

**組織層級：**
1. 組織設定 → 進階安全性 → 全域設定
2. 在安全性組態中設定委派略過

### 管理略過請求

指定的審閱者：
1. 導覽至儲存庫「安全性」索引標籤 →「推送保護略過」
2. 審閱待處理的請求 (包含祕密、提交和貢獻者的評論)
3. **核准** — 貢獻者可以推送該祕密，以及未來任何包含相同祕密的提交
4. **拒絕** — 貢獻者在推送前必須移除祕密

### 略過請求流程 (貢獻者視角)

1. 推送遭封鎖；造訪錯誤訊息中的 URL
2. 新增評論，解釋為何推送該祕密是安全的
3. 點擊 **提交請求**
4. 等待核准/拒絕的電子郵件通知
5. 如果核准：推送提交
6. 如果拒絕：移除祕密並重新推送

## 推送保護模式

推送保護支援祕密掃描模式的子集。並非所有偵測到的祕密類型都會觸發推送保護封鎖。

關鍵考量：
- 推送保護可能不支援較舊/舊有的權杖格式
- 某些模式的誤判率較高，因此被排除在推送保護之外
- 自訂模式可以在發佈後啟用推送保護

有關推送保護支援的完整模式清單，請參閱 [支援的祕密掃描模式](https://docs.github.com/en/code-security/secret-scanning/introduction/supported-secret-scanning-patterns)。

## 為自訂模式設定推送保護

發佈自訂模式後：
1. 導覽至「設定」→「進階安全性」中的自訂模式
2. 點擊推送保護旁邊的 **啟用**
3. 該模式現在將封鎖包含符合祕密的推送

> 自訂模式的推送保護僅適用於啟用了推送保護的儲存庫。為常見模式啟用推送保護可能會對貢獻者造成干擾。
