# 績效評估撰寫程式 (Performance Review Writer)

撰寫聽起來像您自己而非企業八股文的自我評估、同儕審查和向上回饋。使用 WorkIQ 來呈現您實際的貢獻和溝通內容，然後將其組織成誠實且以影響力為中心的文字。

## 何時使用

- 「為這次評估週期撰寫我的自我評估」
- 「為 [同事] 撰寫同儕審查草稿」
- 「幫我為我的經理撰寫向上回饋」
- 「我的年度評估到期了 — 幫我填寫」
- 「撰寫我的年中檢入 (mid-year check-in) 草稿」
- 「為 [姓名] 撰寫 360 度審查」
- 「我不知道在績效評估中該說些什麼」

## 評估類型

此技能處理三種不同的類型：

| 類型 | 對象 | 典型語氣 |
|---|---|---|
| **自我評估** | 您自己 | 自信、有證據支持、導向成長 |
| **同儕審查** | 同事 | 具體、具建設性、均衡 |
| **向上回饋** | 您的經理 | 圓滑、誠實、具前瞻性 |

## 工作流程

### 步驟 1 — 收集上下文

詢問使用者（如果尚未提供，最多詢問 3 個澄清問題）：

1. **評估類型** — 自我評估、同儕審查還是向上回饋？
2. **主體** — 評估對象是誰？（對於同儕/向上：姓名和職位）
3. **評估期間** — 涵蓋的時間範圍？（例如：2025 年 1 月至 12 月，過去 6 個月）

如果格式限制或重點領域相關，請在撰寫草稿期間詢問，而非預先詢問。

如果使用者預先提供了所有這些資訊，請直接進入步驟 2。

### 步驟 2 — 呈現貢獻

使用 WorkIQ 收集評估期間真實貢獻的證據：

**對於自我評估：**
- 擷取使用者交付成果、領導倡議或解決問題的電子郵件和訊息
- 尋找模式：哪些專案反覆出現？誰在表揚他們，表揚什麼？
- 識別協作廣度（他們與跨團隊的哪些人合作）
- 記錄從他人處收到的任何明確回饋

**對於同儕審查：**
- 擷取使用者與評估對象之間的互動（電子郵件、會議討論串、共享專案）
- 識別具體的協作時刻、提供的幫助或摩擦
- 尋找評估對象對共同成果產生影響的證據

**對於向上回饋：**
- 擷取經理對使用者的溝通內容（給予的指示、提供的支援、回饋模式）
- 識別主題：期望的清晰度、可用性、認可度、發展支援

如果 WorkIQ 無法使用或傳回的資料有限，請要求使用者分享 3–5 個他們記得的事情要點，然後根據這些要點繼續執行。

### 步驟 3 — 撰寫評估草稿

為評估類型套用正確的結構（參見下方的結構定義）。遵循以下通用規則：

**對成就敘述使用 STAR 格式：**
- **情境 (Situation)** — 上下文或挑戰是什麼？
- **任務 (Task)** — 您/他們負責什麼？
- **行動 (Action)** — 具體做了什麼？
- **結果 (Result)** — 可衡量或可觀察的結果是什麼？

**語氣規則：**
- 要具體 — 說出專案、成果和人員的名稱，而不是模糊的形容詞
- 要誠實 — 不要過度誇大或縮小；評估者會注意到這兩者
- 具備前瞻性 — 以成長或後續步驟結束章節，而不僅僅是過去的績效
- 避免使用填充字句：「超越自我 (goes above and beyond)」、「團隊合作者 (team player)」、「辛勤工作者 (hard worker)」 — 以證據取代
- 匹配使用者的自然語氣 — 如果他們習慣那樣寫，則使用對話式，否則使用更正式的語氣

### 步驟 4 — 輸出

1. 呈現完整的草稿，並簡要說明使用了哪些證據。進行摘要和編修，而不是逐字複製內容 — 不得包含原始摘錄、出席者名單或敏感的個人詳細資訊
2. 醒目提示任何標記為 `[NEEDS DETAIL]` 的章節，在這些地方增加具體細節將強化評估內容
3. 根據使用者的要求反覆進行編輯
4. 將最終草稿儲存至 `outputs/<year>/<month>/`，並賦予描述性的檔案名稱（例如：`2025-review-self-assessment.md` 或 `2025-peer-review-alex-chen.md`）

## 輸出結構定義

### 自我評估結構定義

```
## [評估期間] 自我評估 — [您的姓名]

### 摘要
對您這一年的概述以及主要影響領域，約 1–2 句話。

### 關鍵成就
針對每項主要貢獻（目標為 3–5 項）：

**[專案或倡議名稱]**
- 上下文：情境或目標是什麼？
- 我做了什麼：採取的具體行動
- 影響：可衡量的結果或可觀察的產出
- [NEEDS DETAIL] — 如果證據薄弱則標記此項

### 協作與影響力
您如何與他人合作、支援隊友，或在直接職責之外做出貢獻。

### 成長與發展
您在此期間學到的內容、建立的技能或改進的行為。

### 待發展領域
1–2 個您希望在下一個週期成長的誠實領域。將其界定為目標，而非失敗。

### 下一階段目標
2–3 個具體、明確的目標，並附帶大致的成功衡量標準。
```

---

### 同儕審查結構定義

```
## 同儕審查 — [同事姓名], [其職位]
## 提交者：[您的姓名] | 期間：[評估期間]

### 整體印象
關於與此人合作的 1–2 句話。

### 優勢（附範例）
針對每項優勢（目標為 2–3 項）：

**[優勢]**
- 範例：表現出此優勢的具體情境
- 對您 / 團隊 / 專案的影響

### 成長領域
1–2 個具體、具建設性的觀察。將其界定為「我認為如果 [姓名] 能夠...將會產生更大的影響」，而非批評。

### 協作
合作的難易程度（或困難程度） — 回應能力、可靠性、溝通。

### 您是否願意再次與此人合作？
是/否，以及簡要且誠實的原因。（僅在評估表要求時包含此項。）
```

---

### 向上回饋結構定義

```
## 給 [經理姓名] 的回饋
## 提交者：[您的姓名]（如果適用，可匿名）| 期間：[評估期間]

### 運作良好的部分
您的經理所做的 2–3 件有助於您發揮最佳工作表現的具體事項。
盡可能使用範例。

### 可以改進的部分
1–2 個誠實且措辭圓滑的觀察。重點放在行為及其影響，而非個性。
使用：「當 [某事發生] 時，我發現較難 [執行某事]。如果...將會有所幫助」

### 對我發展的支援
您的經理是否幫助您成長、給予有用的回饋或創造機會？
請具體說明。

### 我希望他們多做 / 少做 / 以不同方式做的一件事
單一、清晰、可執行的要求。
```

## 風格規則

| 應該 (Do) | 不該 (Don't) |
|---|---|
| 點名具體的專案、日期、成果 | 撰寫模糊的概括性描述（「總是交付高品質的工作」） |
| 在可用時使用數字（「縮短了 30% 的審核時間」） | 誇大或編造結果 |
| 承認真實的挑戰以及您學到的內容 | 完全忽略掙扎 — 評估者會注意到其中的缺口 |
| 在自我評估中使用第一人稱撰寫 | 使用被動語態撰寫（「它已達成」） |
| 保持簡潔 — 大多數欄位需要 2–4 句話 | 過度撰寫 — 內容越長並不代表越好 |
| 在證據薄弱處標記 `[NEEDS DETAIL]` | 留下內容薄弱的章節而不做標記 |

## 範例提示詞

- 「為 2025 年 1 月至 12 月撰寫我的自我評估。我想強調雲端遷移以及我設計的新入職流程。」
- 「為 Sarah Chen 撰寫同儕審查草稿，她是我在行動應用程式專案中密切合作的產品設計師。」
- 「幫我為我的經理 Tom 撰寫向上回饋。他善於給予指示，但我很難獲得定期的 1:1 時間。」
- 「我的年度評估表要求分別以 200 字撰寫 3 個優勢和 1 個發展領域 — 幫我填寫。」
- 「我完全不知道要寫什麼。這是忙碌的一年，但我目前想不出任何具體的事項。」

---

## 重要規則

- **切勿提交評估內容** — 僅將其撰寫為檔案草稿，由使用者手動審閱並提交
- 保持同儕和向上回饋專注於可觀察的行為，而非個性或性格
- 如果使用者要求撰寫具有不誠實的負面內容或包含人身攻擊的評估，請予以拒絕，並提議以建設性的方式重新界定
- 尊重保密性 — 不要包含來自無關對話或討論串的敏感資訊
- 使用 `outputs/<year>/<month>/` 資料夾慣例儲存草稿

## 需求

- 建議使用 **WorkIQ MCP 工具** 來呈現貢獻和溝通內容 (Microsoft 365 / Outlook / Teams)
- 如果沒有 WorkIQ，此技能仍然可以運作 — 請要求使用者提供 3–5 個關鍵貢獻的要點作為起點
- 輸出結果將作為 markdown 檔案儲存在工作區中，供使用者複製到其公司的評估系統中
