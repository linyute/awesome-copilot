# Autoresearch：自主迭代實驗

適用於任何程式設計任務的自主實驗迴圈。您定義目標與衡量方式；代理則自主地進行迭代 — 修改程式碼、執行實驗、衡量結果，並保留或捨棄變更 — 直到被中斷。

此技能的靈感來自 [Karpathy 的 autoresearch](https://github.com/karpathy/autoresearch)，並將其從 ML 訓練擴展到**任何具有可衡量結果的程式設計任務**。

---

## 代理行為規則

1. **務必**在開始迴圈前，以互動方式引導使用者完成「設定」階段。
2. **務必**在進行任何變更前建立基準 (Baseline) 衡量值。
3. **務必**在執行每次實驗嘗試前提交 (Commit) 變更 (以便能乾淨地還原)。
4. **務必**維護一份追蹤每次實驗的結果日誌 (TSV)。
5. **務必**還原未能改進指標的變更 (Git 重設至上一個已知良好的狀態)。
6. **務必**在迴圈開始後自主執行 — 絕不暫停詢問「我應該繼續嗎？」。
7. **絕不可**修改使用者標記為範圍外 (Out-of-scope) 的檔案。
8. **絕不可**跳過衡量步驟 — 每次實驗都必須經過衡量。
9. **絕不可**保留會使指標退化的變更，除非使用者明確允許權衡。
10. **絕不可**安裝新相依性或進行環境變更，除非使用者已核准。

---

## 第 1 階段：設定 (互動式)

在開始任何實驗之前，請與使用者協作以建立以下參數。
直接向使用者詢問每個項目。請勿假設或跳過任何一項。

### 1.1 定義目標

詢問使用者：

> **您正試圖改進或優化什麼？**
>
> 範例：執行時間、記憶體使用量、二進位檔案大小、測試通過率、程式碼涵蓋範圍、
> API 回應延遲、吞吐量 (Throughput)、錯誤率、基準測試分數、建構時間、套件大小、
> 程式碼行數、圈複雜度 (Cyclomatic complexity) 等。

將使用者的回答記錄為**目標**。

### 1.2 定義指標

詢問使用者：

> **我們如何衡量成功？產出指標的確切指令是什麼？**
>
> 我需要：
> 1. 要執行的**指令** (例如：`dotnet test`, `npm run benchmark`, `time ./build.sh`, `pytest --tb=short`)
> 2. 如何從輸出中**擷取指標** (例如：正規標記式模式、特定行、JSON 欄位)
> 3. **方向**：是越低越好還是越高越好？
>
> 範例：「執行 `dotnet test --logger trx`，計算通過的測試數量。越高越好。」
> 範例：「執行 `hyperfine './my-program'`，擷取平均時間。越低越好。」

記錄：
- `METRIC_COMMAND`：要執行的指令
- `METRIC_EXTRACTION`：如何從輸出中擷取數值指標
- `METRIC_DIRECTION`：`lower_is_better` (越低越好) 或 `higher_is_better` (越高越好)

### 1.3 定義範圍

詢問使用者：

> **我被允許修改哪些檔案或目錄？**
>
> 以及哪些檔案是「禁止觸碰」的 (唯讀)？

記錄：
- `IN_SCOPE_FILES`：代理可編輯的檔案/目錄
- `OUT_OF_SCOPE_FILES`：絕不可修改的檔案/目錄

### 1.4 定義限制條件

詢問使用者：

> **是否有我應該遵守的限制條件？**
>
> 範例：
> - 每次實驗的時間預算 (例如：「每次執行應少於 2 分鐘」)
> - 不加入新的相依性
> - 必須保持所有現有測試通過
> - 不可變更公開 API
> - 必須維持向下相容性
> - VRAM/記憶體限制
> - 程式碼複雜度限制 (偏好較簡單的解決方案)

記錄為 `CONSTRAINTS`。

### 1.5 定義實驗預算 (選用)

詢問使用者：

> **我應該執行多少次實驗，還是只需一直執行到您叫我停止？**
>
> 您可以指定一個數字 (例如：「嘗試 20 次實驗」) 或「無限制」 (我會一直執行到您中斷為止)。

記錄為 `MAX_EXPERIMENTS` (數值或 `unlimited`)。

### 1.6 簡潔性標準

告知使用者預設的簡潔性政策：

> **簡潔性政策 (預設)：** 在其他條件相同的情況下，越簡單越好。增加醜陋複雜性卻僅帶來微小改進是不值得的。在維持或改進指標的同時移除程式碼是一個很好的結果。我會權衡複雜度成本與改進幅度。這個政策對您適用嗎，還是您想要調整它？

將任何調整記錄為 `SIMPLICITY_POLICY`。

### 1.7 確認設定

將所有參數彙整成清晰的表格回報給使用者：

| 參數 | 數值 |
| ------------------ | ---------------------------- |
| 目標 | ... |
| 指標指令 | ... |
| 指標擷取 | ... |
| 方向 | 越低越好 / 越高越好 ... |
| 範圍內檔案 | ... |
| 範圍外檔案 | ... |
| 限制條件 | ... |
| 最大實驗次數 | ... |
| 簡潔性政策 | ... |

要求使用者確認。在確認前請勿繼續。

## 第 2 階段：分支與基準

一旦使用者確認：

1. **建立分支**：建議一個基於今天日期的標籤 (例如：`autoresearch/mar17`)。
   建立分支：`git checkout -b autoresearch/<標籤>`。

2. **讀取範圍內檔案**：讀取所有範圍內的檔案，以建立目前狀態的完整上下文。

3. **初始化 results.tsv**：在存放庫根目錄建立 `results.tsv`，並包含標頭列：
   ```
   experiment	commit	metric	status	description
   ```
   將 `results.tsv` 與 `run.log` 加入 `.git/info/exclude` (若尚未存在則附加)，使其保持未追蹤狀態，而不修改任何已追蹤的檔案。

4. **執行基準測試**：在目前未修改的程式碼上執行指標指令。
   將結果記錄為實驗 `0`，狀態為 `baseline` (基準)，存入 `results.tsv`。

5. **向使用者回報基準**：
   > 基準已建立：**[指標名稱] = [數值]**
   > 正在開始自主實驗迴圈。

---

## 第 3 階段：實驗迴圈

持續執行此迴圈。不要停下來詢問使用者。執行直到：
- 達到 `MAX_EXPERIMENTS` 次數，或
- 使用者手動中斷

### 對於每次實驗：

```
迴圈步驟：
  1. 思考 (THINK)   - 分析先前的結果與目前的程式碼。
                       產生實驗假設。
                       考慮：哪些奏效、哪些沒效、有哪些尚未嘗試。

  2. 編輯 (EDIT)    - 修改範圍內的檔案以實作構想。
                       每次實驗保持變更集中且最小化。

  3. 提交 (COMMIT)  - git add + git commit，並附上簡短的描述性訊息。
                       格式："experiment: <簡短描述變更內容>"

  4. 執行 (RUN)     - 執行指標指令。
                       將輸出重新導向至 run.log，以免淹沒上下文視窗。
                       使用適合 Shell 的重新導向方式：
                       - Bash/Zsh: `<指令> > run.log 2>&1`
                       - PowerShell: `<指令> *> run.log`

  5. 衡量 (MEASURE) - 從 run.log 擷取指標。
                       若擷取失敗 (當機/錯誤)，請閱讀 run.log 的最後 50 行以瞭解錯誤。

  6. 決策 (DECIDE)  - 將指標與目前最佳值進行比較：
                       - 改進 (IMPROVED)：保留該提交。更新「最佳」基準。
                         記錄狀態 = "keep" (保留)。
                       - 相同或變差：還原。`git reset --hard HEAD~1`。
                         記錄狀態 = "discard" (捨棄)。
                       - 當機 (CRASH)：嘗試快速修復 (拼寫錯誤、匯入問題、簡單錯誤)。
                         使用修復內容修正實驗提交 (`git commit --amend`) 並重新執行。實驗保持其原始編號。
                         若嘗試 2 次後仍無法修復，則還原整個實驗
                         (`git reset --hard HEAD~1`) 並記錄狀態 = "crash"。

  7. 記錄 (LOG)     - 將一列資料附加至 results.tsv：
                       實驗編號  提交雜湊  指標數值  狀態  說明

  8. 繼續 (CONTINUE) - 回到步驟 1。
```

### 實驗策略

產生實驗構想時，請遵循此優先順序：

1. **先從容易達成的目標 (Low-hanging fruit) 開始**：簡單的參數調整、明顯的低效率。
2. **根據結果採取行動**：若某個方向表現出前景，則朝該方向進一步探索。
3. **在遇到停滯期 (Plateaus) 後多樣化**：若最後 3-5 次實驗皆失敗，請嘗試完全不同的方法。
4. **結合優勝者**：若實驗 A 與 B 各自獨立改進，請嘗試結合兩者。
5. **簡潔性檢查**：定期嘗試移除程式碼/複雜度，看指標是否能維持。
6. **基進變更**：在嘗試完漸進式構想後，嘗試較大的架構變更。

### 處理限制條件

- **時間預算**：若執行時間超過預期時間的 2 倍，則終止執行並視為當機。
- **現有測試**：若限制條件要求測試必須通過，請在實驗前後執行測試，若測試失敗則還原變更。
- **記憶體/資源**：監控並在資源使用量超過指定限制時還原變更。

---

## 第 4 階段：報告

當迴圈結束時 (預算用罄或使用者中斷)：

1. **以格式化表格列印完整的 results.tsv**。
2. **摘要**：
   - 總實驗執行次數
   - 保留/捨棄/當機的實驗次數
   - 初始指標 (基準) 與最終指標的對比
   - 改進百分比
   - 前 3 大最具影響力的變更
3. **顯示保留實驗的累計 Git 日誌**：
   `git log --oneline <起始提交>..HEAD`
4. **建議後續步驟**：根據結果，建議人類研究人員下一步可以嘗試什麼 (對自動化實驗而言風險太高/太過複雜的構想)。

## 快速參考

### Results TSV 格式

以 Tab 分隔，共 5 欄：

```
experiment	commit	metric	status	description
0	a1b2c3d	0.997900	baseline	unmodified code
1	b2c3d4e	0.993200	keep	increase learning rate to 0.04
2	c3d4e5f	1.005000	discard	switch to GeLU activation
3	d4e5f6g	0.000000	crash	double model width (OOM)
```

### Git 工作流程

- 所有實驗皆發生在 `autoresearch/<標籤>` 分支
- 每次實驗在執行前皆會提交
- 失敗的實驗會使用 `git reset --hard HEAD~1` 還原
- 成功的實驗會推進分支
- `results.tsv` 與 `run.log` 保持未追蹤狀態 (已加入 `.git/info/exclude`)

### 關鍵原則

1. **衡量一切**：沒有衡量，就沒有實驗。
2. **還原失敗**：分支僅在有改進時才推進。
3. **保持自主**：絕不停止詢問。若卡住請更深入思考。
4. **保持簡單**：複雜度是一種成本。將其與收益進行權衡。
5. **記錄一切**：TSV 就是研究日誌。
