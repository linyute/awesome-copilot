# 測試框架工程 (Harness Engineering)

測試框架工程將編碼代理重複的錯誤轉化為持久的儲存庫產出：

```text
Harness = 指令 + 限制 + 回饋 + 記憶 + 評估 + 治理
```

當使用者要求執行以下操作時使用此技能：

- 使儲存庫對 GitHub Copilot 或其他編碼代理更加可靠
- 添加持久的代理指令、儲存庫規則或護欄
- 防止 AI 編碼代理重複錯誤
- 記錄已知的失敗路徑以及防止再次發生的檢查
- 為專案規則添加輕量級的漂移檢查
- 審查、刷新或更新現有的代理測試框架

除非使用者要求改善儲存庫的代理運行環境，否則請勿將此技能用於普通的功能實施。

## 核心原則

- 將目標儲存庫視為事實來源。
- 在編輯前進行檢查。保留現有的技術棧、套件管理員、CI、文件、命名和架構。
- 添加最小且有用的測試框架。優先更新現有檔案，而不是添加重複的指導。
- 在可行情況下，透過測試、linter、類型檢查、CI、pre-commit hook 或漂移腳本使重要規則可強制執行。
- 僅在自動化會變得脆弱或具有誤導性時才使用人工審核點。
- 記錄不應再次發生的具有高風險的失敗，並指明捕獲再次發生的檢查或審核點。
- 不要盲目複製通用模板。根據目標儲存庫中的真實證據調整每個產出。

## 發現 (Discovery)

在提議或進行測試框架更改之前，請檢查儲存庫是否存在現有規則和證據。

閱讀這些存在的檔案和資料夾：

- `README.md`
- `AGENTS.md`
- `.github/copilot-instructions.md`
- `.github/instructions/`
- `.github/workflows/`
- `CONTRIBUTING.md`
- 套件清單，例如 `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `pom.xml`, 或 `build.gradle`
- `docs/` 下現有的文件
- `scripts/` 下現有的腳本
- 現有的測試和 CI 檢查

然後總結：

- 技術棧、套件管理員和入口點
- 現有的開發和驗證命令
- 目前的代理指令或儲存庫慣例
- 已知的失敗、事件、不穩定的路徑或重複的評論
- 專案規則未強制執行的缺口

## 採用工作流

遵循以下順序：

1. 選擇適合目標儲存庫的測試框架介面。
2. 編寫特定於目標的代理指令。
3. 為高價值規則添加可強制執行的檢查。
4. 為高風險或反覆發生的失敗記錄失敗記憶。
5. 為可能默默變得陳舊的指導添加漂移檢查。
6. 使用證據、假設和後續行動報告採用情況。

### 1. 選擇測試框架介面 (Harness Surface)

僅挑選適合目標儲存庫的介面：

| 需求 | 首選產出 |
| --- | --- |
| 始終開啟的代理行為 | `AGENTS.md` 或 `.github/copilot-instructions.md` |
| 檔案範圍的指導 | `.github/instructions/*.instructions.md` |
| 反覆發生的專案檢查 | `scripts/check_*.py`、shell 腳本或套件腳本 |
| CI 強制執行 | 現有的工作流檔案或一個小型的新工作流 |
| 已知的失敗 | `docs/failures/*.md` |
| 架構或流程決策 | `docs/decisions/*.md` |
| 採用證據 | `docs/harness/adoption-report.md` 或類似檔案 |

如果儲存庫已經具有等效的位置，請更新它，而不是建立一個平行的系統。

### 2. 編寫代理指令

代理指令應該具體且可操作。包括：

- 專案目的和主要的擁有權邊界
- 設定、測試、lint、構建和驗證命令
- 套件管理員和依賴規則
- 安全編輯規則、產生的檔案規則和禁止路徑
- 對更改代碼的測試期望
- 儲存庫具備的 PR 和 commit 慣例
- 如何記錄新的失敗或決策

避免寬泛的人格指導、通用的最佳實踐以及無法檢查或審核的規則。

### 3. 添加可強制執行的檢查

將高價值規則轉換為檢查。好的測試框架檢查是：

- 足夠窄以避免誤報
- 足夠快，可以在本地和 CI 中執行
- 命名清晰，以便代理在完成之前可以執行它們
- 記錄了它們所保護的規則

示例：

```text
規則：請勿編輯產生的 API 客戶端。
檢查：腳本掃描產生路徑的 diff，並失敗並顯示清晰的消息。

規則：每條失敗記憶筆記都指明一個回歸檢查。
檢查：腳本驗證 docs/failures/*.md 是否包含「Detection (檢測)」部分。

規則：配置文件和模板必須保持一致。
檢查：測試將配置文件 README 檔案與預期的模板檔案進行比較。
```

### 4. 記錄失敗記憶

當失敗是使用者可見的、高風險的或可能反覆發生時，請記錄失敗。
除非現有的筆記已經涵蓋了相同的根本原因，否則請在 `docs/failures/` 下使用一個新檔案。

推薦結構：

```markdown
# 失敗簡短標題

## 總結 (Summary)

失敗了什麼，誰看到了它，以及為什麼它很重要。

## 根本原因 (Root Cause)

技術或流程原因。避免指責。

## 預防 (Prevention)

防止或檢測再次發生的指令、測試、漂移檢查、CI 門檻 (gate)、fixture 或手動審核點。

## 證據 (Evidence)

指向問題 (issue)、PR、測試、日誌、命令輸出或檔案路徑的連結。
```

如果沒有可行的自動化檢查，請記錄手動審核點以及為什麼自動化會是不安全或具有誤導性的。

### 5. 添加漂移檢查 (Drift Checks)

對於可能默默變得陳舊的指導，請使用漂移檢查。常見示例：

- 文件提到了不再存在的命令
- 配置片段和產生的示例發散了
- 失敗筆記遺漏了回歸檢查
- 結構性更改缺少決策記錄
- CI 引用了陳舊的腳本或套件命令

優先使用儲存庫現有語言編寫的小型腳本。如果儲存庫沒有腳本慣例，僅使用標準庫的 Python 是一個可移植的預設選擇。

### 6. 報告採用情況

完成實質性的測試框架工作並提交一份採用報告，其中包括：

- 更改的檔案
- 添加或更新的規則
- 添加或重用的檢查
- 執行的命令和結果
- 假設和手動後續行動
- 建立或特意跳過的失敗記憶
- 將如何衡量有效性

## 審查工作流

當被要求審查測試框架更改時，請採取相反的視角。尋找：

- 未經目標儲存庫證據驗證而複製的通用規則
- 重複或衝突的指令檔案
- 可能在有效更改時失敗的廣泛檢查
- 未強制執行的高風險規則
- 缺少針對重複錯誤或運行時失敗的失敗記憶
- 源檔案更改後未刷新的產生文件
- 未執行相關檢查的 CI 門檻
- 儲存庫慣例被測試框架預設值覆蓋

首先報告發現結果，按嚴重程度排序，並附上檔案和行號引用（如果可用）。除非使用者明確要求修復，否則請勿在審查期間修改檔案。

## 輸出合約

在完成測試框架採用工作之前，請驗證：

- 在編輯前檢查了目標儲存庫
- 新的指導是特定於目標儲存庫的
- 更改的檢查可以在本地執行，或者具有記錄的手動替代方案
- 在需要時記錄了失敗記憶，或者最終回應解釋了為什麼跳過它
- 刷新的產生文件或索引
- 最終報告指明了每個執行的命令及其結果

## 可選參考

`https://github.com/baskduf/harness-starter-kit` 中的 Prompt-first 工作流是這些想法的參考實施。僅當使用者要求或儲存庫已包含它時，才將其用作參考材料。目標儲存庫始終是事實來源。
