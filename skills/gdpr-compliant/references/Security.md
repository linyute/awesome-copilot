當您需要下列詳細實作細節時，請載入此檔案：
加密、密碼雜湊、秘密管理、匿名化/假名化、雲端/DevOps 實務、CI/CD 控制、事件回應、架構模式。

## 加密 (Encryption)

### 靜態加密 (At-Rest Encryption)

| 資料敏感度 | 最低標準 |
|---|---|
| 標準個人資料 (姓名、地址、電子郵件) | AES-256 磁碟/磁碟區加密 (雲端服務預設值) |
| 敏感個人資料 (健康、生物辨識、財務、身分證字號) | AES-256 **欄位層級** 加密 + 透過 KMS 進行信封加密 |
| 加密金鑰 | HSM 支援的 KMS (Azure Key Vault Premium / AWS KMS CMK / GCP Cloud KMS) |

**信封加密模式 (Envelope encryption pattern)：**
1. 使用 **資料加密金鑰 (DEK)** (AES-256，針對每個記錄或每個資料表產生) 加密資料。
2. 使用儲存在 KMS 中的 **金鑰加密金鑰 (KEK)** 加密 DEK。
3. 將加密後的 DEK 與加密資料儲存在一起。
4. 刪除 KEK = 有效銷毀所有使用該金鑰加密的資料 (Crypto-shredding)。

### 傳輸中加密 (In-Transit Encryption)

- **必須** 強制執行 TLS 1.2 以上版本；優先使用 TLS 1.3。
- **必須** 設定 `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`。
- **必須不** 允許 TLS 1.0、TLS 1.1、空加密套件 (null cipher suites) 或出口級加密 (export-grade ciphers)。
- **必須不** 在生產環境中使用自簽憑證。

### 金鑰管理 (Key Management)

- DEK 至少每年旋轉一次；在懷疑遭洩露時立即旋轉。
- 每個環境 (開發/測試/生產) 使用獨立的金鑰命名空間。
- 記錄所有 KMS 金鑰存取事件 — 針對異常存取模式發出警示。
- **必須不** 在原始程式碼或組態檔案中硬編碼加密金鑰。

## 密碼雜湊 (Password Hashing)

| 演算法 | 參數 | 備註 |
|---|---|---|
| **Argon2id** 推薦 | 記憶體 ≥ 64 MB，迭代 ≥ 3，平行處理 ≥ 4 | OWASP 與 NIST 推薦 |
| **bcrypt** 可接受 | 成本係數 ≥ 12 | 廣泛支援；若無法使用 Argon2id 則使用此 |
| **scrypt** 可接受 | N=32768, r=8, p=1 | 良好的替代方案 |
| MD5 | — | 絕不使用 — 已輕易破解 |
| SHA-1 / SHA-256 | — | 絕不使用於密碼 — 並非為此目的設計 |

**必須**
- 為每個密碼使用唯一鹽值 (salt) (上述三種演算法皆已內建)。
- 僅儲存雜湊值 — 絕不儲存明文，絕不使用可還原的編碼。
- 若儲存的雜湊使用過時演算法，請在登入時重新雜湊 — 透明升級。

**建議**
- 新增 **椒鹽 (pepper)** (雜湊前新增的伺服器端秘密)，儲存在 KMS 中，而非資料庫。
- 在註冊時檢查密碼是否曾出現在已知洩露清單中 (`haveibeenpwned` API，k-anonymity 模式)。
- 強制規定密碼最少 12 個字元。

**必須不**
- 以任何形式記錄密碼 — 註冊時、登入失敗時皆然。
- 在 URL 或查詢字串中傳輸密碼。
- 以明文儲存密碼重設權杖 — 儲存前先進行雜湊。

## 秘密管理 (Secrets Management)

**必須**
- 將所有秘密儲存在專用的秘密管理員中：Azure Key Vault, AWS Secrets Manager, GCP Secret Manager 或 HashiCorp Vault。
- 使用預先提交 (pre-commit) 勾子防止秘密被提交：`gitleaks`, `detect-secrets`, GitHub 原生秘密掃描。
- 在下列情況下立即旋轉秘密：開發人員離職、懷疑遭洩露、年度排程。
- 維護一份 **秘密清單文件** — 每個秘密皆列出其用途與旋轉日期。

**建議**
- 使用透過 OIDC 聯盟取得的 **短效憑證** (GitHub Actions → Azure/AWS/GCP)，取代長效 API 金鑰。
- 稽核所有 KMS 秘密存取 — 針對非上班時間或來自未預期來源的存取發出警示。
- 每個環境使用獨立的秘密命名空間。

**`.gitignore` 必須包含：**
```
.env
.env.*
*.pem
*.key
*.pfx
*.p12
secrets/
appsettings.*.json   # 若其中可能包含連接字串
```

**必須不**
- 將秘密提交至原始程式碼儲存庫。
- 將秘密以明文傳遞為 CLI 引數 (它們會出現在處理程序清單和 Shell 歷程記錄中)。
- 在程式碼中將秘密儲存為未加密的環境變數預設值。

## 匿名化與假名化 (Anonymization & Pseudonymization)

### 定義

| 術語 | 可還原嗎？ | GDPR 範圍？ | 用途 |
|---|---|---|---|
| **匿名化** | 否 | GDPR 範圍外 | 刪除後的保留記錄、分析資料集 |
| **假名化** | 是 (需金鑰) | 仍屬個人資料 | 分析管線、稽核記錄、降低風險的處理 |

### 匿名化技術

| 技術 | 方法 | 何時使用 |
|---|---|---|
| 抑制 (Suppression) | 完全移除欄位 | 沒有分析價值的欄位 |
| 遮罩 (Masking) | 替換為固定預留位置 (`"ANONYMIZED_USER"`) | 刪除後的稽核記錄識別碼 |
| 泛化 (Generalization) | 用範圍取代確切值 (34 歲 → "30–40") | 分析 |
| 添加雜訊 (Noise addition) | 對數值添加統計雜訊 | 彙總分析 |
| 彙總 (Aggregation) | 報告群組統計，而非個別值 | 報表 |
| K-匿名 (K-anonymity) | 確保每條記錄無法與其他 k-1 條區分 | 分析資料集 |

### 假名化技術

| 技術 | 方法 |
|---|---|
| 具秘密金鑰的 HMAC-SHA256 | 一致、單向、具金鑰。用於分析中的使用者 ID。金鑰儲存於 KMS。 |
| Tokenization (權杖化) | 用不透明權杖取代數值；對應關係儲存於獨立安全保險庫中。 |
| 使用獨立金鑰加密 | 僅在具備明確 KMS 授權時解密。 |

**必須**
- 當刪除使用者時，**匿名化**必須保留的記錄 (財務、稽核記錄) — 以 `"ANONYMIZED"` 或雜湊預留位置替換識別欄位。
- 將假名化金鑰儲存於 KMS — 絕不儲存於與假名化資料相同的資料庫。
- 使用斷言測試匿名化常式：原值 **必須不可** 從輸出中還原。

**Crypto-shredding 模式 (事件溯源)：**
使用每個使用者的 DEK 加密事件中的個人資料，DEK 儲存於 KMS 中。
刪除時：從 KMS 刪除 DEK → 該使用者的所有歷史事件即有效匿名化，且無需修改事件記錄。

**必須不**
- 若可能透過與其他資料集連結來重新識別，即稱資料為「已匿名化」。
- 進行假名化時，將映射金鑰儲存於與假名化資料相同的資料表中。

## 雲端與 DevOps 實務

**必須**
- 啟用所有雲端儲存的靜態加密：Blob、受管資料庫、佇列、快取。
- 使用 **私有端點 (private endpoints)** — 資料庫 **必須不** 可公開存取。
- 套用網路安全性群組 / 防火牆規則：限制資料庫存取權僅限應用程式層。
- 啟用雲端原生稽核記錄：Azure Monitor / AWS CloudTrail / GCP Cloud Audit Logs。
- 僅將個人資料儲存在 **核准的地理區域** (EEA，或具備適足性認定 / SCCs)。
- 使用 `DataClassification` 標籤標記所有處理個人資料的雲端資源。

**建議**
- 啟用 Microsoft Defender for Cloud / AWS Security Hub / GCP SCC — 每週審查建議。
- 使用 **受控識別 (managed identities)** (Azure) 或 **IAM 角色** (AWS/GCP) 取代長效存取金鑰。
- 在物件儲存上啟用軟刪除 (soft delete) 與版本控制。
- 在雲端儲存上套用 DLP 政策以偵測寫入未受保護儲存體的 PII。
- 針對敏感資料表啟用資料庫層級的 SELECT 稽核記錄。

**必須不**
- 將個人資料儲存在沒有存取控制的公開儲存桶中。
- 在生產環境中部署具備公用 IP 的資料庫。
- 若資料可能跨環境洩露，請勿在生產與非生產環境中使用相同的雲端帳戶/訂閱。

## CI/CD 控制

**必須**
- 在每次提交時執行 **秘密掃描**：`gitleaks`, `detect-secrets`, GitHub 原生秘密掃描。
- 在每次建置時執行 **相依性弱點掃描**：`npm audit`, `dotnet list package --vulnerable`, `trivy`, `snyk`。
- **必須不** 在 CI 測試作業中使用真實個人資料。
- **必須不** 在 CI 管線中記錄環境變數 — 遮罩所有秘密。

**建議**
- 在每個 PR 上執行 **SAST**：SonarQube, Semgrep 或 CodeQL。
- 執行 **容器映像掃描**：`trivy`, Snyk Container 或 AWS ECR 掃描。
- 在管線中加入 **GDPR 合規閘道**：
  - 沒有記錄存留時間的全新遷移 → 失敗。
  - 包含已知 PII 欄位名稱的記錄陳述式 → 警告。

**管線秘密規則：**
```yaml
# 必須：使用前遮罩秘密
- name: Mask secret
  run: echo "::add-mask::${{ secrets.MY_SECRET }}"

# 必須不：將秘密 Echo 到控制台
- run: echo "Key=$API_KEY"   # 絕不

# 建議：使用 OIDC 聯盟 (無長效金鑰)
- uses: azure/login@v1
  with:
    client-id: ${{ vars.AZURE_CLIENT_ID }}
    tenant-id: ${{ vars.AZURE_TENANT_ID }}
    subscription-id: ${{ vars.AZURE_SUBSCRIPTION_ID }}
```

## 事件與洩露處理

### 監管時間軸

| 視窗 | 義務 |
|---|---|
| 自知悉起 **72 小時** | 通知監管當局 (CNIL, APD, ICO...) — 除非洩露不太可能對個人造成風險 |
| **無不當延遲** | 若洩露可能導致 **高風險** 的個人權利損害，需通知受影響的資料主體 |

在內部記錄 **所有** 個人資料洩露 — 即使是那些不需要 DPA 通知的情況。

### 洩露回應執行手冊 (範本)

1. **偵測** — 定義準則：什麼會觸發事件 (憑證洩露、資料庫轉儲公開、勒索軟體、意外的公開儲存桶)。
2. **嚴重性分類** — 基於資料敏感度與數量的 低 / 中 / 高 / 關鍵。
3. **圍堵** — 撤銷遭洩露的憑證；隔離受影響系統；保留證據 (請 **不要** 刪除記錄)。
4. **評估** — 哪些資料外洩？有多少主體？風險等級為何？
5. **DPA 通知** — 使用監管當局的線上入口網站；包含：洩露性質、資料主體類別與近似人數、記錄類別與近似數量、聯絡點、可能後果、採取的措施。
6. **資料主體通知** — 若為高風險：以清晰語言說明洩露性質、可能後果、採取的措施、DPO 聯絡方式。
7. **事後審查** — 根本原因分析；矯正措施；更新執行手冊。

### 自動化洩露偵測警示

針對以下情況設定警示：
- 資料匯出量異常 (每小時閾值)
- 上班時間外的敏感資料表存取
- 大量刪除事件
- 驗證失敗高峰
- 新憑證出現在公開洩露資料庫中 (HaveIBeenPwned 監控)

在內部儲存洩露記錄至少 **5 年**。

## 架構模式

### 資料儲存區分離
將運作資料 (交易資料庫) 與分析資料 (資料倉儲) 分開。
對兩者套用不同的存留時間與存取控制。
分析儲存區 **必須不可** 直接讀取生產運作資料表。

### 專用同意儲存區
將同意記錄為獨立儲存區中的不可變事件記錄，而非使用者資料表中的布林欄位。
這可實現：可稽核的同意歷程、版本追蹤、在不遺失資料的情況下輕鬆撤銷。

### 稽核記錄隔離
將稽核記錄儲存在獨立的、僅能附加 (append-only) 的儲存區中。
應用程式服務帳戶 **必須不可** 刪除稽核記錄條目。
使用僅對稽核資料表具有 INSERT 權限的獨立資料庫使用者。

### DSR 佇列模式
將資料主體請求實作為非同步工作流程：
`POST /api/v1/me/erasure-request` → 加入工作佇列 → 背景執行緒清除所有儲存區 → 完成後通知使用者。
這能可靠地處理多重儲存區清除的複雜性，並提供重試機制。

### 假名化閘道 (Pseudonymization Gateway)
對於分析管線，在運作與分析系統的邊界實作假名化服務。
映射金鑰 (HMAC 秘密或權杖化保險庫) 絕不會離開運作區。
分析區僅接收假名化的識別碼。

### Crypto-shredding (事件溯源)
使用儲存於 KMS 中的每個使用者 DEK 來加密事件中的個人資料。
刪除使用者時：從 KMS 刪除 DEK → 該使用者的所有歷史事件即有效匿名化，無需修改事件記錄。
