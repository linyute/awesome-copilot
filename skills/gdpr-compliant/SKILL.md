# GDPR 工程技能

供工程師、架構師、DevOps 與技術負責人使用的可執行 GDPR 參考指南。靈感來自 CNIL 開發人員指引與 GDPR 第 5、25、32、33、35 條。

> **黃金法則：** 減少收集。減少儲存。減少暴露。減少保留。

如需深入了解，請閱讀 `references/` 中的參考檔案：
- `references/data-rights.md` — 使用者權利端點、DSR 工作流程、RoPA
- `references/security.md` — 加密、雜湊、秘密管理、匿名化
- `references/operations.md` — 雲端、CI/CD、事件回應、架構模式

---

## 1. GDPR 核心原則 (第 5 條)

| 原則 | 工程義務 |
|---|---|
| 合法、公平、透明 | 為 RoPA 中的每一項處理活動記錄法律依據 |
| 目的限制 | 為目的 A 收集的資料 **絕對不可** 在沒有新法律依據的情況下重複用於目的 B |
| 資料最小化 | 僅收集目前有記錄之業務需求的欄位 |
| 準確性 | 提供更新端點；將更正內容傳播至下游儲存區 |
| 儲存限制 | 在結構描述設計時定義 TTL — 之後不可變更 |
| 完整性與機密性 | 靜態與傳輸中加密；限制存取並進行稽核 |
| 當責 | 維護合規證明；隨時準備好 RoPA 以供 DPA 檢查 |

## 2. 隱私設計與預設隱私

**必須**
- 在建立時將 `CreatedAt`、`RetentionExpiresAt` 新增至每個持有個人資料的資料表中。
- 將所有選用的資料收集預設為 **關閉**。使用者需明確加入 (opt in)；永遠不要讓他們在預設開啟的情況下選擇退出。
- 在建構高風險處理 (生物辨識、健康資料、大規模剖析、系統性監控) 前進行 **DPIA**。
- 隨著每項引入處理活動的新功能更新 **RoPA**。
- 在資料傳輸至每個子處理器前與其簽署 **DPA**。

**必須不**
- 在沒有記錄法律依據的情況下發布新的資料收集功能。
- 在沒有明確同意的情況下預設啟用分析、追蹤或遙測。
- 將個人資料儲存在 RoPA 未列出的系統中。

---

## 3. 資料最小化

**必須**
- 將每個 DTO/模型欄位對應至具體的業務需求。移除未記錄的欄位。
- 使用 **獨立的 DTO** 進行建立、讀取與更新 — 永遠不要重複使用同一個物件。
- 僅回傳呼叫者有權查看的內容 — 使用回應投影 (response projections)。
- 在邊緣遮罩敏感值：信用卡號回傳 `****1234`，絕不回傳完整數值。
- 從預設清單/搜尋投影中排除敏感欄位 (出生日期、身分證字號、健康資料)。

**必須不**
- 若請求/回應本文可能包含個人資料，則不要記錄其全文。
- 將個人資料包含在 URL 路徑區段或查詢參數中 (CDN 記錄、瀏覽器歷程記錄)。
- 在沒有明確法律依據的情況下收集 `dateOfBirth`、身分證字號或健康資料。

---

## 4. 目的限制

**必須**
- 在程式碼註解與 RoPA 中記錄每一項處理活動的目的。
- 在重複使用資料進行次要目的之前，取得新的法律依據或執行相容性分析。

**必須不**
- 在沒有明確同意的情況下，與廣告聯播網分享為服務交付而收集的個人資料。
- 在沒有獨立法律依據與使用者通知的情況下，使用支援票證內容來訓練 ML 模型。

---

## 5. 儲存限制與保留

**必須**
- 每個持有個人資料的資料表 **必須** 有定義的保留期限。
- 透過排程作業 (Hangfire, cron) 自動強制執行保留 — 永遠不要採用手動流程。
- 在保留期滿時匿名化或刪除資料 — 永遠不要讓過期資料靜默留在生產環境中。

**建議預設值**

| 資料型別 | 最長保留期 |
|---|---|
| 驗證 / 稽核記錄 | 12–24 個月 |
| 工作階段 / 重新整理權杖 | 30–90 天 |
| 電子郵件 / 通知記錄 | 6 個月 |
| 不活躍的使用者帳戶 | 最後一次登入後 12 個月 → 通知 → 刪除 |
| 付款記錄 | 稅法要求 (7–10 年)，需最小化 |
| 分析事件 | 13 個月 |

**應該**
- 新增 `RetentionExpiresAt` 欄位 — 在插入時計算。
- 使用軟刪除 (`DeletedAt`)，並在清除請求視窗 (30 天) 後進行排程硬刪除。

**必須不**
- 以「萬一以後有用」為由無限期保留個人資料。

## 6. API 設計規則

**必須**
- **必須不** 將個人資料包含在 URL 路徑或查詢參數中。
  - `GET /users/{userId}`
- 對所有回傳或接受個人資料的端點進行驗證。
- 從 JWT 中提取操作使用者的身分 — 絕不要從請求本文中提取。
- 在每個資源上驗證所有權：`if (resource.OwnerId != currentUserId) return 403`。
- 使用 UUID 或不透明識別碼 — 絕不要使用連續整數作為公開資源 ID。

**應該**
- 對敏感端點 (登入、資料匯出、密碼重設) 進行速率限制。
- 設定 `Referrer-Policy: no-referrer` 與明確的 `CORS` 允許清單。

**必須不**
- 在 API 回應中回傳堆疊追蹤、內部路徑或資料庫錯誤。
- 在已驗證的 API 上使用 `Access-Control-Allow-Origin: *`。

---

## 7. 記錄規則 (Logging Rules)

**必須**
- 匿名化應用程式記錄中的 IP — 遮罩最後一個八位元組 (IPv4) 或最後 80 位元 (IPv6)。
  - `192.168.1.xxx`
- **必須不** 記錄：密碼、權杖、工作階段 ID、憑證、卡號、身分證字號、健康資料。
- **必須不** 記錄可能包含個人資料的請求/回應全文。
- 強制執行記錄保留 — 定義期限後自動清除。

**應該**
- 記錄 **事件** 而非資料：`"使用者 {UserId} 更新電子郵件"` 而非 `"電子郵件從 a@b.com 變更為 c@d.com"`。
- 使用結構化記錄 (JSON)，並以 `userId` 作為內部識別碼，而非電子郵件地址。
- 分離稽核記錄（敏感存取、管理動作）與應用程式記錄 — 採用不同的保留期與 ACL。

---

## 8. 錯誤處理

**必須**
- 回傳一般錯誤訊息 — 永遠不要公開堆疊追蹤、內部路徑或資料庫錯誤。
  - `"資料表 'users' 中的 'email' 欄位違反唯一性條件"`
  - `"已有使用此電子郵件地址的帳戶。"`
- 所有錯誤回應皆使用 **問題詳細資訊 (Problem Details, RFC 7807)**。
- 在伺服器端記錄完整的錯誤與關聯 ID；僅將關聯 ID 回傳給用戶端。

**必須不**
- 在錯誤回應中包含檔案路徑、類別名稱或行號。
- 在錯誤訊息中包含個人資料 (例如："找不到使用者 john@example.com")。

## 9. 加密 (摘要 — 詳情請參閱 `references/security.md`)

| 範圍 | 最低標準 |
|---|---|
| 標準個人資料 | AES-256 磁碟/磁碟區加密 |
| 敏感資料 (健康、財務、生物辨識) | AES-256 **欄位層級** 加密 + 透過 KMS 進行信封加密 |
| 傳輸中 | TLS 1.2+ (優先使用 1.3)；強制 HSTS |
| 金鑰 | HSM 支援的 KMS；每年旋轉 DEK |

**必須不** 允許 TLS 1.0/1.1、空加密套件或硬編碼加密金鑰。

---

## 10. 密碼雜湊 (Password Hashing)

**必須**
- 使用 **Argon2id** (推薦) 或 **bcrypt** (成本 ≥ 12)。絕不使用 MD5, SHA-1, 或 SHA-256。
- 為每個密碼使用唯一鹽值。僅儲存雜湊值。

**必須不**
- 以任何形式記錄密碼。以 URL 傳輸密碼。以明文儲存重設權杖。

---

## 11. 秘密管理 (Secrets Management)

**必須**
- 將所有秘密儲存在 KMS：Azure Key Vault, AWS Secrets Manager, GCP Secret Manager 或 HashiCorp Vault。
- 使用預先提交勾子 (`gitleaks`, `detect-secrets`) 防止提交秘密。
- 在開發人員離職、年度排程或懷疑遭洩露時旋轉秘密。

**`.gitignore` 必須包含：** `.env`, `.env.*`, `*.pem`, `*.key`, `*.pfx`, `*.p12`, `secrets/`

**必須不**
- 將秘密提交至原始程式碼。將秘密儲存為程式碼中的純文字環境變數預設值。

---

## 12. 匿名化與假名化 (摘要 — 詳情請參閱 `references/security.md`)

- **匿名化** = 不可逆 → 超出 GDPR 範圍。用於刪除後的保留記錄、分析資料集。
- **假名化** = 可透過金鑰還原 → 仍屬個人資料，風險較低。
- 刪除使用者時，**匿名化**必須保留的記錄 (財務、稽核記錄)，而非刪除它們。
- 將假名化金鑰儲存於 KMS — 絕不儲存於與假名化資料相同的資料庫。

**必須不** 在若可透過連結攻擊重新識別時，將資料稱為「匿名化」。

---

## 13. 使用虛擬資料測試

**必須**
- **必須不** 在開發、測試或 CI 環境中使用生產環境的個人資料。
- **必須不** 在未先清理 PII 的情況下將生產資料庫備份還原至非生產環境。
- 使用虛擬資料產生器：`Bogus` (.NET), `Faker` (JS/Python/Ruby)。
- 所有測試電子郵件地址請使用 `@example.com`。

## 14. 反模式 (Anti-Patterns)

| 反模式 | 正確做法 |
|---|---|
| URL 中包含 PII | 使用不透明 UUID 作為公開識別碼 |
| 記錄完整請求本文 | 僅記錄結構化事件中繼資料 |
| 「永久保留」結構描述 | 在設計時定義 TTL |
| 開發/測試中使用生產資料 | 虛擬資料 + 清理管線 |
| 跨團隊共用憑證 | 個人帳戶 + RBAC |
| 硬編碼秘密 | KMS + 秘密管理員 |
| 在驗證 API 上使用 `Access-Control-Allow-Origin: *` | 明確的 CORS 允許清單 |
| 將同意與設定檔資料儲存在一起 | 專用的同意儲存區 |
| GET 查詢參數包含 PII | 使用 POST 本文或已驗證的工作階段 |
| 公開 URL 中使用連續整數 ID | 使用 UUID |
| 包含準識別碼的「匿名化」資料 | 套用 k-匿名，測試連結抵抗力 |
| 混合 EEA 以外的備份區域 | 備份作業設定明確的區域鎖定 |

## 15. PR 審查檢查清單

### 資料模型
- 每個新的 PII 欄位都有記錄的用途與保留期限。
- 敏感欄位 (健康、財務、身分證字號) 使用欄位層級加密。
- 沒有連續整數 PK 作為公開識別碼。

### API
- URL 路徑或查詢參數中沒有 PII。
- 所有回傳個人資料的端點皆已驗證。
- 擁有權檢查存在 — 使用者無法存取他人的資源。
- 對敏感端點套用速率限制。

### 記錄 (Logging)
- 未記錄密碼、權杖或憑證。
- IP 已匿名化 (遮罩最後一個八位元組)。
- 若可能包含 PII，則不記錄請求/回應全文。

### 基礎架構
- 沒有公開的儲存桶或公開 IP 的資料庫。
- 新的雲端資源標記了 `DataClassification`。
- 新的儲存資源已啟用靜態加密。
- 資料儲存的新地理區域符合 EEA 合規或具備 SCC 涵蓋。

### 秘密與 CI/CD
- 原始程式碼或已提交的組態檔案中沒有秘密。
- 新增的秘密已加入 KMS 與秘密清單文件。
- CI/CD 秘密在管線記錄中已遮罩。

### 保留與刪除
- 新資料儲存區或欄位有保留強制執行作業或政策。
- 刪除管線已更新以涵蓋新資料儲存區。

### 使用者權利與治理
- 資料匯出端點包含任何新的個人資料欄位。
- 若引入新的處理活動，已更新 RoPA。
- 新的子處理器已簽署 DPA 並具備 RoPA 條目。
- 若變更涉及高風險處理，已觸發 DPIA。
