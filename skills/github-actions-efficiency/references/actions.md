# GitHub Actions 效率

僅在任務涉及 GitHub Actions 或 CI 工作流程效率時載入此參考。

如果儲存庫是首次導入 GitHub Actions，請先定義最低限度的基準工作流程，然後使用本指南的其餘部分進行優化。

## 稽核順序

請依照此順序檢查：

1. 如果遺失 `.github/workflows/` 或為空，請先收集基準需求：觸發事件、必要的檢查、執行階段版本以及儲存庫特定的驗證政策。
2. `.github/workflows/*.yml`
3. 說明 CI 預期的文件
4. 如果使用者想要衡量的影響，請檢查現有的報告或執行記錄

對於新的設定，請先從一個能驗證核心檢查的小型工作流程開始，然後僅在需要時才增加矩陣廣度或額外作業 (jobs)。

從常見、低風險的浪費開始：

1. 遺失相依性快取
2. 遺失 `concurrency` 取消設定
3. 過於寬泛的工作流程觸發器
4. 跨檔案或跨作業重複的工作流程涵蓋範圍
5. 無論範圍如何，每次變更都會執行的昂貴作業

## Actions 特定指引

### 觸發範圍設定 (Trigger scoping)

- 當整個工作流程確實不應該針對某些檔案類別執行時，請使用 `paths` 或 `paths-ignore`。
- 當事件層級的篩選器太粗略時，請使用作業層級 (job-level) 的閘控。
- 當可靠性比聰明的篩選表達式更重要時，請優先使用明確的已變更檔案偵測。

### 作業形狀 (Job shaping)

- 不要盲目地合併作業。如果獨立的作業能保持平行處理並縮短關鍵路徑，請讓它們保持獨立。
- 當這能讓跳過行為變得顯而易見時，請將輕量級的協調或變更偵測作業與繁重的執行作業分開。
- 如果僅僅是工作流程本身的變更也會執行完整套件，請將此視為閘控模型太過寬泛的證據。

### 矩陣精簡 (Matrix reduction)

將矩陣廣度與所做的決定相匹配：

- 針對發行或明確的相容性驗證使用完整矩陣
- 針對執行階段、外掛程式、封裝或框架整合變更使用縮減的相容性矩陣
- 針對一般的程式碼變更使用單一代表性的最新版本分段 (leg)
- 當已有更輕量的保護措施存在時，針對明顯非執行階段的變更，不要執行繁重的測試作業

### 選用的維護作業

會寫入分支的格式化或自動修復作業，通常更適合設為「選擇性加入」(opt-in) 的作業。

良好的觸發方式：

- PR 標籤，例如 `ci:format`
- 手動發送 (Manual dispatch)
- 如果儲存庫已支援，則使用明確的註解指令流程

如果您使用標籤觸發器，請記得監聽 PR `labeled`，通常還有 `unlabeled` 事件，否則標籤變更將不會重新評估該作業。

## 安全變更規則

- 不要隱藏必要的發行、遷移或共用程式庫驗證。
- 在取代封裝動作 (wrapper action) 時，不要意外擴大已變更檔案的範圍。
- 將嚴重性漂移 (severity drift) 視為回歸風險。
- 在用原生工具取代廣泛的動作之前，請先確認實際的檢查表面。

## 即時驗證

盡可能優先使用 GitHub 的即時驗證：

- 觸發 `workflow_dispatch` 工作流程一次
- 用兩次快速更新驗證過時執行的取消
- 在現有分支上進行增量式「僅忽略」或「僅工作流程」的變更，以驗證路徑閘控
- 在 UI 中確認繁重作業被跳過，而不是假設它們會被跳過

不要將全新分支上的第一次推送視為乾淨的路徑忽略測試。
