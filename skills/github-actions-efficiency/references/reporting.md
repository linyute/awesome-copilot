# 效率報告與後續審查 (Efficiency Reporting and Follow-Up Review)

當使用者詢問變更了什麼、需要一份變更前後的報告，或要求對剩餘的昂貴作業進行再次檢查時，請載入此參考。

## 報告規則

- 將預期的節省與實際測量的節省分開。
- 若沒有變更前後的執行數據，請勿聲稱確切的時間或成本節省。
- 指出干擾因素，例如快取預熱 (cache warm-up)、變更後的矩陣廣度、執行器 (runner) 變更或異常小的 PR。

當數據不完整時，請使用這樣的說法：

`我可以報告變更的效率機制，但若沒有比較 GitHub Actions 變更前後的執行數據，我無法保證確切節省了多少分鐘。`

## 測量內容

收集：

1. 變更前的基準範例
2. 快取預熱後變更後的範例
3. 每個工作流程或每個作業的執行時間比較
4. 已避免的執行、已跳過的作業或已避免的矩陣分段

務必將以下分開：

- PR 時鐘時間 (wall-clock time)
- 跨作業的總執行器執行時間
- 完全避免的工作

這些回答了不同的問題。一項變更可以在不實質改善最快反饋路徑的情況下，減少執行器的支出。

## 後續審查步驟

在第一輪修復經過驗證後，請檢查剩餘的昂貴作業：

- 比較設定時間與執行時間
- 識別繁重的封裝動作 (wrapper actions)，並確認它們真正強制執行了什麼
- 審查每個矩陣維度是否仍然服務於一個有效的決定
- 在快取預熱後重新檢查
- 在提出進一步變更之前，先分析主導的緩慢步驟

保持後續檢查簡潔。報告接下來幾個最高價值的改進機會，而不是一份冗長的願望清單。
