# 企業級帳號規劃

針對企業級交易的策略性帳號規劃與執行。將複雜的銷售週期轉化為系統性的勝利 — 或是至少在浪費數個月時間之前，及早發現交易是否已經胎死腹中。

## 何時使用

**觸發情境：**
- 「我該如何規劃這筆企業交易？」
- 「這筆交易已經進行 3 個月了，為什麼還沒結案？」
- 「我應該建立完整的帳號計畫，還是簡化版？」
- 「我該如何知道這筆交易是否真的在推進？」
- 「MEDDICC 資格審核」
- 「建立共同行動計畫 (Mutual Action Plan)」

**背景：**
- 高於平均客單價 (ACV) 的策略性交易
- 涉及多方關係人
- 銷售週期超過 60 天
- 複雜的採購流程（法律、採購、安全審核）
- 企業級或中型市場帳號

---

## 核心框架

### 1. 如果你的 MAP 超過 3 週沒更新，這筆交易就完了

**我見過的模式：**

共同行動計畫 (MAP) 是衡量交易健康狀況最準確的指標。不是案源階段，不是口頭承諾，也不是「他們很喜歡這款產品」。

**MAP 會告訴你一切：**

**健康的交易：**
- MAP 每週更新
- 客戶主動增加他們自己的行動項
- 雙方均按時完成任務
- MAP 中出現新的關係人
- 日期提前（而非延後）

**垂死的交易：**
- MAP 最後一次更新是在 3 週前
- 只有你這邊有行動項
- 客戶的任務標示為「待處理 (pending)」已達數週
- 沒有新的關係人參與
- 所有日期都已過期

**為何會這樣：**

當一筆交易是真實的時候，客戶會希望它發生。他們會投入工作、引入關係人、推進他們的流程。

當一筆交易即將失敗時，所有工作都是你在做。他們「太忙了」、他們「下週會再回覆你」、財務買家「出差中」。

**3 週規則：**

如果你的 MAP 超過 3 週沒有更新，這筆交易就已經完了 — 只是你還不知道而已。**在我 11 年的職涯中，從未見過一筆帶著過期 MAP 的交易能成功結案。一次都沒有。**

**該怎麼做：**

**沈默第 1 週：** 寄送 MAP 更新：「這是我們已完成的部分。關於 [特定客戶動作] 的進度如何？」

**沈默第 2 週：** 向上呈報至你的內部支持者 (Champion)：「還沒收到 MAP 的回覆。我們仍按計畫在 [日期] 進行嗎？如果優先順序有變，請讓我知道。」

**沈默第 3 週：** 判定不合格 (Qualify out) 或重啟對話：「看來現在時機可能不對。我們是否應該暫停並在 [時間範圍] 重新聯繫？或者有什麼我可以幫忙排除的障礙嗎？」

**常見錯誤：**

因為「他們說他們想要」就一直把交易留在案源管道中。口頭興趣不等於行動。如果他們不動手做事，他們就不會買。

### 2. 財務買家 (EB) 探索問題（以及為何交易在第 8 週失敗）

**模式：**

交易進行到第 8 週。概念驗證 (POC) 進行得很順利。支持者很喜歡你。技術驗證已完成。你寄出了提案。

然後：音訊全無。

**發生了什麼事？** 你從未見過財務買家 (Economic Buyer, EB)。

**財務買家 (EB) 是指：**
- 控制預算分配的人
- 做出最終採購決策的人
- 在合約上簽名的人

**而不是：**
- 你的內部支持者（他們有影響力，但沒有決策權）
- 技術領導人（他們負責驗證，但不負責購買）
- 參加過一次演示的副總裁（他們提供建議，但不簽名）

**為何沒有見到 EB，交易會失敗：**

你是根據支持者的假設來建立業務案例。但 EB 的優先順序不同：
- 支持者在意：解決他們團隊的痛點
- EB 在意：投資報酬率 (ROI)、風險規避、策略對齊

當你透過支持者將提案轉交給 EB 時，EB 看到的是：
- 沒有情境說明的價標
- 一個他們未曾表述過的解決方案
- 一個他們尚未評估過的風險

**結果：** 交易停滯或失敗。

**框架：EB 驗證檢查清單**

在寄送提案前，請驗證：

- [ ] 你是否已識別出 EB？（姓名、職稱，並獲支持者確認）
- [ ] 你是否見過 EB？（最起碼要視訊通話，實體見面最理想）
- [ ] EB 是否認同該問題？（用他們的話說，而不是你的話）
- [ ] EB 是否認同成功指標？（他們將如何衡量 ROI）
- [ ] EB 是否知道價格範圍？（已討論過概算，不會感到驚訝）
- [ ] EB 是否理解時間表？（實作、上線、價值實現）

**如果你對以上任何一項的回答是「否」，請先不要寄出提案。**

**如何獲得會見 EB 的機會：**

**詢問你的支持者：**
「在我們定案價格之前，我想爭取與 [EB 姓名] 進行 15 分鐘的對談，以確保我們在成果與時間表上達成共識。你能幫我們引薦嗎？」

**如果支持者阻撓：**「我可以處理，你不需要直接跟他們談。」
→ 這是一個紅燈訊號。要麼是支持者沒有權限（不是真正的支持者），要麼是他們擔心 EB 會殺掉這筆交易（這意味著交易基礎很弱）。

**據理力爭：**
「我完全理解。但同時，我也希望確保 [EB] 在看到價格之前先看到完整的價值。根據我的經驗，如果財務買家沒有及早參與，交易往往會在採購階段被延誤。我們能安排一個簡短的對齊會議嗎？」

**常見錯誤：**

將與 EB 的會面視為「有也很好」。對於任何超過 5 萬美元的交易，這都是強制性的。不見 EB = 沒有交易。

### 3. 個人獲勝對應 (Personal Win Mapping)（人們為了自己而買）

**模式：**

企業軟體的採購是由委員會決定的。但委員會不會買東西。**人才會買。**

而人們為了個人理由而買：
- 職涯晉升
- 在老闆面前表現出色
- 減少自己的工作量
- 自保 (Covering their ass, CYA)
- 證明自己是對的
- 避免顯得愚蠢

**框架：個人獲勝識別**

針對每一位關係人，對應以下項目：

**專業層面的獲勝：**
- 如果這件事成功了，他們能得到什麼功勞？
- 他們個人的哪些痛點會消失？
- 這如何讓他們看起來很專業？

**專業層面的風險：**
- 如果這件事失敗了，他們會發生什麼事？
- 如果出錯了，他們的名譽損失為何？
- 內部誰在質疑他們？

**個人動機：**
- 他們是新官上任嗎？（需要快速獲得成績）
- 面臨預算削減嗎？（需要證明支出合理性）
- 快要升遷了嗎？（需要顯眼的成功案例）
- 以前被供應商坑過嗎？（格外規避風險）

**範例：工程副總裁**

**專業層面的獲勝：**
- 減輕團隊的輪值負擔（他們就不會再向她抱怨）
- 更快的事件回應（在季度業務回顧中表現出色）
- 吸引更好的工程人才（現代化的工具）

**專業層面的風險：**
- 團隊拒絕新工具（那是她強加給他們的）
- 遷移過程不順利（停機、事件）
- 供應商倒閉（是她選的）

**個人動機：**
- 新上任（6 個月），需要成績
- 面臨改善系統上線率指標的壓力
- 以前選過的監控工具失敗了

**這如何改變你的說服方式：**

**通用說法：**
「我們的平台能將事件回應時間縮短 40%。」

**個人獲勝說法：**
「你提到輪值負擔讓你的團隊精疲力竭。我們看過有些團隊在第一個月就減少了 40% 的輪值呼叫，這有助於人才留存。而且既然你正專注於向董事會報告上線率指標，缩短的回應時間會立即顯現在你的 QBR 儀表板上。」

**差異點：**

通用 = 業務案例
個人 = 職涯案例

兩者都很重要。但個人獲勝才能促成交易。

**常見錯誤：**

只針對業務問題進行銷售。「這能省錢。這能提高效率。」這固然必要，但還不夠。**人們需要看到這對他們個人有什麼好處。**

### 4. 企業級帳號計畫結構（四個組成部分）

一份完整的帳號計畫包含四個互相關聯的部分。每一部分都為其他部分提供資訊。

**第一部分：帳號摘要**
- 公司基本資訊（總部、規模、產業、子公司）
- 技術現況（基礎架構、工具、平台）
- 首要公司倡議（來自新聞、年報、LinkedIn）
- 假設：「我們能如何提供協助？」（在接觸前寫下）
- LinkedIn 關鍵字分析（量化他們在你的領域的投入程度）

**第二部分：組織圖**
- 對應所有相關聯繫人：姓名、職稱、地點、LinkedIn、Email、電話、備註
- 備註內容：負責領域、技術專長、個人獲勝
- 涵蓋各層級的人員：C 級高管、總監、架構師、領隊、專員
- 不要只對應買家 — 也要對應影響者、使用者和潛在的阻礙者

**第三部分：機會計畫 (MEDDICC)**
- **M - Metrics (指標)**：客戶將如何衡量成功？（已獲 EB 驗證）
- **E - Economic Buyer (財務買家)**：誰擁有預算權限？你見過他們了嗎？
- **D - Decision Criteria (決策標準)**：他們將使用什麼標準來決定？（技術、業務、政治）
- **D - Decision Process (決策流程)**：他們的採購流程為何？（採購、法律、安全審核）
- **I - Identified Pain (已識別痛點)**：他們表述了哪些具體痛點？（用他們的話，而非你的）
- **C - Champion (內部支持者)**：帳號內部誰在積極為你推銷？
- **C - Competition (競爭對手)**：他們還在評估誰？競爭態勢為何？

外加：帶有緩解計畫、所需協助、責任方的問題/風險表

**第四部分：共同行動計畫 (MAP)**
- 聯合時間表，包含：行動、我方負責人、客戶方負責人、其他參與者、截止日期
- 雙方都必須有行動 — 如果只有你的團隊有行動，那不是一筆交易，而只是一場演示
- 追蹤狀態（已完成/進行中）
- 將 MAP 作為定期追蹤會議的議程
- **如果 MAP 超過 3 週沒更新，交易即判定失敗**

**決策標準：**

對於潛在交易規模前 10-20% 的帳號，值得投入建立完整計畫。其餘帳號可使用簡化版（摘要 + MEDDICC + 下一步行動）。

**常見錯誤：**
- 在交易開始後才建立帳號計畫（應在首次接觸前建立）
- 未能每週維護 MAP（過期的 MAP = 停滯的交易）
- MEDDICC 填充的是假設而非經驗證的資訊
- 僅對應明顯的聯繫人而非完整的組織圖
- 未能追蹤每位關係人的個人獲勝

### 5. 用於帳號情報的 LinkedIn 關鍵字分析

在接觸策略性帳號之前，透過 LinkedIn 量化他們在你的領域的投資程度。

**如何執行：**

1. 定義 8-10 個與你的領域相關的關鍵字（例如：類別術語、技術角色、工作流程關鍵字）
2. 在 LinkedIn 搜尋「[公司名稱] + [關鍵字]」並記錄數量
3. 對應分佈：哪些地點？哪些部門？
4. 識別離群值（特定部門的高關鍵字集中度代表成熟度）

**為何這有效：**

如果一家公司有 50 位員工在個人檔案中提到 "SRE"，代表他們在網站可靠性工程方面很成熟。如果只有 2 位，他們可能還沒準備好使用進階的觀測工具。

這能告訴你：
- 是否值得追求該帳號（他們有相應團隊嗎？）
- 該瞄準誰（人才集中在哪裡？）
- 如何進行個人化外展（引用他們的具體情境）

**範例：**

搜尋 "[Company] + DevOps"：
- 120 個結果 → 成熟的 DevOps 組織，契合度高
- 5 個結果 → 早期階段，尚未準備好

搜尋 "[Company] + SRE"：
- 50 個結果 → 他們在意可靠性，主打上線率/事件減少的價值主張
- 0 個結果 → 不要以 SRE 作為首要價值主張

**常見錯誤：**
- 僅搜尋職稱（變化很大）而非關鍵字（較一致）
- 未能將數量與總員工數進行比較
- 未能更新分析（招聘趨勢每季都在變）

### 6. 統一銷售流程（階段關卡）

企業級銷售遵循定義明確的階段，並有清晰的退出標準。在未達標準前不要推進階段。

**階段 0 — 案源產生 (Pipeline Generation)**：開發 → 確認有合格興趣
**階段 1 — 探索 (Discovery)**：環境/痛點/需求 → 識別痛點，對應關係人
**階段 2 — 演示 (Demonstrating)**：產品演示，建立支持者 → 識別出內部支持者
**階段 3 — 價值證明 (Proving Value)**：POC/試用 → 技術驗證完成
**階段 4 — 提案 (Proposal)**：定價、條款、範圍 → 交付提案，與 EB 達成一致
**階段 5 — 簽約流程 (Paper Process)**：法律、採購、安全 → 獲得批准
**階段 6 — 結案贏單 (Closed Won)**：合約簽署 → 客戶成功團隊交接

**退出標準至關重要：**

在擁有內部支持者前，不要從階段 2 移至階段 3。
在 POC 成功標準達成前，不要從階段 3 移至階段 4。
在獲得 EB 批准前，不要從階段 4 移至階段 5。

**常見錯誤：**

根據活動而非標準來推進階段。「我們演示過了，所以我們進入階段 3」— 但如果他們還沒同意進行 POC，你其實還留在階段 2。

## 決策樹

### 我需要建立完整的帳號計畫嗎？

```
交易規模是否高於平均 ACV？
├─ 否 → 簡化版計畫（摘要 + MEDDICC）
└─ 是 → 繼續...
    │
    銷售週期是否 >60 天？
    ├─ 是 → 完整帳號計畫
    └─ 否 → 簡化版計畫
```

### 這筆交易真的在推進嗎？

```
MAP 是否每週更新？
├─ 是 → 健康
└─ 否 → 繼續...
    │
    距離上次 MAP 更新是否已超過 3 週？
    ├─ 是 → 失敗交易（判定不合格或重啟對話）
    └─ 否 → 有風險（向上呈報至支持者）
```

### 我應該寄出提案嗎？

```
你見過財務買家 (EB) 了嗎？
├─ 否 → 先不要寄（先爭取會見 EB）
└─ 是 → 繼續...
    │
    EB 是否認同痛點與成功指標？
    ├─ 是 → 寄出提案
    └─ 否 → 在寄出前先與 EB 達成共識
```

## 常見錯誤

**1. 太晚才建立帳號計畫**
   - 應在首次接觸前建立，而非在交易開始推進後。

**2. MEDDICC 填充的是假設**
   - 與客戶驗證每個元素，不要用猜的。

**3. 過期的共同行動計畫 (MAP)**
   - 如果 MAP 未能每週更新，代表交易陷入停滯。超過 3 週即視為失敗。

**4. 僅對應買家**
   - 需要完整的組織圖：影響者、使用者、阻礙者。

**5. 忽視個人獲勝**
   - 人們為了職涯/名譽理由而買，而不僅僅是業務上的 ROI。

**6. 未能追蹤交易健康狀況**
   - 紅黃綠指標能及早發現垂死的交易。

**7. 跳過支持者驗證**
   - 沒有內部支持者，你是一個人在孤軍奮戰。

## 快速參考

**MAP 健康檢查：**
- 綠色：每週更新，雙方皆有行動，客戶持續完成任務。
- 黃色：每雙週更新，大部分是你的行動，客戶回應緩慢。
- 紅色：過期 3 週以上，只有你的行動，客戶無回應 → **失敗交易**

**MEDDICC 驗證：**
- [ ] 指標 (Metrics)：與 EB 達成共識的成功標準。
- [ ] 財務買家 (Economic Buyer)：已會面，並驗證問題/方案。
- [ ] 決策標準 (Decision Criteria)：理解他們的評估準則。
- [ ] 決策流程 (Decision Process)：了解採購/法律/安全審核步驟。
- [ ] 已識別痛點 (Identified Pain)：用客戶的話表述。
- [ ] 內部支持者 (Champion)：在內部積極為你推銷。
- [ ] 競爭對手 (Competition)：了解他們正在考慮的其他方案。

**個人獲勝提問：**
- 「對你個人而言，成功看起來是什麼樣子？」
- 「如果這件事成功了，對你的團隊有什麼影響？如果失敗了呢？」
- 「今年衡量你表現的指標是什麼？」
- 「內部誰持懷疑態度？為什麼？」

**帳號計畫檢查清單：**
- [ ] 帶有假設的帳號摘要。
- [ ] 對應了個人獲勝的組織圖。
- [ ] 完全經過驗證（非假設）的 MEDDICC。
- [ ] 包含客戶行動（非僅有我方）的 MAP。
- [ ] 已排定每週更新 MAP 的節奏。
