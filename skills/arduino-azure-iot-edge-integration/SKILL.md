# Arduino Azure IoT Edge 整合

當使用者需要將 Arduino 級別的裝置連接到 Azure IoT 時，請使用此技能，特別是在邊緣運算密集的場景中 (閘道、間歇性網路、離線緩衝和本機驅動)。

## 何時使用

針對以下請求使用此技能：

- 「我想將 Arduino 感測器連接到 Azure」
- 「如何將 MQTT 遙測傳送到 IoT Hub？」
- 「我需要一個用於現場裝置的邊緣閘道」
- 「我想要雲端對裝置指令和 OTA 設定更新」

## 強制性文件檢閱

在建議 IoT Edge 拓撲或執行階段行為之前，請先檢閱：

- https://learn.microsoft.com/azure/iot-edge/

如果無法諮詢文件，請以明確的假設繼續進行，並在專用章節中醒目提示。

在提出韌體、配線或通訊實作細節之前，請先諮詢 Arduino 官方來源：

- https://www.arduino.cc/en/Guide
- https://docs.arduino.cc/

## 目標

- 建立一個從 Arduino 裝置到雲端深入分析的安全端對端參考路徑。
- 處理不穩定的連結 (儲存並轉發、重試、等冪性)。
- 定義可執行的裝置和雲端待辦清單 (backlog)。

## 整合模式

### 模式 A：Arduino 直接連接到 IoT Hub

當連線穩定且雲端延遲可接受時使用。

- 協定：MQTT over TLS。
- 身分：每個裝置專屬的認證 (SAS 或 X.509)。
- 遙測酬載：包含時間戳記、裝置 ID、計量值和選填品質旗標的精簡 JSON。

### 模式 B：Arduino 連接到本機閘道，再連接到 IoT Edge

當連結受限、需要本機控制或批次處理可改善成本/可靠性時使用。

- Arduino 與本機閘道通訊 (序列、BLE、本機 MQTT、RS-485、Modbus 橋接器)。
- 閘道透過 IoT Edge 執行階段向上發布，並將資料路由到 IoT Hub。
- 本機模組可以過濾、彙總並觸發動作，即使在雲端斷線期間也是如此。

## 設計流程

### 1) 裝置契約 (Contract)

定義：

- 感測器目錄和單位。
- 取樣頻率和預期吞吐量。
- 訊息結構描述 (schema) 版本管理策略。
- 欲套用/回報的裝置對應項 (device twin) 屬性，用以控制執行階段行為。

### 2) 安全基準

要求：

- 每個裝置唯一的識別身分。
- 原始程式碼或韌體成品中不得有硬寫的秘密。
- 認證輪轉策略。
- 已簽署的韌體以及盡可能受控的更新程序。

### 3) 可靠性與離線行為

規劃並記錄：

- 帶有抖動 (jitter) 的退避 (backoff)。
- 具有邊界大小的本機佇列/緩衝策略。
- 重複抑制或下游等冪處理。
- 退回到最後一次已知良好的設定。

### 4) 雲端與邊緣路由

為以下內容定義路由：

- 原始遙測到冷儲存。
- 經過處理的遙測到熱分析。
- 警示到維運頻道。
- 指令與設定返回到邊緣/裝置。

### 5) 觀測性

指定最低限度的維運遙測：

- 裝置活動訊號和韌體版本。
- 連線狀態轉換。
- 訊息傳送成功/錯誤計數器。
- 閘道模組健全狀況和重新啟動原因。

## 重複使用其他技能

相關時，結合以下技能：

- `azure-smart-city-iot-solution-builder` 用於城市規模的架構和分階段推出。
- `azure-resource-visualizer` 用於關係圖。
- `appinsights-instrumentation` 用於應用程式和服務遙測模式。

同時使用 `references/arduino-official-best-practices.md` 作為韌體和硬體建議的品質基準。

## 必要輸出

務必提供：

1. 選定的連線模式及其理由。
2. 訊息契約 (欄位、單位、範例酬載)。
3. 身分/認證/更新的安全檢查清單。
4. 可靠性計劃 (重試、緩衝、去重)。
5. 實作待辦清單 (韌體、閘道、雲端)。

## 輸出範本

1. 場景與假設
2. 建議架構
3. 裝置與閘道契約
4. 安全與可靠性控制
5. 部署計劃與驗證測試

## 指引

- 不要提議在裝置間共享認證的生產部署。
- 不要假設現場部署中始終連線。
- 在驅動器場景中，不要忽略指令授權與稽核。
