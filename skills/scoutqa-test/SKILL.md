# ScoutQA 測試技能

使用 `scoutqa` CLI 對 Web 應用程式執行 AI 驅動的探索性測試。

**將 ScoutQA 視為您的智慧測試夥伴**，它能自主探索、發現問題並驗證功能。將測試委派給多個平行執行的 ScoutQA，以在節省時間的同時最大化測試覆蓋率。

## 何時使用此技能

在以下兩種情境下使用此技能：

1. **使用者要求測試** - 當使用者明確要求測試網站或驗證功能時。
2. **主動驗證** - 在實作 Web 功能後，自動執行測試以驗證實作是否正確。

**主動使用範例：**

- 實作登入表單後 → 測試驗證流程
- 新增表單驗證後 → 驗證驗證規則與錯誤處理
- 建立結帳流程後 → 測試端到端購買程序
- 修復錯誤後 → 驗證修復是否有效且未破壞其他功能

**最佳實作**：當您完成 Web 功能實作時，請主動在背景啟動 ScoutQA 測試進行驗證，同時繼續處理其他工作。

## 執行測試

### 測試工作流程

複製此檢查表並追蹤您的進度：

測試進度：

- [ ] 撰寫具有明確預期的具體測試提示詞
- [ ] 在背景執行 scoutqa 命令
- [ ] 告知使用者執行 ID 與瀏覽器 URL
- [ ] 擷取並分析結果

**步驟 1：撰寫具體的測試提示詞**

請參閱下方的「撰寫有效提示詞」章節以獲取指南。

**步驟 2：執行 scoutqa 命令**

**重要**：使用 Bash 工具的逾時 (timeout) 參數 (5000ms = 5 秒) 來擷取執行詳細資訊：

呼叫 Bash 工具時，請將 `timeout: 5000` 設定為參數：

- 這是 Claude Code 中 Bash 工具內建的逾時參數 (而非 Unix 的 `timeout` 命令)
- 5 秒後，Bash 工具會回傳控制權並提供一個任務 ID，而程序將在背景繼續執行
- 這不同於 Unix 的 `timeout` 命令會終止程序——在這裡，程序會持續運行
- 前 5 秒用於從 ScoutQA 的輸出中擷取執行 ID 與瀏覽器 URL
- 測試將隨著背景任務在 ScoutQA 的基礎架構上遠端繼續執行

```bash
scoutqa --url "https://example.com" --prompt "您的測試指令"
```

在最初的幾秒鐘內，命令將輸出：

- **執行 ID (Execution ID)** (例如：`019b831d-xxx`)
- **瀏覽器 URL (Browser URL)** (例如：`https://app.scoutqa.ai/t/019b831d-xxx`)
- 顯示測試進度的初始工具呼叫

在 5 秒逾時後，Bash 工具會回傳任務 ID，且命令會在背景繼續執行。您可以在測試執行期間處理其他工作。逾時僅是為了擷取初始輸出 (執行 ID 與瀏覽器 URL)——測試會在本地作為背景任務及在 ScoutQA 基礎架構上遠端持續執行。

**步驟 3：告知使用者執行 ID 與瀏覽器 URL**

在 Bash 工具回傳任務 ID (且已在前 5 秒內擷取執行詳細資訊) 後，請告知使用者：

- ScoutQA 執行 ID 與瀏覽器 URL，以便他們在瀏覽器中監控進度
- 如果他們稍後想檢查本地命令輸出，請提供背景任務 ID

測試會在背景執行，而您可以繼續其他工作。

**步驟 4：擷取並分析結果**

完整格式請參閱下方的「呈現結果」章節。

### 命令選項

- `--url` (必要)：要測試的網站 URL (支援 `localhost` / `127.0.0.1`)
- `--prompt` (必要)：自然語言測試指令
- `--project-id` (選填)：關聯至專案以進行追蹤
- `-v, --verbose` (選填)：顯示所有工具呼叫，包括內部呼叫

### 本地測試支援

ScoutQA 支援自主測試 `localhost` 與 `127.0.0.1` URL——無需手動設定。

```bash
# 在開發應用程式時，無縫測試正在本地執行的應用程式
scoutqa --url "http://localhost:3000" --prompt "測試註冊表單"
```

### 何時使用各個命令

**啟動新測試？** → 使用 `scoutqa --url --prompt`
**驗證已知問題？** → 使用 `scoutqa issue-verify --issue-id <id>`
**從執行紀錄中尋找問題 ID？** → 使用 `scoutqa list-issues --execution-id <id>`
**代理程式需要更多內容？** → 使用 `scoutqa send-message` (參閱「後續處理停滯的執行」)

## 撰寫有效提示詞

重點在於**要探索與驗證什麼**，而非規範性的步驟。ScoutQA 會自主決定如何進行測試。

**範例：使用者註冊流程**

```bash
scoutqa --url "https://example.com" --prompt "
探索使用者註冊流程。測試表單驗證的極端案例 (edge cases)，
驗證錯誤處理，並檢查無障礙性合規情況。
"
```

**範例：電子商務結帳**

```bash
scoutqa --url "https://shop.example.com" --prompt "
測試結帳流程。驗證價格計算、購物車持久性、
付款選項及行動裝置回應式設計。
"
```

**範例：執行平行測試以獲得全面覆蓋**

在單一訊息中進行多次 Bash 工具呼叫，啟動多個平行測試，每次呼叫都將 Bash 工具的 `timeout` 參數設定為 `5000` (毫秒)：

```bash
# 測試 1：驗證與安全性
scoutqa --url "https://app.example.com" --prompt "
探索身分驗證：登入/登出、工作階段處理、密碼重設，
以及安全性極端案例。
"

# 測試 2：核心功能 (平行執行)
scoutqa --url "https://app.example.com" --prompt "
測試儀表板與主要使用者工作流程。驗證資料載入、
CRUD 作業及搜尋功能。
"

# 測試 3：無障礙性 (平行執行)
scoutqa --url "https://app.example.com" --prompt "
執行無障礙性稽核：WCAG 合規性、鍵盤導覽、
螢幕閱讀器支援、色彩對比度。
"
```

**實作方式**：發送含有三次 Bash 工具呼叫的單一訊息。針對每次 Bash 工具呼叫，將 `timeout` 參數設定為 `5000` 毫秒。5 秒後，每次 Bash 呼叫都會回傳任務 ID，而程序會在背景繼續執行。這將從初始輸出中擷取每個測試的執行 ID 與瀏覽器 URL，隨後所有三個測試都會平行執行 (同時作為本地背景任務與 ScoutQA 基礎架構上的遠端任務)。

**關鍵準則：**

- 描述**要測試什麼**，而非**如何測試** (ScoutQA 會自行找出步驟)
- 專注於目標、極端案例與疑慮
- 針對不同測試領域執行多個平行執行任務
- 信任 ScoutQA 能自主探索並發現問題
- 呼叫 scoutqa 命令時，務必將 Bash 工具的 `timeout` 參數設定為 `5000` 毫秒 (這會在 5 秒後回傳控制權，同時程序在背景繼續執行)
- 對於平行測試，請在單一訊息中進行多次 Bash 工具呼叫
- 記住：Bash 工具逾時 ≠ Unix timeout 命令 (Bash 逾時會在背景繼續程序，Unix 逾時會終止程序)

### 常見測試案例

**部署後冒煙測試：**

```bash
scoutqa --url "$URL" --prompt "
冒煙測試：驗證部署後關鍵功能是否正常運作。
檢查首頁、導覽、登入/登出及關鍵使用者流程。
"
```

**無障礙性稽核：**

```bash
scoutqa --url "$URL" --prompt "
無障礙性稽核：WCAG 2.1 AA 合規性、鍵盤導覽、
螢幕閱讀器支援、色彩對比度及語義化 HTML。
"
```

**電子商務測試：**

```bash
scoutqa --url "$URL" --prompt "
探索電子商務功能：產品搜尋/篩選、
購物車操作、結帳流程及價格計算。
"
```

**SaaS 應用程式：**

```bash
scoutqa --url "$URL" --prompt "
測試 SaaS 應用程式：身分驗證、儀表板、CRUD 作業、
權限及資料完整性。
"
```

**表單驗證：**

```bash
scoutqa --url "$URL" --prompt "
測試表單驗證：極端案例、錯誤處理、必填欄位、
格式驗證及成功提交。
"
```

**行動裝置回應式設計：**

```bash
scoutqa --url "$URL" --prompt "
檢查行動裝置體驗：回應式版面配置、導覽、
觸控互動及視埠 (viewport) 行為。
"
```

**已知問題驗證：**

```bash
# 首先，從先前的執行紀錄中尋找問題 ID
scoutqa list-issues --execution-id <executionId>

# 然後驗證問題 (會自動建立一個新的驗證執行任務)
scoutqa issue-verify --issue-id <issueId>
```

`issue-verify` 命令將：

1. 為該問題建立驗證執行任務
2. 顯示執行 ID 與瀏覽器 URL
3. 即時串流代理程式的驗證進度
4. 顯示包含結果連結的完成摘要

**功能驗證 (實作後)：**

```bash
scoutqa --url "$URL" --prompt "
驗證新功能 [功能名稱] 是否正常運作。測試核心功能、
極端案例、錯誤處理以及與現有功能的整合。
"
```

**範例：編寫功能後的主動測試**

實作使用者註冊表單後，自動驗證其運作情況：

```bash
scoutqa --url "http://localhost:3000/register" --prompt "
測試新實作的註冊表單。驗證：
- 表單驗證 (電子郵件格式、密碼強度、必填欄位)
- 錯誤訊息是否正確顯示
- 成功註冊流程
- 極端案例 (重複的電子郵件、特殊字元等)
"
```

這能在實作內容尚處於背景脈絡時，立即發現問題。

## 列出問題

使用 `scoutqa list-issues` 瀏覽先前執行中發現的問題。這對於尋找要用於 `issue-verify` 的問題 ID 非常有用。

```bash
scoutqa list-issues --execution-id <executionId>
```

**選項：**

- `--execution-id` (必要)：執行 ID (來自 `/t/<executionId>` URL 或 CLI 輸出)

**範例輸出：**

```
顯示 3 個問題：

🔴 019c-abc1
   登入按鈕在行動裝置上無回應
   嚴重程度：critical | 類別：usability | 狀態：open

🟠 019c-abc2
   電子郵件欄位缺少表單驗證
   嚴重程度：high | 類別：functional | 狀態：open

🟡 019c-abc3
   頁尾連結的色彩對比度不足
   嚴重程度：medium | 類別：accessibility | 狀態：resolved
```

## 呈現結果

### 立即呈現 (啟動測試後)

執行 scoutqa 命令後，立即向使用者呈現執行詳細資訊：

```markdown
**ScoutQA 測試已啟動**

執行 ID：`019b831d-xxx`
即時檢視：https://app.scoutqa.ai/t/019b831d-xxx

測試正在遠端執行。您可以在上述連結中檢視瀏覽器的即時進度，同時我將繼續處理其他工作。
```

### 最終結果 (完成後)

當執行完成時，使用以下格式呈現發現結果：

```markdown
**ScoutQA 測試結果**

執行 ID：`ex_abc123`

**發現的問題：**

[High] 無障礙性：標誌 (logo) 圖片缺少 alt 文字

- 影響：螢幕閱讀器無法描述標誌
- 位置：頁首導覽

[Medium] 可用性：提交按鈕在行動裝置視埠中不可見

- 影響：使用者無法在行動裝置上完成表單
- 位置：聯絡表單，頁面底部

[Low] 功能性：搜尋對有效查詢回傳無結果

- 影響：搜尋功能看起來已壞掉
- 位置：主搜尋列

**摘要：** 在無障礙性、可用性及功能性類別中發現 3 個問題。請至上述 URL 查看包含螢幕截圖的完整互動式報告。
```

務必包含：

- **執行 ID** (例如：`ex_abc123`) 以供參考
- **發現的問題**，包含嚴重程度、類別 (無障礙性、可用性、功能性)、影響及位置

## 後續處理停滯的執行

若遠端代理程式停滯或需要釐清，請使用 `send-message` 繼續：

```bash
# 範例：代理程式卡在登入頁面，使用者提供憑證
scoutqa send-message --execution-id ex_abc123 --prompt "
請使用這些測試憑證：使用者名稱：testuser@example.com，密碼：TestPass123
"

# 範例：代理程式詢問接下來要測試哪個流程
scoutqa send-message --execution-id ex_abc123 --prompt "
接下來請專注於結帳流程，略過願望清單功能
"
```

## 檢查測試結果

ScoutQA 測試在 ScoutQA 的基礎架構上遠端執行。在以短逾時啟動測試以擷取執行 ID 後：

1. 測試會遠端持續執行 (而非在本地背景執行)
2. 您可以立即繼續其他工作
3. 若要稍後檢查結果，請造訪啟動測試時提供的瀏覽器 URL
4. 或者，使用 `scoutqa get-execution --execution-id <id>` 透過 CLI 獲取結果

**最佳實作**：藉由將 Bash 工具的 `timeout` 參數設定為 `5000` 毫秒來啟動測試。5 秒後，Bash 工具會回傳控制權與任務 ID 以及執行詳細資訊 (執行 ID 與瀏覽器 URL)，同時測試會在背景繼續執行。接著您可以繼續其他工作，並在需要時於 ScoutQA 網站或透過 CLI 檢查結果。

## 疑難排解

| 問題                           | 解決方案                                                     |
| ------------------------------ | ------------------------------------------------------------ |
| `command not found: scoutqa`   | 安裝 CLI：`npm i -g @scoutqa/cli@latest`                     |
| 驗證過期 / 未授權              | 執行 `scoutqa auth login`                                    |
| 測試掛起或需要輸入             | 使用 `scoutqa send-message --execution-id`                   |
| 檢查測試結果                   | 造訪瀏覽器 URL 或使用 `scoutqa get-execution --execution-id` |
| 需要問題 ID 以進行驗證         | 執行 `scoutqa list-issues --execution-id <id>`               |
