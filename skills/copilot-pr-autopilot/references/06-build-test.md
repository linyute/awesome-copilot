# 步驟 6：依儲存庫慣例建構與測試

子代理人類型：`task`（可能平行衍生出數個 `explore` 子代理人
進行探索）；預算：10 分鐘（對於慢速測試套件，延長上限
可達 2 倍）。

## 輸入

- 在步驟 5 中變動的檔案集合（來自每個修正子代理人的
  `files_touched`）。
- 父代理人從先前輪次中快取的關於該儲存庫之
  建構 / 測試 / lint 指令集。

## 回傳協定

```
{ status, failures }
```

其中 `status` ∈ `pass` | `fail`，而 `failures` 是來自
失敗工具輸出的相關摘錄（建構錯誤、測試失敗、
lint 診斷）——足以讓父代理人決定是否返回
步驟 5 進行後續修正，或直接推送。

## 程序

**優先進行探索** — 閱讀並結合：

- `.github/instructions/*.md`，
- `AGENTS.md`，
- `CONTRIBUTING.md`，
- `README.md`，
- `package.json` 腳本，
- `Makefile`，
- 特定語言的工具設定，
- 以及最近的 CI 工作流程執行紀錄（`gh run list`、`gh run view`）以瞭解
  實際使用的指令集。

然後在變更的程式碼上執行這些確切的指令。獨立的
探索面向（建構工具 / 測試執行器 / lint / 拼字檢查 / 格式化）
可以作為獨立的 `explore` 子代理人平行分派；快取
每輪探索到的指令，以便重新執行時不需要重新探索。

## 注意事項

- **切勿憑空捏造通用的建構指令。** 損壞的建構會浪費
  下一個完整的審查週期（3-10 分鐘）。如果探索一無所獲，
  請回報此落差 —— 不要用猜的。
- **尊重儲存庫特定的拼字檢查 / lint / 格式化政策。**
  某些儲存庫偏好重新措辭而非加入白名單項目；某些有
  樣式/正規表示式檔案；某些接受行內忽略指令。請先
  檢查儲存庫現有的設定和最近的提交紀錄，再套用
  通用的修正。
- **按每輪快取探索到的指令，而非按每次迴圈。** 儲存庫設定
  可能會在不同輪次之間發生變化（修正可能會加入新的 lint），
  因此請在每輪開始時重新探索，但在該輪之內重複使用。
- **失敗將會被送回步驟 5。** 當 `status: fail` 時，父代理人
  會帶著失敗摘錄作為新的發現重新進入步驟 5 —— 不要
  為了滿足步驟 7 而推送損壞的建構。
