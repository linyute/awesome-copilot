# 步驟 4: 分流

子代理類型：`general-purpose`；預算：**每個 ≤5 執行緒 5 分鐘**
（若步驟 3 傳回更多，父代以 ≤5 的波次進行批次處理）。

## 輸入

來自步驟 3：
- 分類執行緒表 — 每個開啟的執行緒為 `{ thread_id, file, line, author,
  author_class, severity, summary }`。
- 原始 PR 關聯（說明、受影響檔案的最近提交），用於在震盪情況下從原則重新推導。

## 回傳合約

資料列的表格，每個執行緒一列：

```
{ thread_id, action, rationale }
```

其中 `action` ∈ `fix` | `decline` | `escalate-to-user` 且
`rationale` 是單行說明，引用下方分流指引中觸發的規則。

## 程序

針對每個執行緒，依序套用此檔案中的分流指引：

1. **檢視者類型原則** — `human-or-bot` 執行緒預設為
   `escalate-to-user`，除非使用者明確將其納入範圍。
2. **投資報酬率對照風險** — 評分並決定 `copilot` 執行緒。
3. **修復 / 拒絕規則清單** — 匹配發現的模式。
4. **專案專屬原則連結** — 在套用一般「修復」之前研究存放庫規範。
5. **衝突留言解決** — 如果發現事項翻轉了前一輪的編輯，套用震盪規則；如果代理即將重新編輯其已還原的內容，硬性停止並呈報。
6. **呈報規則** — 針對設計層級的折衷、大型/橫切變更或高風險/不可逆的動作進行呈報。

為每個執行緒向父代傳回 `{ thread_id, action, rationale }`；
步驟 5 僅取用 `fix` 列，步驟 8 取用完整表格。

## 陷阱

- **每次子代理呼叫以 ≤5 的波次進行批次處理。** 較大的批次會模糊原由並超出 5 分鐘的預算。
- **人類 / 進階安全性預設為 `escalate-to-user`。** 自動回覆或自動解決人類審查執行緒會隱藏未處理的問題，且在社交上是不對的。
- **在 `rationale` 中引用觸發的規則** — 具體、單行。例如「依分流指引 §Decline 的風格微修」可以；「看起來很小」則不行。
- **震盪硬性停止是不容妥協的。** 如果代理即將進行在前一輪中已還原的相同編輯，這是震盪迴圈的明確特徵 — 進行呈報。

# 分流指引

決定是否修復或拒絕每個 Copilot 發現事項的決策指引。
目標是正確性，而非姑息 — 在有合理理由時有信心地拒絕。

## 投資報酬率對照風險

對每個建議的變更在兩個軸上進行評分，然後做出決定：

- **投資報酬率** = 修復的價值減去實作成本減去使用者檢視的成本。
- **風險** = 影響範圍、可逆性、對無關程式碼的影響。

| 投資報酬率 | 風險 | 動作 |
|---|---|---|
| 明確正值 | 低 | 單方面修復。 |
| 邊緣值 | 低 | 若成本低則修復；否則說明原由並拒絕。 |
| 明確正值 | 高 / 不可逆 | 先向使用者提議。 |
| 邊緣值 | 高 | 拒絕。 |
| 負值（針對假設性情況過度設計） | 任何 | 拒絕。 |

## 檢視者類型原則（基礎，而非細微處）

每個執行緒的 `author`（由 `03-list-open-threads.ps1` 傳回）決定了誰可以決定它：

| 檢視者 | 預設動作 |
|----------|----------------|
| `copilot-pull-request-reviewer` / `copilot-pull-request-reviewer[bot]` | 迴圈所有 — 使用下方的分流指引進行分流。（`03-list-open-threads.ps1` 回報原始的 `author.login`，其在某些介面上可能帶有 `[bot]` 後綴；匹配這兩種格式。） |
| 人類檢視者 | **預設 `escalate-to-user`**，除非使用者明確將其納入迴圈範圍。自動回覆或自動解決人類執行緒會隱藏未處理的問題，且在社交上是不對的。 |
| `github-advanced-security` / 其他自動化機器人 | **預設 `escalate-to-user`**，除非專案有已記錄的隱藏 / 修復規範可供遵循。 |

## 在發現事項為以下情況時進行修復...

- **真實的正確性錯誤**：釋放後使用 / 生命週期違規、捨棄使用者意圖的競爭條件、略過合法過渡的閘道邏輯、遺失連結相依性、差一錯誤、空值取值、未處理的錯誤路徑。
- **具有乾淨本機修復的橫切關注點**：將互斥鎖向上移動一個範圍、將重複的檢查提取到协助工具中。
- **文件 / 測試計劃偏離**：PR 說明主張的行為與程式碼不再相符；留言區塊描述了錯誤的內容；測試計劃核取方塊與程式碼相反。

## 在發現事項為以下情況時進行拒絕...

- **純粹假設性的競爭條件**，需要跨類別配管，而實際暴露微乎其微。引用您排除的交錯情況。
- **風格、命名或格式化**。超出審查迴圈的範圍。
- **建議新增抽象概念**（「引入策略模式」、「提取介面」），在目前程式碼庫的規模下不划算。
- **與既定專案規範衝突的建議** — 與周圍程式碼的一致性通常比單獨的建議更有價值。
- **微優化**，存在於非熱點路徑的程式碼中。

## 專案專屬原則連結

某些發現事項是由**專案原則**決定，而非一般的正確性推論。在套用一般的「修復」之前，**請先研究存放庫自身的規範** — `.github/instructions/*.md` 檔案（通常有將其固定到變更檔案路徑的 `applyTo` 萬用字元）、`.github/skills/`、`AGENTS.md`、`CONTRIBUTING.md`、CI 組態以及類似檔案的最近提交。當需要檢查多個軸（Lint、格式、拼字檢查、授權標頭等）時，分派多個 `explore` 子代理 — 不要自行發明答案。看似明顯的修復可能會違反專案規則：

- **拼字檢查 / 字典發現事項**。如果專案使用拼字檢查器（`check-spelling`、`cspell`、`typos` 或類似工具），請檢查其組態和最近的提交以了解本機規範（重寫文件、新增模式/正規表示式、擴充字典/允許清單、使用行內忽略）。遵循該規範；不要發明新的機制。
- **Lint 隱藏 / 行內忽略指示**。大多數專案需要行內原由留言。裸隱藏會被推回。
- **授權標頭 / 檔案樣板**。專案專屬格式，通常由 CI 強制執行 — 從鄰近檔案複製現有的標頭。
- **測試框架、模擬程式庫、格式化工具選擇**。遵循周圍測試檔案的規範；絕不要因為 Copilot 的建議而引入新的框架。

在您的回覆中引用專案的組態檔案或規範。

## 衝突留言 — 儘早中斷震盪

失敗模式：第 N 輪「修復」了留言 A 要求的內容；第 N+1 輪留言 B 反對並要求還原它。盲目地來回修改會使修改產生震盪並消耗輪次。

解決規則 — 依序套用，在第一個觸發時停止：

1. **從原則重新推導**。選擇在函式合約 + 周圍模式 + 使用者偏好中獲勝的一方 — 而非最新的留言。
2. **偏好具有明確原由的立場**。具體的失敗模式（安全性、正確性、資料遺失、效能）勝過風格立場。
3. **在直接衝突時偏好人類留言而非機器人留言**。
4. **如果仍然模糊，則呈報給使用者**。並排摘要兩個立場並提供建議；不要無聲地挑選。

**硬性停止**：如果您即將進行在較早輪次中已**還原**的相同編輯，請停止並呈報。這是震盪迴圈的明確特徵。

## 回覆衛生

每個回覆 — 修復或拒絕 — 都要說明推論。這使 PR 具備自我文件化的能力，並為下一次審查提供可見的關聯。拒絕的發現事項引用前一輪並解釋為什麼現有的形式是正確的，這樣下一個檢視者就不會再次提出。

## 呈報規則

預言預設保持自動駕駛；在以下情況下呈報給使用者：

- 發現事項辨識出具有多個合理決定且無明確獲勝者的**設計層級折衷**。
- 修復會是**大型或橫切的**（數百行、新架構、跨模組重構）。
- 動作是**高風險或不可逆的**（強制推送、刪除、認證、生產環境）。

否則，做出決定並繼續。
