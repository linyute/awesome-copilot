# 步驟 5：套用修正

子代理程式類型：`general-purpose`，**每個發現一個子代理程式**，
並行**最多 5 個同時執行**；預算：每個 5 分鐘。如果步驟 4 回傳了
超過 5 個 `fix` 列，父代理程式會以每次 ≤5 個的波次執行步驟 5。

## 輸入

每個子代理程式（每次呼叫一個發現）：
- 來自步驟 3 的 `thread_id`、`file`、`line`。
- 發現的 `summary` 和 Copilot 建議的修正（如果有）。
- 來自步驟 4 的分流 `rationale` — 代理程式已經決定這
  是一個 `fix`；此子代理程式僅負責實作它。

## 回傳協定

```
{ thread_id, files_touched, summary, status }
```

其中 `status` ∈ `complete` | `partial` | `blocked`，且 `summary`
為該變更的一行描述。

## 流程

1. **首先，探索正在編輯之區域的存放庫慣例。**
   在撰寫任何程式碼之前，請先閱讀：
   - 其 `applyTo` glob 模式與該檔案路徑相符的
     `.github/instructions/*.md` 檔案，
   - `.github/skills/`，
   - `AGENTS.md`，
   - `CONTRIBUTING.md`，
   - 同一個目錄中的鄰近檔案模式，以及最近修改過
     類似檔案的提交。
2. 遵循這些慣例來套用修正。
3. 回傳 `files_touched` + 一行 `summary` + `status`。

## 注意事項

- **最多 5 個同時執行的修正子代理程式。** 此上限可防止修正向外擴散的
  混亂；父代理程式會在步驟 6 之前合併結果，並協調波次之間的
  檔案衝突。
- **絕不要想出一個與存放庫慣例相牴觸的通用答案。**
  這是一種「學校裡的象」反模式（anti-pattern）——單獨看
  Copilot 的建議似乎正確，但會破壞專案的 lint、格式、拼字檢查、
  授權標頭或框架慣例。
  探索步驟是強制性的，而非選填的。
- **每個子代理程式一個發現。** 需要修改同一個檔案的修正會由
  父代理程式在不同波次之間進行序列化——請勿將
  多個發現合併到單個子代理程式的呼叫中。
- **專案規約優於 Copilot 建議。** 如果探索發現了
  已文件化記載的慣例（拼字檢查許可清單機制、lint
  隱藏樣式等），且該慣例與建議的修正相衝突，請遵循
  該慣例，並將其反映在步驟 8 中草擬的回覆內文
  （請參閱 [04-triage.md](04-triage.md#project-specific-policy-hooks)）。
- **如果實作該修正會針對假設的邊緣情況進行過度設計，請以**
  書面理由予以拒絕 —— 將分流轉為 `decline` 並回傳
  `status: blocked` 及其理由，
  以便步驟 8 草擬正確的回覆。
