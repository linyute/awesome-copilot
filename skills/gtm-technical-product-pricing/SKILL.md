# 技術產品定價


## 初步評估

在提供定價建議之前，請先瞭解：

1. **產品類型**：API / 平台、開發者工具、SaaS 應用程式、基礎架構？
2. **目前定價**：目前的收費標準為何？維持這種方式多久了？
3. **進入市場 (GTM) 動態**：自助服務、銷售輔助、企業級或是混合模式？
4. **成本結構**：每位客戶 / 使用者 / 單位的邊際成本是多少？
5. **競爭格局**：替代方案的成本為何？（包含「什麼都不做」的成本）

---

## 核心框架

### 1. 沒人察覺的漲價（你的定價可能過低了）

**模式：**

一家處於成長階段的平台公司。自發布以來定價從未變更。企業客戶每年支付 1.5 萬美元購買一款能為其節省超過 20 萬美元工程時間的產品。

領導層的辯論：「如果我們漲價，我們會失去客戶。」

**實際發生的情況：**

將企業層級從每年 1.5 萬美元調升至 4.5 萬美元。加入了專屬支援、SSO、稽核日誌 (audit logs) 來支撐漲價的合理性。

流失客戶數：0。一個企業客戶都沒少。

獲得回報：每個企業帳號的營收成長為 3 倍。此外，留下來的客戶開始更認真地看待這款產品 — 採用率更高、內部支持者更多、擴展機會也更多。

**為何會這樣：**

技術出身的創辦人往往將價格與成本掛鉤（「服務他們需要花費 $X，所以我們收 $2X」）。企業買家則將價格與價值掛鉤（「這幫我們省了 20 萬美元，所以 4.5 萬美元很便宜」）。

**定價合理性檢查：**

針對每個客戶細分市場，計算：

```
價值比率 (Value Ratio) = 客戶的替代成本 / 你的價格

如果價值比率 > 10 倍 → 你的定價嚴重偏低
如果價值比率 > 5 倍  → 定價偏低（多數新創公司處於此區間）
如果價值比率 3-5 倍  → 健康的定價
如果價值比率 < 3 倍  → 接近價格天花板
如果價值比率 < 2 倍  → 你的定價昂貴（需要極強的差異化）
```

**如何計算替代成本：**

- 手動流程花費的小時數 × 時薪 × 頻率
- 自建成本（工程師人數 × 月數 × 包含福利的總成本）
- 現有工具的成本 + 切換成本 + 轉換期間的生產力損失
- *不解決問題* 的成本（事件、停機、流失）

**常見錯誤：**

將你的價格與競爭對手比較，而非與客戶的替代成本比較。競爭對手會把你拉入價格戰的無底洞。價值則會將你錨定在客戶實際節省的成本上。

### 2. 三種定價模型（以及各自失效的時機）

**模型 1：依席次計費 (Seat-Based)（每位使用者每月 $X）**

**適用於：**
- 價值隨使用者人數擴展（協作工具、通訊）
- 各使用者之間的使用量相對均勻
- 追求可預測的營收

**失效於：**
- 進階使用者與輕度使用者價格相同（輕度使用者會流失）
- 產品價值不隨席次擴展（一名管理員為 1,000 名使用者進行設定）
- 客戶為了降低成本而合併席次（使用量增加，營收卻沒增加）

**模型 2：依用量計費 (Usage-Based)（每單位 $X）**

**適用於：**
- 各客戶間的使用量差異極大（API 呼叫、運算、儲存）
- 邊際成本具有實質意義（你需要營收能隨用量同步增長）
- 價值與用量直接相關

**失效於：**
- 客戶無法預測帳單（月底看到帳單感到震驚）
- 低用量客戶不值得投入支援成本
- 高用量客戶要求大宗折扣，進而壓縮毛利

**模型 3：依成果計費 (Outcome-Based)（依成果收費 $X）**

**適用於：**
- 你可以可靠地衡量成果（產生的線索數、解決的工單數、部署的程式碼數）
- 成果直接創造客戶價值
- 你對產品的有效性具備信心

**失效於：**
- 成果取決於你無法控制的因素
- 衡量方式存在爭議（「那筆線索不是來自你的工具」）
- 客戶鑽指標漏洞

**通常勝出的混合模式：**

平台費（涵蓋你的固定成本）+ 用量/成果變動費（隨價值擴展）。

範例：每月 $500 基本費 + 每筆交易（或 API 呼叫、完成的任務、處理的記錄 — 任何你的價值單位）$0.05。

為何有效：
- 基本費確保每位客戶都能覆蓋服務成本
- 變動費讓價格與價值對齊
- 客戶能預測最低支出（基本費），同時隨需求自然規模化
- 你能捕捉客戶成長帶來的收益

### 3. 免費版門檻設計（免費在哪裡結束，付費從哪裡開始）

**模式：**

開發者工具最難的定價決策：在哪裡劃分免費與付費的界線？

**框架：尋找正式環境邊界**

永遠不付錢的免費使用者沒關係 — 他們創造了知名度、社群和內容。問題在於當 *正式環境使用者 (production users)* 也不付錢時。

**如何尋找邊界：**

對應你的使用量分佈：

```
使用量等級            |  使用者類型        |  付費意願
─────────────────────────────────────────────────────────────
每月 <100 單位       |  愛好者/學習者    |  $0 (絕不付費)
每月 100-1K 單位     |  個人專案         |  每月 $0-20 (可能)
每月 1K-10K 單位     |  正式環境使用     |  每月 $50-200 (會付費)
每月 >10K 單位       |  業務關鍵任務     |  每月 $200-2K (必須付費)
```

**將你的免費層級限制設定在正式環境使用量開始之前的門檻。** 在此範例中：每月免費 1,000 單位。

原因：愛好者與學習者維持免費（他們是你的行銷引擎）。正式環境使用者會自然達到限制並轉換（他們擁有預算）。

**三種免費轉付費的觸發點：**

**1. 用量限制**（平台類最常用）
- 免費：每月 1,000 單位（API 呼叫、任務、記錄等價值單位）
- 觸發時機：正式環境用量超過限制
- 轉換訊號：使用者正在建構真實的東西

**2. 團隊/協作門檻**（工具類最適用）
- 免費：個人使用
- 觸發時機：使用者邀請第二個人加入
- 轉換訊號：工具價值高到足以分享

**3. 企業功能門檻**（平台類最適用）
- 免費：核心功能
- 觸發時機：需要 SSO、角色存取控制 (RBAC)、稽核日誌、SLA
- 轉換訊號：IT / 安全部門介入（真實的部署）

**常見錯誤：**

將免費層級設定得太高（「我們希望開發者愛我們」）。如果正式環境使用者碰不到限制，他們就永遠不會轉換。免費層級的慷慨應針對 *學習者*，而非 *正式環境使用者*。

### 4. 企業定價（是場對話，而非一個數字）

**模式：**

企業定價不應是你網站上的一個頁面。它是一場對話。「聯絡銷售」按鈕之所以存在，是因為企業交易有獨特的需求 — 而且你應該基於價值而非菜單來定價。

**企業定價變數：**

**1. 部署模型**（自助式雲端、專屬雲端、地端、混合雲）
- 每種模式的服務成本不同 → 價格底線不同
- 地端部署因支援複雜度較高，通常比雲端貴 2-5 倍

**2. 使用規模**（席次、API 量、資料量）
- 大宗折扣絕不應低於服務成本 + 40% 的毛利
- 應針對定價 (list price) 進行折扣，而非針對已折扣價再次折扣

**3. 支援等級**（社群、標準、進階、專屬）
- 進階支援：基本價的 1.5-2 倍
- 專屬客戶成功經理 (CSM)：基本價的 2-3 倍
- 帶有 SLA 的 24/7 支援：基本價的 3-5 倍

**4. 合規需求** (SOC 2, HIPAA, FedRAMP, 資料駐留)
- 每項合規都會增加實際成本（稽核、基礎架構、流程）
- 相應定價：每項合規標準收費為基本價的 1.5-2 倍

**企業定價對話：**

當潛在客戶問「這要多少錢？」時：

1. **不要回答數字。** 用問題回答：「這取決於你們的部署與規模需求。請幫我瞭解你們的需求。」

2. **對應他們的需求：**
   - 有多少使用者 / 席次 / 單位？
   - 雲端還是地端？
   - 合規需求為何？
   - 支援期望為何？
   - 整合需求為何？

3. **在報價前先錨定價值：**
   - 「根據你們所述，你們目前在處理此問題上花費了 [X]。我們的解決方案通常能將該成本降低 [Y%]。」
   - 然後將價格呈現為省下成本的一小部分。

4. **提供 3 個選項：**
   - 好 (Good)：滿足最低需求
   - 更好 (Better)：滿足需求 + 額外好處（錨定在此選項）
   - 最好 (Best)：包含所有項目，甚至包含他們沒要求的
   - 大多數買家會選擇「更好」。那才是你真正的價格。

**常見錯誤：**

在網站上公布企業定價。一旦你公布了數字，那就是天花板 — 談判只會從那裡往下走。請將企業定價維持在對話層面。

### 5. 價格作為定位訊號

**模式：**

你的價格告訴買家你是為誰而做的。這既是營收決策，也是定位決策。

**價格訊號：**

**$0（開源 / 免費層級）：**
- 訊號：我們是為想在購買前嘗試的開發者而做的
- 吸引對象：個人貢獻者、實驗者
- 風險：被認知為「尚未具備企業級水準」

**每月 $20-100：**
- 訊號：我們是為團隊與小型企業而做的
- 吸引對象：自助式買家、新創公司
- 風險：企業不會認真看待你（太便宜了）

**每月 $500-2,000：**
- 訊號：我們是為正式生產工作負載而做的
- 吸引對象：具備實質預算的成長型公司
- 風險：新創公司負擔不起（可能需要免費層級）

**每年 $5,000-50,000：**
- 訊號：我們是為企業而做的
- 吸引對象：中型市場與企業
- 風險：需要銷售團隊（此價格下無法採取自助服務模式）

**每年 $100K 以上：**
- 訊號：我們是關鍵任務基礎架構
- 吸引對象：大型企業
- 風險：銷售週期長、支援期望高

**定位測試：**

如果你定價每月 $50 但想要吸引企業客戶，你的價格正在削弱你的定位。企業買家會將低價與低價值聯想在一起。你可能需要 *漲價* 才能吸引你想要的客戶。

**常見錯誤：**

為你擁有的客戶定價，而非為你想要的客戶定價。如果你的發展路線圖是企業級的，請為企業定價 — 即使這意味著今天會失去一些中小企業 (SMB) 客戶。

---

### 6. 何時以及如何漲價

**時機訊號：**

- 對於大多數客戶而言，價值比率 > 5 倍
- 贏單率超過 40%（你沒有因為價格而丟單）
- 過去 6 個月內沒有客戶對價格表示反彈
- 客戶擴展速度超乎預期
- 競爭對手漲價且市佔率未下降

**如何在不失去客戶的情況下漲價：**

**1. 保留既有客戶權益 (Grandfathering)** (12-24 個月)
- 新價格僅針對新客戶
- 既有客戶收到通知與緩衝期限
- 為潛在客戶創造緊迫感（「價格即將上調」）

**2. 增加價值以證明漲價合理性**
- 推出更高價位的新層級，並具備新功能
- 將現有層級的功能移至新的高階層級
- 這是重新定位，而不僅僅是漲價

**3. 在合約中加入年度調漲條款**
- 在企業合約中加入 5-10% 的年度遞增條款
- 使漲價常態化
- 避免出現「我們 4 年來都付同樣價格」的對話

**溝通方式：**

「我們正在投入 [具體改進]。為了持續這種程度的投資，我們將於 [日期] 更新定價。在 [寬限截止日] 之前，您的目前方案將鎖定在目前的費率。」

絕不要為了漲價而道歉。請將其框架為對他們所喜愛的產品的投資。

## 決策樹

### 應該採用哪種定價模型？

```
不同客戶間的使用量差異是否 >5 倍？
├─ 是 → 依用量計費（或包含用量組件的混合模式）
└─ 否 → 繼續...
    │
    價值是否隨團隊規模擴展？
    ├─ 是 → 依席次計費
    └─ 否 → 繼續...
        │
        你能可靠地衡量客戶成果嗎？
        ├─ 是 → 依成果計費（或混合模式）
        └─ 否 → 平台費 + 基於功能的階層式定價
```

### 應該漲價嗎？

```
大多數客戶的價值比率是否 > 5 倍？
├─ 是 → 漲價
└─ 否 → 繼續...
    │
    贏單率是否 > 40%？
    ├─ 是 → 價格不是問題，但可考慮漲價
    └─ 否 → 繼續...
        │
        你是否明確因為價格而丟失交易？
        ├─ 是 → 不要漲價。提升價值或進行更好的細分。
        └─ 否 → 其他環節出了問題（產品、定位或銷售）
```
