# 評分準則 — 錨點範例

針對分數 **1**、**5** 和 **10** 的各項標準提供錨點範例。使用這些範例來校準評估。對於中間分數請進行內插。

所有量表均為 1–10。有關正式定義和優先順序公式，請參閱主 [SKILL.md](../SKILL.md)。

錨點取自多個領域 — 請選擇最接近你的應對措施性質的行並進行調整。如果你的領域未列出，請類比推理。

---

## 投資報酬率 (ROI)

*1 = 對步驟和價值流的效率提升較低。10 = 效率提升較高。*

| 分數 | 平台 / DevEx | 安全性 | SRE / 運維 | 應用程式開發 / 治理 |
|-------|------------------|----------|-----------|----------------------|
| **1** | 低流量儲存庫的修飾性儲存庫 Metadata 清理（說明、主題）。 | 輪換一個沒有服務實際使用的憑證。 | 重新命名儀表板小工具。 | 重新命名變數以修復沒人看的 Lint 警告。 |
| **5** | 在前 20 個最活躍的儲存庫中加入 CODEOWNERS — 有針對性的審查覆蓋。 | 為單一高權限管理員群組啟用 MFA。 | 為一個第 2 級服務增加 SLO 警報。 | 透過 OpenAPI 記錄一個服務的 API 合約。 |
| **10** | 全組織強制執行 2FA、啟用秘密掃描 + 推送保護，或為先前合併未經測試的團隊交付 CI 管道。 | 全企業強制執行 SSO + MFA；移除常駐管理員權限。 | 自動修復前 2 個導致呼叫器的警報類別。 | 使用自動化的原則即程式碼 (policy-as-code) 取代失敗的手動核准閘門。 |

## 實作成本

*1 = 便宜（幾分鐘的管理時間）。10 = 非常昂貴（多季度的計畫、新授權、專門的人力）。*

| 分數 | 平台 / DevEx | 安全性 | SRE / 運維 | 應用程式開發 / 治理 |
|-------|------------------|----------|-----------|----------------------|
| **1** | 切換組織層級的切換開關 — 單一管理員，< 1 小時，無需購買。 | 在僅警報模式下啟用免費的諮詢掃描器。 | 調整警報閾值。 | 增加 `CONTRIBUTING.md`。 |
| **5** | 在 50–200 個儲存庫中推廣組織級規則集 — 跨團隊協調、一週的工程時間、無需新 SKU。 | 為單一業務單位實作秘密管理員。 | 為中型規模的資產建立集中式記錄。 | 重構由約 10 個服務使用的共享函式庫。 |
| **10** | 在未獲授權的儲存庫上進行全企業 GHAS 推廣 — 購買新席位、採購、安全性團隊所有權、資助數月的培訓。 | 更換身分提供者；零信任計畫。 | 多區域主動/主動架構重構。 | 將單體架構重新平台化為服務；資助團隊進行多季度開發。 |

## 部署易用性

*1 = 極難部署。10 = 非常容易部署。*

| 分數 | 平台 / DevEx | 安全性 | SRE / 運維 | 應用程式開發 / 治理 |
|-------|------------------|----------|-----------|----------------------|
| **1** | 具備自訂 SAML、自我裝載 runner、LFS 的 GHES → GHEC 遷移。深度切換計畫、停機時間視窗。 | 在沒有寬限期的情況下，首次在大型企業中強制執行 SSO。 | 零停機時間的即時資料庫引擎交換。 | 影響數百個下游使用者的破壞性 API 變更。 |
| **5** | 在所有活躍的儲存庫中啟用 Dependabot — 技術上簡單，維運上不簡單（PR 分流能力、擁有者教育）。 | 在中型規模的機群中部署 EDR 代理程式。 | 為現有服務增加分散式追蹤 — 函式庫升級 + 協調。 | 在約 20 個服務中將共享設定移動到特性旗標服務。 |
| **10** | 在單一儲存庫中增加 `CODEOWNERS` 檔案或啟用自動刪除標頭分支 — 一次提交 / 一個勾選框，立即生效，還原簡單。 | 在一個儲存庫上切換秘密掃描警報。 | 在一個服務中調整重試原則。 | 增加一個 README 章節或單個 Lint 規則。 |

## 風險因素

*1 = 對價值流的風險較低。10 = 風險非常高。*

| 分數 | 平台 / DevEx | 安全性 | SRE / 運維 | 應用程式開發 / 治理 |
|-------|------------------|----------|-----------|----------------------|
| **1** | 在**僅警報**模式下啟用秘密掃描。無開發者端阻塞；最壞情況是吵雜的警報。 | 增加被動稽核記錄匯出。 | 增加一個新的儀表板；唯讀。 | 增加選用文件。 |
| **5** | 在預設分支上推廣必要的狀態檢查 — 如果 CI 不穩定可能會阻塞發佈；可透過放寬規則或修復 CI 來復原。 | 對部分管理員強制執行 MFA，並設有寬限期。 | 變更自動縮放閾值 — 可能暫時導致配置過多或不足。 | 引入一個新的必要審查原則。 |
| **10** | 在沒有寬限期的情況下變更 SAML SSO 提供者或在全資產強制執行 SSO。可能導致使用者被鎖定並停止整個價值流的合併。 | 在沒有階段性切換的情況下同時輪換所有生產秘密。 | 進行主要資料儲存庫的切斷式遷移，且無平行運作。 | 移除廣泛使用的公開 API 版本且無棄用期。 |

## 校準提示

- **交叉檢查**：如果 ROI 為 10 但成本也是 10，優先順序公式在易用性高且風險低的情況下仍會給予獎勵。這是刻意設計的 — 讓高槓桿、高投資的工作保持可見。
- **不要僅根據授權價格來設定成本。** 人力資本時間（安全性分流、入職引導、變更管理、審查負擔）主導了大多數推廣。
- **風險 ≠ 成本。** 一個便宜的變更（切換強制執行）如果階段性實施不當，可能會帶來非常高的風險。
- **易用性 ≠ ROI。** ROI 低的一鍵變更排名仍然適中；只有在風險也非常低的情況下，公式才會將其推高。
- **明確估計。** 當你沒有資料或使用者確認的內容時，請參照錨點評分，並在理由中標註 `(估計)`。
- **領域對應。** 如果你的項目不符合任何列，請選擇最接近的類比（影響範圍的大小、可逆性以及工時才是關鍵 — 而非技術）。
