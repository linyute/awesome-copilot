# 在文件中使用陳腔濫調的資料

在更新或編寫工具文件時，**絕不要包含**在提示、本機配置、指令碼、任務檔案或任何其他特定於實作的來源中提供的真實資料。文件必須僅使用通用的、廣為人知的、無法洩漏敏感資訊的佔位符資料。

## 為什麼這很重要

工具的原始程式碼和本機配置通常包含真實名稱、真實電子郵件地址、真實組織詳細資訊和真實網域名稱。這些值對於工具的功能是必要的，但它們在 **公開文件中沒有立足之地**。將真實資料洩漏到文件中可能會暴露：

- 內部企業名稱和聯絡資訊
- 電子郵件地址和網域名稱
- 客戶或消費者識別碼
- 帳戶名稱和憑證
- 揭露私有營運的組織特定術語

## 核心規則

> **如果資料來自提示、本機檔案、指令碼、配置或任務 — 它就不會進入文件中。**
>
> 文件範例僅使用眾所周知的、虛構的或明顯的佔位符資料。

## 什麼算作真實資料

任何源自以下位置的值：

- **本機配置檔案** (例如 `config.json`, `.env`, 帳戶模組)
- **指令碼和任務檔案** (例如批次指令碼、Shell 指令碼、任務執行器)
- **提示上下文** (例如當使用者要求代理程式建置或更新工具時提供的資料)
- **對應或篩選檔案** (例如 JSON 對應、資料擷取規則)
- **Git 忽略的檔案** (例如從版本控制中排除且包含環境特定值的檔案)

## 文件中批准使用的佔位符資料

在所有文件和範例中使用這些通用的、陳腔濫調的替代品：

| 類別 | 批准使用的佔位符範例 |
| --- | --- |
| **人物** | Jane Doe, John Smith, Alice, Bob |
| **電子郵件地址** | `jane.doe@example.com`, `admin@example.org` |
| **組織** | Acme Corp, Contoso, Northwind Traders |
| **網域** | `example.com`, `example.org`, `example.net` |
| **地址** | 123 Main Street, Suite 100, Springfield |
| **電話號碼** | `(555) 123-4567` |
| **帳戶 / 使用者名稱** | `demo-user`, `test-account` |
| **檔案路徑** | `accounts/acme.mjs`, `config/reports.json` |
| **專案名稱** | My Project, Sample App, Demo Tool |

## 將佔位符與上下文相匹配

只有在 **周圍上下文中合理** 時，佔位符才是正確的。一個違反 OS 約定、工具規範或正在描述的工作流程的通用名稱，與真實值一樣具有誤導性。選擇適合平台、工具和該值所扮演角色的替代品。

### 選擇與平台相匹配的路徑

| 作業系統 / 上下文 | 使用 | 避免 |
| --- | --- | --- |
| Windows, 使用者特定資料 | `C:\Users\<user>\AppData\Local\AcmeApp\` | `/home/user/...`, `~/.config/...` |
| Windows, 機器共享資料 | `C:\ProgramData\AcmeApp\` | `C:\Users\<user>\...` |
| Windows, 暫存 | `%TEMP%\acme\` 或 `C:\Users\<user>\AppData\Local\Temp\acme\` | `/tmp/acme/` |
| POSIX, 使用者特定資料 | `~/.config/acme/`, `~/.local/share/acme/` | `C:\Users\<user>\...` |
| POSIX, 暫存 | `/tmp/acme/` | `%TEMP%\acme\` |
| 跨平台範例 | 兩者皆顯示，或使用 `<config-dir>/acme/` | 靜默地選擇其中一個 |

當周圍的文字或程式碼是特定於作業系統的（一個 `.bat` 檔案、一個在 Windows 上執行的 `.jsx`、一個 `bash` 程式片段）時，路徑佔位符必須與該作業系統相匹配。當文件是平台無關時，顯示這兩種形式，或使用清晰抽象的標記（`<install-dir>`, `<config-dir>`）。

### 將範圍與工作流程相匹配

佔位符必須放置在對其所代表資料類型有意義的位置：

| 資料角色 | 合理的佔位符位置 |
| --- | --- |
| 使用者特定日誌和執行階段輸出 | 使用者設定檔資料夾 (`C:\Users\<user>\AppData\Local\<App>\logs\`, `~/.local/state/<app>/`) |
| 使用者特定設定 | 使用者配置資料夾 (`%APPDATA%\<App>\`, `~/.config/<app>/`) |
| 機器範圍共享狀態 | `C:\ProgramData\<App>\`, `/var/lib/<app>/` |
| 專案本機工作檔案 | 儲存庫相對路徑 (`./build/`, `./tmp/`) |
| 產生的輸出產物 | 專案輸出資料夾 (`./dist/`, `./out/`) |

使用者驅動的指令碼寫入除錯日誌不應將該日誌放置在 `C:\ProgramData\…`（機器共享）中；維護共享狀態的服務不應將其放置在 `~/.config/…`（使用者特定）中。選擇真實實作該角色時會選擇的位置。

### 將識別碼與領域相匹配

當範例使用識別碼（帳戶名稱、專案名稱、資料集鍵）時，選擇與周圍領域詞彙一致的佔位符。

- CRM 範例：`acme-corp`, `northwind-traders`。
- 地理資料集範例：`springfield`, `region-west`。
- 開發工具範例：`demo-app`, `sample-project`。

不要混合領域（`acme-corp` 出現在地理資料範例中，即使兩個名稱都是批准使用的通用名稱，讀起來也是錯誤的）。

### 自我檢查

在提交佔位符之前，請詢問：

- 路徑語法是否與同一程式碼區塊中顯示的作業系統相匹配？
- 位置是否與資料的 **角色** 相匹配（使用者 vs. 機器，執行階段 vs. 配置，本機 vs. 共享）？
- 識別碼是否與範例的 **領域** 相匹配？

如果任何答案為否，請將佔位符更換為合適的佔位符。

## 如何應用此規則

### 新增功能時

如果您使用真實帳戶資料（例如，以真實客戶命名的指令碼）新增功能，請使用虛構的帳戶名稱來記錄該功能。

**真實實作檔案**：為特定企業配置的帳戶模組

**文件範例**：

```javascript
// accounts/acme.mjs — 帳戶配置範例
export default {
  name: 'Acme Corp',
  email: 'reports@example.com',
  folder: 'INBOX',
};
```

### 更新配置文件時

如果設定檔參考真實網域、真實路徑或真實憑證，請在將其包含在文件中之前，用佔位符替換每個真實值。

**文件範例**：

```json
{
  "host": "imap.example.com",
  "user": "admin@example.com",
  "folder": "INBOX/Reports",
  "outputDir": "./downloads"
}
```

### 編寫指令碼範例時

如果指令碼自動化了特定組織的任務，文件範例必須使用通用組織名稱和通用參數。

**文件範例**：

```batch
@echo off
REM 範例：為 Acme Corp 執行擷取任務
node extractEmail.mjs --account acme --task download
```

## 程式碼與文件之間的界線

| 上下文 | 是否允許真實資料？ |
| --- | --- |
| 執行階段使用的本機指令碼和設定檔 | 是 |
| 包含環境特定值的 Git 忽略檔案 | 是 |
| 用於建置或配置工具的提示資料 | 是 (僅限程式碼) |
| README.md, docs/ 資料夾和範例範本 | **否 — 僅使用佔位符** |
| CHANGELOG.md 條目 | **否 — 概括描述變更** |
| 已提交原始程式碼檔案中的程式碼註解 | **否 — 保持通用** |

## 一個例外

真實資料中的單字 **僅在** 作為通用英語單字在其普通意義而非範例上下文中使用時，才可以出現在文件中。例如，單字 "development" 在句子 "This tool is under active development" 中是可以接受的，即使它也出現在真實組織名稱中。

## 總結

文件是公開的。實作資料是私有的。將它們分開。每個文件中的每個範例都應通過一個簡單的測試：*陌生人讀到這些資訊後，是否能從中了解關於該工具背後的真實使用者、客戶或組織的任何資訊？* 如果答案是否定的，請用陳腔濫調的佔位符替換資料。
