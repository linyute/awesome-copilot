# QA 工程最佳實踐

為 GitHub Copilot 提供的一套結構化說明，旨在協助進行 QA 工程任務，包括測試設計、自動化以及跨任何技術堆疊的缺陷管理。

---

## 核心測試原則

- **儘早測試，頻繁測試**：左移測試 — 與程式碼一起編寫測試，而不是在之後才編寫。
- **一次測試一件事**：每個測試案例應驗證單一行為或斷言。
- **測試是第一類程式碼**：對測試程式碼應用與生產程式碼相同的可讀性、命名和重構標準。
- **快速失敗**：測試應產生清晰、可操作的失敗訊息，直接指向損壞的行為。
- **確定性測試**：測試在每次執行時必須產生相同的結果。消除隨機性、時間依賴性和共享的可變狀態。
- **獨立測試**：沒有測試應依賴另一個測試的副作用。測試必須能夠以任何順序執行。

## 測試金字塔

遵循測試金字塔以平衡覆蓋率、速度和維護成本：

| 層級 | 範圍 | 數量 | 速度 |
|-------|-------|----------|-------|
| 單元 | 單一函數 / 類別 | 多 (60–70 %) | 毫秒 |
| 整合 | 模組邊界、資料庫、API 合約 | 中 (20–30 %) | 秒 |
| 端對端 | 跨 UI + 後端的完整用戶旅程 | 少 (5–10 %) | 分鐘 |

- 對於業務邏輯和邊緣案例，優先選擇單元測試。
- 使用整合測試來驗證服務與外部依賴項之間的合約。
- 將端對端測試保留給關鍵用戶路徑和冒煙測試套件。

## 測試命名慣例

一致地使用 **Given / When / Then** (GWT) 或 **should_doX_whenY** 模式。

```typescript
// 好 — 描述場景、動作、預期結果
test('should return 404 when product id does not exist')
test('given an expired token, when the user calls /me, then it returns 401')

// 不好 — 模糊，專注於實作
test('test1')
test('check user')
```

- 將相關測試分組在以待測單元命名的 `describe` / `context` 區塊中。
- 對於個別案例，使用 `it` 或 `test`。
- 測試名稱必須能夠作為獨立句子閱讀。

## 斷言最佳實踐

- **每個測試盡可能只有一個邏輯斷言**；避免斷言多個不相關的事物。
- 使用 **特定的比對器** 而非相等性檢查 (`toContain`, `toBeGreaterThan`, `toMatchObject`)。
- 始終斷言 **確切的預期值**，而不僅僅是真值 (`expect(result).toBe(42)` 而不是 `expect(result).toBeTruthy()`)。
- 對於異常測試，同時斷言異常類型和訊息。
- 測試快樂路徑時，優先使用 **正向斷言** 而非負向斷言。

```typescript
// 好
expect(response.status).toBe(200);
expect(response.body.items).toHaveLength(3);

// 避免
expect(response).toBeTruthy();
expect(response.body).not.toBeNull();
```

## 測試資料管理

- 使用 **工廠模式或產生器** 來建立測試資料 — 避免在每個測試中硬編碼原始物件。
- 保持測試資料 **最小化**：僅包含與測試相關的欄位。
- 每個測試執行使用 **唯一識別碼** 以避免在共享環境中發生衝突。
- 絕不在測試中使用生產資料或 PII (個人識別資訊)。
- 在測試之間重置或隔離狀態（記憶體內資料庫、交易回滾、模擬依賴項）。

## 模擬 (Mocking) 和 Stubbing 指導方針

- 在 **邊界** 進行模擬（HTTP 客戶端、資料庫適配器、訊息佇列） — 而不是在業務邏輯深處。
- 對於純函數和簡單的值物件，優先使用 **真實實作**。
- Stub 返回受控資料；Mock 額外驗證互動 — 選擇正確的工具。
- 在測試之間重置所有 Mock 以防止狀態洩漏。
- 如果模擬的原因不明顯，請記錄為什麼要模擬該依賴項。

## API 測試

- 為每個端點驗證 **狀態碼**、**回應結構 (Schema)**、**標頭** 和 **回應時間**。
- 測試端點公開的所有 **HTTP 方法** (GET, POST, PUT, PATCH, DELETE)。
- 涵蓋 **身份驗證和授權** 路徑：有效的 Token、過期的 Token、遺失的 Token、錯誤的角色。
- 測試輸入的 **邊界值**：空字串、null、最大長度、特殊字元、Unicode。
- 驗證 **錯誤回應主體** 是否遵循一致的結構。
- 斷言 PUT 和 DELETE 操作的 **冪等性**。

## UI / 端對端測試

- 目標是 **用戶可見的行為**，而不是實作細節（避免斷言 CSS 類別或內部狀態）。
- 按優先順序使用 **可存取選擇器**：`role` → `label` → `test-id` → `text`。
- 避免使用 `sleep` / 固定等待；使用對元素狀態（可見、啟用、網路閒置）的 **顯式等待**。
- 在 **穩定、隔離的環境** 中執行 E2E 測試（不要使用共享的 Staging 環境）。
- 保持 E2E 場景 **簡短且集中** — 將長流程拆分為更小、可組合的步驟。
- 在失敗時擷取 **螢幕截圖和追蹤** 以便於偵錯。

---

## 效能測試

- 在編寫效能測試之前定義 **SLO** (服務水準目標)：目標延遲 p50/p95/p99、吞吐量、錯誤率。
- 在負載測試中包含 **預熱 (ramp-up)**、**穩定狀態 (steady state)** 和 **降溫 (ramp-down)** 階段。
- 在 **真實資料量** 下進行測試 — 具有空資料庫的合成測試不具代表性。
- 隨時間追蹤結果以檢測 **效能回歸**。
- 區分 **負載測試** (預期流量)、**壓力測試** (超出容量) 和 **疲勞測試** (長時間持續負載)。

## 錯誤報告標準

一個好的錯誤報告包括：

1. **標題**：簡潔、具體 — 包含組件、動作和症狀 (`[結帳] 套用優惠券後訂單總額不正確`)。
2. **環境**：OS、瀏覽器/執行階段版本、部署環境。
3. **重現步驟**：編號、最小化、確定性。
4. **預期結果**：應該發生什麼。
5. **實際結果**：實際發生什麼，包括錯誤訊息和堆疊追蹤。
6. **嚴重性**：嚴重 / 高 / 中 / 低 (由業務影響定義)。
7. **附件**：螢幕截圖、記錄、網路追蹤、測試 ID。

## 測試覆蓋率指導方針

- 目標是 **有意義的覆蓋率**，而不是百分比目標 — 用無意義的測試達到 100% 行覆蓋率是毫無價值的。
- 優先考慮 **關鍵路徑**、**複雜邏輯** 和 **先前有錯誤區域** 的覆蓋率。
- 追蹤 **分支覆蓋率** 和 **變異分數** 以及行覆蓋率。
- 使用覆蓋率報告來尋找未測試的 **邊緣案例**，而不是為了優化指標。

## CI/CD 整合

- 測試必須在合併到 main/trunk 之前在 CI 中通過 — 沒有例外。
- 在每次提交時執行 **快速測試** (單元、Lint)；在 PR 合併或夜間執行 **慢速測試** (整合、E2E)。
- 使測試失敗在 CI 輸出中 **可見且可操作** — 包含測試名稱、失敗原因和相關記錄。
- 將 **測試報告和產物** (JUnit XML, 覆蓋率 HTML, 追蹤) 歸檔為 CI 建置產物。
- 設定 **不穩定測試檢測**：自動重試一次，重複不一致後標記為不穩定。

## 測試審查清單

在批准變更測試的 PR 之前：

- [ ] 新行為已由適當金字塔層級的測試覆蓋。
- [ ] 測試命名清晰並遵循專案慣例。
- [ ] 沒有 `sleep`、`Thread.Sleep` 或任意逾時。
- [ ] Mock 在每個測試後重置。
- [ ] 沒有硬編碼的環境特定值（URL、憑證）。
- [ ] 測試是獨立的，可以隔離執行。
- [ ] 無需閱讀實作即可閱讀測試程式碼。
