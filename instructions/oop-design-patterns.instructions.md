# 物件導向程式設計設計模式與簡潔程式碼

這些指令設定 GitHub Copilot 在產生或重構程式碼時，優先考慮 Gang of Four (GoF) 設計模式、SOLID 原則和簡潔的物件導向程式設計 (OOP) 實務。

## 核心架構哲學

- **針對介面編寫程式，而非針對實現：** 始終偏好抽象類別或介面，而非具體的實現。使用相依性注入來提供具體實例。
- **偏好物件組合勝過類別繼承：** 使用組合在執行期間動態組合行為。避免深層的繼承樹。在適當的情況下使用委派 (Delegation) 以重複使用行為，且不破壞封裝。
- **封裝變動的部分：** 識別應用程式中變動的部分，並將其與不變的部分分離。使用 Strategy、State 或 Bridge 等模式來隔離這些變動。
- **鬆散耦合：** 最小化類別之間的直接相依性。使用 Mediator、Observer 或抽象工廠來保持元件解耦。

## 生成模式指南 (Creational Patterns Guidelines)

當產生涉及物件建立或實例化的程式碼時，套用這些模式將系統與其物件的建立方式解耦：

- **抽象工廠 (Abstract Factory)：** 當系統必須配置多系列相關產品 (例如跨平台 UI 元件) 中的其中一員時使用。確保用戶端僅與抽象工廠和抽象產品介面互動。
- **工廠方法 (Factory Method)：** 當類別無法預見其必須建立的物件類別時使用。將實例化延遲到子類別。
- **產生器 (Builder)：** 當建構一個複雜物件需要逐步流程時使用，特別是當相同的建構流程可以產生不同的表示形式時。
- **單例模式 (Singleton)：** *僅在* 絕對必要且需要保證類別只有一個實例並提供全域存取點時使用 (例如中央設定管理員或硬體介面)。盡可能偏好相依性注入而非嚴格的單例模式。
- **原型模式 (Prototype)：** 用於避免建立工廠的類別階層，或當從頭建立一個物件比複製現有物件更昂貴時。

## 結構模式指南 (Structural Patterns Guidelines)

當產生定義類別和物件如何組合以形成更大結構的程式碼時，套用以下模式：

- **配接器 (Adapter)：** 用於使不相容的介面協同運作。為了更大的靈活性，偏好物件配接器 (使用組合) 而非類別配接器 (使用多重繼承)。
- **橋接器 (Bridge)：** 用於將抽象與其實現分離，使兩者可以獨立變動 (例如將高階 `Window` 概念與平台特定的 `WindowImpl` 邏輯分離)。
- **組合 (Composite)：** 用於表示部分與整體的階層結構。確保用戶端可以透過共同的 `Component` 介面，統一處理個別物件和物件組合。
- **裝飾器 (Decorator)：** 用於動態地為物件附加額外的職責。偏好此模式勝過透過子類別擴展功能，以防止類別爆炸。確保裝飾器具有與其裝飾元件完全相同的介面。
- **外觀 (Facade)：** 用於為複雜子系統提供簡單、統一的介面。
- **享元 (Flyweight)：** 透過與相似物件盡可能共享，最小化記憶體使用量或計算開銷。
- **代理 (Proxy)：** 用於為另一個物件提供代理或佔位符，以控制對其存取 (例如延遲載入、存取控制或遠端通訊)。

## 行為模式指南 (Behavioral Patterns Guidelines)

當產生涉及演算法、控制流或物件間通訊的程式碼時，套用這些模式：

- **策略 (Strategy)：** 定義一系列演算法，並將其各自封裝，使其可交換。透過委派給 Strategy 物件，消除選擇行為的複雜條件邏輯 (`switch`/`if-else`)。
- **觀察者 (Observer)：** 定義一對多的相依性，當一個物件 (Subject) 變更時，自動通知並更新其他物件 (Observers)。保持主題與觀察者鬆散耦合。
- **命令 (Command)：** 將請求封裝成物件。這對於實現復原/重做功能、佇列或記錄請求至關重要。
- **狀態 (State)：** 當物件的行為高度依賴其內部狀態且必須在執行期間變更行為時使用。將每個狀態表示為一個單獨的類別。
- **範本方法 (Template Method)：** 在基底類別中定義演算法骨架，將特定步驟延遲到子類別，而不變更演算法結構。
- **責任鏈 (Chain of Responsibility)：** 將請求沿著潛在處理程序的鏈條傳遞，直到一個處理程序處理它，避免發送者與特定接收者耦合。
- **中介者 (Mediator)：** 集中處理一組物件之間的複雜通訊與控制邏輯，防止它們明確地相互參照。
- **迭代器 (Iterator)：** 為循序存取聚合物件元素提供標準方法，而不暴露其底層表示。
- **造訪者 (Visitor)：** 在不變更元素類別的情況下，為物件結構定義新作業。這對於在穩定的組合結構 (如抽象語法樹) 上執行不同的分析非常有效。
- **備忘錄 (Memento)：** 在不違反封裝的情況下擷取並外部化物件內部狀態，以便隨後復原 (對複雜的復原機制很有用)。

## Copilot 的程式碼產生規則

- **模式識別：** 當提示解決映射到 GoF 模式的問題 (例如「我需要復原此操作的方法」、「我有幾種計算稅收的方法」) 時，請在註解中明確提及您套用的模式。
- **介面優先：** 在產生具體實現之前，先產生介面或抽象基底類別。
- **不可變性與封裝：** 欄位預設為 `private`。僅在必要時提供 Getter/Setter。偏好不可變物件。
- **命名規範：** 在類別名稱中使用模式名稱，以助於理解 (例如 `TaxCalculationStrategy`, `ButtonDecorator`, `WidgetFactory`)，但在適當時保持自然。
- **避免萬能類別 (God Class)：** 將大型、複雜類別拆分為更小、更專注的類別，透過 Mediator 協調或由更小的 Strategy 物件組成。
- **單一職責原則 (Single Responsibility Principle)：** 確保每個類別只有一個變更的原因。如果類別負責太多，請將其重構為多個類別。
- **開閉原則 (Open/Closed Principle)：** 設計類別對擴展開放，但對修改封閉。使用抽象類別或介面來允許新行為，而無需更改現有程式碼。
- **里氏替換原則 (Liskov Substitution Principle)：** 確保子類別可以替換其基底類別，而不改變程式的正確性。避免違反此原則，確保衍生類別不加強前提條件或削弱後置條件。
- **介面隔離原則 (Interface Segregation Principle)：** 偏好許多特定介面而非單一通用介面。用戶端不應被迫依賴其不使用的介面。
- **相依反向原則 (Dependency Inversion Principle)：** 依賴抽象而非具體。高階模組不應依賴低階模組；兩者皆應依賴抽象。
- **明智地使用設計模式：** 僅當模式解決程式碼庫中的實際問題時才套用。僅當模式在可維護性、靈活性或可讀性方面具有明確優勢時才套用，以避免過度設計。
- **記錄意圖：** 使用設計模式時，加入註解說明為何選擇該模式以及如何套用。這有助於後續維護者瞭解設計決策的依據。
- **可測試性：** 確保產生的程式碼是可測試的。使用有利於單元測試的模式 (例如透過相依性注入以便進行模擬)。編寫測試以驗證使用中模式的行為。
- **迭代重構：** 重構現有程式碼以套用設計模式時，請迭代進行。從改善設計的小型變更開始，且不引入 Bug。使用測試驗證重構期間行為保持正確。
- **效能考量：** 注意設計模式對效能的影響。某些模式可能會增加額外的抽象層。使用剖析工具識別瓶頸，並在不犧牲可維護性的情況下根據需要進行最佳化。
- **一致性：** 在程式碼庫中一致地套用設計模式。如果模式在一個部分中使用，請考慮在其他類似情況下使用，以保持一致的設計語言。
- **檢閱與迭代：** 定期檢閱程式碼庫，尋找套用設計模式或重構現有程式碼以更符合 OOP 原則的機會。鼓勵注重設計品質和遵循指南的程式碼審查。
- **保持更新：** 跟上 OOP 設計模式和最佳實務的最新發展。持續學習並調整編碼風格，以納入可改善品質的新見解和技術。
- **平衡簡潔性與靈活性：** 雖然設計模式可以提供強大的解決方案，但也可能增加複雜性。努力在簡潔與靈活性之間取得平衡，確保程式碼易於理解和維護，同時仍能適應未來變更。當問題可以透過簡單的函式解決時，偏好定義函式而非類別；僅在類別和模式提供明確的組織優勢時才使用。
- **使用儲存庫和型別定義：** 當產生涉及複雜資料結構或互動的程式碼時，考慮使用儲存庫 (Repository) 抽象化資料存取，並使用型別定義以確保型別安全和清晰度。這有助於保持職責分離並改善整體的可維護性。

## 記錄與錯誤處理

- 套用設計模式時，請確保適當地整合記錄和錯誤處理。
- 失敗安全、高聲呼叫、清晰且及時。
- 避免無聲失敗，並確保記錄錯誤時具有足夠的上下文，以便於除錯和維護。
- 在適當的情況下使用自訂例外狀況，以提供更有意義的錯誤訊息，並允許在用戶端程式碼中進行更精確的錯誤處理。
- 明智地使用例外狀況區塊，確保其用於處理預期的錯誤情況，而非控制正常的程式流程。
- 使用記錄框架來管理記錄層級和輸出，以便更好地控制不同環境 (例如開發 vs. 生產) 中的記錄行為。
- 在每個類別和函式中適當使用 info、debug、warning、error 和 critical 記錄層級，以提供對應用程式行為和潛在問題的清晰見解。考慮實施集中式錯誤處理機制 (例如全域例外處理常式)，以確保整個應用程式的一致錯誤回應和記錄。

## 文件

- 套用設計模式時，請確保程式碼有良好的文件說明。
- 使用以英文編寫的 Docstring 說明類別和方法的目的，並加入澄清複雜邏輯或設計決策的註解。除非現有程式碼中使用其他模式，否則在 Docstring 中使用 NumPy 模式記錄參數和傳回值。在此指令的第一次使用時詢問開發人員其偏好的 Docstring 參數和傳回值類型，在此覆蓋 NumPy Docstring 定義，並在以後的所有程式設計任務中使用此類型的 Docstring。這有助於其他開發人員瞭解程式碼背後的意圖以及如何有效地使用它。
- 考慮使用 Sphinx 或 JSDoc 之類的工具從程式碼庫產生文件，使開發人員更容易導覽和瞭解可用的類別、方法及其預期用法。
- 此外，在 README 或專用文件中保持高階架構概觀，解釋不同元件和模式在整體系統架構中如何配合運作。
- 將您的文件分為使用者文件 (如何使用程式碼) 和開發人員文件 (程式碼如何運作以及如何維護)。確保隨著程式碼的演進，這兩種類型的文件都保持更新。
- 在適當的情況下使用圖表 (例如 UML) 視覺化表示類別和模式之間的關係，幫助可能剛接觸 OOP 設計模式的開發人員理解。
- 在開發團隊中鼓勵文件文化，強調其對維持簡潔且可維護程式碼庫的重要性。
- 切勿透過不斷建立包含相同內容的新文件來過度擴張文件。
- 掃描現有的文件檔案以擴充它們，或以現有的相同風格建立所需的新文件。保持簡潔、清晰且專注於程式碼及其設計模式的最重要方面。
- 避免冗長或過度詳細的文件，這可能會淹沒開發人員，並掩蓋他們有效瞭解程式碼庫所需的關鍵資訊。
