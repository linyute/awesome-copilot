# Markdown 協助工具檢閱指引

在檢閱 Markdown 檔案時，請根據 GitHub 的 [「讓您的 GitHub 個人檔案頁面具備協助工具支援的 5 個提示」](https://github.blog/developer-skills/github/5-tips-for-making-your-github-profile-page-accessible/)，檢查是否存在以下協助工具 (accessibility) 問題。標記違規項目並建議修復方案，同時清楚解釋對協助工具支援的影響。

## 1. 描述性連結

- 標記通用的連結文字，例如「點擊此處」、「這裡」、「這個」、「閱讀更多」或「連結」。
- 連結文字必須在脫離上下文時仍具意義，因為輔助技術可以將連結呈現為獨立的清單。
- 標記同一頁面上文字相同但指向不同目的地的多個連結。
- 散文中的原始 URL 應轉換為具描述性的連結。

錯誤：`點擊 [這裡](https://example.com) 閱讀我的部落格文章`
正確：`閱讀我的部落格文章「[撰寫具協助工具支援的履歷](https://example.com)」`

## 2. 圖片替代文字 (Alt Text)

- 標記替代文字為空的圖片 (例如 `![](path/to/image.png)`)，除非它們被明確標記為裝飾性。
- 標記替代文字為檔案名稱 (例如 `img_1234.jpg`) 或通用預留位置 (例如 `螢幕截圖`、`圖片`) 的情況。
- 替代文字應簡潔且具描述性。包含圖片中可見的任何文字。
- 適當時使用「螢幕截圖」，但不要加上「圖片」前綴，因為螢幕閱讀器會自動播報。
- 對於複雜圖片 (圖表、資訊圖表)，建議在替代文字中總結資料，並透過 `<details>` 標籤或連結內容提供較長的說明。
- 在建議改善替代文字時，請將其作為供作者檢閱的建議。替代文字需要對視覺內容和上下文有深入理解，這只有作者才能正確評估。

## 3. 標題階層

- 每個文件僅限一個 H1 (`#`)，用作頁面標題。註：在從 Front Matter 自動產生 H1 的專案中，內容應從 H2 開始。
- 標題必須遵循邏輯階層，且絕不跳過等級 (例如：`##` 後接 `####` 是違規行為)。
- 標記將粗體文字 (`**文字**`) 用作標題視覺替代方案的情況。
- 正確的標題結構允許輔助技術使用者按章節導覽，並幫助視力正常的使用者掃描內容。

## 4. 簡易語言 (Plain Language)

- 標記可以簡化的不必要複雜語言或專業術語。
- 偏好短句、常用詞彙和主動語態。
- 標記可以拆分為較小區段或清單的長且密集的段落。
- 描述使用者介面導覽時，先以簡單明確的自然語言逐步寫出操作（例如：「開啟 設定，然後選擇 偏好設定」）。使用通用且穩定的標籤，而非圖示名稱或視覺描述。
- 可在括號中加入視覺參考作為補充說明（例如：「（齒輪圖示 > 偏好設定）」），但切勿僅以視覺麵包屑或圖示名稱來描述導覽路徑。
- 在建議改善簡易語言時，請將其作為供作者檢閱的建議。語言決策需要理解受眾、上下文和語氣。

## 5. 清單與表情符號的使用

### 清單

- 標記使用表情符號或特殊字元而非正確 Markdown 清單語法 (`-`, `*`, `+`, 或 `1.`) 作為項目符號的情況。
- 標記純文字中應結構化為正確清單的順序項目。
- 正確的清單標記允許螢幕閱讀器播報清單內容 (例如：「第 1 個項目，共 3 個」)。

### 表情符號

- 標記連續多個表情符號的情況，這會對螢幕閱讀器使用者造成干擾，因為每個表情符號名稱都會被完整朗讀 (例如：「火箭」、「火花」、「火」)。
- 標記用於傳達意義但未同時以文字說明的表情符號。
- 表情符號應節制且深思熟慮地使用。

## 檢閱優先順序

當存在多個問題時，請按以下順序排定優先權：

1. 圖片缺失或替代文字為空
2. 跳過標題等級或標題階層問題
3. 非描述性連結文字
4. 將表情符號用作項目符號或清單標記
5. 簡易語言改進

## 檢閱語氣

- 解釋每個問題對協助工具支援的影響，具體說明哪些使用者受到影響 (例如：螢幕閱讀器使用者、認知障礙人士、非母語人士)。
- 不要從寫作中移除個性或語氣。協助工具支援與引人入勝的內容並非互斥。
- 保持建議具備可操作性且具體。
