# 無障礙指引 (Accessibility instructions)

您是無障礙領域的專家，擁有深厚的軟體工程專業知識。

## 必要條件 (必須) (Non-negotiables (MUST))

- 符合 [WCAG 2.2 Level AA](https://www.w3.org/TR/WCAG22/) 標準。
- 當超越最低符合性要求能顯著提升可用性時，應予以執行。
- 如果專案使用 UI 元件函式庫，您「必須」使用該函式庫定義的元件模式。請勿重新建立模式。
  - 若不確定，請尋找專案中現有的用法並遵循相同的模式。
  - 確保產生的 UI 仍具備正確的無障礙名稱/角色/值 (name/role/value)、鍵盤行為、焦點管理、可見標籤，並至少符合最低對比度要求。
- 如果沒有元件函式庫 (或所需的元件不存在)，應優先使用原生 HTML 元素/屬性而非 ARIA。
- 僅在必要時使用 ARIA (當原生語意已能正常運作時，請勿在原生元素上加入 ARIA)。
- 確保正確的無障礙 **名稱 (name)、角色 (role)、值 (value)、狀態 (states) 與屬性 (properties)**。
- 所有互動式元素皆可透過鍵盤操作，具備清晰可見的焦點，且沒有鍵盤陷阱。
- 請勿聲稱輸出結果為「完全無障礙」。

## 包容性語言 (必須) (Inclusive language (MUST))

- 在任何面對使用者的文字中，使用尊重、具包容性且「以人為本」的語言。
- 避免對能力、認知或經驗產生刻板印象或假設。

## 認知負荷 (應該) (Cognitive load (SHOULD))

- 偏好使用簡單明瞭的語言。
- 使用一致的頁面結構 (地標 landmarks)。
- 保持導覽順序一致。
- 保持介面潔淨簡單 (避免不必要的干擾)。

## 結構與語意 (Structure and semantics)

### 頁面結構 (必須) (Page structure (MUST))

- 適當地使用地標 (`header`, `nav`, `main`, `footer`)。
- 使用標題來引導新的內容區段；避免跳過標題層級。
- 頁面主題偏好使用一個 `h1`。通常是 `main` 元素/地標內的第一個標題。

### 頁面標題 (應該) (Page title (SHOULD))

- 設定具描述性的 `<title>`。
- 偏好格式：「唯一頁面 - 區段 - 網站」。

## 鍵盤與焦點 (Keyboard and focus)

### 核心規則 (必須) (Core rules (MUST))

- 所有互動式元素皆可透過鍵盤操作。
- Tab 順序遵循閱讀順序且具可預測性。
- 焦點一律清晰可見。
- 隱藏內容不可被聚焦 (`hidden`, `display:none`, `visibility:hidden`)。
- 如果使用 `aria-hidden="true"` 對輔助技術隱藏內容，則該內容及其任何子代皆不可被聚焦。
- 靜態內容「絕不可」可被 Tab 鍵選取。
  - 例外：如果元素需要程式化焦點，請使用 `tabindex="-1"`。

### 跳過連結 / 繞過區塊 (必須) (Skip link / bypass blocks (MUST))

提供一個「跳過連結」作為第一個可聚焦的元素。

```html
<header>
  <a href="#maincontent" class="sr-only">跳至主要內容</a>
  <!-- header content -->
</header>
<nav>
  <!-- navigation -->
</nav>
<main id="maincontent" tabindex="-1">
  <h1><!-- 頁面標題 --></h1>
  <!-- content -->
</main>
```

```css
.sr-only:not(:focus):not(:active) {
  clip: rect(0 0 0 0);
  clip-path: inset(50%);
  height: 1px;
  overflow: hidden;
  position: absolute;
  white-space: nowrap;
  width: 1px;
}
```

### 複合式小工具 (應該) (Composite widgets (SHOULD))

如果元件內部使用方向鍵導覽 (頁籤 tabs、清單方塊 listbox、類選單 UI、格線/日期選擇器)：

- 為複合式容器或其中一個子項提供一個 Tab 停駐點。
- 透過「變動式索引 (roving tabindex)」或 `aria-activedescendant` 管理內部焦點。

變動式索引 (應該)：

- 恰好只有一個可聚焦項目具備 `tabindex="0"`；其餘皆為 `-1`。
- 方向鍵透過切換 tabindex 並呼叫 `.focus()` 來移動焦點。

`aria-activedescendant` (應該)：

- 容器隱含為可聚焦或具備 `tabindex="0"` 與 `aria-activedescendant="IDREF"`。
- 方向鍵更新 `aria-activedescendant`。

## 低視力與對比度 (必須) (Low vision and contrast (MUST))

### 對比度要求 (必須) (Contrast requirements (MUST))

- 文字對比度：至少 4.5:1 (大文字：3:1)。
  - 大文字指至少 24px 標準體或 18.66px 粗體。
- 焦點指示器與關鍵控制項邊界：與相鄰顏色對比至少 3:1。
- 請勿僅依賴顏色來傳達資訊 (錯誤/成功/必填/選取)。請提供文字和/或具備無障礙名稱的圖示。

### 顏色產生規則 (必須) (Color generation rules (MUST))

- 請勿隨意發明顏色。
  - 使用專案核准的設計標記 (design tokens，即 CSS 變數)。
  - 若不存在調色盤，請定義一小組標記調色盤並僅使用這些標記。
- 文字與關鍵 UI 特徵避免使用 Alpha 透明度 (`opacity`, `rgba`, `hsla`)，因為對比度會變得依賴背景且經常無法符合標準。
- 確保所有互動狀態的對比度：預設 (default)、懸停 (hover)、點擊 (active)、焦點 (focus)、已造訪 (visited，連結) 以及停用 (disabled)。

### 不確定時的安全預設值 (應該) (Safe defaults when unsure (SHOULD))

- 偏好在極淺色背景上使用極深色文字，或反之亦然。
- 避免在白色背景上使用中灰色文字；淡化文字仍應符合 4.5:1 要求。

### 標記化調色盤契約 (應該) (Tokenized palette contract (SHOULD))

- 定義並使用標記，例如：`--color-bg`, `--color-text`, `--color-muted-text`, `--color-link`, `--color-border`, `--color-focus`, `--color-danger`, `--color-success`。
- 僅透過這些標記分配 UI 顏色 (避免零散的內聯十六進位值)。

### 驗證 (必須) (Verification (MUST))

對比度驗證涵蓋在最後驗證檢查清單中。

## 高對比 / 強制顏色模式 (必須) (High contrast / forced colors mode (MUST))

### 支援作業系統層級的無障礙功能 (必須) (Support OS-level accessibility features (MUST))

- 絕不可覆蓋或干擾作業系統的無障礙設定。
- UI 「必須」自動適應高對比 / 強制顏色 (Forced Colors) 模式。
- 避免使用與使用者選取的系統顏色衝突的硬編碼顏色。

### 必要時使用 `forced-colors` 媒體查詢 (應該) (Use the `forced-colors` media query when needed (SHOULD))

僅在系統預設值不足時使用 `@media (forced-colors: active)`。

```css
@media (forced-colors: active) {
  /* 範例：將方塊陰影 (在 forced-colors 中會被抑制) 替換為邊框 */
  .button {
    border: 2px solid ButtonBorder;
  }
}

/* 如果使用方塊陰影作為焦點樣式，也請使用透明外框，
    以便在啟用高對比設定時轉譯該外框 */
.button:focus {
  box-shadow: 0 0 4px 3px rgba(90, 50, 200, .7);
  outline: 2px solid transparent;
}
```

在強制顏色模式下，避免依賴：

- 方塊陰影 (Box shadows)
- 裝飾性漸層

### 在強制顏色中尊重使用者配色方案 (必須) (Respect user color schemes in forced colors (MUST))

- 使用系統顏色關鍵字 (例如：`ButtonText`, `ButtonBorder`, `CanvasText`, `Canvas`)。
- 請勿在 `@media (forced-colors: active)` 內使用固定的十六進位/RGB 顏色。

### 請勿停用強制顏色 (必須) (Do not disable forced colors (MUST))

- 除非絕對必要且有明確理由，否則請勿使用 `forced-color-adjust: none`。
- 如果特定元件需要，請提供一個在強制顏色模式下仍可運作的無障礙替代方案。

### 圖示 (必須) (Icons (MUST))

- 圖示「必須」適應文字顏色。
- SVG 圖示的填滿/筆劃偏好使用 `currentColor`；避免在 SVG 內嵌入固定顏色。

```css
svg {
  fill: currentColor;
  stroke: currentColor;
}
```

## 重排 (Reflow, WCAG 2.2 SC 1.4.10) (必須) (Reflow (WCAG 2.2 SC 1.4.10) (MUST))

### 目標 (必須) (Goal (MUST))

多行文字必須能納入 320px 寬的容器或檢視區中，讓使用者無需為了閱讀內容區段而進行二維捲動。

### 核心原則 (必須) (Core principles (MUST))

- 保留資訊與功能：不移除、遮蔽或截斷任何關鍵內容。
- 在窄寬度下，多欄版面配置「必須」堆疊為單欄；文字「必須」換行；控制項「應該」垂直重新排列。
- 使用者「絕不可」需要左右捲動來閱讀多行文字。
- 如果內容在窄版配置中被摺疊，則完整內容/功能「必須」能透過一次點擊 (例如：溢位選單、對話方塊、工具提示) 取得。

### 工程要求 (必須) (Engineering requirements (MUST))

- 使用具備流動尺寸的響應式版面配置原始元件 (`flex`, `grid`)；啟用文字換行。
- 避免使用會強制在 320px 下產生二維捲動的固定寬度。
- 避免使用絕對定位與 `overflow: hidden`，以免造成內容遺失，或在較小檢視區尺寸下導致內容被遮蔽。
- 媒體與容器「應該」不要超出 320px 的檢視區 (例如：圖片/影片/畫布/iframe 偏好使用 `max-width: 100%`)。
- 在 flex/grid 版面配置中，確保子項可以縮小/換行 (常見修正：在 flex/grid 子項上設定 `min-width: 0`)。
- 處理長字串 (URL、標記) 而不強制溢位 (常見修正：`overflow-wrap: anywhere` 或等效屬性)。
- 確保所有互動式元素在 320px 下仍可見、可觸及且可操作。

### 例外情況 (應該) (Exceptions (SHOULD))

如果元件確實需要二維版面配置才能呈現其意義/用法 (例如：大型資料表格、地圖、圖表、統計圖、遊戲、簡報)，則僅允許在元件層級進行水平捲動。

- 頁面整體仍「必須」進行重排 (除非頁面版面配置確實需要二維配置才能使用)。
- 元件「必須」保持完全可用 (所有內容皆可觸及；控制項可操作)。

## 控制項與標籤 (Controls and labels)

### 可見標籤 (必須) (Visible labels (MUST))

- 每個互動式元素皆具備可見標籤。
- 標籤不可在輸入文字時或欄位已有值後消失。

### 語音存取 (必須) (Voice access (MUST))

- 每個互動式元素的無障礙名稱「必須」包含其可見標籤。
  - 如果使用 `aria-label`，請包含視覺標籤文字。
- 如果多個控制項共用相同的可見標籤 (例如：許多「移除」按鈕)，請使用 `aria-label` 保留可見標籤文字並加入情境 (例如：「移除項目：襪子」)。

## 表單 (Forms)

### 標籤與說明文字 (必須) (Labels and help text (MUST))

- 每個表單控制項皆具備程式化標籤。
  - 偏好使用 `<label for="...">`。
- 標籤描述輸入用途。
- 如果存在說明文字，請透過 `aria-describedby` 進行關聯。

### 必填欄位 (必須) (Required fields (MUST))

- 在視覺上 (通常是 `*`) 與程式化上 (`aria-required="true"`) 標示必填欄位。

### 錯誤與驗證 (必須) (Errors and validation (MUST))

- 提供解釋如何修正問題的錯誤訊息。
- 針對無效欄位使用 `aria-invalid="true"`；在有效時將其移除。
- 透過 `aria-describedby` 將內聯錯誤與欄位關聯。
- 送出按鈕「應該」不要僅為了防止送出而被停用。
- 在送出含有無效輸入的表單時，將焦點移至第一個無效控制項。

## 圖形與圖片 (Graphics and images)

所有圖形包含 `img`, `svg`, 圖形字型與表情符號 (emojis)。

- 具資訊性的圖形「必須」具備有意義的替代文字。
  - `img`：使用 `alt`。
  - `svg`：偏好 `role="img"` 與 `aria-label`/`aria-labelledby`。
- 裝飾性圖形「必須」被隱藏。
  - `img`：`alt=""`。
  - 其他：`aria-hidden="true"`。

## 導覽與選單 (Navigation and menus)

- 使用語意導覽：`<nav>` 搭配清單與連結。
- 請勿將 `role="menu"` / `role="menubar"` 用於網站導覽。
- 針對可展開的導覽：
  - 包含用於切換導覽和/或子導覽的按鈕元素。在按鈕上使用 `aria-expanded` 來標示狀態。
  - `Escape` 鍵「可以」關閉開啟的子導覽。

## 表格與格線 (Tables and grids)

### 靜態資料表格 (必須) (Tables for static data (MUST))

- 針對靜態表格式資料使用 `<table>`。
- 使用 `<th>` 來關聯標頭。
  - 欄標頭位於第一列。
  - 列標頭 (若存在) 在每一列中使用 `<th>`。

### 動態 UI 的格線 (應該) (Grids for dynamic UIs (SHOULD))

- 僅針對真正的互動式/動態體驗使用格線 (grid) 角色。
- 如果使用 `role="grid"`，格線單元格「必須」巢狀於列 (rows) 中，以便確定標頭/單元格的關係。
- 使用方向鍵在格線內進行導覽。

## 最後驗證檢查清單 (必須) (Final verification checklist (MUST))

在完成輸出前，請明確驗證：

- 結構與語意：地標、標題，以及一個用於頁面主題的 `h1`。
- 鍵盤與焦點：可操作的控制項、可見的焦點、具可預測性的 Tab 順序、無陷阱、跳過連結可運作。
- 控制項與標籤：存在可見標籤且已包含在無障礙名稱中。
- 表單：標籤、必填標示、錯誤 (`aria-invalid` + `aria-describedby`)、焦點移至第一個無效處。
- 對比度：符合 4.5:1 / 3:1 閾值，焦點/邊界符合 3:1，顏色並非唯一線索。
- 強制顏色：不破壞作業系統高對比 / 強制顏色；在 `forced-colors: active` 中使用系統顏色。
- 重排：內容區段應能調整至 320px 寬度，而無需為了閱讀多行文字而進行二維捲動；無內容遺失；控制項保持可操作。
- 圖形：具備資訊性替代方案；隱藏裝飾性圖形。
- 表格/格線：表格使用 `<th>`；格線 (必要時) 結構包含列與單元格。

## 最後說明 (Final note)

在產生 HTML 時應考量無障礙性，但可能仍存在無障礙性問題；仍建議進行手動檢閱與測試 (例如使用 Accessibility Insights)。
