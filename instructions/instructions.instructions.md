# 自訂指令檔案指南

建立有效且可維護之自訂指令檔案的說明，引導 GitHub Copilot 生成領域特定程式碼並遵循專案慣例。

## 專案 Context

- 目標受眾：開發人員與使用領域特定程式碼的 GitHub Copilot
- 檔案格式：包含 YAML 前言 (frontmatter) 的 Markdown
- 檔案命名慣例：小寫並以連字號連接 (例如 `react-best-practices.instructions.md`)
- 位置：`.github/instructions/` 目錄
- 目的：為程式碼生成、審查與文件編寫提供具備 Context 的指引

## 必要的 YAML 前言 (Frontmatter)

每個指令檔案皆須包含 YAML 前言，並包含下列欄位：

```yaml
---
description: '簡述指令目的與範圍'
applyTo: '目標檔案的 glob 模式 (例如 **/*.ts, **/*.py)'
---
```

### 前言指南

- **description**：以單引號括起的字串，1-500 字元，清楚說明目的
- **applyTo**：指定這些指令適用於哪些檔案的 glob 模式
  - 單一模式：`'**/*.ts'`
  - 多重模式：`'**/*.ts, **/*.tsx, **/*.js'`
  - 特定檔案：`'src/**/*.py'`
  - 所有檔案：`'**'`

## 檔案結構

結構良好的指令檔案應包含下列區段：

### 1. 標題與概述

- 使用 `#` 標題的清晰描述性標題
- 簡短介紹說明目的與範圍
- 選填：包含關鍵技術與版本的專案 Context 區段

### 2. 核心區段

根據領域將內容組織為邏輯區段：

- **一般指令 (General Instructions)**：高階指南與原則
- **最佳實務 (Best Practices)**：推薦的模式與方法
- **程式碼標準 (Code Standards)**：命名慣例、格式化、樣式規則
- **架構/結構 (Architecture/Structure)**：專案組織與設計模式
- **常見模式 (Common Patterns)**：頻繁使用的實作
- **安全性 (Security)**：安全性考量 (若適用)
- **效能 (Performance)**：優化指南 (若適用)
- **測試 (Testing)**：測試標準與方法 (若適用)

### 3. 範例與程式碼片段

提供具備清晰標籤的具體範例：

```markdown
### 好的範例
\`\`\`language
// 建議方法
程式碼範例
\`\`\`

### 壞的範例
\`\`\`language
// 避免此模式
程式碼範例
\`\`\`
```

### 4. 驗證與確認 (選填，但建議包含)

- 驗證程式碼的建構指令
- Linting 與格式化工具
- 測試需求
- 驗證步驟

## 內容指南

### 寫作風格

- 使用清晰、簡潔的語言
- 使用祈使語氣 ("Use", "Implement", "Avoid")
- 具體且可執行
- 避免使用模糊詞彙，如 "should", "might", "possibly"
- 使用條列式清單以提升可讀性
- 保持區段聚焦且易於掃描

### 最佳實務

- **具體化**：提供具體範例而非抽象概念
- **說明原因**：在有價值時解釋建議背後的理由
- **使用表格**：用於比較選項、列出規則或顯示模式
- **包含範例**：實際程式碼片段比描述更有效
- **保持最新**：參考當前版本與最佳實務
- **連結資源**：包含官方文件與權威來源

### 指令高度 (Goldilocks Zone)

- 從完全定義預期成果的最小規則集開始
- 於觀察到失敗後新增約束，而非針對假設性的邊緣案例
- 偏好高訊號範例，勝過窮舉決策表

| 高度 | 失敗模式 | 結果 |
| --- | --- | --- |
| 過度指定 | 易碎的 if-else 敘述 | 無法處理未列出的情況 |
| 指定不足 | 假設共用 Context | 產出通用內容 |
| 適當高度 |啟發式 + 範例 | 穩定、可推廣的品質 |

### 常見包含模式

1. **命名慣例**：如何命名變數、函式、類別、檔案
2. **程式碼組織**：檔案結構、模組組織、匯入順序
3. **錯誤處理**：偏好的錯誤處理模式
4. **相依性**：如何管理與記錄相依性
5. **註解與文件**：何時及如何為程式碼編寫文件
6. **版本資訊**：目標語言/框架版本

## 遵循模式

### 條列式清單

```markdown
## 安全性最佳實務

- 處理前永遠驗證使用者輸入
- 使用參數化查詢以防止 SQL 注入
- 祕密儲存於環境變數，絕對不要寫入程式碼
- 實作正確的驗證與授權
- 為所有生產環境端點啟用 HTTPS
```

### 用於結構化資訊的表格

```markdown
## 常見問題

| 問題 | 解決方案 | 範例 |
| ---------------- | ------------------- | ----------------------------- |
| 魔術數字 | 使用具名常數 | `const MAX_RETRIES = 3` |
| 過度巢狀 | 提取函式 | 重構巢狀 if 敘述 |
| 硬編碼值 | 使用設定檔 | 將 API URL 存於設定檔 |
```

### 程式碼比較

```markdown
### 好的範例 - 使用 TypeScript 介面
\`\`\`typescript
interface User {
  id: string;
  name: string;
  email: string;
}

function getUser(id: string): User {
  // 實作
}
\`\`\`

### 壞的範例 - 使用 any 型別
\`\`\`typescript
function getUser(id: any): any {
  // 失去型別安全
}
\`\`\`
```

### 條件指引

```markdown
## 框架選擇

- **針對小型專案**：使用 Minimal API 方法
- **針對大型專案**：使用具備清晰分離的控制器架構
- **針對微服務**：考慮領域驅動設計模式
```

## 避免的模式

- **過於冗長的解釋**：保持簡潔且易於掃描
- **過時資訊**：永遠參考當前版本與實務
- **模糊的指引**：明確說明要做什麼或避免什麼
- **缺少範例**：抽象規則沒有具體程式碼範例
- **矛盾建議**：確保整份檔案的一致性
- **從文件複製貼上**：透過提煉與情境化來增加價值
- **假設性規則膨脹**：不要針對尚未發生的失敗新增規則

## 測試你的指令

在定案指令檔案前：

1. **使用 Copilot 測試**：在 VS Code 中以實際 Prompt 測試這些指令
2. **驗證範例**：確保程式碼範例正確且可執行無誤
3. **檢查 Glob 模式**：確認 `applyTo` 模式符合目標檔案

## 範例結構

以下是新指令檔案的最小範例結構：

```markdown
---
description: '簡述目的'
applyTo: '**/*.ext'
---

# 技術名稱開發

簡介與 Context。

## 一般指令

- 高階指南 1
- 高階指南 2

## 最佳實務

- 特定實務 1
- 特定實務 2

## 程式碼標準

### 命名慣例
- 規則 1
- 規則 2

### 檔案組織
- 結構 1
- 結構 2

## 常見模式

### 模式 1
說明與範例

\`\`\`language
程式碼範例
\`\`\`

### 模式 2
說明與範例

## 驗證

- 建構指令：`驗證指令`
- Linting：`lint 指令`
- 測試：`測試指令`
```

## 維護

- 相依性或框架更新時審查指令
- 更新範例以反映當前最佳實務
- 移除過時模式或已棄用功能
- 隨社群演進新增模式
- 隨著專案結構演進，保持 glob 模式準確
