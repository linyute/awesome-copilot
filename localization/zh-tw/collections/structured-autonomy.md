## 運作方式

結構化自主是一個三階段的工作流程，旨在最大化您從AI輔助開發中獲得的價值，同時保持高階請求的低量。該系統遵循一個簡單的原則：**將高階模型用於思考時應節省，將廉價模型用於執行時應自由。**

### 工作流程

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│    sa-plan      │ ──▶ │   sa-generate   │ ──▶ │   sa-implement  │
│  (1 request)    │     │   (1 request)   │     │  (many requests)│
│  Premium Model  │     │  Premium Model  │     │   Cheap Model   │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**階段 1: 規劃** — 您描述您想要建立的內容。規劃代理會研究您的程式碼庫，閱讀文件，並將您的功能分解為邏輯提交。輸出：`plans/{feature-name}/plan.md`


```
/plan Add a user profile page to this application that allows the user to view and edit their profile information. 
```

**階段 2: 生成** — 生成器會根據您的規劃，產生完整、可複製貼上的實作說明，並附有完整的程式碼區塊。沒有佔位符、沒有待辦事項——只有根據您實際程式碼庫模式運作的程式碼。輸出：`plans/{feature-name}/implementation.md`

```
/generate #plan.md
```

**階段 3: 實作** — 實作代理會逐步遵循生成的說明，逐一核對項目，並在定義的提交邊界處停止，以便您驗證和提交。

```
/implement #implementation.md
```

實作將在完成一次提交後停止，讓您驗證更改然後繼續。它將自動從下一步驟繼續。

### 一個分支，一個 PR

每個工作流程都在單一功能分支上操作，目標是單一的拉取請求。實作步驟直接映射到提交，為您提供：

- **具有邏輯、可審查提交的乾淨 Git 歷史記錄**
- **在繼續之前驗證程式碼是否正常運作的自然檢查點**
- **如果出現問題，輕鬆回溯**——只需重設到最後一個良好提交

### 這種方法為何有效

**成本效益** — 高階模型 (Claude Opus, GPT-5.1-Codex) 價格昂貴，但在推理和規劃方面表現出色。您只需使用它們兩次：一次用於理解您的請求，一次用於生成實作。實際的程式碼撰寫工作使用免費模型，可以根據需要進行多次迭代。

**更優的程式碼品質** — 由於規劃階段會首先研究您的程式碼庫，因此生成的程式碼會遵循您現有的模式、命名慣例和架構。您無需對抗不符合您專案的 AI 建議。

**開發人員參與度** — 帶有手動提交點的逐步實作讓您全程參與。您在每個步驟中審查和測試實際的更改，而不是在最後盯著巨大的差異，想知道發生了什麼。

**熟悉的工作流程** — 這不是一種新的工作方式——這是您現有的 Git 工作流程，並輔以 AI 協助。功能分支、增量提交、PR 審查。AI 處理繁瑣的部分，而您則掌控整個過程。
