# 產品需求文件 (PRD)

## 概覽

設計全面、生產等級的產品需求文件 (PRD)，以縮小業務願景與技術執行之間的差距。此技能適用於現代軟體系統，確保需求得到明確定義。

## 何時使用

在以下情況使用此技能：

- 開始新的產品或功能開發週期
- 將模糊的想法轉化為具體的技術規格
- 為人工智慧（AI）驅動的功能定義需求
- 利害關係人需要一個統一的專案範圍「單一事實來源」
- 使用者要求「編寫 PRD」、「記錄需求」或「規劃功能」

---

## 運作工作流

### 第一階段：探索（面談）

在編寫 PRD 的任何一行內容之前，你**必須**詢問使用者以填補資訊空白。不要假設背景資訊。

**詢問關於：**

- **核心問題**：我們現在為什麼要建構這個？
- **成功指標**：我們如何知道它奏效了？
- **限制條件**：預算、技術堆疊或截止日期？

### 第二階段：分析與範圍界定

綜合使用者的輸入。識別相依關係和隱藏的複雜性。

- 規劃**使用者流程 (User Flow)**。
- 定義**非目標 (Non-Goals)** 以保護開發時程。

### 第三階段：技術起草

使用下方的**嚴格 PRD 結構圖 (Schema)** 生成文件。

## PRD 品質標準

### 需求品質

使用具體、可衡量的標準。避免使用「快速」、「簡單」或「直觀」等詞彙。

```diff
# 模糊（差）
- 搜尋應該要很快，並回傳相關結果。
- 使用者介面 (UI) 必須看起來現代且易於使用。

# 具體（好）
+ 搜尋必須在 200 毫秒內針對 1 萬筆記錄的資料集回傳結果。
+ 搜尋演算法在基準評估中必須達到 >= 85% 的 Precision@10。
+ 使用者介面 (UI) 必須遵循 「Vercel/Next.js」 設計系統，並獲得 100 分的 Lighthouse 無障礙分數。
```

## 嚴格 PRD 結構圖 (Schema)

你**必須**遵循以下確切的輸出結構：

### 1. 摘要 (Executive Summary)

- **問題陳述**：用 1-2 句話說明痛點。
- **建議解決方案**：用 1-2 句話說明修復方法。
- **成功標準**：3-5 個可衡量的關鍵績效指標 (KPI)。

### 2. 使用者體驗與功能

- **使用者畫像 (User Personas)**：這是為誰設計的？
- **使用者故事 (User Stories)**：`身為一個 [使用者]，我想要 [動作]，以便於 [獲得益處]。`
- **驗收標準 (Acceptance Criteria)**：每個故事的「完成」定義列表。
- **非目標 (Non-Goals)**：我們**不**打算建構什麼？

### 3. AI 系統需求（如果適用）

- **工具需求**：需要哪些工具和 API？
- **評估策略**：如何衡量輸出的品質和準確性。

### 4. 技術規格

- **架構概覽**：資料流與元件互動。
- **整合點**：API、資料庫 (DB) 和身分驗證 (Auth)。
- **安全性與隱私**：資料處理與合規性。

### 5. 風險與路線圖 (Roadmap)

- **分階段推出**：最小可行產品 (MVP) -> v1.1 -> v2.0。
- **技術風險**：延遲、成本或相依性失敗。

## 實作指南

### 應該要做 (Always)

- **定義測試**：對於 AI 系統，指定如何測試和驗證輸出品質。
- **疊代**：提交草案並針對特定部分徵求意見回饋。

### 不要去做 (Avoid)

- **跳過探索階段**：在未先詢問至少 2 個澄清問題之前，切勿編寫 PRD。
- **憑空想像限制條件**：如果使用者沒有指定技術堆疊，請詢問或將其標記為 `待定 (TBD)`。

## 範例：智慧搜尋系統

### 1. 摘要

**問題**：使用者難以在海量儲存庫中找到特定的文件片段。
**解決方案**：一個智慧搜尋系統，提供帶有來源引用的直接答案。
**成功指標**：

- 減少 50% 的搜尋時間。
- 引用準確度 >= 95%。

### 2. 使用者故事

- **故事**：身為一名開發者，我想要詢問自然語言問題，這樣我就不必猜測關鍵字。
- **驗收標準 (AC)**：
  - 支援多輪澄清。
  - 回傳帶有「複製」按鈕的程式碼區塊。

### 3. AI 系統架構

- **所需工具**：`codesearch`、`grep`、`webfetch`。

### 4. 評估

- **基準 (Benchmark)**：使用 50 個常見的開發者問題進行測試。
- **通過率**：90% 必須符合預期的引用。
