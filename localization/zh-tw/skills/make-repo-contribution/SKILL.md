# 貢獻指南

幾乎每個專案都有一套大家在建立 issue、提取要求 (PR) 或以其他方式貢獻程式碼時需要遵循的貢獻指南。這些可能包括但不限於：

- 在建立 PR 之前先建立 issue，或兩者同時建立
- 必須根據所提出的變更要求使用的 issue 或 PR 範本
- 關於在這些 issue 和 PR 中需要記載內容的指南
- 在推送任何變更之前需要執行的測試、Linter 和其他先決條件

請永遠記住，你是別人存放庫中的客人。因此，在貢獻程式碼時，你需要遵循存放庫擁有者制定的規則和指南。

## 使用現有指南

在建立 PR 或其之前的任何步驟之前，請探索專案以確定是否有任何指引。可以探索的地方包括但不限於：

- README.md
- CONTRIBUTING.md
- 專案文件
- Issue 範本
- 提取要求或 PR 範本

如果存在任何這些文件，或者你在存放庫的其他地方發現了文件，請閱讀你發現的內容，考慮它，並盡你所能遵循指引。如果你有任何問題或困惑，請詢問使用者如何最好地進行。在確定你已遵循實作做法之前，切勿建立 PR。

## 未發現指南

如果未發現任何指引，或者未提供有關某些主題的指引，請使用以下內容作為建立高品質貢獻的基礎。**請務必**以存放庫中提供的指引為準。

## 任務

許多存放庫擁有者會提供在建立 PR 之前需要完成的先決步驟指引。這可以包括但不限於：

- 建構專案或產生資產
- 執行 Linter 並確保解決所有問題
- 命名指南和其他模式
- 需要建立並通過的單元測試、端對端測試或其他測試
  - 相關地，可能有要求的涵蓋率百分比

查看你發現的所有指引，並確保已滿足任何先決條件。

## Issue

請務必先查看是否存在與手頭任務相關的 issue。這可能已經由使用者或其他人建立。如果你發現了一個，請提示使用者以確保他們想要使用該 issue，或者他們希望使用哪一個。

如果未發現 issue，請查看指引以了解建立 issue 是否為必要條件。如果是，請使用存放庫中提供的範本。如果有多個，請選擇與正在進行的工作最一致的一個。如果有任何問題，請詢問使用者使用哪一個。

如果要求是提出 issue，但未提供 issue 範本，請使用[此 issue 範本](./assets/issue-template.md)作為提出內容的指南。

## 分支

在進行任何提交之前，請確保已為該工作建立了分支。遵循存放庫文件提供的任何指引。如果定義了前綴，例如 `feature` 或 `chore`，或者要求使用提出 PR 的人的使用者名稱，請使用該名稱。此分支絕不能是 `main` 或預設分支，而應是專門為正在進行的變更而建立的分支。如果尚未建立分支，請根據正在進行的變更和指引建立一個具有良好名稱的新分支。

## 提交

提交變更時：

1. 審核所有變更
2. 將變更邏輯上分組在一起
3. 為每個群組建立簡短的提交訊息，並遵循存放庫中的任何指引
4. 將分組的程式碼提交到分支。

## 合併

除非使用者明確指示，否則**絕不**合併到 main。

## 提取要求

建立提取要求時，請使用存放庫中現有的範本 (如果存在)，並遵循你發現的指引。

如果未提供範本，請使用[此 PR 範本](./assets/pr-template.md)。它包含一組要使用的標題，每個標題都有關於在特定章節中放置內容的指引。

如果建立或正在使用 issue，請確保在 PR 中引用該 issue。使用 `Closes #NUMBER` 語法來啟用 issue 的自動關閉。
