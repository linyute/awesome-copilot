# 會議紀錄技能 — 簡短內部會議
## 目的 / 概覽

此技能可為 60 分鐘或更短的內部會議產出高品質、一致的會議紀錄。輸出結果旨在清晰、具備行動力，且易於轉換為任務追蹤器（例如 GitHub Issues、Jira）。生成的會議紀錄優先考慮決策和行動項目，以便團隊能夠快速從討論轉向執行。

## 何時使用

在以下情況下使用此技能：

- 內部同步、站立會議、設計審查、分類、規劃或持續時間短的臨時會議
- 需要簡要記錄決策、指派行動項目和後續追蹤的情況
- 從現場會議、逐字稿、錄音或筆記建立標準化的會議紀錄文件

---

## 運作工作流

### 階段 1：攝入（起草前）

- 取得會議 Metadata：標題、日期、開始/結束時間（或持續時間）、召集人以及目標對象。
- 確認可用的輸入：議程、簡報、錄音、逐字稿或原始筆記。
- 如果缺少關鍵細節，在產出會議紀錄前最多詢問 3 個釐清問題（參見下方的「探索」）。

### 階段 2：擷取（會議期間 / 會後立即）

- 記錄出席者和缺席者。
- 為每個議程項目擷取簡短筆記，如果有的話請標註時間戳記。
- 記錄明確的決策、理由摘要（1-2 句話）以及行動項目（負責人 + 截止日期）。

### 階段 3：起草

- 按照**嚴格會議紀錄結構描述**（見下文）生成會議紀錄。
- 確保每個行動項目都包含負責人、截止日期（或時間範圍），以及適用的驗收標準。
- 在「待辦事項 (Parking Lot)」中標註未解決的問題或需要後續追蹤的項目。

### 階段 4：審查與發佈

- 如果可能，將草案發送給會議召集人或指定的審查人進行快速驗證（24 小時內）。
- 將最終會議紀錄發佈到協定的頻道（共用雲端硬碟、儲存庫、票據或電子郵件），並選擇性地在團隊的追蹤器中建立任務。

---

## 探索（必要的釐清問題）

在生成會議紀錄之前，如果缺少以下任何一項，代理程式**必須**詢問最多三個釐清問題：

- 會議標題、日期、開始時間（或持續時間）和召集人為何？
- 是否有議程或逐字稿/錄音可供參考？如果是，請提供。
- 誰應該被指派為會議紀錄的審查人或核准人？

如果使用者回應「無逐字稿」或「無議程」，請繼續執行，但將來源資料標記為「臨時筆記」並標註潛在的缺漏。

## 嚴格會議紀錄結構描述（輸出結構）

您**必須**按照此確切結構產出會議紀錄。如果資訊不可用，請使用 `待定` 或 `未知` 並說明如何取得。

### 1. Metadata

- **標題**：
- **日期 (YYYY-MM-DD)**：
- **開始時間 (UTC)**：
- **結束時間 (UTC) 或持續時間**：
- **召集人**：
- **地點 / 虛擬連結**：
- **會議紀錄作者**（代理程式或人員）：
- **發佈清單**（誰收到會議紀錄）：

### 2. 出席情況

- **出席**：[姓名 + 角色列表]
- **請假 / 缺席**：[列表]
- **記錄員**：[姓名或「代理程式」]

### 3. 議程

議程項目的點列式清單，依序排列：

- 項目 1：簡短標題
- 項目 2：簡短標題
- ...

### 4. 摘要

一段簡潔的摘要（1-3 句話），說明會議的目的和高階成果。
### 5. 已達成的決策

每項作為獨立的項目：

- **決策 1**：決策聲明。
  - 誰決定 / 核准：[姓名或群組]
  - 理由（1-2 句話）：簡短原因。
  - 生效日期（如果適用）：YYYY-MM-DD
- **決策 2**：...

### 6. 行動項目

表格樣式的項目；**必須包含負責人和截止日期**：

- **[ID] 行動**：簡短描述
  - **負責人**：姓名 (團隊)
  - **截止日期**：YYYY-MM-DD 或「儘快」/ 時間範圍
  - **驗收標準**：(完成此行動的條件)
  - **連結的成品 / 票據**：(選填 URL 或票據 ID)

**範例：**

- [A1] 為功能 X 起草部署執行手冊
  - 負責人：Alex (Engineering)
  - 截止日期：2026-02-05
  - 驗收標準：執行手冊包含回滾步驟、健康檢查和監控連結
  - 連結的成品：https://github.com/owner/repo/issues/123
### 7. 按議程項目分類的筆記

簡要、事實，時間戳記選填：

- **議程項目 1**：標題
  - 關鍵點：
    - 點 A (時間戳記 00:05)
    - 點 B (時間戳記 00:12)
  - 未決問題 / 提問：
    - Q1：問題文字 (如果已指派則註明負責人)
- **議程項目 2**：...

### 8. 待辦事項 (Parking Lot) / 未解決項目

- **項目**：簡短描述
  - 為何擱置 / 下一步：
  - 建議的負責人或下次會議解決

### 9. 風險 / 阻礙因素（如果有）

- **風險 1**：簡短描述、影響、緩解負責人
- **風險 2**：...

### 10. 下次會議 / 後續追蹤

- 提議的日期/時間（如果有）
- 下次會議的目標

### 11. 附件 / 參考資料

- 議程文件：URL
- 簡報：URL
- 逐字稿 / 錄音：URL
- 相關票據：URL 或 ID 清單

### 12. 版本與變更紀錄

- **版本**：1.0
- **最後更新**：YYYY-MM-DDTHH:MM:SSZ
- **變更**：關於編輯內容及編輯者的簡短筆記

## 風格與品質規則

- 保持會議紀錄簡潔：對於 <= 30 分鐘的會議，總長度通常應在 1 頁 A4 紙以內；對於接近 60 分鐘的會議，應在 2 頁以內。
- 使用簡單語言和點列式清單以提高可讀性。
- 將決策和行動項目置於文件頂部。
- 請勿包含推測性語言或未經證實的主張。如果不確定，請將其標註為 `待定` 並註明缺少的資訊來源。
- 使用一致的時間戳記和 ISO 8601 日期 (YYYY-MM-DD 或完整 UTC 時間戳記)。

---

## 應該 / 不應該 (DO / DON'T)

**應該：**

- 為每個行動項目包含負責人和截止日期。
- 盡可能為行動項目提供驗收標準。
- 連結到成品（票據、簡報、錄音）以實現可追溯性。
- 如果會議紀錄包含重大決策，請發送草案進行快速審查。

**不應該：**

- 遺漏決策或行動項目 — 這些是會議紀錄的核心價值。
- 將個人觀點與事實混淆。保持評論被清楚地標記為「觀點」或將其排除。
- 除非有必要且獲得授權，否則發佈討論期間收集的原始個人識別資訊 (PII)。

## 範例提示（適用於 Copilot / 代理程式）

**從逐字稿生成會議紀錄的提示：**

> 「根據以下會議逐字稿生成會議紀錄。會議標題：『平台每週同步會議』。日期：2026-02-10。持續時間：45 分鐘。召集人：Priya（平台負責人）。逐字稿：<貼上逐字稿>。遵循嚴格會議紀錄結構描述。強調決策，並在有暗示的情況下建立包含負責人和截止日期的行動項目。」

**從筆記生成會議紀錄的提示：**

> 「我有一份來自 30 分鐘設計審查的原始筆記。標題：『功能 Y 設計審查』。日期：2026-02-11。筆記：<貼上筆記>。按照嚴格會議紀錄結構描述產出簡潔的會議紀錄。如果缺少關鍵欄位，請提出最多 3 個釐清問題。」

## 生成會議紀錄的驗證與驗收標準

生成的會議紀錄文件在以下情況下是可接受的：

- 包含 Metadata、出席情況、決策和行動項目章節。
- 每個行動項目都有指派的負責人和截止日期或明確的時間範圍。
- 所有重大決策都已被擷取，並附有至少 1 行的理由。
- 附件或參考資料已被列出，或明確標記為「無」。
- 文件內容基於事實；不確定的項目標記為 `待定`。
