# C# 文件化最佳實踐

- 公用成員應使用 XML 註解進行文件化。
- 鼓勵文件化內部成員，特別是當它們複雜或不言自明時。

## 所有應用程式介面的指南

- 使用 `<summary>` 提供類型或成員功能的簡要單句描述。摘要應以現在式、第三人稱動詞開頭。
- 使用 `<remarks>` 提供額外資訊，其中可包含實作細節、使用說明或任何其他相關內容。
- 對於語言特定的關鍵字，例如 `null`、`true`、`false`、`int`、`bool` 等，請使用 `<see langword>`。
- 對於內嵌程式碼片段，請使用 `<c>`。
- 對於成員的使用範例，請使用 `<example>`。
  - 對於程式碼區塊，請使用 `<code>`。`<code>` 標籤應放置在 `<example>` 標籤內。使用 `language` 屬性新增程式碼範例的語言，例如 `<code language="csharp">`。
- 使用 `<see cref>` 在行內 (在句子中) 參考其他類型或成員。
- 對於線上文件「另請參閱」部分中對其他類型或成員的獨立 (不在句子中) 參考，請使用 `<seealso>`。
- 使用 `<inheritdoc/>` 從基底類別或介面繼承文件。
  - 除非有重大行為變更，在這種情況下您應文件化差異。

## 方法

- 使用 `<param>` 描述方法參數。
  - 描述應為名詞片語，不指定資料類型。
  - 以介紹性冠詞開頭。
  - 如果參數是旗標列舉，描述應以「指定...的列舉值的位元組合」開頭。
  - 如果參數是非旗標列舉，描述應以「指定...的列舉值之一」開頭。
  - 如果參數是布林值，措辭應為「`<see langword="true" />` 表示...；否則為 `<see langword="false" />`。」。
  - 如果參數是「out」參數，措辭應為「當此方法傳回時，包含...。此參數被視為未初始化。」。
- 使用 `<paramref>` 在文件中參考參數名稱。
- 使用 `<typeparam>` 描述泛型類型或方法中的類型參數。
- 使用 `<typeparamref>` 在文件中參考類型參數。
- 使用 `<returns>` 描述方法傳回的內容。
  - 描述應為名詞片語，不指定資料類型。
  - 以介紹性冠詞開頭。
  - 如果傳回類型是布林值，措辭應為「如果...則為 `<see langword="true" />`；否則為 `<see langword="false" />`。」。

## 建構函式

- 摘要措辭應為「初始化 <Class> 類別 [或結構] 的新實例。」。

## 屬性

- `<summary>` 應以：
  - 「取得或設定...」表示讀寫屬性。
  - 「取得...」表示唯讀屬性。
  - 「取得 [或設定] 指示是否...的值」表示傳回布林值的屬性。
- 使用 `<value>` 描述屬性的值。
  - 描述應為名詞片語，不指定資料類型。
  - 如果屬性有預設值，請在單獨的句子中新增，例如「預設為 `<see langword="false" />`」。
  - 如果值類型是布林值，措辭應為「如果...則為 `<see langword="true" />`；否則為 `<see langword="false" />`。預設為...」。

## 例外狀況

- 使用 `<exception cref>` 文件化建構函式、屬性、索引子、方法、運算子和事件擲回的例外狀況。
- 文件化成員直接擲回的所有例外狀況。
- 對於巢狀成員擲回的例外狀況，僅文件化使用者最可能遇到的例外狀況。
- 例外狀況的描述應描述擲回例外狀況的條件。
  - 省略句子開頭的「如果擲回...」或「如果...」。直接說明條件，例如「存取訊息佇列應用程式介面時發生錯誤。」
