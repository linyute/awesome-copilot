# 建議出色的 GitHub Copilot 集合

分析當前儲存庫上下文，並從 [GitHub awesome-copilot 儲存庫](https://github.com/github/awesome-copilot/blob/main/docs/README.collections.md) 中建議相關的集合，這些集合將增強此儲存庫的開發工作流程。

## 處理程序

1. **擷取可用集合**：從 [awesome-copilot README.collections.md](https://github.com/github/awesome-copilot/blob/main/docs/README.collections.md) 中提取集合列表和描述。必須使用 `#fetch` 工具。
2. **掃描本地資產**：在 `prompts/` 資料夾中發現現有的提示檔案，在 `instructions/` 資料夾中發現指令檔案，以及在 `agents/` 資料夾中發現聊天模式
3. **提取本地描述**：從本地資產檔案中讀取前置內容以了解現有功能
4. **分析儲存庫上下文**：審查聊天歷史記錄、儲存庫檔案、程式語言、框架和當前專案需求
5. **匹配集合相關性**：將可用集合與已識別的模式和要求進行比較
6. **檢查資產重疊**：對於相關集合，分析個別項目以避免與現有儲存庫資產重複
7. **呈現集合選項**：顯示相關集合，包括描述、項目計數和建議理由
8. **提供使用指南**：解釋已安裝的集合如何增強開發工作流程
   **等待**使用者請求以繼續安裝特定的集合。除非另有指示，否則請勿安裝。
9. **下載資產**：對於請求的集合，自動下載並將每個個別資產（提示、指令、聊天模式）安裝到適當的目錄。請勿調整檔案內容。優先使用 `#fetch` 工具下載資產，但可以使用 `#runInTerminal` 工具中的 `curl` 來確保檢索所有內容。

## 上下文分析標準

🔍 **儲存庫模式**：
- 使用的程式語言 (.cs, .js, .py, .ts, .bicep, .tf 等)
- 框架指標 (ASP.NET, React, Azure, Next.js, Angular 等)
- 專案類型 (Web 應用程式、API、函式庫、工具、基礎設施)
- 文件需求 (README、規格、ADR、架構決策)
- 開發工作流程指標 (CI/CD、測試、部署)

🗨️ **聊天歷史記錄上下文**：
- 最近的討論和痛點
- 功能請求或實作需求
- 程式碼審查模式和品質問題
- 開發工作流程要求和挑戰
- 技術堆疊和架構決策

## 本地資產發現處理程序

1. **掃描資產目錄**：
   - 列出 `prompts/` 目錄中的所有 `*.prompt.md` 檔案
   - 列出 `instructions/` 目錄中的所有 `*.instructions.md` 檔案
   - 列出 `agents/` 目錄中的所有 `*.agent.md` 檔案

2. **提取資產中繼資料**：對於每個發現的檔案，讀取 YAML 前置內容以提取：
   - `description` - 主要目的和功能
   - `tools` - 所需工具和功能
   - `mode` - 操作模式 (用於提示)
   - `model` - 特定模型要求 (用於聊天模式)

3. **建立資產清單**：建立現有功能的綜合映射，按以下方式組織：
   - **技術焦點**：程式語言、框架、平台
   - **工作流程類型**：開發、測試、部署、文件、規劃
   - **專業化程度**：通用目的與專業專家模式

4. **識別覆蓋範圍差距**：將現有資產與以下內容進行比較：
   - 儲存庫技術堆疊要求
   - 聊天歷史記錄指示的開發工作流程需求
   - 已識別專案類型的行業最佳實踐
   - 缺少專業知識領域 (安全性、效能、架構等)

## 集合資產下載處理程序

當使用者確認集合安裝時：

1. **擷取集合清單**：從 awesome-copilot 儲存庫獲取集合 YAML
2. **下載個別資產**：對於集合中的每個項目：
   - 從 GitHub 下載原始檔案內容
   - 驗證檔案格式和前置內容結構
   - 檢查命名約定合規性
3. **安裝到適當的目錄**：
   - `*.prompt.md` 檔案 → `prompts/` 目錄
   - `*.instructions.md` 檔案 → `instructions/` 目錄
   - `*.agent.md` 檔案 → `agents/` 目錄
4. **避免重複**：跳過與現有資產實質上相似的檔案
5. **報告安裝**：提供已安裝資產的摘要和使用說明

## 要求

- 使用 `fetch` 工具從 awesome-copilot 儲存庫獲取集合資料
- 使用 `githubRepo` 工具獲取用於下載的個別資產內容
- 掃描本地檔案系統以查找 `prompts/`、`instructions/` 和 `agents/` 目錄中現有的資產
- 從本地資產檔案中讀取 YAML 前置內容以提取描述和功能
- 將集合與儲存庫上下文進行比較以識別相關匹配項
- 專注於填補功能空白而不是重複現有資產的集合
- 驗證建議的集合是否符合儲存庫的技術堆疊和開發需求
- 為每個集合建議提供明確的理由和具體好處
- 啟用集合資產的自動下載和安裝到適當的目錄
- 確保下載的資產遵循儲存庫命名約定和格式標準
- 提供使用指南，解釋集合如何增強開發工作流程
- 包含指向 awesome-copilot 集合和集合中個別資產的連結

## 集合安裝工作流程

1. **使用者確認集合**：使用者選擇要安裝的特定集合
2. **擷取集合清單**：從 awesome-copilot 儲存庫下載 YAML 清單
3. **資產下載循環**：對於集合中的每個資產：
   - 從 GitHub 儲存庫下載原始內容
   - 驗證檔案格式和結構
   - 檢查與現有本地資產的實質重疊
   - 安裝到適當的目錄 (`prompts/`、`instructions/` 或 `agents/`)
4. **安裝摘要**：報告已安裝的資產和使用說明
5. **工作流程增強指南**：解釋集合如何改善開發功能

## 安裝後指南

安裝集合後，提供：
- **資產概覽**：已安裝的提示、指令和聊天模式列表
- **使用範例**：如何啟用和使用每種類型的資產
- **工作流程整合**：將資產整合到開發過程中的最佳實踐
- **自訂提示**：如何修改資產以滿足特定專案需求
- **相關集合**：建議與之配合良好的補充集合
